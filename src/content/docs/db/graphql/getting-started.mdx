# Getting Started with GraphQL

This guide walks you through setting up a GraphQL server with **Apollo Server 4**, writing your first schema, and executing queries.

## Installation

<Steps>
1. Create a new project and install dependencies:

   ```bash
   mkdir my-graphql-api
   cd my-graphql-api
   npm init -y
   npm install @apollo/server graphql
   npm install --save-dev typescript @types/node tsx
   ```

2. Initialize TypeScript:

   ```bash
   npx tsc --init
   ```

3. Update `tsconfig.json` to use `NodeNext` modules:

   ```json
   {
     "compilerOptions": {
       "target": "ES2022",
       "module": "NodeNext",
       "moduleResolution": "NodeNext",
       "outDir": "./dist",
       "strict": true
     }
   }
   ```

4. Add to `package.json`:

   ```json
   {
     "type": "module",
     "scripts": {
       "dev": "tsx watch src/index.ts",
       "build": "tsc",
       "start": "node dist/index.js"
     }
   }
   ```
</Steps>

## Minimal Apollo Server

```typescript
// src/index.ts
import { ApolloServer } from '@apollo/server';
import { startStandaloneServer } from '@apollo/server/standalone';

// 1. Define your schema using SDL (Schema Definition Language)
const typeDefs = `#graphql
  type Book {
    id: ID!
    title: String!
    author: String!
    publishedYear: Int
    genres: [String!]!
  }

  type Query {
    books: [Book!]!
    book(id: ID!): Book
  }

  type Mutation {
    addBook(title: String!, author: String!, publishedYear: Int): Book!
  }
`;

// In-memory data store
const books = [
  { id: '1', title: 'The Pragmatic Programmer', author: 'Dave Thomas', publishedYear: 1999, genres: ['programming'] },
  { id: '2', title: 'Clean Code', author: 'Robert C. Martin', publishedYear: 2008, genres: ['programming'] },
];
let nextId = 3;

// 2. Define resolvers — functions that return data for each field
const resolvers = {
  Query: {
    books: () => books,
    book: (_: unknown, args: { id: string }) =>
      books.find(b => b.id === args.id) ?? null,
  },
  Mutation: {
    addBook: (_: unknown, args: { title: string; author: string; publishedYear?: number }) => {
      const book = {
        id: String(nextId++),
        title: args.title,
        author: args.author,
        publishedYear: args.publishedYear ?? null,
        genres: [],
      };
      books.push(book);
      return book;
    },
  },
};

// 3. Create the server
const server = new ApolloServer({ typeDefs, resolvers });

// 4. Start the standalone server (includes a built-in HTTP server)
const { url } = await startStandaloneServer(server, {
  listen: { port: 4000 },
  context: async ({ req }) => ({
    // Put shared objects here (db connections, auth, etc.)
    authToken: req.headers.authorization,
  }),
});

console.log(`🚀 GraphQL server ready at ${url}`);
```

## SDL Type Definitions

The Schema Definition Language (SDL) is a human-readable syntax for defining your GraphQL schema.

```graphql
# ---- SCALAR TYPES ----
# Built-in: Int, Float, String, Boolean, ID
# Custom scalars (must be implemented in resolvers)
scalar Date
scalar DateTime
scalar JSON
scalar Upload

# ---- ENUM ----
enum OrderStatus {
  PENDING
  PROCESSING
  SHIPPED
  DELIVERED
  CANCELLED
}

enum SortOrder {
  ASC
  DESC
}

# ---- INTERFACE ----
interface Node {
  id: ID!
}

interface Timestamps {
  createdAt: DateTime!
  updatedAt: DateTime!
}

# ---- OBJECT TYPES ----
type User implements Node & Timestamps {
  id: ID!
  name: String!
  email: String!
  role: UserRole!
  posts(first: Int, after: String): PostConnection!
  createdAt: DateTime!
  updatedAt: DateTime!
}

type Post implements Node & Timestamps {
  id: ID!
  title: String!
  slug: String!
  body: String
  status: PostStatus!
  author: User!
  tags: [String!]!
  createdAt: DateTime!
  updatedAt: DateTime!
}

# ---- INPUT TYPES ----
# Used for mutation arguments — cannot have resolvers, cannot be circular
input CreateUserInput {
  name: String!
  email: String!
  password: String!
  role: UserRole = MEMBER
}

input UpdateUserInput {
  name: String
  email: String
}

# ---- UNION ----
union SearchResult = User | Post | Comment

# ---- ROOT TYPES ----
type Query {
  user(id: ID!): User
  users(first: Int = 10, after: String, search: String): UserConnection!
  post(slug: String!): Post
  search(query: String!): [SearchResult!]!
}

type Mutation {
  createUser(input: CreateUserInput!): User!
  updateUser(id: ID!, input: UpdateUserInput!): User!
  deleteUser(id: ID!): Boolean!
}

type Subscription {
  postPublished: Post!
  userOnline(userId: ID!): User!
}
```

## Your First Query

Once the server is running at `http://localhost:4000`, open **Apollo Sandbox** (automatically available in development):

```graphql
# Query — get all books
query GetAllBooks {
  books {
    id
    title
    author
    publishedYear
    genres
  }
}

# Query — get a specific book by ID
query GetBook($id: ID!) {
  book(id: $id) {
    id
    title
    author
  }
}

# Variables panel (JSON):
# { "id": "1" }

# Mutation — add a new book
mutation AddBook($title: String!, $author: String!) {
  addBook(title: $title, author: $author) {
    id
    title
    author
  }
}
```

## Introspection

GraphQL APIs are **self-documenting** through introspection. The server exposes its entire schema via the `__schema` and `__type` meta-fields.

```graphql
# Get the full schema
query IntrospectionQuery {
  __schema {
    types {
      name
      kind
      description
      fields {
        name
        type {
          name
          kind
        }
      }
    }
  }
}

# Inspect a specific type
query GetUserType {
  __type(name: "User") {
    name
    kind
    description
    fields {
      name
      description
      type {
        name
        kind
        ofType {
          name
          kind
        }
      }
    }
  }
}

# Get available query operations
query GetQueryFields {
  __schema {
    queryType {
      fields {
        name
        description
        args {
          name
          type { name kind }
          defaultValue
        }
      }
    }
  }
}
```

:::tip[Disable Introspection in Production]
Introspection reveals your entire schema, which can aid attackers in crafting targeted queries. Disable it in production:

```typescript
const server = new ApolloServer({
  typeDefs,
  resolvers,
  introspection: process.env.NODE_ENV !== 'production',
});
```
:::

## Non-Null and List Modifiers

The type modifier syntax controls nullability and lists:

```graphql
String     # Nullable string — can be null
String!    # Non-null string — never null
[String]   # Nullable list of nullable strings
[String!]  # Nullable list of non-null strings
[String]!  # Non-null list of nullable strings
[String!]! # Non-null list of non-null strings (most common for collections)
```

```graphql
type User {
  id: ID!              # Never null
  name: String!        # Never null
  bio: String          # Can be null (optional field)
  posts: [Post!]!      # Never null list, never null items (always returns array, possibly empty)
  tags: [String]       # Nullable list with nullable strings (avoid — too permissive)
}
```

## GraphQL over HTTP

### POST (Standard)

```bash
# POST request with JSON body
curl -X POST http://localhost:4000/graphql \
  -H "Content-Type: application/json" \
  -d '{
    "query": "query GetBook($id: ID!) { book(id: $id) { title author } }",
    "variables": { "id": "1" },
    "operationName": "GetBook"
  }'
```

### GET (for Queries)

```bash
# GET request — query must be URL-encoded
curl "http://localhost:4000/graphql?query=%7B%20books%20%7B%20title%20%7D%20%7D"

# With variables
curl "http://localhost:4000/graphql?query=query+GetBook(%24id%3AID!)%7Bbook(id%3A%24id)%7Btitle%7D%7D&variables=%7B%22id%22%3A%221%22%7D"
```

:::caution[Only Queries via GET]
Mutations must always use `POST`. Some GraphQL servers reject mutation requests via GET. The `graphql-http` spec requires this.
:::

## graphql-ws for Subscriptions

Subscriptions require a WebSocket connection. The `graphql-ws` protocol is the modern standard (replacing the older `subscriptions-transport-ws`).

```bash
npm install graphql-ws ws
```

```typescript
// src/index.ts — add subscriptions to Apollo Server
import { ApolloServer } from '@apollo/server';
import { expressMiddleware } from '@apollo/server/express4';
import { ApolloServerPluginDrainHttpServer } from '@apollo/server/plugin/drainHttpServer';
import { makeExecutableSchema } from '@graphql-tools/schema';
import { WebSocketServer } from 'ws';
import { useServer } from 'graphql-ws/lib/use/ws';
import express from 'express';
import http from 'http';

const app = express();
const httpServer = http.createServer(app);

const schema = makeExecutableSchema({ typeDefs, resolvers });

// WebSocket server for subscriptions
const wsServer = new WebSocketServer({ server: httpServer, path: '/graphql' });
const serverCleanup = useServer({ schema }, wsServer);

const server = new ApolloServer({
  schema,
  plugins: [
    ApolloServerPluginDrainHttpServer({ httpServer }),
    { async serverWillStart() {
      return { async drainServer() { await serverCleanup.dispose(); } };
    }},
  ],
});

await server.start();
app.use('/graphql', express.json(), expressMiddleware(server));

httpServer.listen(4000, () => console.log('🚀 Server ready at http://localhost:4000/graphql'));
```

## Persisted Queries

Persisted queries (also called Automatic Persisted Queries, APQ) store query documents by hash on the server to reduce request size and enable GET-based caching:

```typescript
// Client side (Apollo Client)
import { ApolloClient, InMemoryCache, HttpLink } from '@apollo/client';
import { createPersistedQueryLink } from '@apollo/client/link/persisted-queries';
import { sha256 } from 'crypto-hash';

const link = createPersistedQueryLink({ sha256 }).concat(
  new HttpLink({ uri: 'http://localhost:4000/graphql' })
);

const client = new ApolloClient({ cache: new InMemoryCache(), link });
// First request: sends hash → server misses → responds with PersistedQueryNotFound
// Client retries with full query + hash → server stores it
// Subsequent requests: sends hash only → server responds from cache
```

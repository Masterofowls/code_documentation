# GraphQL

GraphQL is a **query language for APIs** and a **server-side runtime** for executing those queries. Developed by Facebook (now Meta), it provides a complete and understandable description of your data, gives clients the power to ask for exactly what they need, makes it easier to evolve APIs over time, and enables powerful developer tools.

## History

GraphQL was developed internally at Facebook starting in **2012** to address data-fetching challenges in their mobile applications. The team needed a way to reduce the payload size and number of requests for their News Feed feature on early mobile devices.

Key milestones:

- **2012** — GraphQL development begins at Facebook to replace the existing REST-based approach for their iOS app.
- **2015** — Facebook **open-sources GraphQL** and publishes the specification.
- **2016** — GraphQL gains rapid adoption; first major third-party implementations appear.
- **2018** — The **GraphQL Foundation** is formed under the Linux Foundation, transferring GraphQL's neutrality away from Facebook.
- **2019** — GitHub migrates their public API to GraphQL (v4); Apollo achieves widespread adoption.
- **2021** — Apollo Federation 2 released with improved composition model.
- **2023** — The GraphQL specification continues to evolve with incremental delivery (@defer, @stream), and composite schemas.

## REST vs GraphQL

<Tabs>
  <TabItem label="The Core Problems REST Has">

**Over-fetching**: REST returns the entire resource even if you only need two fields:
```
GET /users/42
→ Returns: id, name, email, address, phone, bio, avatar, created_at, updated_at, ...
  (you only needed name and email)
```

**Under-fetching (N+1)**: Getting a user's posts and each post's author requires multiple round-trips:
```
GET /users/42          → user data
GET /users/42/posts    → list of post IDs
GET /posts/1           → post 1 data
GET /posts/2           → post 2 data
GET /users/7           → author of post 1
GET /users/9           → author of post 2
...
```

  </TabItem>
  <TabItem label="Comparison Table">

| Aspect | REST | GraphQL |
|---|---|---|
| **Data fetching** | Fixed endpoint → fixed response shape | Client declares exactly what it needs |
| **Over-fetching** | Common — server decides shape | Eliminated — client controls fields |
| **Under-fetching** | Requires multiple requests | Single request for nested data |
| **Versioning** | `/v1`, `/v2` endpoints | Schema evolution with `@deprecated` |
| **Type system** | No standard (OpenAPI is optional) | Strongly typed by spec |
| **Documentation** | Requires separate docs/OpenAPI | Self-documenting via introspection |
| **Tooling** | Mature (Postman, Swagger) | Rich (GraphiQL, Playground, Sandbox) |
| **Caching** | HTTP caching (GET requests) | More complex (requires persisted queries) |
| **File uploads** | Simple multipart form | Requires `graphql-multipart-request-spec` |
| **Real-time** | WebSockets or SSE separately | Subscriptions in spec |
| **Learning curve** | Lower | Higher (schema, resolvers, DataLoader) |
| **Performance** | Simple queries are faster | Complex nested queries can be more efficient |

  </TabItem>
</Tabs>

## GraphQL Operations

GraphQL has three fundamental operation types:

### Query

Read data from the server. Queries are **idempotent** — they do not modify state.

```graphql
query GetUser($id: ID!) {
  user(id: $id) {
    id
    name
    email
    posts(first: 5) {
      title
      publishedAt
    }
  }
}
```

### Mutation

Modify data on the server. Mutations are the equivalent of POST/PUT/PATCH/DELETE in REST.

```graphql
mutation CreatePost($input: CreatePostInput!) {
  createPost(input: $input) {
    id
    title
    slug
    createdAt
  }
}
```

### Subscription

Receive real-time updates from the server. Subscriptions maintain a persistent connection (typically WebSocket).

```graphql
subscription OnMessageAdded($channelId: ID!) {
  messageAdded(channelId: $channelId) {
    id
    text
    author {
      name
      avatar
    }
    createdAt
  }
}
```

## Schema-First vs Code-First

<Tabs>
  <TabItem label="Schema-First">

You write the **SDL (Schema Definition Language)** first, then implement resolvers to match.

```graphql
# schema.graphql
type User {
  id: ID!
  name: String!
  email: String!
  posts: [Post!]!
}

type Post {
  id: ID!
  title: String!
  author: User!
}

type Query {
  user(id: ID!): User
  posts: [Post!]!
}
```

**Pros:** Schema is the source of truth; teams can agree on API contract before implementation; great for frontend-backend collaboration.

**Cons:** Schema and resolvers can drift if not kept in sync.

  </TabItem>
  <TabItem label="Code-First">

You write code (usually with decorators or builder APIs) and the schema is **generated** from it.

```typescript
// TypeGraphQL example
@ObjectType()
class User {
  @Field(() => ID)
  id: string;

  @Field()
  name: string;

  @Field(() => [Post])
  posts: Post[];
}

@Resolver(() => User)
class UserResolver {
  @Query(() => User, { nullable: true })
  async user(@Arg('id', () => ID) id: string): Promise<User | null> {
    return UserService.findById(id);
  }
}
```

**Pros:** Single source of truth (TypeScript types = GraphQL schema); no schema/resolver drift; IDE autocompletion.

**Cons:** Requires the specific framework (TypeGraphQL, Pothos, Nexus); schema SDL is generated, not primary.

  </TabItem>
</Tabs>

## GraphQL Servers

<CardGrid>
  <Card title="Apollo Server (Node.js)" icon="seti:javascript">
    The most popular JavaScript/TypeScript GraphQL server. Works with any Node.js HTTP framework or standalone.

    ```bash
    npm install @apollo/server graphql
    ```

    Supports subscriptions, data sources, plugins, and federation.
  </Card>
  <Card title="GraphQL Yoga (Node.js)" icon="seti:javascript">
    Built by The Guild. Spec-compliant, works on Node.js, Deno, Bun, and edge runtimes. Uses `graphql-ws` for subscriptions.

    ```bash
    npm install graphql-yoga graphql
    ```
  </Card>
  <Card title="Strawberry (Python)" icon="seti:python">
    Modern Python GraphQL library using type hints and dataclasses. Code-first approach.

    ```bash
    pip install strawberry-graphql
    ```
  </Card>
  <Card title="Ariadne (Python)" icon="seti:python">
    Schema-first Python library. Bind resolvers to SDL schema types.

    ```bash
    pip install ariadne
    ```
  </Card>
  <Card title="graphql-go / gqlgen (Go)" icon="seti:go">
    `gqlgen` is the leading Go GraphQL library, code-first with type-safe resolvers generated from schema.

    ```bash
    go get github.com/99designs/gqlgen
    ```
  </Card>
  <Card title="Hasura / PostGraphile" icon="seti:db">
    **Hasura**: Auto-generates a GraphQL API from your Postgres schema with subscriptions and access control.

    **PostGraphile**: Similar approach, runs as Node.js middleware on top of Postgres.
  </Card>
</CardGrid>

## GraphQL Clients

<CardGrid>
  <Card title="Apollo Client" icon="seti:javascript">
    Full-featured client for React, Vue, Angular, and vanilla JS. Includes normalized cache, optimistic updates, subscriptions, and React hooks (`useQuery`, `useMutation`, `useSubscription`).

    ```bash
    npm install @apollo/client graphql
    ```
  </Card>
  <Card title="urql" icon="seti:javascript">
    Lightweight, extensible GraphQL client. "Universal React Query Library." Less opinionated than Apollo; exchanges system for customisation.

    ```bash
    npm install urql graphql
    ```
  </Card>
  <Card title="graphql-request" icon="seti:javascript">
    Minimal GraphQL client — just a fetch wrapper with response typing. Ideal when you don't need client-side caching.

    ```bash
    npm install graphql-request graphql
    ```
  </Card>
  <Card title="TanStack Query + fetcher" icon="seti:javascript">
    Use TanStack Query (React Query) with a custom fetcher like `graphql-request`. Gets you server state management without the complexity of Apollo Client.

    ```bash
    npm install @tanstack/react-query graphql-request
    ```
  </Card>
</CardGrid>

## When to Use GraphQL

GraphQL excels when:

- **Multiple clients** (web, mobile, TV) need different data shapes from the same API.
- You have **complex, nested data relationships** that would require many REST endpoints.
- You're building a **BFF (Backend for Frontend)** layer.
- Your frontend teams want **autonomy** to evolve their data requirements without backend changes.
- You need **real-time data** via subscriptions.
- You want **strong typing** and auto-generated documentation.
- Your API is consumed by **third-party developers** who benefit from introspection tools.

## When NOT to Use GraphQL

Consider alternatives when:

- You're building a **simple CRUD app** — REST with a good OpenAPI spec is easier.
- You need **simple HTTP caching** — REST GET requests cache transparently; GraphQL POST requests do not.
- You're serving **large file uploads or binary data** — REST multipart is simpler.
- Your team is **small and moving fast** — GraphQL has more upfront cost.
- You have **performance-critical, well-known query patterns** — a hand-tuned REST endpoint can be faster.
- You're building a **public API** that will be consumed by curl/wget users — REST is more universally accessible.

:::tip[Hybrid Approach]
Many production systems use GraphQL for their primary data API (especially for frontend clients) while keeping REST endpoints for specific use cases: file uploads, webhooks, health checks, and batch processing.
:::

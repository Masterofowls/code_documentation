# GraphQL

GraphQL is a **query language for APIs** and a **runtime for fulfilling those queries** with your existing data. Developed internally at Facebook (Meta) in 2012 and open-sourced in 2015, GraphQL gives clients the power to ask for exactly what they need — nothing more, nothing less.

## History

| Year | Event                                                                               |
| ---- | ----------------------------------------------------------------------------------- |
| 2012 | Facebook engineers create GraphQL internally to power the mobile News Feed rebuild  |
| 2015 | Facebook **open-sources GraphQL** with a specification and reference implementation |
| 2016 | GitHub adopts GraphQL for its public API (v4)                                       |
| 2018 | **GraphQL Foundation** formed under the Linux Foundation                            |
| 2019 | Apollo Federation released for microservices architectures                          |
| 2021 | GraphQL specification moves to annual release cadence                               |
| 2023 | GraphQL Composite Schemas working group (Federation standardization effort)         |

## REST vs GraphQL

<CardGrid>
  <Card title="REST Challenges" icon="warning">
    - **Over-fetching**: endpoint returns more data than needed -
    **Under-fetching**: one endpoint doesn't have enough; need N additional
    requests - **N+1 requests**: fetching a list then fetching details for each
    item - **Versioning complexity**: /v1/, /v2/ proliferation - **Rigid
    contracts**: client must use whatever shape the server defines
  </Card>
  <Card title="GraphQL Advantages" icon="approve-check">
    - Client asks for exactly the fields it needs - Nested queries eliminate N+1
    network round trips - Single endpoint, no versioning (add fields, deprecate
    old ones) - Strongly typed schema serves as living documentation -
    Introspection enables powerful tooling (autocomplete, Playground)
  </Card>
</CardGrid>

| Concern            | REST                                    | GraphQL                                     |
| ------------------ | --------------------------------------- | ------------------------------------------- |
| **Data fetching**  | Fixed response shape per endpoint       | Client specifies exact fields               |
| **Over-fetching**  | Common (unnecessary fields returned)    | Eliminated                                  |
| **Under-fetching** | Common (multiple requests needed)       | Eliminated (nested queries)                 |
| **Versioning**     | URL versioning or headers               | No versioning — additive changes only       |
| **Documentation**  | OpenAPI/Swagger (maintained separately) | Schema IS the documentation (introspection) |
| **Type safety**    | Optional (via OpenAPI codegen)          | Built-in schema types                       |
| **Caching**        | HTTP-native (GET cache, ETags)          | Requires custom caching strategy            |
| **File uploads**   | Multipart form data                     | Needs `graphql-multipart-request-spec`      |
| **Real-time**      | Polling, SSE, WebSocket                 | First-class Subscriptions                   |
| **Learning curve** | Low                                     | Moderate                                    |
| **N+1 problem**    | Handled at network level                | Requires DataLoader on server               |
| **Error format**   | HTTP status codes (4xx, 5xx)            | Always 200 (with `errors` array)            |

:::tip[GraphQL Doesn't Replace REST Everywhere]
GraphQL excels for data-fetching APIs consumed by multiple clients (web, mobile, partner integrations) with different data needs. REST remains excellent for simple CRUD, file serving, webhooks, and public APIs that need straightforward HTTP caching.
:::

## GraphQL Operations

GraphQL defines three operation types:

### Query

Read-only data fetching. Analogous to HTTP GET.

```graphql
query GetUser($id: ID!) {
  user(id: $id) {
    id
    name
    email
    posts(first: 5) {
      title
      publishedAt
    }
  }
}
```

### Mutation

Data modification (create, update, delete). Analogous to HTTP POST/PUT/PATCH/DELETE.

```graphql
mutation CreatePost($input: CreatePostInput!) {
  createPost(input: $input) {
    id
    title
    slug
    author {
      name
    }
  }
}
```

### Subscription

Real-time event streams over WebSocket.

```graphql
subscription OnMessageReceived($roomId: ID!) {
  messageReceived(roomId: $roomId) {
    id
    content
    sender {
      name
      avatarUrl
    }
    sentAt
  }
}
```

## Schema-First vs Code-First

| Approach         | Description                                                                     | When to Use                                                             |
| ---------------- | ------------------------------------------------------------------------------- | ----------------------------------------------------------------------- |
| **Schema-first** | Write SDL (`.graphql` files) first; generate/implement resolvers from schema    | Large teams, when schema is designed collaboratively, design-driven API |
| **Code-first**   | Define schema programmatically (TypeGraphQL, Nexus); SDL is generated from code | TypeScript-heavy projects, when you want type safety from DB to client  |

## GraphQL Servers

<CardGrid>
  <Card title="Apollo Server (Node.js)" icon="seti:javascript">
    The most widely adopted GraphQL server. Version 4 is standalone and
    framework-agnostic. Rich ecosystem (Apollo Studio, Federation, client
    caching). ``` npm install @apollo/server graphql ```
  </Card>
  <Card title="GraphQL Yoga (Node.js)" icon="seti:javascript">
    Lightweight, spec-compliant server from The Guild. Works with any HTTP
    framework. Excellent plugin system. ``` npm install graphql-yoga graphql ```
  </Card>
  <Card title="Mercurius (Fastify)" icon="seti:javascript">
    High-performance GraphQL adapter for Fastify. Excellent throughput
    benchmarks. JIT compilation of resolvers. ``` npm install mercurius graphql
    ```
  </Card>
  <Card title="Strawberry (Python)" icon="seti:python">
    Code-first Python library using type hints and dataclasses. Clean, Pythonic
    API. Supports async resolvers. ``` pip install strawberry-graphql ```
  </Card>
  <Card title="Ariadne (Python)" icon="seti:python">
    Schema-first Python library. Write SDL, then bind Python resolvers. Django,
    Flask, FastAPI integrations. ``` pip install ariadne ```
  </Card>
  <Card title="graphql-go" icon="seti:go">
    Popular Go implementation. Also consider `gqlgen` (code-first, generates Go
    types from schema) for type safety. ``` go get github.com/graphql-go/graphql
    ```
  </Card>
</CardGrid>

### No-Code / Backend-as-a-Service

- **Hasura** — auto-generates a GraphQL API from your PostgreSQL (or MySQL/MSSQL) schema. Permissions, subscriptions, remote schemas, and event triggers built in.
- **PostGraphile** — generates a GraphQL API from a PostgreSQL schema, leveraging PostgreSQL's own security model (row-level security).
- **Supabase** — built on PostgREST but now includes a GraphQL layer via `pg_graphql`.

## When to Use GraphQL

**GraphQL excels when:**

- Multiple clients (web app, mobile app, partner API) consume the same data but need different shapes
- The data model is deeply interconnected (social graphs, e-commerce with products/orders/users/reviews)
- You want a self-documenting API that stays in sync with implementation
- Rapid iteration is needed — add fields to the schema without breaking existing queries
- Real-time features (subscriptions) are needed alongside data fetching

**Consider REST instead when:**

- Simple CRUD with a 1:1 URL-to-resource mapping
- File uploads and downloads are the primary use case
- Heavy HTTP caching (CDN, ETags, Cache-Control) is required
- The consumer is another service that can share types directly (gRPC might be better)
- Team is small and the overhead of schema design isn't justified
- Public APIs where HTTP semantics (status codes, URLs) are important for discoverability

:::caution[GraphQL Is Not Magic]
GraphQL moves complexity from the client to the server. N+1 database queries don't disappear — they move server-side and require DataLoader. Authorization logic must be implemented in every resolver. Plan for these concerns before adopting GraphQL.
:::

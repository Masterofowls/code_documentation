# GraphQL

GraphQL is a query language for your API and a runtime for executing those queries. Created by Facebook, it gives clients the power to ask for exactly what they need and nothing more.

## History

GraphQL was developed internally at Facebook starting in 2012 to solve the challenges of building the Facebook mobile news feed. The core problem: REST APIs were returning too much data, requiring multiple round trips, and making mobile performance suffer on slow 3G networks.

| Year | Milestone |
|------|-----------|
| 2012 | Facebook develops GraphQL internally |
| 2015 | Facebook open-sources GraphQL |
| 2015 | First public spec published alongside React 2015 conference |
| 2016 | GitHub migrates to GraphQL for its v4 API |
| 2018 | GraphQL Foundation formed under Linux Foundation |
| 2019 | Federation specification released by Apollo |
| 2020 | Apollo Federation 2 and schema registry |
| 2023 | GraphQL spec v2023 with incremental delivery (@defer/@stream) |

## REST vs GraphQL

<CardGrid>
  <Card title="Over-fetching" icon="warning">
    **REST**: `GET /users/42` returns the entire user object with 30 fields when you only need name and avatar.

    **GraphQL**: Query exactly `{ user(id: 42) { name avatar } }` — only 2 fields returned.
</Card>
  <Card title="Under-fetching / N+1" icon="warning">
    **REST**: Display a user's posts with author names requires: `GET /users/42` → `GET /posts?userId=42` → N × `GET /users/{authorId}`.

    **GraphQL**: One query: `{ user(id: 42) { posts { title author { name } } } }`.
</Card>
</CardGrid>

### Detailed Comparison

| Concern | REST | GraphQL |
|---------|------|---------|
| **Data fetching** | Fixed endpoints return fixed shapes | Client specifies exact shape |
| **Over-fetching** | Common — fields you don't need | None — you ask for what you need |
| **Under-fetching** | Common — multiple round trips | Rare — one query for nested data |
| **Versioning** | `/v1/`, `/v2/` URL versioning | Add fields (deprecate old ones), no breaking changes |
| **Type system** | OpenAPI/Swagger (optional) | Built-in strongly-typed schema |
| **Introspection** | OpenAPI requires separate docs | Built-in — `__schema`, `__type` |
| **Real-time** | SSE, WebSockets (separate setup) | Subscriptions are first-class |
| **Caching** | HTTP cache (GET cacheable) | Complex — requires APQ or persisted queries |
| **File uploads** | Standard multipart | Non-standard (graphql-upload) |
| **Learning curve** | Lower — HTTP concepts | Higher — SDL, resolvers, DataLoader |
| **Tooling** | Mature (Postman, curl) | Excellent (GraphQL Playground, Apollo Studio) |
| **Error handling** | HTTP status codes | Always HTTP 200, errors in `errors` array |

:::caution[GraphQL Caching Complexity]
REST GET requests are automatically cacheable at the HTTP layer (CDN, browser). GraphQL queries are typically POST requests and require additional setup for caching: Apollo persisted queries, GET-based query caching, or operation-level CDN caching.
:::

## GraphQL Operations

### Query

Queries are read-only operations — the equivalent of GET in REST.

```graphql
# Simple query
query {
  user(id: "42") {
    name
    email
    avatar
  }
}

# Named query with variable
query GetUserProfile($userId: ID!) {
  user(id: $userId) {
    name
    email
    posts(last: 5) {
      title
      publishedAt
    }
  }
}
```

### Mutation

Mutations create, update, or delete data — equivalent to POST/PUT/PATCH/DELETE.

```graphql
mutation CreatePost($input: CreatePostInput!) {
  createPost(input: $input) {
    id
    title
    slug
    createdAt
  }
}
```

### Subscription

Subscriptions maintain a real-time connection and push updates when events occur.

```graphql
subscription OnNewMessage($channelId: ID!) {
  messageAdded(channelId: $channelId) {
    id
    text
    author {
      name
      avatar
    }
    createdAt
  }
}
```

## Schema-First vs Code-First

### Schema-First (SDL-First)

You write the GraphQL Schema Definition Language (SDL) first, then implement resolvers.

```graphql
# schema.graphql
type User {
  id: ID!
  name: String!
  email: String!
  posts: [Post!]!
}

type Post {
  id: ID!
  title: String!
  author: User!
}

type Query {
  user(id: ID!): User
  users: [User!]!
}
```

**Pros**: Schema is the single source of truth. Easy to share with frontend. Tooling generates types.
**Cons**: Schema and resolvers can get out of sync.

### Code-First

You write code (classes, decorators) and the schema is generated from it.

**Pros**: Single source of truth in code. Type safety in resolvers matches schema. No sync issues.
**Cons**: Generated SDL can be less readable. Harder for non-developers to read.

## When GraphQL Excels

✅ **Complex, interconnected data** — social graphs, CMS content trees, e-commerce catalogs

✅ **Multiple clients with different needs** — mobile app needs less data than web app

✅ **Rapid frontend iteration** — frontend teams can add fields without backend changes

✅ **Public developer APIs** — GitHub, Shopify, Stripe all use GraphQL for flexibility

✅ **Real-time features** — subscriptions for live feeds, chat, notifications

✅ **BFF pattern** — Backend-for-Frontend where each client gets a tailored API

## When NOT to Use GraphQL

❌ **Simple CRUD APIs** — REST is simpler and better-understood for basic operations

❌ **File upload heavy** — REST with multipart is simpler and more standardized

❌ **Aggressive HTTP caching needed** — CDN caching is much harder with GraphQL

❌ **Team unfamiliar with resolvers/DataLoader** — N+1 problems can cripple performance

❌ **Microservices with simple contracts** — REST/gRPC is often clearer

❌ **Batch processing / ETL** — Not designed for bulk data operations

:::tip[Start with REST, Add GraphQL When Needed]
For most new projects, start with REST. Add GraphQL when you have concrete problems: multiple clients with different data needs, frontend teams blocked on backend changes, or complex nested data requirements. Don't add the complexity of GraphQL without a clear need.
:::

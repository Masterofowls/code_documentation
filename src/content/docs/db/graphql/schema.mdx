# GraphQL Schema Design

The schema is the contract between your API and its clients. A well-designed schema is intuitive, stable, and easy to evolve.

## Scalar Types

### Built-in Scalars

| Scalar    | Description                              | JS Type   |
| --------- | ---------------------------------------- | --------- |
| `Int`     | 32-bit signed integer                    | `number`  |
| `Float`   | Double-precision floating-point          | `number`  |
| `String`  | UTF-8 string                             | `string`  |
| `Boolean` | `true` or `false`                        | `boolean` |
| `ID`      | Unique identifier (serialized as String) | `string`  |

### Custom Scalars

```graphql
# Declare custom scalars in SDL
scalar Date # ISO 8601 date: "2024-01-15"
scalar DateTime # ISO 8601 datetime: "2024-01-15T10:30:00Z"
scalar JSON # Arbitrary JSON value
scalar Upload # File upload (graphql-upload)
scalar URL # Valid URL string
scalar EmailAddress # Valid email address
scalar NonNegativeInt
scalar PositiveFloat
```

```javascript
// Implement custom scalars in resolvers
import { GraphQLScalarType, Kind } from 'graphql';

const DateScalar = new GraphQLScalarType({
  name: 'Date',
  description: 'ISO 8601 date string (YYYY-MM-DD)',

  // Called when sending to client: JS Date → string
  serialize(value) {
    if (value instanceof Date) {
      return value.toISOString().split('T')[0];
    }
    return value;
  },

  // Called when receiving as a variable: string → JS Date
  parseValue(value) {
    const date = new Date(value);
    if (isNaN(date.getTime())) {
      throw new Error(`Invalid date value: ${value}`);
    }
    return date;
  },

  // Called when receiving as a literal in the query
  parseLiteral(ast) {
    if (ast.kind === Kind.STRING) {
      return new Date(ast.value);
    }
    return null;
  },
});

// In resolvers
const resolvers = {
  Date: DateScalar,
  Query: { ... },
};
```

## Interfaces

An interface defines a set of fields that implementing types must include.

```graphql
interface Node {
  id: ID!
}

interface Timestamped {
  createdAt: DateTime!
  updatedAt: DateTime!
}

interface SearchResult {
  id: ID!
  title: String!
  snippet: String!
  url: URL!
}

# Implementing multiple interfaces
type Article implements Node & Timestamped & SearchResult {
  id: ID!
  title: String!
  snippet: String!
  url: URL!
  createdAt: DateTime!
  updatedAt: DateTime!
  # Additional fields beyond the interface
  body: String!
  author: User!
  category: Category!
  readTime: Int!
}

type Product implements Node & Timestamped & SearchResult {
  id: ID!
  title: String!
  snippet: String!
  url: URL!
  createdAt: DateTime!
  updatedAt: DateTime!
  price: Float!
}
```

```javascript
// Interface resolver: tells GraphQL which concrete type a result is
const resolvers = {
  SearchResult: {
    __resolveType(obj) {
      if (obj.price !== undefined) return 'Product';
      if (obj.body !== undefined) return 'Article';
      return null;
    },
  },
  Node: {
    __resolveType(obj, context, info) {
      return obj.__typename || null;
    },
  },
};
```

## Unions

A union represents one of several object types, with **no shared fields**.

```graphql
union SearchResult = Article | Product | User | Collection

union PaymentResult = PaymentSuccess | PaymentFailure | PaymentPending

type PaymentSuccess {
  transactionId: String!
  amount: Float!
  currency: String!
  receiptUrl: URL!
}

type PaymentFailure {
  code: String!
  message: String!
  declineReason: String
}

type PaymentPending {
  pendingReason: String!
  estimatedResolution: DateTime
}
```

```javascript
// Union resolver
const resolvers = {
  PaymentResult: {
    __resolveType(obj) {
      if (obj.transactionId) return 'PaymentSuccess';
      if (obj.declineReason !== undefined) return 'PaymentFailure';
      return 'PaymentPending';
    },
  },
};
```

Client query using union fragments:

```graphql
query ProcessPayment($input: PaymentInput!) {
  processPayment(input: $input) {
    ... on PaymentSuccess {
      transactionId
      receiptUrl
    }
    ... on PaymentFailure {
      code
      message
      declineReason
    }
    ... on PaymentPending {
      pendingReason
    }
  }
}
```

## Input Types

Input types are used exclusively for mutation arguments and complex filter arguments. They cannot reference output types and cannot have circular references.

```graphql
input CreateProductInput {
  sku: String!
  name: String!
  description: String
  price: Float!
  compareAtPrice: Float
  inventory: Int!
  categoryId: ID!
  tags: [String!]
}

input UpdateProductInput {
  name: String # all fields optional for update
  description: String
  price: Float
  inventory: Int
  categoryId: ID
  tags: [String!]
}

input ProductFilterInput {
  categoryId: ID
  minPrice: Float
  maxPrice: Float
  inStock: Boolean
  tags: [String!]
  search: String
}

input PaginationInput {
  first: Int
  after: String
  last: Int
  before: String
}

# Mutations using input types
type Mutation {
  createProduct(input: CreateProductInput!): CreateProductPayload!
  updateProduct(id: ID!, input: UpdateProductInput!): UpdateProductPayload!
}
```

## Non-Null and List Modifiers Summary

```graphql
type Example {
  # Nullability
  a: String # nullable; resolver can return null
  b: String! # non-null; resolver must return a string
  # Lists
  c: [String] # nullable list of nullable strings
  d: [String!] # nullable list of non-null strings
  e: [String]! # non-null list of nullable strings
  f: [String!]! # non-null list of non-null strings (most common)
  # Nested
  g: [[Int]] # nullable list of nullable lists of nullable ints
  h: [[Int!]!]! # non-null list of non-null lists of non-null ints
}
```

:::caution[Nullability Design Decision]
Be deliberate about nullability. Making everything non-null seems appealing but:

- A single null deep in a tree causes the entire parent to become null (null propagation)
- It makes it harder to add optional fields later without a breaking change
- Facebook's guidance: make fields nullable by default unless they are truly always present

The exception: `ID`, primary keys, and connection fields should generally be `!`.
:::

## Directives

### Built-in Directives

```graphql
# @deprecated — mark fields or enum values as deprecated
type User {
  id: ID!
  name: String!
  username: String! @deprecated(reason: "Use `name` field instead")
  email: String!
}

# @skip — conditionally skip a field (client-side)
query GetUser($includeEmail: Boolean!) {
  user(id: "1") {
    name
    email @skip(if: $includeEmail)
  }
}

# @include — conditionally include a field (client-side)
query GetUser($withPosts: Boolean!) {
  user(id: "1") {
    name
    posts @include(if: $withPosts) {
      title
    }
  }
}

# @specifiedBy — for custom scalar types (GraphQL 2021 spec)
scalar UUID @specifiedBy(url: "https://tools.ietf.org/html/rfc4122")
```

### Custom Directives

```graphql
# Declare custom directives in SDL
directive @auth(requires: UserRole = USER) on FIELD_DEFINITION | OBJECT
directive @rateLimit(max: Int!, window: String!) on FIELD_DEFINITION
directive @cacheControl(
  maxAge: Int
  scope: CacheControlScope
) on FIELD_DEFINITION

enum CacheControlScope {
  PUBLIC
  PRIVATE
}

type Query {
  publicPosts: [Post!]! @cacheControl(maxAge: 60, scope: PUBLIC)
  adminStats: AdminStats!
    @auth(requires: ADMIN)
    @rateLimit(max: 10, window: "1m")
  myProfile: User @auth
}
```

## Naming Conventions

| Element       | Convention                              | Example                                     |
| ------------- | --------------------------------------- | ------------------------------------------- |
| Types         | PascalCase                              | `UserProfile`, `OrderItem`                  |
| Fields        | camelCase                               | `firstName`, `createdAt`                    |
| Arguments     | camelCase                               | `userId`, `pageSize`                        |
| Enums         | PascalCase type, SCREAMING_SNAKE values | `enum OrderStatus { PENDING, IN_TRANSIT }`  |
| Input types   | PascalCase + `Input` suffix             | `CreateUserInput`                           |
| Payload types | PascalCase + `Payload` suffix           | `CreateUserPayload`                         |
| Connections   | PascalCase + `Connection` suffix        | `UserConnection`                            |
| Edges         | PascalCase + `Edge` suffix              | `UserEdge`                                  |
| Mutations     | verb + noun (imperative)                | `createUser`, `deletePost`, `updateProfile` |
| Queries       | noun (descriptive)                      | `user`, `users`, `featuredProducts`         |

## Pagination: Connections Spec

The **Relay Connections Specification** is the most widely adopted pagination pattern in GraphQL.

```graphql
# Connection types
type UserConnection {
  edges: [UserEdge]
  pageInfo: PageInfo!
  totalCount: Int!
}

type UserEdge {
  node: User!
  cursor: String!
}

type PageInfo {
  hasNextPage: Boolean!
  hasPreviousPage: Boolean!
  startCursor: String
  endCursor: String
}

# Query with cursor-based pagination
type Query {
  users(
    first: Int # forward pagination: take N after cursor
    after: String # cursor to paginate after
    last: Int # backward pagination: take N before cursor
    before: String # cursor to paginate before
    # Filters
    search: String
    role: UserRole
  ): UserConnection!
}
```

```graphql
# Client usage
query GetUsers($first: Int!, $after: String) {
  users(first: $first, after: $after) {
    edges {
      cursor
      node {
        id
        name
        email
      }
    }
    pageInfo {
      hasNextPage
      endCursor
    }
    totalCount
  }
}
```

```javascript
// Server-side resolver for connections
const resolvers = {
  Query: {
    users: async (
      _,
      { first = 20, after, last, before, search, role },
      { db },
    ) => {
      // Decode cursor (typically base64-encoded ID or offset)
      const afterId = after ? Buffer.from(after, 'base64').toString() : null;

      const whereClause = {
        ...(search && { name: { contains: search } }),
        ...(role && { role }),
        ...(afterId && { id: { gt: afterId } }),
      };

      const users = await db.user.findMany({
        where: whereClause,
        take: (first ?? 20) + 1, // fetch one extra to check hasNextPage
        orderBy: { id: 'asc' },
      });

      const hasNextPage = users.length > first;
      const nodes = hasNextPage ? users.slice(0, -1) : users;

      return {
        edges: nodes.map((user) => ({
          node: user,
          cursor: Buffer.from(user.id.toString()).toString('base64'),
        })),
        pageInfo: {
          hasNextPage,
          hasPreviousPage: !!after,
          startCursor: nodes[0]
            ? Buffer.from(nodes[0].id.toString()).toString('base64')
            : null,
          endCursor: nodes[nodes.length - 1]
            ? Buffer.from(nodes[nodes.length - 1].id.toString()).toString(
                'base64',
              )
            : null,
        },
        totalCount: await db.user.count({ where: whereClause }),
      };
    },
  },
};
```

## Schema Validation Rules

GraphQL validators enforce these rules at schema build time:

- All types referenced in a schema must be defined
- Field names within a type must be unique
- Argument names within a field must be unique
- Interface fields must be implemented by all implementing types
- Union members must be object types (not scalars, interfaces, or other unions)
- Input types cannot reference output types
- There must be no circular references in input types
- Subscription type must have exactly one root field per subscription operation

:::tip[Schema Linting]
Use `graphql-schema-linter` or `eslint-plugin-graphql` to enforce naming conventions, require descriptions, detect deprecated field usage, and other best practices in CI:

```bash
npm install graphql-schema-linter
npx graphql-schema-linter schema.graphql
```

:::

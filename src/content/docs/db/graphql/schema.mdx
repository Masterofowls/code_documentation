# GraphQL Schema Design

The schema is the contract between your server and clients. A well-designed schema is stable, expressive, and evolves gracefully. This section covers SDL deeply.

## Scalar Types

### Built-In Scalars

| Scalar | Description | Example |
|---|---|---|
| `Int` | 32-bit signed integer | `42` |
| `Float` | Double-precision float | `3.14` |
| `String` | UTF-8 string | `"hello"` |
| `Boolean` | `true` or `false` | `true` |
| `ID` | Unique identifier (serialized as String) | `"abc123"` |

The `ID` scalar is special: it's always serialized as a string in responses but can accept both strings and integers as arguments.

### Custom Scalars

```graphql
# Declare custom scalars in SDL
scalar Date      # YYYY-MM-DD
scalar DateTime  # ISO 8601: 2024-01-15T14:30:00Z
scalar JSON      # Arbitrary JSON value
scalar Upload    # For file uploads
scalar URL       # Validated URL string
scalar EmailAddress
scalar PositiveInt
```

```typescript
// Implement custom scalars in resolvers
import { GraphQLScalarType, Kind } from 'graphql';

const DateScalar = new GraphQLScalarType({
  name: 'Date',
  description: 'Date in YYYY-MM-DD format',

  // Serialize: convert internal value to JSON response
  serialize(value: unknown): string {
    if (value instanceof Date) {
      return value.toISOString().split('T')[0];
    }
    throw new Error('Date must be a Date instance');
  },

  // ParseValue: parse JSON input variable to internal value
  parseValue(value: unknown): Date {
    if (typeof value === 'string') {
      const d = new Date(value);
      if (isNaN(d.getTime())) throw new Error('Invalid date string');
      return d;
    }
    throw new Error('Date must be a string');
  },

  // ParseLiteral: parse inline literal in query document
  parseLiteral(ast): Date {
    if (ast.kind === Kind.STRING) {
      const d = new Date(ast.value);
      if (isNaN(d.getTime())) throw new Error('Invalid date literal');
      return d;
    }
    throw new Error('Date must be a string literal');
  },
});

// Use graphql-scalars library for common scalars
// npm install graphql-scalars
import { DateTimeResolver, EmailAddressResolver, URLResolver } from 'graphql-scalars';

const resolvers = {
  DateTime: DateTimeResolver,
  EmailAddress: EmailAddressResolver,
  URL: URLResolver,
  Date: DateScalar,
  Query: { ... },
};
```

## Object Types

Object types are the most common type â€” they represent an entity with fields.

```graphql
type User {
  id: ID!
  name: String!
  email: String!
  age: Int
  bio: String
  
  # Fields can take arguments
  posts(
    first: Int = 10
    after: String
    status: PostStatus
    orderBy: PostOrderByInput
  ): PostConnection!
  
  # Computed fields (resolved in code, not stored in DB)
  fullName: String!
  isOnline: Boolean!
  followerCount: Int!
}

type Post {
  id: ID!
  title: String!
  slug: String!
  body: String
  excerpt: String
  
  # Nested object types
  author: User!
  comments(first: Int): CommentConnection!
  
  # Custom scalar
  publishedAt: DateTime
  createdAt: DateTime!
  updatedAt: DateTime!
  
  status: PostStatus!
  tags: [String!]!
  
  # Metadata as key-value pairs
  metadata: JSON
}
```

## Interfaces

Interfaces define a set of fields that implementing types must include:

```graphql
interface Node {
  id: ID!
}

interface Timestamped {
  createdAt: DateTime!
  updatedAt: DateTime!
}

interface Searchable {
  searchRank: Float
  highlight: String
}

# A type can implement multiple interfaces
type User implements Node & Timestamped & Searchable {
  id: ID!
  name: String!
  email: String!
  createdAt: DateTime!
  updatedAt: DateTime!
  searchRank: Float
  highlight: String
}

type Post implements Node & Timestamped & Searchable {
  id: ID!
  title: String!
  body: String
  createdAt: DateTime!
  updatedAt: DateTime!
  searchRank: Float
  highlight: String
}

# Query using interface type
type Query {
  node(id: ID!): Node        # Returns any Node-implementing type
  search(query: String!): [Searchable!]!
}
```

```typescript
// Resolver must include __resolveType to determine concrete type
const resolvers = {
  Node: {
    __resolveType(obj: { __typename?: string }) {
      return obj.__typename ?? null;
    },
  },
  Searchable: {
    __resolveType(obj: { title?: string; name?: string }) {
      if ('title' in obj) return 'Post';
      if ('name' in obj) return 'User';
      return null;
    },
  },
};
```

## Unions

Unions allow a field to return one of several object types. Unlike interfaces, union members don't share any fields.

```graphql
union SearchResult = User | Post | Comment | Tag

union MutationResult = User | UserError

type UserError {
  message: String!
  field: String
  code: ErrorCode!
}

type Query {
  search(query: String!): [SearchResult!]!
}

type Mutation {
  createUser(input: CreateUserInput!): MutationResult!
}
```

```graphql
# Client must use inline fragments to access type-specific fields
query Search($q: String!) {
  search(query: $q) {
    ... on User {
      id
      name
      email
    }
    ... on Post {
      id
      title
      slug
    }
    ... on Comment {
      id
      text
    }
  }
}

# Union mutation result pattern
mutation CreateUser($input: CreateUserInput!) {
  createUser(input: $input) {
    ... on User {
      id
      name
    }
    ... on UserError {
      message
      field
      code
    }
  }
}
```

## Enum Types

```graphql
enum UserRole {
  ADMIN
  MODERATOR
  MEMBER
  GUEST
}

enum PostStatus {
  DRAFT
  PUBLISHED
  ARCHIVED
}

enum SortOrder {
  ASC
  DESC
}

# Enums in input types
input PostFilterInput {
  status: PostStatus
  authorId: ID
  tags: [String!]
  orderBy: PostSortField
  order: SortOrder = DESC
}

enum PostSortField {
  CREATED_AT
  UPDATED_AT
  TITLE
  VIEW_COUNT
}
```

## Input Types

Input types are used exclusively as argument types â€” they cannot have resolvers, computed fields, or circular references to output types.

```graphql
input CreatePostInput {
  title: String!
  body: String
  status: PostStatus = DRAFT
  tags: [String!]
  authorId: ID!
  publishAt: DateTime
}

input UpdatePostInput {
  # All fields optional for partial updates
  title: String
  body: String
  status: PostStatus
  tags: [String!]
  publishAt: DateTime
}

input PaginationInput {
  first: Int
  after: String
  last: Int
  before: String
}

# Nested input types
input AddressInput {
  street: String!
  city: String!
  state: String
  country: String!
  postalCode: String!
}

input CreateUserInput {
  name: String!
  email: String!
  password: String!
  address: AddressInput
}
```

:::caution[Input Types Cannot Reference Output Types]
Input types can only reference other input types, scalars, and enums. You cannot use `User` (an output type) as a field type inside an `input` type. Create separate `input` types that mirror the shape you need.
:::

## Directives

### Built-in Directives

```graphql
# @deprecated â€” mark field as deprecated with a reason
type User {
  id: ID!
  name: String!
  fullName: String! @deprecated(reason: "Use `name` instead")
  username: String! @deprecated(reason: "Use `name` instead. Removed in v3.")
}

# @skip â€” skip a field if condition is true (client-side)
query GetUser($id: ID!, $includeEmail: Boolean!) {
  user(id: $id) {
    name
    email @skip(if: $includeEmail)  # Skip email when includeEmail=true
  }
}

# @include â€” include a field only if condition is true
query GetUser($id: ID!, $withPosts: Boolean!) {
  user(id: $id) {
    name
    posts @include(if: $withPosts) {
      title
    }
  }
}
```

### Custom Directives

```graphql
# Declare in schema
directive @auth(requires: UserRole = MEMBER) on FIELD_DEFINITION
directive @rateLimit(max: Int!, window: String!) on FIELD_DEFINITION
directive @cacheControl(maxAge: Int) on FIELD_DEFINITION | OBJECT

type Query {
  me: User @auth
  adminData: AdminData @auth(requires: ADMIN)
  publicFeed: [Post!]! @cacheControl(maxAge: 60)
  expensiveOp: Result! @rateLimit(max: 10, window: "1m")
}
```

## Schema Extension

Use `extend type` to split a schema across multiple files or add fields in federation:

```graphql
# base.graphql
type Query {
  _empty: String  # Placeholder required in some tools
}

# users.graphql
extend type Query {
  user(id: ID!): User
  users: [User!]!
}

# posts.graphql
extend type Query {
  post(slug: String!): Post
  posts(filter: PostFilterInput): [Post!]!
}

extend type Mutation {
  createPost(input: CreatePostInput!): Post!
}
```

## Naming Conventions

| Item | Convention | Example |
|---|---|---|
| Types | `PascalCase` | `UserProfile`, `OrderItem` |
| Fields | `camelCase` | `firstName`, `createdAt` |
| Enums | `PascalCase` type, `SCREAMING_SNAKE_CASE` values | `UserRole { SUPER_ADMIN }` |
| Input types | `PascalCase` with `Input` suffix | `CreateUserInput` |
| Mutations | Verb + Noun pattern | `createUser`, `updatePost`, `deleteComment` |
| Query fields | Noun (singular or plural) | `user`, `users`, `post`, `posts` |
| Arguments | `camelCase` | `userId`, `orderBy` |

## Pagination: Connections Spec

The **Relay Connections Specification** is the GraphQL community standard for pagination:

```graphql
# Connection types
type PostConnection {
  edges: [PostEdge!]!
  pageInfo: PageInfo!
  totalCount: Int!
}

type PostEdge {
  node: Post!
  cursor: String!  # Opaque cursor for this position
}

type PageInfo {
  hasNextPage: Boolean!
  hasPreviousPage: Boolean!
  startCursor: String      # Cursor of first edge
  endCursor: String        # Cursor of last edge
}

# Query using connection
type Query {
  posts(
    first: Int         # Forward pagination: number of items
    after: String      # Forward pagination: cursor to start after
    last: Int          # Backward pagination: number of items
    before: String     # Backward pagination: cursor to start before
    filter: PostFilterInput
    orderBy: PostSortField
    order: SortOrder
  ): PostConnection!
}
```

```graphql
# Client query using connection
query GetPosts($after: String) {
  posts(first: 10, after: $after) {
    edges {
      cursor
      node {
        id
        title
        slug
        publishedAt
        author {
          name
        }
      }
    }
    pageInfo {
      hasNextPage
      endCursor
    }
    totalCount
  }
}
```

:::tip[Why Cursor Pagination?]
Cursor-based pagination is stable â€” if items are added or removed between page requests, cursor pagination won't skip or repeat items. Offset-based pagination (`page: 2, perPage: 10`) suffers from drift when the underlying data changes. Use cursor-based pagination for any large or frequently-updated collection.
:::

## Schema Validation Rules

GraphQL enforces these rules at build time:

1. **All types referenced must be defined** in the schema.
2. **Interface fields must match exactly** in implementing types.
3. **Input types cannot circularly reference** themselves (directly or through other input types).
4. **Enum values must be unique** within an enum.
5. **Field names must be unique** within a type.
6. **Arguments must have unique names** within a field.
7. **Subscription type must have exactly one root field**.

```bash
# Validate your schema
npx graphql-inspector validate schema.graphql
```

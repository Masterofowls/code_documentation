# MySQL Advanced

This section covers MySQL's programmability features, JSON capabilities, partitioning strategies, replication architecture, and deep InnoDB internals.

## Views

A view is a stored `SELECT` statement that can be queried like a table.

```sql
-- Create a basic view
CREATE VIEW active_users AS
SELECT id, name, email, created_at
FROM users
WHERE deleted_at IS NULL;

-- Query the view
SELECT * FROM active_users WHERE created_at > '2024-01-01';

-- Create or replace
CREATE OR REPLACE VIEW active_users AS
SELECT id, name, email, last_login, created_at
FROM users
WHERE deleted_at IS NULL AND is_active = 1;

-- WITH CHECK OPTION: prevent writes through the view that would make the row invisible
CREATE VIEW high_value_orders AS
SELECT * FROM orders WHERE total > 1000
WITH CHECK OPTION;

-- This will ERROR because the inserted row would not be visible through the view:
INSERT INTO high_value_orders (user_id, total) VALUES (1, 50);  -- ERROR

-- Updatable views (must meet all conditions):
-- 1. Based on a single table
-- 2. No DISTINCT, GROUP BY, HAVING, UNION
-- 3. No subqueries in WHERE that reference the same table
-- 4. No aggregate functions
-- 5. No subqueries in SELECT list

-- Show view definition
SHOW CREATE VIEW active_users\G

-- List all views in current database
SELECT TABLE_NAME, VIEW_DEFINITION
FROM information_schema.VIEWS
WHERE TABLE_SCHEMA = DATABASE();

-- Drop a view
DROP VIEW IF EXISTS active_users;
```

## Stored Procedures

Stored procedures encapsulate SQL logic on the server, reducing network round-trips and centralizing business logic.

```sql
-- Change the delimiter so semicolons inside the procedure don't end the statement
DELIMITER $$

CREATE PROCEDURE get_user_orders(
  IN  p_user_id  INT,
  IN  p_status   VARCHAR(50),
  OUT p_total    DECIMAL(10,2)
)
BEGIN
  -- DECLARE variables at the top of the BEGIN block
  DECLARE v_count INT DEFAULT 0;
  DECLARE v_done  BOOLEAN DEFAULT FALSE;
  DECLARE v_order_id INT;
  DECLARE v_amount   DECIMAL(10,2);
  
  -- Cursor for iterating rows
  DECLARE order_cursor CURSOR FOR
    SELECT id, total FROM orders
    WHERE user_id = p_user_id
      AND (p_status IS NULL OR status = p_status);
  
  -- Handler: set v_done=TRUE when cursor runs out of rows
  DECLARE CONTINUE HANDLER FOR NOT FOUND SET v_done = TRUE;
  
  -- Initialize output parameter
  SET p_total = 0;
  
  OPEN order_cursor;
  
  read_loop: LOOP
    FETCH order_cursor INTO v_order_id, v_amount;
    IF v_done THEN
      LEAVE read_loop;
    END IF;
    
    SET p_total = p_total + v_amount;
    SET v_count = v_count + 1;
  END LOOP read_loop;
  
  CLOSE order_cursor;
  
  -- Return result set
  SELECT id, total, status, created_at
  FROM orders
  WHERE user_id = p_user_id
    AND (p_status IS NULL OR status = p_status)
  ORDER BY created_at DESC;
  
END$$

DELIMITER ;

-- Call the procedure
CALL get_user_orders(42, 'delivered', @user_total);
SELECT @user_total AS total_spent;

-- Procedure with WHILE loop
DELIMITER $$
CREATE PROCEDURE populate_calendar(IN p_start DATE, IN p_end DATE)
BEGIN
  DECLARE v_date DATE DEFAULT p_start;
  WHILE v_date <= p_end DO
    INSERT IGNORE INTO calendar (date_value, year, month, day_of_week)
    VALUES (v_date, YEAR(v_date), MONTH(v_date), DAYOFWEEK(v_date));
    SET v_date = DATE_ADD(v_date, INTERVAL 1 DAY);
  END WHILE;
END$$
DELIMITER ;

-- Error handling in procedures
DELIMITER $$
CREATE PROCEDURE safe_transfer(IN from_id INT, IN to_id INT, IN amount DECIMAL(10,2))
BEGIN
  DECLARE EXIT HANDLER FOR SQLEXCEPTION
  BEGIN
    ROLLBACK;
    RESIGNAL;  -- Re-raise the error to caller
  END;
  
  START TRANSACTION;
    UPDATE accounts SET balance = balance - amount WHERE id = from_id;
    UPDATE accounts SET balance = balance + amount WHERE id = to_id;
  COMMIT;
END$$
DELIMITER ;
```

## Stored Functions

Stored functions return a single value and can be used in SQL expressions.

```sql
DELIMITER $$

CREATE FUNCTION calculate_age(p_birth_date DATE)
RETURNS INT
DETERMINISTIC        -- Same input always gives same output (important for replication)
READS SQL DATA       -- Function reads from tables (NO SQL | CONTAINS SQL | MODIFIES SQL DATA)
BEGIN
  RETURN TIMESTAMPDIFF(YEAR, p_birth_date, CURDATE());
END$$

-- Use in queries like a built-in function
SELECT name, calculate_age(birth_date) AS age
FROM users
WHERE calculate_age(birth_date) >= 18;

-- Function with table access
CREATE FUNCTION get_user_email(p_user_id INT)
RETURNS VARCHAR(255)
READS SQL DATA
BEGIN
  DECLARE v_email VARCHAR(255);
  SELECT email INTO v_email FROM users WHERE id = p_user_id LIMIT 1;
  RETURN v_email;
END$$

DELIMITER ;

-- Drop
DROP FUNCTION IF EXISTS calculate_age;

-- List stored procedures and functions
SELECT ROUTINE_TYPE, ROUTINE_NAME, CREATED, LAST_ALTERED
FROM information_schema.ROUTINES
WHERE ROUTINE_SCHEMA = DATABASE()
ORDER BY ROUTINE_TYPE, ROUTINE_NAME;
```

:::tip[DETERMINISTIC Matters for Replication]
Always declare a function `DETERMINISTIC` if the same inputs always produce the same output (no calls to `NOW()`, `RAND()`, etc.). Non-deterministic functions on a replication master using statement-based binary logging require `log_bin_trust_function_creators=1`, which is a security concern.
:::

## Triggers

Triggers fire automatically in response to `INSERT`, `UPDATE`, or `DELETE` events.

```sql
DELIMITER $$

-- AFTER INSERT: audit log when a new user registers
CREATE TRIGGER after_user_insert
AFTER INSERT ON users
FOR EACH ROW
BEGIN
  INSERT INTO audit_log (table_name, action, record_id, new_values, created_at)
  VALUES ('users', 'INSERT', NEW.id,
    JSON_OBJECT('email', NEW.email, 'name', NEW.name),
    NOW());
END$$

-- BEFORE UPDATE: automatically set updated_at
CREATE TRIGGER before_order_update
BEFORE UPDATE ON orders
FOR EACH ROW
BEGIN
  SET NEW.updated_at = NOW();
  
  -- Prevent status from going backwards
  IF OLD.status = 'delivered' AND NEW.status != 'delivered' THEN
    SIGNAL SQLSTATE '45000'
      SET MESSAGE_TEXT = 'Cannot change status of a delivered order';
  END IF;
END$$

-- BEFORE DELETE: soft-delete enforcement
CREATE TRIGGER before_user_delete
BEFORE DELETE ON users
FOR EACH ROW
BEGIN
  -- Archive to deleted_users table before deletion
  INSERT INTO deleted_users (user_id, email, name, deleted_at)
  VALUES (OLD.id, OLD.email, OLD.name, NOW());
END$$

DELIMITER ;

-- NEW = the new row value (available in INSERT and UPDATE)
-- OLD = the old row value (available in UPDATE and DELETE)

-- MySQL trigger limitations:
-- Cannot call a stored procedure that returns a result set
-- Cannot use CALL with transactions in some cases
-- One trigger per event/timing combination per table (no stacking in older versions)
-- MySQL 5.7+ allows multiple triggers per event

-- Show triggers
SHOW TRIGGERS FROM mydb\G
SELECT TRIGGER_NAME, EVENT_MANIPULATION, EVENT_OBJECT_TABLE, ACTION_TIMING
FROM information_schema.TRIGGERS
WHERE TRIGGER_SCHEMA = DATABASE();

-- Drop a trigger
DROP TRIGGER IF EXISTS after_user_insert;
```

## Events (Scheduled Jobs)

MySQL Events let you schedule SQL tasks to run automatically.

```sql
-- Enable the event scheduler (must be ON)
SHOW VARIABLES LIKE 'event_scheduler';
SET GLOBAL event_scheduler = ON;

-- One-time event
CREATE EVENT cleanup_old_sessions
ON SCHEDULE AT NOW() + INTERVAL 1 HOUR
DO DELETE FROM sessions WHERE expires_at < NOW();

-- Recurring event: run every day at 2 AM
CREATE EVENT daily_cleanup
ON SCHEDULE EVERY 1 DAY
STARTS '2024-01-16 02:00:00'
ON COMPLETION PRESERVE    -- Keep the event after it finishes (otherwise it's dropped)
ENABLE
COMMENT 'Purge expired sessions and temp data'
DO
BEGIN
  DELETE FROM sessions WHERE expires_at < NOW() - INTERVAL 7 DAY;
  DELETE FROM temp_exports WHERE created_at < NOW() - INTERVAL 24 HOUR;
  INSERT INTO maintenance_log (task, ran_at) VALUES ('daily_cleanup', NOW());
END;

-- Disable/enable an event
ALTER EVENT daily_cleanup DISABLE;
ALTER EVENT daily_cleanup ENABLE;

-- Show events
SHOW EVENTS FROM mydb\G

-- Drop event
DROP EVENT IF EXISTS daily_cleanup;
```

## MySQL JSON Functions

```sql
-- Sample data
CREATE TABLE config (
  id   INT AUTO_INCREMENT PRIMARY KEY,
  name VARCHAR(100),
  data JSON
);

INSERT INTO config (name, data) VALUES ('app_settings', '{
  "theme": "dark",
  "notifications": {"email": true, "sms": false},
  "limits": {"api_calls": 1000, "storage_mb": 5120},
  "tags": ["beta", "premium"]
}');

-- Extract values
SELECT
  data->'$.theme'                          AS theme,            -- Returns JSON (quoted)
  data->>'$.theme'                         AS theme_text,       -- Returns string (unquoted)
  JSON_EXTRACT(data, '$.notifications.email') AS email_notifs,
  data->'$.limits.api_calls'               AS api_calls,
  data->'$.tags[0]'                        AS first_tag
FROM config;

-- Modify JSON
UPDATE config SET data = JSON_SET(data,
  '$.theme', 'light',
  '$.limits.api_calls', 2000
) WHERE name = 'app_settings';

-- JSON_INSERT (only inserts, doesn't overwrite)
-- JSON_REPLACE (only replaces, doesn't insert)
-- JSON_REMOVE (removes a key)
UPDATE config SET data = JSON_REMOVE(data, '$.notifications.sms');

-- Build JSON values
SELECT
  JSON_ARRAY(1, 2, 3, 'four'),              -- [1, 2, 3, "four"]
  JSON_OBJECT('name', 'Alice', 'age', 30),  -- {"name": "Alice", "age": 30}
  JSON_MERGE_PATCH('{"a":1}', '{"b":2}'),  -- {"a":1,"b":2}
  JSON_PRETTY(data)                         -- Formatted output
FROM config;

-- Search within JSON
SELECT * FROM config WHERE JSON_CONTAINS(data->'$.tags', '"beta"');
SELECT * FROM config WHERE JSON_CONTAINS_PATH(data, 'one', '$.notifications.sms');

-- Aggregate to JSON
SELECT
  JSON_ARRAYAGG(name)                     AS names_array,
  JSON_OBJECTAGG(name, id)               AS name_id_map
FROM config;

-- JSON_TABLE: parse JSON into a relational result
SELECT jt.*
FROM config,
JSON_TABLE(data->'$.tags', '$[*]'
  COLUMNS (tag VARCHAR(50) PATH '$')
) AS jt
WHERE config.name = 'app_settings';

-- Validate JSON
SELECT JSON_VALID('{"key": "value"}');   -- 1
SELECT JSON_VALID('not json');            -- 0

-- JSON type of a value
SELECT JSON_TYPE(data->'$.theme');             -- STRING
SELECT JSON_TYPE(data->'$.limits.api_calls'); -- INTEGER
SELECT JSON_TYPE(data->'$.tags');             -- ARRAY
```

## Partitioning

Partitioning splits a large table into smaller physical pieces while maintaining a single logical interface.

```sql
-- RANGE partitioning: partition by ranges of column values
CREATE TABLE orders_partitioned (
  id         INT NOT NULL,
  user_id    INT NOT NULL,
  total      DECIMAL(10,2),
  created_at DATETIME NOT NULL,
  PRIMARY KEY (id, created_at)  -- Partition key must be in PRIMARY KEY
)
PARTITION BY RANGE (YEAR(created_at)) (
  PARTITION p2021 VALUES LESS THAN (2022),
  PARTITION p2022 VALUES LESS THAN (2023),
  PARTITION p2023 VALUES LESS THAN (2024),
  PARTITION p2024 VALUES LESS THAN (2025),
  PARTITION p_future VALUES LESS THAN MAXVALUE
);

-- LIST partitioning: partition by discrete values
CREATE TABLE orders_by_region (
  id        INT NOT NULL,
  region_id INT NOT NULL,
  total     DECIMAL(10,2),
  PRIMARY KEY (id, region_id)
)
PARTITION BY LIST (region_id) (
  PARTITION p_na   VALUES IN (1, 2, 3),    -- North America
  PARTITION p_eu   VALUES IN (4, 5, 6, 7), -- Europe
  PARTITION p_apac VALUES IN (8, 9, 10)    -- Asia-Pacific
);

-- HASH partitioning: distribute by hash of column
CREATE TABLE sessions (
  id      BIGINT NOT NULL,
  user_id INT NOT NULL,
  data    TEXT,
  PRIMARY KEY (id, user_id)
)
PARTITION BY HASH (user_id)
PARTITIONS 8;

-- KEY partitioning: like HASH but uses MySQL's internal hash function
-- Useful when you can't control the key

-- Partition pruning: MySQL only scans relevant partitions
EXPLAIN SELECT * FROM orders_partitioned WHERE YEAR(created_at) = 2023;
-- Should show "partitions: p2023"

-- Manage partitions
ALTER TABLE orders_partitioned ADD PARTITION (
  PARTITION p2025 VALUES LESS THAN (2026)
);
ALTER TABLE orders_partitioned DROP PARTITION p2021;

-- Truncate a partition efficiently (much faster than DELETE)
ALTER TABLE orders_partitioned TRUNCATE PARTITION p2021;

-- View partition info
SELECT PARTITION_NAME, TABLE_ROWS, DATA_LENGTH
FROM information_schema.PARTITIONS
WHERE TABLE_SCHEMA = 'mydb' AND TABLE_NAME = 'orders_partitioned';
```

:::caution[Partitioning Limitations]
- Foreign keys are not supported on partitioned tables.
- All unique indexes (including PRIMARY KEY) must include the partition key.
- Partitioning doesn't always improve performance — measure first.
- Partition pruning only works when the `WHERE` clause uses the partition expression directly.
:::

## Replication

MySQL replication copies data from a source (master) server to one or more replicas (slaves).

```sql
-- === SOURCE SERVER CONFIGURATION ===
-- In my.cnf:
-- server-id = 1
-- log_bin = /var/log/mysql/mysql-bin.log
-- binlog_format = ROW

-- Create replication user on source
CREATE USER 'replicator'@'%' IDENTIFIED WITH mysql_native_password BY 'replpass';
GRANT REPLICATION SLAVE ON *.* TO 'replicator'@'%';
FLUSH PRIVILEGES;

-- Get binary log position (for traditional replication)
SHOW MASTER STATUS;
-- +------------------+----------+
-- | File             | Position |
-- +------------------+----------+
-- | mysql-bin.000003 | 1234     |

-- === REPLICA SERVER CONFIGURATION ===
-- server-id = 2

-- Traditional (file+position based) replication
CHANGE REPLICATION SOURCE TO
  SOURCE_HOST     = '192.168.1.10',
  SOURCE_USER     = 'replicator',
  SOURCE_PASSWORD = 'replpass',
  SOURCE_LOG_FILE = 'mysql-bin.000003',
  SOURCE_LOG_POS  = 1234;

START REPLICA;
SHOW REPLICA STATUS\G

-- === GTID-Based Replication (recommended) ===
-- On source my.cnf:
-- gtid_mode = ON
-- enforce_gtid_consistency = ON

-- On replica:
CHANGE REPLICATION SOURCE TO
  SOURCE_HOST            = '192.168.1.10',
  SOURCE_USER            = 'replicator',
  SOURCE_PASSWORD        = 'replpass',
  SOURCE_AUTO_POSITION   = 1;  -- Use GTIDs automatically

START REPLICA;

-- Check replication lag
SHOW REPLICA STATUS\G
-- Look for: Seconds_Behind_Source (0 = fully caught up)

-- Binlog formats comparison:
-- STATEMENT: logs the SQL statement (smaller logs, non-deterministic functions are risky)
-- ROW: logs the actual row changes (larger logs, fully deterministic, safest)
-- MIXED: uses STATEMENT normally, falls back to ROW for non-deterministic operations
```

## InnoDB Internals

### MVCC (Multi-Version Concurrency Control)

InnoDB uses MVCC to allow readers and writers to work concurrently without blocking each other.

- Each row has hidden columns: `DB_TRX_ID` (transaction that last modified it) and `DB_ROLL_PTR` (pointer to undo log entry).
- When a transaction reads a row, it sees the version that was committed before the transaction started (snapshot read).
- Write locks only block other writers, not readers.
- The **undo log** stores previous row versions needed for active transactions.

### Redo Log vs Undo Log

| Log | Purpose | Location |
|---|---|---|
| **Redo log** | Crash recovery — replay committed changes after crash | `ib_logfile0`, `ib_logfile1` |
| **Undo log** | MVCC (old row versions), ROLLBACK support | Undo tablespace files |

### Buffer Pool

The InnoDB Buffer Pool is the primary memory cache for data and index pages.

```sql
-- View buffer pool status
SHOW ENGINE INNODB STATUS\G

-- Buffer pool hit rate (should be > 99%)
SELECT
  (1 - (variable_value / (
    SELECT variable_value FROM performance_schema.global_status
    WHERE variable_name = 'Innodb_buffer_pool_read_requests'
  ))) * 100 AS buffer_pool_miss_rate_pct
FROM performance_schema.global_status
WHERE variable_name = 'Innodb_buffer_pool_reads';

-- Innodb_buffer_pool_read_requests: total read requests
-- Innodb_buffer_pool_reads: requests that had to go to disk
```

### Change Buffer

The change buffer is a special data structure that caches changes to secondary index pages when those pages are not in the buffer pool. This avoids expensive random I/O for non-unique index maintenance.

### Adaptive Hash Index (AHI)

InnoDB automatically creates hash indexes on heavily-accessed B-Tree index pages in memory. This provides O(1) lookups for those pages. AHI is fully automatic and maintained in memory only.

## Performance Schema

```sql
-- Top queries by total latency
SELECT
  DIGEST_TEXT,
  COUNT_STAR,
  ROUND(SUM_TIMER_WAIT / 1e12, 3) AS total_latency_s,
  ROUND(AVG_TIMER_WAIT / 1e9, 3)  AS avg_latency_ms,
  SUM_ROWS_EXAMINED
FROM performance_schema.events_statements_summary_by_digest
ORDER BY SUM_TIMER_WAIT DESC
LIMIT 20;

-- Table I/O waits
SELECT OBJECT_NAME, COUNT_READ, COUNT_WRITE,
  ROUND(SUM_TIMER_READ / 1e12, 3) AS read_latency_s
FROM performance_schema.table_io_waits_summary_by_table
WHERE OBJECT_SCHEMA NOT IN ('performance_schema','information_schema','mysql','sys')
ORDER BY SUM_TIMER_READ DESC LIMIT 10;

-- Lock waits
SELECT * FROM performance_schema.data_lock_waits\G
```

## sys Schema

The `sys` schema (available since MySQL 5.7.7) provides human-readable views built on Performance Schema:

```sql
-- Statements by total latency with formatted output
SELECT * FROM sys.statements_with_runtimes_in_95th_percentile LIMIT 10;

-- Queries not using indexes
SELECT * FROM sys.statements_with_full_table_scans LIMIT 10;

-- Tables with the most I/O
SELECT * FROM sys.io_global_by_file_by_latency LIMIT 10;

-- InnoDB buffer pool pages by table
SELECT * FROM sys.innodb_buffer_stats_by_table ORDER BY pages DESC LIMIT 10;

-- Memory usage by thread
SELECT * FROM sys.memory_by_thread_by_current_bytes LIMIT 10;

-- Redundant indexes
SELECT * FROM sys.schema_redundant_indexes WHERE table_schema = 'mydb';

-- Unused indexes
SELECT * FROM sys.schema_unused_indexes WHERE object_schema = 'mydb';
```

# MySQL Indexes

Indexes are the single most important factor in MySQL query performance. A well-indexed schema can make queries thousands of times faster; a missing index on a large table causes a full table scan on every query.

## Index Types

### B-Tree (Default)

InnoDB uses B-tree indexes for PRIMARY, UNIQUE, INDEX, and KEY. B-tree indexes work for:

- Equality: `WHERE col = val`
- Range: `WHERE col BETWEEN a AND b`, `WHERE col > val`
- Prefix: `WHERE col LIKE 'prefix%'` (NOT `'%suffix'`)
- Sorting: `ORDER BY col` (if index order matches)
- Covering: all needed columns are in the index

```sql
-- Basic B-tree index
CREATE INDEX idx_email ON users(email);
ALTER TABLE users ADD INDEX idx_email (email);

-- Composite B-tree index
CREATE INDEX idx_status_created ON orders(status, created_at);

-- Unique index
CREATE UNIQUE INDEX idx_email_unique ON users(email);
ALTER TABLE users ADD UNIQUE (email);

-- Index on expression (MySQL 8.0+)
CREATE INDEX idx_lower_email ON users((LOWER(email)));
```

### HASH Indexes

Available only in the MEMORY storage engine. Hash indexes are O(1) for exact equality lookups but **cannot** be used for range queries or sorting. InnoDB has an _adaptive hash index_ (automatically built in-memory for frequently accessed pages) that cannot be created manually.

```sql
-- HASH index only in MEMORY tables
CREATE TABLE session_cache (
  session_id CHAR(32) NOT NULL,
  data TEXT,
  expires_at DATETIME,
  PRIMARY KEY (session_id)
) ENGINE=MEMORY;

CREATE INDEX idx_session USING HASH ON session_cache(session_id);
```

### FULLTEXT Indexes

For full-text search across large text columns. Available in InnoDB (MySQL 5.6+) and MyISAM.

```sql
CREATE TABLE articles (
  id INT AUTO_INCREMENT PRIMARY KEY,
  title VARCHAR(500),
  body LONGTEXT,
  FULLTEXT INDEX idx_ft_content (title, body)
) ENGINE=InnoDB;

-- Add FULLTEXT to existing table
ALTER TABLE articles ADD FULLTEXT INDEX idx_ft_body (body);
```

### SPATIAL Indexes

For geometric data types (GEOMETRY, POINT, LINESTRING, POLYGON). Requires NOT NULL columns and uses R-tree indexing.

```sql
CREATE TABLE locations (
  id INT AUTO_INCREMENT PRIMARY KEY,
  name VARCHAR(255),
  coordinates POINT NOT NULL,
  SPATIAL INDEX idx_coordinates (coordinates)
) ENGINE=InnoDB;
```

## InnoDB and the Clustered Index

Understanding how InnoDB stores data is crucial to understanding index performance.

### Primary Key = Clustered Index

In InnoDB, **the primary key IS the table**. The entire row data is stored in the B-tree leaf pages of the primary key index. There is no separate "heap" storage — all data is physically ordered by the primary key.

```sql
-- Good: compact integer PK — rows are stored in insertion order
CREATE TABLE orders (
  id BIGINT UNSIGNED NOT NULL AUTO_INCREMENT,
  user_id INT UNSIGNED NOT NULL,
  -- ...
  PRIMARY KEY (id)
);

-- Bad: UUID as PK causes random I/O and index fragmentation
-- Random UUID inserts scatter across the B-tree, causing page splits
CREATE TABLE orders (
  id CHAR(36) NOT NULL DEFAULT (UUID()),  -- random ordering!
  -- ...
  PRIMARY KEY (id)
);
-- If you must use UUID: use UUID_TO_BIN(UUID(), 1) to store as ordered BINARY(16)
-- The second argument `1` swaps time_hi to improve sequential inserts
id BINARY(16) NOT NULL DEFAULT (UUID_TO_BIN(UUID(), 1)),
PRIMARY KEY (id)
```

If a table has no PRIMARY KEY, InnoDB uses the first UNIQUE NOT NULL index. If neither exists, InnoDB silently creates a hidden 6-byte rowid (GEN_CLUST_INDEX) that you cannot query or index on.

:::caution[Always Define a Primary Key]
Always explicitly define a PRIMARY KEY. Tables without a primary key:

- Use a hidden rowid you can't reference in queries
- Have no natural clustering order
- Can cause performance issues in replication and GROUP BY operations
  :::

### Secondary Indexes in InnoDB

Every secondary index in InnoDB stores the **primary key value** as the row pointer (not a physical page pointer). This means:

1. Looking up a row via a secondary index requires two B-tree traversals: first the secondary index (to find the PK), then the primary key index (to fetch the row). This is called a **"double lookup"** or **"bookmark lookup"**.
2. A large primary key means **all secondary indexes are larger** (they each store a copy of the PK).

```sql
-- This secondary index lookup requires 2 B-tree traversals:
SELECT * FROM orders WHERE status = 'pending';
-- Step 1: Traverse idx_status B-tree → find PK values where status='pending'
-- Step 2: For each PK found, traverse the clustered index to get the full row

-- A covering index eliminates step 2:
CREATE INDEX idx_status_covering ON orders(status, user_id, created_at, total);
SELECT user_id, created_at, total FROM orders WHERE status = 'pending';
-- Step 1: Traverse idx_status_covering → all needed columns are IN the index
-- No step 2 needed!
```

## Creating and Managing Indexes

```sql
-- Create index
CREATE INDEX idx_name ON table_name(col1, col2);
CREATE UNIQUE INDEX idx_name ON table_name(col1);
ALTER TABLE table_name ADD INDEX idx_name (col1);
ALTER TABLE table_name ADD UNIQUE (col1, col2);

-- Drop index
DROP INDEX idx_name ON table_name;
ALTER TABLE table_name DROP INDEX idx_name;

-- Show indexes on a table
SHOW INDEXES FROM orders\G
SHOW INDEX FROM orders FROM myapp\G

-- Detailed index info from information_schema
SELECT
  INDEX_NAME,
  SEQ_IN_INDEX,
  COLUMN_NAME,
  NULLABLE,
  INDEX_TYPE,
  CARDINALITY
FROM information_schema.STATISTICS
WHERE TABLE_SCHEMA = 'myapp' AND TABLE_NAME = 'orders'
ORDER BY INDEX_NAME, SEQ_IN_INDEX;
```

## Composite Indexes and the Leftmost Prefix Rule

A composite (multi-column) index on `(a, b, c)` can be used for queries that filter on:

- `a` alone
- `a, b` together
- `a, b, c` together
- `a, c` — **only the `a` part** (MySQL can't skip columns)
- ❌ `b` alone (leftmost prefix rule violated)
- ❌ `c` alone
- ❌ `b, c` together

```sql
CREATE INDEX idx_composite ON orders(status, user_id, created_at);

-- ✅ Uses the full index
SELECT * FROM orders WHERE status = 'pending' AND user_id = 42 AND created_at > '2024-01-01';

-- ✅ Uses first two columns (status, user_id)
SELECT * FROM orders WHERE status = 'pending' AND user_id = 42;

-- ✅ Uses first column only (status)
SELECT * FROM orders WHERE status = 'pending' ORDER BY created_at;
-- Actually: can use (status) part for filtering, then filesort for ORDER BY
-- Better: use ORDER BY created_at only when status is equality, then index covers it

-- ❌ Does NOT use the index (user_id is not leftmost)
SELECT * FROM orders WHERE user_id = 42;

-- ❌ Does NOT use the index
SELECT * FROM orders WHERE created_at > '2024-01-01';
```

### Selectivity and Column Order

```sql
-- Check cardinality (higher = more selective = better leftmost column)
SELECT
  COLUMN_NAME,
  CARDINALITY,
  CARDINALITY / TABLE_ROWS AS selectivity
FROM information_schema.STATISTICS s
JOIN information_schema.TABLES t USING (TABLE_SCHEMA, TABLE_NAME)
WHERE TABLE_SCHEMA = 'myapp' AND TABLE_NAME = 'orders';

-- Rule of thumb: put the most selective column FIRST in composite indexes
-- Exception: if both columns appear in WHERE, order doesn't matter much for filtering
-- But ORDER BY column ordering matters for sort optimization
```

## Covering Indexes

A covering index contains **all columns** needed by a query (in SELECT, WHERE, ORDER BY, GROUP BY). The query can be answered entirely from the index without touching the primary key B-tree.

```sql
-- Example: frequently run report query
SELECT user_id, status, SUM(total) AS revenue, COUNT(*) AS order_count
FROM orders
WHERE created_at BETWEEN '2024-01-01' AND '2024-12-31'
GROUP BY user_id, status;

-- Covering index for this query
CREATE INDEX idx_covering_report
  ON orders (created_at, user_id, status, total);
-- All columns in WHERE (created_at), GROUP BY (user_id, status),
-- and SELECT aggregate (total) are in the index
```

## Invisible Indexes (MySQL 8.0+)

Invisible indexes exist and are maintained by the optimizer but are **not used by the query optimizer**. This lets you test the impact of removing an index without actually dropping it.

```sql
-- Make an index invisible
ALTER TABLE orders ALTER INDEX idx_status INVISIBLE;

-- Query optimizer now ignores idx_status
-- Run your queries and check EXPLAIN

-- Make it visible again
ALTER TABLE orders ALTER INDEX idx_status VISIBLE;

-- Or create invisible from the start
CREATE INDEX idx_test ON orders(status) INVISIBLE;

-- Force a specific session to use invisible indexes (for testing)
SET SESSION optimizer_switch = 'use_invisible_indexes=on';
```

## Functional (Expression) Indexes (MySQL 8.0+)

```sql
-- Index on a function expression
CREATE INDEX idx_lower_email ON users((LOWER(email)));
-- Now this query can use the index:
SELECT * FROM users WHERE LOWER(email) = 'alice@example.com';

-- Index on JSON path
CREATE INDEX idx_json_price ON products((CAST(data->>'$.price' AS DECIMAL(10,2))));

-- Multi-valued index for JSON arrays (MySQL 8.0.17+)
CREATE INDEX idx_tags ON articles ((CAST(tags AS CHAR(100) ARRAY)));
SELECT * FROM articles WHERE JSON_CONTAINS(tags, '"mysql"');
```

## FULLTEXT Search

```sql
-- Create FULLTEXT index
ALTER TABLE articles ADD FULLTEXT idx_ft (title, body);

-- Natural language mode (default): relevance ranking
SELECT id, title,
  MATCH(title, body) AGAINST ('mysql performance tuning') AS relevance
FROM articles
WHERE MATCH(title, body) AGAINST ('mysql performance tuning')
ORDER BY relevance DESC;

-- Boolean mode: precise control
SELECT id, title FROM articles
WHERE MATCH(title, body) AGAINST (
  '+mysql +index -mongodb "full text"'
  IN BOOLEAN MODE
);
-- + = must include
-- - = must exclude
-- "phrase" = exact phrase
-- * = wildcard (word*)
-- > / < = increase/decrease relevance

-- Query expansion: widens search using words from top results
SELECT id, title FROM articles
WHERE MATCH(title, body) AGAINST (
  'mysql' WITH QUERY EXPANSION
);

-- Minimum word length (default 3 for InnoDB, 4 for MyISAM)
SHOW VARIABLES LIKE 'innodb_ft_min_token_size';    -- default 3
-- Words shorter than minimum are not indexed
-- Change requires rebuilding the index: DROP INDEX + CREATE INDEX
```

## EXPLAIN Output

`EXPLAIN` is the primary tool for understanding how MySQL executes a query.

```sql
-- Basic EXPLAIN
EXPLAIN SELECT * FROM orders WHERE status = 'pending' AND user_id = 42\G

-- Extended output (MySQL 5.6+, included in EXPLAIN by default in 8.0)
EXPLAIN FORMAT=JSON SELECT * FROM orders WHERE status = 'pending'\G

-- EXPLAIN ANALYZE (MySQL 8.0.18+): actually runs the query and shows real stats
EXPLAIN ANALYZE SELECT * FROM orders WHERE status = 'pending' AND user_id = 42\G
```

### EXPLAIN Columns

| Column          | Description                                           |
| --------------- | ----------------------------------------------------- |
| `id`            | SELECT identifier; same id = same SELECT              |
| `select_type`   | SIMPLE, PRIMARY, SUBQUERY, DERIVED, UNION             |
| `table`         | Table being accessed                                  |
| `type`          | **Access type** (see below)                           |
| `possible_keys` | Indexes that could be used                            |
| `key`           | Index actually chosen                                 |
| `key_len`       | Bytes of index used (reflects which columns are used) |
| `ref`           | Columns or constants compared to key                  |
| `rows`          | Estimated rows examined                               |
| `filtered`      | Estimated % of rows passing WHERE conditions          |
| `Extra`         | Additional info (Using index, Using filesort, etc.)   |

### Access Types (type column, best to worst)

| type          | Description                                             |
| ------------- | ------------------------------------------------------- |
| `system`      | Table has exactly 1 row                                 |
| `const`       | Single row via PRIMARY KEY or UNIQUE equality           |
| `eq_ref`      | One row from each join, via PRIMARY KEY or UNIQUE       |
| `ref`         | Multiple rows via non-unique index equality             |
| `fulltext`    | FULLTEXT index used                                     |
| `ref_or_null` | ref + IS NULL                                           |
| `index_merge` | Multiple indexes merged                                 |
| `range`       | Index range scan (BETWEEN, >, &lt;, IN, LIKE 'prefix%') |
| `index`       | Full index scan (fast, but still all rows)              |
| `ALL`         | **Full table scan — no useful index found**             |

:::caution[type=ALL on Large Tables]
`type=ALL` on a table with millions of rows means every query reads every row. Add an index. For small tables (< ~1000 rows), a full table scan may still be faster than using an index.
:::

### Extra Column Values

| Extra value                    | Meaning                                                           |
| ------------------------------ | ----------------------------------------------------------------- |
| `Using index`                  | ✅ Covering index — no row lookup needed                          |
| `Using where`                  | WHERE filter applied after index scan                             |
| `Using index condition`        | Index Condition Pushdown (ICP) — filter applied in storage engine |
| `Using filesort`               | ⚠️ Sort operation in memory or disk (no index for ORDER BY)       |
| `Using temporary`              | ⚠️ Temp table created (often for GROUP BY or DISTINCT)            |
| `Using join buffer`            | Join buffering (Block Nested Loop) — join lacks index             |
| `Select tables optimized away` | ✅ Aggregate query answered from index alone                      |
| `Impossible WHERE`             | WHERE is always false (no rows)                                   |
| `Using MRR`                    | Multi-Range Read optimization                                     |

### key_len Calculation

`key_len` tells you how many bytes (and therefore how many columns) of a composite index are actually being used.

```sql
-- Table: idx_composite ON (status VARCHAR(20), user_id INT, created_at DATETIME)
-- With utf8mb4: VARCHAR(20) = 20 * 4 bytes + 2 bytes length prefix = 82 bytes
-- NOT NULL INT = 4 bytes
-- NOT NULL DATETIME = 8 bytes

-- If key_len = 86 → using (status, user_id) = 82 + 4 = 86
-- If key_len = 82 → using (status) only = 82
-- If key_len = 94 → using (status, user_id, created_at) = 82 + 4 + 8 = 94

-- NULL adds 1 byte to key_len (for the null flag)
```

## Index Hints

Override the optimizer's index choice when you know better:

```sql
-- Suggest an index (optimizer may still ignore it)
SELECT * FROM orders USE INDEX (idx_status) WHERE status = 'pending';

-- Force an index (optimizer must use it)
SELECT * FROM orders FORCE INDEX (idx_status) WHERE status = 'pending';

-- Exclude a specific index
SELECT * FROM orders IGNORE INDEX (idx_old_status) WHERE status = 'pending';

-- USE INDEX FOR JOIN / ORDER BY / GROUP BY
SELECT * FROM orders USE INDEX FOR ORDER BY (idx_created_at) ORDER BY created_at;
```

:::tip[When to Use FORCE INDEX]
Use `FORCE INDEX` sparingly. If the optimizer is choosing the wrong index, it usually means the statistics are stale (`ANALYZE TABLE`) or your index design is suboptimal. After MySQL upgrades, forced index hints may cause worse performance if the optimizer improves.
:::

## Table Maintenance

```sql
-- Update index statistics (non-blocking for InnoDB)
ANALYZE TABLE orders;
ANALYZE TABLE orders, users, products;

-- Rebuild table (reclaims fragmented space, updates statistics)
-- WARNING: Locks the table in older MySQL, may be slow on large tables
OPTIMIZE TABLE orders;
-- Prefer: ALTER TABLE orders ENGINE=InnoDB; (same effect, online in MySQL 5.6+)

-- Check table for corruption (MyISAM, read-only for InnoDB)
CHECK TABLE orders;
CHECK TABLE orders EXTENDED;

-- Repair MyISAM table (NOT available for InnoDB)
REPAIR TABLE myisam_table;
```

## Slow Query Log Setup

```sql
-- Enable slow query log at runtime
SET GLOBAL slow_query_log = 'ON';
SET GLOBAL slow_query_log_file = '/var/log/mysql/slow.log';
SET GLOBAL long_query_time = 1;          -- log queries taking > 1 second
SET GLOBAL log_queries_not_using_indexes = 'ON';  -- log full table scans
SET GLOBAL log_throttle_queries_not_using_indexes = 10;  -- max 10/min

-- View slow query log config
SHOW VARIABLES LIKE '%slow%';
SHOW VARIABLES LIKE 'long_query_time';

-- Analyze the slow query log with mysqldumpslow
-- mysqldumpslow -s t -t 10 /var/log/mysql/slow.log  (top 10 by time)

```

## Index Best Practices

<CardGrid>
  <Card title="Do Index" icon="approve-check">
    - Primary keys (always) - Foreign key columns (always — MySQL doesn't
    auto-index FKs) - Columns in WHERE clauses of frequent queries - Columns in
    JOIN conditions - Columns in ORDER BY / GROUP BY - Columns used in covering
    indexes
  </Card>
  <Card title="Don't Over-Index" icon="warning">
    - Every column (wastes space, slows writes) - Low-cardinality boolean
    columns alone (status with 2 values is rarely useful solo) - Very long
    VARCHAR columns (use prefix indexes or functional indexes) - Columns rarely
    used in queries
  </Card>
</CardGrid>

```sql
-- Prefix index for long strings (when you can't use the full column)
CREATE INDEX idx_url_prefix ON pages(url(100));
-- Only indexes first 100 characters — saves space but may reduce selectivity

-- Find tables without a primary key
SELECT t.TABLE_SCHEMA, t.TABLE_NAME
FROM information_schema.TABLES t
LEFT JOIN information_schema.TABLE_CONSTRAINTS c
  ON t.TABLE_SCHEMA = c.TABLE_SCHEMA
  AND t.TABLE_NAME = c.TABLE_NAME
  AND c.CONSTRAINT_TYPE = 'PRIMARY KEY'
WHERE t.TABLE_SCHEMA NOT IN ('mysql','information_schema','performance_schema','sys')
  AND c.CONSTRAINT_NAME IS NULL
  AND t.TABLE_TYPE = 'BASE TABLE';

-- Find unused indexes (requires performance_schema)
SELECT
  object_schema,
  object_name,
  index_name
FROM performance_schema.table_io_waits_summary_by_index_usage
WHERE index_name IS NOT NULL
  AND count_star = 0
  AND object_schema NOT IN ('mysql','performance_schema','information_schema','sys')
ORDER BY object_schema, object_name;
```

# MySQL Indexes

Indexes are data structures that allow MySQL to find rows matching a `WHERE` clause without scanning the entire table. Understanding how MySQL uses indexes — and when it doesn't — is the foundation of query optimization.

## Index Types in MySQL

### B-Tree (Default)

The default index type for InnoDB and MyISAM. B-Tree (balanced tree) indexes support:
- Equality lookups: `WHERE id = 42`
- Range queries: `WHERE price BETWEEN 10 AND 100`
- Prefix lookups: `WHERE name LIKE 'John%'`
- Sorting (can avoid filesort): `ORDER BY indexed_col`
- `IS NULL` and `IS NOT NULL`

**Cannot** use B-Tree for `LIKE '%suffix'` or `LIKE '%middle%'`.

### HASH (MEMORY engine only)

HASH indexes are only available in the `MEMORY` storage engine and for NDB Cluster.

- Only support exact equality lookups (`=`, `IN`)
- Cannot support range queries or ORDER BY
- Generally faster than B-Tree for equality lookups in the MEMORY engine
- InnoDB has an **Adaptive Hash Index** (AHI) built internally, but you cannot create HASH indexes manually in InnoDB

### FULLTEXT

Full-text search indexes allow relevance-based text searching. Available in both MyISAM (older) and InnoDB (since 5.6).

### SPATIAL

For geometry data types (`POINT`, `LINESTRING`, `POLYGON`). Used with spatial queries and the InnoDB/MyISAM engines.

## InnoDB Clustered Index (Primary Key)

In InnoDB, the **primary key IS the clustered index** — the table data is physically stored in primary key order. This is a fundamental architectural difference from MyISAM.

```sql
CREATE TABLE orders (
  -- This is the clustered index: data rows are stored ordered by order_id
  order_id    INT UNSIGNED AUTO_INCREMENT PRIMARY KEY,
  user_id     INT UNSIGNED NOT NULL,
  total       DECIMAL(10,2),
  created_at  DATETIME DEFAULT CURRENT_TIMESTAMP,
  INDEX idx_user (user_id),
  INDEX idx_created (created_at)
);
```

Key implications:

| Consideration | Effect |
|---|---|
| PK lookup | Extremely fast (data is the index) |
| Range scans on PK | Very efficient (sequential I/O) |
| Secondary index lookup | Two lookups: secondary index → PK → clustered data |
| PK should be small | PK value is stored in every secondary index leaf |
| Monotonically increasing PK | Prevents page splits and fragmentation |
| UUIDs as PK | Random insertion causes heavy page splits; use UUIDv7 or `UUID_TO_BIN()` with `swap_flag=1` |

:::caution[Hidden Clustered Index]
If you don't define a PRIMARY KEY, InnoDB looks for a NOT NULL UNIQUE index to use as the clustered key. If none exists, InnoDB creates a hidden 6-byte `rowid` field as the clustered key. You lose the ability to reference rows predictably. **Always define an explicit PRIMARY KEY.**
:::

### Secondary Index Structure in InnoDB

```sql
-- Every secondary index leaf node stores the PK value (not a physical row pointer)
-- Lookup: secondary index → find PK → go to clustered index → fetch row
-- This is called a "double lookup" or "index dive"

-- Covering index: if ALL needed columns are in the index, no clustered lookup needed
CREATE INDEX idx_user_created ON orders (user_id, created_at);

-- This query uses only the index (covering):
SELECT user_id, created_at FROM orders WHERE user_id = 42;

-- This requires a table lookup (total not in index):
SELECT user_id, created_at, total FROM orders WHERE user_id = 42;
```

## Creating Indexes

```sql
-- Create index at table creation time
CREATE TABLE products (
  id          INT AUTO_INCREMENT PRIMARY KEY,
  category_id INT,
  name        VARCHAR(255) NOT NULL,
  price       DECIMAL(10,2),
  sku         VARCHAR(50),
  created_at  DATETIME,
  
  UNIQUE KEY uk_sku (sku),
  INDEX idx_category (category_id),
  INDEX idx_price (price),
  INDEX idx_cat_price (category_id, price)  -- Composite
);

-- Add index to existing table
CREATE INDEX idx_name ON products (name);
ALTER TABLE products ADD INDEX idx_created (created_at);
ALTER TABLE products ADD UNIQUE INDEX uk_email (email);

-- Drop an index
DROP INDEX idx_name ON products;
ALTER TABLE products DROP INDEX idx_name;
ALTER TABLE products DROP PRIMARY KEY;  -- Only if auto_increment is removed first
```

## Composite Indexes and the Leftmost Prefix Rule

A composite (multi-column) index `(a, b, c)` can satisfy queries on:
- `a`
- `a, b`
- `a, b, c`

But **not** on `b` alone, `c` alone, or `b, c` — unless all of these are fixed by equality predicates.

```sql
CREATE INDEX idx_composite ON orders (user_id, status, created_at);

-- Uses the index (leftmost prefix: user_id)
SELECT * FROM orders WHERE user_id = 42;

-- Uses the index (user_id + status)
SELECT * FROM orders WHERE user_id = 42 AND status = 'pending';

-- Uses the full index
SELECT * FROM orders WHERE user_id = 42 AND status = 'pending' AND created_at > '2024-01-01';

-- Uses the index only for user_id (range stops reuse for subsequent columns)
SELECT * FROM orders WHERE user_id = 42 AND created_at > '2024-01-01';

-- Does NOT use the composite index at all
SELECT * FROM orders WHERE status = 'pending';  -- status is not the leftmost column
```

:::tip[Composite Index Column Order]
Place equality columns first, then range columns. High-cardinality columns before low-cardinality ones generally improves selectivity. Column order must match the `WHERE`, `ORDER BY`, and `GROUP BY` clauses to be effective.
:::

## Covering Indexes

A covering index contains all the columns that a query needs, so MySQL can satisfy the query entirely from the index without reading the actual table rows.

```sql
-- Query: find recent orders for a user, return just dates
SELECT created_at FROM orders WHERE user_id = 42 ORDER BY created_at DESC LIMIT 10;

-- A covering index for this query:
CREATE INDEX idx_user_created ON orders (user_id, created_at);

-- EXPLAIN will show "Using index" in Extra column — means covering index used
```

## UNIQUE Indexes

```sql
-- NULL is considered distinct from NULL in UNIQUE indexes
-- Multiple NULL values are allowed even in a UNIQUE column
CREATE TABLE users (
  id       INT AUTO_INCREMENT PRIMARY KEY,
  email    VARCHAR(255),
  username VARCHAR(50),
  UNIQUE KEY uk_email (email),
  UNIQUE KEY uk_username_email (username, email)  -- Composite unique
);

-- Check for duplicate values before adding UNIQUE index
SELECT email, COUNT(*) FROM users GROUP BY email HAVING COUNT(*) > 1;
```

## Invisible Indexes (MySQL 8.0+)

Invisible indexes are maintained by InnoDB but ignored by the query optimizer. This lets you safely test the impact of dropping an index before actually removing it.

```sql
-- Create an invisible index
CREATE INDEX idx_test ON orders (total) INVISIBLE;

-- Make an existing index invisible
ALTER TABLE orders ALTER INDEX idx_old_index INVISIBLE;

-- Make it visible again
ALTER TABLE orders ALTER INDEX idx_old_index VISIBLE;

-- Force optimizer to consider invisible indexes (for testing)
SET SESSION optimizer_switch = 'use_invisible_indexes=on';

-- Show index visibility
SELECT INDEX_NAME, IS_VISIBLE
FROM information_schema.STATISTICS
WHERE TABLE_SCHEMA = 'mydb' AND TABLE_NAME = 'orders';
```

## Functional Indexes (MySQL 8.0+)

Index on an expression rather than a column value.

```sql
-- Index on the year of a date (without generated column)
CREATE INDEX idx_year_created ON orders ((YEAR(created_at)));

-- Index on lowercase email for case-insensitive lookups
CREATE INDEX idx_lower_email ON users ((LOWER(email)));

-- Query MUST match the expression exactly to use the index
SELECT * FROM orders WHERE YEAR(created_at) = 2024;  -- Uses idx_year_created
SELECT * FROM users WHERE LOWER(email) = 'alice@example.com';  -- Uses idx_lower_email
```

## FULLTEXT Indexes

```sql
CREATE TABLE articles (
  id      INT AUTO_INCREMENT PRIMARY KEY,
  title   VARCHAR(255) NOT NULL,
  body    MEDIUMTEXT,
  FULLTEXT INDEX ft_content (title, body)
);

-- Natural Language Mode (default): relevance-ranked results
SELECT id, title,
  MATCH(title, body) AGAINST ('mysql indexing') AS relevance
FROM articles
WHERE MATCH(title, body) AGAINST ('mysql indexing')
ORDER BY relevance DESC;

-- Boolean Mode: supports +, -, "", *, ~
SELECT * FROM articles
WHERE MATCH(title, body) AGAINST (
  '+mysql -oracle "full text" index*'
  IN BOOLEAN MODE
);
-- + required term
-- - excluded term
-- "phrase" exact phrase
-- word* prefix wildcard
-- ~word lower relevance (but don't exclude)

-- Query Expansion: find related terms (two passes)
SELECT * FROM articles
WHERE MATCH(title, body) AGAINST ('database' WITH QUERY EXPANSION);

-- Minimum word length (default 3 for InnoDB, 4 for MyISAM)
SHOW VARIABLES LIKE 'innodb_ft_min_token_size';
-- Words shorter than this are not indexed
```

## EXPLAIN Analysis

`EXPLAIN` shows how MySQL plans to execute a query.

```sql
-- Basic EXPLAIN
EXPLAIN SELECT * FROM orders WHERE user_id = 42\G

-- EXPLAIN FORMAT=JSON (more detail)
EXPLAIN FORMAT=JSON SELECT * FROM orders WHERE user_id = 42;

-- EXPLAIN ANALYZE (MySQL 8.0.18+): actually executes and shows real timing
EXPLAIN ANALYZE SELECT * FROM orders WHERE user_id = 42\G
```

### Key EXPLAIN Columns

| Column | Description |
|---|---|
| `id` | Query identifier (higher = inner subquery) |
| `select_type` | SIMPLE, PRIMARY, SUBQUERY, DERIVED, UNION |
| `table` | Table being accessed |
| `type` | Access type (see below) |
| `possible_keys` | Indexes MySQL considered |
| `key` | Index MySQL actually chose |
| `key_len` | Bytes of key used (smaller = fewer columns used) |
| `ref` | Column or constant compared to key |
| `rows` | Estimated rows to examine |
| `filtered` | % of rows filtered after conditions |
| `Extra` | Important additional information |

### Access Types (type column) — Best to Worst

| Type | Description |
|---|---|
| `system` | Single-row system table |
| `const` | PK or UNIQUE lookup — 1 row at most |
| `eq_ref` | PK or UNIQUE join — 1 row per row from outer |
| `ref` | Non-unique index scan with equality |
| `fulltext` | FULLTEXT index scan |
| `ref_or_null` | Like `ref` but also looks for NULL |
| `index_merge` | Multiple indexes merged |
| `range` | Index range scan |
| `index` | Full index scan (reads all index entries) |
| `ALL` | Full table scan ⚠️ |

### Extra Column Values

```sql
-- "Using index" — covering index (no table lookup needed, FAST)
-- "Using where" — MySQL filters rows after reading them
-- "Using index condition" — Index Condition Pushdown (ICP, 5.6+)
-- "Using filesort" — cannot use index for ORDER BY (potentially slow)
-- "Using temporary" — temp table needed (GROUP BY, UNION, etc.)
-- "Using join buffer" — Block Nested Loop join (no index on join column)
-- "Impossible WHERE" — WHERE is always false (no rows can match)
-- "Select tables optimized away" — query resolved by optimizer alone
```

### key_len Calculation

`key_len` tells you how many bytes of a composite index are being used:

```sql
-- INT: 4 bytes
-- INT NOT NULL: 4 bytes
-- INT nullable: 4 + 1 = 5 bytes (1 byte for NULL flag)
-- VARCHAR(100) utf8mb4 NOT NULL: 100 * 4 + 2 = 402 bytes (4 bytes/char + 2 length)
-- DATETIME: 8 bytes
-- DATE: 3 bytes

-- For a composite index (user_id INT NOT NULL, status ENUM NOT NULL):
-- key_len = 4 (user_id) = only user_id used
-- key_len = 5 (user_id + 1 byte enum) = both columns used
```

## Index Hints

When MySQL chooses a suboptimal index, you can guide it:

```sql
-- Suggest an index (optimizer can ignore the hint)
SELECT * FROM orders USE INDEX (idx_user) WHERE user_id = 42;

-- Force an index (optimizer must use it or table scan)
SELECT * FROM orders FORCE INDEX (idx_user) WHERE user_id = 42;

-- Tell optimizer not to use an index
SELECT * FROM orders IGNORE INDEX (idx_bad_index) WHERE user_id = 42;

-- Hints for JOIN and ORDER BY
SELECT o.*, u.name
FROM orders o
  USE INDEX FOR JOIN (idx_user)
  JOIN users u ON o.user_id = u.id
USE INDEX FOR ORDER BY (idx_created);
```

:::caution[Index Hints Are a Code Smell]
If you need to force indexes, it usually means your statistics are stale or your indexes are poorly designed. Run `ANALYZE TABLE` and review your index strategy before resorting to hints in production code.
:::

## Maintaining Tables and Statistics

```sql
-- Update table statistics (helps optimizer make better decisions)
ANALYZE TABLE orders;
ANALYZE TABLE orders, users, products;  -- Multiple tables

-- Rebuild the table and optimize storage (reclaims space from deleted rows)
-- Also refreshes statistics
-- Note: OPTIMIZE TABLE locks the table and copies data — use pt-online-schema-change for large tables
OPTIMIZE TABLE orders;

-- Check table for errors
CHECK TABLE orders;
CHECK TABLE orders EXTENDED;  -- More thorough

-- Repair a corrupted table (MyISAM mainly)
REPAIR TABLE myisam_table;
```

## Slow Query Log

The slow query log is your primary tool for identifying queries that need indexes:

```sql
-- Check current slow query settings
SHOW VARIABLES LIKE 'slow_query%';
SHOW VARIABLES LIKE 'long_query_time';
SHOW VARIABLES LIKE 'log_queries_not_using_indexes';

-- Enable at runtime
SET GLOBAL slow_query_log = ON;
SET GLOBAL slow_query_log_file = '/var/log/mysql/slow.log';
SET GLOBAL long_query_time = 1;             -- Log queries taking > 1 second
SET GLOBAL log_queries_not_using_indexes = ON;  -- Also log non-indexed queries
SET GLOBAL min_examined_row_limit = 100;    -- Only log if >= 100 rows examined

-- Analyze slow query log with mysqldumpslow
-- mysqldumpslow -s t -t 10 /var/log/mysql/slow.log
-- -s t: sort by query time
-- -t 10: top 10 queries

-- Or use pt-query-digest (Percona Toolkit) for richer analysis
-- pt-query-digest /var/log/mysql/slow.log
```

## Index Best Practices Summary

```sql
-- 1. Always have a PRIMARY KEY
-- 2. Index all foreign key columns
-- 3. Index columns used in WHERE, JOIN ON, ORDER BY, GROUP BY
-- 4. Use composite indexes instead of multiple single-column indexes
-- 5. Put high-selectivity (high cardinality) columns first
-- 6. Put equality conditions before range conditions in composite indexes
-- 7. Consider covering indexes for critical queries
-- 8. Avoid indexing boolean/low-cardinality columns alone
-- 9. Don't over-index — each index slows down writes
-- 10. Use INVISIBLE indexes to safely test before dropping

-- Check index usage statistics
SELECT
  t.TABLE_SCHEMA,
  t.TABLE_NAME,
  s.INDEX_NAME,
  s.COLUMN_NAME,
  s.CARDINALITY,
  t.TABLE_ROWS,
  ROUND(s.CARDINALITY / t.TABLE_ROWS * 100, 2) AS selectivity_pct
FROM information_schema.STATISTICS s
JOIN information_schema.TABLES t
  ON s.TABLE_SCHEMA = t.TABLE_SCHEMA
  AND s.TABLE_NAME = t.TABLE_NAME
WHERE t.TABLE_SCHEMA = 'mydb'
  AND t.TABLE_ROWS > 0
ORDER BY selectivity_pct ASC;

-- Find unused indexes (requires performance_schema)
SELECT
  object_schema,
  object_name,
  index_name
FROM performance_schema.table_io_waits_summary_by_index_usage
WHERE index_name IS NOT NULL
  AND count_star = 0
  AND object_schema NOT IN ('mysql', 'performance_schema', 'information_schema')
ORDER BY object_schema, object_name, index_name;
```

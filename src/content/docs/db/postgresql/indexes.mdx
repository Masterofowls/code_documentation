Indexes are the primary tool for query optimization in PostgreSQL. Choosing the right index type and strategy can improve query performance by orders of magnitude.

## How Indexes Work

An index is a separate data structure that maps column values to their corresponding row locations (heap tuple IDs). When a query filters, sorts, or joins on an indexed column, PostgreSQL can use the index to find rows directly instead of scanning every row in the table.

**Trade-offs**: Indexes speed up reads but slow down writes (every INSERT/UPDATE/DELETE must also update the index). Indexes consume disk space and memory (for caching). The query planner decides whether to use an index based on estimated cost.

## B-tree Indexes (Default)

B-tree (balanced tree) is the default index type — appropriate for the vast majority of cases. It supports equality (`=`), range (`<`, `<=`, `>`, `>=`), `IN`, `BETWEEN`, `IS NULL`, `IS NOT NULL`, `ORDER BY`, and `LIKE 'prefix%'`.

```sql
-- Create a basic B-tree index
CREATE INDEX idx_orders_customer_id ON orders (customer_id);
CREATE INDEX idx_users_email ON users (email);
CREATE INDEX idx_products_price ON products (price);

-- Descending index (useful for ORDER BY col DESC queries)
CREATE INDEX idx_orders_created_desc ON orders (created_at DESC);

-- Composite index (multi-column)
-- Column order matters! Most selective / most commonly filtered first
CREATE INDEX idx_orders_status_created ON orders (status, created_at DESC);

-- Query that uses the above index:
SELECT * FROM orders WHERE status = 'pending' ORDER BY created_at DESC;
```

### Multi-Column Index Column Order

The first column in a composite index can be used independently. Later columns only help when earlier columns are also filtered.

```sql
-- Index on (status, created_at):
-- ✅ Uses index: WHERE status = 'x'
-- ✅ Uses index: WHERE status = 'x' AND created_at > '2024-01-01'
-- ❌ Cannot use index efficiently: WHERE created_at > '2024-01-01' (no status filter)
```

## Hash Indexes

Hash indexes are optimized for equality comparisons only. They do not support range queries or sorting. Since PostgreSQL 10, they are WAL-logged (safe to use on production).

```sql
-- Good for: equality lookups on high-cardinality columns
CREATE INDEX idx_sessions_token ON sessions USING HASH (token);

-- When to use over B-tree:
-- - Column is only used in WHERE col = 'value' (never ranges or LIKE)
-- - Hash index is slightly smaller than B-tree
-- - Hash index lookup is slightly faster than B-tree for equality
```

## GIN Indexes (Generalized Inverted Index)

GIN indexes are designed for columns that contain multiple values per row. Perfect for JSONB, arrays, and full-text search.

```sql
-- Arrays: supports @>, &&, = operators
CREATE INDEX idx_articles_tags ON articles USING GIN (tags);
SELECT * FROM articles WHERE tags @> ARRAY['postgresql'];

-- JSONB: supports @>, ?, ?|, ?& operators
CREATE INDEX idx_events_payload ON events USING GIN (payload);
SELECT * FROM events WHERE payload @> '{"status": "active"}';
SELECT * FROM events WHERE payload ? 'email';

-- jsonb_path_ops: smaller index, only @> operator (often sufficient)
CREATE INDEX idx_events_payload_path ON events USING GIN (payload jsonb_path_ops);

-- Full-text search: tsvector column
CREATE INDEX idx_articles_fts ON articles USING GIN (search_vector);
SELECT * FROM articles WHERE search_vector @@ to_tsquery('postgresql');

-- pg_trgm: trigram index for LIKE/ILIKE queries (requires extension)
CREATE EXTENSION IF NOT EXISTS pg_trgm;
CREATE INDEX idx_users_name_trgm ON users USING GIN (name gin_trgm_ops);
SELECT * FROM users WHERE name ILIKE '%smith%';  -- Uses trigram index!
```

:::tip[GIN vs B-tree for LIKE]
A plain B-tree index only accelerates `LIKE 'prefix%'` (starts-with). For `LIKE '%substring%'` (contains), you need a trigram GIN index with `pg_trgm`. This is the correct solution for search-as-you-type features.
:::

## GiST Indexes (Generalized Search Tree)

GiST is a flexible index framework supporting non-standard data types: geometric shapes, ranges, full-text, and anything with a custom GiST implementation.

```sql
-- Range types: supports @>, &&, <@ operators
CREATE INDEX idx_subscriptions_active ON subscriptions USING GiST (active);
SELECT * FROM subscriptions WHERE active @> NOW()::timestamptz;

-- Geometric types (PostGIS)
CREATE INDEX idx_locations_point ON locations USING GiST (coordinates);
SELECT * FROM locations WHERE ST_DWithin(coordinates, ST_MakePoint(-73.9, 40.7), 1000);

-- Full-text search (alternative to GIN for smaller tables)
CREATE INDEX idx_articles_fts_gist ON articles USING GiST (search_vector);

-- Trigram search (alternative to GIN with pg_trgm)
CREATE INDEX idx_products_name_trgm ON products USING GiST (name gist_trgm_ops);
```

**GIN vs GiST for full-text search:**

- **GIN**: Larger, slower to build, much faster to query. Use for large tables with frequent searches.
- **GiST**: Smaller, faster to build, slightly slower to query. Use for small tables or write-heavy workloads.

## BRIN Indexes (Block Range Index)

BRIN stores min/max values for physical blocks of the table. Extremely small (a few hundred pages regardless of table size) but only useful when the data is physically ordered by the indexed column (like a sequential timestamp).

```sql
-- Events table where rows are inserted in time order
CREATE INDEX idx_events_created_brin ON events USING BRIN (created_at);

-- BRIN is ideal for:
-- - Append-only time-series data
-- - Sequential integer IDs
-- - Any column where value increases as rows are inserted
-- - Tables too large to index with B-tree (BRIN is ~10,000x smaller)

-- Configure pages_per_range (default: 128)
CREATE INDEX ON events USING BRIN (created_at) WITH (pages_per_range = 32);
```

:::caution[BRIN has limited selectivity]
BRIN only helps when data is physically sorted by the indexed column. If data is randomly ordered, BRIN provides no benefit. Always check with `EXPLAIN ANALYZE` before relying on a BRIN index.
:::

## Partial Indexes

A partial index only indexes rows matching a WHERE condition. Smaller and faster than a full index.

```sql
-- Only index active users (not the millions of deleted/inactive ones)
CREATE INDEX idx_active_users_email ON users (email)
  WHERE is_active = true AND deleted_at IS NULL;

-- Only index pending orders (completed orders rarely need fast lookup)
CREATE INDEX idx_pending_orders_customer ON orders (customer_id)
  WHERE status = 'pending';

-- Partial unique constraint (only one active session per user)
CREATE UNIQUE INDEX idx_users_one_active_session ON sessions (user_id)
  WHERE is_active = true;
```

## Expression Indexes (Functional Indexes)

Index the result of a function or expression on a column:

```sql
-- Case-insensitive email search
CREATE INDEX idx_users_email_lower ON users (LOWER(email));
-- Query MUST use the same expression:
SELECT * FROM users WHERE LOWER(email) = 'alice@example.com';

-- Index on JSONB field value
CREATE INDEX idx_events_user_id ON events ((payload->>'user_id'));
SELECT * FROM events WHERE payload->>'user_id' = '42';

-- Index on date part
CREATE INDEX idx_orders_year_month ON orders (EXTRACT(YEAR FROM created_at), EXTRACT(MONTH FROM created_at));

-- Combined partial + expression
CREATE INDEX idx_published_articles_title ON articles (LOWER(title))
  WHERE status = 'published';
```

## Covering Indexes (INCLUDE)

A covering index includes additional columns in the index leaf pages. Queries that only need the indexed columns and the included columns can be satisfied by an **index-only scan** — never touching the heap.

```sql
-- Without INCLUDE: query needs heap access for 'total'
CREATE INDEX idx_orders_customer_id ON orders (customer_id);

-- With INCLUDE: index-only scan possible for (customer_id, total, created_at)
CREATE INDEX idx_orders_customer_covering ON orders (customer_id)
  INCLUDE (total, status, created_at);

-- This query can be served entirely from the index:
SELECT customer_id, total, status, created_at
FROM orders
WHERE customer_id = 42;
```

## Creating Indexes Without Blocking Writes

`CREATE INDEX` (without `CONCURRENTLY`) takes an `AccessShareLock` that blocks all writes during index creation. On large tables, this can cause significant downtime.

```sql
-- Non-blocking index creation (takes longer, but doesn't block)
CREATE INDEX CONCURRENTLY idx_events_created ON events (created_at);

-- Non-blocking re-index
REINDEX INDEX CONCURRENTLY idx_events_created;

-- Re-index entire table (rebuilds all indexes)
REINDEX TABLE CONCURRENTLY events;
```

:::caution[CONCURRENTLY limitations]
`CREATE INDEX CONCURRENTLY` cannot be run inside a transaction. It may leave an "invalid" index if the server crashes during creation — check with `SELECT * FROM pg_indexes WHERE pg_get_indexdef(indexrelid) IS NULL` and drop invalid indexes.
:::

## Monitoring Index Usage

```sql
-- Find unused indexes (no or very few scans since last stats reset)
SELECT
  schemaname,
  tablename,
  indexname,
  idx_scan AS scans,
  idx_tup_read AS tuples_read,
  pg_size_pretty(pg_relation_size(indexrelid)) AS size
FROM pg_stat_user_indexes
WHERE idx_scan < 50    -- Less than 50 uses
ORDER BY pg_relation_size(indexrelid) DESC;

-- Find tables doing the most sequential scans (need indexes!)
SELECT schemaname, tablename, seq_scan, seq_tup_read,
  idx_scan, n_live_tup
FROM pg_stat_user_tables
WHERE seq_scan > 1000
ORDER BY seq_tup_read DESC
LIMIT 20;

-- List all indexes on a table with size
SELECT
  indexname,
  pg_size_pretty(pg_relation_size(indexrelid)) AS index_size,
  idx_scan AS scan_count
FROM pg_stat_user_indexes
WHERE tablename = 'orders'
ORDER BY pg_relation_size(indexrelid) DESC;
```

## Index Bloat and Maintenance

```sql
-- Check for index bloat (dead tuples in index)
SELECT
  indexrelname,
  pg_size_pretty(pg_relation_size(indexrelid)) AS index_size,
  pg_stat_user_indexes.idx_scan
FROM pg_stat_user_indexes
JOIN pg_class ON pg_class.oid = indexrelid
ORDER BY pg_relation_size(indexrelid) DESC;

-- Manual VACUUM to reclaim space (happens automatically via autovacuum)
VACUUM ANALYZE orders;

-- Rebuild a bloated index (blocks reads, use CONCURRENTLY in production)
REINDEX INDEX CONCURRENTLY idx_orders_customer_id;

-- Check if autovacuum is running often enough
SELECT relname, last_autovacuum, last_autoanalyze, n_dead_tup, n_live_tup
FROM pg_stat_user_tables
WHERE n_dead_tup > 10000
ORDER BY n_dead_tup DESC;
```

## Index Strategy Summary

| Scenario                                   | Recommended Index              |
| ------------------------------------------ | ------------------------------ | ------------- |
| Equality and range on standard types       | B-tree (default)               |
| Equality only (tokens, IDs)                | Hash (slightly smaller/faster) |
| JSONB `@>` containment                     | GIN with `jsonb_path_ops`      |
| JSONB key operators `?`, `?                | `, `?&`                        | GIN (default) |
| Arrays (`@>`, `&&`)                        | GIN                            |
| Full-text search (`@@`)                    | GIN on tsvector column         |
| `ILIKE '%substring%'` search               | GIN with `pg_trgm`             |
| Range types (`@>`, `&&`)                   | GiST                           |
| Large time-series table, sequential insert | BRIN                           |
| Sparse data (active rows only)             | Partial B-tree                 |
| Case-insensitive string lookups            | B-tree on `LOWER(col)`         |
| Index-only scan optimization               | B-tree with `INCLUDE`          |

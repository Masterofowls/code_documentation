Indexes are the primary tool for improving query performance in PostgreSQL. Choosing the right index type for each use case is critical — a wrong choice adds overhead without benefit.

## B-Tree Indexes (Default)

B-tree (balanced tree) is the default index type. It works for equality, range queries, and sorting.

```sql
-- Create a basic B-tree index
CREATE INDEX idx_users_email ON users (email);

-- Unique index (enforces uniqueness + index)
CREATE UNIQUE INDEX idx_users_email_unique ON users (email);
-- Note: UNIQUE constraint creates a unique index automatically

-- Multi-column index
CREATE INDEX idx_orders_customer_status ON orders (customer_id, status);

-- Descending order (for ORDER BY col DESC queries)
CREATE INDEX idx_products_price_desc ON products (price DESC);

-- Specify NULL ordering
CREATE INDEX idx_products_deleted_at ON products (deleted_at NULLS LAST);
```

### When B-Tree Works

B-tree indexes support these operators: `=`, `<`, `<=`, `>`, `>=`, `BETWEEN`, `IN`, `LIKE 'prefix%'`, `IS NULL`, `IS NOT NULL`, `ORDER BY`.

```sql
-- All of these can use a B-tree index on (price):
SELECT * FROM products WHERE price = 29.99;
SELECT * FROM products WHERE price > 10 AND price < 50;
SELECT * FROM products WHERE price BETWEEN 10 AND 50;
SELECT * FROM products ORDER BY price;
SELECT * FROM products ORDER BY price DESC NULLS LAST;

-- B-tree on (name) can use prefix matching:
SELECT * FROM products WHERE name LIKE 'Widget%';      -- Works (prefix)
SELECT * FROM products WHERE name LIKE '%Widget%';     -- Does NOT use index
```

## Hash Indexes

Hash indexes store a hash of the indexed value. They only support `=` equality lookups.

```sql
CREATE INDEX idx_sessions_token_hash ON sessions USING HASH (token);

-- Perfect for equality lookups on long strings (UUID, token, hash)
SELECT * FROM sessions WHERE token = 'abc123def456...';
```

:::tip[Hash vs B-tree for Equality]
Since PostgreSQL 10, hash indexes are WAL-logged and crash-safe. They can be slightly faster than B-tree for pure equality lookups, but B-tree is more versatile. Only choose hash when you have pure equality queries on high-cardinality columns.
:::

## GIN Indexes

Generalized Inverted Index — designed for values that contain multiple components (arrays, JSONB, full-text).

```sql
-- For JSONB (full operators: @>, ?, ?|, ?&)
CREATE INDEX idx_products_metadata ON products USING GIN (metadata);

-- jsonb_path_ops: smaller index, only supports @>
CREATE INDEX idx_products_metadata_fast ON products USING GIN (metadata jsonb_path_ops);

-- For arrays
CREATE INDEX idx_users_tags ON users USING GIN (tags);

-- For full-text search
CREATE INDEX idx_articles_search ON articles USING GIN (to_tsvector('english', title || ' ' || body));

-- For pg_trgm (fuzzy search with LIKE '%text%' and similarity)
CREATE EXTENSION IF NOT EXISTS pg_trgm;
CREATE INDEX idx_products_name_trgm ON products USING GIN (name gin_trgm_ops);

-- Now these use the index:
SELECT * FROM products WHERE name ILIKE '%widget%';
SELECT * FROM products WHERE similarity(name, 'wiget') > 0.3;
```

## GiST Indexes

Generalized Search Tree — supports complex data types with custom search strategies: geometric types, full-text, range types, PostGIS.

```sql
-- For range types (exclusion constraints use GiST)
CREATE TABLE reservations (
  room_id   BIGINT NOT NULL,
  during    TSTZRANGE NOT NULL,
  EXCLUDE USING GIST (room_id WITH =, during WITH &&)
);

-- For full-text search (alternative to GIN — smaller but slower for large datasets)
CREATE INDEX idx_articles_search_gist ON articles USING GIST (search_vector);

-- For pg_trgm with GiST (better for UPDATE-heavy workloads than GIN)
CREATE INDEX idx_products_name_gist ON products USING GIST (name gist_trgm_ops);

-- For PostGIS geometry
CREATE INDEX idx_locations_geom ON locations USING GIST (geom);
```

### GIN vs GiST

| Aspect | GIN | GiST |
|---|---|---|
| **Build time** | Slower | Faster |
| **Index size** | Larger | Smaller |
| **Query speed** | Faster | Slower |
| **Update overhead** | Higher (but pending list helps) | Lower |
| **Best for** | Read-heavy, full-text, JSONB | Geometric, range, write-heavy |

## BRIN Indexes

Block Range INdex — stores min/max values per block range. Very small, very fast to build, but only useful for naturally ordered data.

```sql
-- Perfect for timestamp columns in append-only tables (logs, events)
CREATE INDEX idx_events_created_at_brin ON events USING BRIN (created_at);

-- Custom block range size (default: 128 pages)
CREATE INDEX idx_metrics_time_brin ON metrics
  USING BRIN (recorded_at) WITH (pages_per_range = 64);

-- Check effectiveness: BRIN is useful only when physical order ≈ logical order
-- If rows are inserted in created_at order (typical for event/log tables), BRIN is excellent
```

:::tip[BRIN for Time-Series]
BRIN indexes are orders of magnitude smaller than B-tree for time-series data. A BRIN index on a 100 million row event table might be only 100 KB. Use it for append-only tables where you frequently query by time range.
:::

## Partial Indexes

Index only a subset of rows, defined by a WHERE clause. Smaller, faster to maintain:

```sql
-- Only index active users (where most queries focus)
CREATE INDEX idx_users_email_active ON users (email)
  WHERE status = 'active';

-- Only index orders that aren't completed (pending + processing are queried often)
CREATE INDEX idx_orders_customer_open ON orders (customer_id)
  WHERE status NOT IN ('completed', 'cancelled', 'refunded');

-- Index only non-null values
CREATE INDEX idx_products_deleted_at ON products (deleted_at)
  WHERE deleted_at IS NOT NULL;

-- For the query to USE a partial index, the WHERE condition must match
-- This uses the partial index:
SELECT * FROM users WHERE email = 'alice@example.com' AND status = 'active';

-- This does NOT use the partial index (status is not filtered):
SELECT * FROM users WHERE email = 'alice@example.com';
```

## Expression Indexes

Index the result of an expression rather than a raw column value:

```sql
-- Case-insensitive email lookup
CREATE UNIQUE INDEX idx_users_email_lower ON users (LOWER(email));

-- Query that uses this index:
SELECT * FROM users WHERE LOWER(email) = LOWER('Alice@Example.com');

-- Extract a JSONB field and index it
CREATE INDEX idx_products_color ON products ((metadata ->> 'color'));

-- Query that uses this index:
SELECT * FROM products WHERE (metadata ->> 'color') = 'blue';

-- Date part extraction
CREATE INDEX idx_orders_month ON orders (EXTRACT(MONTH FROM created_at));
SELECT * FROM orders WHERE EXTRACT(MONTH FROM created_at) = 6;

-- Computed column
CREATE INDEX idx_order_items_total ON order_items ((quantity * unit_price));
```

## Covering Indexes (INCLUDE)

Include extra columns in the index so the query can be satisfied entirely from the index (index-only scan):

```sql
-- INCLUDE columns are stored in the leaf nodes but not sorted
-- They enable index-only scans when the query only needs these columns
CREATE INDEX idx_products_category_covering
  ON products (category_id)
  INCLUDE (name, price, status);

-- This query does an index-only scan — no table access:
SELECT name, price, status
FROM products
WHERE category_id = 5
  AND status = 'active';

-- Covering index for orders lookup
CREATE INDEX idx_orders_customer_covering
  ON orders (customer_id, status)
  INCLUDE (id, total, created_at);
```

## Multi-Column Index Column Order

The order of columns in a multi-column index critically affects which queries can use it.

```sql
-- Index: (customer_id, status, created_at)
CREATE INDEX idx_orders_multi ON orders (customer_id, status, created_at);

-- CAN use this index (left-prefix rule):
WHERE customer_id = 42
WHERE customer_id = 42 AND status = 'pending'
WHERE customer_id = 42 AND status = 'pending' AND created_at > '2024-01-01'
ORDER BY customer_id, status, created_at

-- CANNOT use this index:
WHERE status = 'pending'                -- Missing first column
WHERE created_at > '2024-01-01'         -- Missing first two columns
```

**Rule of thumb**: Put the most selective (high cardinality) column first when using equality filters; put the sort column last.

## CREATE INDEX CONCURRENTLY

Creating an index normally locks the table for writes. CONCURRENTLY allows writes during the build (takes longer):

```sql
-- Create index without locking writes
CREATE INDEX CONCURRENTLY idx_orders_created_at ON orders (created_at);

-- Note: CONCURRENTLY requires a separate transaction — cannot be in BEGIN/COMMIT block

-- Rebuild a corrupted or bloated index
REINDEX INDEX CONCURRENTLY idx_orders_created_at;

-- Rebuild all indexes on a table
REINDEX TABLE CONCURRENTLY orders;
```

## Index Maintenance

### Finding Unused Indexes

```sql
-- Indexes that have never been used since last stats reset
SELECT
  schemaname,
  tablename,
  indexname,
  idx_scan,
  idx_tup_read,
  idx_tup_fetch,
  pg_size_pretty(pg_relation_size(indexrelid)) AS index_size
FROM pg_stat_user_indexes
WHERE idx_scan = 0
  AND indexrelname NOT LIKE 'pg_%'
ORDER BY pg_relation_size(indexrelid) DESC;
```

### Index Bloat

Deleted and updated rows leave dead tuples in indexes. VACUUM removes them:

```sql
-- Check index bloat
SELECT
  tablename,
  indexname,
  pg_size_pretty(pg_relation_size(indexrelid)) AS index_size,
  round(100 * pg_relation_size(indexrelid) /
    pg_relation_size(indrelid::regclass)::numeric, 1) AS index_to_table_ratio
FROM pg_stat_user_indexes
JOIN pg_index ON indexrelid = pg_stat_user_indexes.indexrelid
ORDER BY pg_relation_size(indexrelid) DESC
LIMIT 20;

-- Manual VACUUM (also done by autovacuum)
VACUUM (VERBOSE, ANALYZE) orders;
```

### pg_stat_user_indexes

```sql
SELECT
  relname           AS table,
  indexrelname      AS index,
  idx_scan          AS scans,
  idx_tup_read      AS tuples_read,
  idx_tup_fetch     AS tuples_fetched,
  pg_size_pretty(pg_relation_size(indexrelid)) AS size
FROM pg_stat_user_indexes
WHERE schemaname = 'public'
ORDER BY idx_scan DESC;
```

PostgreSQL performance optimization requires understanding how the query planner works, what your slow queries are, and how to tune configuration for your workload.

## Reading EXPLAIN Output

`EXPLAIN` shows the query plan without executing the query. `EXPLAIN ANALYZE` executes it and shows actual versus estimated statistics.

```sql
-- Basic EXPLAIN
EXPLAIN SELECT * FROM orders WHERE customer_id = 42;

-- Full analysis with buffer information
EXPLAIN (
  ANALYZE,      -- Actually execute the query
  BUFFERS,      -- Show buffer cache hits/misses
  FORMAT TEXT   -- TEXT | JSON | XML | YAML
)
SELECT
  o.id, c.name, SUM(oi.unit_price * oi.quantity) AS total
FROM orders o
JOIN customers c   ON c.id = o.customer_id
JOIN order_items oi ON oi.order_id = o.id
WHERE o.status = 'completed'
  AND o.created_at >= NOW() - INTERVAL '30 days'
GROUP BY o.id, c.name
ORDER BY total DESC
LIMIT 10;
```

### Scan Types

```
Seq Scan on orders (cost=0.00..4521.00 rows=15000 width=72)
  (actual time=0.05..45.23 rows=14892 loops=1)
  Filter: (status = 'completed')
  Rows Removed by Filter: 85108
```

| Scan Type | Description | When it appears |
|---|---|---|
| **Seq Scan** | Full table scan, reads every row | No usable index, or table is tiny |
| **Index Scan** | B-tree traversal + heap fetch | Selective query with B-tree index |
| **Index Only Scan** | All data from index, no heap | Covering index satisfies all columns |
| **Bitmap Index Scan** | Collect matching page locations | Multiple index conditions combined |
| **Bitmap Heap Scan** | Fetch pages identified by bitmap | After Bitmap Index Scan |

### Join Strategies

| Join Type | Description | Best for |
|---|---|---|
| **Nested Loop** | For each outer row, scan inner | Small inner table or indexed inner |
| **Hash Join** | Build hash table from smaller relation | Medium-to-large unsorted inputs |
| **Merge Join** | Both inputs pre-sorted on join key | Large pre-sorted or indexed inputs |

### Understanding Cost Values

```
Seq Scan on orders (cost=0.00..4521.00 rows=15000 width=72)
                          ^^^^ ^^^^^^^^ ^^^^^^^ ^^^^^
                          |    |        |       bytes per row (estimated)
                          |    |        estimated output rows
                          |    total cost (arbitrary units)
                          startup cost (before first row)

(actual time=0.05..45.23 rows=14892 loops=1)
              ^^^^  ^^^^^  ^^^^^^  ^^^^^^
              |     |      |       how many times this node ran
              |     |      actual rows returned
              |     total actual time (ms)
              startup actual time (ms)
```

## pg_stat_statements

Tracks execution statistics for all SQL statements. One of the most valuable PostgreSQL extensions.

```sql
-- Enable (requires postgresql.conf change + restart)
-- shared_preload_libraries = 'pg_stat_statements'

CREATE EXTENSION IF NOT EXISTS pg_stat_statements;

-- Top 10 slowest queries by total time
SELECT
  round(total_exec_time::numeric, 2)   AS total_ms,
  calls,
  round(mean_exec_time::numeric, 2)    AS mean_ms,
  round(stddev_exec_time::numeric, 2)  AS stddev_ms,
  round((100 * total_exec_time /
    SUM(total_exec_time) OVER ())::numeric, 2) AS pct_of_total,
  LEFT(query, 120)                     AS query_preview
FROM pg_stat_statements
ORDER BY total_exec_time DESC
LIMIT 10;

-- Top 10 most frequently called queries
SELECT
  calls,
  round(mean_exec_time::numeric, 2) AS mean_ms,
  round(total_exec_time::numeric, 2) AS total_ms,
  LEFT(query, 120) AS query_preview
FROM pg_stat_statements
ORDER BY calls DESC
LIMIT 10;

-- High variance queries (inconsistent performance)
SELECT
  calls,
  round(mean_exec_time::numeric, 2)   AS mean_ms,
  round(stddev_exec_time::numeric, 2) AS stddev_ms,
  round(stddev_exec_time / NULLIF(mean_exec_time, 0) * 100, 1) AS cv_pct,
  LEFT(query, 120) AS query_preview
FROM pg_stat_statements
WHERE calls > 10
ORDER BY stddev_exec_time / NULLIF(mean_exec_time, 0) DESC
LIMIT 10;

-- Reset statistics
SELECT pg_stat_statements_reset();
```

## Autovacuum Tuning

PostgreSQL's MVCC leaves dead tuples after UPDATE and DELETE operations. Autovacuum cleans them up. Tuning it prevents table bloat.

### Key Configuration Parameters

```sql
-- Check current autovacuum settings
SHOW autovacuum;
SHOW autovacuum_vacuum_scale_factor;
SHOW autovacuum_analyze_scale_factor;
SHOW autovacuum_vacuum_cost_delay;

-- Per-table autovacuum tuning (overrides global settings)
ALTER TABLE orders SET (
  autovacuum_vacuum_scale_factor = 0.01,    -- Vacuum when 1% of rows are dead (default: 20%)
  autovacuum_analyze_scale_factor = 0.005,  -- Analyze when 0.5% of rows change
  autovacuum_vacuum_cost_delay = 2          -- Less throttling for busy tables
);

-- For large tables, absolute thresholds are better than scale factors
ALTER TABLE events SET (
  autovacuum_vacuum_scale_factor = 0.0,
  autovacuum_vacuum_threshold = 10000,      -- Vacuum after 10K dead tuples
  autovacuum_analyze_scale_factor = 0.0,
  autovacuum_analyze_threshold = 5000       -- Analyze after 5K changes
);
```

### Monitoring Autovacuum

```sql
-- Tables needing vacuum
SELECT
  schemaname,
  relname,
  n_dead_tup,
  n_live_tup,
  round(n_dead_tup::numeric / NULLIF(n_live_tup, 0) * 100, 1) AS dead_pct,
  last_vacuum,
  last_autovacuum,
  last_analyze,
  last_autoanalyze
FROM pg_stat_user_tables
WHERE n_dead_tup > 1000
ORDER BY n_dead_tup DESC;

-- Currently running autovacuum
SELECT pid, query, now() - pg_stat_activity.query_start AS duration
FROM pg_stat_activity
WHERE query LIKE 'autovacuum:%'
ORDER BY duration DESC;

-- Vacuum manually when needed
VACUUM (VERBOSE, ANALYZE) orders;
VACUUM FULL orders;   -- Reclaims disk space but locks table — use with caution
```

## Connection Pooling with PgBouncer

Each PostgreSQL connection uses ~5-10MB RAM and creates a backend process. With many application servers, connection overhead becomes a bottleneck. PgBouncer is the standard solution.

### pgbouncer.ini Configuration

```ini
[databases]
myapp = host=127.0.0.1 port=5432 dbname=myapp

[pgbouncer]
listen_addr = 0.0.0.0
listen_port = 5432
auth_type = scram-sha-256
auth_file = /etc/pgbouncer/userlist.txt

# Pool mode:
# session: connection held for entire client session (like direct connection)
# transaction: connection returned to pool after each transaction (recommended)
# statement: connection returned after each statement (limited use)
pool_mode = transaction

# Max server connections (to PostgreSQL)
max_client_conn = 1000
default_pool_size = 25
min_pool_size = 5

# Behavior when pool is full
server_idle_timeout = 600
client_idle_timeout = 0
max_db_connections = 100

# Logging
log_connections = 1
log_disconnections = 1
log_stats = 1
stats_period = 60
```

### Monitoring PgBouncer

```bash
# Connect to PgBouncer admin console
psql -h localhost -p 5432 -U pgbouncer pgbouncer

-- Show pool status
SHOW POOLS;

-- Show client connections
SHOW CLIENTS;

-- Show server connections
SHOW SERVERS;

-- Show statistics
SHOW STATS;
SHOW STATS_AVERAGES;

-- Reload config
RELOAD;

-- Pause/resume (for maintenance)
PAUSE myapp;
RESUME myapp;
```

:::caution[Transaction Mode Limitations]
PgBouncer transaction mode doesn't support: session-level advisory locks, prepared statements (unless `prepared_statement_cache_queries = 0`), `SET` session variables that persist across queries, or `LISTEN/NOTIFY`. Use session mode for these features.
:::

## Query Optimization Patterns

### Avoid SELECT *

```sql
-- BAD: fetches all columns, prevents index-only scans
SELECT * FROM orders WHERE customer_id = 42;

-- GOOD: fetch only what you need
SELECT id, status, total, created_at FROM orders WHERE customer_id = 42;
```

### Use Covering Indexes

```sql
-- Index includes all columns the query needs
CREATE INDEX idx_orders_customer_covering
  ON orders (customer_id)
  INCLUDE (id, status, total, created_at);

-- Now this query does an index-only scan (no table access):
SELECT id, status, total, created_at FROM orders WHERE customer_id = 42;
```

### Avoid Functions on Indexed Columns in WHERE

```sql
-- BAD: function prevents index use
SELECT * FROM users WHERE UPPER(email) = 'ALICE@EXAMPLE.COM';
SELECT * FROM orders WHERE DATE(created_at) = '2024-01-15';

-- GOOD: expression index or rewrite to avoid the function
CREATE INDEX idx_users_email_upper ON users (UPPER(email));

-- Or rewrite the query:
SELECT * FROM orders
WHERE created_at >= '2024-01-15' AND created_at < '2024-01-16';
```

### Avoid Implicit Type Casts

```sql
-- BAD: integer column with text comparison — implicit cast prevents index use
SELECT * FROM users WHERE id = '42';   -- id is BIGINT

-- GOOD: match types explicitly
SELECT * FROM users WHERE id = 42;

-- BAD: comparing TIMESTAMPTZ to text
SELECT * FROM orders WHERE created_at = '2024-01-15';  -- Implicit cast

-- GOOD: explicit cast
SELECT * FROM orders WHERE created_at = '2024-01-15'::timestamptz;
```

### work_mem and maintenance_work_mem

```sql
-- work_mem: memory per sort/hash operation per query (multiplied by concurrent sorts)
-- Default is 4MB — often too low for complex queries
SET work_mem = '256MB';   -- For a specific session
-- Or in postgresql.conf: work_mem = 64MB

-- Higher work_mem helps:
-- - Sorts that spill to disk (look for "Sort Method: external merge Disk" in EXPLAIN)
-- - Hash joins that spill to disk ("Batches: N" where N > 1 in EXPLAIN)

-- maintenance_work_mem: for VACUUM, CREATE INDEX, ALTER TABLE
SET maintenance_work_mem = '1GB';   -- For large table operations

-- Check current values
SHOW work_mem;
SHOW maintenance_work_mem;
```

## Table Statistics

PostgreSQL's query planner relies on statistics to estimate row counts and choose plans.

```sql
-- Manually update statistics
ANALYZE orders;
ANALYZE;  -- Analyze all tables in current database

-- Increase statistics target for better estimates (default: 100)
ALTER TABLE orders ALTER COLUMN status SET STATISTICS 500;
ALTER TABLE orders ALTER COLUMN customer_id SET STATISTICS 1000;

-- Check column statistics
SELECT
  attname,
  n_distinct,        -- Negative = fraction of total rows; positive = estimate of distinct values
  correlation,       -- Physical vs logical order correlation (1.0 = perfectly correlated)
  most_common_vals,
  most_common_freqs
FROM pg_stats
WHERE tablename = 'orders'
ORDER BY attname;

-- Table size and row estimates
SELECT
  relname,
  reltuples::bigint AS estimated_rows,
  pg_size_pretty(pg_table_size(oid)) AS table_size,
  pg_size_pretty(pg_indexes_size(oid)) AS indexes_size,
  pg_size_pretty(pg_total_relation_size(oid)) AS total_size
FROM pg_class
WHERE relname = 'orders';
```

## Parallel Query

PostgreSQL can use multiple CPU cores for a single query.

```sql
-- Check current parallel settings
SHOW max_parallel_workers_per_gather;
SHOW max_parallel_workers;
SHOW min_parallel_table_scan_size;
SHOW min_parallel_index_scan_size;

-- Force parallel execution for testing
SET max_parallel_workers_per_gather = 4;
SET parallel_tuple_cost = 0;
SET parallel_setup_cost = 0;

-- Parallel sequential scan appears in EXPLAIN as:
-- Gather (cost=... rows=... width=...)
--   Workers Planned: 4
--   -> Parallel Seq Scan on orders (cost=... rows=... width=...)

-- Disable parallel for a specific query
SET max_parallel_workers_per_gather = 0;
```

## Streaming Replication Overview

```sql
-- Check replication status (on primary)
SELECT
  client_addr,
  state,            -- startup | catchup | streaming | backup | stopped
  sent_lsn,         -- LSN sent to replica
  write_lsn,        -- LSN written to WAL on replica
  flush_lsn,        -- LSN flushed to disk on replica
  replay_lsn,       -- LSN replayed on replica
  write_lag,        -- Time lag for write
  flush_lag,        -- Time lag for flush
  replay_lag        -- Time lag for replay
FROM pg_stat_replication;

-- Check if running as primary or replica
SELECT pg_is_in_recovery();   -- false = primary, true = replica

-- On replica: check replication lag
SELECT
  now() - pg_last_xact_replay_timestamp() AS replication_lag;
```

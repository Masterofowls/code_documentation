## Transactions

A transaction is a unit of work that is atomic — either all statements succeed and are committed, or the entire unit is rolled back.

```sql
-- Basic transaction
BEGIN;

UPDATE accounts SET balance = balance - 500 WHERE id = 1;
UPDATE accounts SET balance = balance + 500 WHERE id = 2;
INSERT INTO transfers (from_id, to_id, amount, created_at) VALUES (1, 2, 500, NOW());

COMMIT;  -- permanently write all changes

-- On error, roll back
BEGIN;

UPDATE accounts SET balance = balance - 500 WHERE id = 1;
-- If the next statement errors, nothing is written:
UPDATE accounts SET balance = balance + 500 WHERE id = 99;  -- id 99 doesn't exist

ROLLBACK;  -- undo all changes in this transaction
```

### SAVEPOINT

Savepoints let you partially roll back a transaction:

```sql
BEGIN;

INSERT INTO orders (customer_id, total) VALUES (42, 150.00);
SAVEPOINT before_items;

INSERT INTO order_items (order_id, product_id, quantity, unit_price)
VALUES (currval('orders_id_seq'), 101, 2, 75.00);

-- If item insert fails, rollback only to the savepoint
ROLLBACK TO SAVEPOINT before_items;

-- Try a different item
INSERT INTO order_items (order_id, product_id, quantity, unit_price)
VALUES (currval('orders_id_seq'), 102, 1, 150.00);

RELEASE SAVEPOINT before_items;  -- release the savepoint (optional, auto-released on COMMIT)
COMMIT;
```

:::tip[Transactions in Application Code]
Always use explicit transactions for multi-step operations in application code. Most database drivers start in auto-commit mode — each statement is its own transaction. Use `BEGIN`/`COMMIT` explicitly or your ORM's transaction API.
:::

## Isolation Levels

Isolation levels control how transactions interact with each other. Higher isolation = fewer anomalies but lower concurrency.

### Isolation Level Phenomena

| Phenomenon              | Description                                                                                          |
| ----------------------- | ---------------------------------------------------------------------------------------------------- |
| **Dirty Read**          | Reading uncommitted changes from another transaction                                                 |
| **Non-repeatable Read** | Reading the same row twice yields different values (another transaction committed between reads)     |
| **Phantom Read**        | A range query returns different rows when re-executed (rows inserted/deleted by another transaction) |

### Isolation Level Comparison

| Level            | Dirty Read | Non-repeatable Read | Phantom Read | Notes                  |
| ---------------- | ---------- | ------------------- | ------------ | ---------------------- |
| READ UNCOMMITTED | Possible   | Possible            | Possible     | Fastest, least safe    |
| READ COMMITTED   | Prevented  | Possible            | Possible     | **PostgreSQL default** |
| REPEATABLE READ  | Prevented  | Prevented           | Possible     | MySQL InnoDB default   |
| SERIALIZABLE     | Prevented  | Prevented           | Prevented    | Slowest, safest        |

```sql
-- Set isolation level for current transaction
BEGIN;
SET TRANSACTION ISOLATION LEVEL REPEATABLE READ;
-- ... your statements ...
COMMIT;

-- Set default for current session
SET SESSION CHARACTERISTICS AS TRANSACTION ISOLATION LEVEL SERIALIZABLE;

-- Serializable is needed for correctness-critical financial operations
BEGIN ISOLATION LEVEL SERIALIZABLE;
SELECT balance FROM accounts WHERE id = 1;  -- 1000
-- Another transaction cannot commit a change to this row before we commit
UPDATE accounts SET balance = balance + 100 WHERE id = 1;
COMMIT;
```

## Views

Views are named queries stored in the database that behave like read-only (or sometimes writable) virtual tables.

### Creating Views

```sql
-- Simple view
CREATE VIEW active_users AS
SELECT id, email, name, created_at
FROM users
WHERE is_active = true
  AND deleted_at IS NULL;

-- Complex view
CREATE VIEW order_summary AS
SELECT
  o.id,
  o.created_at,
  c.name AS customer_name,
  c.email,
  COUNT(oi.id)                    AS item_count,
  SUM(oi.quantity * oi.unit_price) AS subtotal,
  o.total,
  o.status
FROM orders o
JOIN customers c    ON c.id = o.customer_id
JOIN order_items oi ON oi.order_id = o.id
GROUP BY o.id, o.created_at, c.name, c.email, o.total, o.status;

-- Query views just like tables
SELECT * FROM active_users WHERE email LIKE '%@example.com';
SELECT * FROM order_summary WHERE status = 'pending' ORDER BY created_at;
```

### Updatable Views

A view is automatically updatable if it satisfies certain conditions (single table, no GROUP BY, DISTINCT, aggregate functions, etc.):

```sql
-- This view is updatable
CREATE VIEW public_users AS
SELECT id, email, display_name, avatar_url
FROM users
WHERE is_public = true;

-- This UPDATE goes to the underlying table
UPDATE public_users SET display_name = 'Alice' WHERE id = 42;

-- WITH CHECK OPTION prevents updates that would hide the row from the view
CREATE VIEW active_products AS
SELECT * FROM products WHERE is_active = true
WITH CHECK OPTION;

-- This would fail — setting is_active = false would make the row invisible to the view
UPDATE active_products SET is_active = false WHERE id = 1;  -- ERROR
```

### Materialized Views (PostgreSQL)

Materialized views store the result set physically and must be refreshed:

```sql
-- Create materialized view
CREATE MATERIALIZED VIEW monthly_revenue AS
SELECT
  DATE_TRUNC('month', created_at) AS month,
  SUM(total)                       AS revenue,
  COUNT(*)                         AS orders
FROM orders
WHERE status = 'delivered'
GROUP BY DATE_TRUNC('month', created_at)
WITH DATA;  -- populate immediately

-- Create unique index for CONCURRENTLY refresh
CREATE UNIQUE INDEX idx_monthly_revenue_month ON monthly_revenue(month);

-- Refresh (blocks reads unless CONCURRENTLY)
REFRESH MATERIALIZED VIEW monthly_revenue;

-- Refresh without blocking reads (requires unique index)
REFRESH MATERIALIZED VIEW CONCURRENTLY monthly_revenue;

-- Query it like a regular table (fast!)
SELECT * FROM monthly_revenue WHERE month >= '2024-01-01';
```

## Stored Procedures and Functions

### PostgreSQL PL/pgSQL Functions

```sql
-- Function: returns a value
CREATE OR REPLACE FUNCTION get_customer_lifetime_value(p_customer_id BIGINT)
RETURNS NUMERIC AS $$
DECLARE
  v_total NUMERIC;
BEGIN
  SELECT COALESCE(SUM(total), 0)
  INTO v_total
  FROM orders
  WHERE customer_id = p_customer_id
    AND status = 'delivered';

  RETURN v_total;
END;
$$ LANGUAGE plpgsql STABLE;

-- Call it:
SELECT get_customer_lifetime_value(42);

-- Function: returns a table
CREATE OR REPLACE FUNCTION get_top_customers(p_limit INTEGER DEFAULT 10)
RETURNS TABLE(customer_id BIGINT, name TEXT, total_spent NUMERIC) AS $$
BEGIN
  RETURN QUERY
    SELECT c.id, c.name, SUM(o.total)::NUMERIC
    FROM customers c
    JOIN orders o ON o.customer_id = c.id
    WHERE o.status = 'delivered'
    GROUP BY c.id, c.name
    ORDER BY SUM(o.total) DESC
    LIMIT p_limit;
END;
$$ LANGUAGE plpgsql STABLE;

-- Call it:
SELECT * FROM get_top_customers(5);
```

### Procedures (PostgreSQL 11+)

```sql
-- Procedures: don't return values, can COMMIT/ROLLBACK
CREATE OR REPLACE PROCEDURE archive_old_orders(p_days INTEGER DEFAULT 365)
LANGUAGE plpgsql AS $$
DECLARE
  v_count INTEGER;
BEGIN
  INSERT INTO orders_archive
  SELECT * FROM orders
  WHERE created_at < NOW() - (p_days || ' days')::INTERVAL
    AND status IN ('delivered', 'cancelled');

  GET DIAGNOSTICS v_count = ROW_COUNT;

  DELETE FROM orders
  WHERE created_at < NOW() - (p_days || ' days')::INTERVAL
    AND status IN ('delivered', 'cancelled');

  RAISE NOTICE 'Archived % orders', v_count;
END;
$$;

-- Call with CALL
CALL archive_old_orders(365);
```

## Triggers

Triggers automatically execute a function before or after DML events.

```sql
-- Trigger function: automatically update updated_at column
CREATE OR REPLACE FUNCTION set_updated_at()
RETURNS TRIGGER AS $$
BEGIN
  NEW.updated_at = NOW();
  RETURN NEW;
END;
$$ LANGUAGE plpgsql;

-- Apply to any table
CREATE TRIGGER users_set_updated_at
BEFORE UPDATE ON users
FOR EACH ROW EXECUTE FUNCTION set_updated_at();

-- Audit log trigger
CREATE TABLE audit_log (
  id         BIGSERIAL PRIMARY KEY,
  table_name TEXT NOT NULL,
  operation  TEXT NOT NULL,
  old_data   JSONB,
  new_data   JSONB,
  changed_by TEXT,
  changed_at TIMESTAMPTZ DEFAULT NOW()
);

CREATE OR REPLACE FUNCTION audit_changes()
RETURNS TRIGGER AS $$
BEGIN
  INSERT INTO audit_log (table_name, operation, old_data, new_data, changed_by)
  VALUES (
    TG_TABLE_NAME,
    TG_OP,
    CASE WHEN TG_OP = 'DELETE' THEN row_to_json(OLD)::jsonb ELSE NULL END,
    CASE WHEN TG_OP != 'DELETE' THEN row_to_json(NEW)::jsonb ELSE NULL END,
    current_user
  );
  RETURN COALESCE(NEW, OLD);
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER users_audit
AFTER INSERT OR UPDATE OR DELETE ON users
FOR EACH ROW EXECUTE FUNCTION audit_changes();
```

## EXPLAIN and Query Plans

```sql
-- Basic EXPLAIN (shows plan without executing)
EXPLAIN SELECT * FROM orders WHERE customer_id = 42;

-- EXPLAIN ANALYZE (executes query and shows actual timing)
EXPLAIN (ANALYZE, BUFFERS, FORMAT TEXT)
SELECT o.id, c.name, o.total
FROM orders o
JOIN customers c ON c.id = o.customer_id
WHERE o.status = 'delivered'
  AND c.country = 'US'
ORDER BY o.total DESC
LIMIT 10;
```

Key plan nodes to understand:

| Node               | Meaning                                            |
| ------------------ | -------------------------------------------------- |
| `Seq Scan`         | Full table scan — check if index would help        |
| `Index Scan`       | Uses an index to find rows                         |
| `Index Only Scan`  | All data from index — no heap access               |
| `Bitmap Heap Scan` | Used when index finds many rows                    |
| `Hash Join`        | Join using a hash table in memory                  |
| `Merge Join`       | Join on sorted inputs                              |
| `Nested Loop`      | For each outer row, scan inner — good with indexes |

## Normalization

### First Normal Form (1NF)

Every column contains atomic values; no repeating groups:

```sql
-- VIOLATES 1NF: storing multiple values in one column
-- BAD: phones = '555-1234, 555-5678'
-- GOOD: separate rows in a related table
CREATE TABLE user_phones (
  user_id    BIGINT REFERENCES users(id),
  phone_type TEXT,     -- 'mobile', 'home', 'work'
  phone_number TEXT NOT NULL,
  PRIMARY KEY (user_id, phone_type)
);
```

### Second Normal Form (2NF)

1NF + every non-key column depends on the **entire** primary key (no partial dependencies):

```sql
-- VIOLATES 2NF if PK is (order_id, product_id) but product_name only depends on product_id
-- BAD: order_items(order_id, product_id, quantity, product_name, product_price)
-- GOOD: separate products table
CREATE TABLE order_items (
  order_id   BIGINT REFERENCES orders(id),
  product_id BIGINT REFERENCES products(id),  -- product details in products table
  quantity   INTEGER NOT NULL,
  unit_price NUMERIC(10,2) NOT NULL,           -- snapshot price at time of order
  PRIMARY KEY (order_id, product_id)
);
```

### Third Normal Form (3NF)

2NF + no transitive dependencies (non-key columns must depend only on the key):

```sql
-- VIOLATES 3NF: zip_code -> city -> state (city depends on zip, state depends on city)
-- BAD: users(id, name, zip_code, city, state)
-- GOOD: separate address lookup
CREATE TABLE zip_codes (
  zip_code VARCHAR(10) PRIMARY KEY,
  city     TEXT NOT NULL,
  state    CHAR(2) NOT NULL
);
CREATE TABLE users (
  id       BIGINT PRIMARY KEY,
  name     TEXT,
  zip_code VARCHAR(10) REFERENCES zip_codes(zip_code)
);
```

### Boyce-Codd Normal Form (BCNF)

Stricter than 3NF: every determinant must be a candidate key:

```sql
-- Every column's value must be determined by a key
-- A table with overlapping candidate keys can violate 3NF but satisfy BCNF needs restructuring
-- Practical rule: if EXPLAIN shows unexpected joins, re-examine your normalization
```

:::tip[When to Denormalize]
Normalization reduces redundancy and ensures consistency. But for read-heavy analytics workloads, controlled denormalization (storing pre-computed aggregates, materialized views, or summary tables) can dramatically improve performance. Always profile before denormalizing.
:::

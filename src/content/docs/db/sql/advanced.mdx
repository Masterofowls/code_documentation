## Transactions

A transaction is a unit of work that either completes entirely or not at all. Transactions protect your data from partial updates.

### Basic Transaction Control

```sql
-- Begin an explicit transaction
BEGIN;  -- or START TRANSACTION;

-- Do work
UPDATE accounts SET balance = balance - 100 WHERE id = 1;
UPDATE accounts SET balance = balance + 100 WHERE id = 2;

-- Commit if everything succeeded
COMMIT;

-- Or roll back to undo all changes since BEGIN
ROLLBACK;
```

### Savepoints

Savepoints allow partial rollbacks within a transaction:

```sql
BEGIN;

INSERT INTO orders (customer_id, total) VALUES (42, 199.99);
SAVEPOINT after_order;

INSERT INTO order_items (order_id, product_id, quantity) VALUES (1001, 5, 2);
-- If this fails, roll back just to after_order
ROLLBACK TO SAVEPOINT after_order;

-- Re-try with different data
INSERT INTO order_items (order_id, product_id, quantity) VALUES (1001, 5, 1);

RELEASE SAVEPOINT after_order;  -- Remove the savepoint (optional)
COMMIT;
```

### Transaction Isolation Levels

Isolation controls what concurrent transactions can see of each other's uncommitted changes. Higher isolation = more safety, lower performance.

```sql
-- Set isolation level for the current transaction
BEGIN ISOLATION LEVEL SERIALIZABLE;

-- Or set for the session
SET TRANSACTION ISOLATION LEVEL READ COMMITTED;
```

| Isolation Level | Dirty Read | Non-repeatable Read | Phantom Read | Performance |
|---|---|---|---|---|
| `READ UNCOMMITTED` | ✅ Possible | ✅ Possible | ✅ Possible | Highest |
| `READ COMMITTED` | ❌ Prevented | ✅ Possible | ✅ Possible | High |
| `REPEATABLE READ` | ❌ Prevented | ❌ Prevented | ✅ Possible | Medium |
| `SERIALIZABLE` | ❌ Prevented | ❌ Prevented | ❌ Prevented | Lowest |

**Dirty Read** — reading uncommitted data from another transaction. If that transaction rolls back, you read data that never existed.

**Non-repeatable Read** — reading the same row twice within a transaction and getting different values because another transaction committed between your reads.

**Phantom Read** — running the same query twice and getting different sets of rows because another transaction inserted/deleted rows matching your WHERE clause.

:::tip[PostgreSQL ignores READ UNCOMMITTED]
PostgreSQL implements `READ COMMITTED` as its minimum — even if you set `READ UNCOMMITTED`, you still get `READ COMMITTED` behavior. Most applications work fine with `READ COMMITTED`. Use `SERIALIZABLE` for financial transactions or operations requiring strict consistency.
:::

## Views

A view is a named, saved query. It behaves like a table but computes its data on demand.

```sql
-- Create a view
CREATE VIEW active_users AS
SELECT id, email, username, created_at
FROM users
WHERE is_active = true AND deleted_at IS NULL;

-- Query a view just like a table
SELECT * FROM active_users WHERE created_at >= NOW() - INTERVAL '30 days';

-- Update a view definition
CREATE OR REPLACE VIEW active_users AS
SELECT id, email, username, role, created_at
FROM users
WHERE is_active = true AND deleted_at IS NULL;

-- Drop a view
DROP VIEW IF EXISTS active_users;
```

### Updatable Views

Simple views (single table, no aggregation, no DISTINCT) can be updated:

```sql
CREATE VIEW customer_view AS
SELECT id, name, email, country
FROM customers
WHERE country = 'US';

-- This works on simple views
UPDATE customer_view SET email = 'new@example.com' WHERE id = 42;

-- WITH CHECK OPTION prevents updates that would make the row invisible in the view
CREATE OR REPLACE VIEW us_customers AS
SELECT * FROM customers WHERE country = 'US'
WITH CHECK OPTION;

-- This would fail — the update removes the row from the view
UPDATE us_customers SET country = 'CA' WHERE id = 1;
```

### Materialized Views (PostgreSQL)

Materialized views store the query results on disk. They must be refreshed when the underlying data changes.

```sql
CREATE MATERIALIZED VIEW monthly_revenue AS
SELECT
  DATE_TRUNC('month', created_at) AS month,
  SUM(total) AS revenue,
  COUNT(*) AS order_count
FROM orders
WHERE status = 'completed'
GROUP BY 1;

-- Create an index on the materialized view
CREATE INDEX ON monthly_revenue (month);

-- Refresh (blocks reads until done)
REFRESH MATERIALIZED VIEW monthly_revenue;

-- Refresh concurrently (non-blocking, requires unique index)
CREATE UNIQUE INDEX ON monthly_revenue (month);
REFRESH MATERIALIZED VIEW CONCURRENTLY monthly_revenue;
```

## Stored Procedures and Functions

Stored procedures encapsulate reusable logic in the database. The SQL syntax varies significantly by database.

<Tabs>
<TabItem label="PostgreSQL (PL/pgSQL)">
```sql
-- Function (returns a value)
CREATE OR REPLACE FUNCTION get_user_order_count(p_user_id INTEGER)
RETURNS INTEGER
LANGUAGE plpgsql
AS $$
DECLARE
  v_count INTEGER;
BEGIN
  SELECT COUNT(*) INTO v_count
  FROM orders
  WHERE user_id = p_user_id AND status = 'completed';

  RETURN v_count;
END;
$$;

-- Usage
SELECT get_user_order_count(42);

-- Procedure (no return value, can use COMMIT/ROLLBACK)
CREATE OR REPLACE PROCEDURE archive_old_orders(p_days INTEGER DEFAULT 365)
LANGUAGE plpgsql
AS $$
DECLARE
  v_count INTEGER;
BEGIN
  INSERT INTO orders_archive
  SELECT * FROM orders
  WHERE created_at < NOW() - (p_days || ' days')::INTERVAL;

  GET DIAGNOSTICS v_count = ROW_COUNT;

  DELETE FROM orders
  WHERE created_at < NOW() - (p_days || ' days')::INTERVAL;

  RAISE NOTICE 'Archived % orders', v_count;
END;
$$;

-- Usage
CALL archive_old_orders(730);
```
</TabItem>

-- Simple SQL function (inline optimizable)
CREATE OR REPLACE FUNCTION user_full_name(p_user_id INTEGER)
RETURNS TEXT
LANGUAGE sql STABLE
AS $$
  SELECT first_name || ' ' || last_name
  FROM users
  WHERE id = p_user_id;
$$;

## Triggers

Triggers automatically execute a function before or after data changes.

```sql
-- 1. Create the trigger function
CREATE OR REPLACE FUNCTION update_updated_at()
RETURNS TRIGGER
LANGUAGE plpgsql
AS $$
BEGIN
  NEW.updated_at = NOW();
  RETURN NEW;
END;
$$;

-- 2. Create the trigger
CREATE TRIGGER users_updated_at
  BEFORE UPDATE ON users
  FOR EACH ROW
  EXECUTE FUNCTION update_updated_at();

-- Audit log trigger
CREATE OR REPLACE FUNCTION audit_user_changes()
RETURNS TRIGGER
LANGUAGE plpgsql
AS $$
BEGIN
  INSERT INTO audit_log (
    table_name, operation, old_data, new_data, changed_at, changed_by
  ) VALUES (
    TG_TABLE_NAME,
    TG_OP,
    CASE WHEN TG_OP = 'DELETE' THEN TO_JSONB(OLD) ELSE NULL END,
    CASE WHEN TG_OP != 'DELETE' THEN TO_JSONB(NEW) ELSE NULL END,
    NOW(),
    CURRENT_USER
  );
  RETURN COALESCE(NEW, OLD);
END;
$$;

CREATE TRIGGER users_audit
  AFTER INSERT OR UPDATE OR DELETE ON users
  FOR EACH ROW
  EXECUTE FUNCTION audit_user_changes();
```

:::caution[Triggers add hidden complexity]
Triggers execute invisibly from the application's perspective, which can make debugging difficult. Use triggers for cross-cutting concerns (audit logging, updated_at timestamps), but avoid complex business logic in triggers — keep that in the application layer.
:::

## Query Planning with EXPLAIN

`EXPLAIN` shows the query plan without executing. `EXPLAIN ANALYZE` executes and shows actual vs estimated statistics.

```sql
-- Show estimated plan (no execution)
EXPLAIN SELECT * FROM orders WHERE customer_id = 42;

-- Execute and show actual timings
EXPLAIN ANALYZE SELECT * FROM orders WHERE customer_id = 42;

-- Full analysis with buffer information
EXPLAIN (ANALYZE, BUFFERS, FORMAT TEXT)
SELECT o.*, c.name
FROM orders o
JOIN customers c ON c.id = o.customer_id
WHERE o.status = 'pending';
```

Key output to look for:
- **Seq Scan** — full table scan; consider an index if the table is large
- **Index Scan** — using an index; good
- **Index Only Scan** — all data from the index, no heap access; best
- **Bitmap Heap Scan** — efficient for moderate selectivity
- **cost=X..Y** — estimated startup cost..total cost (in arbitrary units)
- **rows=N** — estimated row count (compare to actual)
- **actual time=X..Y** — actual timing in milliseconds

## Common Performance Anti-Patterns

```sql
-- ❌ SELECT * — fetches more data than needed, breaks index-only scans
SELECT * FROM orders WHERE customer_id = 42;
-- ✅ Select only what you need
SELECT id, total, status, created_at FROM orders WHERE customer_id = 42;

-- ❌ Function on indexed column in WHERE (disables index)
SELECT * FROM users WHERE LOWER(email) = 'alice@example.com';
-- ✅ Use a functional index, or store lowercase
CREATE INDEX idx_users_email_lower ON users (LOWER(email));

-- ❌ Implicit type cast (disables index)
SELECT * FROM orders WHERE id = '42';  -- id is INTEGER, '42' is TEXT
-- ✅ Match types
SELECT * FROM orders WHERE id = 42;

-- ❌ N+1 query pattern (1 query + N queries)
-- Application loop: for each user, run: SELECT * FROM orders WHERE user_id = ?
-- ✅ Single join query
SELECT u.id, u.email, o.id AS order_id, o.total
FROM users u
LEFT JOIN orders o ON o.user_id = u.id
WHERE u.id = ANY(ARRAY[1, 2, 3, 4, 5]);
```

## Normalization

Normalization organizes a database to reduce redundancy and improve integrity. Each normal form builds on the previous.

### First Normal Form (1NF)

Each column contains atomic (indivisible) values. No repeating groups or arrays.

```sql
-- ❌ Violates 1NF: multiple values in one column
-- phone_numbers: '555-1234, 555-5678'

-- ✅ 1NF: separate table for multi-valued data
CREATE TABLE user_phone_numbers (
  user_id  INTEGER REFERENCES users(id),
  phone    VARCHAR(20) NOT NULL,
  type     TEXT,
  PRIMARY KEY (user_id, phone)
);
```

### Second Normal Form (2NF)

All non-key columns must depend on the **entire** primary key (eliminates partial dependencies). Only applies to tables with composite primary keys.

```sql
-- ❌ Violates 2NF: product_name depends only on product_id, not the full PK
-- order_items (order_id, product_id, product_name, quantity)

-- ✅ 2NF: move product_name to the products table
-- order_items (order_id, product_id, quantity)
-- products (product_id, product_name, price)
```

### Third Normal Form (3NF)

No non-key column depends on another non-key column (eliminates transitive dependencies).

```sql
-- ❌ Violates 3NF: zip_code determines city and state (transitive dependency)
-- customers (id, zip_code, city, state, name)

-- ✅ 3NF: separate zip code data
-- customers (id, zip_code, name)
-- zip_codes (zip_code, city, state)
```

### Boyce-Codd Normal Form (BCNF)

Stricter than 3NF — every determinant must be a candidate key. Most practical schemas that satisfy 3NF also satisfy BCNF.

:::tip[Denormalization for performance]
While normalization is the ideal starting point, sometimes denormalization (intentionally adding redundancy) is necessary for performance. Materialized views, cached aggregate columns, and reporting tables are common denormalization strategies. Always start normalized and denormalize with measurement.
:::

## Index Types Overview

| Index Type | Best For | Notes |
|---|---|---|
| **B-tree** (default) | Equality, range, ORDER BY, most WHERE clauses | Default for all databases |
| **Hash** | Equality only (`=`) | Faster than B-tree for equality; no range support |
| **GIN** (PostgreSQL) | Arrays, JSONB, full-text search | Multiple values per row |
| **GiST** (PostgreSQL) | Geometric types, full-text, ranges | Generalized — supports custom types |
| **BRIN** (PostgreSQL) | Sequential data (timestamps, IDs) | Very small, best for append-only data |
| **Partial index** | Queries with WHERE conditions | Small, fast for filtered queries |
| **Expression index** | Functions on columns | Index `LOWER(email)` for case-insensitive search |

```sql
-- B-tree (default)
CREATE INDEX idx_orders_created ON orders (created_at DESC);

-- Partial index (only indexes active users)
CREATE INDEX idx_active_users ON users (email) WHERE is_active = true;

-- Expression index (case-insensitive email lookup)
CREATE INDEX idx_users_email_ci ON users (LOWER(email));

-- Composite index (order matters: most selective first, or by query pattern)
CREATE INDEX idx_orders_status_created ON orders (status, created_at DESC);
```

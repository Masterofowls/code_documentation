This page covers the advanced SQL features needed for building production systems: transactions, isolation, views, procedural code, triggers, query planning, and normalization.

## Transactions

A transaction groups multiple SQL statements into a single atomic unit of work.

### Basic Transaction Syntax

```sql
-- PostgreSQL / MySQL / SQLite
BEGIN;                     -- or: START TRANSACTION;

UPDATE accounts SET balance = balance - 500 WHERE id = 1;
UPDATE accounts SET balance = balance + 500 WHERE id = 2;

COMMIT;                    -- Make changes permanent

-- OR to undo all changes:
ROLLBACK;
```

### SAVEPOINT

Savepoints allow partial rollback within a transaction:

```sql
BEGIN;

INSERT INTO orders (customer_id, total) VALUES (1, 150.00)
RETURNING id INTO order_id;

SAVEPOINT after_order;     -- Checkpoint

INSERT INTO order_items (order_id, product_id, quantity)
VALUES (order_id, 5, 2);   -- This might fail

-- If the above fails:
ROLLBACK TO SAVEPOINT after_order;   -- Undo only the order_items insert
-- order still exists; we can try a different product

RELEASE SAVEPOINT after_order;       -- Optional: remove the savepoint

COMMIT;
```

## Isolation Levels

Isolation levels control how much a transaction is isolated from the effects of concurrent transactions. Higher isolation = more consistency but more contention.

### The Four Levels

```sql
-- Set for current transaction
BEGIN TRANSACTION ISOLATION LEVEL READ COMMITTED;

-- Or set as session default (PostgreSQL)
SET SESSION CHARACTERISTICS AS TRANSACTION ISOLATION LEVEL SERIALIZABLE;
```

| Level | Dirty Read | Non-Repeatable Read | Phantom Read |
|---|---|---|---|
| READ UNCOMMITTED | ✅ Possible | ✅ Possible | ✅ Possible |
| READ COMMITTED | ❌ Prevented | ✅ Possible | ✅ Possible |
| REPEATABLE READ | ❌ Prevented | ❌ Prevented | ✅ Possible |
| SERIALIZABLE | ❌ Prevented | ❌ Prevented | ❌ Prevented |

**Dirty Read**: Reading uncommitted changes from another transaction.

**Non-Repeatable Read**: Re-reading a row within the same transaction gets different data because another transaction committed a change.

**Phantom Read**: A query within a transaction returns different rows on re-execution because another transaction inserted or deleted rows.

:::tip[PostgreSQL Default]
PostgreSQL defaults to `READ COMMITTED`. Most applications work correctly with this level. Use `SERIALIZABLE` only for financial transactions or complex multi-step operations that require true isolation.
:::

## Views

A view is a named, stored SELECT query that acts like a table.

### Creating Views

```sql
-- Simple view
CREATE VIEW active_products AS
SELECT id, sku, name, price, category_id
FROM products
WHERE status = 'active';

-- View with joins
CREATE VIEW order_summary AS
SELECT
  o.id            AS order_id,
  o.created_at,
  o.status,
  o.total,
  c.name          AS customer_name,
  c.email         AS customer_email,
  COUNT(oi.product_id) AS item_count
FROM orders o
JOIN customers c  ON c.id = o.customer_id
JOIN order_items oi ON oi.order_id = o.id
GROUP BY o.id, o.created_at, o.status, o.total, c.name, c.email;

-- Replace an existing view
CREATE OR REPLACE VIEW active_products AS
SELECT id, sku, name, price, category_id, created_at
FROM products
WHERE status = 'active';

-- Drop
DROP VIEW IF EXISTS active_products;
```

### Updatable Views

Simple views (single table, no GROUP BY, DISTINCT, aggregates, or joins) are automatically updatable:

```sql
CREATE VIEW active_customers AS
SELECT id, name, email, country
FROM customers
WHERE status = 'active';

-- This works because the view is simple
UPDATE active_customers SET email = 'new@example.com' WHERE id = 1;

-- WITH CHECK OPTION prevents updates that would remove the row from the view
CREATE VIEW active_customers AS
SELECT id, name, email, country, status
FROM customers
WHERE status = 'active'
WITH CHECK OPTION;

-- This would fail: setting status = 'inactive' would make the row invisible in the view
UPDATE active_customers SET status = 'inactive' WHERE id = 1;
```

### Materialized Views (PostgreSQL)

A materialized view stores the query result on disk — it's fast to read but must be refreshed.

```sql
CREATE MATERIALIZED VIEW product_category_stats AS
SELECT
  c.id            AS category_id,
  c.name          AS category_name,
  COUNT(p.id)     AS product_count,
  AVG(p.price)    AS avg_price,
  MIN(p.price)    AS min_price,
  MAX(p.price)    AS max_price
FROM categories c
LEFT JOIN products p ON p.category_id = c.id AND p.status = 'active'
GROUP BY c.id, c.name;

-- Create index on the materialized view
CREATE INDEX ON product_category_stats(category_id);

-- Refresh the data (blocking)
REFRESH MATERIALIZED VIEW product_category_stats;

-- Refresh without blocking reads (requires UNIQUE index)
CREATE UNIQUE INDEX ON product_category_stats(category_id);
REFRESH MATERIALIZED VIEW CONCURRENTLY product_category_stats;
```

## Stored Procedures and Functions

### SQL Functions (Simple)

```sql
-- Scalar function: returns one value
CREATE OR REPLACE FUNCTION calculate_tax(price NUMERIC, rate NUMERIC DEFAULT 0.10)
RETURNS NUMERIC AS $$
  SELECT ROUND(price * rate, 2);
$$ LANGUAGE SQL IMMUTABLE;

SELECT name, price, calculate_tax(price) AS tax FROM products;

-- Table-returning function
CREATE OR REPLACE FUNCTION get_customer_orders(p_customer_id BIGINT)
RETURNS TABLE (order_id BIGINT, total NUMERIC, status TEXT, created_at TIMESTAMPTZ) AS $$
  SELECT id, total, status, created_at
  FROM orders
  WHERE customer_id = p_customer_id
  ORDER BY created_at DESC;
$$ LANGUAGE SQL STABLE;

SELECT * FROM get_customer_orders(42);
```

### PL/pgSQL Functions (PostgreSQL)

```sql
CREATE OR REPLACE FUNCTION transfer_funds(
  p_from_id  BIGINT,
  p_to_id    BIGINT,
  p_amount   NUMERIC
)
RETURNS VOID AS $$
DECLARE
  v_from_balance NUMERIC;
BEGIN
  -- Lock both rows to prevent concurrent modification
  SELECT balance INTO v_from_balance
  FROM accounts
  WHERE id = p_from_id
  FOR UPDATE;

  IF v_from_balance < p_amount THEN
    RAISE EXCEPTION 'Insufficient funds: balance % < amount %', v_from_balance, p_amount;
  END IF;

  UPDATE accounts SET balance = balance - p_amount WHERE id = p_from_id;
  UPDATE accounts SET balance = balance + p_amount WHERE id = p_to_id;

  INSERT INTO transfer_log (from_id, to_id, amount, transferred_at)
  VALUES (p_from_id, p_to_id, p_amount, NOW());

  RAISE NOTICE 'Transfer complete: % → % for %', p_from_id, p_to_id, p_amount;
END;
$$ LANGUAGE plpgsql;

-- Call it
SELECT transfer_funds(1, 2, 100.00);
```

## Triggers

Triggers execute stored code automatically when data changes.

### PostgreSQL Trigger

```sql
-- Trigger function (must return trigger)
CREATE OR REPLACE FUNCTION update_updated_at()
RETURNS TRIGGER AS $$
BEGIN
  NEW.updated_at = NOW();
  RETURN NEW;
END;
$$ LANGUAGE plpgsql;

-- Attach trigger to a table
CREATE TRIGGER trg_products_updated_at
  BEFORE UPDATE ON products
  FOR EACH ROW
  EXECUTE FUNCTION update_updated_at();

-- Audit log trigger
CREATE OR REPLACE FUNCTION audit_changes()
RETURNS TRIGGER AS $$
BEGIN
  IF TG_OP = 'DELETE' THEN
    INSERT INTO audit_log (table_name, operation, old_data, changed_by, changed_at)
    VALUES (TG_TABLE_NAME, 'DELETE', row_to_json(OLD), current_user, NOW());
    RETURN OLD;
  ELSE
    INSERT INTO audit_log (table_name, operation, old_data, new_data, changed_by, changed_at)
    VALUES (TG_TABLE_NAME, TG_OP, row_to_json(OLD), row_to_json(NEW), current_user, NOW());
    RETURN NEW;
  END IF;
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER trg_orders_audit
  AFTER INSERT OR UPDATE OR DELETE ON orders
  FOR EACH ROW
  EXECUTE FUNCTION audit_changes();
```

## EXPLAIN — Query Planning

`EXPLAIN` shows how the database plans to execute a query without running it. `EXPLAIN ANALYZE` runs it and shows actual times.

```sql
-- Show query plan (PostgreSQL)
EXPLAIN SELECT * FROM orders WHERE customer_id = 42;

-- Run and show actual performance
EXPLAIN (ANALYZE, BUFFERS, FORMAT TEXT)
SELECT o.id, c.name, SUM(oi.unit_price * oi.quantity)
FROM orders o
JOIN customers c ON c.id = o.customer_id
JOIN order_items oi ON oi.order_id = o.id
WHERE o.status = 'completed'
GROUP BY o.id, c.name
ORDER BY 3 DESC
LIMIT 10;
```

### Reading EXPLAIN Output

```
Sort  (cost=1234.56..1234.78 rows=89 width=48) (actual time=12.34..12.45 rows=10 loops=1)
  -> HashAggregate  (cost=1228.00..1230.00 rows=89 width=48)
      -> Hash Join  (cost=45.00..900.00 rows=1200 width=40)
          Hash Cond: (oi.order_id = o.id)
          -> Seq Scan on order_items oi  (cost=0..800 rows=15000 width=24)
          -> Hash  (cost=44.00..44.00 rows=89 width=28)
              -> Index Scan using idx_orders_status on orders o  (cost=0.29..44 rows=89 width=28)
                  Index Cond: (status = 'completed')
```

Key terms:
- **Seq Scan**: Full table scan — may indicate a missing index
- **Index Scan**: Using a B-tree index to find rows
- **Index Only Scan**: All data served from the index (no table access)
- **Bitmap Heap Scan**: Multiple index entries collected, then table accessed
- **Hash Join**: Build hash table from smaller table, probe with larger
- **Merge Join**: Both inputs sorted on join key; efficient for large sorted sets
- **Nested Loop**: For each row in outer, scan inner — good for small inner tables
- `cost=start..total`: Estimated cost (arbitrary units)
- `actual time=start..end`: Measured time in milliseconds
- `rows=N`: Number of rows returned

## Normalization

Normalization organizes tables to reduce data redundancy and improve data integrity.

### First Normal Form (1NF)

**Rule**: Each column contains only atomic (indivisible) values; no repeating groups.

```sql
-- VIOLATES 1NF: phone_numbers is multi-valued
CREATE TABLE contacts_bad (
  id           INT PRIMARY KEY,
  name         TEXT,
  phone_numbers TEXT   -- "555-1234, 555-5678" — NOT atomic!
);

-- SATISFIES 1NF: separate table for phone numbers
CREATE TABLE contacts (
  id   INT PRIMARY KEY,
  name TEXT NOT NULL
);

CREATE TABLE contact_phones (
  contact_id   INT REFERENCES contacts(id),
  phone_number VARCHAR(20) NOT NULL,
  phone_type   TEXT,
  PRIMARY KEY (contact_id, phone_number)
);
```

### Second Normal Form (2NF)

**Rule**: 1NF + every non-key attribute is fully dependent on the entire primary key (no partial dependencies).

```sql
-- VIOLATES 2NF: product_name depends only on product_id, not the full PK
CREATE TABLE order_items_bad (
  order_id     INT,
  product_id   INT,
  product_name TEXT,   -- Depends only on product_id — partial dependency!
  quantity     INT,
  PRIMARY KEY (order_id, product_id)
);

-- SATISFIES 2NF: move product_name to products table
CREATE TABLE products    (id INT PRIMARY KEY, name TEXT NOT NULL);
CREATE TABLE order_items (
  order_id   INT REFERENCES orders(id),
  product_id INT REFERENCES products(id),
  quantity   INT NOT NULL,
  PRIMARY KEY (order_id, product_id)
);
```

### Third Normal Form (3NF)

**Rule**: 2NF + no transitive dependencies (non-key attributes depend only on the primary key, not on other non-key attributes).

```sql
-- VIOLATES 3NF: zip_code → city (transitive dependency)
CREATE TABLE customers_bad (
  id       INT PRIMARY KEY,
  name     TEXT,
  zip_code VARCHAR(10),
  city     TEXT,       -- Determined by zip_code, not by id — transitive!
  state    TEXT        -- Also determined by zip_code
);

-- SATISFIES 3NF: move city/state to a zip_codes table
CREATE TABLE zip_codes (
  zip_code VARCHAR(10) PRIMARY KEY,
  city     TEXT NOT NULL,
  state    CHAR(2) NOT NULL
);

CREATE TABLE customers (
  id       INT PRIMARY KEY,
  name     TEXT NOT NULL,
  zip_code VARCHAR(10) REFERENCES zip_codes(zip_code)
);
```

### BCNF (Boyce-Codd Normal Form)

**Rule**: For every non-trivial functional dependency X → Y, X must be a superkey.

BCNF handles edge cases where 3NF is insufficient due to overlapping candidate keys.

:::tip[When to Denormalize]
High normalization is ideal for write-heavy OLTP systems. For read-heavy analytics and reporting (OLAP), deliberate denormalization can dramatically improve query performance by reducing joins. Always profile before denormalizing.
:::

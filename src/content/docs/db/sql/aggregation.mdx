Aggregation collapses multiple rows into a single summary value. Window functions extend this concept — performing calculations across a set of related rows without collapsing them.

## Aggregate Functions

```sql
-- COUNT: number of rows
SELECT COUNT(*) FROM orders;                  -- All rows including NULLs
SELECT COUNT(shipped_at) FROM orders;         -- Only non-NULL shipped_at values
SELECT COUNT(DISTINCT customer_id) FROM orders; -- Distinct customers with orders

-- SUM
SELECT SUM(total) FROM orders WHERE status = 'completed';
SELECT SUM(quantity * unit_price) AS revenue FROM order_items;

-- AVG (ignores NULLs)
SELECT AVG(total) FROM orders;
SELECT ROUND(AVG(price)::numeric, 2) AS avg_price FROM products;

-- MIN / MAX
SELECT MIN(price), MAX(price) FROM products;
SELECT MIN(created_at) AS first_order, MAX(created_at) AS last_order FROM orders;

-- String aggregation
SELECT
  customer_id,
  STRING_AGG(product_name, ', ' ORDER BY product_name) AS products
FROM order_items oi
JOIN products p ON p.id = oi.product_id
GROUP BY customer_id;

-- Array aggregation (PostgreSQL)
SELECT
  category_id,
  ARRAY_AGG(name ORDER BY name) AS product_names
FROM products
GROUP BY category_id;

-- JSON aggregation (PostgreSQL)
SELECT
  order_id,
  JSON_AGG(
    JSON_BUILD_OBJECT('product', name, 'qty', quantity)
    ORDER BY name
  ) AS items
FROM order_items oi
JOIN products p ON p.id = oi.product_id
GROUP BY order_id;
```

## GROUP BY

```sql
-- Single column grouping
SELECT status, COUNT(*) AS count
FROM orders
GROUP BY status;

-- Multiple columns
SELECT
  DATE_TRUNC('month', created_at) AS month,
  status,
  COUNT(*) AS count,
  SUM(total) AS revenue
FROM orders
GROUP BY DATE_TRUNC('month', created_at), status
ORDER BY month DESC, status;

-- Group by expression
SELECT
  EXTRACT(YEAR FROM created_at) AS year,
  EXTRACT(MONTH FROM created_at) AS month,
  COUNT(*) AS signups
FROM users
GROUP BY 1, 2   -- Can reference by position
ORDER BY 1 DESC, 2 DESC;
```

## HAVING

`HAVING` filters aggregated results (like `WHERE` but applied after `GROUP BY`):

```sql
-- Only categories with more than 10 products
SELECT category_id, COUNT(*) AS product_count
FROM products
GROUP BY category_id
HAVING COUNT(*) > 10;

-- Customers who spent more than $1000 in the last 90 days
SELECT
  customer_id,
  SUM(total) AS total_spent
FROM orders
WHERE created_at >= NOW() - INTERVAL '90 days'
  AND status = 'completed'
GROUP BY customer_id
HAVING SUM(total) > 1000
ORDER BY total_spent DESC;
```

:::tip[WHERE vs HAVING]
`WHERE` filters individual rows **before** grouping. `HAVING` filters groups **after** aggregation. Use `WHERE` whenever possible — it reduces the number of rows that need to be grouped. Only use `HAVING` when filtering by an aggregate value.
:::

## ROLLUP, CUBE, and GROUPING SETS

These extensions generate multiple groupings in a single query.

### ROLLUP — Hierarchical Subtotals

```sql
-- Sales by year → month → total
SELECT
  EXTRACT(YEAR FROM created_at)  AS year,
  EXTRACT(MONTH FROM created_at) AS month,
  SUM(total)                     AS revenue
FROM orders
WHERE status = 'completed'
GROUP BY ROLLUP (
  EXTRACT(YEAR FROM created_at),
  EXTRACT(MONTH FROM created_at)
)
ORDER BY year NULLS LAST, month NULLS LAST;
-- NULLs in year = grand total; NULLs in month = year subtotal
```

### CUBE — All Dimension Combinations

```sql
-- Sales by every combination of year, month, and country
SELECT
  EXTRACT(YEAR FROM o.created_at) AS year,
  c.country,
  SUM(o.total)                    AS revenue
FROM orders o
JOIN customers c ON c.id = o.customer_id
GROUP BY CUBE (
  EXTRACT(YEAR FROM o.created_at),
  c.country
);
```

### GROUPING SETS — Explicit Groups

```sql
-- Explicit control over which groupings to compute
SELECT
  category,
  region,
  SUM(revenue) AS total
FROM sales_data
GROUP BY GROUPING SETS (
  (category, region),  -- By category and region together
  (category),          -- By category only
  (region),            -- By region only
  ()                   -- Grand total
);
```

## FILTER Clause (Conditional Aggregation)

The `FILTER` clause applies a condition inside an aggregate function, computing the aggregate over only the matching rows:

```sql
SELECT
  COUNT(*)                                      AS total_orders,
  COUNT(*) FILTER (WHERE status = 'completed')  AS completed,
  COUNT(*) FILTER (WHERE status = 'refunded')   AS refunded,
  COUNT(*) FILTER (WHERE status = 'cancelled')  AS cancelled,
  SUM(total) FILTER (WHERE status = 'completed') AS completed_revenue,
  ROUND(
    100.0 * COUNT(*) FILTER (WHERE status = 'completed') / COUNT(*), 1
  ) AS completion_rate_pct
FROM orders;
```

```sql
-- Cohort retention: count users active in each month after signup
SELECT
  DATE_TRUNC('month', u.created_at) AS cohort_month,
  COUNT(DISTINCT u.id) AS cohort_size,
  COUNT(DISTINCT u.id) FILTER (
    WHERE o.created_at < u.created_at + INTERVAL '30 days'
  ) AS active_month_1,
  COUNT(DISTINCT u.id) FILTER (
    WHERE o.created_at BETWEEN u.created_at + INTERVAL '30 days'
                           AND u.created_at + INTERVAL '60 days'
  ) AS active_month_2
FROM users u
LEFT JOIN orders o ON o.customer_id = u.id
GROUP BY cohort_month;
```

## Window Functions

Window functions compute a value for each row based on a "window" of related rows — without collapsing the result set. They use the `OVER` clause.

### Basic Syntax

```sql
function_name(args) OVER (
  [PARTITION BY col1, col2, ...]    -- Divides rows into groups
  [ORDER BY col3 ASC/DESC]          -- Orders rows within the window
  [frame_clause]                    -- Specifies the frame extent
)
```

```sql
-- Running total of revenue per customer
SELECT
  customer_id,
  order_id,
  total,
  SUM(total) OVER (PARTITION BY customer_id ORDER BY created_at) AS running_total
FROM orders
WHERE status = 'completed';
```

## Ranking Functions

```sql
SELECT
  name,
  department,
  salary,

  -- No gaps in rank, doesn't skip numbers
  DENSE_RANK() OVER (PARTITION BY department ORDER BY salary DESC) AS dept_rank,

  -- Skips numbers after ties
  RANK()       OVER (PARTITION BY department ORDER BY salary DESC) AS rank_with_gaps,

  -- Unique sequential number regardless of ties
  ROW_NUMBER() OVER (PARTITION BY department ORDER BY salary DESC, id) AS row_num,

  -- Divide into N buckets (quartiles, percentiles)
  NTILE(4)     OVER (ORDER BY salary DESC) AS salary_quartile
FROM employees;
```

### Using Ranking to Deduplicate

A common pattern — keep only the most recent row per group:

```sql
-- Keep only the latest address per user
WITH ranked_addresses AS (
  SELECT *,
    ROW_NUMBER() OVER (
      PARTITION BY user_id
      ORDER BY created_at DESC
    ) AS rn
  FROM user_addresses
)
SELECT * FROM ranked_addresses WHERE rn = 1;
```

## Offset Functions

```sql
SELECT
  date,
  revenue,

  -- Previous row's value (NULL for first row)
  LAG(revenue, 1) OVER (ORDER BY date) AS prev_day_revenue,

  -- Next row's value (NULL for last row)
  LEAD(revenue, 1) OVER (ORDER BY date) AS next_day_revenue,

  -- Day-over-day change
  revenue - LAG(revenue, 1) OVER (ORDER BY date) AS daily_change,

  -- % change
  ROUND(
    100.0 * (revenue - LAG(revenue, 1) OVER (ORDER BY date))
    / NULLIF(LAG(revenue, 1) OVER (ORDER BY date), 0),
    1
  ) AS pct_change,

  -- First and last in window
  FIRST_VALUE(revenue) OVER (ORDER BY date ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW) AS first_revenue,
  LAST_VALUE(revenue)  OVER (ORDER BY date ROWS BETWEEN CURRENT ROW AND UNBOUNDED FOLLOWING) AS last_revenue
FROM daily_revenue;
```

## Window Frame Clauses

The frame clause specifies which rows relative to the current row are included in the window:

```sql
-- ROWS BETWEEN: frame based on physical row positions
SUM(revenue) OVER (
  ORDER BY date
  ROWS BETWEEN 6 PRECEDING AND CURRENT ROW   -- Last 7 days (7-day rolling sum)
)

-- RANGE BETWEEN: frame based on value ranges
SUM(revenue) OVER (
  ORDER BY date
  RANGE BETWEEN INTERVAL '7 days' PRECEDING AND CURRENT ROW
)

-- Running total from start to current row
SUM(revenue) OVER (
  ORDER BY date
  ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW
)

-- Total of all rows (same for every row — equivalent to a subquery)
SUM(revenue) OVER (
  ROWS BETWEEN UNBOUNDED PRECEDING AND UNBOUNDED FOLLOWING
)
```

### Running Totals and Moving Averages

```sql
SELECT
  date,
  revenue,

  -- Cumulative sum
  SUM(revenue) OVER (ORDER BY date
    ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW
  ) AS cumulative_revenue,

  -- 7-day moving average
  ROUND(
    AVG(revenue) OVER (ORDER BY date
      ROWS BETWEEN 6 PRECEDING AND CURRENT ROW
    )::numeric, 2
  ) AS moving_avg_7d,

  -- % of total
  ROUND(
    100.0 * revenue / SUM(revenue) OVER (),
    2
  ) AS pct_of_total
FROM daily_revenue
ORDER BY date;
```

## Named Windows

Use the `WINDOW` clause to define a named window and reuse it across multiple functions:

```sql
SELECT
  department,
  name,
  salary,
  RANK()         OVER dept_salary AS rank,
  DENSE_RANK()   OVER dept_salary AS dense_rank,
  ROW_NUMBER()   OVER dept_salary AS row_num,
  SUM(salary)    OVER dept_window AS dept_total,
  AVG(salary)    OVER dept_window AS dept_avg
FROM employees
WINDOW
  dept_salary AS (PARTITION BY department ORDER BY salary DESC),
  dept_window AS (PARTITION BY department)
ORDER BY department, rank;
```

## Practical Example: Sales Dashboard

```sql
WITH daily_metrics AS (
  SELECT
    DATE_TRUNC('day', created_at) AS day,
    COUNT(*)    AS order_count,
    SUM(total)  AS revenue,
    COUNT(DISTINCT customer_id) AS unique_customers
  FROM orders
  WHERE status = 'completed'
    AND created_at >= NOW() - INTERVAL '90 days'
  GROUP BY 1
)
SELECT
  day,
  order_count,
  revenue,
  unique_customers,

  -- Running totals
  SUM(order_count) OVER (ORDER BY day ROWS UNBOUNDED PRECEDING) AS cumulative_orders,
  SUM(revenue)     OVER (ORDER BY day ROWS UNBOUNDED PRECEDING) AS cumulative_revenue,

  -- 7-day rolling averages
  ROUND(AVG(revenue)       OVER (ORDER BY day ROWS 6 PRECEDING)::numeric, 2) AS revenue_7d_avg,
  ROUND(AVG(order_count)   OVER (ORDER BY day ROWS 6 PRECEDING)::numeric, 1) AS orders_7d_avg,

  -- Day-over-day change
  revenue - LAG(revenue) OVER (ORDER BY day) AS revenue_vs_yesterday,

  -- Week-over-week change
  revenue - LAG(revenue, 7) OVER (ORDER BY day) AS revenue_vs_last_week
FROM daily_metrics
ORDER BY day DESC;
```

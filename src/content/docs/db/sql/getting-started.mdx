This guide covers the fundamentals of defining database structure using DDL (Data Definition Language) — creating tables with appropriate data types, constraints, and relationships, then modifying and removing those structures as requirements evolve.

## Creating a Database

```sql
-- PostgreSQL
CREATE DATABASE myapp
    ENCODING 'UTF8'
    LC_COLLATE 'en_US.UTF-8'
    LC_CTYPE 'en_US.UTF-8';

-- Connect to the database
\c myapp     -- psql command

-- MySQL
CREATE DATABASE myapp CHARACTER SET utf8mb4 COLLATE utf8mb4_unicode_ci;
USE myapp;
```

## CREATE TABLE

The `CREATE TABLE` statement defines a new table with its columns and constraints.

```sql
CREATE TABLE users (
    id          BIGSERIAL PRIMARY KEY,
    email       VARCHAR(255) NOT NULL UNIQUE,
    username    VARCHAR(50)  NOT NULL UNIQUE,
    full_name   VARCHAR(200),
    password_hash TEXT       NOT NULL,
    is_active   BOOLEAN      NOT NULL DEFAULT TRUE,
    created_at  TIMESTAMPTZ  NOT NULL DEFAULT NOW(),
    updated_at  TIMESTAMPTZ  NOT NULL DEFAULT NOW()
);
```

## Data Types

### Numeric Types

```sql
-- Integer types
id          SMALLINT          -- -32,768 to 32,767 (2 bytes)
count       INTEGER           -- -2.1B to 2.1B (4 bytes)
big_id      BIGINT            -- ±9.2 quintillion (8 bytes)

-- Auto-incrementing (PostgreSQL)
id          SERIAL            -- INTEGER with auto-sequence
id          BIGSERIAL         -- BIGINT with auto-sequence
id          INTEGER GENERATED ALWAYS AS IDENTITY    -- SQL standard
id          INTEGER GENERATED BY DEFAULT AS IDENTITY

-- Exact decimal (use for money, measurements)
price       NUMERIC(10, 2)    -- 10 total digits, 2 after decimal
amount      DECIMAL(15, 4)    -- DECIMAL is alias for NUMERIC

-- Floating point (avoid for money — imprecise)
ratio       REAL              -- 4-byte float (~6 decimal digits precision)
coefficient DOUBLE PRECISION  -- 8-byte float (~15 decimal digits precision)
```

### Character Types

```sql
-- Variable-length (most common)
name        VARCHAR(100)      -- up to 100 chars; stored actual length
bio         TEXT              -- unlimited length; PostgreSQL stores same as VARCHAR

-- Fixed-length (padded with spaces)
country_code CHAR(2)          -- always exactly 2 characters

-- PostgreSQL: VARCHAR vs TEXT are equally performant
-- Use TEXT for variable-length strings without a known maximum
```

### Date and Time Types

```sql
birth_date  DATE              -- '2024-01-15' (date only)
start_time  TIME              -- '09:30:00' (time only, no timezone)
scheduled   TIMESTAMP         -- '2024-01-15 09:30:00' (no timezone)
created_at  TIMESTAMPTZ       -- '2024-01-15 09:30:00+00' (with timezone)
event_time  TIME WITH TIME ZONE

-- Use TIMESTAMPTZ for most timestamps — stores UTC, displays in session tz
```

### Other Types

```sql
active      BOOLEAN           -- TRUE / FALSE / NULL
content     TEXT              -- unbounded text
tags        JSONB             -- PostgreSQL binary JSON
data        JSON              -- PostgreSQL text JSON
image_data  BYTEA             -- binary data (PostgreSQL)
identifier  UUID              -- 'a0eebc99-9c0b-4ef8-bb6d-6bb9bd380a11'
```

## Constraints

Constraints enforce data integrity rules. They can be column-level (inline) or table-level (after all column definitions).

### NOT NULL

```sql
CREATE TABLE products (
    name    TEXT NOT NULL,    -- must provide a value
    sku     TEXT NOT NULL
);
```

### UNIQUE

```sql
-- Column-level unique
email   VARCHAR(255) UNIQUE,

-- Table-level unique (for composite unique constraints)
CONSTRAINT uq_user_email UNIQUE (user_id, email_type)
```

### DEFAULT

```sql
status      VARCHAR(20) DEFAULT 'pending',
created_at  TIMESTAMPTZ DEFAULT NOW(),
score       INTEGER     DEFAULT 0,
settings    JSONB       DEFAULT '{}'::jsonb
```

### CHECK

```sql
-- Inline check
age         INTEGER     CHECK (age >= 0 AND age <= 150),
price       NUMERIC(10,2) CHECK (price >= 0),
status      VARCHAR(20) CHECK (status IN ('pending', 'active', 'cancelled')),

-- Named check constraint
CONSTRAINT chk_price_positive CHECK (price > 0),
CONSTRAINT chk_start_before_end CHECK (start_date < end_date)
```

### PRIMARY KEY

```sql
-- Single column
id  BIGSERIAL PRIMARY KEY,

-- Composite primary key (table-level only)
CREATE TABLE order_items (
    order_id    BIGINT NOT NULL,
    product_id  BIGINT NOT NULL,
    quantity    INTEGER NOT NULL DEFAULT 1,
    PRIMARY KEY (order_id, product_id)
);
```

### FOREIGN KEY

```sql
CREATE TABLE orders (
    id          BIGSERIAL PRIMARY KEY,
    customer_id BIGINT NOT NULL,
    -- Inline foreign key
    CONSTRAINT fk_orders_customer
        FOREIGN KEY (customer_id)
        REFERENCES customers(id)
        ON DELETE RESTRICT       -- prevent deleting customers with orders
        ON UPDATE CASCADE        -- update order.customer_id if customer.id changes
);
```

#### ON DELETE / ON UPDATE Actions

| Action | Description |
|---|---|
| `CASCADE` | Delete/update child rows when parent is deleted/updated |
| `SET NULL` | Set foreign key column to NULL when parent is deleted/updated |
| `SET DEFAULT` | Set foreign key column to its default value |
| `RESTRICT` | Prevent delete/update if child rows exist (checked immediately) |
| `NO ACTION` | Prevent delete/update if child rows exist (checked at end of transaction) |

```sql
-- Common patterns
REFERENCES users(id) ON DELETE CASCADE       -- delete posts when user deleted
REFERENCES categories(id) ON DELETE SET NULL -- keep products, null out category
REFERENCES orders(id) ON DELETE RESTRICT     -- can't delete order with line items
```

## Complete Table Example

```sql
CREATE TABLE blog_posts (
    id              BIGSERIAL PRIMARY KEY,
    author_id       BIGINT          NOT NULL REFERENCES users(id) ON DELETE RESTRICT,
    category_id     INTEGER         REFERENCES categories(id) ON DELETE SET NULL,
    slug            VARCHAR(255)    NOT NULL UNIQUE,
    title           VARCHAR(500)    NOT NULL,
    content         TEXT            NOT NULL,
    excerpt         VARCHAR(1000),
    status          VARCHAR(20)     NOT NULL DEFAULT 'draft',
    view_count      INTEGER         NOT NULL DEFAULT 0,
    published_at    TIMESTAMPTZ,
    created_at      TIMESTAMPTZ     NOT NULL DEFAULT NOW(),
    updated_at      TIMESTAMPTZ     NOT NULL DEFAULT NOW(),

    CONSTRAINT chk_status CHECK (status IN ('draft', 'published', 'archived')),
    CONSTRAINT chk_view_count CHECK (view_count >= 0),
    CONSTRAINT chk_published_has_date
        CHECK (status != 'published' OR published_at IS NOT NULL)
);
```

## ALTER TABLE

Modify an existing table's structure without recreating it.

```sql
-- Add a column
ALTER TABLE users
    ADD COLUMN phone VARCHAR(20);

ALTER TABLE users
    ADD COLUMN last_login TIMESTAMPTZ NOT NULL DEFAULT NOW();

-- Drop a column
ALTER TABLE users
    DROP COLUMN phone;

ALTER TABLE users
    DROP COLUMN IF EXISTS legacy_field;

-- Rename a column
ALTER TABLE users
    RENAME COLUMN full_name TO display_name;

-- Change a column's type
ALTER TABLE products
    ALTER COLUMN price TYPE NUMERIC(15, 4);

-- Change a column's default
ALTER TABLE products
    ALTER COLUMN status SET DEFAULT 'active';
ALTER TABLE products
    ALTER COLUMN status DROP DEFAULT;

-- Add/remove NOT NULL
ALTER TABLE users
    ALTER COLUMN email SET NOT NULL;
ALTER TABLE users
    ALTER COLUMN phone DROP NOT NULL;

-- Add a constraint
ALTER TABLE orders
    ADD CONSTRAINT chk_total_positive CHECK (total_amount > 0);

ALTER TABLE users
    ADD CONSTRAINT uq_email UNIQUE (email);

-- Drop a constraint
ALTER TABLE orders
    DROP CONSTRAINT chk_total_positive;

-- Rename a table
ALTER TABLE users RENAME TO app_users;
```

:::caution[ALTER TABLE and Locks]
Most `ALTER TABLE` operations take an `ACCESS EXCLUSIVE` lock, blocking all reads and writes for the duration. For large tables in production, use `CREATE INDEX CONCURRENTLY`, add nullable columns (instant), then backfill separately, or use tools like `pg_repack` and `pgroll`.
:::

## DROP TABLE

```sql
-- Basic drop (fails if table doesn't exist)
DROP TABLE products;

-- Safe drop
DROP TABLE IF EXISTS products;

-- Drop and cascade (also drops dependent objects: foreign keys, views)
DROP TABLE IF EXISTS customers CASCADE;

-- Drop multiple tables
DROP TABLE IF EXISTS order_items, orders, products CASCADE;
```

## TRUNCATE

Removes all rows from a table faster than `DELETE` (no row-by-row logging):

```sql
-- Truncate a table
TRUNCATE products;

-- Truncate and reset SERIAL sequence
TRUNCATE products RESTART IDENTITY;

-- Truncate and cascade to tables with foreign keys pointing here
TRUNCATE customers CASCADE;
```

:::tip[TRUNCATE vs DELETE]
`TRUNCATE` is much faster for large tables but cannot be rolled back in some databases (it can in PostgreSQL). It doesn't fire row-level triggers and cannot use a `WHERE` clause. Use `DELETE` when you need to selectively remove rows or fire triggers.
:::

## Schema Organization

Schemas are namespaces that group related database objects:

```sql
-- Create a schema
CREATE SCHEMA api;
CREATE SCHEMA audit;

-- Create a table in a specific schema
CREATE TABLE api.endpoints (
    id      SERIAL PRIMARY KEY,
    path    TEXT NOT NULL,
    method  VARCHAR(10) NOT NULL
);

-- Query with schema-qualified name
SELECT * FROM api.endpoints;

-- Set the search path to find tables without schema prefix
SET search_path TO api, public;
SELECT * FROM endpoints;  -- finds api.endpoints

-- Drop a schema and all its objects
DROP SCHEMA audit CASCADE;
```

## information_schema

Query database metadata using the standard `information_schema`:

```sql
-- List all tables in the current database
SELECT table_name, table_type
FROM information_schema.tables
WHERE table_schema = 'public'
ORDER BY table_name;

-- List all columns for a table
SELECT column_name, data_type, is_nullable, column_default
FROM information_schema.columns
WHERE table_name = 'users'
  AND table_schema = 'public'
ORDER BY ordinal_position;

-- List all constraints
SELECT constraint_name, constraint_type
FROM information_schema.table_constraints
WHERE table_name = 'orders';

-- List foreign key relationships
SELECT
    tc.constraint_name,
    tc.table_name AS referencing_table,
    kcu.column_name AS referencing_column,
    ccu.table_name AS referenced_table,
    ccu.column_name AS referenced_column
FROM information_schema.table_constraints tc
JOIN information_schema.key_column_usage kcu
    ON tc.constraint_name = kcu.constraint_name
JOIN information_schema.constraint_column_usage ccu
    ON tc.constraint_name = ccu.constraint_name
WHERE tc.constraint_type = 'FOREIGN KEY'
  AND tc.table_schema = 'public';
```

## COMMENT ON

Add documentation directly to database objects:

```sql
COMMENT ON TABLE users IS 'Application user accounts';
COMMENT ON COLUMN users.password_hash IS 'bcrypt hash with cost factor 12';
COMMENT ON COLUMN orders.status IS 'pending | confirmed | shipped | delivered | cancelled';
COMMENT ON CONSTRAINT chk_price_positive ON products IS 'Prices must be positive; use 0 for free items';
```

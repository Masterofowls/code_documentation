## CREATE TABLE

The `CREATE TABLE` statement defines a new table with columns and constraints.

```sql
CREATE TABLE users (
  id          SERIAL         PRIMARY KEY,
  email       VARCHAR(255)   NOT NULL UNIQUE,
  username    VARCHAR(50)    NOT NULL,
  full_name   TEXT,
  age         INTEGER        CHECK (age >= 0 AND age <= 150),
  is_active   BOOLEAN        NOT NULL DEFAULT true,
  role        TEXT           NOT NULL DEFAULT 'member',
  created_at  TIMESTAMPTZ    NOT NULL DEFAULT NOW(),
  updated_at  TIMESTAMPTZ    NOT NULL DEFAULT NOW()
);
```

## Data Types

### Numeric Types

```sql
-- Integers
id        SMALLINT         -- -32,768 to 32,767
quantity  INTEGER          -- -2,147,483,648 to 2,147,483,647
big_id    BIGINT           -- ±9.2 × 10^18

-- Auto-increment (PostgreSQL)
id        SERIAL           -- Integer + sequence
id        BIGSERIAL        -- BigInt + sequence
id        INTEGER GENERATED ALWAYS AS IDENTITY   -- SQL standard

-- Exact decimals (for money, never use FLOAT for currency)
price     NUMERIC(10, 2)   -- 10 total digits, 2 decimal places
rate      DECIMAL(5, 4)    -- NUMERIC and DECIMAL are equivalent

-- Floating point (approximate — avoid for financial data)
ratio     REAL             -- 4-byte float (~6 decimal digits)
score     DOUBLE PRECISION -- 8-byte float (~15 decimal digits)
```

### Text Types

```sql
code      CHAR(3)          -- Fixed-length, padded with spaces
username  VARCHAR(50)      -- Variable-length, max 50 chars
notes     TEXT             -- Unlimited length (PostgreSQL/SQLite)
```

:::tip[Use TEXT in PostgreSQL]
In PostgreSQL, `TEXT` and `VARCHAR(n)` have identical performance. Use `TEXT` unless you specifically need to enforce a length limit as a business constraint (e.g., a country code that must be exactly 2 characters).
:::

### Boolean

```sql
is_active  BOOLEAN    NOT NULL DEFAULT true
-- Valid literals: TRUE/FALSE, 'true'/'false', 't'/'f', '1'/'0', 'yes'/'no' (PostgreSQL)
```

### Date and Time

```sql
birthday       DATE           -- '2024-01-15'
start_time     TIME           -- '14:30:00'
event_at       TIMESTAMP      -- '2024-01-15 14:30:00' (no timezone)
created_at     TIMESTAMPTZ    -- '2024-01-15 14:30:00+00' (with timezone — recommended)
duration       INTERVAL       -- '2 hours 30 minutes', '3 days'
```

:::caution[Always use TIMESTAMPTZ]
Store timestamps with timezone (`TIMESTAMPTZ`) — it stores UTC internally and converts on read. Plain `TIMESTAMP` stores exactly what you give it without timezone awareness, which leads to subtle bugs when servers or clients are in different timezones.
:::

### Identifiers

```sql
-- PostgreSQL UUID
id  UUID  DEFAULT gen_random_uuid()   -- Requires pgcrypto or pg 13+

-- Or use a text-based ID
id  TEXT  NOT NULL DEFAULT concat('usr_', gen_random_uuid()::text)
```

### JSON

```sql
-- MySQL / PostgreSQL
metadata  JSON     -- Stored as text, validated as JSON
settings  JSONB    -- PostgreSQL: binary JSON, indexable, faster queries
```

## Constraints

Constraints enforce data integrity rules at the database level — they apply regardless of which application, tool, or script modifies the data.

### Column-Level Constraints

```sql
CREATE TABLE products (
  id          INTEGER       PRIMARY KEY,           -- Unique + NOT NULL
  sku         VARCHAR(50)   NOT NULL UNIQUE,        -- Required, no duplicates
  name        TEXT          NOT NULL,
  price       NUMERIC(10,2) NOT NULL CHECK (price >= 0),
  category    TEXT          NOT NULL DEFAULT 'uncategorized',
  stock       INTEGER       NOT NULL DEFAULT 0 CHECK (stock >= 0),
  created_at  TIMESTAMPTZ   NOT NULL DEFAULT NOW()
);
```

### Table-Level Constraints

```sql
CREATE TABLE order_items (
  order_id    INTEGER   NOT NULL,
  product_id  INTEGER   NOT NULL,
  quantity    INTEGER   NOT NULL CHECK (quantity > 0),
  unit_price  NUMERIC(10,2) NOT NULL,

  -- Composite primary key
  PRIMARY KEY (order_id, product_id),

  -- Named constraints (recommended — easier to drop/modify)
  CONSTRAINT fk_order_items_order
    FOREIGN KEY (order_id) REFERENCES orders (id) ON DELETE CASCADE,

  CONSTRAINT fk_order_items_product
    FOREIGN KEY (product_id) REFERENCES products (id) ON DELETE RESTRICT,

  -- Multi-column check
  CONSTRAINT positive_total CHECK (quantity * unit_price >= 0)
);
```

### Foreign Key Actions

When the referenced row is deleted or updated, the database can automatically handle dependent rows:

| Action | `ON DELETE` behavior | `ON UPDATE` behavior |
|---|---|---|
| `CASCADE` | Delete all dependent rows | Update foreign key values |
| `SET NULL` | Set FK columns to NULL | Set FK columns to NULL |
| `SET DEFAULT` | Set FK columns to their default | Set FK columns to their default |
| `RESTRICT` | Reject the delete/update (error) | Reject the delete/update (error) |
| `NO ACTION` | Like RESTRICT, but checked at end of transaction | Same |

```sql
-- Common patterns
REFERENCES users (id) ON DELETE CASCADE      -- Delete orders when user is deleted
REFERENCES categories (id) ON DELETE SET NULL -- Keep product, clear category
REFERENCES plans (id) ON DELETE RESTRICT     -- Cannot delete a plan with active users
```

### Named Constraints

Always name constraints for clarity and to make them easier to drop:

```sql
CREATE TABLE sessions (
  id         UUID        PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id    INTEGER     NOT NULL,
  token      TEXT        NOT NULL,
  expires_at TIMESTAMPTZ NOT NULL,

  CONSTRAINT sessions_user_fk
    FOREIGN KEY (user_id) REFERENCES users (id) ON DELETE CASCADE,

  CONSTRAINT sessions_token_unique
    UNIQUE (token),

  CONSTRAINT sessions_expiry_future
    CHECK (expires_at > created_at)
);
```

## ALTER TABLE

Modify an existing table's structure without dropping it.

### Adding Columns

```sql
-- Add a column with a default (safe — existing rows get the default)
ALTER TABLE users
  ADD COLUMN bio TEXT;

ALTER TABLE users
  ADD COLUMN preferences JSONB NOT NULL DEFAULT '{}';

ALTER TABLE users
  ADD COLUMN email_verified BOOLEAN NOT NULL DEFAULT false;
```

### Dropping Columns

```sql
ALTER TABLE users DROP COLUMN bio;
ALTER TABLE users DROP COLUMN IF EXISTS legacy_field;

-- CASCADE drops dependent objects (views, foreign keys) referencing this column
ALTER TABLE users DROP COLUMN old_id CASCADE;
```

### Renaming

```sql
ALTER TABLE users RENAME COLUMN username TO handle;
ALTER TABLE users RENAME TO accounts;
```

### Changing Column Type

```sql
-- PostgreSQL: requires explicit USING if types are incompatible
ALTER TABLE products
  ALTER COLUMN price TYPE NUMERIC(12, 2);

ALTER TABLE logs
  ALTER COLUMN created_at TYPE TIMESTAMPTZ
  USING created_at AT TIME ZONE 'UTC';
```

### Managing Constraints

```sql
-- Add constraint
ALTER TABLE orders
  ADD CONSTRAINT orders_total_positive CHECK (total >= 0);

ALTER TABLE users
  ADD CONSTRAINT users_email_unique UNIQUE (email);

-- Drop constraint
ALTER TABLE orders DROP CONSTRAINT orders_total_positive;

-- Set/drop NOT NULL
ALTER TABLE users ALTER COLUMN full_name SET NOT NULL;
ALTER TABLE users ALTER COLUMN nickname DROP NOT NULL;

-- Set/drop default
ALTER TABLE users ALTER COLUMN is_active SET DEFAULT true;
ALTER TABLE users ALTER COLUMN timezone DROP DEFAULT;
```

:::caution[ALTER TABLE on large tables]
On large tables (millions of rows), adding a column with a default value rewrites the entire table in older PostgreSQL versions (&lt; 11). In PostgreSQL 11+, adding a column with a non-volatile default is instant. Dropping a column is always instant (it just marks it as hidden). Use `CONCURRENTLY` when creating indexes.
:::

## DROP TABLE

```sql
-- Drop a table
DROP TABLE sessions;

-- Only drop if it exists (no error if missing)
DROP TABLE IF EXISTS sessions;

-- Drop and all dependent objects (views, foreign keys referencing this table)
DROP TABLE users CASCADE;

-- Drop multiple tables
DROP TABLE IF EXISTS sessions, password_resets, email_verifications;
```

## TRUNCATE

Removes all rows from a table much faster than `DELETE` (no per-row logging):

```sql
-- Remove all rows
TRUNCATE users;

-- Reset auto-increment sequences (PostgreSQL)
TRUNCATE users RESTART IDENTITY;

-- Truncate and all tables with FK references to this table
TRUNCATE users CASCADE;

-- Truncate multiple tables atomically
TRUNCATE orders, order_items RESTART IDENTITY CASCADE;
```

## Schema Organization

Schemas provide a namespace for organizing tables. PostgreSQL supports multiple schemas per database.

```sql
-- Create a schema
CREATE SCHEMA app;
CREATE SCHEMA audit;

-- Create table in a schema
CREATE TABLE app.users ( ... );
CREATE TABLE audit.user_changes ( ... );

-- Set search path so you don't need to prefix every query
SET search_path TO app, public;

-- Grant schema access
GRANT USAGE ON SCHEMA app TO app_user;
GRANT CREATE ON SCHEMA app TO app_user;
```

## Querying the Information Schema

The `information_schema` provides metadata about your database structure:

```sql
-- List all tables in the current schema
SELECT table_name, table_type
FROM information_schema.tables
WHERE table_schema = 'public'
ORDER BY table_name;

-- List columns for a specific table
SELECT column_name, data_type, is_nullable, column_default
FROM information_schema.columns
WHERE table_name = 'users'
ORDER BY ordinal_position;

-- List all constraints
SELECT constraint_name, constraint_type, table_name
FROM information_schema.table_constraints
WHERE table_schema = 'public';
```

## COMMENT ON

Document your schema directly in the database:

```sql
COMMENT ON TABLE users IS 'Registered user accounts';
COMMENT ON COLUMN users.email IS 'Primary email address — used for login and notifications';
COMMENT ON COLUMN users.created_at IS 'UTC timestamp when the account was first created';
COMMENT ON CONSTRAINT users_email_unique ON users IS 'Each email may only be registered once';
```

Comments are visible in `psql` with `\d+ tablename` and in tools like pgAdmin and DBeaver.

## Complete Table Example

```sql
CREATE TABLE articles (
  id           BIGSERIAL      PRIMARY KEY,
  slug         VARCHAR(255)   NOT NULL,
  title        TEXT           NOT NULL,
  body         TEXT           NOT NULL DEFAULT '',
  excerpt      TEXT,
  author_id    INTEGER        NOT NULL,
  status       TEXT           NOT NULL DEFAULT 'draft'
                              CHECK (status IN ('draft', 'published', 'archived')),
  tags         TEXT[]         NOT NULL DEFAULT '{}',
  metadata     JSONB          NOT NULL DEFAULT '{}',
  view_count   INTEGER        NOT NULL DEFAULT 0 CHECK (view_count >= 0),
  published_at TIMESTAMPTZ,
  created_at   TIMESTAMPTZ    NOT NULL DEFAULT NOW(),
  updated_at   TIMESTAMPTZ    NOT NULL DEFAULT NOW(),

  CONSTRAINT articles_slug_unique UNIQUE (slug),
  CONSTRAINT articles_author_fk
    FOREIGN KEY (author_id)
    REFERENCES users (id)
    ON DELETE RESTRICT
    ON UPDATE CASCADE,
  CONSTRAINT articles_published_requires_title
    CHECK (status != 'published' OR (title != '' AND body != ''))
);

COMMENT ON TABLE articles IS 'Blog/CMS article content';
COMMENT ON COLUMN articles.slug IS 'URL-safe unique identifier for the article';
COMMENT ON COLUMN articles.status IS 'draft | published | archived';

CREATE INDEX articles_author_id_idx ON articles (author_id);
CREATE INDEX articles_status_published_idx ON articles (status, published_at DESC)
  WHERE status = 'published';
CREATE INDEX articles_tags_idx ON articles USING GIN (tags);
```

SQL (Structured Query Language) is the standard language for managing and querying relational databases. It was developed in the early 1970s at IBM and has remained the dominant interface for structured data storage for over five decades.

## What Is SQL?

SQL allows you to create database structures, insert and modify data, query records with complex filtering and aggregation, control access permissions, and manage transactions — all through a declarative language where you describe **what** you want rather than **how** to compute it.

```sql
-- A simple but complete demonstration of SQL's power
SELECT
    customers.name,
    COUNT(orders.id) AS order_count,
    SUM(orders.total_amount) AS lifetime_value
FROM customers
LEFT JOIN orders ON customers.id = orders.customer_id
WHERE customers.created_at >= '2024-01-01'
GROUP BY customers.id, customers.name
HAVING SUM(orders.total_amount) > 1000
ORDER BY lifetime_value DESC
LIMIT 10;
```

## The Relational Model

The relational model, introduced by E.F. Codd in 1970, organizes data into **tables** (also called relations) with well-defined structures.

### Core Components

**Table (Relation)**: A named collection of rows sharing the same structure. Every column has a defined name and data type.

**Row (Tuple)**: A single record in a table representing one entity or event.

**Column (Attribute)**: A named field with a specific data type that applies to every row in the table.

**Primary Key**: One or more columns that uniquely identify each row. No two rows can share the same primary key value.

**Foreign Key**: A column (or set of columns) that references the primary key of another table, establishing a relationship between the two tables.

**Schema**: The structure definition of a database — the tables, columns, constraints, indexes, views, and other objects.

```sql
-- A normalized schema with relationships
CREATE TABLE customers (
    id          BIGSERIAL PRIMARY KEY,
    email       VARCHAR(255) UNIQUE NOT NULL,
    name        VARCHAR(100) NOT NULL,
    created_at  TIMESTAMPTZ DEFAULT NOW()
);

CREATE TABLE orders (
    id          BIGSERIAL PRIMARY KEY,
    customer_id BIGINT NOT NULL REFERENCES customers(id),
    total_amount NUMERIC(12, 2) NOT NULL,
    status      VARCHAR(20) DEFAULT 'pending',
    created_at  TIMESTAMPTZ DEFAULT NOW()
);
```

## SQL Standards

SQL has been standardized by ANSI and ISO through a series of progressively expanding specifications:

| Standard | Year | Key Additions |
|---|---|---|
| SQL-86 | 1986 | First ANSI standard |
| SQL-89 | 1989 | Integrity constraints |
| SQL-92 | 1992 | Major expansion: JOINs, transactions, ALTER TABLE |
| SQL:1999 | 1999 | CTEs, recursion, triggers, OO extensions |
| SQL:2003 | 2003 | Window functions, MERGE, sequences, XML |
| SQL:2008 | 2008 | TRUNCATE, FETCH FIRST, improved window functions |
| SQL:2011 | 2011 | Temporal tables, enhanced period predicates |
| SQL:2016 | 2016 | JSON support, row pattern matching |
| SQL:2023 | 2023 | Property graph queries, multi-dimensional arrays |

:::tip[Practical Compliance]
No database implements 100% of any SQL standard. Most implement a useful subset plus proprietary extensions. Focus on the commonly supported core — SELECT, INSERT, UPDATE, DELETE, JOINs, CTEs, window functions — and check dialect docs for advanced features.
:::

## SQL Dialects

While SQL is standardized, each database system has its own dialect with varying syntax, functions, and capabilities.

| Feature | PostgreSQL | MySQL | SQLite | SQL Server | Oracle |
|---|---|---|---|---|---|
| **String concat** | `\|\|` or `concat()` | `concat()` | `\|\|` | `+` or `concat()` | `\|\|` |
| **Auto increment** | `SERIAL` / `IDENTITY` | `AUTO_INCREMENT` | `AUTOINCREMENT` | `IDENTITY` | `SEQUENCE` |
| **Limit rows** | `LIMIT n` | `LIMIT n` | `LIMIT n` | `TOP n` / `FETCH FIRST` | `ROWNUM` / `FETCH FIRST` |
| **String to date** | `TO_DATE()` | `STR_TO_DATE()` | `DATE()` | `CONVERT()` | `TO_DATE()` |
| **Current time** | `NOW()` | `NOW()` | `DATETIME('now')` | `GETDATE()` | `SYSDATE` |
| **Case-insensitive LIKE** | `ILIKE` | `LIKE` (default) | `LIKE` (ASCII) | `LIKE` (default) | `UPPER() LIKE` |
| **Boolean type** | `BOOLEAN` | `TINYINT(1)` | No native | `BIT` | No native |
| **JSONB** | ✅ Native | ✅ JSON | ✅ JSON functions | ✅ JSON | ✅ JSON |
| **Full-text search** | ✅ Native | ✅ Full-text | ✅ FTS5 | ✅ Full-text | ✅ Oracle Text |
| **Window functions** | ✅ Full | ✅ 8.0+ | ✅ 3.25+ | ✅ Full | ✅ Full |
| **Recursive CTEs** | ✅ | ✅ 8.0+ | ✅ | ✅ | ✅ |

## SQL Statement Categories

SQL statements are grouped into four categories based on what they do:

### DDL — Data Definition Language

DDL defines and modifies the database structure. Changes are usually auto-committed.

```sql
CREATE TABLE products (id SERIAL PRIMARY KEY, name TEXT NOT NULL);
ALTER TABLE products ADD COLUMN price NUMERIC(10, 2);
DROP TABLE products;
TRUNCATE products;
CREATE INDEX idx_products_name ON products (name);
CREATE VIEW active_products AS SELECT * FROM products WHERE active = true;
```

### DML — Data Manipulation Language

DML reads and modifies data within tables. Operations within a transaction can be rolled back.

```sql
INSERT INTO products (name, price) VALUES ('Widget', 9.99);
SELECT name, price FROM products WHERE price < 100;
UPDATE products SET price = price * 1.1 WHERE category = 'electronics';
DELETE FROM products WHERE discontinued = true;
```

### DCL — Data Control Language

DCL manages permissions and access control.

```sql
GRANT SELECT, INSERT ON products TO app_user;
REVOKE DELETE ON products FROM app_user;
GRANT ALL PRIVILEGES ON DATABASE mydb TO admin_user;
```

### TCL — Transaction Control Language

TCL manages transactions — grouping DML statements into atomic units.

```sql
BEGIN;
UPDATE accounts SET balance = balance - 100 WHERE id = 1;
UPDATE accounts SET balance = balance + 100 WHERE id = 2;
COMMIT;  -- or ROLLBACK to undo both changes
```

## ACID Properties

ACID guarantees ensure database operations are reliable even in the face of failures:

### Atomicity

A transaction is **all or nothing**. If any part of a transaction fails, the entire transaction is rolled back. There is no partial execution.

```sql
BEGIN;
-- Transfer $100 from Alice to Bob
UPDATE accounts SET balance = balance - 100 WHERE user_id = 1;
UPDATE accounts SET balance = balance + 100 WHERE user_id = 2;
-- If the second UPDATE fails, the first is automatically rolled back
COMMIT;
```

### Consistency

Every transaction brings the database from one **valid state** to another valid state. All constraints, rules, and triggers are enforced.

```sql
-- This transaction fails because it would violate the NOT NULL constraint
BEGIN;
INSERT INTO orders (customer_id, total_amount) VALUES (1, NULL); -- fails
ROLLBACK;
```

### Isolation

Concurrent transactions execute as if they were running **serially**. Each transaction cannot see the intermediate state of other transactions (to varying degrees depending on isolation level).

### Durability

Once a transaction is **committed**, it remains committed even in the event of a system crash. The database writes changes to persistent storage before acknowledging the commit.

:::caution[ACID vs BASE]
NoSQL databases often trade ACID guarantees for higher scalability using BASE (Basically Available, Soft state, Eventually consistent). For financial data, inventory, or any system requiring correctness guarantees, use an ACID-compliant relational database.
:::

SQL (Structured Query Language) is the standard language for managing and querying relational databases. It has been the dominant data query language since the 1970s and remains essential for application development, data analysis, and systems engineering.

## What is SQL?

SQL is a **declarative language** — you describe *what* data you want, not *how* to retrieve it. The database engine's query planner figures out the most efficient execution path. This abstraction is what makes SQL powerful: the same query runs correctly whether your table has 100 rows or 100 million rows (performance varies, but correctness doesn't).

```sql
-- Describe what you want, not how to get it
SELECT
  c.name,
  COUNT(o.id)    AS order_count,
  SUM(o.total)   AS lifetime_value
FROM customers c
LEFT JOIN orders o ON c.id = o.customer_id
WHERE c.created_at >= '2024-01-01'
GROUP BY c.id, c.name
HAVING SUM(o.total) > 1000
ORDER BY lifetime_value DESC
LIMIT 25;
```

## The Relational Model

The relational model, introduced by Edgar F. Codd in 1970, organizes data into **relations** (tables). Each concept has a precise definition:

| Concept | Relational term | SQL term | Description |
|---|---|---|---|
| Relation | Relation | Table | A named set of rows with a fixed schema |
| Tuple | Tuple | Row / Record | A single entry in a table |
| Attribute | Attribute | Column / Field | A named, typed property of a relation |
| Domain | Domain | Data type | The set of valid values for an attribute |
| Key | Key | Primary key | A set of columns uniquely identifying a row |
| Foreign key | Foreign key | Foreign key | A reference to a row in another table |

### Tables, Rows, and Columns

```sql
-- A table defines the schema (structure)
CREATE TABLE products (
  id          BIGINT       PRIMARY KEY,
  sku         VARCHAR(50)  UNIQUE NOT NULL,
  name        TEXT         NOT NULL,
  price       NUMERIC(10,2) NOT NULL CHECK (price >= 0),
  category_id BIGINT       REFERENCES categories(id),
  created_at  TIMESTAMP    DEFAULT NOW()
);

-- Rows are the data instances
INSERT INTO products (sku, name, price, category_id)
VALUES ('WIDGET-001', 'Blue Widget', 29.99, 1);
```

### Keys

**Primary key**: Uniquely identifies each row. Cannot be NULL. Every table should have one.

**Foreign key**: References the primary key of another table, enforcing referential integrity.

**Candidate key**: Any column or set of columns that could be a primary key.

**Composite key**: A primary key made of multiple columns.

```sql
-- Single-column primary key
CREATE TABLE users (
  id    BIGINT PRIMARY KEY,
  email TEXT   UNIQUE NOT NULL
);

-- Composite primary key
CREATE TABLE order_items (
  order_id   BIGINT REFERENCES orders(id),
  product_id BIGINT REFERENCES products(id),
  quantity   INTEGER NOT NULL CHECK (quantity > 0),
  PRIMARY KEY (order_id, product_id)
);
```

## SQL Standards

SQL is standardized by ISO/IEC. Each standard revision adds new features:

| Standard | Year | Key additions |
|---|---|---|
| SQL-86 (SQL-87) | 1987 | First standard — basic SELECT, INSERT, UPDATE, DELETE |
| SQL-89 | 1989 | Integrity constraints (CHECK, NOT NULL) |
| SQL-92 | 1992 | Major revision — JOINs, subqueries, CASE, CAST, string functions |
| SQL:1999 | 1999 | Recursive CTEs, triggers, OO extensions, OLAP functions |
| SQL:2003 | 2003 | Window functions, XML, MERGE, sequences, identity columns |
| SQL:2006 | 2006 | XML query |
| SQL:2008 | 2008 | TRUNCATE, FETCH FIRST, additional window functions |
| SQL:2011 | 2011 | Temporal tables (AS OF syntax) |
| SQL:2016 | 2016 | JSON support, row pattern recognition |
| SQL:2019 | 2019 | Multidimensional arrays |
| SQL:2023 | 2023 | Property graph queries, improved JSON, UUIDs |

:::tip[Standards vs Reality]
No database fully implements every SQL standard feature. PostgreSQL has the best standards compliance overall. MySQL/MariaDB historically lag on standard SQL. Always check your target database's documentation for specific feature availability.
:::

## SQL Dialects

While SQL is standardized, each database has its own dialect with extensions, syntax variations, and non-standard features.

| Feature | PostgreSQL | MySQL/MariaDB | SQLite | SQL Server | Oracle |
|---|---|---|---|---|---|
| **String concat** | `\|\|` or `concat()` | `concat()` | `\|\|` | `+` or `concat()` | `\|\|` or `concat()` |
| **Auto-increment** | `SERIAL` / `IDENTITY` | `AUTO_INCREMENT` | `AUTOINCREMENT` | `IDENTITY(1,1)` | `GENERATED AS IDENTITY` |
| **Boolean** | `BOOLEAN` (true/false) | `TINYINT(1)` | No BOOLEAN | `BIT` | No BOOLEAN |
| **Limit rows** | `LIMIT n` | `LIMIT n` | `LIMIT n` | `TOP n` / `FETCH FIRST` | `FETCH FIRST n ROWS` |
| **String case** | `ILIKE` | Case-insensitive by default | `LIKE` (case-insensitive for ASCII) | `LIKE` (configurable collation) | `UPPER()/LOWER()` |
| **JSON** | JSONB (full) | JSON (limited) | JSON functions (3.38+) | JSON (limited) | JSON (20c+) |
| **CTE recursion** | `WITH RECURSIVE` | `WITH RECURSIVE` (8.0+) | `WITH RECURSIVE` | `WITH … (recursive)` | `WITH … CYCLE` |
| **Window functions** | Full support | Full support (8.0+) | Limited (3.25+) | Full support | Full support |
| **Upsert** | `ON CONFLICT` | `ON DUPLICATE KEY UPDATE` | `ON CONFLICT` | `MERGE` | `MERGE` |
| **Arrays** | Native array type | No | No | No | No |

## SQL Statement Categories

SQL statements are organized into four categories:

### DDL — Data Definition Language

Defines and modifies the database schema (structure):

```sql
CREATE TABLE employees (...);
ALTER TABLE employees ADD COLUMN salary NUMERIC(10,2);
DROP TABLE employees;
TRUNCATE TABLE employees;
CREATE INDEX idx_email ON employees(email);
CREATE VIEW active_employees AS SELECT * FROM employees WHERE active = true;
```

### DML — Data Manipulation Language

Reads and modifies data:

```sql
SELECT * FROM employees WHERE department_id = 5;
INSERT INTO employees (name, email) VALUES ('Alice', 'alice@example.com');
UPDATE employees SET salary = salary * 1.10 WHERE performance_rating = 5;
DELETE FROM employees WHERE termination_date < NOW();
```

### DCL — Data Control Language

Controls access permissions:

```sql
GRANT SELECT, INSERT ON employees TO hr_user;
REVOKE DELETE ON employees FROM hr_user;
GRANT ALL PRIVILEGES ON DATABASE mydb TO admin_user;
```

### TCL — Transaction Control Language

Manages transactions:

```sql
BEGIN;                                -- Start transaction
SAVEPOINT before_salary_update;
UPDATE employees SET salary = 100000 WHERE id = 1;
ROLLBACK TO before_salary_update;    -- Undo to savepoint
COMMIT;                               -- Finalize transaction
```

## ACID Properties

ACID is the set of properties that guarantee database transactions are processed reliably.

### Atomicity

A transaction is **all or nothing**. If any step fails, the entire transaction is rolled back — no partial changes are committed.

```sql
-- Transfer $100 from account A to account B
BEGIN;
UPDATE accounts SET balance = balance - 100 WHERE id = 'A';
UPDATE accounts SET balance = balance + 100 WHERE id = 'B';
-- If the second UPDATE fails, the first is also rolled back
COMMIT;
```

### Consistency

A transaction takes the database from one **valid state** to another valid state. All data integrity constraints must be satisfied before the transaction can commit.

```sql
-- Constraint ensures consistency
ALTER TABLE accounts ADD CONSTRAINT balance_positive CHECK (balance >= 0);

-- This transaction will be rejected if A's balance would go negative
BEGIN;
UPDATE accounts SET balance = balance - 1000 WHERE id = 'A';  -- May violate CHECK
COMMIT;  -- Will fail if balance < 0
```

### Isolation

Concurrent transactions execute as if they were **serial** (one after the other). The degree of isolation is configurable via isolation levels.

```sql
-- Set isolation level for this transaction
BEGIN TRANSACTION ISOLATION LEVEL SERIALIZABLE;
-- Other transactions cannot see your uncommitted changes
-- You cannot see uncommitted changes from others
COMMIT;
```

### Durability

Once a transaction **commits**, its changes are permanent — even if the system crashes immediately after.

The database achieves durability via the **Write-Ahead Log (WAL)**: changes are written to a log before being applied to data files. On crash recovery, the log is replayed.

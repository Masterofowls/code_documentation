SQL (Structured Query Language) is the standard language for managing and querying relational databases. It is declarative â€” you describe *what* data you want, not *how* to retrieve it. The database engine's query planner determines the most efficient execution strategy.

## What is SQL?

SQL was developed at IBM in the early 1970s based on Edgar F. Codd's relational model paper. It became an ANSI standard in 1986 and an ISO standard in 1987. Today, virtually every relational database system implements SQL, making it one of the most widely used programming languages in the world.

SQL operates on **relations** (tables) â€” collections of rows (tuples) with a defined set of columns (attributes). Queries produce new relations, enabling powerful data transformations without modifying the underlying data.

## The Relational Model

The relational model organizes data into tables with strict structure:

- **Table (Relation)**: A named collection of rows. Each table has a defined schema.
- **Row (Tuple/Record)**: A single data entry in a table. All rows in a table share the same columns.
- **Column (Attribute)**: A named field with a defined data type. Each column holds one piece of information per row.
- **Primary Key**: One or more columns that uniquely identify a row in its table. Cannot be NULL; must be unique.
- **Foreign Key**: One or more columns that reference the primary key of another table, enforcing referential integrity.
- **Index**: A data structure built on one or more columns to speed up queries at the cost of storage and write performance.
- **Schema**: The overall structure of a database â€” its tables, columns, data types, constraints, and relationships.
- **Constraint**: A rule enforced by the database engine (NOT NULL, UNIQUE, CHECK, FOREIGN KEY).

## SQL Standards

SQL has been standardized multiple times, adding features with each revision:

| Standard | Year | Major additions |
|---|---|---|
| SQL-86 (SQL1) | 1986 | First ANSI standard â€” basic SELECT, INSERT, UPDATE, DELETE |
| SQL-89 | 1989 | Integrity constraints |
| SQL-92 (SQL2) | 1992 | JOINs, CASE, subqueries, NULL handling, transaction levels |
| SQL:1999 (SQL3) | 1999 | CTEs (WITH), recursive queries, OLAP, OO extensions |
| SQL:2003 | 2003 | Window functions, MERGE, XML support, IDENTITY columns |
| SQL:2008 | 2008 | TRUNCATE, FETCH FIRST, INSTEAD OF triggers |
| SQL:2011 | 2011 | Temporal tables, PERIOD FOR, system-versioned tables |
| SQL:2016 | 2016 | JSON support, polymorphic table functions, row pattern matching |
| SQL:2023 | 2023 | Graph queries (SQL/PGQ), UNIQUE null treatment improvements |

No database implements the full SQL standard. Most implement SQL-92 plus selective features from later standards.

## SQL Command Categories

SQL commands are divided into four categories based on their purpose:

### DDL â€” Data Definition Language

Commands that define and modify the **structure** of the database:

```sql
CREATE TABLE users (id SERIAL PRIMARY KEY, email TEXT NOT NULL);
ALTER TABLE users ADD COLUMN created_at TIMESTAMPTZ;
DROP TABLE sessions;
TRUNCATE audit_logs;
CREATE INDEX idx_users_email ON users (email);
```

### DML â€” Data Manipulation Language

Commands that **read and modify** the data:

```sql
SELECT id, email FROM users WHERE active = true;
INSERT INTO users (email) VALUES ('alice@example.com');
UPDATE users SET last_login = NOW() WHERE id = 42;
DELETE FROM sessions WHERE expired_at < NOW();
```

### DCL â€” Data Control Language

Commands that manage **permissions**:

```sql
GRANT SELECT ON users TO readonly_role;
REVOKE INSERT ON orders FROM app_user;
GRANT ALL PRIVILEGES ON DATABASE mydb TO admin;
```

### TCL â€” Transaction Control Language

Commands that manage **transactions**:

```sql
BEGIN;
SAVEPOINT before_update;
UPDATE accounts SET balance = balance - 100 WHERE id = 1;
UPDATE accounts SET balance = balance + 100 WHERE id = 2;
COMMIT;
-- or
ROLLBACK TO before_update;
```

## ACID Properties

Every transaction in a relational database must satisfy ACID properties to guarantee data integrity:

**Atomicity** â€” A transaction is all-or-nothing. If any part fails, the entire transaction is rolled back. There are no partial updates. *Example: transferring money deducts from one account and credits another â€” both succeed or neither does.*

**Consistency** â€” A transaction moves the database from one valid state to another. All constraints, rules, and cascades are enforced. A transaction that violates a constraint is rejected. *Example: a foreign key constraint ensures you cannot insert an order for a non-existent customer.*

**Isolation** â€” Concurrent transactions behave as if they ran sequentially. The intermediate state of a transaction is not visible to others (at appropriate isolation levels). *Example: two users booking the last seat on a flight cannot both succeed.*

**Durability** â€” Once a transaction is committed, it is permanent â€” even if the system crashes immediately after. Data is written to disk before acknowledgment. *Example: a completed payment is still recorded after a power outage.*

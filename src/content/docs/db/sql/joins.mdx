Joins are how SQL combines rows from two or more tables based on a related column. Understanding joins is essential for working with normalized relational data.

## Sample Schema

All examples on this page use this schema:

```sql
CREATE TABLE customers (
  id         BIGINT PRIMARY KEY,
  name       TEXT   NOT NULL,
  email      TEXT   UNIQUE NOT NULL,
  country    TEXT,
  created_at TIMESTAMPTZ DEFAULT NOW()
);

CREATE TABLE orders (
  id          BIGINT PRIMARY KEY,
  customer_id BIGINT NOT NULL REFERENCES customers(id),
  status      TEXT   NOT NULL DEFAULT 'pending',
  total       NUMERIC(12,2) NOT NULL DEFAULT 0,
  created_at  TIMESTAMPTZ DEFAULT NOW()
);

CREATE TABLE products (
  id          BIGINT PRIMARY KEY,
  name        TEXT           NOT NULL,
  price       NUMERIC(10,2)  NOT NULL,
  category_id BIGINT REFERENCES categories(id)
);

CREATE TABLE categories (
  id        BIGINT PRIMARY KEY,
  name      TEXT   NOT NULL,
  parent_id BIGINT REFERENCES categories(id)
);

CREATE TABLE order_items (
  order_id   BIGINT REFERENCES orders(id)   ON DELETE CASCADE,
  product_id BIGINT REFERENCES products(id) ON DELETE RESTRICT,
  quantity   INTEGER      NOT NULL CHECK (quantity > 0),
  unit_price NUMERIC(10,2) NOT NULL,
  PRIMARY KEY (order_id, product_id)
);
```

## INNER JOIN

Returns only rows where the join condition is satisfied in **both** tables. Rows without a match are excluded.

```sql
-- ON clause: explicit join condition
SELECT
  o.id          AS order_id,
  c.name        AS customer_name,
  o.total,
  o.status
FROM orders o
INNER JOIN customers c ON c.id = o.customer_id;

-- USING clause: shorthand when column names match
SELECT o.id, o.total, c.name
FROM orders o
INNER JOIN customers c USING (customer_id);  -- Only works if both tables have customer_id

-- Equivalent without USING:
SELECT o.id, o.total, c.name
FROM orders o
INNER JOIN customers c ON o.customer_id = c.id;
```

:::tip[INNER JOIN is the Default]
The `INNER` keyword is optional — `JOIN` alone means `INNER JOIN`. However, writing `INNER JOIN` explicitly improves readability in complex queries.
:::

## LEFT JOIN (LEFT OUTER JOIN)

Returns all rows from the **left** table plus matching rows from the right table. If there is no match, right-table columns are NULL.

```sql
-- All customers, with their order count (including customers with no orders)
SELECT
  c.id,
  c.name,
  COUNT(o.id) AS order_count,    -- Counts non-NULL values; 0 for no orders
  SUM(o.total) AS total_spent    -- NULL for customers with no orders
FROM customers c
LEFT JOIN orders o ON o.customer_id = c.id
GROUP BY c.id, c.name
ORDER BY total_spent DESC NULLS LAST;
```

### Find Unmatched Rows

LEFT JOIN + NULL filter is an efficient anti-join pattern:

```sql
-- Customers who have NEVER placed an order
SELECT c.id, c.name, c.email
FROM customers c
LEFT JOIN orders o ON o.customer_id = c.id
WHERE o.id IS NULL;   -- NULL means no matching order was found

-- Products that have never been ordered
SELECT p.id, p.name, p.price
FROM products p
LEFT JOIN order_items oi ON oi.product_id = p.id
WHERE oi.order_id IS NULL;
```

## RIGHT JOIN (RIGHT OUTER JOIN)

Returns all rows from the **right** table plus matching rows from the left table. Much less common than LEFT JOIN — you can always rewrite a RIGHT JOIN as a LEFT JOIN by swapping table order.

```sql
-- All orders, with customer info (RIGHT JOIN)
SELECT c.name, o.id, o.total
FROM customers c
RIGHT JOIN orders o ON c.id = o.customer_id;

-- Same query using LEFT JOIN (preferred style)
SELECT c.name, o.id, o.total
FROM orders o
LEFT JOIN customers c ON c.id = o.customer_id;
```

## FULL OUTER JOIN

Returns all rows from **both** tables. Where there is no match, the missing side's columns are NULL.

```sql
-- All customers and all orders, matched where possible
SELECT
  c.name        AS customer_name,
  o.id          AS order_id,
  o.total
FROM customers c
FULL OUTER JOIN orders o ON c.id = o.customer_id;

-- Find rows that only exist in one table or the other
SELECT
  c.id AS customer_id,
  o.id AS order_id
FROM customers c
FULL OUTER JOIN orders o ON c.id = o.customer_id
WHERE c.id IS NULL OR o.customer_id IS NULL;
```

## CROSS JOIN

Returns every combination of rows from both tables — the Cartesian product.

```sql
-- Generate all combinations of sizes and colors
SELECT s.name AS size, c.name AS color
FROM sizes s
CROSS JOIN colors c
ORDER BY s.id, c.id;

-- Same without explicit CROSS JOIN keyword
SELECT s.name, c.name FROM sizes s, colors c;  -- Implicit cross join (avoid this style)
```

:::caution[CROSS JOIN Row Count]
If table A has 100 rows and table B has 200 rows, the CROSS JOIN produces 20,000 rows. On large tables, this can easily overwhelm memory and disk.
:::

## SELF JOIN

Join a table to itself. Useful for hierarchical data and finding relationships within a single table.

```sql
-- Employee management hierarchy
SELECT
  e.name        AS employee,
  e.title,
  m.name        AS manager,
  m.title       AS manager_title
FROM employees e
LEFT JOIN employees m ON m.id = e.manager_id    -- e is the employee, m is their manager
ORDER BY m.name NULLS FIRST, e.name;

-- Category parent-child
SELECT
  child.name  AS category,
  parent.name AS parent_category
FROM categories child
LEFT JOIN categories parent ON child.parent_id = parent.id;

-- Find pairs of customers in the same country
SELECT
  a.name AS customer_a,
  b.name AS customer_b,
  a.country
FROM customers a
INNER JOIN customers b
  ON a.country = b.country
  AND a.id < b.id      -- Avoid (A,B) and (B,A) duplicates
ORDER BY a.country, a.name;
```

## Multiple Joins

Real-world queries often join four or more tables:

```sql
-- Complete order detail: customer → order → items → products → categories
SELECT
  c.name                                     AS customer,
  o.id                                       AS order_id,
  o.created_at                               AS order_date,
  p.name                                     AS product,
  cat.name                                   AS category,
  oi.quantity,
  oi.unit_price,
  oi.quantity * oi.unit_price                AS line_total
FROM orders o
INNER JOIN customers  c   ON c.id   = o.customer_id
INNER JOIN order_items oi ON oi.order_id = o.id
INNER JOIN products   p   ON p.id   = oi.product_id
LEFT  JOIN categories cat ON cat.id = p.category_id
WHERE o.status = 'completed'
  AND o.created_at >= '2024-01-01'
ORDER BY o.created_at DESC, p.name;
```

## Join Conditions

Joins aren't limited to equality:

```sql
-- Range join: match events to time periods
SELECT e.name, p.plan_name
FROM user_events e
INNER JOIN plans p
  ON e.occurred_at BETWEEN p.started_at AND COALESCE(p.ended_at, NOW());

-- Function in join condition
SELECT * FROM users u
INNER JOIN email_domains d ON LOWER(SPLIT_PART(u.email, '@', 2)) = LOWER(d.domain);

-- Non-equi join: find all products cheaper than this one
SELECT p1.name AS product, p2.name AS cheaper_alternative
FROM products p1
INNER JOIN products p2
  ON p2.category_id = p1.category_id
  AND p2.price < p1.price
ORDER BY p1.name, p2.price;
```

## LATERAL JOIN

A LATERAL join allows a subquery on the right side to reference columns from the left side — it's a correlated subquery that behaves like a table. Supported in PostgreSQL (and newer versions of MySQL/SQL Server).

```sql
-- Get the 3 most recent orders for each customer
SELECT
  c.name,
  recent.id    AS order_id,
  recent.total,
  recent.created_at
FROM customers c
CROSS JOIN LATERAL (
  SELECT id, total, created_at
  FROM orders o
  WHERE o.customer_id = c.id      -- References outer table
  ORDER BY created_at DESC
  LIMIT 3
) recent;

-- Equivalent to: for each customer, execute the subquery
-- Much more efficient than a correlated subquery in SELECT
```

```sql
-- Expand JSON array into rows (LATERAL is implicit with set-returning functions)
SELECT u.id, tag
FROM users u,
     LATERAL jsonb_array_elements_text(u.tags) AS tag;
```

## Anti-Join Pattern

Find rows in the left table with **no match** in the right table. Two equivalent approaches:

```sql
-- Method 1: LEFT JOIN + WHERE NULL (often more readable)
SELECT c.id, c.name
FROM customers c
LEFT JOIN orders o ON o.customer_id = c.id
WHERE o.id IS NULL;

-- Method 2: NOT EXISTS (often more efficient)
SELECT c.id, c.name
FROM customers c
WHERE NOT EXISTS (
  SELECT 1 FROM orders o WHERE o.customer_id = c.id
);

-- Method 3: NOT IN (avoid — NULLs in subquery cause unexpected behavior)
-- If any row in orders has NULL customer_id, this returns NO results:
SELECT id, name FROM customers
WHERE id NOT IN (SELECT customer_id FROM orders);  -- DANGEROUS if NULLs exist
```

:::tip[NOT EXISTS vs NOT IN]
Always prefer `NOT EXISTS` over `NOT IN` with subqueries. If the subquery returns any NULL values, `NOT IN` returns zero rows due to SQL's three-valued logic — a very common bug.
:::

## Semi-Join Pattern

Find rows in the left table where a match **exists** in the right table, without returning duplicate rows from the left:

```sql
-- Customers who have placed at least one order (semi-join via EXISTS)
SELECT c.id, c.name
FROM customers c
WHERE EXISTS (
  SELECT 1 FROM orders o WHERE o.customer_id = c.id
);

-- Less efficient equivalent (may produce duplicate customers):
SELECT DISTINCT c.id, c.name
FROM customers c
INNER JOIN orders o ON o.customer_id = c.id;
```

## Performance Considerations

### Index Join Columns

Always index columns used in ON conditions:

```sql
-- These indexes support the common join patterns
CREATE INDEX idx_orders_customer_id ON orders(customer_id);
CREATE INDEX idx_order_items_order_id ON order_items(order_id);
CREATE INDEX idx_order_items_product_id ON order_items(product_id);
```

### Join Order Matters

The query planner usually figures out the optimal join order, but understanding it helps:

- The **driving table** (usually the smaller one) is accessed first
- Subsequent tables are joined using the index on the join column
- LEFT JOINs constrain the order: the left table must be accessed before the right

```sql
-- Hint to planner (PostgreSQL): disable nested loop for this query
SET enable_nestloop = off;

-- Check what plan the optimizer chose
EXPLAIN (ANALYZE, BUFFERS) SELECT ...;
```

### Avoid Implicit Joins

Never use the comma syntax — it produces a cross join that silently becomes an inner join only via the WHERE clause, and is easy to accidentally turn into a Cartesian product:

```sql
-- BAD: implicit cross join (comma syntax)
SELECT c.name, o.id
FROM customers c, orders o
WHERE c.id = o.customer_id;

-- GOOD: explicit INNER JOIN
SELECT c.name, o.id
FROM customers c
INNER JOIN orders o ON c.id = o.customer_id;
```

Joins combine rows from two or more tables based on a related column. Understanding join types, conditions, and performance characteristics is essential to writing effective SQL.

## Sample Schema

All examples on this page use the following tables:

```sql
CREATE TABLE customers (
  id         SERIAL PRIMARY KEY,
  name       TEXT NOT NULL,
  email      TEXT UNIQUE,
  country    TEXT,
  created_at TIMESTAMPTZ DEFAULT NOW()
);

CREATE TABLE orders (
  id          SERIAL PRIMARY KEY,
  customer_id INTEGER REFERENCES customers(id),
  total       NUMERIC(10,2),
  status      TEXT DEFAULT 'pending',
  created_at  TIMESTAMPTZ DEFAULT NOW()
);

CREATE TABLE order_items (
  id         SERIAL PRIMARY KEY,
  order_id   INTEGER REFERENCES orders(id) ON DELETE CASCADE,
  product_id INTEGER REFERENCES products(id),
  quantity   INTEGER NOT NULL,
  unit_price NUMERIC(10,2) NOT NULL
);

CREATE TABLE products (
  id          SERIAL PRIMARY KEY,
  name        TEXT NOT NULL,
  category_id INTEGER REFERENCES categories(id),
  price       NUMERIC(10,2)
);

CREATE TABLE categories (
  id        SERIAL PRIMARY KEY,
  name      TEXT NOT NULL,
  parent_id INTEGER REFERENCES categories(id)
);
```

## INNER JOIN

Returns only rows where there is a match in **both** tables. Rows with no match in either table are excluded.

```sql
-- Customers with their orders (excludes customers with no orders)
SELECT
  c.id AS customer_id,
  c.name,
  o.id AS order_id,
  o.total,
  o.status
FROM customers c
INNER JOIN orders o ON o.customer_id = c.id;

-- Using USING when column names match exactly
SELECT c.name, o.total
FROM customers c
JOIN orders o USING (id);  -- Joins on customers.id = orders.id
-- Note: USING requires the join column to have the same name
```

:::tip[INNER JOIN vs JOIN]
`JOIN` without a keyword is always `INNER JOIN`. The `INNER` keyword is optional. Most developers omit it.
:::

### Multiple Table Joins

```sql
-- Customers → Orders → Order Items → Products → Categories
SELECT
  c.name AS customer,
  o.id   AS order_id,
  p.name AS product,
  cat.name AS category,
  oi.quantity,
  oi.unit_price,
  oi.quantity * oi.unit_price AS line_total
FROM customers c
JOIN orders o      ON o.customer_id = c.id
JOIN order_items oi ON oi.order_id = o.id
JOIN products p    ON p.id = oi.product_id
JOIN categories cat ON cat.id = p.category_id
WHERE o.status = 'completed'
ORDER BY o.id, p.name;
```

## LEFT JOIN (LEFT OUTER JOIN)

Returns **all rows from the left table**, plus matching rows from the right table. Where there is no match, right-table columns are NULL.

```sql
-- All customers, with their order count (including customers with 0 orders)
SELECT
  c.id,
  c.name,
  COUNT(o.id) AS order_count,
  COALESCE(SUM(o.total), 0) AS total_spent
FROM customers c
LEFT JOIN orders o ON o.customer_id = c.id
GROUP BY c.id, c.name
ORDER BY total_spent DESC;
```

### Finding Unmatched Rows

The most common LEFT JOIN pattern — find rows in the left table with no corresponding row in the right table:

```sql
-- Customers who have never placed an order
SELECT c.id, c.name, c.email
FROM customers c
LEFT JOIN orders o ON o.customer_id = c.id
WHERE o.id IS NULL;   -- NULL means no match was found

-- Products never ordered
SELECT p.id, p.name, p.price
FROM products p
LEFT JOIN order_items oi ON oi.product_id = p.id
WHERE oi.id IS NULL;
```

## RIGHT JOIN (RIGHT OUTER JOIN)

Returns **all rows from the right table**, plus matching rows from the left table. Rarely needed — you can always rewrite a RIGHT JOIN as a LEFT JOIN by swapping table order.

```sql
-- Equivalent queries:
SELECT c.name, o.id FROM orders o RIGHT JOIN customers c ON c.id = o.customer_id;
SELECT c.name, o.id FROM customers c LEFT JOIN orders o ON o.customer_id = c.id;
```

## FULL OUTER JOIN

Returns **all rows from both tables**. Where there is no match, the missing side produces NULLs.

```sql
-- Find all customers and all orders — even orders with no customer (orphaned data)
SELECT
  c.id   AS customer_id,
  c.name AS customer_name,
  o.id   AS order_id,
  o.total
FROM customers c
FULL OUTER JOIN orders o ON o.customer_id = c.id
WHERE c.id IS NULL OR o.id IS NULL;   -- Show only the unmatched rows
```

## CROSS JOIN

Produces the **Cartesian product** — every row in the left table combined with every row in the right table. A CROSS JOIN of 100 rows × 200 rows = 20,000 result rows.

```sql
-- Generate all possible size/color combinations for a product
SELECT s.name AS size, c.name AS color
FROM sizes s
CROSS JOIN colors c
ORDER BY s.sort_order, c.name;

-- Implicit syntax (same result — avoid for clarity)
SELECT s.name, c.name FROM sizes s, colors c;
```

## SELF JOIN

A table joined to itself. Common for hierarchical data or comparing rows within the same table.

```sql
-- Employee and their manager (both in the same employees table)
SELECT
  e.id,
  e.name     AS employee,
  m.name     AS manager,
  e.salary,
  m.salary   AS manager_salary
FROM employees e
LEFT JOIN employees m ON m.id = e.manager_id;

-- Find pairs of products in the same category within a certain price range
SELECT
  p1.name AS product_a,
  p2.name AS product_b,
  ABS(p1.price - p2.price) AS price_diff
FROM products p1
JOIN products p2 ON p2.category_id = p1.category_id
  AND p2.id > p1.id    -- Avoid duplicates and self-pairing
  AND ABS(p1.price - p2.price) < 10
ORDER BY price_diff;
```

## LATERAL JOIN

A `LATERAL` join allows the right-side subquery to reference columns from the left side. It's like a correlated subquery that can return multiple rows.

```sql
-- For each customer, get their 3 most recent orders
SELECT c.name, recent.*
FROM customers c
CROSS JOIN LATERAL (
  SELECT id, total, created_at
  FROM orders
  WHERE customer_id = c.id
  ORDER BY created_at DESC
  LIMIT 3
) AS recent;

-- Without LATERAL, this would require window functions or complex subqueries
```

```sql
-- Apply a function to each row's JSONB array elements
SELECT p.name, elem.value AS tag
FROM products p
CROSS JOIN LATERAL jsonb_array_elements_text(p.tags) AS elem(value);
```

## Anti-Join Pattern

Find rows in table A with no match in table B. Two common approaches:

```sql
-- Method 1: LEFT JOIN + IS NULL (readable, often fastest)
SELECT c.* FROM customers c
LEFT JOIN orders o ON o.customer_id = c.id
WHERE o.id IS NULL;

-- Method 2: NOT EXISTS (often optimized well by the query planner)
SELECT c.* FROM customers c
WHERE NOT EXISTS (
  SELECT 1 FROM orders o WHERE o.customer_id = c.id
);

-- Method 3: NOT IN (dangerous with NULLs — if any customer_id is NULL, returns empty set!)
SELECT c.* FROM customers c
WHERE c.id NOT IN (SELECT customer_id FROM orders WHERE customer_id IS NOT NULL);
```

:::caution[Avoid NOT IN with NULLs]
If the subquery in `NOT IN` returns any NULL values, the entire result is empty. This is a common bug. Use `LEFT JOIN ... IS NULL` or `NOT EXISTS` instead.
:::

## Semi-Join Pattern

Find rows in table A that have at least one match in table B (without duplicating A rows):

```sql
-- EXISTS semi-join (returns each customer at most once, even with multiple orders)
SELECT c.* FROM customers c
WHERE EXISTS (
  SELECT 1 FROM orders o
  WHERE o.customer_id = c.id AND o.status = 'completed'
);

-- Alternative with DISTINCT (less efficient)
SELECT DISTINCT c.*
FROM customers c
JOIN orders o ON o.customer_id = c.id AND o.status = 'completed';
```

## Join Conditions Beyond Equality

Joins can use any condition, not just equality:

```sql
-- Range join: find the pricing tier for each order amount
SELECT o.id, o.total, t.tier_name
FROM orders o
JOIN pricing_tiers t
  ON o.total >= t.min_amount
 AND o.total <  t.max_amount;

-- Find overlapping date ranges (room bookings that overlap)
SELECT a.room_id, a.guest AS guest_a, b.guest AS guest_b
FROM bookings a
JOIN bookings b
  ON a.room_id = b.room_id
 AND a.id < b.id
 AND a.check_in < b.check_out
 AND b.check_in < a.check_out;
```

## Implicit Joins (Avoid)

The old comma-separated syntax is equivalent to a CROSS JOIN, with filtering in WHERE:

```sql
-- Implicit join (avoid — hard to read, easy to miss join conditions)
SELECT c.name, o.total
FROM customers c, orders o
WHERE c.id = o.customer_id;

-- Explicit join (preferred)
SELECT c.name, o.total
FROM customers c
JOIN orders o ON o.customer_id = c.id;
```

## Join Performance

### Index the Join Columns

Always create an index on foreign key columns:

```sql
-- These indexes make JOINs fast
CREATE INDEX idx_orders_customer_id ON orders (customer_id);
CREATE INDEX idx_order_items_order_id ON order_items (order_id);
CREATE INDEX idx_order_items_product_id ON order_items (product_id);
```

### Think About the Driving Table

In a NESTED LOOP join, the outer (driving) table is scanned once; for each row, the inner table is looked up via index. The driving table should have fewer rows or more selective filters.

```sql
-- Efficient: filter first, then join (small driving set)
SELECT c.name, o.total
FROM orders o           -- Filtered orders is small
JOIN customers c ON c.id = o.customer_id
WHERE o.status = 'pending'
  AND o.created_at > NOW() - INTERVAL '7 days';
```

## Practical Full-Query Example

```sql
-- Monthly revenue report: revenue by category, with % of total
WITH monthly_sales AS (
  SELECT
    cat.id   AS category_id,
    cat.name AS category,
    DATE_TRUNC('month', o.created_at) AS month,
    SUM(oi.quantity * oi.unit_price)  AS revenue
  FROM orders o
  JOIN order_items oi ON oi.order_id = o.id
  JOIN products p     ON p.id = oi.product_id
  JOIN categories cat ON cat.id = p.category_id
  WHERE o.status = 'completed'
    AND o.created_at >= DATE_TRUNC('year', CURRENT_DATE)
  GROUP BY cat.id, cat.name, month
),
monthly_totals AS (
  SELECT month, SUM(revenue) AS total_revenue
  FROM monthly_sales
  GROUP BY month
)
SELECT
  ms.month,
  ms.category,
  ms.revenue,
  mt.total_revenue,
  ROUND(100.0 * ms.revenue / mt.total_revenue, 1) AS pct_of_total
FROM monthly_sales ms
JOIN monthly_totals mt USING (month)
ORDER BY ms.month DESC, ms.revenue DESC;
```

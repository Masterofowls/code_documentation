Joins combine rows from two or more tables based on a related column. They are the primary tool for working with normalized relational data.

## Setup: Example Schema

All examples use this e-commerce schema:

```sql
CREATE TABLE customers (
  id         SERIAL PRIMARY KEY,
  name       TEXT NOT NULL,
  email      TEXT UNIQUE NOT NULL,
  country    TEXT,
  created_at TIMESTAMPTZ DEFAULT NOW()
);

CREATE TABLE products (
  id          SERIAL PRIMARY KEY,
  name        TEXT NOT NULL,
  price       NUMERIC(10,2) NOT NULL,
  category_id INTEGER REFERENCES categories(id),
  stock       INTEGER NOT NULL DEFAULT 0
);

CREATE TABLE categories (
  id        SERIAL PRIMARY KEY,
  name      TEXT NOT NULL,
  parent_id INTEGER REFERENCES categories(id)
);

CREATE TABLE orders (
  id          SERIAL PRIMARY KEY,
  customer_id INTEGER REFERENCES customers(id) ON DELETE SET NULL,
  status      TEXT NOT NULL DEFAULT 'pending',
  total       NUMERIC(10,2),
  created_at  TIMESTAMPTZ DEFAULT NOW()
);

CREATE TABLE order_items (
  id         SERIAL PRIMARY KEY,
  order_id   INTEGER NOT NULL REFERENCES orders(id) ON DELETE CASCADE,
  product_id INTEGER NOT NULL REFERENCES products(id),
  quantity   INTEGER NOT NULL,
  unit_price NUMERIC(10,2) NOT NULL
);
```

## INNER JOIN

Returns only rows where the join condition matches in **both** tables. Non-matching rows from either side are excluded.

```sql
-- ON clause
SELECT
  o.id    AS order_id,
  c.name  AS customer_name,
  o.total,
  o.status
FROM orders o
INNER JOIN customers c ON c.id = o.customer_id
WHERE o.status = 'pending';

-- USING clause (when both tables have the same column name)
SELECT o.id, c.name, o.total
FROM orders o
JOIN customers c USING (customer_id);    -- JOIN = INNER JOIN

-- Multiple INNER JOINs
SELECT
  o.id          AS order_id,
  c.name        AS customer,
  p.name        AS product,
  oi.quantity,
  oi.unit_price,
  cat.name      AS category
FROM orders o
JOIN customers c    ON c.id = o.customer_id
JOIN order_items oi ON oi.order_id = o.id
JOIN products p     ON p.id = oi.product_id
JOIN categories cat ON cat.id = p.category_id
WHERE o.created_at >= NOW() - INTERVAL '30 days'
ORDER BY o.created_at DESC;
```

## LEFT JOIN (LEFT OUTER JOIN)

Returns **all rows from the left table**, plus matching rows from the right. Where no match exists, right table columns are NULL.

```sql
-- All customers with their order count (including 0-order customers)
SELECT
  c.id,
  c.name,
  c.email,
  COUNT(o.id)       AS order_count,
  SUM(o.total)      AS lifetime_value
FROM customers c
LEFT JOIN orders o ON o.customer_id = c.id
GROUP BY c.id, c.name, c.email
ORDER BY lifetime_value DESC NULLS LAST;

-- Find customers with NO orders (NULL right side = no match)
SELECT c.id, c.name, c.email
FROM customers c
LEFT JOIN orders o ON o.customer_id = c.id
WHERE o.id IS NULL;   -- filter to unmatched rows only (anti-join pattern)

-- All products with their order count (including never-ordered products)
SELECT
  p.name,
  p.price,
  COUNT(oi.id)         AS times_ordered,
  COALESCE(SUM(oi.quantity), 0) AS units_sold
FROM products p
LEFT JOIN order_items oi ON oi.product_id = p.id
GROUP BY p.id, p.name, p.price
ORDER BY units_sold DESC;
```

:::tip[NULL Handling in LEFT JOIN]
After a LEFT JOIN, use `COALESCE(value, 0)` to replace NULL numeric results with 0. Use `IS NULL` in WHERE (not `= NULL`) to find non-matching rows.
:::

## RIGHT JOIN (RIGHT OUTER JOIN)

Returns **all rows from the right table** plus matching rows from the left. The mirror image of LEFT JOIN. In practice, most developers rewrite RIGHT JOINs as LEFT JOINs by swapping table order (easier to read).

```sql
-- Equivalent queries:
SELECT c.name, o.id FROM orders o RIGHT JOIN customers c ON c.id = o.customer_id;
SELECT c.name, o.id FROM customers c LEFT JOIN orders o ON o.customer_id = c.id;
```

## FULL OUTER JOIN

Returns **all rows from both tables**. Where no match exists, the non-matching side's columns are NULL.

```sql
-- Find all mismatches between two tables (e.g., data sync audit)
SELECT
  a.id   AS source_id,
  b.id   AS dest_id,
  a.email AS source_email,
  b.email AS dest_email,
  CASE
    WHEN b.id IS NULL THEN 'only in source'
    WHEN a.id IS NULL THEN 'only in destination'
    ELSE 'matched'
  END AS sync_status
FROM source_users a
FULL OUTER JOIN dest_users b ON a.email = b.email
WHERE a.id IS NULL OR b.id IS NULL;   -- show only mismatches
```

:::caution[MySQL FULL OUTER JOIN]
MySQL does not support FULL OUTER JOIN natively. Emulate it with `LEFT JOIN UNION ALL RIGHT JOIN WHERE left.id IS NULL`:

```sql
SELECT * FROM a LEFT JOIN b ON a.id = b.a_id
UNION ALL
SELECT * FROM a RIGHT JOIN b ON a.id = b.a_id WHERE a.id IS NULL;
```

:::

## CROSS JOIN

Returns the **Cartesian product** — every combination of rows from both tables. Rarely used directly but useful for generating test data or combinations.

```sql
-- Generate all possible size/color product variants
SELECT
  s.name AS size,
  c.name AS color,
  s.name || '-' || c.name AS variant_code
FROM sizes s
CROSS JOIN colors c
ORDER BY s.sort_order, c.name;

-- Generate a date series (PostgreSQL alternative using generate_series)
SELECT
  d::date AS report_date,
  s.store_name
FROM generate_series('2024-01-01', '2024-12-31', INTERVAL '1 day') d
CROSS JOIN stores s
ORDER BY d, s.store_name;
```

## SELF JOIN

A table joining to itself. Common for hierarchical or comparison queries.

```sql
-- Employee hierarchy: employee with their manager's name
SELECT
  e.id,
  e.name       AS employee,
  m.name       AS manager,
  m.email      AS manager_email
FROM employees e
LEFT JOIN employees m ON m.id = e.manager_id
ORDER BY m.name NULLS LAST, e.name;

-- Find all pairs of products in the same category (within same price range)
SELECT
  a.name  AS product_a,
  b.name  AS product_b,
  a.price AS price_a,
  b.price AS price_b
FROM products a
JOIN products b
  ON a.category_id = b.category_id
  AND a.id < b.id           -- avoid duplicates (a,b) and (b,a)
  AND ABS(a.price - b.price) < 5
ORDER BY a.category_id, a.name;
```

## LATERAL JOIN

A LATERAL join allows the right-side subquery to reference columns from the left side — essentially a correlated subquery as a table.

```sql
-- Get the last 3 orders for each customer (PostgreSQL)
SELECT
  c.id,
  c.name,
  recent.order_id,
  recent.total,
  recent.created_at
FROM customers c
LEFT JOIN LATERAL (
  SELECT id AS order_id, total, created_at
  FROM orders
  WHERE customer_id = c.id     -- references c from outer query
  ORDER BY created_at DESC
  LIMIT 3
) recent ON true
ORDER BY c.name, recent.created_at DESC;

-- Apply a function to each row (useful with set-returning functions)
SELECT p.name, kw.word
FROM products p
LEFT JOIN LATERAL unnest(string_to_array(p.keywords, ',')) AS kw(word) ON true;
```

## Anti-Join Pattern

Find rows in table A that have **no match** in table B.

```sql
-- Method 1: LEFT JOIN + IS NULL (most common)
SELECT c.id, c.name
FROM customers c
LEFT JOIN orders o ON o.customer_id = c.id
WHERE o.id IS NULL;

-- Method 2: NOT EXISTS (more explicit intent, often faster)
SELECT c.id, c.name
FROM customers c
WHERE NOT EXISTS (
  SELECT 1 FROM orders o WHERE o.customer_id = c.id
);

-- Method 3: NOT IN (dangerous with NULLs — avoid if subquery may return NULL)
SELECT c.id, c.name
FROM customers c
WHERE c.id NOT IN (
  SELECT customer_id FROM orders WHERE customer_id IS NOT NULL  -- must add IS NOT NULL
);
```

## Semi-Join Pattern

Find rows in table A **where a match exists** in table B, without returning B's data.

```sql
-- Method 1: EXISTS (most common)
SELECT DISTINCT c.id, c.name
FROM customers c
WHERE EXISTS (
  SELECT 1 FROM orders o
  WHERE o.customer_id = c.id
  AND o.status = 'delivered'
);

-- Method 2: INNER JOIN + DISTINCT (may be slower due to dedup)
SELECT DISTINCT c.id, c.name
FROM customers c
JOIN orders o ON o.customer_id = c.id
WHERE o.status = 'delivered';
```

## Join Performance Guidance

### Indexing Join Columns

Always index foreign key columns and any column used in a join condition:

```sql
-- Essential indexes for the example schema
CREATE INDEX idx_orders_customer_id ON orders(customer_id);
CREATE INDEX idx_order_items_order_id ON order_items(order_id);
CREATE INDEX idx_order_items_product_id ON order_items(product_id);
CREATE INDEX idx_products_category_id ON products(category_id);

-- Covering index for a common query
CREATE INDEX idx_orders_customer_status ON orders(customer_id, status)
INCLUDE (total, created_at);
```

### Understanding Join Algorithms

PostgreSQL chooses among three join algorithms:

| Algorithm       | When used                               | Notes                           |
| --------------- | --------------------------------------- | ------------------------------- |
| **Nested Loop** | Small inner table, index on join column | Fast for small result sets      |
| **Hash Join**   | Large tables, no useful index           | Builds hash table in memory     |
| **Merge Join**  | Both inputs sorted on join key          | Efficient for large sorted sets |

```sql
-- Check which algorithm PostgreSQL chose
EXPLAIN (ANALYZE, BUFFERS)
SELECT o.id, c.name
FROM orders o
JOIN customers c ON c.id = o.customer_id
WHERE o.status = 'pending';
```

## Complex Real-World Query

Order summary report joining four tables:

```sql
SELECT
  DATE_TRUNC('week', o.created_at)   AS week,
  cat.name                            AS category,
  COUNT(DISTINCT o.id)                AS order_count,
  COUNT(DISTINCT o.customer_id)       AS unique_customers,
  SUM(oi.quantity)                    AS units_sold,
  SUM(oi.quantity * oi.unit_price)    AS revenue,
  AVG(o.total)                        AS avg_order_value,
  RANK() OVER (
    PARTITION BY DATE_TRUNC('week', o.created_at)
    ORDER BY SUM(oi.quantity * oi.unit_price) DESC
  ) AS revenue_rank_this_week
FROM orders o
JOIN customers c    ON c.id = o.customer_id
JOIN order_items oi ON oi.order_id = o.id
JOIN products p     ON p.id = oi.product_id
JOIN categories cat ON cat.id = p.category_id
WHERE o.status IN ('delivered', 'shipped')
  AND o.created_at >= NOW() - INTERVAL '12 weeks'
  AND c.country = 'US'
GROUP BY DATE_TRUNC('week', o.created_at), cat.id, cat.name
HAVING SUM(oi.quantity * oi.unit_price) > 1000
ORDER BY week DESC, revenue DESC;
```

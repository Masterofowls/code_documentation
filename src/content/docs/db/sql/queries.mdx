This page covers the full range of DML (Data Manipulation Language) operations — from simple SELECT queries to recursive CTEs, conditional expressions, and multi-row mutations.

## SELECT Basics

```sql
-- Select all columns
SELECT * FROM users;

-- Select specific columns
SELECT id, email, created_at FROM users;

-- Column aliases
SELECT
  id,
  email AS user_email,
  CONCAT(first_name, ' ', last_name) AS full_name,
  DATE_PART('year', AGE(birthday)) AS age_years
FROM users;

-- Remove duplicate rows
SELECT DISTINCT country FROM users;
SELECT DISTINCT ON (country) country, city FROM users ORDER BY country, city;

-- Limit and offset (pagination)
SELECT id, title FROM articles
ORDER BY created_at DESC
LIMIT 20 OFFSET 40;   -- Page 3 (0-indexed), 20 per page

-- SQL standard pagination
SELECT id, title FROM articles
ORDER BY created_at DESC
FETCH FIRST 20 ROWS ONLY;

FETCH FIRST 20 ROWS WITH TIES;  -- Include tied rows
```

## WHERE Clause

```sql
-- Equality and comparison
SELECT * FROM orders WHERE status = 'pending';
SELECT * FROM products WHERE price > 100;
SELECT * FROM events WHERE created_at >= '2024-01-01';

-- Multiple conditions
SELECT * FROM users
WHERE is_active = true
  AND role = 'admin'
  AND created_at >= '2024-01-01';

-- OR and NOT
SELECT * FROM orders
WHERE status = 'pending' OR status = 'processing';

SELECT * FROM users WHERE NOT is_deleted;

-- IN / NOT IN
SELECT * FROM products
WHERE category_id IN (1, 2, 5, 8);

SELECT * FROM users
WHERE country NOT IN ('US', 'CA', 'MX');

-- BETWEEN (inclusive on both ends)
SELECT * FROM orders
WHERE created_at BETWEEN '2024-01-01' AND '2024-12-31';

SELECT * FROM products
WHERE price BETWEEN 10.00 AND 99.99;

-- Pattern matching
SELECT * FROM users WHERE email LIKE '%@gmail.com';    -- Ends with
SELECT * FROM products WHERE sku LIKE 'PROD-%';        -- Starts with
SELECT * FROM articles WHERE title LIKE '%SQL%';       -- Contains (slow — no index)

-- Case-insensitive LIKE (PostgreSQL)
SELECT * FROM users WHERE email ILIKE '%@GMAIL.COM';

-- Single character wildcard
SELECT * FROM codes WHERE code LIKE 'A_C';  -- A, any char, C

-- NULL checks
SELECT * FROM users WHERE deleted_at IS NULL;       -- Active users
SELECT * FROM orders WHERE shipped_at IS NOT NULL;  -- Shipped orders
```

:::caution[NULL comparisons]
Never compare NULL with `=` or `!=` — they always evaluate to NULL (unknown), not true or false. Always use `IS NULL` or `IS NOT NULL`. The expression `NULL = NULL` is `NULL`, not `TRUE`.
:::

## ORDER BY

```sql
-- Single column
SELECT * FROM users ORDER BY created_at DESC;

-- Multiple columns
SELECT * FROM products
ORDER BY category ASC, price DESC;

-- NULLS FIRST / LAST
SELECT * FROM articles
ORDER BY published_at DESC NULLS LAST;   -- Unpublished articles go to end

-- Order by expression
SELECT * FROM users
ORDER BY LENGTH(email) ASC, email ASC;

-- Order by column alias
SELECT *, EXTRACT(YEAR FROM created_at) AS year
FROM orders
ORDER BY year DESC;
```

## Subqueries

### Non-Correlated Subqueries

A non-correlated subquery executes once and its result is used by the outer query:

```sql
-- Scalar subquery (returns one value)
SELECT
  id,
  title,
  price,
  price - (SELECT AVG(price) FROM products) AS price_vs_avg
FROM products;

-- IN with subquery
SELECT * FROM users
WHERE id IN (
  SELECT DISTINCT user_id FROM orders
  WHERE created_at >= NOW() - INTERVAL '30 days'
);

-- NOT IN with subquery (be careful with NULLs!)
SELECT * FROM products
WHERE id NOT IN (
  SELECT product_id FROM order_items WHERE product_id IS NOT NULL
);
```

### Correlated Subqueries

A correlated subquery references the outer query and executes once per row:

```sql
-- Find users who have placed at least one order
SELECT u.id, u.email
FROM users u
WHERE EXISTS (
  SELECT 1 FROM orders o WHERE o.user_id = u.id
);

-- Find users with no orders
SELECT u.id, u.email
FROM users u
WHERE NOT EXISTS (
  SELECT 1 FROM orders o WHERE o.user_id = u.id
);

-- Correlated subquery in SELECT (executes for every row — can be slow)
SELECT
  u.id,
  u.email,
  (SELECT COUNT(*) FROM orders o WHERE o.user_id = u.id) AS order_count
FROM users u;
```

:::tip[EXISTS is more efficient than IN for correlated checks]
`EXISTS` stops scanning as soon as it finds one matching row. `IN` collects all matching values first. For large tables, `EXISTS` can be significantly faster for existence checks.
:::

## CTEs (Common Table Expressions)

CTEs use the `WITH` clause to name intermediate result sets. They improve readability and allow referencing the same subquery multiple times.

```sql
-- Basic CTE
WITH active_users AS (
  SELECT id, email, created_at
  FROM users
  WHERE is_active = true AND deleted_at IS NULL
),
recent_orders AS (
  SELECT user_id, COUNT(*) AS order_count, SUM(total) AS total_spent
  FROM orders
  WHERE created_at >= NOW() - INTERVAL '90 days'
  GROUP BY user_id
)
SELECT
  u.email,
  COALESCE(o.order_count, 0) AS orders_last_90_days,
  COALESCE(o.total_spent, 0) AS spent_last_90_days
FROM active_users u
LEFT JOIN recent_orders o ON o.user_id = u.id
ORDER BY spent_last_90_days DESC;
```

### Multiple CTEs

```sql
WITH
  category_stats AS (
    SELECT category_id, AVG(price) AS avg_price
    FROM products
    GROUP BY category_id
  ),
  top_products AS (
    SELECT p.*, c.avg_price AS category_avg
    FROM products p
    JOIN category_stats c ON c.category_id = p.category_id
    WHERE p.price > c.avg_price * 1.5
  )
SELECT t.name, t.price, t.category_avg
FROM top_products t
ORDER BY t.price DESC
LIMIT 10;
```

### Recursive CTEs

Recursive CTEs query hierarchical data (trees, graphs) without knowing the depth:

```sql
-- Employee hierarchy — find all reports under a manager
WITH RECURSIVE org_chart AS (
  -- Base case: start with the root manager
  SELECT id, name, manager_id, 0 AS depth, name AS path
  FROM employees
  WHERE manager_id IS NULL   -- CEO or root node

  UNION ALL

  -- Recursive case: join each employee to their manager's row
  SELECT
    e.id,
    e.name,
    e.manager_id,
    oc.depth + 1,
    oc.path || ' > ' || e.name
  FROM employees e
  JOIN org_chart oc ON oc.id = e.manager_id
  WHERE oc.depth < 10   -- Guard against infinite loops
)
SELECT depth, name, path
FROM org_chart
ORDER BY path;
```

```sql
-- Category breadcrumb path
WITH RECURSIVE category_path AS (
  SELECT id, name, parent_id, name::TEXT AS breadcrumb
  FROM categories
  WHERE id = 42   -- Start category

  UNION ALL

  SELECT c.id, c.name, c.parent_id,
         c.name || ' > ' || cp.breadcrumb
  FROM categories c
  JOIN category_path cp ON cp.parent_id = c.id
)
SELECT breadcrumb FROM category_path
WHERE parent_id IS NULL;
```

## CASE Expression

### Simple CASE

```sql
SELECT
  id,
  status,
  CASE status
    WHEN 'pending'    THEN 'Awaiting payment'
    WHEN 'paid'       THEN 'Payment received'
    WHEN 'shipped'    THEN 'On the way'
    WHEN 'delivered'  THEN 'Delivered'
    ELSE 'Unknown status'
  END AS status_label
FROM orders;
```

### Searched CASE

```sql
SELECT
  id,
  price,
  CASE
    WHEN price < 10   THEN 'Budget'
    WHEN price < 50   THEN 'Standard'
    WHEN price < 200  THEN 'Premium'
    ELSE 'Luxury'
  END AS price_tier
FROM products;
```

### CASE in ORDER BY

```sql
-- Sort by custom priority order
SELECT * FROM tickets
ORDER BY
  CASE priority
    WHEN 'critical' THEN 1
    WHEN 'high'     THEN 2
    WHEN 'medium'   THEN 3
    WHEN 'low'      THEN 4
    ELSE 5
  END ASC,
  created_at ASC;
```

## INSERT

```sql
-- Single row
INSERT INTO users (email, username, role)
VALUES ('alice@example.com', 'alice', 'member');

-- Multi-row
INSERT INTO tags (name, slug)
VALUES
  ('JavaScript', 'javascript'),
  ('TypeScript', 'typescript'),
  ('SQL', 'sql');

-- Insert from SELECT
INSERT INTO archive_orders (id, user_id, total, created_at)
SELECT id, user_id, total, created_at
FROM orders
WHERE created_at < NOW() - INTERVAL '2 years';

-- Returning inserted values
INSERT INTO users (email, username)
VALUES ('bob@example.com', 'bob')
RETURNING id, created_at;
```

## ON CONFLICT (UPSERT)

Handle conflicts when inserting a row that violates a UNIQUE or PRIMARY KEY constraint:

```sql
-- Do nothing on conflict
INSERT INTO user_preferences (user_id, key, value)
VALUES (42, 'theme', 'dark')
ON CONFLICT (user_id, key) DO NOTHING;

-- Update on conflict
INSERT INTO user_preferences (user_id, key, value)
VALUES (42, 'theme', 'dark')
ON CONFLICT (user_id, key)
DO UPDATE SET
  value = EXCLUDED.value,
  updated_at = NOW();

-- EXCLUDED refers to the row that was rejected by the conflict
INSERT INTO page_views (page_id, date, view_count)
VALUES (1, CURRENT_DATE, 1)
ON CONFLICT (page_id, date)
DO UPDATE SET
  view_count = page_views.view_count + EXCLUDED.view_count;
```

## UPDATE

```sql
-- Basic update
UPDATE users SET is_active = false WHERE last_login < NOW() - INTERVAL '1 year';

-- Update multiple columns
UPDATE products
SET
  price = price * 1.05,
  updated_at = NOW()
WHERE category_id = 3 AND is_active = true;

-- Update with JOIN (FROM clause — PostgreSQL/SQL Server)
UPDATE orders o
SET status = 'shipped', shipped_at = NOW()
FROM shipments s
WHERE s.order_id = o.id
  AND s.tracking_number IS NOT NULL
  AND o.status = 'packed';

-- Returning updated rows
UPDATE users
SET last_login = NOW()
WHERE id = 42
RETURNING id, email, last_login;
```

## DELETE

```sql
-- Basic delete
DELETE FROM sessions WHERE expired_at < NOW();

-- Delete with conditions
DELETE FROM users WHERE is_active = false AND created_at < NOW() - INTERVAL '2 years';

-- Delete with JOIN (using EXISTS or subquery)
DELETE FROM orders
WHERE user_id IN (
  SELECT id FROM users WHERE is_deleted = true
);

-- Returning deleted rows
DELETE FROM temp_tokens WHERE token = 'abc123'
RETURNING user_id, created_at;

-- Delete all rows (use TRUNCATE for large tables)
DELETE FROM session_data;
```

## Complete Query Example

```sql
-- Find high-value customers who haven't ordered recently, with their order history
WITH customer_stats AS (
  SELECT
    u.id,
    u.email,
    u.created_at AS member_since,
    COUNT(o.id)   AS lifetime_orders,
    SUM(o.total)  AS lifetime_value,
    MAX(o.created_at) AS last_order_date
  FROM users u
  LEFT JOIN orders o ON o.user_id = u.id AND o.status = 'completed'
  WHERE u.is_active = true
  GROUP BY u.id, u.email, u.created_at
)
SELECT
  email,
  lifetime_orders,
  lifetime_value,
  last_order_date,
  CASE
    WHEN last_order_date IS NULL               THEN 'Never ordered'
    WHEN last_order_date < NOW() - INTERVAL '180 days' THEN 'Lapsed'
    WHEN last_order_date < NOW() - INTERVAL '90 days'  THEN 'At risk'
    ELSE 'Active'
  END AS customer_status
FROM customer_stats
WHERE lifetime_value > 500
  AND (last_order_date IS NULL OR last_order_date < NOW() - INTERVAL '90 days')
ORDER BY lifetime_value DESC
LIMIT 100;
```

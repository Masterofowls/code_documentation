# Getting Started with Astro.js

This guide walks you through creating a new Astro project from scratch, understanding its structure,
writing your first `.astro` component, and preparing a production build.

## Prerequisites

- Node.js 18.17.1 or later (`node --version`)
- A modern package manager: npm 9+, pnpm 8+, or yarn 1.22+

## Creating a New Project

<Steps>

1. **Run the creation wizard**

   ```bash
   npm create astro@latest
   ```

   The interactive wizard asks:
   - Project directory name
   - Template to start from
   - Whether to install dependencies
   - Whether to initialize a git repository
   - TypeScript strictness level

2. **Enter the project directory**

   ```bash
   cd my-astro-project
   ```

3. **Start the development server**

   ```bash
   npm run dev
   ```

   Your site is now live at `http://localhost:4321`.

</Steps>

## Project Templates

When running `npm create astro@latest`, you can choose from several starting templates:

| Template | Description |
|---|---|
| `minimal` | Bare bones: `index.astro` only |
| `blog` | Blog with MDX, content collections, RSS |
| `portfolio` | Portfolio with project showcase |
| `docs` | Documentation site (Starlight) |
| `starlight` | Full Starlight docs framework |

You can also use any GitHub template:

```bash
npm create astro@latest -- --template withastro/astro/examples/blog
```

## Folder Structure

A standard Astro project looks like this:

```
my-astro-project/
├── public/              # Static assets (copied as-is, no processing)
│   ├── favicon.svg
│   └── robots.txt
├── src/
│   ├── components/      # Reusable .astro or framework components
│   │   └── Card.astro
│   ├── layouts/         # Page layout templates
│   │   └── BaseLayout.astro
│   ├── pages/           # File-based routing (each file = a route)
│   │   ├── index.astro
│   │   ├── about.astro
│   │   └── blog/
│   │       └── [slug].astro
│   ├── content/         # Content Collections (type-safe Markdown/MDX)
│   │   └── blog/
│   │       └── first-post.md
│   └── styles/          # Global CSS files
│       └── global.css
├── astro.config.mjs     # Astro configuration
├── tsconfig.json        # TypeScript configuration
└── package.json
```

:::tip[src/pages is Special]
Only files inside `src/pages/` are automatically turned into routes. Files in `src/components/`,
`src/layouts/`, and `src/content/` are not routes — they must be imported explicitly.
:::

## astro.config.mjs

The central configuration file for your Astro project:

```typescript
// astro.config.mjs
import { defineConfig } from 'astro/config';
import react from '@astrojs/react';
import tailwind from '@astrojs/tailwind';
import mdx from '@astrojs/mdx';
import sitemap from '@astrojs/sitemap';

export default defineConfig({
  // Your production site URL (required for sitemap, canonical URLs)
  site: 'https://example.com',

  // Optional base path if deploying to a subdirectory
  base: '/',

  // Output mode: 'static' (default) | 'server' | 'hybrid'
  output: 'static',

  // Integrations
  integrations: [
    react(),
    tailwind({ applyBaseStyles: false }),
    mdx(),
    sitemap(),
  ],

  // Vite configuration passthrough
  vite: {
    define: {
      __APP_VERSION__: JSON.stringify(process.env.npm_package_version),
    },
  },

  // Markdown configuration
  markdown: {
    shikiConfig: {
      theme: 'github-dark',
      wrap: true,
    },
  },
});
```

## TypeScript Configuration

Astro comes with TypeScript support out of the box. The recommended strict config:

```json
// tsconfig.json
{
  "extends": "astro/tsconfigs/strict",
  "compilerOptions": {
    "baseUrl": ".",
    "paths": {
      "@components/*": ["src/components/*"],
      "@layouts/*": ["src/layouts/*"],
      "@utils/*": ["src/utils/*"]
    }
  }
}
```

Astro provides three built-in TypeScript presets:
- `astro/tsconfigs/base` — Minimal settings
- `astro/tsconfigs/strict` — Recommended for most projects
- `astro/tsconfigs/strictest` — Maximum strictness

## Your First .astro Component

An `.astro` file has three sections: the **frontmatter script** (server-side), the **HTML template**, and
optional **scoped styles**:

```astro
---
// src/pages/index.astro
// Frontmatter script: runs on the server only
// Import components, fetch data, define variables

import Layout from '../layouts/BaseLayout.astro';
import Card from '../components/Card.astro';

const siteTitle = 'My Astro Site';
const posts = await fetch('https://jsonplaceholder.typicode.com/posts?_limit=3')
  .then(r => r.json());
---

<!-- HTML Template: rendered to static HTML -->
<Layout title={siteTitle}>
  <main>
    <h1>Welcome to {siteTitle}</h1>

    <p>I have <strong>{posts.length}</strong> posts to share.</p>

    <ul>
      {posts.map((post) => (
        <Card title={post.title} body={post.body} />
      ))}
    </ul>
  </main>
</Layout>

<!-- Scoped styles: only affect this component -->
<style>
  main {
    max-width: 800px;
    margin: 0 auto;
    padding: 2rem;
  }

  h1 {
    font-size: 2rem;
    color: var(--color-primary);
  }
</style>
```

## Astro vs HTML

Astro's template syntax is a superset of HTML with a few key differences:

| Feature | HTML | Astro |
|---|---|---|
| `class` attribute | `class="..."` | `class="..."` (same) |
| Dynamic expressions | Not supported | `{expression}` |
| Conditional rendering | Not supported | `{condition && <El />}` |
| List rendering | Not supported | `{arr.map(item => <El />)}` |
| Component imports | Not supported | `import Comp from './Comp.astro'` |
| Server-side data | Not supported | In frontmatter script |
| Scoped styles | Not supported | `<style>` in `.astro` |
| Raw HTML | Not supported | `<Fragment set:html={html} />` |

## Template Expressions

Astro uses `{}` for template expressions — similar to JSX but without the JSX quirks:

```astro
---
const isLoggedIn = true;
const items = ['Apple', 'Banana', 'Cherry'];
const htmlString = '<strong>Bold text</strong>';
---

<!-- Conditional rendering -->
{isLoggedIn ? <p>Welcome back!</p> : <a href="/login">Sign in</a>}

<!-- Short-circuit rendering -->
{isLoggedIn && <nav>Dashboard</nav>}

<!-- List rendering -->
<ul>
  {items.map((item, index) => (
    <li data-index={index}>{item}</li>
  ))}
</ul>

<!-- Rendering raw HTML (sanitize first if from user input!) -->
<div set:html={htmlString} />

<!-- Attribute expressions -->
<a href={isLoggedIn ? '/dashboard' : '/login'} class="btn">
  {isLoggedIn ? 'Dashboard' : 'Sign In'}
</a>
```

## Building a Page Layout

Layouts are regular `.astro` components that accept a `<slot />` for child content:

```astro
---
// src/layouts/BaseLayout.astro
interface Props {
  title: string;
  description?: string;
}

const { title, description = 'My Astro Site' } = Astro.props;
---

<!doctype html>
<html lang="en">
  <head>
    <meta charset="UTF-8" />
    <meta name="viewport" content="width=device-width, initial-scale=1" />
    <meta name="description" content={description} />
    <link rel="icon" type="image/svg+xml" href="/favicon.svg" />
    <title>{title}</title>
  </head>
  <body>
    <header>
      <nav>
        <a href="/">Home</a>
        <a href="/about">About</a>
        <a href="/blog">Blog</a>
      </nav>
    </header>

    <main>
      <slot />  <!-- Child content injected here -->
    </main>

    <footer>
      <p>&copy; {new Date().getFullYear()} My Site</p>
    </footer>
  </body>
</html>

<style is:global>
  * { box-sizing: border-box; }
  body { margin: 0; font-family: system-ui, sans-serif; }
</style>
```

## Build and Preview

```bash
# Build for production (outputs to dist/)
npm run build

# Preview the production build locally
npm run preview

# Check for TypeScript errors
npx astro check
```

The build output in `dist/` is a fully static site ready for any CDN or static host.

## Environment Variables

Astro uses Vite's environment variable handling:

```bash
# .env
DATABASE_URL=postgresql://localhost:5432/mydb
SECRET_API_KEY=super-secret-key
PUBLIC_SITE_URL=https://example.com
PUBLIC_GA_ID=G-XXXXXXXXXX
```

```typescript
// In your .astro frontmatter or TS files:
const dbUrl = import.meta.env.DATABASE_URL;     // Server-only
const siteUrl = import.meta.env.PUBLIC_SITE_URL; // Available in browser too
```

:::caution[PUBLIC_ Prefix]
Only variables prefixed with `PUBLIC_` are included in the client-side JavaScript bundle.
All other variables are server-only and will be `undefined` in the browser. Never prefix secrets
with `PUBLIC_`.
:::

Environment variable types can be declared for TypeScript autocompletion:

```typescript
// src/env.d.ts
interface ImportMetaEnv {
  readonly DATABASE_URL: string;
  readonly SECRET_API_KEY: string;
  readonly PUBLIC_SITE_URL: string;
  readonly PUBLIC_GA_ID: string;
}

interface ImportMeta {
  readonly env: ImportMetaEnv;
}
```

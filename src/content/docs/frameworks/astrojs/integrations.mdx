# Astro Integrations

Astro's integration system lets you extend your project with UI frameworks, CSS tooling, SSR adapters,
and third-party libraries. Most integrations can be added with a single command using `npx astro add`.

## Adding Integrations

The `astro add` command installs and configures integrations automatically:

```bash
# Add React support
npx astro add react

# Add Vue support
npx astro add vue

# Add Tailwind CSS
npx astro add tailwind

# Add multiple at once
npx astro add react tailwind mdx sitemap
```

`astro add` performs three steps:

1. Installs the npm package(s)
2. Updates `astro.config.mjs` with the integration
3. Adds any required config files (e.g., `tailwind.config.mjs`)

## @astrojs/mdx

Enables MDX (Markdown + JSX) support throughout your project:

```bash
npx astro add mdx
```

```typescript
// astro.config.mjs
import mdx from '@astrojs/mdx';

export default defineConfig({
  integrations: [
    mdx({
      // Add remark plugins
      remarkPlugins: [remarkGfm, remarkSmartypants],
      // Add rehype plugins
      rehypePlugins: [
        rehypeSlug,
        [rehypeAutolinkHeadings, { behavior: 'wrap' }],
      ],
      // Optimize images in MDX
      optimize: true,
    }),
  ],
});
```

With MDX, you can import and use components inside Markdown files:

```mdx
---
title: My MDX Post
---

import Alert from '../../components/Alert.astro';
import ReactChart from '../../components/Chart.tsx';

# My MDX Post

<Alert type="info">
  This is an interactive alert component inside Markdown!
</Alert>

Here is some regular Markdown text with **bold** and _italic_.

<ReactChart client:visible data={[1, 2, 3, 4, 5]} />
```

## @astrojs/tailwind

```bash
npx astro add tailwind
```

```typescript
// astro.config.mjs
import tailwind from '@astrojs/tailwind';

export default defineConfig({
  integrations: [
    tailwind({
      // Prevent Astro from applying Tailwind's base styles (you control them)
      applyBaseStyles: false,
      // Point to a custom config file
      configFile: './tailwind.config.mjs',
    }),
  ],
});
```

```typescript
// tailwind.config.mjs
import { fontFamily } from 'tailwindcss/defaultTheme';

export default {
  content: ['./src/**/*.{astro,html,js,jsx,md,mdx,svelte,ts,tsx,vue}'],
  theme: {
    extend: {
      fontFamily: {
        sans: ['Inter var', ...fontFamily.sans],
      },
    },
  },
  plugins: [require('@tailwindcss/typography'), require('@tailwindcss/forms')],
};
```

## @astrojs/sitemap

Automatically generates a `sitemap-index.xml` on build:

```bash
npx astro add sitemap
```

```typescript
import sitemap from '@astrojs/sitemap';

export default defineConfig({
  site: 'https://example.com', // Required!
  integrations: [
    sitemap({
      filter: (page) => !page.includes('/private/'),
      changefreq: 'weekly',
      priority: 0.7,
      lastmod: new Date(),
      i18n: {
        defaultLocale: 'en',
        locales: {
          en: 'en-US',
          fr: 'fr-FR',
          es: 'es-ES',
        },
      },
    }),
  ],
});
```

## SSR Adapters

To enable server-side rendering (SSR), install an adapter for your deployment target:

```typescript
// astro.config.mjs
export default defineConfig({
  output: 'server', // 'static' | 'server' | 'hybrid'
  adapter: vercel(), // or netlify(), node(), cloudflare()
});
```

## Hybrid Rendering

`output: 'hybrid'` defaults all pages to static pre-rendering, but lets individual pages opt in to SSR:

```typescript
// astro.config.mjs
export default defineConfig({
  output: 'hybrid',
  adapter: vercel(),
});
```

```astro
---
// src/pages/dashboard.astro — Opt OUT of pre-rendering (becomes SSR)
export const prerender = false;

const user = Astro.locals.user;
if (!user) return Astro.redirect('/login');
---

<h1>Welcome, {user.name}!</h1>
```

```astro
---
// src/pages/about.astro — No export = pre-rendered (static) in hybrid mode
---

<h1>About Us</h1>
```

## Starlight Integration

[Starlight](https://starlight.astro.build) is Astro's official documentation framework:

```bash
npm create astro@latest -- --template starlight
# Or add to existing project:
npx astro add starlight
```

```typescript
// astro.config.mjs
import starlight from '@astrojs/starlight';

export default defineConfig({
  integrations: [
    starlight({
      title: 'My Documentation',
      description: 'Comprehensive guides and API reference.',
      logo: { src: './src/assets/logo.svg' },
      social: {
        github: 'https://github.com/myorg/myrepo',
      },
      sidebar: [
        { label: 'Guides', autogenerate: { directory: 'guides' } },
        { label: 'Reference', autogenerate: { directory: 'reference' } },
      ],
      defaultLocale: 'root',
      locales: {
        root: { label: 'English', lang: 'en' },
        fr: { label: 'Français', lang: 'fr' },
      },
    }),
  ],
});
```

## Writing a Custom Integration

Custom integrations are objects with a `name` and `hooks` property:

```typescript
// my-integration.ts
import type { AstroIntegration } from 'astro';

interface MyIntegrationOptions {
  token: string;
  endpoint?: string;
}

export function myIntegration(options: MyIntegrationOptions): AstroIntegration {
  return {
    name: 'my-integration',
    hooks: {
      // Runs first — modify the Astro config before anything else
      'astro:config:setup': ({
        config,
        updateConfig,
        injectScript,
        injectRoute,
        addMiddleware,
        logger,
      }) => {
        logger.info('Setting up my integration...');

        // Inject a script on every page
        injectScript('head-inline', `console.log('My Integration Active');`);

        // Add a virtual route
        injectRoute({
          pattern: '/my-integration/status',
          entrypoint: './src/my-integration/status.astro',
        });

        // Add middleware
        addMiddleware({
          entrypoint: './src/my-integration/middleware.ts',
          order: 'pre',
        });

        // Update the Astro config
        updateConfig({
          vite: {
            define: {
              __MY_TOKEN__: JSON.stringify(options.token),
            },
          },
        });
      },

      // Runs after the build completes
      'astro:build:done': async ({ dir, routes, pages, logger }) => {
        logger.info(`Build complete! ${pages.length} pages generated.`);
        // Post-process build output, submit to analytics, etc.
      },

      'astro:server:setup': ({ server, logger }) => {
        // Modify the Vite dev server
        server.middlewares.use('/api/health', (req, res) => {
          res.end(JSON.stringify({ ok: true }));
        });
      },
    },
  };
}
```

Usage:

```typescript
// astro.config.mjs
import { myIntegration } from './my-integration.ts';

export default defineConfig({
  integrations: [
    myIntegration({
      token: process.env.MY_API_TOKEN!,
      endpoint: 'https://api.example.com',
    }),
  ],
});
```

## Vite Plugin in astro.config

Astro accepts Vite plugins directly for lower-level customizations:

```typescript
// astro.config.mjs
import { defineConfig } from 'astro/config';

export default defineConfig({
  vite: {
    plugins: [
      // Any Vite plugin works here
      {
        name: 'vite-plugin-example',
        transform(code, id) {
          if (id.endsWith('.astro')) {
            return code.replace('__VERSION__', '1.0.0');
          }
        },
      },
    ],
    optimizeDeps: {
      include: ['my-heavy-lib'],
    },
    build: {
      rollupOptions: {
        output: {
          manualChunks: {
            vendor: ['react', 'react-dom'],
          },
        },
      },
    },
  },
});
```

:::caution[Integration vs Vite Plugin]
Prefer Astro integrations over raw Vite plugins when possible — they have access to Astro-specific
hooks like `astro:config:setup`, `astro:build:done`, and can inject routes, scripts, and middleware.
Use Vite plugins only when you need Vite-specific behavior (transforms, asset handling, etc.).
:::

Electron is an open-source framework developed by GitHub that enables developers to build native desktop applications using web technologies — HTML, CSS, and JavaScript. Under the hood it bundles **Chromium** for rendering and **Node.js** for system access, giving your app a full browser environment plus unfettered OS integration in a single binary.

## Architecture Overview

Every Electron application runs two kinds of processes:

| Process      | Environment      | Role                                             |
| ------------ | ---------------- | ------------------------------------------------ |
| **Main**     | Node.js          | App lifecycle, native OS APIs, window management |
| **Renderer** | Chromium (Blink) | UI rendering, one per `BrowserWindow`            |

The main process is the entry point. It creates and manages `BrowserWindow` instances. Each window spawns its own renderer process — an isolated Chromium context that renders your frontend. The two sides communicate exclusively through **IPC** (Inter-Process Communication), enforced by `contextBridge` when `contextIsolation: true` is set (the secure default).

```
┌─────────────────────────────────┐
│         Main Process            │
│  (Node.js + Electron APIs)      │
│  app · BrowserWindow · ipcMain  │
└────────────┬────────────────────┘
             │  IPC (ipcMain / ipcRenderer)
    ┌────────┴────────┐
    │                 │
┌───▼───┐         ┌───▼───┐
│ Win 1 │         │ Win 2 │
│Renderer│        │Renderer│
│(Chromium)│      │(Chromium)│
└───────┘         └───────┘
```

## Main vs Renderer Process Model

### Main Process

- Runs once per application instance
- Full Node.js API access (`fs`, `path`, `child_process`, etc.)
- Controls native menus, tray icons, dialogs, notifications
- Manages the application lifecycle (`app.on('ready')`, `app.quit()`)
- Creates and destroys `BrowserWindow` instances

### Renderer Process

- Runs once per `BrowserWindow` (or `BrowserView`/`WebContentsView`)
- Chromium sandbox; **no Node.js** by default (and never enable it)
- Receives Electron capabilities via `contextBridge` in a preload script
- Runs your React / Vue / Svelte / Angular frontend normally

:::caution[Never enable nodeIntegration]
Setting `nodeIntegration: true` grants renderer processes direct Node.js access. This is a critical security vulnerability — any XSS attack or malicious iframe can read your filesystem, execute arbitrary code, and exfiltrate data. Always keep `contextIsolation: true` and `nodeIntegration: false`.
:::

## Electron vs Alternatives

| Feature             | Electron         | Tauri          | NW.js            | PWA             |
| ------------------- | ---------------- | -------------- | ---------------- | --------------- |
| **Backend runtime** | Node.js          | Rust           | Node.js          | Browser only    |
| **Renderer**        | Bundled Chromium | System WebView | Bundled Chromium | System browser  |
| **Bundle size**     | ~150 MB          | ~5–10 MB       | ~150 MB          | N/A             |
| **Memory baseline** | ~100–150 MB      | ~10–30 MB      | ~100 MB          | ~50 MB          |
| **Node.js APIs**    | Full             | Via plugins    | Full             | None            |
| **Native modules**  | Yes              | Via Rust       | Yes              | No              |
| **Mobile support**  | No               | Yes (v2)       | No               | Yes             |
| **Security model**  | contextIsolation | Capabilities   | contextIsolation | Browser sandbox |
| **Maturity**        | Very mature      | Growing fast   | Mature           | Wide support    |
| **TypeScript**      | First-class      | Frontend TS    | Yes              | Yes             |
| **Hot reload dev**  | Yes              | Yes            | Yes              | Yes             |

:::tip[Choosing Electron]
Electron is the right choice when: you need maximum npm ecosystem access, your team is JavaScript-only, you require proven stability (VS Code, Slack, Discord all use it), or you need advanced WebRTC / media APIs unavailable in system WebViews.
:::

## Security Concerns Overview

Electron's power comes with responsibility. The most common pitfalls:

1. **`nodeIntegration: true`** — exposes Node.js to renderer; never enable
2. **`contextIsolation: false`** — allows renderer to access Electron internals
3. **`sandbox: false`** — disables Chromium OS-level sandboxing
4. **Loading remote content** — `loadURL('https://...')` brings web security risks into a privileged context
5. **`allowRunningInsecureContent: true`** — mixed HTTP/HTTPS; avoid
6. **Unvalidated IPC handlers** — always validate and sanitize all data received in `ipcMain.handle`
7. **`webSecurity: false`** — disables CORS and same-origin policy; only for specific local dev scenarios

Follow the [Electron Security Checklist](https://www.electronjs.org/docs/latest/tutorial/security) before shipping.

## Packaging Tools

### Electron Forge

The officially recommended tool for packaging and publishing Electron apps. Provides templates, plugin system, and makers for every platform.

```bash
npm create electron-app@latest my-app -- --template=vite-typescript
```

**Makers available:** `zip`, `squirrel.windows`, `dmg`, `deb`, `rpm`, `appx`, `snap`

### electron-builder

A mature alternative with comprehensive build configuration, auto-updater integration, and code signing support — popular for complex enterprise builds.

```bash
npm install --save-dev electron-builder
```

Both tools produce platform-native installers (NSIS/MSI, DMG, AppImage/deb) and support auto-update workflows.

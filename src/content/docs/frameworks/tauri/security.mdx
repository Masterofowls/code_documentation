Security is a first-class design goal in Tauri. The framework is architected around the principle of **least privilege** — the frontend WebView has no system access by default, and every capability must be explicitly declared. This page explains the security model, how to configure it correctly, and what to audit before shipping.

## The Tauri Security Model

```
WebView (untrusted zone)
  │   Can call only registered commands
  │   Has no filesystem, OS, or shell access by default
  │   Governed by CSP
  ▼
Capabilities Layer (explicit allowlist)
  │   Declares which permissions are granted
  │   Scoped to specific window labels
  │   Scoped to specific file paths and URLs
  ▼
Rust Commands & Plugins (privileged zone)
  │   All inputs are Rust-type-validated before reaching your code
  │   Your code runs with OS user privileges
  ▼
Operating System
```

No code path from the WebView to the OS exists that bypasses this chain. If a permission is not in a capability file, the corresponding API call fails at the Tauri layer before any Rust code runs.

## Capabilities System (Tauri 2)

### What Is a Capability?

A capability is a JSON file in `src-tauri/capabilities/` that grants a specific set of permissions to one or more windows:

```json
// src-tauri/capabilities/default.json
{
  "$schema": "../gen/schemas/desktop-schema.json",
  "identifier": "default",
  "description": "Grants basic functionality to the main window",
  "windows": ["main"],
  "permissions": [
    "core:default",
    "fs:allow-read-text-file",
    "fs:allow-write-text-file",
    "shell:allow-open",
    "dialog:allow-open",
    "dialog:allow-save"
  ]
}
```

Multiple capability files can coexist:

```json
// src-tauri/capabilities/admin.json
{
  "identifier": "admin",
  "description": "Elevated capabilities for the admin panel",
  "windows": ["admin"],
  "permissions": [
    "core:default",
    "fs:allow-read-text-file",
    "fs:allow-write-text-file",
    "fs:allow-read-dir",
    "fs:allow-mkdir",
    "fs:allow-remove"
  ]
}
```

### Tauri 1 Allowlist vs Tauri 2 Capabilities

| Tauri 1 (`allowlist`)                   | Tauri 2 (`capabilities`)                 |
| --------------------------------------- | ---------------------------------------- |
| Single global config block              | Multiple granular JSON files             |
| Boolean on/off per feature              | Scoped permissions with allow/deny lists |
| No path-level scoping                   | Path-level and URL-level scoping         |
| All windows shared the same permissions | Per-window permission assignment         |

### Permission Scoping

Restrict permissions to specific file paths or URLs using `allow` and `deny`:

```json
{
  "identifier": "scoped-fs",
  "windows": ["main"],
  "permissions": [
    // Read only from app data and user documents
    {
      "identifier": "fs:allow-read-text-file",
      "allow": [{ "path": "$APPDATA/**" }, { "path": "$DOCUMENT/**" }],
      "deny": [
        { "path": "$DOCUMENT/.ssh/**" },
        { "path": "$DOCUMENT/.gnupg/**" },
        { "path": "$HOME/.env" }
      ]
    },
    // Write only to app-controlled directories
    {
      "identifier": "fs:allow-write-text-file",
      "allow": [
        { "path": "$APPDATA/config/**" },
        { "path": "$APPDATA/exports/**" },
        { "path": "$TEMP/tauri-**" }
      ]
    },
    // HTTP: only allow specific domains
    {
      "identifier": "http:allow-fetch",
      "allow": [
        { "url": "https://api.yourservice.com/**" },
        { "url": "https://cdn.yourservice.com/**" }
      ]
    }
  ]
}
```

:::caution[Deny Rules Take Precedence]
`deny` entries are always evaluated before `allow`. A path matching both a `deny` and an `allow` rule will be denied. Use this to create exceptions within broad allow patterns (e.g., allow `$DOCUMENT/**` but deny `$DOCUMENT/.ssh/**`).
:::

## Content Security Policy

Configure the CSP in `tauri.conf.json` to mitigate XSS and restrict resource loading:

```json
// tauri.conf.json
{
  "app": {
    "security": {
      "csp": "default-src 'self'; script-src 'self'; style-src 'self' 'unsafe-inline'; img-src 'self' asset: https://asset.localhost data: blob:; font-src 'self' data:; connect-src 'self' ipc: http://ipc.localhost; media-src 'none'; object-src 'none'; frame-src 'none';",

      "dangerousDisableAssetCspModification": false,
      "devCsp": "default-src 'self' 'unsafe-eval'; connect-src ipc: http://ipc.localhost ws://localhost:*"
    }
  }
}
```

### CSP Directive Explanations

| Directive     | Recommended Value                  | Reason                                 |
| ------------- | ---------------------------------- | -------------------------------------- |
| `default-src` | `'self'`                           | Fallback for unspecified directives    |
| `script-src`  | `'self'`                           | No inline scripts, no eval             |
| `style-src`   | `'self' 'unsafe-inline'`           | `'unsafe-inline'` needed for CSS-in-JS |
| `img-src`     | `'self' asset: data: blob:`        | Local images + Tauri asset protocol    |
| `connect-src` | `'self' ipc: http://ipc.localhost` | Tauri IPC protocol required            |
| `object-src`  | `'none'`                           | Block all plugins (Flash, etc.)        |
| `frame-src`   | `'none'`                           | Block iframes (unless needed)          |
| `media-src`   | `'none'`                           | Block media unless your app needs it   |

:::tip[Development CSP]
The `devCsp` key applies only in development mode. Use it to allow `'unsafe-eval'` that some dev tools require without weakening your production CSP.
:::

## Disabling Dangerous Features

```json
// tauri.conf.json
{
  "app": {
    "security": {
      "csp": "...",

      // Set to true to prevent Tauri from modifying your CSP headers
      // (only do this if you fully control all CSP headers yourself)
      "dangerousDisableAssetCspModification": false,

      // Allow specific remote domains to use IPC (Tauri 2 only — be very careful)
      // This should NEVER be used unless you fully trust the remote origin
      "dangerousRemoteDomainIpcAccess": []
    }
  }
}
```

:::caution[dangerousRemoteDomainIpcAccess]
Granting IPC access to remote domains is equivalent to enabling `nodeIntegration: true` in Electron for that domain. An XSS vulnerability on the remote site would have full access to your Tauri commands. Never enable this unless you fully control the remote origin and understand the implications.
:::

## Input Validation in Rust

Tauri validates that inputs match the Rust parameter types before your code runs, but you must still validate business logic:

```rust
use std::path::{Path, PathBuf};
use tauri::{AppHandle, Manager};

#[tauri::command]
pub async fn safe_read_file(
    app: AppHandle,
    path: String,
) -> Result<String, String> {
    let requested = PathBuf::from(&path);

    // 1. Resolve to an absolute path (prevent relative path traversal)
    let canonical = requested.canonicalize()
        .map_err(|_| "Invalid path".to_string())?;

    // 2. Check that the path is within an allowed base directory
    let allowed_base = app.path()
        .app_data_dir()
        .map_err(|e| e.to_string())?;

    if !canonical.starts_with(&allowed_base) {
        return Err("Access denied: path outside app data directory".to_string());
    }

    // 3. Check file size before reading
    let metadata = tokio::fs::metadata(&canonical)
        .await
        .map_err(|e| e.to_string())?;

    const MAX_FILE_SIZE: u64 = 10 * 1024 * 1024; // 10 MB
    if metadata.len() > MAX_FILE_SIZE {
        return Err(format!("File too large: {} bytes (max {})", metadata.len(), MAX_FILE_SIZE));
    }

    // 4. Read
    tokio::fs::read_to_string(&canonical)
        .await
        .map_err(|e| e.to_string())
}
```

## Avoiding shell:execute

The `shell` plugin's `execute` command can run arbitrary system commands. Prefer specific Rust implementations over shell execution:

```typescript
// ❌ Dangerous — if userInput is tainted, this is command injection
await invoke('shell_execute', { program: 'rm', args: ['-rf', userInput] });
```

```rust
// ✅ Safe — use a specific Rust command with validated inputs
#[tauri::command]
pub async fn delete_export_file(
    filename: String,
    app: AppHandle,
) -> Result<(), String> {
    // Validate filename — no path separators, no special chars
    if filename.contains('/') || filename.contains('\\') || filename.contains("..") {
        return Err("Invalid filename".to_string());
    }

    let file_path = app.path()
        .app_data_dir()
        .map_err(|e| e.to_string())?
        .join("exports")
        .join(&filename);

    tokio::fs::remove_file(&file_path)
        .await
        .map_err(|e| e.to_string())
}
```

If you must use `shell:allow-open` for opening URLs in the system browser:

```json
// Grant only shell:allow-open (not shell:allow-execute)
{
  "permissions": [
    {
      "identifier": "shell:allow-open",
      "allow": [{ "url": "https://**" }, { "url": "mailto:**" }]
    }
  ]
}
```

## DevTools in Production

Tauri automatically disables DevTools in release builds. Verify this is the case:

```rust
// src-tauri/src/lib.rs
.setup(|app| {
    // Only open DevTools in debug builds
    #[cfg(debug_assertions)]
    {
        let window = app.get_webview_window("main").unwrap();
        window.open_devtools();
    }
    Ok(())
})
```

## HTTPS for Remote Content

If your app loads content from a remote URL, enforce HTTPS:

```json
// tauri.conf.json
{
  "app": {
    "windows": [
      {
        "url": "https://app.yourservice.com"
      }
    ],
    "security": {
      "csp": "default-src 'self' https://app.yourservice.com;"
    }
  }
}
```

:::caution[Loading Remote Content]
If you load a remote URL in your Tauri window, that remote origin can call any IPC command that the window's capability allows. Use `dangerousRemoteDomainIpcAccess` only if absolutely necessary, and scope it tightly to specific commands with minimal permissions.
:::

## Security Audit Checklist

Before releasing your Tauri app, verify every item:

### Capabilities

- [ ] All capability files are reviewed — no overly broad permissions
- [ ] `fs` permissions are scoped to specific directories, not `**` globally
- [ ] `http` permissions are scoped to specific domains
- [ ] `shell:allow-execute` is not granted unless absolutely necessary
- [ ] `dangerousRemoteDomainIpcAccess` is empty or absent
- [ ] `dangerousDisableAssetCspModification` is `false`

### Commands

- [ ] All command inputs are validated in Rust (not just TypeScript)
- [ ] Path inputs are canonicalized and verified against an allowed base
- [ ] File size limits are enforced before reading
- [ ] No `unwrap()` in command handlers — use `?` or `map_err`
- [ ] Sensitive data is not logged with `println!` or `dbg!`

### Content Security Policy

- [ ] CSP is configured in `tauri.conf.json`
- [ ] `script-src` does not include `'unsafe-eval'` in production
- [ ] `object-src` is `'none'`
- [ ] `frame-src` is `'none'` (unless iframes are needed)

### Dependencies

- [ ] Run `cargo audit` — no HIGH or CRITICAL CVEs in Rust dependencies
- [ ] Run `npm audit` — no HIGH or CRITICAL CVEs in frontend dependencies
- [ ] Dependencies are pinned to specific versions in `Cargo.lock` and `package-lock.json`

### Distribution

- [ ] App is code-signed for macOS (Developer ID + notarized)
- [ ] App is code-signed for Windows (EV or standard certificate)
- [ ] Update mechanism uses signature verification (pubkey in `tauri-plugin-updater`)
- [ ] Release builds have debug symbols stripped (`strip = true` in `[profile.release]`)

## Known Vulnerabilities to Avoid

| Pattern                                       | Risk                        | Mitigation                                                 |
| --------------------------------------------- | --------------------------- | ---------------------------------------------------------- |
| Loading user-provided URLs in the main window | Remote code gets IPC access | Validate and open in system browser via `shell:allow-open` |
| Passing unsanitized paths to Rust commands    | Path traversal              | Canonicalize and verify against allowed base               |
| Using `eval()` in frontend                    | XSS amplification           | CSP blocks eval; use JSON.parse instead                    |
| Storing secrets in frontend JS                | Secret exposure             | Keep secrets Rust-side; never embed API keys in frontend   |
| Shell execution with user input               | Command injection           | Use specific Rust implementations instead of shell         |
| HTTP (not HTTPS) for remote content           | MITM attacks                | Enforce HTTPS; CSP blocks http://                          |

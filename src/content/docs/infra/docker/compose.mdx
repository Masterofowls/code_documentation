# Docker Compose

Docker Compose is a tool for defining and running multi-container applications. You describe your entire stack — services, networks, volumes, configs, and secrets — in a single `compose.yml` file, then bring the whole environment up or down with a single command.

:::tip[Compose v2]
Docker Compose v2 is the current version, shipped as a Docker CLI plugin (`docker compose`, no hyphen). The old standalone `docker-compose` (Python, v1) is deprecated. All examples in this guide use v2 syntax.
:::

## File Format

The Compose file is named `compose.yml` (preferred) or `docker-compose.yml` (legacy, still supported). It is valid YAML at the top level with five root keys:

```yaml
# compose.yml
name: myapp  # Optional project name (default: directory name)

services:    # Container definitions
  web: ...
  db: ...

networks:    # Custom network definitions
  frontend: ...
  backend: ...

volumes:     # Named volume definitions
  pgdata: ...

configs:     # Configuration file definitions (Swarm / Compose 2.23+)
  nginx_conf: ...

secrets:     # Secret definitions
  db_password: ...
```

## Service Definition

A service is a container template. Compose creates one or more containers from each service definition.

```yaml
services:
  api:
    # Image to use (mutually exclusive with build)
    image: myorg/api:1.2.3

    # Build an image from a Dockerfile
    build:
      context: ./api
      dockerfile: Dockerfile
      target: runtime           # Multi-stage build target
      args:
        NODE_ENV: production
      cache_from:
        - myorg/api:latest
      labels:
        - "com.example.version=1.2.3"

    # Port mappings: "host:container"
    ports:
      - "3000:3000"
      - "127.0.0.1:9229:9229"  # Bind debug port to localhost only

    # Environment variables (inline)
    environment:
      NODE_ENV: production
      PORT: 3000
      DATABASE_URL: postgres://postgres:secret@db:5432/myapp

    # Environment from file
    env_file:
      - .env
      - .env.production

    # Volume mounts
    volumes:
      - uploads:/app/uploads              # Named volume
      - ./config:/app/config:ro           # Bind mount, read-only
      - /tmp/cache:/tmp/cache             # Absolute bind mount

    # Dependency ordering
    depends_on:
      db:
        condition: service_healthy        # Wait for health check to pass
      redis:
        condition: service_started        # Wait for container to start (no health check)

    # Restart policy
    restart: unless-stopped

    # Health check
    healthcheck:
      test: ["CMD", "wget", "-qO-", "http://localhost:3000/health"]
      interval: 30s
      timeout: 10s
      retries: 3
      start_period: 15s

    # Override Dockerfile CMD
    command: ["node", "dist/server.js"]

    # Override Dockerfile ENTRYPOINT
    entrypoint: ["/sbin/tini", "--"]

    # Container and hostname
    container_name: myapp-api
    hostname: api

    # Network membership
    networks:
      - frontend
      - backend

    # Resource limits
    deploy:
      resources:
        limits:
          cpus: "0.5"
          memory: 512M
        reservations:
          cpus: "0.25"
          memory: 256M

    # Profiles — only start when explicitly requested
    profiles:
      - production

    # Labels
    labels:
      com.example.team: backend
      traefik.enable: "true"

    # Secrets
    secrets:
      - db_password
      - api_key

    # Logging driver
    logging:
      driver: json-file
      options:
        max-size: "10m"
        max-file: "3"
```

## Networks

```yaml
networks:
  # Default bridge network (created automatically)
  default:
    driver: bridge

  # Custom named network
  frontend:
    driver: bridge
    ipam:
      config:
        - subnet: 172.20.0.0/16

  backend:
    driver: bridge
    internal: true  # No external connectivity

  # Use a pre-existing external network
  monitoring:
    external: true
    name: monitoring_network
```

Services on the same custom network can reach each other by **service name** as the hostname. Services on different networks cannot communicate unless they share at least one network.

```yaml
services:
  api:
    networks:
      frontend:
        aliases:
          - api
          - backend-api
      backend: {}

  db:
    networks:
      - backend  # Only on backend — api can reach db, but frontend cannot
```

## Volumes

```yaml
volumes:
  # Simple named volume (Docker manages storage in /var/lib/docker/volumes/)
  pgdata: {}

  # With driver options (NFS example)
  nfsdata:
    driver: local
    driver_opts:
      type: nfs
      o: addr=192.168.1.100,rw
      device: ":/exports/data"

  # Use a pre-existing volume
  external_uploads:
    external: true
    name: production_uploads
```

### Named Volumes vs Bind Mounts

```yaml
services:
  db:
    volumes:
      # Named volume — persists in Docker's storage area, survives container removal
      - pgdata:/var/lib/postgresql/data

  app:
    volumes:
      # Bind mount — mounts a host path, great for development hot reload
      - ./src:/app/src

      # Read-only bind mount
      - ./nginx.conf:/etc/nginx/nginx.conf:ro

      # tmpfs — in-memory, not persisted
      - type: tmpfs
        target: /tmp
        tmpfs:
          size: 100m
```

## Override Files

Compose merges multiple files, with later files overriding earlier ones. This enables a base config plus environment-specific overrides.

```yaml
# compose.yml — base configuration
services:
  api:
    build: .
    environment:
      NODE_ENV: development
    volumes:
      - .:/app
      - /app/node_modules
```

```yaml
# compose.override.yml — automatically merged in development
services:
  api:
    ports:
      - "3000:3000"
      - "9229:9229"   # Debug port
    command: ["node", "--inspect=0.0.0.0:9229", "dist/server.js"]
```

```yaml
# compose.prod.yml — explicitly specified for production
services:
  api:
    image: myorg/api:${IMAGE_TAG:-latest}
    environment:
      NODE_ENV: production
    deploy:
      replicas: 3
      resources:
        limits:
          memory: 512M
```

```bash
# Development (uses compose.yml + compose.override.yml automatically)
docker compose up

# Production (explicitly specify files)
docker compose -f compose.yml -f compose.prod.yml up -d
```

## Essential Commands

```bash
# Start all services (build images if needed)
docker compose up

# Start in detached mode
docker compose up -d

# Force rebuild images before starting
docker compose up --build

# Start specific services only
docker compose up api db

# Stop and remove containers, networks (keeps volumes and images)
docker compose down

# Also remove volumes
docker compose down --volumes

# Also remove images built by Compose
docker compose down --rmi all --volumes

# List running services
docker compose ps

# List all services (including stopped)
docker compose ps -a

# View logs
docker compose logs

# Follow logs for specific services
docker compose logs -f api db

# Last 50 lines
docker compose logs --tail 50 api

# Execute a command in a running service
docker compose exec api sh
docker compose exec db psql -U postgres

# Run a one-off command in a new container
docker compose run --rm api npm run migrate

# Build images without starting
docker compose build

# Pull latest images
docker compose pull

# Restart specific service
docker compose restart api

# Show service config after merging all files
docker compose config

# Validate compose file syntax
docker compose config --quiet

# List compose projects
docker compose ls
```

## Scaling Services

```bash
# Scale a service to 3 replicas
docker compose up -d --scale api=3

# Scale down
docker compose up -d --scale api=1
```

:::caution[Port Conflicts When Scaling]
You cannot scale a service that has a fixed host port mapping (e.g., `- "3000:3000"`) because multiple containers cannot bind to the same host port. Use a load balancer (like Nginx) in front, or omit the host port and let Docker assign random ports.
:::

## Secrets in Compose

```yaml
secrets:
  db_password:
    file: ./secrets/db_password.txt   # Read from file
  api_key:
    environment: API_KEY              # Read from environment variable

services:
  api:
    secrets:
      - db_password
      - api_key
    # Secrets are mounted at /run/secrets/<secret_name>
    # Read in app: fs.readFileSync('/run/secrets/db_password', 'utf8')
```

## Health Checks and `depends_on`

```yaml
services:
  db:
    image: postgres:16-alpine
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U postgres"]
      interval: 10s
      timeout: 5s
      retries: 5
      start_period: 30s

  api:
    depends_on:
      db:
        condition: service_healthy   # Won't start until db passes health check
```

## Profiles

Profiles let you selectively start services. Services without a profile always start; services with a profile only start when requested.

```yaml
services:
  api:
    image: myorg/api:latest       # No profile — always starts

  worker:
    image: myorg/worker:latest
    profiles: [worker]            # Only starts when "worker" profile is active

  mailhog:
    image: mailhog/mailhog
    profiles: [dev-tools]         # Local email testing only in dev

  pgadmin:
    image: dpage/pgadmin4
    profiles: [dev-tools]
```

```bash
# Start only base services
docker compose up -d

# Start base + worker profile
docker compose --profile worker up -d

# Start base + dev-tools profile
docker compose --profile dev-tools up -d
```

## Complete Full-Stack Example

This example runs a Node.js API, PostgreSQL database, Redis cache, and Nginx reverse proxy.

```yaml
# compose.yml
name: fullstack-app

services:

  # ── Nginx Reverse Proxy ──────────────────────────────────
  nginx:
    image: nginx:1.27-alpine
    ports:
      - "80:80"
      - "443:443"
    volumes:
      - ./nginx/nginx.conf:/etc/nginx/nginx.conf:ro
      - ./nginx/ssl:/etc/nginx/ssl:ro
    depends_on:
      api:
        condition: service_healthy
    networks:
      - frontend
    restart: unless-stopped
    logging:
      driver: json-file
      options:
        max-size: "10m"
        max-file: "3"

  # ── Node.js API ──────────────────────────────────────────
  api:
    build:
      context: .
      target: runtime
    environment:
      NODE_ENV: production
      PORT: 3000
      DATABASE_URL: postgres://app:${DB_PASSWORD}@db:5432/myapp
      REDIS_URL: redis://redis:6379
    env_file:
      - .env
    volumes:
      - uploads:/app/uploads
    depends_on:
      db:
        condition: service_healthy
      redis:
        condition: service_healthy
    networks:
      - frontend
      - backend
    restart: unless-stopped
    healthcheck:
      test: ["CMD", "wget", "-qO-", "http://localhost:3000/health"]
      interval: 30s
      timeout: 10s
      retries: 3
      start_period: 20s
    logging:
      driver: json-file
      options:
        max-size: "20m"
        max-file: "5"
    deploy:
      resources:
        limits:
          cpus: "0.5"
          memory: 512M

  # ── PostgreSQL Database ──────────────────────────────────
  db:
    image: postgres:16-alpine
    environment:
      POSTGRES_USER: app
      POSTGRES_PASSWORD: ${DB_PASSWORD}
      POSTGRES_DB: myapp
    volumes:
      - pgdata:/var/lib/postgresql/data
      - ./db/init:/docker-entrypoint-initdb.d:ro
    networks:
      - backend
    restart: unless-stopped
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U app -d myapp"]
      interval: 10s
      timeout: 5s
      retries: 5
      start_period: 30s
    logging:
      driver: json-file
      options:
        max-size: "10m"
        max-file: "3"

  # ── Redis Cache ──────────────────────────────────────────
  redis:
    image: redis:7-alpine
    command: redis-server --appendonly yes --maxmemory 256mb --maxmemory-policy allkeys-lru
    volumes:
      - redisdata:/data
    networks:
      - backend
    restart: unless-stopped
    healthcheck:
      test: ["CMD", "redis-cli", "ping"]
      interval: 10s
      timeout: 5s
      retries: 5
    logging:
      driver: json-file
      options:
        max-size: "5m"
        max-file: "2"

networks:
  frontend:
    driver: bridge
  backend:
    driver: bridge
    internal: false

volumes:
  pgdata:
    name: fullstack_pgdata
  redisdata:
    name: fullstack_redisdata
  uploads:
    name: fullstack_uploads
```

```nginx
# nginx/nginx.conf
upstream api {
    server api:3000;
}

server {
    listen 80;
    server_name _;

    location /api/ {
        proxy_pass http://api/;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
    }

    location /uploads/ {
        alias /var/www/uploads/;
    }
}
```

Start the stack:

```bash
# Create .env file
echo "DB_PASSWORD=supersecret" > .env

# Start everything
docker compose up -d

# Check status
docker compose ps

# Run database migrations
docker compose exec api npm run migrate

# Tail logs
docker compose logs -f api db
```

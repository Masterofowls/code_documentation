# Writing Dockerfiles

A `Dockerfile` is a text document containing all the commands needed to assemble a Docker image. Understanding how to write efficient, secure Dockerfiles is one of the most valuable Docker skills.

---

## Dockerfile Instructions Reference

### `FROM` — Base Image

Every Dockerfile must begin with `FROM`. It sets the base image for subsequent instructions.

```dockerfile
FROM ubuntu:22.04
FROM node:20-alpine
FROM scratch                    # Empty base — for statically compiled binaries
FROM node:20-alpine AS builder  # Named stage for multi-stage builds
```

### `RUN` — Execute Commands

Executes commands in a new layer on top of the current image and commits the result.

```dockerfile
# Shell form (runs in /bin/sh -c on Linux)
RUN apt-get update && apt-get install -y curl

# Exec form (no shell processing, preferred for clarity)
RUN ["apt-get", "install", "-y", "curl"]

# Chain commands to minimize layers
RUN apt-get update \
    && apt-get install -y --no-install-recommends \
       curl \
       git \
       ca-certificates \
    && rm -rf /var/lib/apt/lists/*
```

:::tip[Clean up in the same RUN]
Always clean package manager caches in the same `RUN` instruction. If you clean in a separate layer, the original cache layer is preserved in the image, wasting space.
:::

### `COPY` and `ADD`

```dockerfile
# COPY — preferred for most cases
COPY package.json package-lock.json ./
COPY src/ ./src/
COPY --chown=node:node . .         # Set ownership on copy

# ADD — has extra features (auto-extract archives, URL fetch)
# Prefer COPY unless you specifically need these features
ADD archive.tar.gz /app/           # Auto-extracted
ADD https://example.com/file.txt . # URL download (use curl/wget in RUN instead)
```

### `WORKDIR` — Set Working Directory

```dockerfile
WORKDIR /app                       # Creates dir if it doesn't exist
WORKDIR /usr/local/src             # Absolute path recommended
```

### `ENV` and `ARG`

```dockerfile
# ENV — runtime environment variable (persists in container)
ENV NODE_ENV=production
ENV PORT=3000 HOST=0.0.0.0        # Multiple on one line

# ARG — build-time variable (not available in running container)
ARG NODE_VERSION=20
ARG BUILD_DATE
FROM node:${NODE_VERSION}-alpine

# ARG before FROM is only available in FROM
# ARG after FROM is available in subsequent instructions
ARG APP_VERSION
ENV APP_VERSION=${APP_VERSION}    # Promote ARG to ENV for runtime access
```

### `EXPOSE`

Documents which ports the container will listen on. **Does not actually publish ports** — use `-p` at runtime.

```dockerfile
EXPOSE 3000
EXPOSE 80/tcp
EXPOSE 53/udp
```

### `VOLUME`

Declares a mount point that should be externally mounted. Creates an anonymous volume if not mounted at runtime.

```dockerfile
VOLUME /app/data
VOLUME ["/var/log", "/var/db"]
```

### `USER` — Non-Root Security

```dockerfile
# Create a non-root user and switch to it
RUN addgroup --system appgroup && adduser --system --ingroup appgroup appuser
USER appuser

# Or use numeric IDs for clarity
RUN addgroup -S -g 1001 app && adduser -S -u 1001 -G app app
USER 1001:1001
```

### `LABEL`

Metadata as key-value pairs. Useful for automation, maintenance, and tooling.

```dockerfile
LABEL maintainer="team@example.com"
LABEL org.opencontainers.image.title="My App"
LABEL org.opencontainers.image.version="1.0.0"
LABEL org.opencontainers.image.source="https://github.com/org/repo"
LABEL org.opencontainers.image.licenses="MIT"
```

### `HEALTHCHECK`

```dockerfile
HEALTHCHECK --interval=30s --timeout=10s --start-period=30s --retries=3 \
  CMD curl -f http://localhost:3000/health || exit 1

# Disable inherited healthcheck
HEALTHCHECK NONE
```

### `ENTRYPOINT` and `CMD`

```dockerfile
# ENTRYPOINT — the executable (cannot be overridden without --entrypoint flag)
ENTRYPOINT ["node", "server.js"]

# CMD — default arguments (can be overridden by docker run args)
CMD ["--port", "3000"]

# Together: ENTRYPOINT + CMD provides executable + default args
ENTRYPOINT ["nginx", "-g"]
CMD ["daemon off;"]

# Shell form (runs in sh -c — avoids signal issues, prefer exec form)
ENTRYPOINT node server.js    # Avoid: SIGTERM won't reach Node
```

:::caution[ENTRYPOINT + CMD interaction]
When both are defined in exec form, CMD provides default arguments to ENTRYPOINT. If you override CMD at runtime (`docker run my-image --debug`), the full command becomes `[ENTRYPOINT, "--debug"]`.
:::

### `SHELL`, `ONBUILD`, `STOPSIGNAL`

```dockerfile
# SHELL — change the default shell for RUN/CMD/ENTRYPOINT shell form
SHELL ["/bin/bash", "-o", "pipefail", "-c"]

# ONBUILD — triggers instruction in child images that use this as base
ONBUILD COPY . /app
ONBUILD RUN npm install

# STOPSIGNAL — signal sent to container to stop it
STOPSIGNAL SIGTERM
STOPSIGNAL SIGQUIT
```

## Layer Caching

Docker caches each instruction as a layer. If nothing has changed up to a given layer, Docker reuses the cache. **Order matters enormously.**

### The Wrong Way

```dockerfile
# BAD: Copying everything first invalidates cache on any source change
FROM node:20-alpine
WORKDIR /app
COPY . .                    # ← Any file change busts all subsequent cache
RUN npm ci                  # ← Re-runs on every source code change!
CMD ["node", "server.js"]
```

### The Right Way

```dockerfile
# GOOD: Copy dependency files first, install, then copy source
FROM node:20-alpine
WORKDIR /app
COPY package.json package-lock.json ./   # ← Only changes when deps change
RUN npm ci                               # ← Cache hit on most source changes
COPY . .                                 # ← Source change only affects this layer
CMD ["node", "server.js"]
```

**Cache invalidation rules:**
- `COPY` or `ADD` — invalidated if file content or metadata changes
- `RUN` — invalidated if preceding instructions changed
- `ENV`/`ARG` — invalidated if value changes
- `FROM` — invalidated if base image digest changes

:::tip[BuildKit cache mounts]
Use BuildKit cache mounts for package managers to persist the cache across builds without embedding it in the layer:
```dockerfile
RUN --mount=type=cache,target=/root/.npm \
    npm ci
```
:::

## Multi-Stage Builds

Multi-stage builds allow you to use multiple `FROM` instructions, copying only what you need between stages. This dramatically reduces final image size.

```dockerfile
# Stage 1: Build
FROM node:20-alpine AS builder
WORKDIR /app
COPY package*.json ./
RUN npm ci
COPY . .
RUN npm run build

# Stage 2: Production runtime (lean image)
FROM node:20-alpine AS production
WORKDIR /app

# Create non-root user
RUN addgroup -S -g 1001 app && adduser -S -u 1001 -G app app

# Copy ONLY the built artifacts from the builder stage
COPY --from=builder --chown=app:app /app/dist ./dist
COPY --from=builder --chown=app:app /app/node_modules ./node_modules
COPY --from=builder --chown=app:app /app/package.json ./

USER app
EXPOSE 3000
HEALTHCHECK --interval=30s --timeout=5s CMD wget -qO- http://localhost:3000/health || exit 1
CMD ["node", "dist/server.js"]
```

```bash
# Build the final (default last) stage
docker build -t my-app .

# Build a specific stage (useful for CI testing)
docker build --target builder -t my-app:builder .
docker build --target production -t my-app:prod .
```

### Copy from External Images

```dockerfile
FROM alpine AS final
# Copy a binary from a pre-built official image
COPY --from=golang:1.22-alpine /usr/local/go /usr/local/go
```

## `.dockerignore`

Like `.gitignore`, this file prevents files from being sent to the build context (and thus from being copied into images via `COPY . .`).

```
# .dockerignore
.git
.gitignore
node_modules
npm-debug.log
dist
build
.DS_Store
*.md
.env
.env.*
!.env.example
coverage
.nyc_output
.cache
*.log
Dockerfile*
docker-compose*
.dockerignore
```

:::tip[Build context size]
A large build context slows down every build. Run `docker build` and watch the "Sending build context" line. If it's large, your `.dockerignore` needs work.
:::

## Base Image Selection

| Base | Size | Best For |
|---|---|---|
| `ubuntu:22.04` | ~70 MB | Full Linux environment, APT packages |
| `debian:bookworm-slim` | ~75 MB | Lighter Debian, fewer packages pre-installed |
| `alpine:3.19` | ~5 MB | Minimal size; uses musl libc (can cause issues) |
| `node:20-alpine` | ~60 MB | Node.js on Alpine |
| `node:20-slim` | ~200 MB | Node.js on Debian Slim |
| `gcr.io/distroless/nodejs20` | ~120 MB | No shell, no package manager — very secure |
| `scratch` | 0 B | Statically compiled binaries (Go, Rust) |

:::caution[Alpine and musl libc]
Alpine uses `musl libc` instead of `glibc`. Some native Node.js modules (and Python C extensions) expect glibc. If you see segfaults or mysterious errors with Alpine, switch to `-slim` or `-debian` variants.
:::

## Security: Non-Root Users

Running containers as root is a significant security risk. Always create and use a non-root user.

```dockerfile
# Debian/Ubuntu style
RUN groupadd --gid 1001 appgroup \
    && useradd --uid 1001 --gid appgroup --shell /bin/bash --create-home appuser
USER appuser

# Alpine style
RUN addgroup -S -g 1001 app && adduser -S -u 1001 -G app app
USER 1001

# Node.js official images include a 'node' user
USER node
```

## tini as PID 1

In containers, PID 1 has special responsibilities (reaping zombie processes, forwarding signals). Using your app as PID 1 can cause problems. Use `tini` as a proper init process.

```dockerfile
FROM node:20-alpine
# tini is included in node:alpine — enable with --init flag
# or install it:
RUN apk add --no-cache tini
ENTRYPOINT ["/sbin/tini", "--"]
CMD ["node", "server.js"]
```

```bash
# Alternatively, use --init flag at runtime
docker run --init my-app
```

## Build Secrets

Pass secrets to the build process without embedding them in image layers.

```dockerfile
# syntax=docker/dockerfile:1
FROM alpine
RUN --mount=type=secret,id=npmrc,target=/root/.npmrc \
    npm install
```

```bash
docker build --secret id=npmrc,src=$HOME/.npmrc .
```

## BuildKit and `buildx`

BuildKit is Docker's modern build engine — faster, more features, better caching.

```bash
# Enable BuildKit
DOCKER_BUILDKIT=1 docker build .

# Or set in daemon.json
# { "features": { "buildkit": true } }

# buildx — extends BuildKit with multi-platform and more
docker buildx build \
  --platform linux/amd64,linux/arm64 \
  --tag myrepo/myapp:latest \
  --push \
  .

# Create a builder instance
docker buildx create --name mybuilder --use
docker buildx inspect --bootstrap
```

## Linting with hadolint

```bash
# Install
brew install hadolint           # macOS
docker run --rm -i hadolint/hadolint < Dockerfile  # Docker

# Check your Dockerfile
hadolint Dockerfile

# Ignore specific rules
hadolint --ignore DL3008 --ignore DL3009 Dockerfile
```

Common hadolint rules:
- `DL3007` — Use pinned tags, not `latest`
- `DL3008` — Pin apt-get package versions
- `DL3015` — Avoid `apt-get upgrade`
- `DL3025` — Use exec form of CMD
- `SC2086` — Shell quoting issues

## Complete Examples

### Node.js Application

```dockerfile
# syntax=docker/dockerfile:1
FROM node:20-alpine AS deps
WORKDIR /app
COPY package*.json ./
RUN --mount=type=cache,target=/root/.npm \
    npm ci --only=production

FROM node:20-alpine AS builder
WORKDIR /app
COPY package*.json ./
RUN --mount=type=cache,target=/root/.npm \
    npm ci
COPY . .
RUN npm run build

FROM node:20-alpine AS runner
WORKDIR /app
ENV NODE_ENV=production

RUN addgroup -S -g 1001 nodejs \
    && adduser -S -u 1001 -G nodejs nextjs

COPY --from=builder --chown=nextjs:nodejs /app/dist ./dist
COPY --from=deps --chown=nextjs:nodejs /app/node_modules ./node_modules
COPY --chown=nextjs:nodejs package.json ./

USER nextjs
EXPOSE 3000
ENV PORT=3000 HOST=0.0.0.0

HEALTHCHECK --interval=30s --timeout=5s --start-period=10s --retries=3 \
  CMD wget -qO- http://localhost:3000/health || exit 1

ENTRYPOINT ["/sbin/tini", "--"]
CMD ["node", "dist/server.js"]
```

### Python Application

```dockerfile
# syntax=docker/dockerfile:1
FROM python:3.12-slim AS builder
WORKDIR /app

RUN apt-get update \
    && apt-get install -y --no-install-recommends gcc \
    && rm -rf /var/lib/apt/lists/*

COPY requirements.txt ./
RUN --mount=type=cache,target=/root/.cache/pip \
    pip install --user --no-cache-dir -r requirements.txt

FROM python:3.12-slim AS runner
WORKDIR /app

RUN addgroup --system --gid 1001 appgroup \
    && adduser --system --uid 1001 --gid 1001 appuser

COPY --from=builder --chown=appuser:appgroup /root/.local /home/appuser/.local
COPY --chown=appuser:appgroup . .

USER appuser
ENV PATH=/home/appuser/.local/bin:$PATH \
    PYTHONDONTWRITEBYTECODE=1 \
    PYTHONUNBUFFERED=1

EXPOSE 8000
HEALTHCHECK --interval=30s --timeout=5s CMD curl -f http://localhost:8000/health || exit 1
CMD ["uvicorn", "app.main:app", "--host", "0.0.0.0", "--port", "8000"]
```

### Go Application (scratch)

```dockerfile
# syntax=docker/dockerfile:1
FROM golang:1.22-alpine AS builder
WORKDIR /app

COPY go.mod go.sum ./
RUN --mount=type=cache,target=/go/pkg/mod \
    go mod download

COPY . .
RUN --mount=type=cache,target=/go/pkg/mod \
    --mount=type=cache,target=/root/.cache/go-build \
    CGO_ENABLED=0 GOOS=linux GOARCH=amd64 \
    go build -ldflags="-w -s" -o server ./cmd/server

# Final stage: zero dependencies
FROM scratch
COPY --from=builder /etc/ssl/certs/ca-certificates.crt /etc/ssl/certs/
COPY --from=builder /app/server /server
EXPOSE 8080
ENTRYPOINT ["/server"]
```

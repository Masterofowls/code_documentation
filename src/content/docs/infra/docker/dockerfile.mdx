# Dockerfile

A `Dockerfile` is a text file containing sequential instructions that Docker reads to build an image. Each instruction creates a new read-only **layer** in the image filesystem. Understanding how layers work is the key to writing fast, lean Dockerfiles.

## Dockerfile Instructions Reference

### FROM — Base Image

Every Dockerfile must start with `FROM`. It sets the base image for subsequent instructions.

```dockerfile
# Pinned tag (recommended for reproducibility)
FROM node:20.11.1-alpine3.19

# Digest pinning (strongest — survives tag mutation)
FROM node:20-alpine@sha256:a1b2c3d4...

# Scratch — empty image for compiled binaries (Go, Rust)
FROM scratch

# Named build stage (multi-stage builds)
FROM node:20-alpine AS builder
FROM nginx:alpine AS runtime
```

:::caution[Never use `latest` in production]
`latest` is a moving target. If the base image updates and introduces a breaking change, your build silently breaks. Always pin to a specific tag or digest.
:::

### RUN — Execute Commands

```dockerfile
# Single command (creates one layer)
RUN apt-get update

# Chained commands (best practice — single layer, smaller image)
RUN apt-get update && \
    apt-get install -y --no-install-recommends \
      curl \
      ca-certificates && \
    rm -rf /var/lib/apt/lists/*

# Use heredoc syntax (BuildKit, Docker 1.20+)
RUN <<EOF
apt-get update
apt-get install -y curl
rm -rf /var/lib/apt/lists/*
EOF

# Shell vs exec form
RUN echo hello           # shell form: /bin/sh -c "echo hello"
RUN ["echo", "hello"]    # exec form: no shell, no variable expansion
```

### COPY and ADD

```dockerfile
# COPY — preferred for local files
COPY package.json package-lock.json ./
COPY src/ ./src/
COPY --chown=node:node . .

# COPY from another build stage
COPY --from=builder /app/dist ./dist

# ADD — use only for tar extraction or URL fetching (COPY otherwise)
ADD https://example.com/file.tar.gz /tmp/
ADD archive.tar.gz /app/   # auto-extracts tar
```

:::tip[Prefer COPY over ADD]
`ADD` has implicit behaviours (URL fetching, tar extraction) that make Dockerfiles harder to reason about. Use `COPY` for all local file copies. Use `RUN curl` for URL fetching so intent is explicit.
:::

### WORKDIR — Set Working Directory

```dockerfile
# Creates directory if it doesn't exist
WORKDIR /app

# All subsequent COPY, RUN, CMD paths are relative to WORKDIR
COPY package.json .     # copies to /app/package.json
RUN npm install         # runs in /app

# Avoid using RUN cd /path — use WORKDIR instead
```

### ENV and ARG — Variables

```dockerfile
# ENV — set at build AND runtime (baked into image)
ENV NODE_ENV=production
ENV PORT=3000

# ARG — set at build time only (not in final image)
ARG BUILD_VERSION
ARG REGISTRY=docker.io

# Use ARG in RUN commands
RUN echo "Building version: ${BUILD_VERSION}"

# ARG with default value
ARG NODE_VERSION=20

# Dynamic base image using ARG
ARG BASE_IMAGE=node:20-alpine
FROM ${BASE_IMAGE}
```

:::caution[ARG vs ENV for secrets]
Neither `ARG` nor `ENV` is safe for secrets — both are visible in `docker history`. Use `--secret` mount (BuildKit) for build-time secrets.
:::

### EXPOSE — Document Ports

```dockerfile
# Documents which port the container listens on (informational only)
EXPOSE 3000
EXPOSE 8080/tcp
EXPOSE 53/udp

# Does NOT actually publish the port — use -p flag with docker run
```

### USER — Run as Non-Root

```dockerfile
# Create a non-root user and switch to it
RUN addgroup --system --gid 1001 nodejs && \
    adduser --system --uid 1001 nextjs

USER nextjs

# Or use numeric UID/GID directly
USER 1001:1001
```

### LABEL — Image Metadata

```dockerfile
LABEL maintainer="team@example.com"
LABEL org.opencontainers.image.source="https://github.com/org/repo"
LABEL org.opencontainers.image.version="1.2.3"
LABEL org.opencontainers.image.description="My application"
LABEL build_date="2024-01-15"
```

### HEALTHCHECK — Container Health

```dockerfile
# HTTP health check
HEALTHCHECK --interval=30s \
            --timeout=10s \
            --start-period=40s \
            --retries=3 \
  CMD curl -f http://localhost:3000/health || exit 1

# TCP health check (when curl not available)
HEALTHCHECK --interval=15s --timeout=5s \
  CMD wget --no-verbose --tries=1 --spider http://localhost:3000/health || exit 1

# Disable inherited health check
HEALTHCHECK NONE
```

### ENTRYPOINT and CMD

```dockerfile
# CMD — default arguments, overridable at runtime
CMD ["node", "server.js"]
CMD ["nginx", "-g", "daemon off;"]

# ENTRYPOINT — fixed executable, CMD provides default args
ENTRYPOINT ["node"]
CMD ["server.js"]         # docker run myapp → runs: node server.js
                          # docker run myapp other.js → runs: node other.js

# Shell form (not recommended — can't receive Unix signals properly)
ENTRYPOINT node server.js

# Exec form (recommended)
ENTRYPOINT ["node", "server.js"]

# Using tini as PID 1 (handles zombie processes and signal forwarding)
RUN apk add --no-cache tini
ENTRYPOINT ["/sbin/tini", "--"]
CMD ["node", "server.js"]
```

### VOLUME, STOPSIGNAL, ONBUILD, SHELL

```dockerfile
# VOLUME — creates a mount point (better to declare in compose/run)
VOLUME ["/data"]
VOLUME /var/log/app

# STOPSIGNAL — signal sent to container on docker stop
STOPSIGNAL SIGTERM
STOPSIGNAL SIGQUIT

# ONBUILD — triggers in child images
ONBUILD COPY package.json .
ONBUILD RUN npm install

# SHELL — change default shell for RUN commands
SHELL ["/bin/bash", "-o", "pipefail", "-c"]
RUN set -eux; echo "now using bash"
```

## Layer Caching

Docker caches each layer. If an instruction and all layers before it are unchanged, Docker uses the cached layer — dramatically speeding up builds. **Order your instructions from least to most frequently changing.**

```dockerfile
# BAD: source code changes invalidate npm install cache
FROM node:20-alpine
WORKDIR /app
COPY . .                    # ← changes every time
RUN npm ci                  # ← re-runs every time (slow!)
CMD ["node", "server.js"]

# GOOD: copy package files first, install, then copy source
FROM node:20-alpine
WORKDIR /app
COPY package.json package-lock.json ./  # ← only changes when deps change
RUN npm ci                               # ← cached when deps unchanged
COPY . .                                 # ← source changes don't bust install cache
CMD ["node", "server.js"]
```

:::tip[Cache-busting pattern]
Always copy dependency manifest files (`package.json`, `requirements.txt`, `go.mod`, `Gemfile`) before copying source code. The `npm install` / `pip install` / `go mod download` layer will only re-run when those files change.
:::

## Multi-Stage Builds

Multi-stage builds let you use one image for building and a smaller image for the final artifact — eliminating build tools, source code, and test dependencies from the production image.

```dockerfile
# ─── Stage 1: Build ────────────────────────────────────────────
FROM node:20-alpine AS builder

WORKDIR /app

COPY package.json package-lock.json ./
RUN npm ci

COPY . .
RUN npm run build

# ─── Stage 2: Production ───────────────────────────────────────
FROM node:20-alpine AS runtime

# Install only production deps
WORKDIR /app
COPY package.json package-lock.json ./
RUN npm ci --only=production && npm cache clean --force

# Copy built artifacts from builder stage
COPY --from=builder /app/dist ./dist

# Non-root user
RUN addgroup --system --gid 1001 nodejs && \
    adduser --system --uid 1001 nodeuser
USER nodeuser

EXPOSE 3000
HEALTHCHECK --interval=30s --timeout=5s --retries=3 \
  CMD wget -qO- http://localhost:3000/health || exit 1

CMD ["node", "dist/server.js"]
```

### Building Specific Stages

```bash
# Build only to the "builder" stage (useful for testing)
docker build --target builder -t myapp:builder .

# Build the final (default) stage
docker build -t myapp:latest .
```

## .dockerignore

Create a `.dockerignore` file in your build context to exclude files. Smaller build context = faster builds and smaller images.

```
# .dockerignore
node_modules/
.git/
.gitignore
*.md
.env
.env.*
dist/
build/
coverage/
.nyc_output/
*.log
.DS_Store
Thumbs.db
.vscode/
.idea/
**/__pycache__/
**/*.pyc
**/*.pyo
*.test.ts
*.spec.ts
```

## Base Image Selection

| Base                         | Size   | Use Case                                             |
| ---------------------------- | ------ | ---------------------------------------------------- |
| `ubuntu:22.04`               | ~77MB  | Full environment, max compatibility                  |
| `debian:bookworm-slim`       | ~74MB  | Lighter Debian, apt available                        |
| `alpine:3.19`                | ~7MB   | Minimal, musl libc (may have compat issues)          |
| `node:20-slim`               | ~220MB | Slim Debian, Node preinstalled                       |
| `node:20-alpine`             | ~50MB  | Alpine + Node (very common)                          |
| `gcr.io/distroless/nodejs20` | ~115MB | No shell, no package manager, minimal attack surface |
| `scratch`                    | 0MB    | Empty — for statically compiled binaries only        |

:::tip[Distroless for Production]
Google's distroless images contain only the application and runtime, with no shell or package manager. This minimises the attack surface significantly. Use them in the final stage of multi-stage builds.

```dockerfile
FROM gcr.io/distroless/nodejs20-debian12
COPY --from=builder /app/dist /app
CMD ["/app/server.js"]
```

:::

## Build Secrets (BuildKit)

Never put secrets in `ARG` or `ENV` — they're visible in `docker history`. Use BuildKit's `--secret` flag:

```dockerfile
# syntax=docker/dockerfile:1
FROM python:3.12-slim

# Mount secret at build time — NOT baked into the layer
RUN --mount=type=secret,id=pip_token \
    pip install \
      --extra-index-url https://$(cat /run/secrets/pip_token)@pypi.example.com/simple \
      private-package
```

```bash
# Pass secret at build time
docker build \
  --secret id=pip_token,src=./pip_token.txt \
  -t myapp .
```

### SSH Agent Forwarding

```dockerfile
# syntax=docker/dockerfile:1
FROM python:3.12-slim

RUN --mount=type=ssh \
    pip install git+ssh://git@github.com/org/private-repo.git
```

```bash
docker build --ssh default -t myapp .
```

## Using BuildKit and Buildx

```bash
# Enable BuildKit (Docker 23.0+ enables by default)
DOCKER_BUILDKIT=1 docker build -t myapp .

# Or set in Docker config
echo '{"features":{"buildkit":true}}' > ~/.docker/daemon.json

# buildx — extended build capabilities
docker buildx build -t myapp .

# Build with cache export (speeds up CI)
docker buildx build \
  --cache-from type=registry,ref=myregistry/myapp:cache \
  --cache-to type=registry,ref=myregistry/myapp:cache,mode=max \
  -t myapp .
```

## Linting with hadolint

```bash
# Install hadolint
brew install hadolint          # macOS
winget install hadolint        # Windows

# Lint a Dockerfile
hadolint Dockerfile

# Lint in CI (exit code 1 on warnings)

# Ignore specific rules

```

## Non-Root User and Security

```dockerfile
# Alpine
RUN addgroup -S appgroup && adduser -S appuser -G appgroup
USER appuser

# Debian/Ubuntu
RUN groupadd --system --gid 1001 appgroup && \
    useradd --system --uid 1001 --gid appgroup --no-create-home appuser
USER 1001:1001
```

:::caution[Root containers are dangerous]
A process escaping a root container has root access on the host. Always drop to a non-root user before your CMD/ENTRYPOINT. Many Kubernetes security policies (PodSecurityAdmission) will reject root containers.
:::

## Using tini as PID 1

When Docker starts a container, the `ENTRYPOINT`/`CMD` process runs as **PID 1**. PID 1 is responsible for reaping zombie child processes and forwarding Unix signals. Node.js, Python, and most apps aren't designed to do this correctly.

`tini` is a minimal init system that handles these responsibilities:

```dockerfile
# Alpine
RUN apk add --no-cache tini
ENTRYPOINT ["/sbin/tini", "--"]
CMD ["node", "server.js"]

# Or use Docker's built-in init
# docker run --init myapp
```

# Dockerfile

A `Dockerfile` is a text file containing sequential instructions that Docker executes to build an image. Understanding layer caching, multi-stage builds, and security practices is essential for producing small, fast, and secure images.

## Dockerfile Instructions Reference

### FROM

The first instruction in every Dockerfile. Specifies the base image.

```dockerfile
# Use a specific tag — never `latest` in production
FROM node:22-alpine3.20

# Multi-stage: name the stage
FROM node:22-alpine3.20 AS builder

# Scratch — empty base for compiled binaries
FROM scratch
```

### RUN

Executes commands in a new layer. Combine related commands with `&&` to minimise layers.

```dockerfile
# Bad — three separate layers
RUN apt-get update
RUN apt-get install -y curl
RUN rm -rf /var/lib/apt/lists/*

# Good — single layer, clean cache in same step
RUN apt-get update && \
    apt-get install -y --no-install-recommends \
      curl \
      ca-certificates && \
    rm -rf /var/lib/apt/lists/*
```

### COPY vs ADD

Prefer `COPY` for copying files. Use `ADD` only for its special features (URL fetch, tar auto-extraction).

```dockerfile
# COPY — simple and explicit
COPY package.json package-lock.json ./
COPY --chown=node:node src/ ./src/

# ADD — auto-extracts tar archives
ADD archive.tar.gz /app/

# ADD with URL (prefer curl/wget in RUN instead)
ADD https://example.com/config.json /etc/app/
```

### WORKDIR

Sets the working directory for subsequent instructions. Creates the directory if it does not exist.

```dockerfile
WORKDIR /app

# Equivalent to: RUN mkdir -p /app && cd /app
# Prefer WORKDIR over RUN cd
```

### ENV and ARG

```dockerfile
# ENV — available at build time AND runtime
ENV NODE_ENV=production
ENV PORT=3000 HOST=0.0.0.0

# ARG — available only at build time
ARG VERSION=1.0.0
ARG BUILD_DATE

# Use ARG to parameterise the base image tag
ARG NODE_VERSION=22-alpine3.20
FROM node:${NODE_VERSION}
```

:::tip[Build args vs env vars]
Use `ARG` for values that differ between builds (version numbers, registry URLs). Use `ENV` for values the application needs at runtime. Never put secrets in either — they appear in `docker history`.
:::

### EXPOSE

Documents which ports the container listens on. Does not actually publish the port.

```dockerfile
EXPOSE 3000
EXPOSE 8080/tcp 8081/udp
```

### VOLUME

Declares a mount point. The directory is managed as a Docker volume if no mount is specified.

```dockerfile
VOLUME ["/var/lib/postgresql/data"]
VOLUME /app/uploads
```

### USER

Switch to a non-root user. Always do this before the final `CMD` or `ENTRYPOINT`.

```dockerfile
# Create a dedicated user with no home directory and no shell
RUN addgroup -S appgroup && adduser -S appuser -G appgroup
USER appuser
```

### LABEL

Adds metadata to the image. Useful for CI traceability and tooling.

```dockerfile
LABEL org.opencontainers.image.title="My App" \
      org.opencontainers.image.version="1.2.3" \
      org.opencontainers.image.source="https://github.com/org/repo" \
      org.opencontainers.image.created="2024-01-15T12:00:00Z"
```

### HEALTHCHECK

Tells Docker how to test whether the container is healthy.

```dockerfile
HEALTHCHECK --interval=30s --timeout=5s --start-period=10s --retries=3 \
  CMD curl -f http://localhost:3000/health || exit 1

# Disable inherited healthcheck
HEALTHCHECK NONE
```

### ENTRYPOINT and CMD

```dockerfile
# ENTRYPOINT — the executable that always runs
ENTRYPOINT ["node", "dist/server.js"]

# CMD — default arguments, overridable at `docker run`
CMD ["--port", "3000"]

# Together: `docker run myapp --port 8080` overrides CMD
# ENTRYPOINT stays fixed, CMD becomes the argument

# Shell form — runs via /bin/sh -c, loses signals
ENTRYPOINT node dist/server.js   # avoid in production

# Exec form — direct exec, receives signals properly
ENTRYPOINT ["node", "dist/server.js"]   # preferred
```

### ONBUILD, STOPSIGNAL, SHELL

```dockerfile
# ONBUILD — trigger instructions when image is used as a base
ONBUILD COPY package.json .
ONBUILD RUN npm ci

# STOPSIGNAL — signal sent to stop the container (default SIGTERM)
STOPSIGNAL SIGQUIT

# SHELL — override default shell (/bin/sh -c on Linux)
SHELL ["/bin/bash", "-o", "pipefail", "-c"]
```

## Layer Caching

Docker caches each layer. A layer is only rebuilt when its instruction changes **or a previous layer changes**. Order your instructions to maximise cache hits.

```dockerfile
# Bad — source copy invalidates npm install cache on every code change
FROM node:22-alpine
WORKDIR /app
COPY . .            # any code change busts all subsequent cache
RUN npm ci
CMD ["node", "index.js"]

# Good — copy dependency manifests first
FROM node:22-alpine
WORKDIR /app
COPY package.json package-lock.json ./   # cached until deps change
RUN npm ci --omit=dev                    # only re-runs when deps change
COPY . .                                 # code changes don't bust npm ci
CMD ["node", "index.js"]
```

:::tip[Cache busting strategy]
Files that change infrequently (`package.json`, `requirements.txt`, `go.mod`) should be copied before files that change often (application source code).
:::

## Multi-Stage Builds

Multi-stage builds let you use a fat build image and copy only the artefacts into a lean runtime image.

```dockerfile
# Stage 1: Build
FROM node:22-alpine AS builder
WORKDIR /app
COPY package.json package-lock.json ./
RUN npm ci
COPY . .
RUN npm run build

# Stage 2: Runtime — only the compiled output
FROM node:22-alpine AS runtime
WORKDIR /app
RUN addgroup -S app && adduser -S app -G app
COPY --chown=app:app --from=builder /app/dist ./dist
COPY --chown=app:app --from=builder /app/node_modules ./node_modules
USER app
EXPOSE 3000
CMD ["node", "dist/server.js"]
```

### Building a Specific Stage

```bash
# Build only the builder stage (useful for CI caching)
docker build --target builder -t myapp:builder .

# Build the final runtime stage
docker build -t myapp:latest .
```

### COPY --from

```dockerfile
# Copy from a named stage
COPY --from=builder /app/dist ./dist

# Copy from an external image
COPY --from=nginx:alpine /etc/nginx/nginx.conf /etc/nginx/nginx.conf

# Copy specific file with permissions
COPY --from=builder --chown=app:app /app/dist/server ./server
```

## .dockerignore

The `.dockerignore` file prevents files from being sent to the build context, speeding up builds and avoiding accidental inclusion of secrets.

```bash
# .dockerignore
node_modules/
dist/
.git/
.github/
*.log
.env
.env.*
!.env.example
coverage/
.nyc_output/
*.test.ts
*.spec.ts
README.md
docker-compose*.yml
.DS_Store
Thumbs.db
```

:::caution[Never include secrets in the build context]
Even if you don't `COPY` a file, it exists in the build context tarball sent to the daemon. Always add `.env` and credential files to `.dockerignore`.
:::

## Base Image Selection

| Base | Size | Use case |
|---|---|---|
| `ubuntu:24.04` | ~78MB | Full OS utilities, debugging |
| `debian:bookworm-slim` | ~75MB | Smaller Debian, most packages available |
| `alpine:3.20` | ~7MB | Minimal, uses musl libc |
| `node:22-slim` | ~75MB | Node on Debian slim |
| `node:22-alpine` | ~45MB | Node on Alpine |
| `gcr.io/distroless/nodejs22` | ~60MB | No shell, no package manager — minimal attack surface |
| `scratch` | 0MB | For statically compiled binaries only |

:::tip[Distroless for production]
Google's distroless images contain only the runtime and its dependencies. No shell, no package manager, no debugging tools. This significantly reduces the attack surface. Use a regular image for the build stage and distroless for the final stage.
:::

## Using tini as PID 1

Linux containers need a proper init process to handle zombie reaping and signal forwarding. `tini` is a minimal init designed for containers.

```dockerfile
FROM node:22-alpine
# Install tini
RUN apk add --no-cache tini
# ...
ENTRYPOINT ["/sbin/tini", "--"]
CMD ["node", "dist/server.js"]
```

Alternatively, Node.js 20+ and Docker's `--init` flag solve this without modifying the Dockerfile:

```bash
docker run --init myapp
```

## Build Secrets (BuildKit)

Never put secrets in build args or environment variables — they appear in image history. Use BuildKit's secret mount instead.

```dockerfile
# syntax=docker/dockerfile:1
FROM node:22-alpine AS builder
WORKDIR /app
COPY package.json package-lock.json ./
# Mount a secret at build time — not stored in any layer
RUN --mount=type=secret,id=npmrc,target=/root/.npmrc \
    npm ci
```

```bash
# Pass the secret file at build time
docker build --secret id=npmrc,src=$HOME/.npmrc -t myapp .
```

## hadolint — Dockerfile Linting

```bash
# Install
brew install hadolint              # macOS
docker pull hadolint/hadolint      # any OS

# Lint a Dockerfile
hadolint Dockerfile

# Use in CI
docker run --rm -i hadolint/hadolint < Dockerfile
```

Common hadolint rules:
- `DL3008` — pin apt-get package versions
- `DL3018` — pin apk package versions
- `DL3009` — delete apt-get lists after installing
- `DL4006` — set SHELL options when using pipes

## BuildKit and buildx

Enable BuildKit for faster builds, better caching, and secrets support:

```bash
# BuildKit is enabled by default in Docker 23+
# For older versions:
export DOCKER_BUILDKIT=1

# Build with buildx (supports multi-arch)
docker buildx build -t myapp:latest .

# Inline cache for CI
docker buildx build \
  --cache-from type=registry,ref=myregistry/myapp:cache \
  --cache-to type=registry,ref=myregistry/myapp:cache,mode=max \
  -t myregistry/myapp:latest \
  --push .
```

## Summary Checklist

- [ ] Pin base image to a specific tag (never `latest`)
- [ ] Use multi-stage builds to keep the final image lean
- [ ] Copy dependency manifests before source for cache efficiency
- [ ] Clean package manager caches in the same `RUN` layer
- [ ] Add a `.dockerignore` file
- [ ] Run the process as a non-root user
- [ ] Use exec form for `ENTRYPOINT` and `CMD`
- [ ] Add a `HEALTHCHECK`
- [ ] Use `tini` or `--init` for signal handling
- [ ] Use BuildKit secrets for sensitive build-time values
- [ ] Lint with `hadolint` in CI

# Dockerfile Reference

A `Dockerfile` is a text file containing instructions that Docker executes sequentially to build an image. Each instruction creates a new layer in the image filesystem. Understanding how layers work is essential for writing fast, small, and secure images.

## Dockerfile Instructions

### FROM

Every Dockerfile must start with `FROM`. It sets the base image.

```dockerfile
# Use a specific version tag — never use :latest in production
FROM node:22-alpine

# Multi-stage: name the stage for reference later
FROM node:22-alpine AS builder

# Scratch — empty filesystem, for fully static binaries
FROM scratch
```

### RUN

Executes commands during the image build. Each `RUN` creates a new layer.

```dockerfile
# Combine commands to reduce layers
RUN apt-get update && \
    apt-get install -y --no-install-recommends \
        curl \
        ca-certificates && \
    rm -rf /var/lib/apt/lists/*

# Alpine package manager
RUN apk add --no-cache curl ca-certificates

# Use shell form (runs via /bin/sh -c)
RUN echo "hello"

# Use exec form (no shell interpolation — preferred for commands)
RUN ["npm", "ci", "--omit=dev"]
```

:::tip[Clean Up in the Same RUN]
Always delete package manager caches in the **same** `RUN` instruction that installs packages. If you run `rm` in a separate layer, the cache data still exists in the earlier layer and contributes to the image size.
:::

### COPY and ADD

```dockerfile
# COPY is preferred for most cases — it is transparent
COPY package.json package-lock.json ./
COPY src/ ./src/

# COPY with owner flag (BuildKit feature)
COPY --chown=node:node src/ ./src/

# ADD can auto-extract archives and fetch URLs — use sparingly
ADD https://example.com/file.tar.gz /tmp/
ADD app.tar.gz /app/
```

:::caution[Prefer COPY over ADD]
`ADD` has implicit behaviour (URL fetching, auto-extraction) that makes Dockerfiles harder to reason about. Use `COPY` for local files and `RUN curl` or `RUN wget` for remote files, so the intent is explicit.
:::

### WORKDIR

Sets the working directory for subsequent instructions. Creates the directory if it does not exist.

```dockerfile
WORKDIR /app

# Equivalent to: RUN mkdir -p /app && cd /app
# Prefer WORKDIR over RUN cd — it is cleaner and affects all subsequent instructions
```

### ENV and ARG

```dockerfile
# ENV — available at build time and in the running container
ENV NODE_ENV=production \
    PORT=3000 \
    LOG_LEVEL=info

# ARG — only available at build time
ARG NODE_VERSION=22
FROM node:${NODE_VERSION}-alpine

ARG BUILD_DATE
ARG GIT_COMMIT
LABEL build.date="${BUILD_DATE}" build.commit="${GIT_COMMIT}"

# Pass ARG at build time
# docker build --build-arg NODE_VERSION=22 .
```

:::caution[Secrets in ENV]
`ENV` values are embedded in the image and visible via `docker inspect`. Never use `ENV` for passwords, API keys, or tokens. Use build secrets (shown below) or runtime environment injection instead.
:::

### EXPOSE

Documents which ports the container listens on. It does **not** actually publish ports — that is done with `-p` at runtime.

```dockerfile
EXPOSE 3000
EXPOSE 5432
EXPOSE 80 443
```

### USER

Switches the user for subsequent `RUN`, `CMD`, and `ENTRYPOINT` instructions.

```dockerfile
# Create a non-root user and switch to it
RUN addgroup --system --gid 1001 nodejs && \
    adduser --system --uid 1001 --ingroup nodejs nextjs

USER nextjs
```

### LABEL

Adds metadata to the image. Useful for tooling, documentation, and filtering.

```dockerfile
LABEL org.opencontainers.image.title="My App" \
      org.opencontainers.image.version="1.2.3" \
      org.opencontainers.image.source="https://github.com/myorg/myapp" \
      org.opencontainers.image.licenses="MIT" \
      maintainer="team@example.com"
```

### HEALTHCHECK

Defines a command Docker uses to check if the container is healthy.

```dockerfile
HEALTHCHECK --interval=30s --timeout=10s --start-period=15s --retries=3 \
  CMD curl -f http://localhost:3000/health || exit 1

# For images without curl, use wget
HEALTHCHECK --interval=30s --timeout=5s --retries=3 \
  CMD wget --no-verbose --tries=1 --spider http://localhost:3000/health || exit 1

# Disable health check (override base image's check)
HEALTHCHECK NONE
```

### ENTRYPOINT and CMD

```dockerfile
# CMD — default command, easily overridden at runtime
CMD ["node", "server.js"]

# ENTRYPOINT — always executed, arguments are appended
ENTRYPOINT ["docker-entrypoint.sh"]
CMD ["postgres"]  # default argument to entrypoint

# Exec form (preferred) vs shell form
ENTRYPOINT ["npm", "start"]        # exec form — PID 1 is npm
ENTRYPOINT npm start               # shell form — PID 1 is /bin/sh

# Combined: entrypoint is the executable, CMD provides default args
ENTRYPOINT ["node"]
CMD ["server.js"]
# docker run myapp             → node server.js
# docker run myapp worker.js   → node worker.js
```

### VOLUME

Declares a mount point. Creates an anonymous volume if no volume is provided at runtime.

```dockerfile
VOLUME ["/var/lib/postgresql/data"]
VOLUME /app/uploads
```

### STOPSIGNAL and SHELL

```dockerfile
# Change the signal sent to stop the container (default: SIGTERM)
STOPSIGNAL SIGQUIT

# Change the default shell for RUN instructions
SHELL ["/bin/bash", "-o", "pipefail", "-c"]
```

### ONBUILD

Triggers instructions in child images that use this image as a base.

```dockerfile
# In a base image
ONBUILD COPY package.json ./
ONBUILD RUN npm ci
```

## Layer Caching

Docker caches each layer. If a layer's instruction and its context haven't changed, Docker reuses the cached layer. **Layer cache invalidation propagates downward** — once a layer changes, all subsequent layers are rebuilt.

The golden rule: **copy dependency manifests before source code**.

```dockerfile
# ❌ Bad — every code change invalidates the npm ci cache
COPY . .
RUN npm ci

# ✅ Good — npm ci only reruns when package files change
COPY package.json package-lock.json ./
RUN npm ci
COPY . .
```

```dockerfile
# Python example — same principle
COPY requirements.txt ./
RUN pip install --no-cache-dir -r requirements.txt
COPY src/ ./src/
```

```dockerfile
# Go example — module downloads cached separately from compilation
COPY go.mod go.sum ./
RUN go mod download
COPY . .
RUN go build -o /app/server ./cmd/server
```

## Multi-Stage Builds

Multi-stage builds use multiple `FROM` instructions in a single Dockerfile. Earlier stages compile or build; later stages copy only the artifacts needed to run. The final image contains no compiler, build tools, or intermediate files.

```dockerfile
# ─── Stage 1: Build ───────────────────────────────────────
FROM node:22-alpine AS builder
WORKDIR /build

COPY package.json package-lock.json ./
RUN npm ci

COPY . .
RUN npm run build

# ─── Stage 2: Runtime ─────────────────────────────────────
FROM node:22-alpine AS runtime
WORKDIR /app

# Copy only the built artifacts from the builder stage
COPY --from=builder /build/dist ./dist
COPY --from=builder /build/node_modules ./node_modules
COPY --from=builder /build/package.json ./

RUN addgroup -S nodejs && adduser -S -G nodejs nodejs
USER nodejs

EXPOSE 3000
CMD ["node", "dist/server.js"]
```

Build a specific stage with `--target`:

```bash
docker build --target builder -t myapp:builder .
docker build --target runtime -t myapp:latest .
```

### Go Multi-Stage Example

```dockerfile
FROM golang:1.23-alpine AS builder
WORKDIR /build
COPY go.mod go.sum ./
RUN go mod download
COPY . .
RUN CGO_ENABLED=0 GOOS=linux go build -ldflags="-s -w" -o server ./cmd/server

# Final image — distroless has no shell, no package manager
FROM gcr.io/distroless/static:nonroot
COPY --from=builder /build/server /server
USER nonroot:nonroot
EXPOSE 8080
ENTRYPOINT ["/server"]
```

### Python Multi-Stage Example

```dockerfile
FROM python:3.12-slim AS builder
WORKDIR /app
RUN pip install --no-cache-dir uv
COPY pyproject.toml uv.lock ./
RUN uv export --frozen --no-dev -o requirements.txt && \
    pip install --no-cache-dir --prefix=/install -r requirements.txt

FROM python:3.12-slim AS runtime
WORKDIR /app
COPY --from=builder /install /usr/local
COPY src/ ./src/

RUN useradd --system --no-create-home appuser
USER appuser

EXPOSE 8000
CMD ["python", "-m", "uvicorn", "src.main:app", "--host", "0.0.0.0", "--port", "8000"]
```

## .dockerignore

A `.dockerignore` file prevents files from being sent to the build context (and thus `COPY`-ed into the image). It uses the same syntax as `.gitignore`.

```
# Dependencies
node_modules/
vendor/
.venv/
__pycache__/
*.pyc

# Build outputs
dist/
build/
target/
*.o
*.a

# Version control
.git/
.gitignore

# Development config
.env
.env.local
*.env

# IDE files
.idea/
.vscode/
*.swp

# Documentation
docs/
*.md
!README.md

# Test files
**/*.test.ts
**/*.spec.ts
coverage/

# Logs
*.log
logs/

# Docker files (not needed in image)
Dockerfile*
docker-compose*.yml
.dockerignore
```

:::tip[Always Create .dockerignore]
Without `.dockerignore`, every file in the project directory is sent to the daemon as the build context. This includes `node_modules/` (potentially GBs) and `.git/` (your entire commit history). A good `.dockerignore` file dramatically speeds up builds.
:::

## Base Image Selection

| Base Image | Size | Shell | Package Manager | Use Case |
|---|---|---|---|---|
| `ubuntu:24.04` | ~77 MB | bash | apt | Maximum compatibility |
| `debian:bookworm-slim` | ~74 MB | bash | apt | Slim Debian with glibc |
| `alpine:3.20` | ~7 MB | ash | apk | Minimal, musl libc |
| `node:22-alpine` | ~60 MB | ash | apk | Node.js on Alpine |
| `node:22-slim` | ~80 MB | bash | apt | Node.js on Debian slim |
| `gcr.io/distroless/nodejs22` | ~50 MB | none | none | No shell attack surface |
| `scratch` | 0 MB | none | none | Fully static binaries only |

:::tip[Alpine Caveats]
Alpine uses `musl` libc instead of `glibc`. Most software works fine, but some native Node.js modules, Python packages with C extensions, and certain binaries may fail or behave differently. Test Alpine images thoroughly before adopting them.
:::

## Non-Root User

Running as root inside a container is a security risk. If an attacker exploits a vulnerability in your application, they gain root access inside the container — and potentially on the host if the container runtime is misconfigured.

```dockerfile
# Alpine / minimal distros
RUN addgroup -S -g 1001 appgroup && \
    adduser -S -u 1001 -G appgroup appuser

# Debian/Ubuntu
RUN groupadd --gid 1001 appgroup && \
    useradd --uid 1001 --gid appgroup --shell /bin/bash --create-home appuser

# Set ownership and switch
COPY --chown=appuser:appgroup . .
USER appuser
```

## tini as PID 1

By default, your application becomes PID 1 inside the container. PID 1 has special responsibilities: it must reap zombie processes and forward signals to children. Most applications don't handle this correctly.

[tini](https://github.com/krallin/tini) is a minimal init process designed specifically for containers:

```dockerfile
# Alpine has tini in its package repositories
RUN apk add --no-cache tini
ENTRYPOINT ["/sbin/tini", "--"]
CMD ["node", "server.js"]
```

Alternatively, use Docker's built-in init:

```bash
docker run --init myapp
```

## Build Secrets (BuildKit)

Build secrets allow you to pass sensitive data (SSH keys, tokens, credentials) to the build without embedding them in any image layer.

```dockerfile
# syntax=docker/dockerfile:1
FROM node:22-alpine AS builder
WORKDIR /app
COPY package.json package-lock.json ./

# Mount a secret — available only during this RUN, never in any layer
RUN --mount=type=secret,id=npmrc,target=/root/.npmrc \
    npm ci

COPY . .
RUN npm run build
```

Pass the secret at build time:

```bash
docker build --secret id=npmrc,src=$HOME/.npmrc -t myapp .
```

For SSH keys (private Git repos):

```dockerfile
RUN --mount=type=ssh git clone git@github.com:myorg/private-pkg.git /tmp/pkg
```

```bash
docker build --ssh default -t myapp .
```

## BuildKit and buildx

BuildKit is Docker's modern build engine with improved performance, build secrets, and advanced caching.

```bash
# Enable BuildKit for a single build
DOCKER_BUILDKIT=1 docker build -t myapp .

# Set BuildKit as the default in daemon.json
# { "features": { "buildkit": true } }

# buildx — extended build capabilities (multi-arch, cache export)
docker buildx build \
  --platform linux/amd64,linux/arm64 \
  --cache-from type=registry,ref=myregistry/myapp:cache \
  --cache-to type=registry,ref=myregistry/myapp:cache,mode=max \
  -t myregistry/myapp:latest \
  --push .
```

## Linting with hadolint

[hadolint](https://github.com/hadolint/hadolint) is a Dockerfile linter that catches common mistakes.

```bash
# Install on macOS
brew install hadolint

# Lint a Dockerfile
hadolint Dockerfile

# Ignore a specific rule
hadolint --ignore DL3008 Dockerfile

# Output as JSON for CI
hadolint -f json Dockerfile
```

Common rules:
- `DL3008` — Pin versions in `apt-get install`
- `DL3018` — Pin versions in `apk add`
- `DL3025` — Use exec form for CMD/ENTRYPOINT
- `SC2086` — Quote variables in shell commands

## Complete Node.js Production Dockerfile

```dockerfile
# syntax=docker/dockerfile:1
FROM node:22-alpine AS base
WORKDIR /app
RUN apk add --no-cache tini

# ── Dependency stage ──────────────────────────────────────
FROM base AS deps
COPY package.json package-lock.json ./
RUN npm ci --omit=dev

# ── Build stage ───────────────────────────────────────────
FROM base AS builder
COPY package.json package-lock.json ./
RUN npm ci
COPY . .
RUN npm run build

# ── Runtime stage ─────────────────────────────────────────
FROM base AS runtime

RUN addgroup --system --gid 1001 nodejs && \
    adduser --system --uid 1001 nodejs

COPY --from=deps --chown=nodejs:nodejs /app/node_modules ./node_modules
COPY --from=builder --chown=nodejs:nodejs /app/dist ./dist
COPY --chown=nodejs:nodejs package.json ./

USER nodejs

EXPOSE 3000
HEALTHCHECK --interval=30s --timeout=5s --start-period=10s --retries=3 \
  CMD wget -qO- http://localhost:3000/health || exit 1

ENTRYPOINT ["/sbin/tini", "--"]
CMD ["node", "dist/server.js"]
```

Build and run:

```bash
docker build -t myapp:latest .
docker run -d -p 3000:3000 --name myapp myapp:latest
```

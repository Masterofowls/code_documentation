# Docker

Docker is an open-source platform for developing, shipping, and running applications in **containers** — lightweight, portable, self-sufficient units that package an application's code alongside all its dependencies, libraries, and configuration. Write once, run anywhere.

---

## Containers vs Virtual Machines

Both containers and VMs provide isolation, but they achieve it at different layers of the stack.

| Feature | Containers | Virtual Machines |
|---|---|---|
| **Isolation** | Process-level (shared OS kernel) | Full OS-level (hypervisor) |
| **Startup time** | Milliseconds | Seconds to minutes |
| **Image size** | Megabytes | Gigabytes |
| **Memory overhead** | Minimal | High (full OS per VM) |
| **Portability** | Excellent | Good |
| **Security isolation** | Good (namespaces + cgroups) | Excellent (hardware boundary) |
| **Use case** | Microservices, CI/CD, dev environments | Legacy apps, full OS requirements |
| **Density** | Hundreds per host | Tens per host |
| **Persistent storage** | Volumes / bind mounts | Virtual disks |
| **Networking** | Virtual Ethernet, overlay networks | Full virtual NIC |

:::tip[Best of both worlds]
Many production environments run containers **inside** VMs — you get the security isolation of VMs with the density and speed of containers.
:::

## Docker Architecture

Docker uses a **client-server architecture**. The Docker client talks to the Docker daemon, which does the heavy lifting of building, running, and distributing containers.

```
┌─────────────────────────────────────────────────────────────────┐
│                        Docker Host                              │
│                                                                 │
│  ┌──────────────┐     REST API / Unix socket                   │
│  │ Docker Client│ ──────────────────────────────►  Docker      │
│  │  (docker CLI)│                                  Daemon      │
│  └──────────────┘                               (dockerd)      │
│                                                      │          │
│                         ┌────────────────────────────┤          │
│                         ▼          ▼          ▼       │          │
│                    ┌─────────┐ ┌──────┐ ┌────────┐   │          │
│                    │ Images  │ │ Nets │ │ Volumes│   │          │
│                    └────┬────┘ └──────┘ └────────┘   │          │
│                         │                             │          │
│              ┌──────────▼──────────┐                  │          │
│              │     Containers      │                  │          │
│              │  (running images)   │                  │          │
│              └─────────────────────┘                  │          │
└─────────────────────────────────────────────────────────────────┘
                              │
                              ▼
                    ┌──────────────────┐
                    │  Docker Registry │
                    │  (Docker Hub,    │
                    │   GHCR, ECR...)  │
                    └──────────────────┘
```

### Core Components

**Docker Daemon (`dockerd`)**
The long-running background service that manages Docker objects (images, containers, networks, volumes). It listens for Docker API requests and delegates container lifecycle to `containerd`.

**Docker Client (`docker`)**
The primary way users interact with Docker. When you run `docker run`, the client sends the command to `dockerd` via the Docker API (Unix socket `/var/run/docker.sock` or TCP).

**Docker Registry**
A storage and distribution system for Docker images. Docker Hub (`hub.docker.com`) is the default public registry. You can run private registries and use cloud-hosted options like GHCR, ECR, and GAR.

**Images**
Read-only templates built from a `Dockerfile`. Each instruction in a Dockerfile creates a new layer. Layers are shared and cached, making images efficient.

**Containers**
A runnable instance of an image. You can create, start, stop, move, or delete a container using the Docker API or CLI. Containers are isolated from other containers and the host by default.

**Volumes**
Persistent storage that exists independently of container lifecycle. Named volumes are managed by Docker; bind mounts map host paths into containers.

**Networks**
Virtual networks that allow containers to communicate. Docker provides several network drivers: bridge (default), host, none, overlay, and macvlan.

## containerd and the OCI Standard

Docker doesn't manage containers directly — it delegates to **containerd**, a CNCF-graduated container runtime.

```
docker CLI
    │
    ▼
dockerd (Docker Daemon)
    │
    ▼
containerd
    │
    ▼
runc (OCI runtime)
    │
    ▼
Linux namespaces + cgroups
```

**OCI (Open Container Initiative)** defines two open standards:
- **Image Spec** — how container images are structured and stored
- **Runtime Spec** — how containers are created and run

Because Docker images conform to the OCI Image Spec, they can run on any OCI-compliant runtime — not just Docker.

## Docker Desktop vs Docker Engine

| | Docker Desktop | Docker Engine |
|---|---|---|
| **Platform** | macOS, Windows, Linux | Linux only |
| **GUI** | Yes (dashboard, extensions) | No |
| **Docker Compose** | Bundled | Separate install |
| **Kubernetes** | Built-in single-node cluster | Manual setup |
| **License** | Free for personal/small biz; paid for large enterprise | Apache 2.0 (free) |
| **VM layer** | Runs in Linux VM (HyperKit/WSL2/VZ) | Runs natively |
| **Dev experience** | Easiest on macOS/Windows | Best on Linux |

:::caution[Docker Desktop License]
Since 2022, Docker Desktop requires a paid subscription for companies with more than 250 employees or over $10 million in annual revenue. Docker Engine remains free and open source under Apache 2.0.
:::

## Docker vs Podman

**Podman** is a daemonless, rootless container engine from Red Hat that is largely CLI-compatible with Docker.

| Feature | Docker | Podman |
|---|---|---|
| **Daemon** | Requires `dockerd` | Daemonless (fork-exec model) |
| **Root requirement** | Historically required root | Rootless by default |
| **Compose support** | docker compose v2 | podman-compose (third-party) |
| **Pod support** | Via Kubernetes/Compose | Native pod concept |
| **systemd integration** | Extra configuration | Native `podman generate systemd` |
| **CLI compatibility** | `docker` | `podman` (alias-compatible) |
| **OCI compliant** | Yes | Yes |

Most `docker` commands work identically with `podman` — you can often alias `docker=podman`.

## Use Cases

Docker excels at the following scenarios:

- **Microservices** — Package each service with its exact dependencies; deploy, scale, and update independently
- **CI/CD pipelines** — Reproducible build and test environments; no "works on my machine" problems
- **Development environments** — `docker compose up` to spin up a full stack (app + DB + cache + proxy) in seconds
- **Application isolation** — Run multiple versions of the same software side-by-side without conflicts
- **Cloud-native deployment** — The foundation for Kubernetes and container orchestration platforms
- **Legacy app modernization** — Containerize existing apps for easier deployment without full rewrites
- **Data science / ML** — Reproducible environments with pinned CUDA/library versions

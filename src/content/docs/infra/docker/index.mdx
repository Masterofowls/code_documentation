# Docker

Docker is an open-source platform for developing, shipping, and running applications in **containers** — lightweight, portable, self-sufficient units that package code and all its dependencies together. Since its public launch in 2013, Docker has become the de-facto standard for containerisation, powering everything from local developer workflows to large-scale cloud deployments.

## Containers vs Virtual Machines

Containers and VMs both provide isolation, but they do so at different layers of the stack.

| Feature | Containers | Virtual Machines |
|---|---|---|
| **Isolation level** | Process-level (OS namespaces) | Hardware-level (hypervisor) |
| **Boot time** | Milliseconds | Seconds to minutes |
| **Disk size** | Megabytes | Gigabytes |
| **Memory overhead** | Very low (shared kernel) | High (full OS per VM) |
| **Portability** | Very high (OCI image) | Medium (hypervisor-dependent) |
| **Security boundary** | Shared kernel — less strict | Full kernel isolation — stricter |
| **Density** | Hundreds per host | Tens per host |
| **Use case** | Microservices, CI/CD, dev environments | Legacy apps, strong isolation requirements |

:::tip[Best of Both Worlds]
Many production setups run containers **inside** VMs: VMs provide the security boundary between tenants, while containers provide density and fast iteration within each VM. Kubernetes nodes are typically VMs.
:::

## Docker Architecture

Docker uses a client–server architecture. Understanding the components helps you reason about what happens when you run a `docker` command.

```
┌─────────────────────────────────────────────────────────┐
│  Docker Client (CLI / SDK)                              │
│  docker build · docker pull · docker run · docker push  │
└────────────────────┬────────────────────────────────────┘
                     │ REST API (Unix socket / TCP)
┌────────────────────▼────────────────────────────────────┐
│  Docker Daemon  (dockerd)                               │
│  ┌──────────────────────────────────────────────────┐  │
│  │  containerd  (container lifecycle manager)        │  │
│  │  ┌──────────────────────────────────────────┐    │  │
│  │  │  runc  (OCI runtime — creates namespaces) │    │  │
│  │  └──────────────────────────────────────────┘    │  │
│  └──────────────────────────────────────────────────┘  │
└────────────────────┬────────────────────────────────────┘
                     │
┌────────────────────▼────────────────────────────────────┐
│  Registry  (Docker Hub / GHCR / ECR / private)          │
└─────────────────────────────────────────────────────────┘
```

### Docker Daemon (`dockerd`)

The daemon is a long-running background process that manages Docker objects — images, containers, networks, and volumes. It exposes a REST API over a Unix socket (`/var/run/docker.sock`) or an optional TCP port.

### Docker Client (`docker`)

The CLI you interact with daily. It translates commands into API calls sent to the daemon. The client and daemon can run on the same host or on different machines.

### Registry

A registry stores and distributes **Docker images**. Docker Hub (`registry-1.docker.io`) is the default public registry. You can also use private registries such as GitHub Container Registry (`ghcr.io`), AWS ECR, Google Artifact Registry, or a self-hosted registry.

### Images

An **image** is a read-only, layered filesystem snapshot built from a `Dockerfile`. Each instruction in a `Dockerfile` adds a new layer. Images are addressed by `name:tag` (e.g., `nginx:1.27-alpine`).

### Containers

A **container** is a runnable instance of an image. Docker adds a thin writable layer on top of the read-only image layers when a container starts. Multiple containers can share the same image without duplicating the underlying data.

### Volumes

**Volumes** are Docker-managed directories that persist data beyond the lifecycle of a container. They are stored in `/var/lib/docker/volumes/` on the host and are the recommended way to persist database data, uploads, and other stateful content.

### Networks

Docker creates virtual networks to allow containers to communicate. Each container gets its own network namespace. By default, containers on the same user-defined bridge network can reach each other by name.

## Docker Desktop vs Docker Engine

| | Docker Desktop | Docker Engine |
|---|---|---|
| **Platforms** | macOS, Windows, Linux | Linux only (natively) |
| **GUI** | Yes | No |
| **Kubernetes** | Built-in single-node cluster | Not included |
| **License** | Free for personal/small biz; paid for enterprise | Apache 2.0 (free) |
| **VM layer** | Runs daemon inside a Linux VM (HyperKit/WSL2/QEMU) | Runs natively on Linux kernel |
| **Dev extensions** | Extension marketplace | N/A |

:::caution[Docker Desktop Licensing]
As of 2022, Docker Desktop requires a paid subscription for companies with more than 250 employees or more than $10 M in annual revenue. Docker Engine on Linux remains free under the Apache 2.0 licence.
:::

## The OCI Standard

The **Open Container Initiative (OCI)** defines open standards for container formats and runtimes:

- **OCI Image Spec** — defines the image layer and manifest format. Any OCI-compliant image can run on any OCI-compliant runtime.
- **OCI Runtime Spec** — defines how a container runtime (like `runc`) creates and runs containers.
- **OCI Distribution Spec** — defines the API for pushing and pulling images from registries.

Docker images are fully OCI-compliant, which is why they can be run with alternative runtimes like Podman.

## containerd

`containerd` is an industry-standard container runtime that was originally part of Docker and later donated to the CNCF. Docker Engine now uses `containerd` internally for all container lifecycle management. Kubernetes also uses `containerd` directly via the Container Runtime Interface (CRI), bypassing the Docker daemon entirely.

## Docker vs Podman

[Podman](https://podman.io) is a daemonless, rootless container engine from Red Hat that is fully OCI-compatible and Docker CLI-compatible.

| | Docker | Podman |
|---|---|---|
| **Daemon** | Required (`dockerd`) | Daemonless |
| **Root required** | Historically yes; rootless mode available | Rootless by default |
| **Compose support** | `docker compose` (v2 plugin) | `podman-compose` (third-party) |
| **Kubernetes pods** | Via `kompose` | Native `podman pod` commands |
| **CLI compatibility** | Reference implementation | Drop-in replacement (`alias docker=podman`) |

## Use Cases

Docker shines in a variety of scenarios:

- **Development environments** — Consistent "works on my machine" environments across the team. Developers run the same database version as production with a single command.
- **CI/CD pipelines** — Build, test, and scan an image in every pull request. Ship the exact same artifact to production.
- **Microservices** — Each service runs in its own container with its own dependencies, enabling independent deployments and scaling.
- **Serverless and edge** — Lightweight containers start in milliseconds, ideal for function-as-a-service platforms.
- **Legacy application modernisation** — Containerise existing apps without rewriting them to benefit from orchestration and portability.
- **Local development databases** — Spin up PostgreSQL, Redis, or MongoDB locally without installing them on the host OS.

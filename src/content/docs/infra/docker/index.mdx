# Docker

Docker is an open-source platform for building, shipping, and running applications in **containers** — lightweight, portable, self-sufficient units that package application code alongside all its dependencies, libraries, and runtime configuration.

Released in 2013 by Solomon Hykes at dotCloud (later renamed Docker, Inc.), Docker popularized the Linux container model and became the de facto standard for packaging and distributing software. Today it underpins most modern CI/CD pipelines, microservice architectures, and cloud-native deployments.

---

## Containers vs Virtual Machines

Both containers and VMs provide isolation, but they operate at fundamentally different layers of the stack.

| Feature                | Containers                           | Virtual Machines             |
| ---------------------- | ------------------------------------ | ---------------------------- |
| **Isolation level**    | Process-level (namespaces + cgroups) | Full hardware virtualization |
| **Boot time**          | Milliseconds                         | Seconds to minutes           |
| **Image size**         | Megabytes                            | Gigabytes                    |
| **Resource overhead**  | Minimal (shared OS kernel)           | High (full OS per VM)        |
| **Density**            | Hundreds per host                    | Tens per host                |
| **OS portability**     | Same kernel required                 | Any guest OS                 |
| **Security boundary**  | Weaker (shared kernel)               | Stronger (hypervisor)        |
| **Persistent storage** | Volumes / bind mounts                | Full disk images             |
| **Networking**         | Virtual bridges, overlays            | Virtual NICs, full stack     |
| **Use case**           | Microservices, CI/CD, dev parity     | Strong isolation, mixed OS   |

:::tip[When to Use Each]
Use containers for application isolation, fast iteration, and density. Use VMs when you need full OS-level isolation, different kernel versions, or are running untrusted workloads. In practice, containers often _run inside_ VMs in cloud environments — you get both.
:::

## Docker Architecture

Docker uses a client–server architecture. The Docker client communicates with the Docker daemon (server), which does the heavy lifting of building, running, and managing containers.

```
┌─────────────────────────────────────────────────────────────────┐
│  Docker Host                                                    │
│                                                                 │
│  ┌──────────┐   REST API   ┌─────────────────────────────────┐  │
│  │  Docker  │◄────────────►│         Docker Daemon           │  │
│  │  Client  │  (Unix sock  │  (dockerd)                      │  │
│  │  (CLI)   │  /TCP/TLS)   │                                 │  │
│  └──────────┘              │  ┌──────────┐  ┌────────────┐  │  │
│                            │  │  Images  │  │ Containers │  │  │
│                            │  └──────────┘  └────────────┘  │  │
│                            │  ┌──────────┐  ┌────────────┐  │  │
│                            │  │ Volumes  │  │  Networks  │  │  │
│                            │  └──────────┘  └────────────┘  │  │
│                            └─────────────────────────────────┘  │
└─────────────────────────────────────────────────────────────────┘
         │
         ▼
┌─────────────────┐
│  Docker Registry │  (Docker Hub, GHCR, ECR, ACR, private)
└─────────────────┘
```

### Core Components

**Docker Daemon (`dockerd`)**
The background service that manages Docker objects. It listens for API requests and manages images, containers, networks, and volumes. On Linux it runs as a systemd service; on macOS/Windows it runs inside a lightweight VM.

**Docker Client (`docker`)**
The primary CLI tool users interact with. Commands like `docker run`, `docker build`, and `docker pull` are sent via the REST API to the daemon. The client can connect to remote daemons via TCP with TLS.

**Docker Registry**
A storage and distribution system for Docker images. Docker Hub (`hub.docker.com`) is the default public registry. Organizations typically also maintain private registries.

**Images**
Read-only templates built from a `Dockerfile`. Each image is composed of stacked, immutable **layers** stored in a content-addressable cache. Layers are shared across images, making storage efficient.

**Containers**
A runnable instance of an image. Docker adds a thin read-write layer on top of the image layers. Containers are isolated via Linux namespaces (pid, net, mnt, uts, ipc, user) and constrained via cgroups (CPU, memory, I/O).

**Volumes**
Managed storage that persists beyond the container lifecycle. Stored in Docker's data directory (`/var/lib/docker/volumes/`) and decoupled from the container filesystem.

**Networks**
Virtual networks that containers attach to. Docker provides several built-in drivers (bridge, host, overlay, macvlan) and supports third-party network plugins.

## The OCI Standard

Docker helped establish the **Open Container Initiative (OCI)**, a set of open industry standards for container formats and runtimes. The two key specs are:

- **OCI Image Spec** — defines the image format (layers, manifest, config)
- **OCI Runtime Spec** — defines how to run a container (`runc` is the reference implementation)

Any OCI-compliant tool can build, push, pull, and run OCI images — regardless of whether Docker was involved.

### containerd

`containerd` is the high-level container runtime that Docker (and Kubernetes) use under the hood. It handles image pulling, storage, networking attachment, and lifecycle management. Docker wraps `containerd` with its daemon API and CLI experience.

```
docker CLI → dockerd → containerd → runc → container process
```

Kubernetes communicates directly with `containerd` via the **Container Runtime Interface (CRI)**, bypassing Docker entirely since Kubernetes 1.24.

## Docker Desktop vs Docker Engine

|                | Docker Desktop                                   | Docker Engine         |
| -------------- | ------------------------------------------------ | --------------------- |
| **Platform**   | macOS, Windows, Linux                            | Linux only (natively) |
| **GUI**        | Yes (dashboard, extensions)                      | No                    |
| **Kubernetes** | Built-in single-node cluster                     | Not included          |
| **VM**         | Uses Apple Hypervisor / WSL2                     | Runs on host kernel   |
| **License**    | Free for personal/small biz; paid for enterprise | Apache 2.0 (free)     |
| **Best for**   | Developer workstations                           | CI servers, Linux VMs |

:::caution[Docker Desktop Licensing]
Docker Desktop requires a paid subscription for companies with more than 250 employees or $10M in annual revenue. Docker Engine itself remains free and open source under Apache 2.0. For CI/CD runners and Linux servers, install Docker Engine directly.
:::

## Docker vs Podman

**Podman** is a daemonless, rootless container tool developed by Red Hat. It's API-compatible with Docker and can act as a drop-in replacement.

| Feature                 | Docker                       | Podman                                  |
| ----------------------- | ---------------------------- | --------------------------------------- |
| **Architecture**        | Daemon-based (`dockerd`)     | Daemonless (fork/exec)                  |
| **Root required**       | By default (can be rootless) | Rootless by default                     |
| **Compose**             | `docker compose` (built-in)  | `podman-compose` (separate)             |
| **Pods**                | Not native                   | Native (mirrors K8s pods)               |
| **systemd integration** | Limited                      | First-class (`podman generate systemd`) |
| **OCI compliant**       | Yes                          | Yes                                     |

For most developers on Linux, `alias docker=podman` works for the majority of use cases.

## Key Use Cases

**Local Development**
Run databases (PostgreSQL, MySQL, Redis), message brokers (Kafka, RabbitMQ), and third-party services locally without polluting your host OS. Destroy and recreate in seconds.

**Consistent Environments**
Eliminate "works on my machine" problems. The container carries its own OS libraries, runtimes, and dependencies — identical across developer laptops, CI runners, and production servers.

**Microservices**
Package each service with its own runtime and dependencies. Independent deployments, independent scaling, clear service boundaries.

**CI/CD Pipelines**
Build once, push to registry, pull in any environment. Docker images are the universal artifact format for modern pipelines.

**Application Packaging and Distribution**
Ship software as a self-contained image. No installation scripts, no dependency conflicts, no OS compatibility matrix.

## Quick Reference

```bash
# Lifecycle
docker run -d -p 8080:80 --name my-app nginx:alpine
docker stop my-app && docker start my-app
docker rm -f my-app

# Images
docker build -t my-image:latest .
docker pull postgres:16-alpine
docker images
docker rmi my-image:latest

# Inspect
docker ps -a
docker logs -f my-app
docker exec -it my-app sh
docker inspect my-app

# Cleanup
docker system prune -af --volumes
```

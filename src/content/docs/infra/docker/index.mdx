# Docker

Docker is an open-source platform for developing, shipping, and running applications in **containers** — lightweight, portable, self-sufficient units that package an application alongside all its dependencies, libraries, and configuration.

Since its release in 2013, Docker has become the de facto standard for container tooling and fundamentally changed how teams build, ship, and operate software.

---

## Containers vs Virtual Machines

Containers and VMs both provide isolation, but they do so at different layers of the stack.

| Feature | Containers | Virtual Machines |
|---|---|---|
| **Boot time** | Milliseconds | Minutes |
| **Size** | MBs (image layers) | GBs (full OS) |
| **OS sharing** | Shares host kernel | Full guest OS per VM |
| **Isolation** | Process-level (namespaces + cgroups) | Hardware-level (hypervisor) |
| **Portability** | Extremely high | Moderate |
| **Overhead** | Minimal | Significant CPU/RAM |
| **Startup** | Near-instant | Slow (OS boot required) |
| **Security boundary** | Kernel shared — weaker | Stronger hardware boundary |
| **Density** | Hundreds per host | Dozens per host |
| **Use case** | Microservices, CI/CD, packaging | Full OS isolation, legacy apps |

:::tip[When to use each]
Use containers for modern application workloads and CI/CD pipelines. Prefer VMs when you need strong security boundaries, different kernels, or must run legacy workloads that assume full OS access.
:::

## Docker Architecture

Docker uses a **client–server** architecture. The Docker client communicates with the Docker daemon via a REST API over a Unix socket or TCP.

```
┌──────────────────────────────────────────────────────────┐
│                      Docker Host                         │
│                                                          │
│  ┌──────────┐     REST API    ┌────────────────────────┐ │
│  │  Docker  │ ─────────────► │    Docker Daemon        │ │
│  │  Client  │                │    (dockerd)            │ │
│  └──────────┘                │                         │ │
│                              │  ┌──────┐  ┌─────────┐  │ │
│                              │  │Image │  │Container│  │ │
│                              │  └──────┘  └─────────┘  │ │
│                              │  ┌──────┐  ┌─────────┐  │ │
│                              │  │Volume│  │ Network │  │ │
│                              │  └──────┘  └─────────┘  │ │
│                              └────────────────────────┘ │
└──────────────────────────────────────────────────────────┘
                                        │
                                        ▼
                              ┌──────────────────┐
                              │  Docker Registry  │
                              │  (Docker Hub /    │
                              │   private)        │
                              └──────────────────┘
```

### Core Components

**Docker Daemon (`dockerd`)**
The background service running on the host that manages Docker objects. It listens for Docker API requests and manages images, containers, networks, and volumes.

**Docker Client (`docker`)**
The primary interface for interacting with Docker. When you run `docker run`, the client sends the command to `dockerd`.

**Docker Registry**
A storage and distribution system for Docker images. Docker Hub is the default public registry. You can also run private registries using `registry:2` or cloud-hosted options like GHCR and ECR.

**Images**
Read-only templates used to create containers. Images are built from a `Dockerfile` and consist of stacked, cached layers. Each layer represents a set of filesystem changes.

**Containers**
Runnable instances of images. Containers add a writable layer on top of the image layers. They are isolated via Linux namespaces (pid, net, ipc, mnt, uts) and resource-controlled via cgroups.

**Volumes**
Persistent data storage managed by Docker. Volumes survive container removal, can be shared between multiple containers, and are stored in a Docker-managed location on the host filesystem.

**Networks**
Virtual networks that allow containers to communicate with each other and the outside world. Docker provides multiple network drivers: bridge, host, overlay, macvlan, and none.

## Docker Desktop vs Docker Engine

| | Docker Desktop | Docker Engine |
|---|---|---|
| **Platform** | macOS, Windows, Linux | Linux only |
| **GUI** | Yes — includes a dashboard | No |
| **Kubernetes** | Built-in single-node cluster | Not included |
| **Licensing** | Commercial license required for large organizations | Apache 2.0, free |
| **VM layer** | Uses a Linux VM (HyperKit/WSL2/Lima) | Runs natively |
| **Dev Extensions** | Marketplace extensions available | Not applicable |
| **BuildKit** | Enabled by default | Configurable |
| **Dev Environments** | Supported | Not included |

:::caution[Docker Desktop Licensing]
Docker Desktop requires a paid subscription for companies with more than 250 employees or more than $10M in annual revenue. Docker Engine on Linux remains free under Apache 2.0 for all use cases, including commercial production workloads.
:::

## The OCI Standard

The **Open Container Initiative (OCI)** defines open, vendor-neutral standards for container formats and runtimes — ensuring interoperability between tools.

- **OCI Image Spec** — defines the format for container images (layers, manifests, configs)
- **OCI Runtime Spec** — defines the interface for running containers (lifecycle, config.json format)
- **OCI Distribution Spec** — defines the API for pushing and pulling images from registries

Docker images and containers are fully OCI-compliant, meaning they work with any OCI-compatible runtime including containerd, CRI-O, and Podman.

---

## containerd and the Runtime Stack

Docker uses **containerd** as its high-level container runtime, which in turn delegates to **runc** (an OCI runtime) to actually create and run containers via Linux syscalls.

```
docker CLI
    │
    ▼
dockerd (Docker daemon — image management, networking, volumes)
    │
    ▼
containerd (container lifecycle management — pull, create, start, stop)
    │
    ▼
runc (OCI runtime — spawns container processes via Linux namespaces + cgroups)
    │
    ▼
Linux kernel (namespaces: pid, net, mnt, ipc, uts; cgroups v2)
```

containerd is also used directly by Kubernetes via the **Container Runtime Interface (CRI)**, which is why Docker images work seamlessly in Kubernetes clusters even though Docker itself is no longer the default Kubernetes runtime.

## Docker vs Podman

**Podman** is a daemonless container engine developed by Red Hat, fully OCI-compatible and Docker CLI-compatible.

| Feature | Docker | Podman |
|---|---|---|
| **Daemon** | Requires `dockerd` background service | Daemonless — no background service |
| **Default user** | Runs as root | Rootless by default |
| **CLI compatibility** | Docker CLI | `podman` — drop-in replacement |
| **Compose** | `docker compose` | `podman-compose` |
| **Pods** | No native pod concept | Supports OCI-style pods |
| **Systemd** | Limited integration | First-class `podman generate systemd` |
| **Image format** | OCI / Docker v2 | OCI compatible |
| **Desktop app** | Docker Desktop | Podman Desktop |

:::tip[Podman compatibility alias]
On many systems: `alias docker=podman` works transparently for common workflows since Podman implements the same CLI interface. Commands like `docker run`, `docker build`, and `docker compose` all work with this alias.
:::

## Common Use Cases

**Local Development**
Spin up databases, message queues, and services locally without installing them on your host machine. Destroy and recreate environments cleanly at any time with `docker compose down -v && docker compose up -d`.

**CI/CD Pipelines**
Build reproducible build environments. Run tests in isolated containers. Produce container images as build artifacts that can be deployed identically to any environment.

**Microservices**
Package each service independently with its exact dependencies. Deploy, scale, and update services without affecting others. Each service owns its runtime.

**Serverless and Containers-as-a-Service**
Platforms like AWS Lambda (container image support), Google Cloud Run, and Azure Container Instances run OCI-compatible container images.

**Database and Middleware**
Run PostgreSQL, Redis, Kafka, RabbitMQ, Elasticsearch, and any other stateful service with a single command. No installation, no version conflicts, instant cleanup.

**Edge Computing**
Lightweight Alpine-based containers are ideal for resource-constrained edge devices running ARM processors.

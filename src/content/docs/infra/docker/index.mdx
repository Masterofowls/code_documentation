# Docker

Docker is an open-source platform for developing, shipping, and running applications in **containers** — lightweight, portable, self-sufficient units that bundle an application and all its dependencies into a single deployable artifact.

## What Is Docker?

Docker solves the classic "it works on my machine" problem by packaging an application together with its runtime, libraries, configuration, and system tools. The resulting image runs identically on a developer's laptop, a CI server, and a production cluster.

Docker was first released in 2013 by Solomon Hykes at dotCloud. It popularised Linux containers (which had existed since 2006 via cgroups and namespaces) with a developer-friendly CLI, image format, and registry. Today it is the dominant container toolchain and the foundation of modern cloud-native development.

## Containers vs Virtual Machines

| Feature | Container | Virtual Machine |
|---|---|---|
| Isolation level | Process-level (namespaces + cgroups) | Full OS virtualisation (hypervisor) |
| Boot time | Milliseconds | Seconds to minutes |
| Image size | Megabytes (shared layers) | Gigabytes |
| Overhead | Minimal — shares host kernel | Significant — runs full guest OS |
| Portability | Very high — OCI image runs anywhere | Medium — depends on hypervisor |
| Security boundary | Weaker (shared kernel) | Stronger (separate kernel) |
| Use case | Microservices, CI/CD, cloud-native | Strong isolation, legacy OS, desktop VMs |
| Examples | Docker, containerd, Podman | VMware, VirtualBox, KVM, Hyper-V |

:::tip[Containers complement VMs]
In production you will often run containers *inside* VMs — Kubernetes nodes are typically VMs. The two technologies work together rather than replacing each other.
:::

## Docker Architecture

Docker uses a client–server architecture. The Docker CLI client talks to the Docker daemon over a Unix socket or TCP.

│  │   (docker)   │                │   (dockerd)    │  │
│  └──────────────┘                └───────┬────────┘  │
│                                          │            │
│                               ┌──────────▼─────────┐ │
│                               │     containerd     │ │

### Core Components

**Docker Daemon (`dockerd`)**
The long-running background service that manages Docker objects. It listens for API requests from the Docker client and delegates to containerd.

**Docker Client (`docker`)**
The CLI tool users interact with. Commands like `docker run`, `docker build`, and `docker push` are translated into API calls to the daemon.

**Docker Registry**
A storage and distribution service for Docker images. Docker Hub (`hub.docker.com`) is the default public registry. Private registries include GitHub Container Registry (ghcr.io), AWS ECR, Google Artifact Registry, and self-hosted solutions.

**Images**
Read-only, layered file system snapshots built from a `Dockerfile`. Each layer represents an instruction in the Dockerfile. Layers are cached and shared between images, making pulls fast and storage efficient.

**Containers**
Running instances of images. A container adds a thin writable layer on top of the image layers. Multiple containers can share the same image layers simultaneously.

**Volumes**
Persistent storage managed by Docker that exists outside the container's writable layer. Data in volumes survives container removal.

**Networks**
Virtual networks that allow containers to communicate with each other and the outside world. Docker creates default bridge, host, and none networks automatically.

### containerd

`dockerd` delegates container lifecycle management to **containerd**, a CNCF-graduated container runtime that handles image pulls, container execution via `runc`, and storage. containerd can be used directly without Docker (as Kubernetes does in modern clusters).

## OCI Standard

The **Open Container Initiative (OCI)** defines portable standards for container images and runtimes:

- **OCI Image Spec** — defines the image format (layers, manifests, config)
- **OCI Runtime Spec** — defines how a container is started from an image (implemented by `runc`)
- **OCI Distribution Spec** — defines how images are pushed/pulled from registries

Any OCI-compliant tool can build, run, and distribute containers. This means images built with Docker run on containerd, Podman, and CRI-O without modification.

## Docker Desktop vs Docker Engine

| | Docker Desktop | Docker Engine |
|---|---|---|
| Platform | macOS, Windows, Linux | Linux only |
| Includes | Docker Engine, Compose, BuildKit, Extensions, Dev Environments, K8s | Core daemon + CLI |
| License | Free for personal/small biz; paid for enterprise | Apache 2.0, always free |
| Installation | GUI installer | Package manager (`apt`, `yum`) |
| VM required | Yes (Linux VM on macOS/Win) | No |
| Best for | Developer workstations | Servers, CI runners |

:::caution[Docker Desktop Licensing]
As of 2022, Docker Desktop requires a paid subscription for companies with more than 250 employees or more than $10M annual revenue. Docker Engine on Linux remains free under Apache 2.0.
:::

## Docker vs Podman

**Podman** is a daemonless, rootless OCI container runtime developed by Red Hat. Key differences:

| Feature | Docker | Podman |
|---|---|---|
| Daemon | Yes (`dockerd`) | No — direct `fork/exec` |
| Root required | By default (rootful) | No — fully rootless by default |
| Compose support | `docker compose` (v2 built-in) | `podman-compose` (third-party) |
| CLI compatibility | Reference | Drop-in compatible (`alias docker=podman`) |
| Pod concept | No native pods | Yes — mirrors Kubernetes pods |
| systemd integration | Limited | First-class via `podman generate systemd` |

Podman is a compelling choice for security-conscious environments and is the default on newer RHEL/Fedora systems.

## Use Cases

- **Local development** — consistent environments across team members
- **Microservices** — isolate each service with its own dependencies
- **CI/CD pipelines** — reproducible build and test environments
- **Legacy application isolation** — run old software without polluting the host
- **Kubernetes** — Docker images are the deployment unit for K8s workloads
- **Edge computing** — small OCI images deploy to resource-constrained devices
- **Data science** — reproducible Jupyter environments with pinned package versions

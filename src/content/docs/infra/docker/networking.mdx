# Docker Networking

Docker networking enables containers to communicate with each other and the outside world. Understanding network types and DNS resolution is essential for building robust multi-container applications.

---

## Network Types Overview

| Driver | Use Case | Scope |
|---|---|---|
| **bridge** | Default for standalone containers; container-to-container on same host | Local |
| **host** | Remove network isolation; container shares host network stack | Local |
| **none** | Completely isolated container; no networking | Local |
| **overlay** | Multi-host networking for Docker Swarm | Swarm |
| **macvlan** | Assign MAC address to container; appear as physical device on network | Local |
| **ipvlan** | Like macvlan but shares MAC; useful in environments where MAC spoofing is blocked | Local |

## Managing Networks

```bash
# List all networks
docker network ls

# Inspect a network
docker network inspect bridge
docker network inspect my-network

# Create a network
docker network create my-network
docker network create --driver bridge my-bridge
docker network create \
  --driver bridge \
  --subnet 172.20.0.0/16 \
  --ip-range 172.20.240.0/20 \
  --gateway 172.20.0.1 \
  my-custom-network

# Remove a network
docker network rm my-network

# Remove all unused networks
docker network prune

# Connect a running container to a network
docker network connect my-network my-container
docker network connect --alias db my-network my-postgres

# Disconnect a container from a network
docker network disconnect my-network my-container
```

## Bridge Networking

The **bridge** network is the default network driver. When you start a container without specifying a network, it joins the default `bridge` network.

### Default Bridge Network

```bash
# Start two containers on the default bridge network
docker run -d --name container1 alpine sleep 3600
docker run -d --name container2 alpine sleep 3600

# Inspect the bridge network to see connected containers
docker network inspect bridge

# container1 can reach container2 only via IP, NOT by name
# (DNS name resolution does NOT work on the default bridge network)
docker exec container1 ping 172.17.0.3   # works (by IP)
docker exec container1 ping container2    # FAILS on default bridge
```

### User-Defined Bridge Network

User-defined bridge networks are **strongly preferred**. They provide automatic DNS resolution between containers.

```bash
# Create a custom bridge network
docker network create app-network

# Start containers on the custom network
docker run -d --name web --network app-network nginx
docker run -d --name api --network app-network my-api
docker run -d --name db  --network app-network postgres

# Now containers can reach each other by name!
docker exec api ping db          # works!
docker exec api curl http://web  # works!
```

:::tip[Default bridge vs user-defined bridge]
| Feature | Default Bridge | User-Defined Bridge |
|---|---|---|
| DNS resolution by container name | ❌ No | ✅ Yes |
| Automatic service discovery | ❌ No | ✅ Yes |
| Network isolation | Shared by all | Per-network isolation |
| `--link` required for name res | Yes (deprecated) | Not needed |
| Recommended | No | **Yes** |
:::

### Port Mapping

Port mapping exposes a container port on the host, enabling external access.

```bash
# Host port 8080 → Container port 80
docker run -p 8080:80 nginx

# Bind only to localhost (don't expose publicly)
docker run -p 127.0.0.1:8080:80 nginx

# Let Docker choose an available host port
docker run -p 80 nginx
docker port <container-id>          # Find the assigned port

# Map multiple ports
docker run -p 80:80 -p 443:443 nginx

# Map UDP port
docker run -p 53:53/udp dns-server

# Map all exposed ports to random host ports
docker run -P nginx                 # -P is capital P
```

**EXPOSE vs -p:**
- `EXPOSE` in a Dockerfile is **documentation only** — it does not publish the port
- `-p` (or `--publish`) in `docker run` **actually publishes** the port on the host

## Host Network

With the **host** network driver, the container shares the host's network namespace directly. There's no network isolation — the container behaves as if it's a process directly on the host.

```bash
docker run --network host nginx
# nginx now listens on host port 80 directly (no -p needed)
```

**When to use:**
- Maximum network performance (no NAT overhead)
- Applications that need to bind to specific host interfaces
- Network monitoring tools that need to see all host traffic

:::caution[Linux only]
The host network driver **only works on Linux**. On macOS and Windows, Docker runs in a VM, so `--network host` connects to the VM's network, not your Mac/Windows host network.
:::

## Container DNS Resolution

Docker runs an embedded DNS server at `127.0.0.11` for containers on user-defined networks. This provides automatic service discovery.

```bash
# From within a container on a user-defined network:
# Resolve by container name
nslookup my-container           # → container's IP

# Resolve by service alias
nslookup db.backend              # → if alias 'db' is set on 'backend' network

# DNS search domains are set automatically
# In docker-compose, 'postgres' resolves to the postgres service IP
```

### DNS in Docker Compose

Services in a Compose file automatically get DNS entries equal to their service name.

```yaml
services:
  api:
    image: my-api
    # Can reach postgres at: postgres://db:5432/mydb
    # Can reach redis at: redis://cache:6379
    environment:
      DATABASE_URL: postgres://db:5432/mydb
      REDIS_URL: redis://cache:6379

  db:
    image: postgres:16

  cache:
    image: redis:7
```

## Overlay Networks (Docker Swarm)

Overlay networks span multiple Docker hosts. They're used in Docker Swarm mode to allow containers on different machines to communicate as if on the same network.

```bash
# Initialize Swarm
docker swarm init

# Create an overlay network
docker network create --driver overlay --attachable my-overlay

# Services on the overlay network can communicate across hosts
docker service create --network my-overlay --name web nginx
docker service create --network my-overlay --name api my-api
```

## Network Aliases

Network aliases allow a container to be reachable by multiple names on a network.

```bash
# Assign alias when connecting to a network
docker network connect --alias db --alias primary my-network postgres

# In docker-compose:
```

```yaml
services:
  postgres:
    image: postgres:16
    networks:
      backend:
        aliases:
          - db
          - primary-db
          - database
```

## The `--link` Flag (Deprecated)

`--link` was the original mechanism for container-to-container communication. It is **deprecated** and should not be used in new projects.

```bash
# Old way (don't use this)
docker run --link db:database my-app

# Modern way — use user-defined networks instead
docker network create app-net
docker run --network app-net --name db postgres
docker run --network app-net my-app  # can reach 'db' by name
```

## IPv6 Support

```bash
# Enable IPv6 in daemon.json
# /etc/docker/daemon.json
```

```json
{
  "ipv6": true,
  "fixed-cidr-v6": "2001:db8:1::/64"
}
```

```bash
# Create IPv6-enabled network
docker network create \
  --driver bridge \
  --ipv6 \
  --subnet 172.20.0.0/16 \
  --subnet "2001:db8:2::/64" \
  ip6net
```

## Inter-Service Communication in Compose

With Docker Compose, services communicate via their **service name** as hostname on the default network.

```yaml
services:
  nginx:
    image: nginx
    # nginx proxies to api:3000

  api:
    image: my-node-api
    # api connects to database:5432 and redis:6379

  database:
    image: postgres:16

  redis:
    image: redis:7
```

```nginx
# nginx.conf — upstream using service name as hostname
upstream api {
  server api:3000;
}

server {
  listen 80;
  location /api/ {
    proxy_pass http://api;
  }
}
```

## macvlan Networks

macvlan networks assign a unique MAC address to each container, making them appear as physical network devices. Useful for legacy applications that need to be on the physical network.

```bash
docker network create \
  --driver macvlan \
  --subnet 192.168.1.0/24 \
  --gateway 192.168.1.1 \
  -o parent=eth0 \
  macvlan-net

docker run --network macvlan-net \
  --ip 192.168.1.100 \
  nginx
```

:::caution[macvlan and promiscuous mode]
Many cloud providers and managed switches disable promiscuous mode, which macvlan requires. If macvlan doesn't work, your network interface may have MAC spoofing protection enabled.
:::

## Network Troubleshooting

The `nicolaka/netshoot` image is an all-in-one network troubleshooting toolkit.

```bash
# Run a temporary troubleshooting container on the same network
docker run --rm -it \
  --network my-network \
  nicolaka/netshoot

# Check DNS resolution
nslookup db
nslookup db.my-network

# Test connectivity
ping api
curl http://api:3000/health
wget -qO- http://web

# Port scan
nmap -p 3000 api

# Trace route
traceroute api

# Check open ports
ss -tlnp
netstat -tlnp

# DNS lookup from inside a container
docker exec my-container nslookup postgres
docker exec my-container cat /etc/resolv.conf
docker exec my-container curl -v http://api:3000

# Inspect network settings of a running container
docker inspect --format '{{json .NetworkSettings}}' my-container | jq
docker inspect --format '{{range .NetworkSettings.Networks}}{{.IPAddress}}{{end}}' my-container
```

### Common Network Issues

| Symptom | Likely Cause | Fix |
|---|---|---|
| `Name or service not known` | Container on default bridge, not user-defined | Create and use a user-defined network |
| `Connection refused` | Port not published or service not listening | Check `EXPOSE` + `-p` flag, verify service is running |
| `No route to host` | Containers on different networks | Connect both to same network |
| DNS works, connections fail | Firewall / iptables rules | Check `docker network inspect` for subnet; check host firewall |
| macOS/Windows host not reachable | Host networking limitation | Use `host.docker.internal` as hostname to reach the host machine |

```bash
# On macOS/Windows, reach the host from a container:
curl http://host.docker.internal:8080

# On Linux, use the host gateway IP:
docker run --add-host=host.docker.internal:host-gateway my-app
```

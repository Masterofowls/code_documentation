# Docker Networking

Docker networking provides isolated communication channels between containers, the host, and external networks. Understanding how Docker networks work is essential for designing secure, reliable multi-container applications.

## Network Types Overview

| Driver | Description | Use Case |
|---|---|---|
| `bridge` | Software bridge on the host. Default for standalone containers. | Development, single-host apps |
| `host` | Container shares host network stack. No isolation. | High-performance, Linux only |
| `none` | No networking. Container is fully isolated. | Security-sensitive workloads |
| `overlay` | Multi-host networking for Docker Swarm | Distributed apps across nodes |
| `macvlan` | Container gets a MAC address and appears as physical device | Legacy apps needing direct network access |
| `ipvlan` | Similar to macvlan but shares MAC address | Low-overhead L2/L3 networking |

## Managing Networks

```bash
# List all networks
docker network ls

# Create a custom bridge network
docker network create my-network

# Create with custom subnet and gateway
docker network create \
  --driver bridge \
  --subnet 172.20.0.0/16 \
  --gateway 172.20.0.1 \
  --ip-range 172.20.0.0/24 \
  my-network

# Inspect a network (JSON output)
docker network inspect my-network

# Inspect with Go template
docker network inspect -f '{{range .Containers}}{{.Name}} {{.IPv4Address}}{{"\n"}}{{end}}' my-network

# Remove a network
docker network rm my-network

# Remove all unused networks
docker network prune

# Connect a running container to a network
docker network connect my-network my-container

# Connect with a specific alias
docker network connect --alias db my-network postgres

# Disconnect a container from a network
docker network disconnect my-network my-container
```

## Bridge Network

Bridge is the default network driver. Docker creates a `docker0` virtual bridge on the host.

### Default Bridge

```bash
# Run two containers on the default bridge
docker run -d --name container1 alpine sleep 3600
docker run -d --name container2 alpine sleep 3600

# Containers can ping each other by IP, but NOT by name
docker exec container1 ping 172.17.0.3    # works (IP)
docker exec container1 ping container2   # FAILS — no DNS on default bridge
```

### User-Defined Bridge

User-defined bridge networks provide automatic DNS resolution between containers by name.

```bash
# Create a named network
docker network create app-network

# Run containers on the named network
docker run -d --name api --network app-network myapp/api
docker run -d --name postgres --network app-network postgres:16

# api can reach postgres by name — DNS works!
docker exec api ping postgres   # works
docker exec api curl http://api:3000/health   # works
```

:::tip[Always use user-defined networks]
The default `bridge` network does not support container name resolution. Create a named network for every project to enable hostname-based communication.
:::

### Port Mapping

Port mapping exposes container ports to the host:

```bash
# Map host 8080 → container 80
docker run -d -p 8080:80 nginx

# Bind to specific host IP (localhost only)
docker run -d -p 127.0.0.1:8080:80 nginx

# Map multiple ports
docker run -d -p 8080:80 -p 8443:443 nginx

# Assign random host port
docker run -d -p 80 nginx
docker port <container_id>    # show assigned port

# Map all EXPOSE'd ports to random host ports
docker run -d -P nginx
```

### EXPOSE vs -p

```dockerfile
# In Dockerfile — documents intent, does NOT publish
EXPOSE 3000
```

```bash
# At runtime — actually publishes the port
docker run -p 3000:3000 myapp
```

`EXPOSE` is documentation. The actual publication happens with `-p` at runtime or `ports:` in Compose.

## Host Network

The container shares the host's network stack entirely. There is no network namespace isolation.

```bash
# Run with host networking
docker run -d --network host nginx

# Nginx now listens on host port 80 directly
# No -p flag needed or allowed
```

:::caution[Host network is Linux-only]
`--network host` has no effect on Docker Desktop for macOS and Windows because Docker runs inside a Linux VM. It works as expected on Linux hosts only.
:::

## None Network

Completely disables networking. The container has only a loopback interface.

```bash
docker run -d --network none alpine sleep 3600
docker exec <id> ping google.com  # FAILS — no connectivity
```

Use for containers that process data from mounted volumes without any network access.

## Container DNS Resolution

Docker runs an embedded DNS server at `127.0.0.11` for user-defined networks.

```bash
# Inside a container on a named network
# These all resolve via Docker's embedded DNS:
ping api               # service name
ping api.myapp         # service.network name
nslookup postgres      # returns container IP

# Check DNS config inside container
cat /etc/resolv.conf
# nameserver 127.0.0.11
# options ndots:0
```

## Network Aliases

Aliases allow a container to be reachable under multiple names on a network:

```bash
# Connect with alias
docker network connect --alias primary --alias db my-network postgres

# In Compose
services:
  postgres:
    networks:
      backend:
        aliases:
          - db
          - primary-db
```

```yaml
# Full compose example with aliases
networks:
  backend:
    driver: bridge

services:
  postgres:
    image: postgres:16-alpine
    networks:
      backend:
        aliases:
          - db
  api:
    image: myapp/api
    environment:
      DB_HOST: db    # uses the alias
    networks:
      - backend
```

## Inter-Service Communication in Compose

In Docker Compose, services on the same network communicate using the service name as the hostname.

```yaml
services:
  api:
    environment:
      # Use service name "postgres" as hostname
      DATABASE_URL: postgresql://app:secret@postgres:5432/myapp
      # Use service name "redis" as hostname
      REDIS_URL: redis://redis:6379
    networks:
      - backend

  postgres:
    image: postgres:16-alpine
    networks:
      - backend    # same network as api

  redis:
    image: redis:7-alpine
    networks:
      - backend
```

## Overlay Networks (Swarm)

Overlay networks span multiple Docker hosts in a Swarm cluster:

```bash
# Create an overlay network (requires Swarm mode)
docker network create \
  --driver overlay \
  --attachable \
  my-overlay

# Deploy a service to the overlay network
docker service create \
  --network my-overlay \
  --name api \
  myapp/api
```

## IPv6 Support

```bash
# Enable IPv6 in daemon.json
# /etc/docker/daemon.json
{
  "ipv6": true,
  "fixed-cidr-v6": "2001:db8:1::/64"
}

# Create IPv6-enabled network
docker network create \
  --driver bridge \
  --ipv6 \
  --subnet 2001:db8:2::/64 \
  ipv6-net
```

## `--link` (Deprecated)

The legacy `--link` flag connected containers and added `/etc/hosts` entries. It is deprecated and should not be used in new projects. Use named networks instead.

```bash
# Old way — DEPRECATED
docker run -d --name db postgres
docker run -d --link db:database myapp

# New way — use named network
docker network create mynet
docker run -d --network mynet --name db postgres
docker run -d --network mynet --name app myapp
# app can reach db by hostname "db"
```

## Network Troubleshooting

### Check Container Connectivity

```bash
# Get a shell in the container
docker exec -it my-container sh

# Test DNS resolution
nslookup postgres
cat /etc/resolv.conf

# Test connectivity
ping postgres
curl -v http://api:3000/health
wget -qO- http://nginx/

# Check listening ports inside container
ss -tlnp
netstat -tlnp
```

### Inspect Network Configuration

```bash
# See which network(s) a container is on
docker inspect --format='{{json .NetworkSettings.Networks}}' my-container | jq

# List all containers on a network
docker network inspect my-network | jq '.[0].Containers'

# Check iptables rules (Linux host)
sudo iptables -L -n -t nat
sudo iptables -L DOCKER -n -v
```

### Install Networking Tools in Alpine

```bash
# Alpine doesn't include curl, ping, or nslookup by default
docker exec -it my-container sh
apk add --no-cache curl bind-tools iputils
```

### Temporary Debug Container

```bash
# Attach a debugging container to a network
docker run -it --rm \
  --network myapp_backend \
  nicolaka/netshoot \
  sh

# Inside netshoot — full suite of network tools available
curl http://api:3000/health
nslookup postgres
traceroute redis
tcpdump -i eth0 port 5432
```

:::tip[nicolaka/netshoot]
`nicolaka/netshoot` is a container image packed with networking tools: `curl`, `dig`, `nslookup`, `netstat`, `ss`, `ping`, `traceroute`, `tcpdump`, `iperf`, `iftop`, and more. Perfect for debugging network issues in containers.
:::

## Security Considerations

```bash
# Use internal networks to prevent external access
docker network create \
  --driver bridge \
  --internal \
  db-net

# Containers on db-net cannot reach the internet
# Only containers explicitly connected can communicate

# In Compose
networks:
  backend:
    driver: bridge
    internal: true   # no route to internet
```

### Network Segmentation in Compose

```yaml
services:
  nginx:
    networks:
      - public    # internet-facing

  api:
    networks:
      - public    # receives requests from nginx
      - private   # connects to db

  postgres:
    networks:
      - private   # isolated — not reachable from outside

networks:
  public:
    driver: bridge
  private:
    driver: bridge
    internal: true
```

This pattern ensures that `postgres` is only reachable from `api`, and `api` is only reachable from `nginx`.

# Docker in Production

Running Docker containers in production requires a mindset shift from local development. This guide covers the practices, configurations, and tooling needed to run containers reliably, securely, and observably in production environments.

---

## Production Dockerfile Best Practices

A production-ready Dockerfile follows these principles:

```dockerfile
# syntax=docker/dockerfile:1

# 1. Pin to a specific digest, not just a tag
FROM node:20.11.1-alpine3.19

# 2. Apply meaningful labels (OCI standard)
LABEL org.opencontainers.image.title="My API"
LABEL org.opencontainers.image.description="Production API server"
LABEL org.opencontainers.image.version="${APP_VERSION}"
LABEL org.opencontainers.image.vendor="My Company"
LABEL org.opencontainers.image.licenses="MIT"

# 3. Set working directory
WORKDIR /app

# 4. Install dependencies first (better caching)
COPY package*.json ./
RUN npm ci --only=production && npm cache clean --force

# 5. Copy application code
COPY --chown=node:node . .

# 6. Drop to non-root user
USER node

# 7. Expose only necessary port
EXPOSE 3000

# 8. Set HEALTHCHECK
HEALTHCHECK --interval=30s --timeout=10s --start-period=30s --retries=3 \
  CMD wget -qO- http://localhost:3000/health || exit 1

# 9. Use exec form for proper signal handling
# 10. Use tini for PID 1 (included in node:alpine as 'tini')
ENTRYPOINT ["/sbin/tini", "--"]
CMD ["node", "server.js"]
```

**Checklist:**
- [ ] Pinned base image tag (not `:latest`)
- [ ] `LABEL` with OCI metadata
- [ ] Non-root `USER`
- [ ] `HEALTHCHECK` defined
- [ ] Exec form `ENTRYPOINT`/`CMD` (not shell form)
- [ ] No secrets in any layer
- [ ] Minimal base image (alpine/slim/distroless)
- [ ] Multi-stage build (dev deps excluded)
- [ ] `.dockerignore` excludes `.git`, `node_modules`, etc.

## Resource Limits

Without resource limits, a single runaway container can consume all host resources and bring down other workloads.

```bash
# Memory limit: container is OOM-killed if it exceeds this
docker run -d --memory="512m" nginx

# Memory + swap (total memory including swap)
docker run -d --memory="512m" --memory-swap="1g" nginx

# CPU limit (fractional cores)
docker run -d --cpus="1.5" nginx

# CPU shares (relative weight, default 1024)
docker run -d --cpu-shares=512 low-priority-app

# CPU pinning (bind to specific CPUs)
docker run -d --cpuset-cpus="0,1" nginx

# Process limit (prevent fork bombs)
docker run -d --pids-limit=100 nginx

# Block I/O throttle
docker run -d --device-write-bps /dev/sda:10mb nginx
docker run -d --device-read-bps  /dev/sda:10mb nginx
```

```yaml
# In docker-compose / Swarm deploy
services:
  api:
    image: my-api
    deploy:
      resources:
        limits:
          cpus: "0.5"
          memory: 512M
          pids: 100
        reservations:
          cpus: "0.25"
          memory: 256M
```

:::tip[Set both limits and reservations]
`limits` = maximum allowed, `reservations` = guaranteed minimum. Setting reservations helps the scheduler place containers on nodes with enough resources and prevents over-subscription.
:::

## Logging Drivers

Docker supports multiple logging drivers. The default `json-file` is fine for development; production deployments often route to a centralized logging system.

### json-file (Default)

```bash
# Run with size-limited json-file logging
docker run -d \
  --log-driver json-file \
  --log-opt max-size=10m \
  --log-opt max-file=3 \
  nginx
```

```json
// /etc/docker/daemon.json — set defaults for all containers
{
  "log-driver": "json-file",
  "log-opts": {
    "max-size": "10m",
    "max-file": "3",
    "compress": "true"
  }
}
```

    ```bash
    docker run -d \

    ```
  </TabItem>

:::caution[Logging driver and `docker logs`]
When using drivers other than `json-file` or `journald`, the `docker logs` command may not work or return an error. Logs are only available through the external system (CloudWatch, fluentd, etc.).
:::

## Restart Policies

```bash
# Do not restart (default)
docker run --restart=no nginx

# Always restart (even on manual stop — use unless-stopped instead)
docker run --restart=always nginx

# Restart unless explicitly stopped
docker run --restart=unless-stopped nginx

# Restart on failure, with a retry limit
docker run --restart=on-failure:3 nginx

# Check restart policy
docker inspect --format '{{.HostConfig.RestartPolicy}}' my-container
```

| Policy | Behavior |
|---|---|
| `no` | Never restart (default) |
| `always` | Always restart, even after `docker stop` + daemon restart |
| `unless-stopped` | Restart unless manually stopped; persists across daemon restarts |
| `on-failure[:N]` | Restart only on non-zero exit; optional max retry count |

**Use `unless-stopped`** for most production services. Use `on-failure:3` for tasks that should retry a limited number of times.

## Secrets Management

Never bake secrets into images or environment variables that appear in `docker inspect`.

### Docker Secrets (Swarm)

```bash
# Create a secret from a file
echo "supersecretpassword" | docker secret create db_password -

# Create from stdin
docker secret create ssl_cert ./ssl.crt

# List secrets
docker secret ls

# Use in a Swarm service
docker service create \
  --name api \
  --secret db_password \
  --secret ssl_cert \
  my-api
# Secrets are mounted at /run/secrets/<name>
```

### Compose Secrets

```yaml
# compose.yml
services:
  api:
    secrets:
      - db_password
      - api_key
    environment:
      # Read from file mounted by secrets
      DB_PASSWORD_FILE: /run/secrets/db_password

secrets:
  db_password:
    file: ./secrets/db_password.txt    # From file

  api_key:
    environment: API_KEY               # From host environment variable
```

```bash
# In application code, read the secret from file:
```

```javascript
const fs = require('fs');
const dbPassword = fs.readFileSync('/run/secrets/db_password', 'utf8').trim();
```

## Health Checks

Health checks allow Docker to determine if your application is actually healthy (not just running).

### In Dockerfile

```dockerfile
# HTTP health check
HEALTHCHECK --interval=30s --timeout=10s --start-period=30s --retries=3 \
  CMD curl -f http://localhost:3000/health || exit 1

# TCP check (no curl available)
HEALTHCHECK --interval=30s --timeout=5s \
  CMD nc -z localhost 3000 || exit 1

# Application-native check
HEALTHCHECK --interval=30s --timeout=10s \
  CMD node healthcheck.js || exit 1

# Disable inherited healthcheck from base image
HEALTHCHECK NONE
```

### In docker-compose

```yaml
services:
  api:
    image: my-api
    healthcheck:
      test: ["CMD", "wget", "-qO-", "http://localhost:3000/health"]
      interval: 30s
      timeout: 10s
      retries: 3
      start_period: 30s   # Grace period before first check; failures don't count

  db:
    image: postgres:16
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U postgres -d mydb"]
      interval: 10s
      timeout: 5s
      retries: 5
      start_period: 20s
```

### Health Check States

```bash
# Inspect health status
docker inspect --format '{{.State.Health.Status}}' my-container
# starting | healthy | unhealthy | none

# View health check log
docker inspect --format '{{json .State.Health}}' my-container | jq

# Watch health status
watch docker inspect --format '{{.State.Health.Status}}' my-container
```

:::tip[Health check endpoint design]
Your `/health` endpoint should:
- Return `200 OK` if fully ready to serve traffic
- Return `503 Service Unavailable` if degraded
- Check actual dependencies (DB connection, cache, etc.)
- Respond in &lt;2 seconds (health check timeout is typically 5–10s)
- Not require authentication
:::

## Container Security

### Linux Capabilities

Containers run with a default set of Linux capabilities. For maximum security, drop all and add only what's needed.

```bash
# Drop all capabilities, add only what's needed
docker run -d \
  --cap-drop ALL \
  --cap-add NET_BIND_SERVICE \   # Bind to ports < 1024
  --cap-add CHOWN \              # Change file ownership
  nginx

# Common capabilities to add selectively:
# NET_BIND_SERVICE  — bind to privileged ports
# CHOWN             — change file ownership
# SETUID/SETGID     — change process UID/GID
# NET_RAW           — raw sockets (ping)
```

### Read-Only Root Filesystem

```bash
# Make root filesystem read-only (prevent writes outside volumes/tmpfs)
docker run -d \
  --read-only \
  --tmpfs /tmp \
  --tmpfs /var/run \
  -v app-data:/app/data \
  my-api
```

### Security Options

```bash
# Prevent privilege escalation
docker run -d \
  --security-opt no-new-privileges \
  nginx

# Apply seccomp profile (restrict syscalls)
docker run -d \
  --security-opt seccomp=/path/to/seccomp.json \
  nginx

# Disable seccomp (not recommended for production)
docker run -d \
  --security-opt seccomp=unconfined \
  nginx

# AppArmor profile
docker run -d \
  --security-opt apparmor=docker-nginx \
  nginx

# User namespace remapping (run as root inside but non-root on host)
# Configure in /etc/docker/daemon.json:
# { "userns-remap": "default" }
```

### Full Security Hardening Example

```yaml
# compose.yml production security settings
services:
  api:
    image: my-api:1.0.0
    read_only: true
    tmpfs:
      - /tmp
      - /app/tmp
    security_opt:
      - no-new-privileges:true
      - seccomp:./seccomp/default.json
    cap_drop:
      - ALL
    cap_add:
      - NET_BIND_SERVICE
    user: "1001:1001"
    volumes:
      - app-data:/app/data     # Only writable path
```

## Monitoring with cAdvisor and Prometheus

```yaml
# compose.yml monitoring stack
services:
  cadvisor:
    image: gcr.io/cadvisor/cadvisor:latest
    ports:
      - "8080:8080"
    volumes:
      - /:/rootfs:ro
      - /var/run:/var/run:rw
      - /sys:/sys:ro
      - /var/lib/docker:/var/lib/docker:ro
    restart: unless-stopped

  prometheus:
    image: prom/prometheus:latest
    ports:
      - "9090:9090"
    volumes:
      - ./prometheus.yml:/etc/prometheus/prometheus.yml:ro
    restart: unless-stopped
```

## Graceful Shutdown

Your application must handle `SIGTERM` gracefully, completing in-flight requests before exiting.

```javascript
// Node.js graceful shutdown
const server = app.listen(3000);

process.on('SIGTERM', () => {
  console.log('SIGTERM received, shutting down gracefully');
  server.close(() => {
    console.log('HTTP server closed');
    // Close DB connections, flush buffers, etc.
    process.exit(0);
  });

  // Force exit after 30 seconds
  setTimeout(() => {
    console.error('Forced shutdown after timeout');
    process.exit(1);
  }, 30000);
});
```

```python
# Python graceful shutdown (FastAPI)
import signal, sys

def handle_sigterm(*args):
    print("SIGTERM received, shutting down...")
    sys.exit(0)

signal.signal(signal.SIGTERM, handle_sigterm)
```

### Docker Stop Grace Period

```bash
# docker stop sends SIGTERM, waits 10 seconds, then sends SIGKILL
docker stop my-container                    # 10s grace period (default)
docker stop --time 30 my-container          # 30s grace period
```

```yaml
# In compose.yml
services:
  api:
    stop_grace_period: 30s        # Override default 10s
    stop_signal: SIGTERM          # Signal to send (default: SIGTERM)
```

:::caution[Shell form and SIGTERM]
If your Dockerfile uses shell form `CMD node server.js`, the process runs as a child of `/bin/sh`. Docker's SIGTERM goes to `sh`, which may not forward it to your app. Always use **exec form** `CMD ["node", "server.js"]` so your app is PID 1 and receives signals directly.
:::

## Production Checklist

| Category | Check |
|---|---|
| **Image** | Pinned base image tag |
| **Image** | Non-root user |
| **Image** | Multi-stage build (no dev deps in runtime) |
| **Image** | Minimal base (alpine/slim/distroless) |
| **Image** | No secrets in layers or environment |
| **Runtime** | Memory and CPU limits set |
| **Runtime** | `--restart unless-stopped` |
| **Runtime** | `--read-only` with tmpfs for writable paths |
| **Runtime** | `--cap-drop ALL` + only needed caps |
| **Runtime** | `--security-opt no-new-privileges` |

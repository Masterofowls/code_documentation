# Docker in Production

Running containers in production requires hardening beyond the defaults. This guide covers security, resource management, logging, health checks, secrets, and observability for production Docker deployments.

## Production Dockerfile Checklist

```dockerfile
# syntax=docker/dockerfile:1

# ✅ Pin to a specific digest or immutable tag
FROM node:22.11.0-alpine3.20

# ✅ Label for traceability
LABEL org.opencontainers.image.source="https://github.com/org/repo" \
      org.opencontainers.image.version="1.2.3"

WORKDIR /app

# ✅ Copy dependency manifests first (layer caching)
COPY package.json package-lock.json ./
RUN npm ci --omit=dev && npm cache clean --force

# ✅ Copy application code after dependencies
COPY --chown=node:node dist/ ./dist/

# ✅ Non-root user
USER node

# ✅ Declare listening port
EXPOSE 3000

# ✅ Health check
HEALTHCHECK --interval=30s --timeout=5s --start-period=30s --retries=3 \
  CMD wget -qO- http://localhost:3000/health || exit 1

# ✅ Exec form entrypoint (receives signals properly)
ENTRYPOINT ["node", "dist/server.js"]
```

:::caution[Never store secrets in image layers]
Anything set via `ENV`, `ARG`, `COPY`, or `RUN` is visible in `docker history`. Use BuildKit secret mounts at build time and runtime secrets (Docker Swarm secrets, mounted files, or a secret manager) for runtime values.
:::

## Resource Limits

Without limits, a single runaway container can starve the host of CPU and memory.

### Memory

```bash
# Hard memory limit (container is OOM-killed if exceeded)
docker run -d --memory 512m nginx

# Memory + swap (total = memory + memory-swap)
docker run -d --memory 512m --memory-swap 1g nginx

# Disable swap
docker run -d --memory 512m --memory-swap 512m nginx

# Soft limit (kernel can reclaim memory under pressure)
docker run -d --memory 512m --memory-reservation 256m nginx
```

### CPU

```bash
# Limit to 0.5 CPU cores
docker run -d --cpus 0.5 nginx

# Assign to specific CPUs
docker run -d --cpuset-cpus "0,1" nginx

# CPU shares (relative weight, default 1024)
docker run -d --cpu-shares 512 nginx   # half the default priority
```

### Process Limits

```bash
# Limit number of processes / threads
docker run -d --pids-limit 100 nginx
```

### In Compose

```yaml
services:
  api:
    deploy:
      resources:
        limits:
          cpus: "1.0"
          memory: 512M
          pids: 100
        reservations:
          cpus: "0.25"
          memory: 128M
```

## Logging Drivers

The default `json-file` driver writes JSON logs to disk. Configure it to prevent unbounded disk growth, or switch to a centralised driver.

### json-file (default)

```json
// /etc/docker/daemon.json
{
  "log-driver": "json-file",
  "log-opts": {
    "max-size": "10m",
    "max-file": "3",
    "compress": "true"
  }
}
```

```bash
# Per-container override
docker run -d \
  --log-driver json-file \
  --log-opt max-size=10m \
  --log-opt max-file=3 \
  nginx
```

### syslog

```bash
docker run -d \
  --log-driver syslog \
  --log-opt syslog-address=udp://localhost:514 \
  --log-opt tag="myapp/{{.Name}}" \
  nginx
```

### fluentd

```bash
docker run -d \
  --log-driver fluentd \
  --log-opt fluentd-address=localhost:24224 \
  --log-opt tag="myapp.api" \
  nginx
```

### AWS CloudWatch Logs

```bash
docker run -d \
  --log-driver awslogs \
  --log-opt awslogs-region=us-east-1 \
  --log-opt awslogs-group=/myapp/api \
  --log-opt awslogs-stream=production \
  --log-opt awslogs-create-group=true \
  myapp
```

### In Compose

```yaml
services:
  api:
    logging:
      driver: json-file
      options:
        max-size: "10m"
        max-file: "3"
        compress: "true"
        labels: "env,app"
        tag: "{{.Name}}/{{.ID}}"
```

## Restart Policies

```bash
# Don't restart (default)
docker run -d --restart no nginx

# Always restart (even on clean exit)
docker run -d --restart always nginx

# Restart unless manually stopped
docker run -d --restart unless-stopped nginx

# Restart on failure, max 3 attempts
docker run -d --restart on-failure:3 nginx
```

| Policy | Restart on failure | Restart on stop | Restart on daemon start |
|---|---|---|---|
| `no` | ❌ | ❌ | ❌ |
| `always` | ✅ | ✅ | ✅ |
| `unless-stopped` | ✅ | ❌ (if manually stopped) | ✅ |
| `on-failure[:N]` | ✅ (up to N times) | ❌ | ✅ |

:::tip[Use `unless-stopped` for production services]
`unless-stopped` is the safest choice for long-running services. It restarts automatically on crashes and system reboots, but respects intentional `docker stop` commands.
:::

## Health Checks

Health checks let Docker determine if a container is truly ready to serve traffic.

### HEALTHCHECK in Dockerfile

```dockerfile
# HTTP health check via curl
HEALTHCHECK --interval=30s \
            --timeout=5s \
            --start-period=30s \
            --retries=3 \
  CMD curl -f http://localhost:3000/health || exit 1

# HTTP health check via wget (available in Alpine)
HEALTHCHECK --interval=30s --timeout=5s --retries=3 \
  CMD wget -qO- http://localhost:3000/health || exit 1

# Native check (no curl/wget dependency)
HEALTHCHECK --interval=30s --timeout=5s --retries=3 \
  CMD node -e "require('http').get('http://localhost:3000/health', r => process.exit(r.statusCode === 200 ? 0 : 1))"

# PostgreSQL
HEALTHCHECK --interval=10s --timeout=5s --retries=5 \
  CMD pg_isready -U app -d myapp || exit 1

# Redis
HEALTHCHECK --interval=10s --timeout=3s --retries=5 \
  CMD redis-cli ping | grep PONG || exit 1
```

### Health Check States

- **starting** — container started, within `start-period`
- **healthy** — last N checks succeeded
- **unhealthy** — last N checks failed
- **none** — no HEALTHCHECK defined

```bash
# View health status
docker ps --format "table {{.Names}}\t{{.Status}}"
docker inspect --format='{{.State.Health.Status}}' myapp
docker inspect --format='{{json .State.Health}}' myapp | jq
```

## Docker Secrets

### Swarm Secrets

```bash
# Create a secret
echo "supersecretpassword" | docker secret create db_password -
cat ./ssl.key | docker secret create ssl_key -

# Use in a service
docker service create \
  --name api \
  --secret db_password \
  --env DB_PASSWORD_FILE=/run/secrets/db_password \
  myapp/api
```

### Compose Secrets

```yaml
# compose.yml
services:
  api:
    secrets:
      - db_password
      - jwt_secret
    environment:
      # Application reads from file
      DB_PASSWORD_FILE: /run/secrets/db_password

secrets:
  db_password:
    file: ./secrets/db_password.txt   # read from file
  jwt_secret:
    environment: JWT_SECRET            # read from env var
```

The secret is mounted as a file at `/run/secrets/<name>` inside the container. The application reads it at startup instead of from an environment variable.

## Container Security

### Drop Capabilities

Containers run with a reduced set of Linux capabilities by default. Drop all and add back only what is needed.

```bash
# Drop all capabilities, add back only what's needed
docker run -d \
  --cap-drop ALL \
  --cap-add NET_BIND_SERVICE \  # bind to port < 1024
  nginx

# Common capabilities and their use:
# NET_BIND_SERVICE — bind privileged ports (< 1024)
# CHOWN — change file ownership
# DAC_OVERRIDE — bypass file permission checks
# SETUID, SETGID — change user/group ID
# SYS_PTRACE — used by debuggers (never in prod)
```

### Prevent Privilege Escalation

```bash
# Prevent setuid/setgid escalation
docker run -d \
  --security-opt no-new-privileges \
  nginx
```

### Read-Only Filesystem

```bash
# Mount root filesystem as read-only
# Explicitly allow writes to specific paths with tmpfs
docker run -d \
  --read-only \
  --tmpfs /tmp:noexec,nosuid,size=64m \
  --tmpfs /run:noexec,nosuid,size=32m \
  nginx
```

### Seccomp Profiles

```bash
# Use default seccomp profile (enabled by default)
docker run -d --security-opt seccomp=default nginx

# Use custom profile
docker run -d \
  --security-opt seccomp=/path/to/seccomp-profile.json \
  nginx

# Disable (dangerous — only for debugging)
docker run -d --security-opt seccomp=unconfined nginx
```

### Full Security Hardening Example

```yaml
# compose.yml
services:
  api:
    image: myapp/api:1.2.3
    read_only: true
    tmpfs:
      - /tmp:noexec,nosuid,size=64m
    security_opt:
      - no-new-privileges:true
      - seccomp:./seccomp-profile.json
    cap_drop:
      - ALL
    cap_add:
      - NET_BIND_SERVICE
    user: "1001:1001"
```

## Graceful Shutdown

Docker sends `SIGTERM` to the container's PID 1 when `docker stop` is called. After the `stop_grace_period` (default 10s), it sends `SIGKILL`.

### Node.js Example

```javascript
// Handle SIGTERM gracefully
process.on('SIGTERM', () => {
  console.log('SIGTERM received — shutting down gracefully');
  server.close(() => {
    console.log('HTTP server closed');
    db.end(() => {
      console.log('Database connections closed');
      process.exit(0);
    });
  });
});
```

### Compose Configuration

```yaml
services:
  api:
    stop_grace_period: 30s    # wait 30s before SIGKILL (default: 10s)
    stop_signal: SIGTERM      # signal to send (default: SIGTERM)
```

```bash
# Give the container 30 seconds to shut down
docker stop --time 30 myapp
```

## Monitoring

### cAdvisor

cAdvisor collects container metrics and exposes a Prometheus endpoint.

```yaml
services:
  cadvisor:
    image: gcr.io/cadvisor/cadvisor:latest
    ports:
      - "8080:8080"
    volumes:
      - /:/rootfs:ro
      - /var/run:/var/run:ro
      - /sys:/sys:ro
      - /var/lib/docker/:/var/lib/docker:ro
    devices:
      - /dev/kmsg:/dev/kmsg
    privileged: true
    restart: unless-stopped
```

### Docker Daemon Prometheus Metrics

Enable the experimental metrics endpoint in `daemon.json`:

```json
{
  "metrics-addr": "0.0.0.0:9323",
  "experimental": true
}
```

Prometheus scrape config:

```yaml
scrape_configs:
  - job_name: docker
    static_configs:
      - targets: ["docker-host:9323"]
```

### Grafana + Prometheus + cAdvisor Stack

```yaml
services:
  prometheus:
    image: prom/prometheus:latest
    volumes:
      - ./prometheus.yml:/etc/prometheus/prometheus.yml:ro
      - promdata:/prometheus
    ports:
      - "9090:9090"

  grafana:
    image: grafana/grafana:latest
    ports:
      - "3000:3000"
    volumes:
      - grafanadata:/var/grafana

  cadvisor:
    image: gcr.io/cadvisor/cadvisor:latest
    volumes:
      - /:/rootfs:ro
      - /var/run:/var/run:ro
      - /sys:/sys:ro
      - /var/lib/docker:/var/lib/docker:ro
    privileged: true

volumes:
  promdata:
  grafanadata:
```

## Production Hardening Checklist

- [ ] Pin image to specific tag or digest
- [ ] Multi-stage build — minimal runtime image
- [ ] Non-root user
- [ ] Read-only root filesystem with targeted tmpfs
- [ ] `--cap-drop ALL` + minimal `--cap-add`
- [ ] `--security-opt no-new-privileges`
- [ ] Resource limits (`--memory`, `--cpus`, `--pids-limit`)
- [ ] Log rotation configured (`max-size`, `max-file`)
- [ ] `restart: unless-stopped`
- [ ] HEALTHCHECK defined
- [ ] Secrets via files or secret manager — never environment variables
- [ ] No sensitive data in image layers
- [ ] Image scanned with Trivy or Docker Scout in CI
- [ ] Network segmentation — internal networks for databases
- [ ] `stop_grace_period` set appropriately
- [ ] Monitoring with cAdvisor + Prometheus

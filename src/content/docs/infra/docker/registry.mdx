# Docker Registry

A Docker registry stores and distributes container images. Docker Hub is the default public registry, but most production teams use private registries for security, performance, and compliance.

---

## Image Naming Convention

Understanding the full image name format is essential for working with multiple registries.

```
[registry/][namespace/]name[:tag][@digest]

Examples:
  nginx:1.25-alpine
  ↑ implicit docker.io/library/nginx:1.25-alpine

  myuser/myapp:1.2.3
  ↑ implicit docker.io/myuser/myapp:1.2.3

  ghcr.io/myorg/myapp:latest
  ↑ GitHub Container Registry

  123456789.dkr.ecr.us-east-1.amazonaws.com/myapp:v1.2.3
  ↑ AWS ECR

  europe-west1-docker.pkg.dev/myproject/myrepo/myapp:prod
  ↑ Google Artifact Registry
```

## Docker Hub

Docker Hub is the default public registry at `docker.io`. Free accounts get unlimited public repositories and one private repository.

### Authentication

```bash
# Log in (prompts for username and password)
docker login

# Log in to Docker Hub explicitly
docker login docker.io

# Log in with credentials (avoid — password visible in shell history)
docker login -u myusername -p mypassword

# Log in with stdin (safer in CI)
echo "$DOCKER_PASSWORD" | docker login -u "$DOCKER_USERNAME" --password-stdin

# Verify login
cat ~/.docker/config.json

# Log out
docker logout
```

### Push and Pull

```bash
# Tag an image for Docker Hub
docker build -t myusername/myapp:1.2.3 .
docker tag myusername/myapp:1.2.3 myusername/myapp:latest

# Push to Docker Hub
docker push myusername/myapp:1.2.3
docker push myusername/myapp:latest

# Push all tags at once
docker push --all-tags myusername/myapp

# Pull from Docker Hub
docker pull myusername/myapp:1.2.3

# Pull with digest (fully immutable reference)
docker pull myusername/myapp@sha256:abc123def456...
```

### Tagging Best Practices

```bash
# Semantic versioning tags
docker tag myapp:build myusername/myapp:2.1.3
docker tag myapp:build myusername/myapp:2.1
docker tag myapp:build myusername/myapp:2
docker tag myapp:build myusername/myapp:latest

# Git SHA tag for traceability
GIT_SHA=$(git rev-parse --short HEAD)
docker tag myapp:build myusername/myapp:$GIT_SHA

# Environment + version tag
docker tag myapp:build myusername/myapp:prod-2.1.3
```

## GitHub Container Registry (GHCR)

GHCR (`ghcr.io`) is tightly integrated with GitHub repositories and Actions. Images inherit repository permissions.

### Authentication

```bash
# Authenticate with a Personal Access Token (PAT)
# Required scopes: read:packages, write:packages, delete:packages
echo "$GITHUB_TOKEN" | docker login ghcr.io -u USERNAME --password-stdin

# In GitHub Actions, use the built-in GITHUB_TOKEN
echo "${{ secrets.GITHUB_TOKEN }}" | docker login ghcr.io -u ${{ github.actor }} --password-stdin
```

### Push and Pull

```bash
# Build and tag for GHCR
docker build -t ghcr.io/myorg/myapp:1.2.3 .

# Push
docker push ghcr.io/myorg/myapp:1.2.3

# Pull
docker pull ghcr.io/myorg/myapp:1.2.3
```

### GitHub Actions Workflow

```yaml
# .github/workflows/docker.yml
name: Build and Push

on:
  push:
    branches: [main]
    tags: ['v*']

jobs:
  build:
    runs-on: ubuntu-latest
    permissions:
      contents: read
      packages: write

    steps:
      - uses: actions/checkout@v4

      - name: Log in to GHCR
        uses: docker/login-action@v3
        with:
          registry: ghcr.io
          username: ${{ github.actor }}
          password: ${{ secrets.GITHUB_TOKEN }}

      - name: Extract metadata
        id: meta
        uses: docker/metadata-action@v5
        with:
          images: ghcr.io/${{ github.repository }}
          tags: |
            type=semver,pattern={{version}}
            type=semver,pattern={{major}}.{{minor}}
            type=sha

      - name: Build and push
        uses: docker/build-push-action@v5
        with:
          context: .
          push: true
          tags: ${{ steps.meta.outputs.tags }}
          labels: ${{ steps.meta.outputs.labels }}
          cache-from: type=gha
          cache-to: type=gha,mode=max
```

## AWS Elastic Container Registry (ECR)

ECR is the AWS-native private registry. Authentication uses the AWS CLI to generate a temporary token.

```bash
# Authenticate (valid for 12 hours)
aws ecr get-login-password --region us-east-1 \
  | docker login --username AWS \
    --password-stdin 123456789.dkr.ecr.us-east-1.amazonaws.com

# Create a repository (one-time)
aws ecr create-repository \
  --repository-name myapp \
  --region us-east-1

# Tag and push
ECR_URI="123456789.dkr.ecr.us-east-1.amazonaws.com/myapp"
docker build -t $ECR_URI:1.2.3 .
docker push $ECR_URI:1.2.3

# Pull
docker pull $ECR_URI:1.2.3

# Enable image scanning on push
aws ecr put-image-scanning-configuration \
  --repository-name myapp \
  --image-scanning-configuration scanOnPush=true
```

### ECR Lifecycle Policy (Cost Management)

```json
{
  "rules": [
    {
      "rulePriority": 1,
      "description": "Keep only last 10 images per tag prefix",
      "selection": {
        "tagStatus": "tagged",
        "tagPrefixList": ["dev-"],
        "countType": "imageCountMoreThan",
        "countNumber": 10
      },
      "action": { "type": "expire" }
    },
    {
      "rulePriority": 2,
      "description": "Expire untagged images after 7 days",
      "selection": {
        "tagStatus": "untagged",
        "countType": "sinceImagePushed",
        "countUnit": "days",
        "countNumber": 7
      },
      "action": { "type": "expire" }
    }
  ]
}
```

## Google Artifact Registry

```bash
# Configure gcloud authentication
gcloud auth configure-docker europe-west1-docker.pkg.dev

# Create repository
gcloud artifacts repositories create myrepo \
  --repository-format=docker \
  --location=europe-west1

# Build, tag, push
REPO="europe-west1-docker.pkg.dev/myproject/myrepo/myapp"
docker build -t $REPO:1.2.3 .
docker push $REPO:1.2.3
```

## Private Registry (Self-Hosted)

Run your own registry using the official `registry:2` image.

```bash
# Start a basic private registry on port 5000
docker run -d \
  --name registry \
  -p 5000:5000 \
  -v regdata:/var/lib/registry \
  registry:2

# Push to local registry
docker tag myapp:latest localhost:5000/myapp:latest
docker push localhost:5000/myapp:latest

# Pull from local registry
docker pull localhost:5000/myapp:latest

# List repositories
curl http://localhost:5000/v2/_catalog
# {"repositories":["myapp"]}

# List tags for a repository
curl http://localhost:5000/v2/myapp/tags/list
```

### Registry with TLS and Authentication

```yaml
# compose.yml for a production-grade self-hosted registry
services:
  registry:
    image: registry:2
    ports:
      - "5000:5000"
    environment:
      REGISTRY_HTTP_TLS_CERTIFICATE: /certs/domain.crt
      REGISTRY_HTTP_TLS_KEY: /certs/domain.key
      REGISTRY_AUTH: htpasswd
      REGISTRY_AUTH_HTPASSWD_PATH: /auth/htpasswd
      REGISTRY_AUTH_HTPASSWD_REALM: "Registry Realm"
    volumes:
      - regdata:/var/lib/registry
      - ./certs:/certs:ro
      - ./auth:/auth:ro

volumes:
  regdata:
```

## Image Scanning

### Docker Scout

```bash
# Scan a local image
docker scout cves myapp:latest

# Scan with policy check
docker scout policy myapp:latest

# Compare two images
docker scout compare myapp:1.2.2 --to myapp:1.2.3

# Show SBOMs
docker scout sbom myapp:latest
```

### Trivy (Open Source Scanner)

```bash
# Install trivy
brew install aquasecurity/trivy/trivy   # macOS
apt-get install -y trivy                 # Debian/Ubuntu

# Scan a local image
trivy image myapp:latest

# Scan and fail on CRITICAL and HIGH vulnerabilities
trivy image --exit-code 1 --severity CRITICAL,HIGH myapp:latest

# Scan a remote image
trivy image ghcr.io/myorg/myapp:1.2.3

# Output as JSON for CI processing
trivy image --format json --output trivy-results.json myapp:latest

# Scan filesystem (for use in multi-stage Dockerfiles)
trivy fs --security-checks vuln,secret .
```

In CI:

```yaml
# GitHub Actions step
- name: Scan image with Trivy
  uses: aquasecurity/trivy-action@master
  with:
    image-ref: myapp:${{ github.sha }}
    format: sarif
    output: trivy-results.sarif
    severity: CRITICAL,HIGH
    exit-code: '1'
```

## Image Signing (cosign)

Image signing ensures the images you pull are exactly what was built and signed by your CI system.

```bash
# Install cosign
brew install cosign

# Generate a key pair
cosign generate-key-pair

# Sign an image (after pushing)
cosign sign --key cosign.key ghcr.io/myorg/myapp:1.2.3

# Verify a signature
cosign verify \
  --key cosign.pub \
  ghcr.io/myorg/myapp:1.2.3

# Keyless signing with Sigstore (OIDC-based, no key management)
cosign sign ghcr.io/myorg/myapp:1.2.3   # Uses OIDC identity

# Attach SBOM
cosign attach sbom \
  --sbom sbom.spdx \
  ghcr.io/myorg/myapp:1.2.3
```

## Multi-Architecture Builds

Build images for multiple CPU architectures (amd64, arm64, armv7) from a single build command.

```bash
# Create a new builder with multi-platform support
docker buildx create \
  --name multibuilder \
  --driver docker-container \
  --use

# Inspect the builder (bootstrap downloads BuildKit)
docker buildx inspect --bootstrap

# List available platforms
docker buildx ls

# Build for multiple platforms and push
docker buildx build \
  --platform linux/amd64,linux/arm64,linux/arm/v7 \
  --tag myregistry/myapp:1.2.3 \
  --tag myregistry/myapp:latest \
  --push \
  .

# Build for multiple platforms without pushing (local cache)
docker buildx build \
  --platform linux/amd64,linux/arm64 \
  --tag myapp:local \
  --load \
  .

# Inspect the manifest list
docker buildx imagetools inspect myregistry/myapp:1.2.3
```

:::tip[QEMU for cross-compilation]
When building for ARM on an x86 host (or vice versa), Docker uses QEMU emulation. Enable it with:
```bash
docker run --rm --privileged tonistiigi/binfmt --install all
```
Cross-compilation is slower than native; consider using AWS Graviton or Apple M-series runners for native ARM builds.
:::

## Analyzing Image Size with dive

`dive` is a tool for exploring a Docker image's layers and finding wasted space.

```bash
# Install
brew install dive

# Analyze an image
dive myapp:latest

# Run in CI mode (fails if wasted space exceeds threshold)
dive --ci myapp:latest
```

In the dive interactive UI:
- Arrow keys navigate layers
- Tab switches between layer view and file tree
- Space bar toggle file changes per layer
- Ctrl+U shows unmodified files

### Layer Squashing

```bash
# Using docker export/import (flattens all layers into one)
docker create --name temp myapp:latest
docker export temp | docker import - myapp:squashed
docker rm temp

# Multi-stage build naturally squashes by only copying what you need
FROM builder AS final
COPY --from=builder /app/dist /dist
# Only the final COPY creates a layer — no build tools included
```

:::caution[Squashing breaks layer sharing]
Squashing layers saves disk space for a single image but breaks layer sharing between images that would otherwise share base layers. For most use cases, a well-constructed multi-stage Dockerfile is better than post-build squashing.
:::

# Docker Registry

A Docker registry is a storage and distribution service for container images. Understanding tagging conventions, authentication, and multi-arch builds is essential for shipping production-grade images.

## Image Naming and Tagging

The full image reference format is:

```
[REGISTRY/][NAMESPACE/]NAME[:TAG][@DIGEST]
```

Examples:

```bash
nginx                              # Docker Hub official image
nginx:1.27-alpine                  # specific tag
library/nginx:1.27-alpine          # fully qualified Docker Hub path
ubuntu:22.04                       # official image with tag
myuser/myapp:1.2.3                 # Docker Hub user namespace
ghcr.io/myorg/myapp:latest         # GitHub Container Registry
123456789.dkr.ecr.us-east-1.amazonaws.com/myapp:v1.0.0   # AWS ECR
us-central1-docker.pkg.dev/project/repo/myapp:latest      # Google AR

# Reference by immutable digest (never changes, even if tag moves)
nginx@sha256:a5127daff3d6f4606be3100a252419bfa84fd6ee5cd74d0feaca1a5068f97dcf
```

## Docker Hub

Docker Hub (`hub.docker.com`) is the default public registry.

### Authentication

```bash
# Log in interactively
docker login

# Log in with credentials (avoid — password visible in shell history)
docker login -u myuser -p mypassword

# Log in using stdin (secure — no shell history)
echo "$DOCKER_PASSWORD" | docker login -u "$DOCKER_USERNAME" --password-stdin

# Log out
docker logout
```

:::caution[Protect your Docker Hub token]
Create an access token at `hub.docker.com/settings/security` instead of using your account password. Scope it to Read/Write and store it in a secret manager or CI environment variable.
:::

### Push and Pull

```bash
# Pull a public image
docker pull nginx:1.27-alpine

# Tag a local image for Docker Hub
docker tag myapp:latest myuser/myapp:1.2.3
docker tag myapp:latest myuser/myapp:latest

# Push to Docker Hub
docker push myuser/myapp:1.2.3
docker push myuser/myapp:latest

# Push all tags at once
docker push --all-tags myuser/myapp
```

### Build and Push in One Step

```bash
docker build -t myuser/myapp:1.2.3 -t myuser/myapp:latest .
docker push myuser/myapp:1.2.3
docker push myuser/myapp:latest

# Or with buildx in a single command
docker buildx build \
  -t myuser/myapp:1.2.3 \
  -t myuser/myapp:latest \
  --push .
```

## GitHub Container Registry (ghcr.io)

GHCR is integrated with GitHub Actions and supports fine-grained access control via `GITHUB_TOKEN`.

```bash
# Authenticate with a personal access token (PAT) with read:packages scope
echo "$GITHUB_TOKEN" | docker login ghcr.io -u USERNAME --password-stdin

# Tag and push
docker tag myapp:latest ghcr.io/myorg/myapp:1.2.3
docker push ghcr.io/myorg/myapp:1.2.3
```

### In GitHub Actions

```yaml
# .github/workflows/docker.yml
name: Build and Push

on:
  push:
    branches: [main]
    tags: ["v*"]

jobs:
  build:
    runs-on: ubuntu-latest
    permissions:
      contents: read
      packages: write

    steps:
      - uses: actions/checkout@v4

      - name: Log in to GHCR
        uses: docker/login-action@v3
        with:
          registry: ghcr.io
          username: ${{ github.actor }}
          password: ${{ secrets.GITHUB_TOKEN }}

      - name: Extract metadata
        id: meta
        uses: docker/metadata-action@v5
        with:
          images: ghcr.io/${{ github.repository }}
          tags: |
            type=semver,pattern={{version}}
            type=semver,pattern={{major}}.{{minor}}
            type=sha,prefix=sha-

      - name: Build and push
        uses: docker/build-push-action@v5
        with:
          context: .
          push: true
          tags: ${{ steps.meta.outputs.tags }}
          labels: ${{ steps.meta.outputs.labels }}
          cache-from: type=gha
          cache-to: type=gha,mode=max
```

## AWS Elastic Container Registry (ECR)

ECR is a fully managed private registry integrated with AWS IAM.

```bash
# Authenticate (requires AWS CLI configured)
aws ecr get-login-password --region us-east-1 | \
  docker login --username AWS --password-stdin \
  123456789012.dkr.ecr.us-east-1.amazonaws.com

# Create a repository
aws ecr create-repository --repository-name myapp --region us-east-1

# Tag and push
docker tag myapp:latest \
  123456789012.dkr.ecr.us-east-1.amazonaws.com/myapp:1.2.3

docker push 123456789012.dkr.ecr.us-east-1.amazonaws.com/myapp:1.2.3
```

### ECR Lifecycle Policy (prevent unbounded growth)

```json
{
  "rules": [
    {
      "rulePriority": 1,
      "description": "Keep last 10 tagged images",
      "selection": {
        "tagStatus": "tagged",
        "tagPrefixList": ["v"],
        "countType": "imageCountMoreThan",
        "countNumber": 10
      },
      "action": { "type": "expire" }
    },
    {
      "rulePriority": 2,
      "description": "Remove untagged after 7 days",
      "selection": {
        "tagStatus": "untagged",
        "countType": "sinceImagePushed",
        "countUnit": "days",
        "countNumber": 7
      },
      "action": { "type": "expire" }
    }
  ]
}
```

## Self-Hosted Private Registry

For air-gapped environments or cost control, run the official `registry:2` image.

```bash
# Basic registry (no auth, not for production)
docker run -d \
  -p 5000:5000 \
  --name registry \
  -v registrydata:/var/lib/registry \
  --restart always \
  registry:2

# Push to local registry
docker tag myapp:latest localhost:5000/myapp:1.2.3
docker push localhost:5000/myapp:1.2.3
docker pull localhost:5000/myapp:1.2.3
```

```yaml
# compose.yml for a registry with htpasswd auth
services:
  registry:
    image: registry:2
    ports:
      - "5000:5000"
    environment:
      REGISTRY_AUTH: htpasswd
      REGISTRY_AUTH_HTPASSWD_REALM: Registry
      REGISTRY_AUTH_HTPASSWD_PATH: /auth/htpasswd
      REGISTRY_STORAGE_FILESYSTEM_ROOTDIRECTORY: /data
    volumes:
      - ./auth:/auth:ro
      - registrydata:/data
    restart: always

volumes:
  registrydata:
```

## Image Scanning

### Docker Scout

```bash
# Enable Docker Scout (free tier available)
docker scout cves nginx:1.27-alpine

# Compare two images
docker scout compare nginx:1.27-alpine nginx:1.26-alpine

# Quickview
docker scout quickview myapp:latest

# Recommendations
docker scout recommendations myapp:latest
```

### Trivy

```bash
# Install
brew install aquasecurity/trivy/trivy

# Scan an image
trivy image nginx:1.27-alpine

# Scan and fail on HIGH or CRITICAL
trivy image --exit-code 1 --severity HIGH,CRITICAL myapp:latest

# Scan in CI (JSON output)
trivy image --format json --output trivy-report.json myapp:latest

# Scan a Dockerfile
trivy config Dockerfile
```

## Image Signing with cosign

```bash
# Install cosign
brew install cosign

# Generate a key pair
cosign generate-key-pair

# Sign an image (after pushing)
cosign sign --key cosign.key myregistry/myapp:1.2.3

# Verify a signature
cosign verify --key cosign.pub myregistry/myapp:1.2.3

# Keyless signing (using OIDC identity — GitHub Actions, etc.)
cosign sign myregistry/myapp:1.2.3
```

## Multi-Architecture Builds

Build images for `linux/amd64` (x86_64) and `linux/arm64` (Apple Silicon, Graviton) simultaneously using `docker buildx`.

```bash
# Create and use a new builder with multi-arch support
docker buildx create --name multiarch --use --bootstrap

# List available builders
docker buildx ls

# Build for multiple platforms and push
docker buildx build \
  --platform linux/amd64,linux/arm64 \
  -t myuser/myapp:1.2.3 \
  -t myuser/myapp:latest \
  --push .

# Build and load locally (single platform only)
docker buildx build \
  --platform linux/amd64 \
  -t myapp:latest \
  --load .

# Inspect the multi-arch manifest
docker buildx imagetools inspect myuser/myapp:latest
```

### Multi-arch in GitHub Actions

```yaml
- name: Set up QEMU (for cross-compilation)
  uses: docker/setup-qemu-action@v3

- name: Set up Docker Buildx
  uses: docker/setup-buildx-action@v3

- name: Build and push multi-arch
  uses: docker/build-push-action@v5
  with:
    context: .
    platforms: linux/amd64,linux/arm64
    push: true
    tags: ghcr.io/myorg/myapp:latest
    cache-from: type=gha
    cache-to: type=gha,mode=max
```

## Image Size Optimization

### dive — Layer Inspection Tool

```bash
# Install
brew install dive

# Analyse image layers interactively
dive myapp:latest

# CI mode — fail if wasted space > 10%
CI=true dive myapp:latest
```

### Size Reduction Techniques

```bash
# Check image size
docker image inspect myapp:latest --format='{{.Size}}' | \
  awk '{printf "%.1f MB\n", $1/1048576}'

# Compare sizes
docker images --format "table {{.Repository}}\t{{.Tag}}\t{{.Size}}" | \
  grep myapp
```

```dockerfile
# 1. Use Alpine or distroless as base
FROM node:22-alpine   # 45MB vs 900MB for node:22

# 2. Multi-stage — exclude dev dependencies
FROM node:22-alpine AS builder
RUN npm ci   # includes devDependencies

FROM node:22-alpine AS runtime
COPY --from=builder /app/dist ./dist
COPY --from=builder /app/node_modules ./node_modules  # or only prod deps

# 3. Clean package manager cache in same layer
RUN apt-get install -y curl && rm -rf /var/lib/apt/lists/*
RUN apk add --no-cache curl   # Alpine never caches

# 4. Remove unnecessary files with .dockerignore
# (tests, docs, CI configs, dev tools)
```

### Layer Squashing

```bash
# Squash all layers into one (BuildKit)
docker buildx build --squash -t myapp:squashed .

# Note: squashing loses layer sharing benefits.
# Only useful for shipping the absolute smallest single image.
```

# Docker Registry

A Docker registry is a storage and distribution system for container images. Understanding registries, image tagging conventions, and multi-architecture builds is essential for CI/CD pipelines and production deployments.

---

## Image Naming and Tagging Convention

Docker image names follow a consistent pattern:

```
[registry/][namespace/]name[:tag][@digest]

Examples:
nginx                               # Docker Hub official image, latest tag
nginx:1.25-alpine                   # Docker Hub official, specific tag
library/nginx:1.25                  # Fully qualified Docker Hub official
myuser/myapp:v1.2.3                 # Docker Hub user image
myuser/myapp:latest                 # Docker Hub, latest tag
ghcr.io/org/myapp:main              # GitHub Container Registry
123456789.dkr.ecr.us-east-1.amazonaws.com/myapp:v2  # AWS ECR
europe-west1-docker.pkg.dev/project/repo/app:1.0    # Google Artifact Registry
registry.example.com:5000/myapp:dev # Private registry
```

### Tagging Best Practices

```bash
# Build and tag simultaneously
docker build -t myapp:1.2.3 .
docker build -t myapp:1.2.3 -t myapp:latest .

# Tag an existing image with a new name
docker tag myapp:1.2.3 myuser/myapp:1.2.3
docker tag myapp:1.2.3 myuser/myapp:latest
docker tag myapp:1.2.3 ghcr.io/myorg/myapp:1.2.3
docker tag myapp:1.2.3 myapp:stable
```

**Recommended tagging strategy:**
- `myapp:1.2.3` — exact semantic version (immutable)
- `myapp:1.2` — minor version (updated with patches)
- `myapp:1` — major version (updated with minor/patches)
- `myapp:latest` — latest stable (updated with each release)
- `myapp:main` or `myapp:edge` — latest from default branch (dev/unstable)
- `myapp:sha-a1b2c3d` — git commit SHA for traceability

:::caution[Never use `latest` in production]
`latest` is just a convention — it's not automatically the newest image. Using `:latest` in production deployments means you can't tell which version is running and rollbacks are unreliable. Always pin to a specific version tag.
:::

## Docker Hub

[Docker Hub](https://hub.docker.com) is the default public registry. It hosts official images, public repositories, and private repositories.

### Authentication

```bash
# Login to Docker Hub
docker login
# Enter your Docker Hub username and password (or access token)

# Login with credentials (for scripts)
echo $DOCKER_PASSWORD | docker login -u $DOCKER_USERNAME --password-stdin

# Logout
docker logout
```

:::tip[Use Access Tokens]
Generate a Docker Hub Access Token (Account Settings → Security → New Access Token) instead of using your password. Access tokens can be scoped and revoked independently.
:::

### Pushing and Pulling

```bash
# Pull a public image
docker pull nginx:alpine
docker pull ubuntu:22.04

# Push to your Docker Hub namespace
docker build -t myusername/myapp:1.0.0 .
docker push myusername/myapp:1.0.0

# Push multiple tags
docker push myusername/myapp:1.0.0
docker push myusername/myapp:latest

# Pull your image on another machine
docker pull myusername/myapp:1.0.0

# Make a repository private (from Docker Hub web UI or CLI)
# In docker-compose pull with auth:
docker compose pull
```

### Docker Hub Rate Limits

Anonymous pulls: 100 pulls per 6 hours per IP
Authenticated free accounts: 200 pulls per 6 hours
Paid subscriptions: higher limits

```bash
# Check your remaining rate limit
TOKEN=$(curl -s "https://auth.docker.io/token?service=registry.docker.io&scope=repository:ratelimitpreview/test:pull" | jq -r .token)
curl -s --head -H "Authorization: Bearer $TOKEN" https://registry-1.docker.io/v2/ratelimitpreview/test/manifests/latest \
  | grep -i ratelimit
```

## GitHub Container Registry (GHCR)

GHCR is tightly integrated with GitHub Actions and uses GitHub's permission model.

```bash
# Authenticate with a Personal Access Token (PAT)
# Required scopes: read:packages, write:packages, delete:packages
echo $GITHUB_TOKEN | docker login ghcr.io -u USERNAME --password-stdin

# Build and push
docker build -t ghcr.io/myorg/myapp:latest .
docker push ghcr.io/myorg/myapp:latest

# Pull
docker pull ghcr.io/myorg/myapp:latest
```

### GitHub Actions Integration

```yaml
# .github/workflows/publish.yml
name: Publish Docker Image

on:
  push:
    branches: [main]
    tags: ["v*"]

jobs:
  publish:
    runs-on: ubuntu-latest
    permissions:
      contents: read
      packages: write

    steps:
      - uses: actions/checkout@v4

      - name: Log in to GHCR
        uses: docker/login-action@v3
        with:
          registry: ghcr.io
          username: ${{ github.actor }}
          password: ${{ secrets.GITHUB_TOKEN }}  # No extra secret needed!

      - name: Extract metadata
        id: meta
        uses: docker/metadata-action@v5
        with:
          images: ghcr.io/${{ github.repository }}
          tags: |
            type=ref,event=branch
            type=semver,pattern={{version}}
            type=semver,pattern={{major}}.{{minor}}
            type=sha

      - name: Build and push
        uses: docker/build-push-action@v5
        with:
          context: .
          push: true
          tags: ${{ steps.meta.outputs.tags }}
          labels: ${{ steps.meta.outputs.labels }}
          cache-from: type=gha
          cache-to: type=gha,mode=max
```

## AWS Elastic Container Registry (ECR)

ECR is a fully managed private container registry integrated with AWS IAM.

```bash
# Authenticate (requires AWS CLI configured)
aws ecr get-login-password --region us-east-1 \
  | docker login --username AWS --password-stdin \
    123456789.dkr.ecr.us-east-1.amazonaws.com

# Create a repository
aws ecr create-repository --repository-name myapp --region us-east-1

# Build and push
docker build -t 123456789.dkr.ecr.us-east-1.amazonaws.com/myapp:latest .
docker push 123456789.dkr.ecr.us-east-1.amazonaws.com/myapp:latest

# Pull
docker pull 123456789.dkr.ecr.us-east-1.amazonaws.com/myapp:latest
```

ECR supports **lifecycle policies** to automatically clean up old images:

```json
{
  "rules": [
    {
      "rulePriority": 1,
      "description": "Keep last 10 images",
      "selection": {
        "tagStatus": "any",
        "countType": "imageCountMoreThan",
        "countNumber": 10
      },
      "action": {
        "type": "expire"
      }
    }
  ]
}
```

## Google Artifact Registry

```bash
# Configure Docker authentication
gcloud auth configure-docker europe-west1-docker.pkg.dev
```

## Private Registry

Run your own registry using the official `registry` image.

```bash
# Start a local private registry on port 5000
docker run -d \
  --name registry \
  -p 5000:5000 \
  --restart unless-stopped \
  -v registry-data:/var/lib/registry \
  registry:2

# Push to local registry
docker tag myapp:latest localhost:5000/myapp:latest
docker push localhost:5000/myapp:latest

# Pull from local registry
docker pull localhost:5000/myapp:latest

# List images in the registry
curl http://localhost:5000/v2/_catalog
curl http://localhost:5000/v2/myapp/tags/list
```

### Secured Private Registry

```yaml
# compose.yml for production registry
services:
  registry:
    image: registry:2
    ports:
      - "5000:5000"
    environment:
      REGISTRY_AUTH: htpasswd
      REGISTRY_AUTH_HTPASSWD_PATH: /auth/htpasswd
      REGISTRY_AUTH_HTPASSWD_REALM: "Registry Realm"
      REGISTRY_HTTP_TLS_CERTIFICATE: /certs/domain.crt
      REGISTRY_HTTP_TLS_KEY: /certs/domain.key
    volumes:
      - registry-data:/var/lib/registry
      - ./auth:/auth
      - ./certs:/certs
    restart: unless-stopped

volumes:
  registry-data:
```

```bash
# Create htpasswd file
docker run --rm httpd htpasswd -Bbn username password > auth/htpasswd

# Login to secured registry
docker login myregistry.example.com:5000
```

## Image Scanning

Scan images for vulnerabilities before pushing to production.

### Docker Scout

```bash
# Docker's built-in scanner (integrated into Docker Desktop)
docker scout quickview myapp:latest
docker scout cves myapp:latest
docker scout cves --only-fixed myapp:latest       # Only show patchable CVEs
docker scout recommendations myapp:latest
docker scout compare myapp:v1 myapp:v2            # Compare two images
```

### Trivy

```bash
# Install trivy
brew install aquasecurity/trivy/trivy

# Scan an image
trivy image nginx:latest
trivy image --severity HIGH,CRITICAL myapp:latest

# Scan and exit with error code on HIGH/CRITICAL
trivy image --exit-code 1 --severity HIGH,CRITICAL myapp:latest

# Scan in CI
docker run --rm aquasec/trivy image myapp:latest
```

## Image Signing with Cosign

[Cosign](https://github.com/sigstore/cosign) is part of the Sigstore project for supply chain security.

```bash
# Install cosign
brew install cosign

# Generate a key pair
cosign generate-key-pair

# Sign an image after pushing
cosign sign --key cosign.key myregistry/myapp:1.0.0

# Verify the signature
cosign verify --key cosign.pub myregistry/myapp:1.0.0

# Keyless signing (uses OIDC identity — works in GitHub Actions)
cosign sign ghcr.io/myorg/myapp:latest
```

## Multi-Architecture Builds

Build images for multiple CPU architectures (amd64 for servers, arm64 for Apple Silicon/Graviton).

```bash
# Set up a multi-platform builder
docker buildx create --name multiarch --driver docker-container --use
docker buildx inspect --bootstrap

# Build for multiple platforms and push
docker buildx build \
  --platform linux/amd64,linux/arm64,linux/arm/v7 \
  --tag myregistry/myapp:latest \
  --push \
  .

# Build for local testing (single platform)
docker buildx build \
  --platform linux/arm64 \
  --tag myapp:arm64 \
  --load \      # Load into local docker images
  .

# Inspect the manifest
docker buildx imagetools inspect myregistry/myapp:latest

# List available platforms for an image
docker manifest inspect myregistry/myapp:latest | jq '.manifests[].platform'
```

### Multi-Arch in GitHub Actions

```yaml
- name: Set up QEMU (for cross-platform emulation)
  uses: docker/setup-qemu-action@v3

- name: Set up Docker Buildx
  uses: docker/setup-buildx-action@v3

- name: Build and push multi-arch
  uses: docker/build-push-action@v5
  with:
    platforms: linux/amd64,linux/arm64
    push: true
    tags: ghcr.io/myorg/myapp:latest
```

## Image Size Optimization

### Analyze with `dive`

```bash
# Install dive
brew install dive

# Analyze an image's layers
dive myapp:latest

# CI-friendly check for wasted space
dive --ci myapp:latest
```

### Size Reduction Techniques

```dockerfile
# 1. Use minimal base images
FROM node:20-alpine        # ~60 MB vs node:20 (~1 GB)
FROM python:3.12-slim      # ~150 MB vs python:3.12 (~900 MB)

# 2. Combine RUN commands to avoid extra layers
RUN apt-get update \
    && apt-get install -y --no-install-recommends curl \
    && rm -rf /var/lib/apt/lists/*

# 3. Multi-stage builds — only ship what's needed at runtime
FROM node:20-alpine AS builder
RUN npm ci && npm run build

FROM node:20-alpine AS runner
COPY --from=builder /app/dist ./dist

# 4. Use .dockerignore to reduce build context
# 5. Remove dev dependencies
RUN npm ci --only=production

# 6. Clear package manager cache
RUN pip install -r requirements.txt && pip cache purge
```

```bash
# Check image sizes
docker images --format "table {{.Repository}}\t{{.Tag}}\t{{.Size}}"

# Inspect each layer's contribution
docker history --no-trunc myapp:latest
```

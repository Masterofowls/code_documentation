# Docker Registry

A **registry** is a stateless, highly scalable server-side application that stores and distributes Docker images. Understanding how to work with registries is essential for any deployment pipeline.

## Image Naming Convention

```
[registry/][namespace/]name[:tag][@digest]

# Examples:
nginx                                       # Docker Hub official: library/nginx:latest
nginx:1.25-alpine                           # Docker Hub official, pinned tag
myuser/myapp:v2.1.0                         # Docker Hub personal namespace
docker.io/myuser/myapp:v2.1.0              # Explicit Docker Hub
ghcr.io/myorg/myapp:main                   # GitHub Container Registry
123456789.dkr.ecr.us-east-1.amazonaws.com/myapp:prod  # AWS ECR
europe-west1-docker.pkg.dev/proj/repo/app:v1          # Google Artifact Registry
registry.example.com:5000/myapp:latest     # Private registry with port
myapp@sha256:a1b2c3d4e5f6...              # Digest pinning
```

## Docker Hub

Docker Hub is the default public registry at `hub.docker.com`.

### Authentication

```bash
# Login to Docker Hub
docker login

# Login with credentials (avoid — password in shell history)
docker login -u myusername -p mypassword

# Login with token (recommended — create at hub.docker.com/settings/security)
docker login -u myusername --password-stdin <<< "dckr_pat_XXXX"

# Or pipe from environment variable
echo "$DOCKER_PAT" | docker login -u "$DOCKER_USERNAME" --password-stdin

# Logout
docker logout

# Check current login
cat ~/.docker/config.json
```

### Push to Docker Hub

```bash
# Build and tag for Docker Hub
docker build -t myusername/myapp:v1.0.0 .
docker build -t myusername/myapp:latest .

# Or tag an existing image
docker tag myapp:local myusername/myapp:v1.0.0
docker tag myapp:local myusername/myapp:latest

# Push
docker push myusername/myapp:v1.0.0
docker push myusername/myapp:latest

# Push all tags at once
docker push myusername/myapp --all-tags

# Pull from Docker Hub
docker pull myusername/myapp:v1.0.0
docker pull myusername/myapp:latest
```

### Tagging Strategy

```bash
# Semantic versioning
docker tag myapp:build myusername/myapp:2.1.0
docker tag myapp:build myusername/myapp:2.1
docker tag myapp:build myusername/myapp:2
docker tag myapp:build myusername/myapp:latest

# Git-based tagging
docker tag myapp:build myusername/myapp:$(git rev-parse --short HEAD)
docker tag myapp:build myusername/myapp:$(git describe --tags --always)

# Branch-based (for CI)
docker tag myapp:build myusername/myapp:main
docker tag myapp:build myusername/myapp:feature-auth
```

:::tip[Immutable Tags]
Never push to a tag that already exists (except `latest` and branch tags). Use immutable version tags for deployments so you can always roll back to exactly the same image. Many registries support tag immutability policies.
:::

## GitHub Container Registry (GHCR)

GHCR is GitHub's container registry at `ghcr.io`. Images are linked to GitHub repositories.

```bash
# Authenticate with GitHub Personal Access Token or GITHUB_TOKEN
echo $GITHUB_TOKEN | docker login ghcr.io -u USERNAME --password-stdin

# Build and push
docker build -t ghcr.io/OWNER/REPO:TAG .
docker push ghcr.io/OWNER/REPO:TAG

# Example
docker build -t ghcr.io/myorg/myapp:v1.0.0 .
docker push ghcr.io/myorg/myapp:v1.0.0

# Pull a public image
docker pull ghcr.io/myorg/myapp:v1.0.0
```

### GHCR in GitHub Actions

```yaml
# .github/workflows/docker.yml
name: Build and Push

on:
  push:
    branches: [main]
    tags: ['v*']

env:
  REGISTRY: ghcr.io
  IMAGE_NAME: ${{ github.repository }}

jobs:
  build-push:
    runs-on: ubuntu-latest
    permissions:
      contents: read
      packages: write
      id-token: write

    steps:
      - uses: actions/checkout@v4

      - name: Log in to GHCR
        uses: docker/login-action@v3
        with:
          registry: ghcr.io
          username: ${{ github.actor }}
          password: ${{ secrets.GITHUB_TOKEN }}

      - name: Extract metadata (tags, labels)
        id: meta
        uses: docker/metadata-action@v5
        with:
          images: ${{ env.REGISTRY }}/${{ env.IMAGE_NAME }}
          tags: |
            type=ref,event=branch
            type=ref,event=pr
            type=semver,pattern={{version}}
            type=semver,pattern={{major}}.{{minor}}
            type=sha

      - name: Build and push
        uses: docker/build-push-action@v5
        with:
          context: .
          push: true
          tags: ${{ steps.meta.outputs.tags }}
          labels: ${{ steps.meta.outputs.labels }}
          cache-from: type=gha
          cache-to: type=gha,mode=max
```

## AWS Elastic Container Registry (ECR)

```bash
# Authenticate (requires AWS CLI configured)
aws ecr get-login-password --region us-east-1 \
  | docker login \
      --username AWS \
      --password-stdin \
      123456789.dkr.ecr.us-east-1.amazonaws.com

# Create a repository
aws ecr create-repository \
  --repository-name myapp \
  --region us-east-1 \
  --image-scanning-configuration scanOnPush=true \
  --encryption-configuration encryptionType=AES256

# Build, tag, and push
docker build -t myapp:latest .
docker tag myapp:latest 123456789.dkr.ecr.us-east-1.amazonaws.com/myapp:latest
docker push 123456789.dkr.ecr.us-east-1.amazonaws.com/myapp:latest

# Pull
docker pull 123456789.dkr.ecr.us-east-1.amazonaws.com/myapp:latest

# List repositories
aws ecr describe-repositories --region us-east-1

# List images in a repository
aws ecr list-images --repository-name myapp --region us-east-1

# Set lifecycle policy (auto-cleanup old images)
aws ecr put-lifecycle-policy \
  --repository-name myapp \
  --lifecycle-policy-text file://lifecycle-policy.json
```

```json
// lifecycle-policy.json — keep last 10 images, expire untagged after 1 day
{
  "rules": [
    {
      "rulePriority": 1,
      "description": "Expire untagged images after 1 day",
      "selection": {
        "tagStatus": "untagged",
        "countType": "sinceImagePushed",
        "countUnit": "days",
        "countNumber": 1
      },
      "action": { "type": "expire" }
    },
    {
      "rulePriority": 2,
      "description": "Keep last 10 tagged images",
      "selection": {
        "tagStatus": "tagged",
        "tagPrefixList": ["v"],
        "countType": "imageCountMoreThan",
        "countNumber": 10
      },
      "action": { "type": "expire" }
    }
  ]
}
```

## Private Registry

Run your own registry using the official `registry` image:

```bash
# Start a local registry
docker run -d \
  --name registry \
  --restart always \
  -p 5000:5000 \
  -v registry-data:/var/lib/registry \
  registry:2

# Push to local registry
docker tag myapp localhost:5000/myapp:v1
docker push localhost:5000/myapp:v1

# Pull from local registry
docker pull localhost:5000/myapp:v1

# List images
curl http://localhost:5000/v2/_catalog
curl http://localhost:5000/v2/myapp/tags/list
```

### Registry with TLS and Auth (Production)

```yaml
# compose.yml for production private registry
services:
  registry:
    image: registry:2
    ports:
      - '5000:5000'
    environment:
      REGISTRY_HTTP_TLS_CERTIFICATE: /certs/domain.crt
      REGISTRY_HTTP_TLS_KEY: /certs/domain.key
      REGISTRY_AUTH: htpasswd
      REGISTRY_AUTH_HTPASSWD_REALM: 'Registry Realm'
      REGISTRY_AUTH_HTPASSWD_PATH: /auth/htpasswd
      REGISTRY_STORAGE_DELETE_ENABLED: 'true'
    volumes:
      - registry-data:/var/lib/registry
      - ./certs:/certs:ro
      - ./auth:/auth:ro
    restart: always
```

```bash
# Create htpasswd credentials
docker run --rm httpd:alpine \
  htpasswd -nbB admin securepassword > ./auth/htpasswd
```

## Image Scanning

### Docker Scout

```bash
# Scan an image for vulnerabilities (requires Docker Desktop or Scout CLI)
docker scout cves nginx:alpine
docker scout cves --only-severity critical,high myapp:latest

# Compare two images
docker scout compare nginx:1.24 nginx:1.25

# SBOM (Software Bill of Materials)
docker scout sbom myapp:latest

# Quick summary
docker scout quickview myapp:latest
```

### Trivy (Open Source)

```bash
# Install
brew install aquasecurity/trivy/trivy

# Scan an image
trivy image nginx:alpine
trivy image myapp:latest

# Only critical vulnerabilities
trivy image --severity CRITICAL myapp:latest

# Output JSON
trivy image --format json --output trivy-report.json myapp:latest

# Scan a tarball
docker save myapp:latest | trivy image --input -

# Fail CI on HIGH+ vulnerabilities
trivy image --exit-code 1 --severity HIGH,CRITICAL myapp:latest
```

```yaml
# GitHub Actions integration
- name: Scan image with Trivy
  uses: aquasecurity/trivy-action@master
  with:
    image-ref: myapp:latest
    format: sarif
    output: trivy-results.sarif
    severity: CRITICAL,HIGH

- name: Upload Trivy scan results to GitHub Security tab
  uses: github/codeql-action/upload-sarif@v3
  with:
    sarif_file: trivy-results.sarif
```

## Image Signing with Cosign

```bash
# Install cosign
brew install cosign

# Generate key pair
cosign generate-key-pair

# Sign an image after pushing
cosign sign --key cosign.key myregistry/myapp:v1.0.0

# Verify a signature
cosign verify --key cosign.pub myregistry/myapp:v1.0.0

# Sign with GitHub Actions OIDC (keyless — no key management)
cosign sign --oidc-issuer https://token.actions.githubusercontent.com \
  ghcr.io/myorg/myapp:v1.0.0
```

## Multi-Architecture Builds

Build images that run natively on both `linux/amd64` (x86 servers) and `linux/arm64` (Apple Silicon, AWS Graviton, Raspberry Pi).

```bash
# Create a buildx builder with multi-arch support
docker buildx create --name multiarch --driver docker-container --use
docker buildx inspect --bootstrap

# Build and push multi-arch image
docker buildx build \
  --platform linux/amd64,linux/arm64 \
  --tag myregistry/myapp:v1.0.0 \
  --push \
  .

# Build for specific platform (local testing)
docker buildx build \
  --platform linux/arm64 \
  --tag myapp:arm64 \
  --load \
  .

# Inspect a multi-arch manifest
docker buildx imagetools inspect myregistry/myapp:v1.0.0

# Create and push a manifest list manually
docker manifest create myapp:latest \
  myapp:latest-amd64 \
  myapp:latest-arm64

docker manifest push myapp:latest
```

:::tip[Multi-Arch in CI]
Use `docker/setup-qemu-action` in GitHub Actions to enable QEMU emulation for cross-platform builds:

```yaml
- uses: docker/setup-qemu-action@v3
- uses: docker/setup-buildx-action@v3
- uses: docker/build-push-action@v5
  with:
    platforms: linux/amd64,linux/arm64
    push: true
    tags: ghcr.io/myorg/myapp:latest
```

:::

## Image Size Optimisation

### Using dive

```bash
# Install dive — visualise image layers
brew install dive

# Analyse an image
dive myapp:latest

# CI mode — fail if efficiency is below threshold
dive --ci myapp:latest
CI_MISMATCH_THRESHOLD=0.1 dive myapp:latest
```

### Layer Optimisation

```bash
# Compare image sizes
docker images --format "table {{.Repository}}\t{{.Tag}}\t{{.Size}}" | sort -k3

# Show all layers with sizes
docker history --human --format "table {{.CreatedBy}}\t{{.Size}}" myapp:latest

# Flatten all layers into one (use sparingly — loses build cache)
docker export myapp | docker import - myapp:flat
```

### Practical Size Reduction

| Technique                        | Typical Saving |
| -------------------------------- | -------------- |
| Alpine base instead of Debian    | 100–300MB      |
| Multi-stage build (Node)         | 200–400MB      |
| `npm ci --only=production`       | 50–200MB       |
| `--no-install-recommends` in apt | 20–100MB       |
| `rm -rf /var/lib/apt/lists/*`    | 10–50MB        |
| `npm cache clean --force`        | 30–100MB       |
| Distroless final stage           | 50–150MB       |

```dockerfile
# Before: 1.2GB
FROM node:20
COPY . .
RUN npm install

# After: ~150MB
FROM node:20-alpine AS builder
WORKDIR /app
COPY package*.json .
RUN npm ci
COPY . .
RUN npm run build

FROM node:20-alpine
WORKDIR /app
COPY --from=builder /app/dist ./dist
COPY package*.json .
RUN npm ci --only=production && npm cache clean --force
CMD ["node", "dist/server.js"]
```

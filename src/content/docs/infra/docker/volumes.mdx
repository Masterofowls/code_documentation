# Docker Volumes

By default, all files created inside a container are stored in a writable container layer that's lost when the container stops. **Volumes** are the preferred mechanism for persisting data generated by and used by Docker containers.

---

## Volume Types

| Type | Description | Use Case |
|---|---|---|
| **Named volume** | Docker-managed storage, lives at `/var/lib/docker/volumes/` | Databases, persistent app data |
| **Bind mount** | Maps a host filesystem path into a container | Dev hot-reload, config files, code |
| **tmpfs** | In-memory storage, never written to disk | Secrets, sensitive temp data, performance |
| **Anonymous volume** | Like named volume but no explicit name (generated ID) | Throwaway or auto-created by Dockerfile `VOLUME` |

---

## Named Volumes

Named volumes are the recommended way to persist data. They are completely managed by Docker and portable across systems.

### Creating and Managing Volumes

```bash
# Create a named volume
docker volume create my-data

# List volumes
docker volume ls

# Inspect a volume (shows mountpoint, driver, labels)
docker volume inspect my-data

# Remove a volume
docker volume rm my-data

# Remove all unused volumes (not mounted by any container)
docker volume prune

# Remove with filter
docker volume prune --filter "label!=keep"
```

### Using Named Volumes

```bash
# Mount named volume into a container (-v)
docker run -d \
  --name postgres \
  -v postgres-data:/var/lib/postgresql/data \
  postgres:16

# Long-form mount flag (--mount) — more explicit, recommended for scripts
docker run -d \
  --name postgres \
  --mount type=volume,source=postgres-data,target=/var/lib/postgresql/data \
  postgres:16

# Volume is automatically created if it doesn't exist
docker run -v new-volume:/app/data my-app
```

**Key characteristic:** Named volumes **survive container removal**. If you `docker rm postgres` and then recreate the container mounting the same volume name, your data is still there.

## Bind Mounts

Bind mounts map a host directory or file into a container. The host path must exist.

```bash
# Short form: host-path:container-path
docker run -v /host/path:/container/path nginx

# Current directory
docker run -v $(pwd):/app node:20 npm test

# Read-only bind mount
docker run -v $(pwd)/config.json:/app/config.json:ro nginx

# Long form (--mount)
docker run \
  --mount type=bind,source="$(pwd)/src",target=/app/src \
  node:20-alpine

# Read-only with --mount
docker run \
  --mount type=bind,source="$(pwd)/nginx.conf",target=/etc/nginx/nginx.conf,readonly \
  nginx
```

:::caution[Bind mount path errors]
With bind mounts, Docker will create the path as a **directory** on the host if it doesn't exist, even if you intended it as a file. Always create files before mounting them as bind mounts. Use `--mount` syntax to get clearer errors.
:::

### Development Hot-Reload Pattern

```bash
# Mount source code for live editing (Node.js example)
docker run -d \
  --name dev-server \
  -p 3000:3000 \
  -v $(pwd)/src:/app/src \
  -v $(pwd)/package.json:/app/package.json \
  -e NODE_ENV=development \
  node:20-alpine \
  npm run dev

# In compose.yml for dev environment:
```

```yaml
# compose.override.yml (development)
services:
  api:
    volumes:
      - ./src:/app/src             # Live reload source changes
      - ./public:/app/public       # Static assets
      # Don't mount node_modules from host:
      - /app/node_modules          # Anonymous volume "shadows" the bind mount
    command: npm run dev
```

The anonymous volume `/app/node_modules` trick prevents the host's `node_modules` directory (which may be empty or built for a different platform) from overriding the container's installed modules.

## tmpfs Mounts

tmpfs mounts are stored in host memory and never written to disk. Perfect for sensitive data that shouldn't persist.

```bash
# Short form (no tmpfs options)
docker run --tmpfs /tmp my-app

# Long form with options
docker run \
  --mount type=tmpfs,target=/tmp,tmpfs-size=100m,tmpfs-mode=1777 \
  my-app

# In compose.yml
```

```yaml
services:
  api:
    tmpfs:
      - /tmp
      - /run
    # Long form:
    volumes:
      - type: tmpfs
        target: /tmp
        tmpfs:
          size: 104857600          # 100 MB in bytes
```

## Volumes in Dockerfiles

The `VOLUME` instruction in a Dockerfile creates a mount point and marks it as holding externally mounted volumes.

```dockerfile
# Declare a volume in Dockerfile
FROM postgres:16
VOLUME /var/lib/postgresql/data

# Multiple volumes
VOLUME ["/app/data", "/app/logs"]
```

:::tip[VOLUME instruction behavior]
When a container starts with a Dockerfile-declared `VOLUME` path, Docker creates an **anonymous volume** for it automatically if you don't provide a named or bind mount. Data written to that path is not lost when the container stops, but the volume has a random generated name.

Always provide an explicit volume name at runtime to maintain control.
:::

## Sharing Volumes Between Containers

```bash
# Create a volume
docker volume create shared-data

# Container A writes to it
docker run -d \
  --name writer \
  -v shared-data:/data \
  alpine sh -c "echo 'hello' > /data/message.txt && sleep 3600"

# Container B reads from it
docker run --rm \
  -v shared-data:/data:ro \
  alpine cat /data/message.txt

# Using --volumes-from (inherit volumes from another container)
docker run -d --name source-container \
  -v /app/data \
  my-app

docker run --rm \
  --volumes-from source-container \
  alpine ls /app/data
```

## Backup and Restore

### Backup a Named Volume

```bash
# Method 1: tar pipe backup
docker run --rm \
  -v my-volume:/source:ro \
  -v $(pwd):/backup \
  alpine tar czf /backup/my-volume-backup.tar.gz -C /source .

# Method 2: using --volumes-from to backup a running container's data
docker run --rm \
  --volumes-from my-container \
  -v $(pwd):/backup \
  alpine tar czf /backup/data-backup.tar.gz /app/data

# Restore from backup
docker run --rm \
  -v my-volume:/target \
  -v $(pwd):/backup:ro \
  alpine tar xzf /backup/my-volume-backup.tar.gz -C /target
```

### Postgres Database Backup

```bash
# Backup
docker exec postgres pg_dumpall -U postgres | gzip > backup-$(date +%Y%m%d).sql.gz

# Restore
gunzip < backup-20240115.sql.gz | docker exec -i postgres psql -U postgres

# Backup specific database
docker exec postgres pg_dump -U postgres mydb | gzip > mydb-backup.sql.gz
```

## Volume Drivers

Docker's volume driver system allows volumes to be stored on remote systems.

```bash
# Default local driver
docker volume create --driver local my-local-vol

# NFS driver
docker volume create \
  --driver local \
  --opt type=nfs \
  --opt o=addr=192.168.1.100,rw,nfsvers=4 \
  --opt device=:/exports/data \
  nfs-volume

# CIFS/SMB driver
docker volume create \
  --driver local \
  --opt type=cifs \
  --opt o=addr=192.168.1.100,username=user,password=pass,file_mode=0777,dir_mode=0777 \
  --opt device=//192.168.1.100/share \
  smb-volume
```

**Third-party volume drivers:**
- `rexray` — Multi-cloud, supports AWS EBS, GCE PD, Azure, etc.
- `portworx` — Cloud-native storage for Kubernetes and Docker
- `convoy` — Backup and recovery oriented

## Database Data Persistence Patterns

### PostgreSQL

```yaml
# compose.yml
services:
  db:
    image: postgres:16-alpine
    volumes:
      - postgres-data:/var/lib/postgresql/data
      - ./db/init:/docker-entrypoint-initdb.d:ro  # Init scripts
    environment:
      PGDATA: /var/lib/postgresql/data/pgdata     # Subdirectory prevents permission issues

volumes:
  postgres-data:
    driver: local
```

### MySQL / MariaDB

```yaml
services:
  db:
    image: mysql:8
    volumes:
      - mysql-data:/var/lib/mysql
    environment:
      MYSQL_ROOT_PASSWORD: rootpass
      MYSQL_DATABASE: mydb

volumes:
  mysql-data:
```

### MongoDB

```yaml
services:
  mongo:
    image: mongo:7
    volumes:
      - mongo-data:/data/db
      - mongo-config:/data/configdb

volumes:
  mongo-data:
  mongo-config:
```

## Volume Performance on macOS and Windows

Docker on macOS and Windows runs containers inside a Linux VM. Bind mounts crossing the VM boundary have historically been slow.

### macOS Performance

```bash
# Modern Docker Desktop uses VirtioFS by default (very fast)
# Check in Docker Desktop → Settings → General → VirtioFS

# Older systems use gRPC-FUSE or osxfs (slower)
# If experiencing slow performance, enable VirtioFS

# For best performance on macOS, use named volumes instead of bind mounts
# for node_modules and other large directories
```

### Windows and WSL 2

```yaml
# Best performance: store project files inside WSL 2 filesystem
# /home/user/projects/ (NOT /mnt/c/Users/...)

# In WSL 2 terminal:
# cd ~/projects/myapp
# docker compose up
# This gives near-native Linux performance
```

:::tip[Development tip on macOS]
If you're experiencing slow `npm install` or build times in containers on macOS, try storing your project inside the WSL 2 or Docker VM volume instead of crossing the VirtioFS boundary. Alternatively, exclude `node_modules` from bind mounts and use a named volume or anonymous volume for it.
:::

## Volume Inspection and Cleanup

```bash
# See all volumes and their sizes
docker system df -v

# Find which containers are using a volume
docker ps -a --filter volume=my-volume

# Check what's in a volume (useful for debugging)
docker run --rm \
  -v my-volume:/data:ro \
  alpine ls -la /data

# Find dangling volumes (not referenced by any container)
docker volume ls -f dangling=true

# Clean up dangling volumes
docker volume prune

# Nuclear option: remove ALL unused volumes
# WARNING: This deletes data!
docker volume prune --all --force
```

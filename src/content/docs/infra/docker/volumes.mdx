# Docker Volumes

Containers are ephemeral — when a container is removed, all data written to its writable layer is lost. Docker volumes provide persistent storage that exists independently of any container's lifecycle.

## Volume Types Comparison

| Type | Managed by | Host Path | Persistence | Best for |
|---|---|---|---|---|
| Named volume | Docker | `/var/lib/docker/volumes/<name>` | Survives container removal | Production databases, stateful apps |
| Bind mount | User | Any host path | Depends on host | Development (source code), config files |
| tmpfs | Kernel | Memory only | Lost on container stop | Sensitive data, temp files, performance |

## Named Volumes

Named volumes are managed by Docker and stored under `/var/lib/docker/volumes/` on the host.

### CLI Operations

```bash
# Create a named volume
docker volume create pgdata

# Create with options (NFS example)
docker volume create \
  --driver local \
  --opt type=nfs \
  --opt o=addr=192.168.1.100,rw \
  --opt device=:/path/to/nfs \
  nfs-volume

# List all volumes
docker volume ls

# Filter volumes
docker volume ls -f dangling=true    # unused volumes

# Inspect a volume
docker volume inspect pgdata

# Remove a specific volume
docker volume rm pgdata

# Remove all unused (dangling) volumes
docker volume prune

# Remove all volumes (including those in use)
docker volume prune -a
```

### Volume Lifecycle

```bash
# The volume persists even after the container is removed
docker run -d -v pgdata:/var/lib/postgresql/data postgres:16-alpine
docker rm -f postgres_container    # container gone — data stays

# A new container can reattach to the existing volume
docker run -d -v pgdata:/var/lib/postgresql/data postgres:16-alpine
```

:::tip[Named volumes survive `docker rm`]
Named volumes are not deleted when a container is removed. They are only removed with `docker volume rm` or `docker volume prune`. This is the key difference from anonymous volumes.
:::

### Anonymous Volumes

Anonymous volumes are named volumes created without a name — Docker assigns a UUID.

```bash
# Anonymous volume created by -v /data
docker run -d -v /data myapp

# Also created by VOLUME instruction in Dockerfile with no explicit mapping
docker run -d myapp   # creates anonymous volume for each VOLUME declaration
```

:::caution[Anonymous volumes accumulate]
Anonymous volumes are not automatically removed when a container stops. They remain until `docker volume prune` or `docker rm -v`. Use named volumes in production to avoid leaking disk space.
:::

## Bind Mounts

Bind mounts mount a directory or file from the host filesystem directly into the container.

```bash
# Mount a directory (read-write)
docker run -d \
  -v /host/path:/container/path \
  myapp

# Current working directory
docker run -d \
  -v $(pwd)/src:/app/src \
  myapp

# Read-only mount
docker run -d \
  -v $(pwd)/config:/app/config:ro \
  myapp

# Single file mount
docker run -d \
  -v $(pwd)/nginx.conf:/etc/nginx/nginx.conf:ro \
  nginx:alpine

# Using --mount syntax (more explicit)
docker run -d \
  --mount type=bind,source=$(pwd)/src,target=/app/src,readonly \
  myapp
```

### Development Workflow with Bind Mounts

Bind mounts enable live code reloading without rebuilding images:

```yaml
# compose.yml for development
services:
  api:
    build:
      context: .
      target: development    # dev stage with nodemon/hot reload
    volumes:
      # Source code — live reload
      - ./src:/app/src
      # Preserve container's node_modules (faster than host)
      - /app/node_modules
    environment:
      NODE_ENV: development
```

:::caution[Bind mount performance on macOS and Windows]
Bind mounts have lower I/O performance on macOS (hypervisor overhead) and Windows (WSL2 filesystem boundary). Use Docker's `virtiofs` (macOS) or keep source code in the WSL2 filesystem (Windows) for best performance.
:::

## tmpfs Mounts

tmpfs mounts are stored in host memory. They are never written to disk and are lost when the container stops.

```bash
# Basic tmpfs
docker run -d --tmpfs /tmp myapp

# With options (size limit, noexec, nosuid)
docker run -d \
  --tmpfs /tmp:noexec,nosuid,size=128m \
  myapp

# Using --mount syntax
docker run -d \
  --mount type=tmpfs,target=/tmp,tmpfs-size=128m,tmpfs-mode=1777 \
  myapp
```

Use cases for tmpfs:
- Session data that should never persist to disk
- Temporary processing files (image transforms, archives)
- In-memory SQLite databases for testing
- Sensitive intermediate data

## Volumes in Compose

```yaml
volumes:
  pgdata:           # Docker-managed named volume
    driver: local
  redisdata:
    driver: local
  nfs-data:
    driver: local
    driver_opts:
      type: nfs
      o: "addr=192.168.1.100,rw,nfsvers=4"
      device: ":/exports/data"
  external-vol:
    external: true  # must exist before `docker compose up`
    name: legacy-pgdata

services:
  postgres:
    image: postgres:16-alpine
    volumes:
      # Named volume
      - pgdata:/var/lib/postgresql/data
      # Bind mount for init scripts
      - ./postgres/init:/docker-entrypoint-initdb.d:ro
      # tmpfs for tmp
      - type: tmpfs
        target: /tmp

  api:
    volumes:
      # Development bind mount
      - ./src:/app/src
      # Preserve container's node_modules
      - type: volume
        target: /app/node_modules
```

## Sharing Volumes Between Containers

```bash
# Share a named volume between two containers
docker run -d --name writer -v shared:/data alpine \
  sh -c "while true; do date >> /data/log.txt; sleep 5; done"

docker run -d --name reader -v shared:/data alpine \
  sh -c "while true; do tail -5 /data/log.txt; sleep 5; done"

# In Compose
services:
  writer:
    volumes:
      - shared:/data
  reader:
    volumes:
      - shared:/data:ro

volumes:
  shared:
```

## Backup and Restore

### Backup a Named Volume

```bash
# Backup pattern: run a temporary container that mounts the volume,
# tars it, and streams the tar to the host

docker run --rm \
  -v pgdata:/data \
  -v $(pwd)/backups:/backup \
  alpine \
  tar czf /backup/pgdata-$(date +%Y%m%d-%H%M%S).tar.gz -C /data .
```

### Restore a Volume

```bash
# Create a new volume and restore from backup
docker volume create pgdata-restored

docker run --rm \
  -v pgdata-restored:/data \
  -v $(pwd)/backups:/backup:ro \
  alpine \
  sh -c "cd /data && tar xzf /backup/pgdata-20240115-120000.tar.gz"
```

### Using --volumes-from

```bash
# Clone volumes from one container to another
docker run --rm \
  --volumes-from source-container \
  -v $(pwd)/backups:/backup \
  alpine \
  tar czf /backup/backup.tar.gz /app/data
```

### PostgreSQL-Specific Backup

```bash
# Dump database using pg_dump (better than raw volume backup)
docker exec postgres pg_dump -U app myapp > backup.sql

# Restore
docker exec -i postgres psql -U app myapp < backup.sql

# Compressed dump
docker exec postgres pg_dump -U app -Fc myapp > backup.dump
docker exec -i postgres pg_restore -U app -d myapp < backup.dump
```

## Volume Drivers

The default `local` driver supports basic NFS and tmpfs options. Extend with third-party drivers for cloud storage.

```bash
# NFS volume
docker volume create \
  --driver local \
  --opt type=nfs \
  --opt o=nfsvers=4,addr=192.168.1.50,rw \
  --opt device=:/mnt/data \
  nfs-vol

# CIFS/SMB volume
docker volume create \
  --driver local \
  --opt type=cifs \
  --opt o=username=user,password=pass,domain=DOMAIN \
  --opt device=//server/share \
  smb-vol
```

Popular third-party volume drivers:
- **rexray/ebs** — AWS Elastic Block Store
- **rexray/gcepd** — Google Compute Engine Persistent Disk
- **rexray/azureud** — Azure Unmanaged Disk
- **vieux/sshfs** — SSH/SFTP
- **Portworx** — Enterprise distributed storage

## Database Data Persistence Patterns

### PostgreSQL

```yaml
services:
  postgres:
    image: postgres:16-alpine
    environment:
      POSTGRES_DB: myapp
      POSTGRES_USER: app
      POSTGRES_PASSWORD: ${DB_PASSWORD}
    volumes:
      - pgdata:/var/lib/postgresql/data          # persistent data
      - ./postgres/init:/docker-entrypoint-initdb.d:ro  # init scripts
      - ./postgres/postgresql.conf:/etc/postgresql/postgresql.conf:ro
    command: postgres -c config_file=/etc/postgresql/postgresql.conf

volumes:
  pgdata:
```

### Redis

```yaml
services:
  redis:
    image: redis:7-alpine
    command: redis-server --appendonly yes --save 60 1
    volumes:
      - redisdata:/data

volumes:
  redisdata:
```

### MongoDB

```yaml
services:
  mongo:
    image: mongo:7
    environment:
      MONGO_INITDB_ROOT_USERNAME: admin
      MONGO_INITDB_ROOT_PASSWORD: ${MONGO_PASSWORD}
    volumes:
      - mongodata:/data/db
      - ./mongo/init:/docker-entrypoint-initdb.d:ro

volumes:
  mongodata:
```

## Volume Performance on macOS and Windows

### macOS — VirtioFS

Docker Desktop for macOS mounts volumes via a Linux VM. Enable VirtioFS for the best bind mount performance:

1. Docker Desktop → Settings → General → **Use VirtioFS** (default in recent versions)
2. VirtioFS uses Apple's Virtualization.framework for near-native I/O

```bash
# Check mount type
docker run --rm -it -v $(pwd):/test alpine \
  dd if=/dev/zero of=/test/testfile bs=1M count=100
```

### Windows — WSL 2

Keep source code inside the WSL 2 filesystem (`/home/user/projects`) rather than the Windows filesystem (`/mnt/c/Users`). Cross-filesystem I/O is significantly slower.

```bash
# Good — project in WSL filesystem
cd ~/projects/myapp
docker run -v $(pwd):/app myapp

# Slow — project on Windows drive
cd /mnt/c/Users/user/projects/myapp
docker run -v $(pwd):/app myapp   # poor I/O performance
```

## Inspect Volume Contents

```bash
# Mount a volume to explore its contents
docker run --rm -it \
  -v pgdata:/data:ro \
  alpine \
  sh -c "ls -la /data && du -sh /data/*"

# Find large files in a volume
docker run --rm \
  -v pgdata:/data:ro \
  alpine \
  find /data -size +10M -exec ls -lh {} \;
```

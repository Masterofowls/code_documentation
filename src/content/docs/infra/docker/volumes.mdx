# Docker Volumes

Containers are **ephemeral** — their writable layer is deleted when the container is removed. Volumes solve this by providing persistent storage that lives outside the container lifecycle.

## Volume Types Overview

| Type                 | Managed By      | Persists After Container Removal           | Best For                              |
| -------------------- | --------------- | ------------------------------------------ | ------------------------------------- |
| **Named volume**     | Docker          | ✅ Yes                                     | Database data, app state              |
| **Bind mount**       | Host filesystem | ✅ Yes (it's a host path)                  | Development hot-reload, configs       |
| **tmpfs mount**      | Host memory     | ❌ No (in-memory)                          | Temp files, sensitive runtime data    |
| **Anonymous volume** | Docker          | ❌ No (auto-named, removed with container) | Temporary per-container scratch space |

## Named Volumes

Named volumes are the recommended way to persist data. Docker manages the storage location (under `/var/lib/docker/volumes/` on Linux).

### Managing Named Volumes

```bash
# List all volumes
docker volume ls

# Create a volume explicitly
docker volume create my-data
docker volume create --driver local \
  --opt type=nfs \
  --opt o=addr=192.168.1.100,rw \
  --opt device=:/path/to/export \
  nfs-volume

# Inspect a volume
docker volume inspect my-data
# Output:
# [
#   {
#     "Name": "my-data",
#     "Driver": "local",
#     "Mountpoint": "/var/lib/docker/volumes/my-data/_data",
#     "Labels": {},
#     "Scope": "local"
#   }
# ]

# Remove a volume (only works if no containers are using it)
docker volume rm my-data

# Remove all unused volumes (caution!)
docker volume prune

# Remove specific unused volumes
docker volume prune --filter label=temporary=true
```

### Using Named Volumes

```bash
# docker run — mount named volume
docker run -d \
  --name postgres \
  -v postgres-data:/var/lib/postgresql/data \
  postgres:16-alpine

# Multiple volumes
docker run -d \
  --name app \
  -v app-logs:/var/log/app \
  -v app-uploads:/app/uploads \
  myapp:latest

# Inspect which volumes a container uses
docker inspect --format '{{json .Mounts}}' postgres | python3 -m json.tool
```

```yaml
# In Docker Compose
services:
  db:
    image: postgres:16-alpine
    volumes:
      - postgres-data:/var/lib/postgresql/data # named volume mount

  redis:
    image: redis:7-alpine
    volumes:
      - redis-data:/data
      - ./redis.conf:/etc/redis/redis.conf:ro # bind mount for config

volumes:
  postgres-data: # declared — Docker creates it
  redis-data:
    driver: local # explicit driver (default)
    labels:
      app.component: cache

  external-vol:
    external: true # must exist before compose up
    name: my-pre-created-volume
```

### Volume Lifecycle

Named volumes **survive** container removal:

```bash
docker run -d --name db -v mydb-data:/var/lib/postgresql/data postgres:16
docker stop db && docker rm db           # container gone
docker volume ls | grep mydb-data        # volume still exists!

# Create a new container reusing the same volume
docker run -d --name db-v2 -v mydb-data:/var/lib/postgresql/data postgres:16
# Data is preserved
```

## Bind Mounts

Bind mounts map a **host path** directly into a container. Changes on either side are immediately reflected.

```bash
# Absolute host path : container path
docker run -d \
  -v /home/user/myapp:/app \
  -v /home/user/config/nginx.conf:/etc/nginx/nginx.conf:ro \
  myapp:latest

# Current directory ($(pwd) on Linux/macOS, %cd% on Windows cmd)
docker run -d -v $(pwd):/app node:20-alpine

# Read-only bind mount
docker run -d -v $(pwd)/config:/etc/app:ro myapp

# Long-form syntax (clearer)
docker run -d \
  --mount type=bind,source=$(pwd)/src,target=/app/src,readonly \
  myapp
```

### Bind Mounts in Compose (Development Workflow)

```yaml
services:
  app:
    build: .
    volumes:
      # Mount source code for hot-reload
      - .:/app
      # Prevent host node_modules from overwriting container's
      - /app/node_modules
    command: npm run dev
    environment:
      NODE_ENV: development

  python-app:
    build: .
    volumes:
      - ./src:/app/src # only mount source, not entire context
      - ./tests:/app/tests
```

:::tip[node_modules Trick]
When you bind-mount your project into a Node container, the host's `node_modules` (built for your OS) overwrites the container's `node_modules` (built for Linux). Fix it by declaring an anonymous volume on `node_modules`:

```yaml
volumes:
  - .:/app
  - /app/node_modules # anonymous volume "shadows" the bind mount here
```

:::

### Bind Mount Permissions

```bash
# Container process writes files owned by its UID
# On the host, those files are owned by that UID

# Ensure the host directory is writable by the container user
# Option 1: Make directory world-writable (quick but insecure)
chmod 777 ./data

# Option 2: Change ownership to container UID
sudo chown -R 1001:1001 ./data

# Option 3: Use --user to run as your host UID
docker run --user $(id -u):$(id -g) -v $(pwd):/app myapp
```

## tmpfs Mounts

tmpfs mounts are stored in the host's memory. They're never written to disk — perfect for sensitive data that shouldn't persist.

```bash
# tmpfs via --tmpfs flag
docker run -d \
  --tmpfs /tmp \
  --tmpfs /run:rw,noexec,nosuid,size=65536k \
  myapp

# tmpfs via --mount
docker run -d \
  --mount type=tmpfs,target=/tmp,tmpfs-size=64m,tmpfs-mode=1777 \
  myapp
```

```yaml
# Compose tmpfs
services:
  app:
    tmpfs:
      - /tmp
      - /run
    # Or with options
    volumes:
      - type: tmpfs
        target: /tmp
        tmpfs:
          size: 67108864 # 64MB in bytes
```

## Volume in Dockerfile

```dockerfile
# Declare a mount point in the image
# If no volume is mounted at runtime, Docker creates an anonymous volume
VOLUME /var/lib/postgresql/data
VOLUME ["/app/uploads", "/app/logs"]
```

:::caution[VOLUME in Dockerfile Pitfall]
If you declare `VOLUME /data` in a Dockerfile, **any `COPY` or `RUN` commands that write to `/data` after that line are discarded** — because Docker has already initialised an anonymous volume there. Always copy data **before** the `VOLUME` instruction.
:::

## Sharing Volumes Between Containers

```bash
# Share a volume between multiple containers
docker volume create shared-uploads

docker run -d --name writer -v shared-uploads:/uploads myapp-writer
docker run -d --name reader -v shared-uploads:/uploads:ro myapp-reader
```

```yaml
# Compose: multiple services sharing a volume
services:
  uploader:
    image: myapp-uploader
    volumes:
      - uploads:/app/uploads

  processor:
    image: myapp-processor
    volumes:
      - uploads:/data/incoming:ro

  cdn-sync:
    image: myapp-cdn-sync
    volumes:
      - uploads:/sync/source:ro

volumes:
  uploads:
```

### Volumes-From (Legacy)

```bash
# Share all volumes from another container (legacy approach)
docker run -d --name data-container -v /data busybox

docker run -d --volumes-from data-container myapp

# Compose equivalent — use named volumes instead:
# volumes: [shared-data:/data]
```

## Backup and Restore

### Backup a Named Volume

```bash
# Create a backup of a named volume using a temporary container
docker run --rm \
  -v my-volume:/source:ro \
  -v $(pwd):/backup \
  alpine \
  tar czf /backup/my-volume-backup.tar.gz -C /source .

# Or using docker cp if the container is running
docker run -d --name temp-backup -v my-volume:/data busybox sleep infinity
docker cp temp-backup:/data ./backup-data
docker rm -f temp-backup
```

### Restore a Volume

```bash
# Restore from backup archive
docker volume create my-volume-restored

docker run --rm \
  -v my-volume-restored:/target \
  -v $(pwd):/backup:ro \
  alpine \
  tar xzf /backup/my-volume-backup.tar.gz -C /target
```

### Backup a Running Database

```bash
# PostgreSQL backup
docker exec postgres \
  pg_dump -U myuser mydb | gzip > backup-$(date +%Y%m%d_%H%M%S).sql.gz

# PostgreSQL restore
gunzip -c backup.sql.gz | docker exec -i postgres psql -U myuser mydb

# MySQL backup
docker exec mysql \
  mysqldump -u root -p"${MYSQL_ROOT_PASSWORD}" mydb \
  > backup-$(date +%Y%m%d).sql

# Redis backup (copy RDB file)
docker exec redis redis-cli BGSAVE
docker cp redis:/data/dump.rdb ./redis-backup.rdb
```

## Volume Drivers

```bash
# Local driver (default) — stores on host filesystem
docker volume create --driver local mydata

# NFS volume (shares across multiple hosts)
docker volume create \
  --driver local \
  --opt type=nfs \
  --opt o=addr=nfs-server.example.com,rw,nfsvers=4 \
  --opt device=:/exports/data \
  nfs-data

# CIFS/SMB volume
docker volume create \
  --driver local \
  --opt type=cifs \
  --opt o=addr=smb-server,username=user,password=pass \
  --opt device=//smb-server/share \
  smb-data
```

## Database Data Persistence Patterns

### Pattern 1: Named Volume (Recommended)

```yaml
services:
  postgres:
    image: postgres:16-alpine
    volumes:
      - pgdata:/var/lib/postgresql/data
    environment:
      POSTGRES_PASSWORD: ${DB_PASSWORD}

volumes:
  pgdata:
```

### Pattern 2: Custom Volume with Backup Labels

```yaml
services:
  postgres:
    image: postgres:16-alpine
    volumes:
      - pgdata:/var/lib/postgresql/data

volumes:
  pgdata:
    labels:
      backup.enabled: 'true'
      backup.schedule: '0 2 * * *'
      backup.retention: '7d'
```

### Pattern 3: Pre-initialise Database

```yaml
services:
  postgres:
    image: postgres:16-alpine
    volumes:
      - pgdata:/var/lib/postgresql/data
      - ./init-scripts:/docker-entrypoint-initdb.d:ro # runs on first start
```

```bash
# init-scripts/01-schema.sql
# init-scripts/02-seed-data.sql
# Scripts run alphabetically on first volume init
```

## Volume Performance on macOS and Windows

Docker Desktop runs Linux containers inside a VM. File I/O between the host and the VM adds latency.

<Tabs>
  <TabItem label="macOS (VirtioFS)">
    Docker Desktop uses **VirtioFS** for file sharing — significantly faster than the older gRPC-FUSE.

    ```bash
    # Check VirtioFS is enabled (Settings → Resources → File Sharing)
    # Enabled by default in Docker Desktop 4.6+

    # For maximum performance, use named volumes for heavy I/O (e.g., databases)
    # Use bind mounts only for source code (read-heavy, occasional writes)
    ```

</TabItem>
  <TabItem label="Windows (WSL2)">
    Bind mounts to Windows paths (`C:\Users\...`) go through WSL2 filesystem translation — slow for I/O-intensive workloads.

    ```bash
    # Fastest option: keep files in the WSL2 filesystem
    # Access them at \\wsl$\Ubuntu\home\user\project in Windows

    # In WSL2 terminal — work here for best performance
    cd ~/projects/myapp
    docker compose up

    # Named volumes are stored inside WSL2 VM — fast
    # Bind mounts from /home/... are fast (native WSL2 path)
    # Bind mounts from /mnt/c/... are slow (cross-OS boundary)
    ```

    :::tip[WSL2 Performance Tip]
    Clone your repos into the WSL2 filesystem (`~/projects/`) instead of the Windows filesystem (`/mnt/c/Users/`). Bind mounts from the WSL2 filesystem are 5–10× faster for I/O-heavy workloads like `npm install` or database writes.
    :::

</TabItem>
</Tabs>

## Volume Inspection and Cleanup

```bash
# See how much space volumes are using
docker system df -v

# Remove dangling volumes (not referenced by any container)
docker volume prune

# Force remove with confirmation bypass
docker volume prune -f

# Remove specific volume
docker volume rm myapp_postgres-data

# Remove all volumes (DANGER — irreversible data loss)
docker volume prune -a

# Find which container is using a volume
docker ps -a --filter volume=my-volume

# Inspect volume details
docker volume inspect my-volume
```

:::caution[Volume Prune is Irreversible]
`docker volume prune` permanently deletes data. Always ensure you have backups before running prune commands, especially in environments where you manage stateful services.
:::

# Docker Volumes

By default, files created inside a container exist only as long as the container does. When a container is removed, its writable layer disappears. Docker provides three storage mechanisms to persist data beyond container lifecycles.

---

## Volume Types

| Type | Host Location | Managed By | Use Case |
|---|---|---|---|
| **Named Volume** | Docker-managed path | Docker | Database data, persistent app state |
| **Bind Mount** | Any host path | User | Dev hot-reload, config injection |
| **tmpfs** | Host RAM | Docker | Sensitive ephemeral data, caches |

---

## Named Volumes

Named volumes are the recommended way to persist data. Docker manages the storage location (`/var/lib/docker/volumes/` on Linux) and the volume lifecycle is independent of any container.

### Volume Commands

```bash
# Create a named volume
docker volume create pgdata

# Create with options
docker volume create \
  --driver local \
  --opt type=nfs \
  --opt o=addr=192.168.1.100,rw \
  --opt device=:/exports/data \
  nfs-volume

# List all volumes
docker volume ls

# List only volume names
docker volume ls -q

# Inspect a volume (see mountpoint and labels)
docker volume inspect pgdata

# Remove a volume (must not be in use by any container)
docker volume rm pgdata

# Remove all unused volumes
docker volume prune

# Remove unused volumes without confirmation
docker volume prune -f
```

### Using Named Volumes with docker run

```bash
# Mount a named volume into a container
docker run -d \
  --name postgres \
  -v pgdata:/var/lib/postgresql/data \
  postgres:16-alpine

# The volume persists after the container is removed
docker rm -f postgres
docker volume ls   # pgdata still exists

# Start a new container using the same data
docker run -d \
  --name postgres2 \
  -v pgdata:/var/lib/postgresql/data \
  postgres:16-alpine
```

### Named Volumes in Docker Compose

```yaml
services:
  db:
    image: postgres:16-alpine
    volumes:
      - pgdata:/var/lib/postgresql/data
      - pgconfig:/etc/postgresql

volumes:
  pgdata:                      # Docker manages location
  pgconfig:
    driver: local
    labels:
      com.example.environment: production
```

## Bind Mounts

Bind mounts link a specific path on the host into the container. The host directory **must already exist**. Changes on either side are immediately reflected.

```bash
# Bind mount current directory to /app
docker run -v "$(pwd)":/app myapp

# Bind mount with read-only flag
docker run -v "$(pwd)/config.json":/app/config.json:ro myapp

# Absolute host path
docker run -v /etc/ssl/certs:/etc/ssl/certs:ro myapp

# Windows paths
docker run -v "C:\Users\dev\myapp:/app" myapp
```

### Bind Mounts in Compose

```yaml
services:
  api:
    volumes:
      # Long syntax (explicit)
      - type: bind
        source: ./src
        target: /app/src
        read_only: false

      # Short syntax
      - ./src:/app/src

      # Read-only config
      - ./nginx.conf:/etc/nginx/nginx.conf:ro

      # Absolute path
      - /var/log/myapp:/app/logs
```

:::tip[Bind mounts for development hot reload]
Mount your source code as a bind mount during development. The container sees file changes instantly, enabling hot reload with tools like `nodemon`, `uvicorn --reload`, or `air` (Go).

```yaml
# compose.override.yml (dev only)
services:
  api:
    volumes:
      - ./src:/app/src
    command: ["npm", "run", "dev"]    # Uses nodemon
```
:::

## tmpfs (In-Memory Storage)

tmpfs mounts are stored in the host's RAM. They are not written to disk and are destroyed when the container stops. Ideal for sensitive temporary data.

```bash
# Mount tmpfs at /tmp (no size limit)
docker run --tmpfs /tmp myapp

# Mount with options
docker run \
  --tmpfs /tmp:size=100m,mode=1777 \
  --tmpfs /run:size=50m \
  myapp
```

In Compose:

```yaml
services:
  api:
    tmpfs:
      - /tmp:size=100m
      - /run:size=50m
      - /app/cache

    # Or using volumes with tmpfs type
    volumes:
      - type: tmpfs
        target: /tmp
        tmpfs:
          size: 104857600    # 100MB in bytes
```

## Volume in Dockerfile

Declaring a `VOLUME` in a Dockerfile creates an anonymous volume automatically at that path when a container starts without a matching `-v` mount.

```dockerfile
# Creates anonymous volume if not mounted explicitly
VOLUME /var/lib/postgresql/data
```

:::caution[Anonymous volumes are hard to manage]
Anonymous volumes generated by `VOLUME` in Dockerfiles have random IDs and are easy to lose track of. They accumulate with `docker volume prune -f` risks. Prefer explicit named volumes in `docker run -v pgdata:/path` or Compose.
:::

## Sharing Volumes Between Containers

Multiple containers can mount the same volume simultaneously. This enables data sharing and sidecar patterns.

```bash
# Container 1: write logs
docker run -d \
  --name app \
  -v applogs:/app/logs \
  myapp

# Container 2: ship logs (sidecar)
docker run -d \
  --name logshipper \
  -v applogs:/logs:ro \
  fluent/fluent-bit
```

In Compose:

```yaml
services:
  app:
    volumes:
      - applogs:/app/logs

  logshipper:
    image: fluent/fluent-bit
    volumes:
      - applogs:/logs:ro    # Read-only access

volumes:
  applogs:
```

## Backup and Restore

### Backing Up a Named Volume

```bash
# Backup: spin up a temporary container, mount the volume,
# and tar it to stdout piped to the host
docker run --rm \
  -v pgdata:/data:ro \
  -v "$(pwd)/backups":/backups \
  alpine \
  tar czf /backups/pgdata-$(date +%Y%m%d).tar.gz -C /data .

# Verify backup
ls -lh backups/
```

### Restoring a Volume from Backup

```bash
# Create a fresh volume
docker volume create pgdata-restored

# Restore into the new volume
docker run --rm \
  -v pgdata-restored:/data \
  -v "$(pwd)/backups":/backups:ro \
  alpine \
  tar xzf /backups/pgdata-20240101.tar.gz -C /data

# Verify
docker run --rm -v pgdata-restored:/data alpine ls -la /data
```

### Using --volumes-from (Copy Between Containers)

```bash
# Copy data from one container's volume to another
docker run --rm \
  --volumes-from source-container \
  -v "$(pwd)":/backup \
  alpine \
  tar czf /backup/data.tar.gz /app/data

# Restore into a new container
docker run --rm \
  --volumes-from target-container \
  -v "$(pwd)":/backup:ro \
  alpine \
  tar xzf /backup/data.tar.gz -C /
```

## Volume Drivers

Volume drivers let you mount external storage systems as Docker volumes.

### local (Default)

```bash
# NFS mount via local driver
docker volume create \
  --driver local \
  --opt type=nfs \
  --opt o=addr=192.168.1.100,rw,nfsvers=4 \
  --opt device=:/exports/mydata \
  nfs-data
```

### Cloud Volume Drivers

```bash
# AWS EFS (via rexray/efs or efs-utils)
docker volume create \
  --driver rexray/efs \
  --opt size=20 \
  efs-volume

# Azure File Share (via docker-volume-netshare)
docker volume create \
  --driver azurefile \
  --opt share=myshare \
  azure-volume
```

### Docker Compose Volume Driver Options

```yaml
volumes:
  nfs-data:
    driver: local
    driver_opts:
      type: nfs
      o: "addr=192.168.1.100,rw,nfsvers=4.1"
      device: ":/exports/mydata"

  tmpfs-cache:
    driver: local
    driver_opts:
      type: tmpfs
      device: tmpfs
      o: "size=256m"
```

## Database Data Persistence Patterns

### PostgreSQL

```yaml
services:
  db:
    image: postgres:16-alpine
    environment:
      POSTGRES_PASSWORD_FILE: /run/secrets/pg_password
    volumes:
      # Data directory — survives container updates
      - pgdata:/var/lib/postgresql/data
      # Custom postgres config
      - ./postgres.conf:/etc/postgresql/postgresql.conf:ro
      # Init scripts (run once on first start)
      - ./db/init:/docker-entrypoint-initdb.d:ro
    secrets:
      - pg_password

volumes:
  pgdata:
```

### MySQL / MariaDB

```yaml
services:
  db:
    image: mysql:8.0
    environment:
      MYSQL_ROOT_PASSWORD: rootsecret
      MYSQL_DATABASE: myapp
    volumes:
      - mysqldata:/var/lib/mysql
      - ./my.cnf:/etc/mysql/conf.d/custom.cnf:ro

volumes:
  mysqldata:
```

### Redis

```yaml
services:
  redis:
    image: redis:7-alpine
    command: redis-server --appendonly yes --save 60 1000
    volumes:
      - redisdata:/data

volumes:
  redisdata:
```

## Performance Considerations

### macOS Performance (virtiofs)

On macOS, Docker runs in a VM. Bind mounts cross the VM boundary and were historically slow. Modern Docker Desktop uses **virtiofs** for near-native performance.

```bash
# Enable virtiofs in Docker Desktop settings:
# Settings → General → "Use VirtioFS for file sharing"

# Verify
docker run --rm -v "$(pwd)":/data alpine stat /data
```

### Windows Performance (WSL2)

On Windows, use the **WSL2** backend and store project files inside the WSL2 filesystem (e.g., `~/projects/`) rather than on the Windows drive (`/mnt/c/`). Cross-filesystem I/O between Windows and WSL2 is slow.

```bash
# Fast — inside WSL2 filesystem
docker run -v ~/projects/myapp:/app myapp

# Slow — crossing WSL2/Windows boundary
docker run -v /mnt/c/Users/dev/myapp:/app myapp
```

:::tip[WSL2 file location tip]
Always keep Docker project files under `~/` inside WSL2, not under `/mnt/c/`. Mount performance inside WSL2 is near-native; cross-boundary performance can be 10× slower.
:::

### Anonymous Volume Trick for node_modules

A common pattern to prevent bind-mounted `node_modules` from the host from overriding the container's installed modules:

```yaml
services:
  api:
    volumes:
      - ./src:/app/src             # Bind mount source
      - /app/node_modules          # Anonymous volume — shadows host node_modules
```

The anonymous volume at `/app/node_modules` takes precedence over the bind mount, so the container uses its own installed modules rather than the host's.

## Volume Inspection and Debugging

```bash
# Find where a named volume is stored on the host
docker volume inspect pgdata --format '{{.Mountpoint}}'
# /var/lib/docker/volumes/pgdata/_data

# Browse volume contents with a temporary container
docker run --rm -it \
  -v pgdata:/data:ro \
  alpine \
  sh -c "ls -la /data && du -sh /data/*"

# Check volume disk usage
docker system df -v | grep VOLUME -A 100
```

## The `.github/workflows/` Directory

GitHub Actions workflows are YAML files stored in `.github/workflows/` at the root of your repository. GitHub automatically discovers and registers every `.yml` or `.yaml` file in that directory.

```
your-repo/
├── .github/
│   └── workflows/
│       ├── ci.yml          # runs on push/PR
│       ├── release.yml     # runs on version tags
│       └── nightly.yml     # scheduled cron job
├── src/
└── package.json
```

You can have multiple workflow files — they all run independently and in parallel when their triggers fire.

## Anatomy of a Workflow File

Every workflow has three top-level sections:

```yaml
name: CI # Display name in GitHub UI

on: # Trigger(s) — what starts this workflow
  push:
    branches: [main]
  pull_request:
    branches: [main]

jobs: # One or more jobs to run
  build: # Job ID (used for references)
    runs-on: ubuntu-latest # Runner OS
    steps: # Ordered list of tasks
      - uses: actions/checkout@v4
      - run: echo "Hello, world!"
```

## Your First CI Workflow

A practical Node.js CI pipeline that runs on every push and pull request:

```yaml
name: CI

on:
  push:
    branches: [main, develop]
  pull_request:
    branches: [main, develop]

jobs:
  test:
    name: Test (Node ${{ matrix.node-version }})
    runs-on: ubuntu-latest

    strategy:
      matrix:
        node-version: [18, 20, 22]

    steps:
      - name: Checkout repository
        uses: actions/checkout@v4

      - name: Set up Node.js ${{ matrix.node-version }}
        uses: actions/setup-node@v4
        with:
          node-version: ${{ matrix.node-version }}
          cache: npm

      - name: Install dependencies
        run: npm ci

      - name: Run linter
        run: npm run lint

      - name: Run tests
        run: npm test

      - name: Build
        run: npm run build
```

:::tip[Use `npm ci` in CI]
Always use `npm ci` instead of `npm install` in CI pipelines. It installs exact versions from `package-lock.json`, is faster, and fails if the lockfile is out of sync with `package.json`.
:::

## Trigger Syntax (`on:`)

### Push Triggers

```yaml
on:
  push:
    branches:
      - main
      - 'release/**' # glob pattern
    branches-ignore:
      - 'dependabot/**'
    paths:
      - 'src/**' # only trigger when src/ changes
      - 'package.json'
    paths-ignore:
      - '**.md' # ignore markdown changes
    tags:
      - 'v*' # trigger on version tags
```

### Pull Request Triggers

```yaml
on:
  pull_request:
    branches: [main]
    types:
      - opened
      - synchronize # new commits pushed to PR
      - reopened
      - ready_for_review
```

### Manual Trigger with Inputs

```yaml
on:
  workflow_dispatch:
    inputs:
      environment:
        description: 'Target environment'
        required: true
        default: staging
        type: choice
        options: [staging, production]
      run_migrations:
        description: 'Run database migrations'
        required: false
        type: boolean
        default: false
      version:
        description: 'Version tag to deploy'
        required: true
        type: string
```

### Scheduled Trigger (Cron)

```yaml
on:
  schedule:
    - cron: '0 2 * * 1-5' # 2 AM UTC, Monday-Friday
    - cron: '0 8 * * 0' # 8 AM UTC, every Sunday
```

:::caution[Schedule Minimum Interval]
The minimum cron interval is **5 minutes**. Scheduled workflows only run on the **default branch**. They may be delayed during high-load periods on GitHub's infrastructure.
:::

## Job Dependencies with `needs`

Jobs run in parallel by default. Use `needs` to create a dependency chain:

```yaml
jobs:
  lint:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: npm ci && npm run lint

  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: npm ci && npm test

  build:
    runs-on: ubuntu-latest
    needs: [lint, test] # waits for both lint and test to succeed
    steps:
      - uses: actions/checkout@v4
      - run: npm ci && npm run build

  deploy:
    runs-on: ubuntu-latest
    needs: build # waits for build
    steps:
      - run: echo "Deploying..."
```

## Step Types: `uses` vs `run`

### Using a Pre-built Action (`uses`)

```yaml
- name: Checkout code
  uses: actions/checkout@v4 # GitHub official action
  with:
    fetch-depth: 0 # full history (default: 1)
    token: ${{ secrets.PAT }} # custom token if needed

- name: Set up Node.js
  uses: actions/setup-node@v4
  with:
    node-version: '20'
    node-version-file: '.nvmrc' # read version from .nvmrc
    cache: 'npm' # or 'yarn' or 'pnpm'
    registry-url: 'https://registry.npmjs.org'

- name: Cache dependencies
  uses: actions/cache@v4
  with:
    path: ~/.npm
    key: ${{ runner.os }}-npm-${{ hashFiles('**/package-lock.json') }}
    restore-keys: |
      ${{ runner.os }}-npm-
```

### Running Shell Commands (`run`)

```yaml
- name: Single-line command
  run: npm test

- name: Multi-line script
  run: |
    echo "Running database migrations..."
    npm run migrate
    echo "Migration complete"

- name: With working directory
  run: npm install
  working-directory: ./packages/api

- name: Explicit shell
  run: |
    Write-Host "Windows PowerShell step"
    Get-ChildItem
  shell: pwsh

- name: Python step
  run: |
    import json, os
    print(json.dumps({"runner": os.environ.get("RUNNER_OS")}))
  shell: python
```

## Environment Variables

Environment variables can be set at three levels — workflow, job, and step:

```yaml
name: Env Demo

env: # workflow-level (available to all jobs/steps)
  NODE_ENV: test
  API_BASE_URL: https://api.example.com

jobs:
  build:
    runs-on: ubuntu-latest
    env: # job-level (available to all steps in this job)
      DATABASE_URL: postgres://localhost/testdb
    steps:
      - uses: actions/checkout@v4

      - name: Build with step-level env
        run: npm run build
        env: # step-level (only this step)
          SENTRY_DSN: ${{ secrets.SENTRY_DSN }}
          BUILD_VERSION: ${{ github.sha }}

      - name: Access env vars
        run: |
          echo "Node env: $NODE_ENV"
          echo "DB: $DATABASE_URL"
          echo "SHA: $BUILD_VERSION"
```

## Step Control Options

### `continue-on-error`

Allow subsequent steps to run even if this step fails:

```yaml
- name: Run optional lint check
  run: npm run lint:strict
  continue-on-error: true # workflow continues even if this fails

- name: Always runs after lint
  run: echo "Lint check done"
```

### `timeout-minutes`

Cap how long a step or job can run:

```yaml
jobs:
  test:
    runs-on: ubuntu-latest
    timeout-minutes: 20 # job-level timeout

    steps:
      - name: Run e2e tests
        run: npm run test:e2e
        timeout-minutes: 15 # step-level timeout
```

### `if` Conditions

Conditionally run steps or jobs:

```yaml
- name: Deploy to production
  if: github.ref == 'refs/heads/main' && github.event_name == 'push'
  run: ./deploy.sh production

- name: Notify on failure
  if: failure()
  uses: actions/github-script@v7
  with:
    script: |
      github.rest.issues.createComment({
        issue_number: context.issue.number,
        owner: context.repo.owner,
        repo: context.repo.repo,
        body: '❌ CI failed on this PR'
      })
```

## Working with `actions/checkout@v4`

The checkout action clones your repository onto the runner:

```yaml
- uses: actions/checkout@v4
  with:
    # Repository to check out (default: current repo)
    repository: ${{ github.repository }}

    # Branch/tag/SHA to checkout (default: the triggering ref)
    ref: ${{ github.head_ref }}

    # Number of commits to fetch (0 = full history)
    fetch-depth: 0

    # Fetch tags even when fetch-depth > 0
    fetch-tags: true

    # Check out submodules
    submodules: recursive

    # Use a PAT instead of GITHUB_TOKEN
    token: ${{ secrets.MY_PAT }}

    # Persist credentials for subsequent git commands
    persist-credentials: true
```

## Working with `actions/setup-node@v4`

```yaml
- uses: actions/setup-node@v4
  with:
    # Exact version, range, or 'lts/*'
    node-version: '20.x'

    # Read version from .nvmrc or .node-version
    node-version-file: '.nvmrc'

    # Enable built-in caching (npm, yarn, or pnpm)
    cache: npm

    # Path to package-lock for cache key (auto-detected if not set)
    cache-dependency-path: package-lock.json

    # Configure npm registry for publishing
    registry-url: 'https://registry.npmjs.org'
    scope: '@my-org'
```

## Complete Real-World CI Example

```yaml
name: CI Pipeline

on:
  push:
    branches: [main]
  pull_request:
    types: [opened, synchronize, reopened]

concurrency:
  group: ${{ github.workflow }}-${{ github.ref }}
  cancel-in-progress: true

jobs:
  quality:
    name: Code Quality
    runs-on: ubuntu-latest
    timeout-minutes: 10
    steps:
      - uses: actions/checkout@v4

      - uses: actions/setup-node@v4
        with:
          node-version: 20
          cache: npm

      - run: npm ci

      - run: npm run lint
      - run: npm run type-check
      - run: npm run format:check

  test:
    name: Tests
    runs-on: ubuntu-latest
    timeout-minutes: 15
    needs: quality
    steps:
      - uses: actions/checkout@v4

      - uses: actions/setup-node@v4
        with:
          node-version: 20
          cache: npm

      - run: npm ci

      - name: Run tests with coverage
        run: npm run test:coverage

      - name: Upload coverage report
        uses: actions/upload-artifact@v4
        if: always()
        with:
          name: coverage-report
          path: coverage/
          retention-days: 7

  build:
    name: Build
    runs-on: ubuntu-latest
    timeout-minutes: 15
    needs: test
    steps:
      - uses: actions/checkout@v4

      - uses: actions/setup-node@v4
        with:
          node-version: 20
          cache: npm

      - run: npm ci
      - run: npm run build

      - uses: actions/upload-artifact@v4
        with:
          name: dist
          path: dist/
          retention-days: 1
```

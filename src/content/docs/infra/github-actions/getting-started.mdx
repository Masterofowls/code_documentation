## The `.github/workflows/` Directory

All workflow files live in `.github/workflows/` in your repository root. Each `.yml` or `.yaml` file in that directory is a separate workflow. GitHub automatically discovers and registers them.

```bash
.github/
└── workflows/
    ├── ci.yml           # Runs on push/PR
    ├── release.yml      # Runs on new tags
    └── codeql.yml       # Security scanning
```

File names don't affect behavior — only the `name:` field and `on:` trigger block matter.

## YAML Workflow Structure

Every workflow file follows this top-level structure:

```yaml
name: CI                          # Display name in the GitHub UI

on:                               # Trigger events
  push:
    branches: [main, develop]
  pull_request:
    branches: [main]

env:                              # Workflow-level environment variables
  NODE_VERSION: '20'

jobs:                             # One or more jobs
  build:                          # Job ID (arbitrary key)
    runs-on: ubuntu-latest        # Runner
    steps:                        # Sequential steps
      - uses: actions/checkout@v4
      - run: echo "Hello, world!"
```

The four top-level keys are:
- `name` — displayed in GitHub UI (optional but recommended)
- `on` — what triggers the workflow (required)
- `env` — environment variables available to all jobs
- `jobs` — the actual work to perform (required)

## Your First CI Workflow

This workflow runs on every push and pull request, checks out your code, installs Node.js dependencies, and runs tests:

```yaml
name: CI

on:
  push:
    branches: [main, develop]
  pull_request:
    branches: [main]

jobs:
  test:
    runs-on: ubuntu-latest

    steps:
      - name: Checkout code
        uses: actions/checkout@v4

      - name: Set up Node.js
        uses: actions/setup-node@v4
        with:
          node-version: '20'
          cache: 'npm'

      - name: Install dependencies
        run: npm ci

      - name: Run linter
        run: npm run lint

      - name: Run tests
        run: npm test

      - name: Build
        run: npm run build
```

<Steps>
1. Create the file at `.github/workflows/ci.yml`
2. Push it to your default branch
3. Navigate to the **Actions** tab in your GitHub repository
4. Watch the workflow run in real time
</Steps>

:::tip[Use `npm ci` not `npm install`]
`npm ci` is faster and more reliable in CI — it installs exactly what's in `package-lock.json` and fails if the lockfile is out of sync. Always use `npm ci` in workflows.
:::

## Workflow Triggers (`on:`)

### Push and Pull Request

```yaml
on:
  push:
    branches:
      - main
      - 'release/**'    # glob pattern
    tags:
      - 'v*'            # trigger on version tags
    paths:
      - 'src/**'        # only when src/ changes
      - '!src/**/*.md'  # exclude markdown files
  pull_request:
    branches: [main]
    types: [opened, synchronize, reopened]
    paths-ignore:
      - 'docs/**'
      - '*.md'
```

### Manual Trigger (`workflow_dispatch`)

Add a button to run the workflow manually from the GitHub UI or API:

```yaml
on:
  workflow_dispatch:
    inputs:
      environment:
        description: 'Target environment'
        required: true
        type: choice
        options: [staging, production]
        default: staging
      debug:
        description: 'Enable debug logging'
        required: false
        type: boolean
        default: false
      version:
        description: 'Version to deploy'
        required: false
        type: string

jobs:
  deploy:
    runs-on: ubuntu-latest
    steps:
      - run: echo "Deploying to ${{ inputs.environment }}"
      - run: echo "Version: ${{ inputs.version }}"
```

### Scheduled Workflows

```yaml
on:
  schedule:
    - cron: '0 6 * * 1-5'   # 6am UTC, Monday–Friday
    - cron: '0 0 * * 0'     # midnight Sunday
```

Cron syntax: `minute hour day-of-month month day-of-week`

:::caution[Scheduled workflows and forks]
Scheduled workflows are disabled in forked repositories. They only run in the original repository.
:::

## Job Dependencies with `needs`

By default, jobs run in parallel. Use `needs:` to create a dependency chain:

```yaml
jobs:
  lint:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: npm run lint

  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: npm test

  build:
    runs-on: ubuntu-latest
    needs: [lint, test]        # waits for both to succeed
    steps:
      - uses: actions/checkout@v4
      - run: npm run build

  deploy:
    runs-on: ubuntu-latest
    needs: build               # waits for build
    steps:
      - run: echo "Deploying..."
```

## Steps: `uses` vs `run`

### `uses` — Call a Pre-Built Action

```yaml
- name: Checkout
  uses: actions/checkout@v4
  with:
    fetch-depth: 0         # full history
    token: ${{ secrets.GITHUB_TOKEN }}
    submodules: recursive
```

Always pin actions to a specific version tag. Avoid `@main` or `@master` in production workflows.

### `run` — Execute Shell Commands

```yaml
- name: Build and test
  run: |
    npm ci
    npm run build
    npm test
  shell: bash
  working-directory: ./packages/app
```

Multi-line `run` commands use `|` for literal block scalar — each line runs as part of the same shell session.

## `actions/checkout@v4`

The most commonly used action — checks out your repository so subsequent steps can access the code:

```yaml
- uses: actions/checkout@v4
  with:
    # Default: checks out the triggering commit
    ref: ${{ github.head_ref }}         # PR branch
    fetch-depth: 0                       # 0 = full history (needed for git log)
    token: ${{ secrets.PAT_TOKEN }}      # use PAT to trigger downstream workflows
    path: my-app                         # checkout to subdirectory
    sparse-checkout: |                   # only checkout specific paths
      src
      package.json
```

## `actions/setup-node@v4`

Sets up the Node.js version and optionally enables dependency caching:

```yaml
- uses: actions/setup-node@v4
  with:
    node-version: '20'           # exact version
    node-version-file: '.nvmrc'  # read from .nvmrc / .node-version
    cache: 'npm'                 # npm | yarn | pnpm
    cache-dependency-path: |     # custom lockfile path(s)
      packages/*/package-lock.json
    registry-url: 'https://registry.npmjs.org'
    scope: '@myorg'
```

Similar setup actions exist for other languages:
- `actions/setup-python@v5` — Python
- `actions/setup-java@v4` — Java / Kotlin
- `actions/setup-go@v5` — Go
- `actions/setup-dotnet@v4` — .NET
- `oven-sh/setup-bun@v2` — Bun

## Step Options

### `working-directory`

Run a step in a specific directory:

```yaml
- name: Install frontend deps
  run: npm ci
  working-directory: ./frontend

- name: Install backend deps
  run: pip install -r requirements.txt
  working-directory: ./backend
```

### `shell`

Override the default shell for a step:

```yaml
- name: PowerShell step
  run: Get-ChildItem
  shell: pwsh

- name: Python inline
  run: |
    import sys
    print(f"Python {sys.version}")
  shell: python

- name: Bash explicit
  run: echo $SHELL
  shell: bash
```

Available shells: `bash`, `sh`, `pwsh`, `powershell`, `cmd`, `python`

### `continue-on-error`

Allow a step to fail without failing the job:

```yaml
- name: Optional lint check
  run: npm run lint:strict
  continue-on-error: true

- name: Always runs
  run: echo "This step always runs"
```

### `timeout-minutes`

Prevent runaway steps from consuming excessive minutes:

```yaml
- name: Long-running test suite
  run: npm run test:e2e
  timeout-minutes: 30      # fail if exceeds 30 minutes
```

You can also set `timeout-minutes` at the job level.

## Environment Variables

Variables can be set at three scopes, with more specific scopes overriding broader ones:

### Workflow Level

```yaml
env:
  NODE_ENV: test
  API_URL: https://api.example.com

jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - run: echo $NODE_ENV    # "test"
```

### Job Level

```yaml
jobs:
  deploy:
    runs-on: ubuntu-latest
    env:
      DEPLOY_TARGET: production
    steps:
      - run: echo $DEPLOY_TARGET
```

### Step Level

```yaml
steps:
  - name: Run with specific env
    run: echo $MY_VAR
    env:
      MY_VAR: hello-from-step
      DATABASE_URL: ${{ secrets.DATABASE_URL }}
```

## A Complete Real-World CI Workflow

```yaml
name: CI Pipeline

on:
  push:
    branches: [main]
  pull_request:
    branches: [main]
    types: [opened, synchronize, reopened]

concurrency:
  group: ci-${{ github.ref }}
  cancel-in-progress: true

env:
  NODE_VERSION: '20'

jobs:
  lint:
    name: Lint & Type Check
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - uses: actions/setup-node@v4
        with:
          node-version: ${{ env.NODE_VERSION }}
          cache: 'npm'

      - run: npm ci
      - run: npm run lint
      - run: npx tsc --noEmit

  test:
    name: Unit Tests
    runs-on: ubuntu-latest
    needs: lint
    strategy:
      matrix:
        node: ['18', '20', '22']
    steps:
      - uses: actions/checkout@v4

      - uses: actions/setup-node@v4
        with:
          node-version: ${{ matrix.node }}
          cache: 'npm'

      - run: npm ci
      - run: npm test -- --coverage

      - uses: actions/upload-artifact@v4
        if: matrix.node == '20'
        with:
          name: coverage-report
          path: coverage/
          retention-days: 7

  build:
    name: Build
    runs-on: ubuntu-latest
    needs: test
    steps:
      - uses: actions/checkout@v4

      - uses: actions/setup-node@v4
        with:
          node-version: ${{ env.NODE_VERSION }}
          cache: 'npm'

      - run: npm ci
      - run: npm run build

      - uses: actions/upload-artifact@v4
        with:
          name: build-output
          path: dist/
          retention-days: 1
```

:::tip[Concurrency cancellation]
The `concurrency` block at the top cancels any in-progress workflow run for the same branch when a new push arrives. This saves minutes and keeps your run queue clean on active branches.
:::

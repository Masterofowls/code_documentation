GitHub Actions workflows live in your repository — no external service account needed. This page walks you through the directory structure, YAML syntax, and a complete working CI workflow.

## The `.github/workflows/` Directory

All workflow files must be placed inside `.github/workflows/` at the root of your repository. GitHub automatically detects any `.yml` or `.yaml` file in this directory.

```
your-repo/
├── .github/
│   └── workflows/
│       ├── ci.yml          # Runs on push/PR
│       ├── release.yml     # Runs on tag push
│       └── nightly.yml     # Runs on schedule
├── src/
└── package.json
```

A repository can have as many workflow files as needed. Each workflow is independent and can have different triggers, runners, and job definitions.

## Workflow YAML Structure

Every GitHub Actions workflow file shares the same top-level structure:

```yaml
name: CI                     # Optional: display name in the Actions tab

on:                          # Required: event trigger(s)
  push:
    branches: [main]

jobs:                        # Required: one or more jobs
  build:                     # Job ID (used for dependencies)
    runs-on: ubuntu-latest   # Runner label
    steps:                   # Ordered list of steps
      - name: Checkout
        uses: actions/checkout@v4
```

The three required top-level keys are `on`, and `jobs`. `name` is optional but strongly recommended for readability in the GitHub UI.

## Your First CI Workflow

Here is a complete Node.js CI workflow that runs on every push and pull request:

```yaml
name: CI

on:
  push:
    branches: [main, develop]
  pull_request:
    branches: [main, develop]

jobs:
  test:
    name: Test (Node ${{ matrix.node-version }})
    runs-on: ubuntu-latest

    strategy:
      matrix:
        node-version: [18, 20, 22]

    steps:
      - name: Checkout repository
        uses: actions/checkout@v4

      - name: Set up Node.js ${{ matrix.node-version }}
        uses: actions/setup-node@v4
        with:
          node-version: ${{ matrix.node-version }}
          cache: 'npm'

      - name: Install dependencies
        run: npm ci

      - name: Run linter
        run: npm run lint

      - name: Run tests
        run: npm test

      - name: Build
        run: npm run build
```

This workflow:
1. Triggers on pushes to `main`/`develop` and on any PR targeting those branches
2. Runs three parallel jobs — one per Node.js version — using a matrix strategy
3. Caches npm dependencies automatically via `setup-node`'s `cache: 'npm'` option
4. Runs lint, test, and build in sequence within each job

## Trigger Syntax in Detail

### Push and Pull Request

```yaml
on:
  push:
    branches:
      - main
      - 'releases/**'     # Glob pattern
    branches-ignore:
      - 'dependabot/**'
    paths:
      - 'src/**'          # Only trigger when src/ changes
      - 'package.json'
    paths-ignore:
      - '**.md'           # Skip markdown-only changes
    tags:
      - 'v*'              # Trigger on version tags

  pull_request:
    types:
      - opened
      - synchronize       # New commits pushed to PR
      - reopened
    branches:
      - main
```

### Manual Trigger (workflow_dispatch)

```yaml
on:
  workflow_dispatch:
    inputs:
      environment:
        description: 'Target environment'
        required: true
        default: 'staging'
        type: choice
        options:
          - staging
          - production
      debug_mode:
        description: 'Enable debug logging'
        required: false
        type: boolean
        default: false
      version:
        description: 'Version to deploy'
        required: true
        type: string
```

Access inputs in the workflow with `${{ inputs.environment }}`.

### Schedule (Cron)

```yaml
on:
  schedule:
    - cron: '0 2 * * 1-5'   # 2 AM UTC, Monday–Friday
    - cron: '0 6 * * 0'     # 6 AM UTC, Sundays
```

Cron format: `minute hour day-of-month month day-of-week`. Scheduled workflows only run on the **default branch**.

:::caution[Schedule precision]
GitHub does not guarantee exact schedule execution. During periods of high load, scheduled workflows may be delayed by several minutes. Do not use schedules for time-critical operations.
:::

## Job Dependencies with `needs`

Jobs run in parallel by default. Use `needs:` to create a dependency chain:

```yaml
jobs:
  lint:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: npm run lint

  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: npm test

  build:
    runs-on: ubuntu-latest
    needs: [lint, test]     # Both must pass first
    steps:
      - uses: actions/checkout@v4
      - run: npm run build

  deploy:
    runs-on: ubuntu-latest
    needs: build            # Runs after build
    steps:
      - run: echo "Deploying..."
```

## Step Types: `uses` vs `run`

Every step is either an **action** (`uses:`) or a **shell command** (`run:`):

```yaml
steps:
  # Action step — runs a prebuilt action
  - name: Checkout code
    uses: actions/checkout@v4
    with:
      fetch-depth: 0        # Full git history

  # Shell step — runs inline shell command(s)
  - name: Print Node version
    run: node --version

  # Multi-line shell step
  - name: Build and verify
    run: |
      npm run build
      ls -la dist/
      echo "Build complete"

  # Shell step with specific shell
  - name: PowerShell step
    shell: pwsh
    run: Write-Host "Hello from PowerShell"
```

## Essential Actions

### `actions/checkout@v4`

Checks out your repository code onto the runner. Almost every workflow starts with this step.

```yaml
- uses: actions/checkout@v4
  with:
    # Number of commits to fetch. 0 = full history (needed for git log, tags)
    fetch-depth: 1          # Default: 1 (shallow clone, fastest)

    # Check out a specific ref (branch, tag, SHA)
    ref: ${{ github.event.pull_request.head.sha }}

    # Use a different token (e.g., to push back to repo)
    token: ${{ secrets.GITHUB_TOKEN }}

    # Check out submodules
    submodules: recursive

    # Persist credentials for subsequent git commands
    persist-credentials: true
```

### `actions/setup-node@v4`

Installs a specific Node.js version and optionally enables dependency caching.

```yaml
- uses: actions/setup-node@v4
  with:
    node-version: '20'        # Exact version
    # node-version: '20.x'   # Latest 20.x patch
    # node-version-file: '.nvmrc'  # Read from file

    # Enable caching for npm, yarn, or pnpm
    cache: 'npm'
    # cache-dependency-path: 'subdir/package-lock.json'

    # Use a specific registry
    registry-url: 'https://registry.npmjs.org'
```

After `setup-node` with `cache: 'npm'`, the action automatically caches `~/.npm` based on `package-lock.json`. Subsequent runs restore the cache, skipping the network download.

## Step Configuration Options

### `working-directory`

Run a step in a subdirectory:

```yaml
- name: Run tests in backend
  working-directory: ./backend
  run: npm test
```

### `continue-on-error`

Allow a step to fail without failing the job:

```yaml
- name: Lint (non-blocking)
  continue-on-error: true
  run: npm run lint
```

### `timeout-minutes`

Prevent a hung step from running forever:

```yaml
- name: Integration tests
  timeout-minutes: 30
  run: npm run test:integration
```

The default job timeout is 6 hours. Individual step timeouts override the job default.

### `if` Conditions on Steps

Run a step only under certain conditions:

```yaml
steps:
  - name: Run on main only
    if: github.ref == 'refs/heads/main'
    run: ./deploy.sh

  - name: Run on failure
    if: failure()
    run: ./notify-failure.sh

  - name: Always run cleanup
    if: always()
    run: ./cleanup.sh
```

## Environment Variables

Define environment variables at the workflow, job, or step level:

```yaml
name: CI

env:
  NODE_ENV: test            # Available to all jobs
  API_URL: https://api.dev

jobs:
  test:
    runs-on: ubuntu-latest
    env:
      DATABASE_URL: sqlite://test.db   # Available to all steps in this job
    steps:
      - name: Run with step-level env
        env:
          DEBUG: 'true'               # Only this step
        run: |
          echo "NODE_ENV=$NODE_ENV"
          echo "DEBUG=$DEBUG"
```

Environment variables at a narrower scope override those at a wider scope (step > job > workflow).

## A Complete Real-World CI Workflow

```yaml
name: CI Pipeline

on:
  push:
    branches: [main]
  pull_request:
    types: [opened, synchronize, reopened]

concurrency:
  group: ci-${{ github.ref }}
  cancel-in-progress: true

jobs:
  lint-and-typecheck:
    name: Lint & Type Check
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - uses: actions/setup-node@v4
        with:
          node-version: '20'
          cache: 'npm'

      - run: npm ci

      - name: Lint
        run: npm run lint

      - name: Type check
        run: npx tsc --noEmit

  test:
    name: Test Suite
    runs-on: ubuntu-latest
    needs: lint-and-typecheck
    steps:
      - uses: actions/checkout@v4

      - uses: actions/setup-node@v4
        with:
          node-version: '20'
          cache: 'npm'

      - run: npm ci

      - name: Run tests with coverage
        run: npm test -- --coverage

      - name: Upload coverage report
        uses: actions/upload-artifact@v4
        with:
          name: coverage-report
          path: coverage/
          retention-days: 7

  build:
    name: Build
    runs-on: ubuntu-latest
    needs: test
    steps:
      - uses: actions/checkout@v4

      - uses: actions/setup-node@v4
        with:
          node-version: '20'
          cache: 'npm'

      - run: npm ci

      - name: Build production bundle
        run: npm run build

      - name: Upload build artifacts
        uses: actions/upload-artifact@v4
        with:
          name: dist
          path: dist/
          retention-days: 1
```

:::tip[Use `npm ci` instead of `npm install`]
`npm ci` is faster in CI environments — it installs exactly what's in `package-lock.json`, never updates it, and clears `node_modules` first. Always use `npm ci` in CI workflows.
:::

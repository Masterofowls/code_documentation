GitHub Actions is a native CI/CD and automation platform built directly into GitHub. It lets you automate your build, test, and deployment pipelines using YAML workflows stored alongside your code in `.github/workflows/`.

## What Is GitHub Actions?

GitHub Actions turns your repository into an automation engine. Every time something happens in your repo — a push, a pull request, a comment, a release, a scheduled time — GitHub Actions can respond by running a workflow: a sequence of jobs and steps that execute code on virtual machines (runners) hosted by GitHub or on your own infrastructure.

:::tip[Why GitHub Actions?]
Because your CI/CD pipeline lives in the same repository as your code, it's versioned, reviewable, and testable alongside every change. No separate CI server configuration to manage.
:::

## CI/CD Concepts

**Continuous Integration (CI)** is the practice of automatically building and testing code every time a change is pushed to the repository. The goal is to catch integration problems early, before they compound.

**Continuous Delivery (CD)** extends CI by automatically deploying the tested build to a staging or preview environment, keeping it always ready to ship.

**Continuous Deployment** goes one step further — every passing build is automatically deployed to production without human intervention.

GitHub Actions supports all three stages. A single workflow file can lint, test, build, and deploy your application end-to-end.

## GitHub Actions vs Other CI/CD Platforms

| Feature | GitHub Actions | Jenkins | CircleCI | GitLab CI |
|---|---|---|---|---|
| **Hosting** | GitHub-hosted + self-hosted | Self-hosted | Cloud + self-hosted | GitLab-hosted + self-hosted |
| **Config format** | YAML in repo | Jenkinsfile (Groovy) | YAML in repo | YAML in repo |
| **Setup effort** | Zero (native to GitHub) | High (server, plugins) | Low | Low (if using GitLab) |
| **Free tier** | 2,000 min/month (private repos) | Free (self-hosted) | 6,000 min/month | 400 min/month |
| **Marketplace** | 20,000+ actions | Plugin ecosystem | Orbs | Components |
| **OIDC cloud auth** | Native | Via plugins | Supported | Supported |
| **Reusable workflows** | First-class | Shared libraries | Reusable config | `include` templates |
| **Matrix builds** | Built-in | Via plugins | Built-in | Built-in |
| **Self-hosted runners** | Yes | Yes (it IS self-hosted) | Yes | Yes |
| **Repository integration** | Native | Webhook-based | OAuth | Native |

:::caution[Jenkins Trade-offs]
Jenkins is extremely flexible but requires significant operational overhead — server maintenance, plugin updates, and security patching. For most teams using GitHub, GitHub Actions eliminates this burden entirely.
:::

## Key Concepts

### Workflow
A **workflow** is an automated process defined in a YAML file inside `.github/workflows/`. A repository can have multiple workflows. Each workflow is triggered by one or more events.

### Trigger (Event)
A **trigger** or **event** is what causes a workflow to run. Events include `push`, `pull_request`, `schedule`, `workflow_dispatch` (manual), `release`, and dozens more.

### Job
A **job** is a set of steps that execute on the same runner. Jobs within a workflow run in parallel by default, but you can create dependencies with `needs:`.

### Step
A **step** is a single task within a job. Steps run sequentially and can either execute a shell command (`run:`) or invoke a pre-built action (`uses:`).

### Action
An **action** is a reusable unit of code — either a JavaScript application, a Docker container, or a composite set of steps — packaged to perform a specific task (e.g., `actions/checkout`, `actions/setup-node`).

### Runner
A **runner** is the virtual machine or container that executes your jobs. GitHub provides hosted runners; you can also register your own self-hosted runners.

### Artifact
An **artifact** is a file or set of files produced during a workflow run (build outputs, test reports, coverage data) that can be uploaded, retained, and downloaded.

### Environment
An **environment** is a named deployment target (e.g., `staging`, `production`) with its own secrets, protection rules (required reviewers, wait timers), and deployment history.

## GitHub-Hosted Runners

GitHub provides fully managed virtual machines for running your workflows. Each job gets a **fresh, isolated runner** — no state leaks between runs.

### Standard Hosted Runner Specifications

| Runner Label | OS | CPU | RAM | Storage |
|---|---|---|---|---|
| `ubuntu-latest` | Ubuntu 22.04 | 4 vCPUs | 16 GB | 14 GB SSD |
| `ubuntu-22.04` | Ubuntu 22.04 | 4 vCPUs | 16 GB | 14 GB SSD |
| `ubuntu-20.04` | Ubuntu 20.04 | 4 vCPUs | 16 GB | 14 GB SSD |
| `windows-latest` | Windows Server 2022 | 4 vCPUs | 16 GB | 14 GB SSD |
| `windows-2022` | Windows Server 2022 | 4 vCPUs | 16 GB | 14 GB SSD |
| `macos-latest` | macOS 14 (Sonoma) | 3 vCPUs (Apple M1) | 7 GB | 14 GB SSD |
| `macos-13` | macOS 13 (Ventura) | 4 vCPUs (Intel) | 14 GB | 14 GB SSD |

### Larger Runners (GitHub Team / Enterprise)

For compute-intensive workloads, GitHub offers larger hosted runners:

| Runner | CPU | RAM | Notes |
|---|---|---|---|
| `ubuntu-latest-4-cores` | 4 vCPUs | 16 GB | 2× standard |
| `ubuntu-latest-8-cores` | 8 vCPUs | 32 GB | 4× standard |
| `ubuntu-latest-16-cores` | 16 vCPUs | 64 GB | 8× standard |
| `ubuntu-latest-64-cores` | 64 vCPUs | 256 GB | GPU available |
| `macos-latest-xlarge` | 6 vCPUs (M1) | 14 GB | macOS ARM |

### Pre-Installed Software

GitHub-hosted runners come with a large suite of pre-installed tools:

- **Languages**: Node.js, Python, Ruby, Java, Go, Rust, .NET, PHP
- **Tools**: Docker, Git, npm, pip, Homebrew (macOS), Chocolatey (Windows), curl, jq, wget
- **Cloud CLIs**: AWS CLI, Azure CLI, Google Cloud SDK
- See the full list at [actions/runner-images](https://github.com/actions/runner-images)

## Self-Hosted Runners

You can register your own machines as runners for cases where GitHub-hosted runners are insufficient:

- **Compliance**: data cannot leave your network
- **Performance**: specialized hardware (GPU, high-memory, ARM)
- **Cost**: reduce billable minutes for large-scale builds
- **Custom software**: proprietary tools pre-installed

```bash
# Download and configure a self-hosted runner (Linux example)
mkdir actions-runner && cd actions-runner
curl -o actions-runner-linux-x64-2.319.0.tar.gz -L \
  https://github.com/actions/runner/releases/download/v2.319.0/actions-runner-linux-x64-2.319.0.tar.gz
tar xzf ./actions-runner-linux-x64-2.319.0.tar.gz
./config.sh --url https://github.com/YOUR_ORG/YOUR_REPO --token YOUR_REGISTRATION_TOKEN
./run.sh
```

:::caution[Self-Hosted Runner Security]
Never use self-hosted runners on public repositories — malicious pull requests could execute arbitrary code on your machines. Always use them only with private repositories or apply strict fork policies.
:::

## Billing

### Free Tier (GitHub-Hosted Minutes)

| Plan | Free minutes/month | Storage |
|---|---|---|
| GitHub Free (public repos) | **Unlimited** | 500 MB |
| GitHub Free (private repos) | 2,000 min | 500 MB |
| GitHub Pro | 3,000 min | 1 GB |
| GitHub Team | 3,000 min | 2 GB |
| GitHub Enterprise | 50,000 min | 50 GB |

### Minute Multipliers

Windows and macOS runners consume minutes at a higher rate due to higher infrastructure costs:

| OS | Multiplier |
|---|---|
| Linux | 1× |
| Windows | 2× |
| macOS | 10× |

So a 10-minute job on macOS costs 100 Linux minutes from your quota.

### Cost After Free Tier

| OS | Price per minute |
|---|---|
| Linux | $0.008 |
| Windows | $0.016 |
| macOS | $0.08 |

:::tip[Cost Optimization]
Use `ubuntu-latest` whenever possible. Cache dependencies with `actions/cache`. Cancel redundant runs with `concurrency`. Skip CI with `[skip ci]` in commit messages for doc-only changes.
:::

## GITHUB_TOKEN

Every workflow run automatically receives a `GITHUB_TOKEN` — a short-lived installation token scoped to the repository. It expires when the workflow finishes.

```yaml
- name: Create a comment
  uses: actions/github-script@v7
  with:
    github-token: ${{ secrets.GITHUB_TOKEN }}
    script: |
      github.rest.issues.createComment({
        issue_number: context.issue.number,
        owner: context.repo.owner,
        repo: context.repo.repo,
        body: 'Thanks for your contribution!'
      })
```

Default permissions are **read-only** for most scopes. You can grant additional permissions at the workflow or job level using the `permissions:` key.

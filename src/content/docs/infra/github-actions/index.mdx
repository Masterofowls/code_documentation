GitHub Actions is GitHub's built-in continuous integration and continuous delivery (CI/CD) platform. It lets you automate your build, test, and deployment pipeline directly from your GitHub repository — no external service required.

## What Is GitHub Actions?

GitHub Actions allows you to create **workflows** — automated processes triggered by events in your repository. When a developer pushes code, opens a pull request, or publishes a release, Actions can automatically run tests, build Docker images, deploy to production, or do virtually anything else you can script.

Each workflow is defined in a YAML file stored in `.github/workflows/` inside your repository. Workflows consist of **jobs**, which consist of **steps**, which run shell commands or call pre-built **actions** from the GitHub Marketplace.

:::tip[Why GitHub Actions?]
Because the CI/CD pipeline lives in the same repository as your code, you get version-controlled workflows, no external service credentials, and tight integration with GitHub pull requests, issues, deployments, and environments.
:::

## CI/CD Concepts

**Continuous Integration (CI)** is the practice of automatically building and testing code every time a change is pushed. This catches bugs early, before they compound.

**Continuous Delivery (CD)** extends CI by automatically deploying validated builds to staging or production environments, ensuring you can release at any time with confidence.

**Continuous Deployment** goes one step further: every passing build is automatically deployed to production with no manual gate.

GitHub Actions supports all three patterns with native integrations for GitHub's environments, secrets, approvals, and deployment APIs.

| Secrets management | Built-in | Credentials plugin | Built-in | Built-in |
| Environments & approvals | Built-in | Manual config | Built-in | Built-in |

| Self-hosted runners | Yes | Core feature | Yes | Yes |

:::caution[GitHub-only]
GitHub Actions workflows only trigger on events from GitHub repositories. If your source code is hosted elsewhere, you'll need a different CI/CD platform or a mirror/webhook setup.
:::

## Key Concepts

### Workflow

A **workflow** is an automated process defined in a YAML file. Workflows live in `.github/workflows/` and are triggered by **events**. A repository can have multiple workflows for different purposes — CI, CD, release management, dependency updates, etc.

### Trigger / Event

An **event** is what causes a workflow to run. Common events include:
- `push` — code pushed to a branch
- `pull_request` — PR opened, synchronized, or merged
- `workflow_dispatch` — manually triggered from the UI or API
- `schedule` — cron-based scheduling
- `release` — a GitHub Release is published

### Job

A **job** is a set of steps that run on the same runner. Jobs run **in parallel by default**; use `needs:` to create sequential dependencies. Each job gets a fresh runner environment.

### Step

A **step** is an individual task within a job. Steps run sequentially in the order defined. Each step either:
- Runs a shell command (`run:`)
- Calls a pre-built action (`uses:`)

### Action

An **action** is a reusable unit of automation — a pre-built step you call with `uses:`. Actions are published to the GitHub Marketplace or defined locally in `.github/actions/`. Examples: `actions/checkout@v4`, `actions/setup-node@v4`.

### Runner

A **runner** is the server that executes the workflow jobs. GitHub provides hosted runners; you can also register your own self-hosted runners.

### Artifact

An **artifact** is a file or directory produced by a workflow and uploaded for later use. Artifacts persist beyond the workflow run and can be downloaded from the UI or by other jobs using `actions/download-artifact`.

### Environment

An **environment** is a named deployment target (e.g., `staging`, `production`) with its own secrets, protection rules, and required reviewers. Jobs can target an environment to trigger approval gates.

## GitHub-Hosted Runners

GitHub provides managed runner machines for every workflow. You specify the runner with `runs-on:`.

For compute-intensive workloads, GitHub offers **larger runners** with configurable specs:

| Label pattern | vCPUs | RAM |

Larger runners also support static IP addresses (useful for firewall allowlisting) and GPU runners for ML workloads.

## Self-Hosted Runners

You can register your own machines as runners — on-premises servers, cloud VMs, or even Raspberry Pis. Self-hosted runners are useful when:

- You need access to internal network resources
- Your build requires proprietary hardware or software licenses
- You want to avoid per-minute billing
- You need specific OS versions or architectures (ARM, Windows Server 2019, etc.)

```bash
# Register a self-hosted runner (example)
# Download runner package from GitHub → Settings → Actions → Runners
./config.sh --url https://github.com/ORG/REPO --token TOKEN
./run.sh
```

:::caution[Self-hosted security]
Self-hosted runners on public repositories are a significant security risk. Any fork can trigger a PR that runs malicious code on your runner. Use self-hosted runners only with private repositories or restrict which workflows can use them.
:::

### Runner Groups

Enterprise and organization accounts can create **runner groups** to restrict which repositories can use which self-hosted runners.

## Billing

### Free Tier

| Repository type | Free minutes/month | Storage |
|---|---|---|
| Public repositories | Unlimited | Unlimited |
| Private repositories | 2,000 minutes | 500 MB |

### Per-Minute Rates for Private Repos

| OS | Multiplier | Effective rate |
|---|---|---|
| Linux | 1× | $0.008/min |
| Windows | 2× | $0.016/min |
| macOS | 10× | $0.08/min |

Larger runners have higher per-minute multipliers. Minutes reset monthly. You can set spending limits in your organization billing settings.

:::tip[Optimize costs]
Use Linux runners whenever possible (10× cheaper than macOS). Cache dependencies with `actions/cache`. Use `paths` filters to skip workflows when unrelated files change.
:::

## GITHUB_TOKEN

Every workflow run automatically receives a `GITHUB_TOKEN` secret — a short-lived token scoped to the repository that expires when the workflow ends. It can:

- Create pull request comments
- Push commits to the repository
- Publish packages to GitHub Packages
- Create releases and deployment statuses
- Trigger other workflow runs (with limitations)

```yaml
- name: Comment on PR
  uses: actions/github-script@v7
  with:
    script: |
      github.rest.issues.createComment({
        issue_number: context.issue.number,
        owner: context.repo.owner,
        repo: context.repo.repo,
        body: '✅ Tests passed!'
      })
```

The default permissions of `GITHUB_TOKEN` can be configured at the repository, organization, or workflow level.

GitHub Actions is a native CI/CD and automation platform built directly into GitHub. It lets you automate workflows — building, testing, and deploying code — triggered by virtually any GitHub event, from a push to a pull request review to a scheduled cron job.

## What Is GitHub Actions?

GitHub Actions allows you to define automated pipelines called **workflows** using YAML files stored in your repository under `.github/workflows/`. Each workflow responds to one or more events and runs a series of jobs on hosted or self-hosted machines.

Unlike external CI systems, GitHub Actions has deep integration with GitHub APIs, pull request checks, deployment environments, packages, and the secrets manager — all without a separate service account or webhook setup.

## Core Concepts

| Concept             | Description                                                                          |
| ------------------- | ------------------------------------------------------------------------------------ |
| **Workflow**        | A YAML-defined automation pipeline stored in `.github/workflows/`                    |
| **Trigger / Event** | A GitHub event (push, pull_request, schedule, etc.) that starts a workflow           |
| **Job**             | A group of steps that execute on the same runner                                     |
| **Step**            | A single task in a job — either a shell command (`run`) or an Action (`uses`)        |
| **Action**          | A reusable unit of automation — can be from the Marketplace or local                 |
| **Runner**          | The virtual machine or container that executes a job                                 |
| **Artifact**        | Files produced by a workflow (build outputs, test reports) persisted for download    |
| **Environment**     | A named deployment target with optional protection rules and secrets                 |
| **Context**         | Objects like `github`, `secrets`, `env`, `matrix` available via `${{ }}` expressions |
| **Expression**      | `${{ }}` interpolation for dynamic values in workflow YAML                           |

## CI/CD Concepts Refresher

**Continuous Integration (CI)** automatically builds and tests code on every commit, detecting integration issues early before they compound.

**Continuous Delivery (CD)** automatically prepares a verified release artifact so a deployment can happen at any time with minimal human intervention.

**Continuous Deployment** goes one step further — every passing commit is automatically deployed to production without manual approval.

GitHub Actions supports all three patterns through a single unified workflow file format.

## GitHub Actions vs Other CI Systems

| Feature                 | GitHub Actions                         | Jenkins                    | CircleCI                      | GitLab CI               |
| ----------------------- | -------------------------------------- | -------------------------- | ----------------------------- | ----------------------- |
| **Hosting**             | Cloud (GitHub-hosted) + self-hosted    | Self-hosted only           | Cloud + self-hosted           | Cloud + self-hosted     |
| **Config format**       | YAML in `.github/workflows/`           | Groovy (Jenkinsfile)       | YAML (`.circleci/config.yml`) | YAML (`.gitlab-ci.yml`) |
| **GitHub integration**  | Native (checks, deployments, packages) | Plugin-based               | Via API                       | Via mirror/webhook      |
| **Marketplace**         | 20,000+ Actions                        | Plugin ecosystem           | Orbs                          | Includes / Templates    |
| **Matrix builds**       | Built-in strategy                      | Manual scripting           | `matrix` key                  | `parallel` + `matrix`   |
| **Reusable workflows**  | `workflow_call` + composite actions    | Shared libraries           | Orbs                          | `include:` + `extends:` |
| **OIDC cloud auth**     | Native (AWS, GCP, Azure)               | Plugin-dependent           | Native                        | Native                  |
| **Free tier**           | 2,000 min/mo (public repos: unlimited) | None (infrastructure cost) | 6,000 credits/mo              | 400 min/mo              |
| **Self-hosted runners** | Yes (any OS, ARM, GPU)                 | Yes (agents)               | Yes (resource classes)        | Yes (runners)           |
| **Approval gates**      | Environment protection rules           | Input step plugin          | Approval jobs                 | Manual jobs             |

:::tip[Public Repositories]
GitHub-hosted runners are **completely free** for public repositories with no minute limits. This makes GitHub Actions the go-to choice for open-source projects.
:::

## GitHub-Hosted Runners

GitHub provides managed virtual machines on three operating systems. Each job gets a fresh, isolated runner instance.

### Standard Runners

| Label            | OS                  | CPU           | RAM  | Storage   |
| ---------------- | ------------------- | ------------- | ---- | --------- |
| `ubuntu-latest`  | Ubuntu 22.04        | 2 cores       | 7 GB | 14 GB SSD |
| `ubuntu-22.04`   | Ubuntu 22.04        | 2 cores       | 7 GB | 14 GB SSD |
| `ubuntu-20.04`   | Ubuntu 20.04        | 2 cores       | 7 GB | 14 GB SSD |
| `windows-latest` | Windows Server 2022 | 2 cores       | 7 GB | 14 GB SSD |
| `macos-latest`   | macOS 14 (Sonoma)   | 3 cores (M1)  | 7 GB | 14 GB SSD |
| `macos-13`       | macOS 13 (Ventura)  | 3 cores (x64) | 7 GB | 14 GB SSD |

### Larger Runners (Paid)

For compute-intensive workloads, GitHub offers larger hosted runners on a per-minute billing model:

| Label                    | CPU          | RAM    | Notes                  |
| ------------------------ | ------------ | ------ | ---------------------- |
| `ubuntu-latest-4-cores`  | 4 cores      | 16 GB  | GitHub Team/Enterprise |
| `ubuntu-latest-8-cores`  | 8 cores      | 32 GB  | GitHub Team/Enterprise |
| `ubuntu-latest-16-cores` | 16 cores     | 64 GB  | GitHub Team/Enterprise |
| `ubuntu-latest-32-cores` | 32 cores     | 128 GB | GitHub Enterprise      |
| `macos-latest-xlarge`    | 6 cores (M1) | 14 GB  | GitHub Team/Enterprise |

### Pre-installed Software

GitHub-hosted runners include an extensive software suite:

- **Build tools**: Node.js (multiple versions via nvm), Python, Java, Ruby, Go, .NET, Rust
- **Containers**: Docker, Docker Compose, Podman (Linux)
- **Cloud CLIs**: AWS CLI, Azure CLI, Google Cloud SDK
- **Utilities**: git, curl, wget, jq, yq, make, cmake, GitHub CLI (`gh`)

```bash
# Check what's installed on a runner
- name: List installed tools
  run: |
    node --version
    python --version
    docker --version
    gh --version
```

:::caution[macOS Runner Costs]
macOS runners consume minutes at a **10x multiplier** compared to Linux. A macOS job that runs for 10 minutes uses 100 billed minutes. Plan accordingly for iOS/macOS builds.
:::

## Self-Hosted Runners

Self-hosted runners let you run jobs on your own infrastructure — on-premise servers, cloud VMs, ARM hardware, or GPU machines.

```bash
# Register a self-hosted runner (from GitHub repo Settings > Actions > Runners)
./config.sh --url https://github.com/YOUR_ORG/YOUR_REPO \
            --token YOUR_REGISTRATION_TOKEN \
            --labels self-hosted,linux,x64,production

# Start the runner
./run.sh
```

Self-hosted runner use cases:

- **Custom hardware**: ARM (Raspberry Pi, AWS Graviton), GPU (ML workloads)
- **Private network access**: databases, internal APIs, on-premise services
- **Cost optimization**: unlimited minutes on your own machines
- **Compliance**: data never leaves your network

Reference self-hosted runners in workflows:

```yaml
jobs:
  build:
    runs-on: [self-hosted, linux, x64, production]
```

:::caution[Self-Hosted Runner Security]
Never use self-hosted runners for public repositories. A malicious pull request could execute arbitrary code on your infrastructure. Use ephemeral runners (auto-delete after each job) in production via GitHub Actions Runner Controller (ARC).
:::

| ------- | ----------------- |
| Linux   | $0.008            |
| Windows | $0.016 (2× Linux) |

```yaml
# Always set timeout-minutes to cap costs
jobs:
  build:
    runs-on: ubuntu-latest
    timeout-minutes: 30 # prevents runaway jobs from draining budget
```

## The GITHUB_TOKEN

Every workflow run automatically receives a short-lived `GITHUB_TOKEN` — a scoped installation token for the `github-actions` bot. It expires when the workflow ends.

```yaml
- name: Create a comment
  uses: actions/github-script@v7
  with:
    github-token: ${{ secrets.GITHUB_TOKEN }}
    script: |
      github.rest.issues.createComment({
        issue_number: context.issue.number,
        owner: context.repo.owner,
        repo: context.repo.repo,
        body: '✅ All checks passed!'
      })
```

Default GITHUB_TOKEN permissions vary by repository setting (restricted vs permissive). Always declare the permissions your workflow needs explicitly:

```yaml
permissions:
  contents: read
  pull-requests: write
  issues: write
```

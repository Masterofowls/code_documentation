GitHub Actions is a continuous integration and continuous delivery (CI/CD) platform built directly into GitHub. It lets you automate your build, test, and deployment pipelines with YAML-defined workflows that respond to any GitHub event.

## What is GitHub Actions?

GitHub Actions enables you to create automated workflows triggered by events in your repository — pushes, pull requests, issue comments, releases, schedules, and more. Workflows run on **runners** (GitHub-hosted or self-hosted virtual machines) and are composed of one or more **jobs**, each containing a sequence of **steps**.

Unlike separate CI services, GitHub Actions is natively integrated with your repository: it has direct access to your code, can read and write issues and pull requests, manage releases, and interact with the GitHub API using the built-in `GITHUB_TOKEN`.

## Core CI/CD Concepts

**Continuous Integration (CI)** is the practice of automatically building and testing code changes when they are pushed. The goal is to detect integration issues early.

**Continuous Delivery (CD)** extends CI by automatically deploying tested changes to staging or production environments, ensuring every successful build is potentially shippable.

**Continuous Deployment** goes one step further — every passing build is automatically pushed to production without manual approval.

GitHub Actions supports all three patterns through configurable workflow triggers, environment protection rules, and deployment gates.

| **Matrix builds** | ✅ Native | ✅ With plugin | ✅ Native | ✅ Native |
| **Reusable workflows** | ✅ Native | Shared libs | Orbs | Templates |

| **Environments + gates** | ✅ Native | Manual | ✅ Approval | ✅ Environments |
| **Setup complexity** | None | High | Low | Low |

:::tip[Best for GitHub users]
If your code lives on GitHub, Actions is the path of least resistance — no extra accounts, no webhook configuration, and `GITHUB_TOKEN` gives you free read/write access to your repository without storing credentials.
:::

## Key Concepts

### Workflow
A YAML file stored in `.github/workflows/` that defines the automation. A repository can have multiple workflows, each triggered by different events. Workflows can be triggered by push events, pull requests, schedules (cron), webhooks, or manually.

### Trigger (Event)
The `on:` key in a workflow file specifies what causes the workflow to run. Events include `push`, `pull_request`, `schedule`, `workflow_dispatch` (manual), `release`, `issues`, and dozens more. See the [Workflows](/infra/github-actions/workflows) page for the full list.

### Job
A set of steps that execute on the same runner. Jobs run in parallel by default; use `needs:` to create sequential dependencies. Each job gets a fresh runner environment.

### Step
An individual task within a job. Steps run sequentially and can either execute a shell command (`run:`) or invoke a reusable action (`uses:`). Steps within a job share the same runner's filesystem and environment variables.

### Action
A reusable unit of automation — either a JavaScript package, Docker container, or composite of shell steps. Actions are referenced with `uses:` and parameterized with `with:`. The [GitHub Marketplace](https://github.com/marketplace?type=actions) hosts thousands of community actions.

### Runner
The server that executes your jobs. GitHub provides hosted runners (Ubuntu, Windows, macOS) or you can host your own. Each job runs on a fresh, isolated runner instance.

### Artifact
Files produced during a workflow run that you want to persist or share between jobs. Uploaded with `actions/upload-artifact` and downloaded with `actions/download-artifact`.

### Environment
A deployment target (e.g., `staging`, `production`) with optional protection rules such as required reviewers, wait timers, and environment-scoped secrets.

## GitHub-Hosted Runners

GitHub provides managed runners for every major operating system. These are ephemeral — provisioned fresh for each job and torn down when the job completes.

| Label | OS | Cores | RAM | Storage |
|---|---|---|---|---|
| `ubuntu-latest` | Ubuntu 24.04 | 4 | 16 GB | 14 GB SSD |
| `ubuntu-22.04` | Ubuntu 22.04 | 4 | 16 GB | 14 GB SSD |
| `ubuntu-20.04` | Ubuntu 20.04 | 4 | 16 GB | 14 GB SSD |
| `windows-latest` | Windows Server 2022 | 4 | 16 GB | 14 GB SSD |
| `windows-2022` | Windows Server 2022 | 4 | 16 GB | 14 GB SSD |
| `macos-latest` | macOS 14 (Sonoma) | 3 (M1) | 7 GB | 14 GB SSD |
| `macos-13` | macOS 13 (Ventura) | 4 (Intel) | 14 GB | 14 GB SSD |

### Larger Runners

For CPU/memory-intensive workloads, GitHub offers **larger runners** with up to 64 cores and 256 GB RAM. These require a GitHub Team or Enterprise plan and are configured in your organization settings.

```yaml
jobs:
  heavy-build:
    runs-on: ubuntu-latest-16-cores  # 16-core larger runner
```

### Preinstalled Software

GitHub-hosted runners come with an extensive software suite preinstalled:

- **Languages**: Node.js, Python, Ruby, Java, Go, .NET, PHP, Rust
- **Package managers**: npm, pip, Bundler, Maven, Gradle, Cargo
- **Tools**: Docker, kubectl, Helm, Terraform, AWS CLI, Azure CLI, GCloud CLI
- **Shells**: bash, PowerShell, sh

Full software manifests are published at [github.com/actions/runner-images](https://github.com/actions/runner-images).

## Self-Hosted Runners

Self-hosted runners let you use your own hardware or cloud VMs. They are useful for:
- Private network access (databases, internal APIs)
- Specialized hardware (GPUs, ARM, specific OS versions)
- Cost savings on very high-volume workloads
- Compliance requirements

```yaml
jobs:
  build:
    runs-on: self-hosted        # or use labels
    # runs-on: [self-hosted, linux, gpu]
```

:::caution[Self-hosted runner security]
Never use self-hosted runners on **public** repositories — malicious PRs can execute arbitrary code on your runner. Self-hosted runners are only recommended for private repositories or with strict fork controls enabled.
:::

Self-hosted runners are registered at the repository, organization, or enterprise level. The runner application is a lightweight agent that polls the GitHub Actions API for queued jobs.

## Billing

### Free Tier

| Repository type | Free minutes/month | Storage |
|---|---|---|
| Public repositories | **Unlimited** | Unlimited |
| Private repositories | **2,000 minutes** | 500 MB |

GitHub Free (personal) and GitHub Team include these allocations. GitHub Enterprise includes 50,000 minutes/month.

### Minute Multipliers

Minutes are counted differently per OS:

| OS | Multiplier |
|---|---|
| Linux | 1× |
| Windows | 2× |
| macOS | 10× |

A 10-minute macOS job consumes **100 billed minutes**.

### Cost per Minute (over free tier)

| OS | Cost |
|---|---|
| Linux | $0.008/min |
| Windows | $0.016/min |
| macOS | $0.08/min |

:::tip[Optimize costs]
Run as many steps as possible on Linux. Only switch to Windows or macOS when platform-specific behavior is required. Use caching (`actions/cache`) to reduce build times.
:::

## The GITHUB_TOKEN

Every workflow run receives an automatically generated `GITHUB_TOKEN` secret. This token authenticates to the GitHub API with scoped permissions:

- **Default permissions**: Read access to repository contents; write access only to checks and status
- **Configurable permissions**: Extend using the `permissions:` block in your workflow
- **Scope**: Automatically expires when the job finishes
- **Never share**: Do not pass `GITHUB_TOKEN` to external services

```yaml
jobs:
  release:
    runs-on: ubuntu-latest
    permissions:
      contents: write       # Allow creating releases
      pull-requests: write  # Allow commenting on PRs
    steps:
      - uses: actions/checkout@v4
        with:
          token: ${{ secrets.GITHUB_TOKEN }}
```

The token's permissions can be set at the workflow level (all jobs) or the job level (override per job). Setting `permissions:` at the workflow level removes all defaults, so you must explicitly grant what each job needs — this is the principle of least privilege.

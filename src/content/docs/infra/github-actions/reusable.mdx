## Why Reuse?

As your workflow library grows, you'll find yourself copying the same deploy steps, test setups, or notification logic across dozens of workflows. GitHub Actions offers two primary mechanisms for DRY workflow automation:

| Mechanism | Defined in | Called with | Cross-repo | Secrets |
|---|---|---|---|---|
| **Reusable workflow** | `.github/workflows/*.yml` | `uses: org/repo/.github/workflows/name.yml@ref` | Yes | Explicit or inherit |
| **Composite action** | `action.yml` in any directory | `uses: org/repo/path@ref` | Yes | Via `inputs` |
| **JavaScript action** | `action.yml` + `.js` file | `uses: org/repo@ref` | Yes | Via `inputs` |
| **Docker action** | `action.yml` + `Dockerfile` | `uses: org/repo@ref` | Yes | Via `inputs` |

## Reusable Workflows

### Creating a Reusable Workflow

A reusable workflow is a normal workflow file that uses `workflow_call` as its trigger:

```yaml
# .github/workflows/deploy.yml
name: Deploy (reusable)

on:
  workflow_call:
    inputs:
      environment:
        description: 'Target environment'
        type: string
        required: true
      image-tag:
        description: 'Docker image tag to deploy'
        type: string
        required: true
      dry-run:
        description: 'Perform a dry run'
        type: boolean
        required: false
        default: false
    secrets:
      DEPLOY_KEY:
        required: true
      AWS_ROLE_ARN:
        required: false
    outputs:
      deployed-url:
        description: 'URL of the deployed service'
        value: ${{ jobs.deploy.outputs.url }}

jobs:
  deploy:
    name: Deploy to ${{ inputs.environment }}
    runs-on: ubuntu-latest
    environment: ${{ inputs.environment }}
    permissions:
      id-token: write
      contents: read
    outputs:
      url: ${{ steps.get-url.outputs.url }}
    steps:
      - uses: actions/checkout@v4

      - name: Authenticate to AWS
        uses: aws-actions/configure-aws-credentials@v4
        with:
          role-to-assume: ${{ secrets.AWS_ROLE_ARN }}
          aws-region: us-east-1

      - name: Deploy
        if: ${{ !inputs.dry-run }}
        id: deploy-step
        run: |
          ./scripts/deploy.sh \
            --env ${{ inputs.environment }} \
            --tag ${{ inputs.image-tag }} \
            --key ${{ secrets.DEPLOY_KEY }}

      - name: Get URL
        id: get-url
        run: |
          URL=$(./scripts/get-deploy-url.sh ${{ inputs.environment }})
          echo "url=$URL" >> $GITHUB_OUTPUT
```

### Calling a Reusable Workflow

```yaml
# .github/workflows/ci-cd.yml
name: CI/CD

on:
  push:
    branches: [main]

jobs:
  build:
    runs-on: ubuntu-latest
    outputs:
      image-tag: ${{ steps.tag.outputs.tag }}
    steps:
      - uses: actions/checkout@v4
      - id: tag
        run: echo "tag=$(git rev-parse --short HEAD)" >> $GITHUB_OUTPUT
      - run: docker build -t myapp:$(git rev-parse --short HEAD) .

  deploy-staging:
    needs: build
    uses: myorg/devops/.github/workflows/deploy.yml@main
    with:
      environment: staging
      image-tag: ${{ needs.build.outputs.image-tag }}
    secrets:
      DEPLOY_KEY: ${{ secrets.STAGING_DEPLOY_KEY }}
      AWS_ROLE_ARN: ${{ secrets.AWS_ROLE_STAGING }}

  deploy-production:
    needs: deploy-staging
    uses: myorg/devops/.github/workflows/deploy.yml@v2
    with:
      environment: production
      image-tag: ${{ needs.build.outputs.image-tag }}
    secrets: inherit    # pass ALL caller secrets to the called workflow
```

:::tip[Version pinning]
Always reference reusable workflows at a specific version (`@v2`, `@main`, `@<sha>`). Using `@main` means unexpected changes in the shared workflow can break your callers. For production-critical workflows, pin to a SHA.
:::

### Reusable Workflow Limitations

- A reusable workflow can call other reusable workflows (nesting up to 4 levels deep)
- The `env` context is not passed to called workflows
- Called workflows run with the permissions of the caller
- Matrix jobs can call reusable workflows

## Composite Actions

Composite actions package a sequence of steps (including `uses:` and `run:`) into a reusable unit callable with `uses:`.

### Directory Structure

```bash
.github/actions/setup-environment/
└── action.yml

# Or in a dedicated repo:
my-org/my-actions/
├── setup/
│   └── action.yml
├── notify/
│   └── action.yml
└── README.md
```

### Creating a Composite Action

```yaml
# .github/actions/setup-environment/action.yml
name: 'Setup Environment'
description: 'Checkout, setup Node.js, and install dependencies'
author: 'MyOrg'

branding:
  icon: 'package'
  color: 'blue'

inputs:
  node-version:
    description: 'Node.js version to use'
    required: false
    default: '20'
  install-command:
    description: 'Command to install dependencies'
    required: false
    default: 'npm ci'
  working-directory:
    description: 'Working directory'
    required: false
    default: '.'

outputs:
  cache-hit:
    description: 'Whether dependencies were restored from cache'
    value: ${{ steps.cache.outputs.cache-hit }}

runs:
  using: 'composite'
  steps:
    - name: Checkout
      uses: actions/checkout@v4
      with:
        fetch-depth: 0

    - name: Setup Node.js
      uses: actions/setup-node@v4
      with:
        node-version: ${{ inputs.node-version }}
        cache: 'npm'

    - name: Cache dependencies
      id: cache
      uses: actions/cache@v4
      with:
        path: ${{ inputs.working-directory }}/node_modules
        key: node-modules-${{ runner.os }}-${{ hashFiles('**/package-lock.json') }}

    - name: Install dependencies
      if: steps.cache.outputs.cache-hit != 'true'
      shell: bash
      working-directory: ${{ inputs.working-directory }}
      run: ${{ inputs.install-command }}
```

### Using a Composite Action

```yaml
jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - name: Setup environment
        uses: ./.github/actions/setup-environment    # local action
        with:
          node-version: '20'
          working-directory: ./frontend

      # Or from another repository:
      - uses: myorg/my-actions/setup@v1
        with:
          node-version: '20'

      - run: npm test
```

## JavaScript Actions

For more complex logic, write actions in JavaScript/TypeScript using Node.js:

### Directory Structure

```bash
my-action/
├── action.yml
├── dist/
│   └── index.js       # bundled with @vercel/ncc or esbuild
├── src/
│   └── index.ts
├── package.json
└── README.md
```

### `action.yml` for JavaScript Actions

```yaml
name: 'Post Deploy Comment'
description: 'Posts a deployment URL comment on the PR'
author: 'MyOrg'

inputs:
  token:
    description: 'GitHub token'
    required: true
  deploy-url:
    description: 'Deployment URL'
    required: true
  pr-number:
    description: 'PR number'
    required: false

outputs:
  comment-id:
    description: 'ID of the created/updated comment'

runs:
  using: 'node20'
  main: 'dist/index.js'
  post: 'dist/cleanup.js'    # optional cleanup step
```

### JavaScript Action Implementation

```javascript
// src/index.ts
import * as core from '@actions/core';
import * as github from '@actions/github';

async function run(): Promise<void> {
  try {
    const token = core.getInput('token', { required: true });
    const deployUrl = core.getInput('deploy-url', { required: true });

    const octokit = github.getOctokit(token);
    const context = github.context;

    // Get PR number from input or context
    const prNumber = parseInt(core.getInput('pr-number')) ||
      context.payload.pull_request?.number;

    if (!prNumber) {
      core.info('No PR number found, skipping comment');
      return;
    }

    const body = `## 🚀 Deployment Preview\n\n**URL:** ${deployUrl}\n**Commit:** \`${context.sha.slice(0, 7)}\``;

    // Check for existing comment
    const { data: comments } = await octokit.rest.issues.listComments({
      owner: context.repo.owner,
      repo: context.repo.repo,
      issue_number: prNumber,
    });

    const existing = comments.find(c =>
      c.user?.login === 'github-actions[bot]' &&
      c.body?.includes('## 🚀 Deployment Preview')
    );

    let commentId: number;

    if (existing) {
      const { data } = await octokit.rest.issues.updateComment({
        owner: context.repo.owner,
        repo: context.repo.repo,
        comment_id: existing.id,
        body,
      });
      commentId = data.id;
    } else {
      const { data } = await octokit.rest.issues.createComment({
        owner: context.repo.owner,
        repo: context.repo.repo,
        issue_number: prNumber,
        body,
      });
      commentId = data.id;
    }

    core.setOutput('comment-id', String(commentId));
    core.info(`Comment ${existing ? 'updated' : 'created'}: ${commentId}`);

  } catch (error) {
    if (error instanceof Error) {
      core.setFailed(error.message);
    }
  }
}

run();
```

### Building the Action

```bash
# Install dependencies
npm install @actions/core @actions/github

# Bundle with ncc (no node_modules needed at runtime)
npx @vercel/ncc build src/index.ts -o dist --source-map

# Commit dist/
git add dist/ && git commit -m "chore: build action"
```

## Docker Container Actions

Run actions in a custom Docker environment:

```yaml
# action.yml
name: 'Terraform Plan'
description: 'Run terraform plan with output'

inputs:
  working-directory:
    required: false
    default: '.'

outputs:
  plan-output:
    description: 'Terraform plan output'

runs:
  using: 'docker'
  image: 'Dockerfile'
  args:
    - ${{ inputs.working-directory }}
```

```dockerfile
# Dockerfile
FROM hashicorp/terraform:1.6

COPY entrypoint.sh /entrypoint.sh
RUN chmod +x /entrypoint.sh

ENTRYPOINT ["/entrypoint.sh"]
```

## Action Versioning

When publishing actions, use semantic version tags:

```bash
# After release
git tag -a v1.0.0 -m "Release v1.0.0"
git tag -f v1                           # move major version tag
git push origin v1.0.0
git push origin v1 --force
```

Users then reference `uses: myorg/my-action@v1` and automatically get patch/minor updates.

:::tip[Action versioning best practice]
Always provide a major version tag (`v1`, `v2`) that you move forward with each non-breaking release. Pin to a SHA in security-sensitive workflows: `uses: myorg/my-action@abc1234`.
:::

## When to Use Reusable Workflow vs Composite Action

| Consideration | Reusable Workflow | Composite Action |
|---|---|---|
| Multiple jobs | ✅ Yes | ❌ No (single job) |
| Environments & approvals | ✅ Yes | ❌ No |
| Matrix strategy | ✅ Yes | ❌ No |
| Secrets forwarding | Explicit or inherit | Via inputs |
| Steps only | ❌ No | ✅ Yes |
| Can call other actions | ✅ Yes | ✅ Yes |
| Marketplace publishable | ❌ No | ✅ Yes |
| Best for | Multi-job pipelines | Step bundles |

## Publishing to the Marketplace

<Steps>
1. Create a **public** repository for your action
2. Add `action.yml` at the repository root with `name`, `description`, `branding`, `inputs`, `outputs`, `runs`
3. Include a `README.md` with usage examples
4. Navigate to the repository on GitHub — click **Draft a release**
5. Check **Publish this Action to the GitHub Marketplace**
6. Fill in the category and release notes
7. Publish the release
</Steps>

```yaml
# Required action.yml branding for Marketplace
branding:
  icon: 'zap'           # feather icon name
  color: 'yellow'       # gray | blue | green | orange | red | purple | yellow | white
```

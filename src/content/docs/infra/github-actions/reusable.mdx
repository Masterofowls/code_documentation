GitHub Actions provides multiple reuse mechanisms. Understanding when to use each is key to maintainable automation.

| Mechanism | Scope | Language | Inputs | Outputs | Secrets |
|---|---|---|---|---|---|
| **Reusable workflow** | Entire workflow | YAML | Yes | Yes | inherit or explicit |
| **Composite action** | Steps within a job | YAML + shell | Yes | Yes | No direct access |
| **JavaScript action** | Single step | Node.js | Yes | Yes | Via env |
| **Docker action** | Single step | Any | Yes | Yes | Via env |

## Reusable Workflows

A reusable workflow is a complete workflow file triggered by `workflow_call`. Other workflows can invoke it as a job.

### Creating a Reusable Workflow

```yaml
# .github/workflows/deploy.yml (reusable workflow)
name: Deploy

on:
  workflow_call:
    inputs:
      environment:
        description: 'Target environment'
        required: true
        type: string
      image-tag:
        required: true
        type: string
      dry-run:
        required: false
        type: boolean
        default: false

    secrets:
      KUBE_CONFIG:
        required: true
      REGISTRY_TOKEN:
        required: false

    outputs:
      deployed-url:
        description: 'URL of the deployed application'
        value: ${{ jobs.deploy.outputs.app-url }}

jobs:
  deploy:
    runs-on: ubuntu-latest
    environment: ${{ inputs.environment }}
    outputs:
      app-url: ${{ steps.deploy.outputs.url }}

    steps:
      - uses: actions/checkout@v4

      - name: Set up kubectl
        uses: azure/setup-kubectl@v3

      - name: Configure kubeconfig
        run: |
          echo "${{ secrets.KUBE_CONFIG }}" | base64 -d > kubeconfig.yaml
          echo "KUBECONFIG=$PWD/kubeconfig.yaml" >> $GITHUB_ENV

      - name: Deploy
        id: deploy
        run: |
          if [ "${{ inputs.dry-run }}" = "true" ]; then
            kubectl diff -f k8s/${{ inputs.environment }}/
          else
            kubectl set image deployment/app \
              app=${{ inputs.image-tag }}
            APP_URL=$(kubectl get svc app -o jsonpath='{.status.loadBalancer.ingress[0].hostname}')
            echo "url=https://$APP_URL" >> $GITHUB_OUTPUT
          fi
```

### Calling a Reusable Workflow

```yaml
# .github/workflows/ci-cd.yml (caller workflow)
name: CI/CD

on:
  push:
    branches: [main]

jobs:
  build:
    runs-on: ubuntu-latest
    outputs:
      image-tag: ${{ steps.build.outputs.tag }}
    steps:
      - uses: actions/checkout@v4
      - id: build
        run: |
          TAG="sha-${GITHUB_SHA::8}"
          docker build -t myapp:$TAG .
          echo "tag=myapp:$TAG" >> $GITHUB_OUTPUT

  deploy-staging:
    needs: build
    uses: ./.github/workflows/deploy.yml    # Same repo
    with:
      environment: staging
      image-tag: ${{ needs.build.outputs.image-tag }}
    secrets:
      KUBE_CONFIG: ${{ secrets.STAGING_KUBE_CONFIG }}

  deploy-production:
    needs: [build, deploy-staging]
    uses: myorg/shared-workflows/.github/workflows/deploy.yml@v2  # Another repo
    with:
      environment: production
      image-tag: ${{ needs.build.outputs.image-tag }}
    secrets: inherit    # Pass all caller's secrets to the called workflow
```

:::tip[`secrets: inherit` vs explicit]
Use `secrets: inherit` when the reusable workflow needs many of the same secrets as the caller. Use explicit secret mapping when you want to clearly document which secrets are required and rename them.
:::

### Reusable Workflow Constraints

- A caller can use at most **3 levels** of nested reusable workflows
- Reusable workflows cannot call other reusable workflows in the same call chain
- A `workflow_call` workflow cannot be triggered by events other than being called
- Environment variables set by the caller are not available in the called workflow unless explicitly passed as inputs

## Composite Actions

A composite action runs a series of steps (using `run` and `uses`) as if they were inline in the caller's job. Unlike reusable workflows, composite actions run in the **caller's job context** — they share the same runner, environment, and working directory.

### Creating a Composite Action

```yaml
# .github/actions/setup-app/action.yml
name: 'Setup Application'
description: 'Checkout, install Node, install deps, and configure environment'

inputs:
  node-version:
    description: 'Node.js version to use'
    required: false
    default: '20'
  install-playwright:
    description: 'Whether to install Playwright browsers'
    required: false
    default: 'false'
  working-directory:
    description: 'Working directory'
    required: false
    default: '.'

outputs:
  node-version-used:
    description: 'The actual Node.js version installed'
    value: ${{ steps.node-setup.outputs.node-version }}

runs:
  using: 'composite'
  steps:
    - name: Checkout
      uses: actions/checkout@v4

    - name: Set up Node.js
      id: node-setup
      uses: actions/setup-node@v4
      with:
        node-version: ${{ inputs.node-version }}
        cache: 'npm'
        cache-dependency-path: ${{ inputs.working-directory }}/package-lock.json

    - name: Install dependencies
      shell: bash
      working-directory: ${{ inputs.working-directory }}
      run: npm ci

    - name: Install Playwright browsers
      if: inputs.install-playwright == 'true'
      shell: bash
      working-directory: ${{ inputs.working-directory }}
      run: npx playwright install --with-deps
```

:::caution[Shell required in composite actions]
Every `run` step in a composite action **must** specify `shell:` explicitly. Unlike regular workflows, there is no inherited default shell.
:::

### Using a Composite Action

```yaml
# .github/workflows/ci.yml
jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: ./.github/actions/setup-app    # Local action
        with:
          node-version: '20'
          install-playwright: 'true'

      - run: npm test

  lint:
    runs-on: ubuntu-latest
    steps:
      - uses: ./.github/actions/setup-app   # Reuse in another job
        with:
          node-version: '20'

      - run: npm run lint
```

## JavaScript Actions

JavaScript actions run Node.js code directly on the runner — no Docker container, fast startup.

### `action.yml` for a JavaScript Action

```yaml
# .github/actions/my-js-action/action.yml
name: 'My JavaScript Action'
description: 'Does something useful with the GitHub API'

inputs:
  message:
    description: 'Message to post'
    required: true
  pr-number:
    description: 'Pull request number'
    required: false

outputs:
  comment-id:
    description: 'ID of the created comment'

runs:
  using: 'node20'   # node16, node20
  main: 'dist/index.js'
  post: 'dist/cleanup.js'   # Optional: runs after job finishes
  post-if: always()
```

### JavaScript Action Code

```javascript
// src/index.js
const core = require('@actions/core');
const github = require('@actions/github');

async function run() {
  try {
    const message = core.getInput('message', { required: true });
    const prNumber = parseInt(core.getInput('pr-number'));

    const token = process.env.GITHUB_TOKEN;
    const octokit = github.getOctokit(token);
    const { context } = github;

    if (prNumber) {
      const { data: comment } = await octokit.rest.issues.createComment({
        owner: context.repo.owner,
        repo: context.repo.repo,
        issue_number: prNumber,
        body: message,
      });

      core.setOutput('comment-id', comment.id);
      core.info(`Created comment #${comment.id}`);
    }

    // Set a failed status
    // core.setFailed('Something went wrong');

    // Mask a sensitive value
    // core.setSecret(sensitiveValue);

    // Write to step summary
    await core.summary
      .addHeading('Action Result')
      .addRaw(`Posted: ${message}`)
      .write();

  } catch (error) {
    core.setFailed(error.message);
  }
}

run();
```

### Building and Distributing

```bash
# package.json scripts
npm run build   # tsc + esbuild/ncc to bundle into dist/index.js

# Use @vercel/ncc to bundle all dependencies into one file
npx @vercel/ncc build src/index.js -o dist
```

:::caution[Always commit `dist/`]
JavaScript actions must include the compiled `dist/` directory in the repository. Unlike npm packages, GitHub does not run `npm install` when using an action — the compiled output must be committed.
:::

## Docker Container Actions

Docker actions run inside a container — ideal for actions requiring specific system tools or non-Node runtimes.

```yaml
# action.yml
name: 'Python Analysis Action'
description: 'Runs Python analysis tools'

inputs:
  python-version:
    required: false
    default: '3.12'

runs:
  using: 'docker'
  image: 'Dockerfile'         # Build from local Dockerfile
  # image: 'docker://python:3.12-slim'   # Or use a prebuilt image
  args:
    - ${{ inputs.python-version }}
```

```dockerfile
# Dockerfile
FROM python:3.12-slim
COPY entrypoint.sh /entrypoint.sh
RUN chmod +x /entrypoint.sh
ENTRYPOINT ["/entrypoint.sh"]
```

## Action Inputs and Outputs Reference

```yaml
inputs:
  my-input:
    description: 'Human-readable description shown in docs'
    required: true              # or false
    default: 'default-value'    # Only valid when required: false
    deprecationMessage: 'Use new-input instead'

outputs:
  my-output:
    description: 'Description of what this output contains'
    # For composite: value: ${{ steps.step-id.outputs.something }}
    # For JS: set with core.setOutput('my-output', value)
```

## Action Versioning

When publishing an action to the Marketplace, use semantic versioning with a major-version tag:

```bash
# Tag a release
git tag -a v2.1.0 -m "Release v2.1.0"
git push origin v2.1.0

# Update the floating major version tag
git tag -fa v2 -m "Update v2 tag to v2.1.0"
git push origin v2 --force
```

Users can then pin to:
- `uses: owner/action@v2` — latest v2.x (recommended for stability)
- `uses: owner/action@v2.1.0` — exact version
- `uses: owner/action@main` — latest (risky for production)
- `uses: owner/action@a1b2c3d` — specific commit SHA (most secure)

:::tip[Pin to commit SHA for security]
For actions from untrusted or third-party repositories, pin to a full commit SHA. A version tag (like `@v2`) can be moved to point to different code. A commit SHA cannot.
:::

## Publishing to the GitHub Marketplace

<Steps>
1. Create your action repository with an `action.yml` in the root
2. Add a `README.md` describing inputs, outputs, and usage examples
3. Create a release with a semantic version tag
4. On the GitHub release page, check **Publish this Action to the GitHub Marketplace**
5. Select a category and add a description
6. Click **Release**
</Steps>


## Reusable Workflow vs Composite Action

| Question | Reusable Workflow | Composite Action |
|---|---|---|
| Needs its own job? | ✅ Yes | ❌ No (runs in caller's job) |
| Can use `services:`? | ✅ Yes | ❌ No |
| Can use `matrix:`? | ✅ Yes | ❌ No |
| Runs in parallel? | ✅ Yes (separate jobs) | ❌ No (sequential steps) |
| Shares runner filesystem with caller? | ❌ No | ✅ Yes |
| Can access caller secrets? | Via `secrets: inherit` | Via caller's `env:` |
| Cross-repo use | ✅ Yes | ✅ Yes |
| Versioned independently | ✅ Yes | ✅ Yes |
| When to use | Complex multi-job flows, deployment pipelines | Setup steps, linting, simple utilities |

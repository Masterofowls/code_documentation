## Reusable Workflows vs Composite Actions

Before building either, understand when to use which:

| Feature              | Reusable Workflow                      | Composite Action                     |
| -------------------- | -------------------------------------- | ------------------------------------ |
| **Unit**             | Entire workflow (multiple jobs)        | Single step (`uses:`)                |
| **Uses jobs?**       | Yes — can have multiple jobs           | No — sequence of steps               |
| **Runner selection** | Caller or callee can define            | Inherited from caller job            |
| **Inputs**           | `inputs:` with type, required, default | `inputs:` in action.yml              |
| **Secrets**          | `secrets: inherit` or explicit         | Not supported directly               |
| **Nesting**          | Up to 4 levels deep                    | No depth limit                       |
| **Storage**          | `.github/workflows/*.yml` in any repo  | `action.yml` in root or subdirectory |
| **Best for**         | Full CI/CD pipeline templates          | Reusable step sequences              |

## Reusable Workflows

### Creating a Reusable Workflow

Store the workflow in any GitHub repository (public, private, or internal). The `on: workflow_call:` trigger makes it callable.

```yaml
# .github/workflows/deploy-service.yml
name: Deploy Service (Reusable)

on:
  workflow_call:
    inputs:
      environment:
        description: 'Target environment'
        required: true
        type: string
      image-tag:
        description: 'Docker image tag to deploy'
        required: true
        type: string
      dry-run:
        description: 'Run without applying changes'
        required: false
        type: boolean
        default: false
      replicas:
        description: 'Number of replicas'
        required: false
        type: number
        default: 2
    secrets:
      DEPLOY_TOKEN:
        required: true
      REGISTRY_PASSWORD:
        required: false
    outputs:
      deploy-url:
        description: 'URL of the deployed application'
        value: ${{ jobs.deploy.outputs.url }}
      deploy-status:
        description: 'success or failure'
        value: ${{ jobs.deploy.outputs.status }}

permissions:
  id-token: write
  contents: read

jobs:
  validate:
    runs-on: ubuntu-latest
    steps:
      - name: Validate inputs
        run: |
          if [[ "${{ inputs.environment }}" != "staging" && "${{ inputs.environment }}" != "production" ]]; then
            echo "Invalid environment: ${{ inputs.environment }}"
            exit 1
          fi

  deploy:
    runs-on: ubuntu-latest
    needs: validate
    environment: ${{ inputs.environment }}
    outputs:
      url: ${{ steps.deploy.outputs.url }}
      status: ${{ steps.deploy.outputs.status }}
    steps:
      - uses: actions/checkout@v4

      - name: Deploy
        id: deploy
        run: |
          if [[ "${{ inputs.dry-run }}" == "true" ]]; then
            echo "DRY RUN: would deploy ${{ inputs.image-tag }} to ${{ inputs.environment }}"
            echo "url=https://dry-run.example.com" >> $GITHUB_OUTPUT
          else
            ./deploy.sh \
              --env "${{ inputs.environment }}" \
              --image "${{ inputs.image-tag }}" \
              --replicas "${{ inputs.replicas }}"
            echo "url=https://${{ inputs.environment }}.example.com" >> $GITHUB_OUTPUT
          fi
          echo "status=success" >> $GITHUB_OUTPUT
        env:
          DEPLOY_TOKEN: ${{ secrets.DEPLOY_TOKEN }}
```

### Calling a Reusable Workflow

```yaml
# .github/workflows/release.yml
name: Release Pipeline

on:
  push:
    tags: ['v*']

jobs:
  build:
    runs-on: ubuntu-latest
    outputs:
      image-tag: ${{ steps.meta.outputs.version }}
    steps:
      - uses: actions/checkout@v4
      - name: Build and tag
        id: meta
        run: |
          echo "version=${GITHUB_REF_NAME}" >> $GITHUB_OUTPUT
          docker build -t myapp:${GITHUB_REF_NAME} .

  deploy-staging:
    needs: build
    uses: my-org/shared-workflows/.github/workflows/deploy-service.yml@main
    with:
      environment: staging
      image-tag: ${{ needs.build.outputs.image-tag }}
      replicas: 1
    secrets:
      DEPLOY_TOKEN: ${{ secrets.STAGING_DEPLOY_TOKEN }}
      # Or inherit all secrets from the calling workflow:
      # secrets: inherit

  deploy-production:
    needs: deploy-staging
    uses: my-org/shared-workflows/.github/workflows/deploy-service.yml@main
    with:
      environment: production
      image-tag: ${{ needs.build.outputs.image-tag }}
      replicas: 3
    secrets: inherit

  notify:
    needs: [deploy-staging, deploy-production]
    runs-on: ubuntu-latest
    if: always()
    steps:
      - run: |
          echo "Staging URL: ${{ needs.deploy-staging.outputs.deploy-url }}"
          echo "Production URL: ${{ needs.deploy-production.outputs.deploy-url }}"
```

:::tip[Pin Reusable Workflow Versions]
Always pin reusable workflow references to a specific SHA or tag for production workflows: `uses: my-org/shared/.github/workflows/ci.yml@abc1234`. Using `@main` means changes to the shared workflow immediately affect all callers.
:::

## Composite Actions

Composite actions bundle multiple steps into a single reusable action callable with `uses:`.

### Creating a Composite Action

```yaml
# actions/setup-and-cache/action.yml
name: 'Setup Node.js with Cache'
description: 'Sets up Node.js, installs deps with full caching strategy'

inputs:
  node-version:
    description: 'Node.js version to use'
    required: false
    default: '20'
  working-directory:
    description: 'Directory containing package.json'
    required: false
    default: '.'
  install-command:
    description: 'Install command to run'
    required: false
    default: 'npm ci'

outputs:
  cache-hit:
    description: 'Whether the cache was hit'
    value: ${{ steps.cache.outputs.cache-hit }}
  node-version:
    description: 'The Node.js version that was installed'
    value: ${{ steps.setup.outputs.node-version }}

runs:
  using: composite
  steps:
    - name: Set up Node.js
      id: setup
      uses: actions/setup-node@v4
      with:
        node-version: ${{ inputs.node-version }}
        cache: npm
        cache-dependency-path: ${{ inputs.working-directory }}/package-lock.json

    - name: Cache node_modules
      id: cache
      uses: actions/cache@v4
      with:
        path: ${{ inputs.working-directory }}/node_modules
        key: ${{ runner.os }}-node${{ inputs.node-version }}-${{ hashFiles(format('{0}/package-lock.json', inputs.working-directory)) }}
        restore-keys: |
          ${{ runner.os }}-node${{ inputs.node-version }}-

    - name: Install dependencies
      if: steps.cache.outputs.cache-hit != 'true'
      shell: bash
      working-directory: ${{ inputs.working-directory }}
      run: ${{ inputs.install-command }}
```

:::caution[shell: is Required in Composite Actions]
Every `run:` step inside a composite action **must** specify `shell:`. Unlike regular workflow steps, there is no default shell in composite actions.
:::

### Using a Composite Action

```yaml
# Reference from the same repo
- uses: ./.github/actions/setup-and-cache
  with:
    node-version: '20'

# Reference from another repo at a tag
- uses: my-org/shared-actions/setup-and-cache@v2
  with:
    node-version: '18'
    working-directory: packages/api

# Use outputs
- id: setup
  uses: ./.github/actions/setup-and-cache
- run: echo "Cache hit: ${{ steps.setup.outputs.cache-hit }}"
```

### Composite Action with `$GITHUB_ACTION_PATH`

```yaml
runs:
  using: composite
  steps:
    - name: Run bundled script
      shell: bash
      run: |
        # $GITHUB_ACTION_PATH points to the action's directory
        bash $GITHUB_ACTION_PATH/scripts/setup.sh
        python $GITHUB_ACTION_PATH/scripts/validate.py
```

## JavaScript Actions

JavaScript actions run directly on the runner without a container, making them faster to start.

### `action.yml` for JS Action

```yaml
# action.yml
name: 'My JavaScript Action'
description: 'Does something useful with Node.js'

inputs:
  api-url:
    description: 'API endpoint URL'
    required: true
  timeout:
    description: 'Request timeout in seconds'
    required: false
    default: '30'
  debug:
    description: 'Enable debug logging'
    required: false
    default: 'false'

outputs:
  response-body:
    description: 'The API response body'
  status-code:
    description: 'HTTP status code'

runs:
  using: node20
  main: dist/index.js # compiled/bundled entry point
  pre: dist/pre.js # optional: runs before main
  post: dist/post.js # optional: runs after main (for cleanup)
```

```bash
# Install required packages
```

### Building and Publishing

```yaml
# .github/workflows/release-action.yml
name: Release Action

on:
  push:
    tags: ['v*']

jobs:
  release:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - uses: actions/setup-node@v4
        with:
          node-version: 20

      - run: npm ci && npm run build && npm run package

      - name: Create release
        uses: softprops/action-gh-release@v2
        with:
          files: dist/
          generate_release_notes: true
```

## Docker Container Actions

Docker actions run in a container, giving full control over the environment:

```yaml
# action.yml for Docker action
name: 'Docker Action'
description: 'Runs in a custom Docker container'

inputs:
  input-file:
    description: 'Path to input file'
    required: true

outputs:
  result:
    description: 'Processing result'

runs:
  using: docker
  image: Dockerfile # build from local Dockerfile
  # OR:
  # image: docker://alpine:3.18  # use a pre-built image
  args:
    - ${{ inputs.input-file }}
  env:
    DEBUG: ${{ inputs.debug }}
```

## Action Versioning

### Semantic Versioning Tags

```yaml
# Specific patch version (most locked-down)
uses: actions/checkout@v4.1.2

# Major version tag (recommended — gets patches automatically)
uses: actions/checkout@v4

# Full commit SHA (most secure — immutable)
uses: actions/checkout@11bd71901bbe5b1630ceea73d27597364c9af683

# Branch (avoid in production — mutable)
uses: actions/checkout@main
```

### Maintaining Version Tags

```bash
# When releasing a new version, move the major tag forward
git tag -fa v4 -m "Update v4 tag to v4.2.0"
git push origin v4 --force
git tag v4.2.0
git push origin v4.2.0
```

## When to Use Which Pattern

```
Need to reuse an entire CI/CD pipeline across repos?
→ Reusable Workflow (workflow_call)

Need to bundle 3-4 setup steps used in every job?
→ Composite Action

Need to call a GitHub API or do complex logic?
→ JavaScript Action (@actions/core + @actions/github)

Need a specific runtime environment (Go, Python, Rust binary)?
→ Docker Container Action

Need to run a one-off script shared across steps?
→ $GITHUB_ACTION_PATH script in a composite action
```

## Shared Action Repository Pattern

```
my-org/
└── github-actions/                  # shared actions repo
    ├── .github/
    │   └── workflows/
    │       ├── ci.yml               # reusable: test any Node.js service
    │       ├── docker-build.yml     # reusable: build & push Docker images
    │       └── deploy-k8s.yml       # reusable: deploy to Kubernetes
    ├── setup-node-cache/
    │   └── action.yml               # composite: setup + cache
    ├── notify-slack/
    │   ├── action.yml               # JS: send Slack notification
    │   ├── src/index.ts
    │   └── dist/index.js            # pre-compiled bundle
    └── README.md
```

Caller repository:

```yaml
# Use reusable workflow from shared repo
jobs:
  test:
    uses: my-org/github-actions/.github/workflows/ci.yml@v2
    with:
      node-version: '20'
    secrets: inherit

  # Use composite action from shared repo
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: my-org/github-actions/setup-node-cache@v2
        with:
          node-version: '20'

      - uses: my-org/github-actions/notify-slack@v2
        if: failure()
        with:
          message: 'Build failed on ${{ github.ref }}'
        env:
          SLACK_WEBHOOK: ${{ secrets.SLACK_WEBHOOK }}
```

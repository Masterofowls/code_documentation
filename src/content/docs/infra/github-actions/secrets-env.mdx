## Secrets

GitHub Actions provides encrypted secret storage at three scopes:

### Repository Secrets

Stored in `Settings → Secrets and variables → Actions`. Available to all workflows in the repository.

```yaml
steps:
  - name: Deploy
    run: ./deploy.sh
    env:
      API_KEY: ${{ secrets.API_KEY }}
      DATABASE_URL: ${{ secrets.DATABASE_URL }}
```

### Environment Secrets

Scoped to a specific GitHub Environment (staging, production, etc.). Only accessible when a job targets that environment. Useful for separating credentials per environment.

```yaml
jobs:
  deploy-prod:
    environment: production    # activates production environment secrets
    runs-on: ubuntu-latest
    steps:
      - run: deploy.sh
        env:
          PROD_API_KEY: ${{ secrets.PROD_API_KEY }}  # env-scoped secret
```

### Organization Secrets

Stored at the organization level and shared across multiple repositories. Administrators control which repositories can access each org secret.

### Dependabot Secrets

Dependabot workflows can't access regular repository secrets for security reasons. Store credentials needed by Dependabot separately at `Settings → Secrets and variables → Dependabot`.

:::caution[Secrets in logs]
GitHub automatically redacts secret values in logs, but never print secrets with `echo`. Multi-step composition can sometimes bypass redaction. Use `::add-mask::` for dynamically created sensitive values.
:::

## GITHUB_TOKEN

Every workflow receives an auto-generated `GITHUB_TOKEN` — a short-lived, scoped token that expires when the workflow ends.

```yaml
jobs:
  publish:
    runs-on: ubuntu-latest
    permissions:
      contents: write       # write files and create releases
      packages: write       # push to GitHub Packages
      pull-requests: write  # comment on PRs
    steps:
      - uses: actions/checkout@v4
        with:
          token: ${{ secrets.GITHUB_TOKEN }}

      - name: Create release
        uses: softprops/action-gh-release@v2
        with:
          token: ${{ secrets.GITHUB_TOKEN }}
          tag_name: v${{ needs.version.outputs.version }}
```

### Default GITHUB_TOKEN Permissions

| Permission | Default (permissive) | Default (restricted) |
|---|---|---|
| `contents` | write | read |
| `metadata` | read | read |
| `packages` | write | read |
| `pull-requests` | write | read |
| `issues` | write | read |
| All others | none | none |

Configure the default in `Settings → Actions → General → Workflow permissions`. Prefer **restricted** with explicit grants per workflow.

:::tip[GITHUB_TOKEN vs PAT]
`GITHUB_TOKEN` cannot trigger other workflows (to prevent loops). If you need a pushed commit to trigger CI, use a Personal Access Token stored as a repository secret.
:::

## OIDC for Cloud Authentication

OpenID Connect (OIDC) lets workflows authenticate to cloud providers **without storing long-lived credentials**. The cloud provider verifies the workflow's identity directly with GitHub.

### How OIDC Works

<Steps>
1. Workflow requests an OIDC token from GitHub (`id-token: write` permission required)
2. GitHub issues a short-lived JWT signed with GitHub's private key
3. Cloud provider verifies the JWT against GitHub's public JWKS endpoint
4. Cloud provider issues short-lived credentials (access tokens, temporary roles)
5. Workflow uses those credentials — they expire when the workflow ends
</Steps>

### AWS via OIDC

```yaml
permissions:
  id-token: write
  contents: read

jobs:
  deploy:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Configure AWS credentials
        uses: aws-actions/configure-aws-credentials@v4
        with:
          role-to-assume: arn:aws:iam::123456789:role/GitHubActionsRole
          role-session-name: GitHubActions-${{ github.run_id }}
          aws-region: us-east-1

      - name: Deploy to S3
        run: aws s3 sync dist/ s3://my-bucket/

      - name: Invalidate CloudFront
        run: |
          aws cloudfront create-invalidation \
            --distribution-id ${{ vars.CF_DISTRIBUTION_ID }} \
            --paths "/*"
```

### Google Cloud via OIDC

```yaml
permissions:
  id-token: write
  contents: read

jobs:
  deploy:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Authenticate to GCP
        uses: google-github-actions/auth@v2
        with:
          workload_identity_provider: projects/123/locations/global/workloadIdentityPools/github/providers/github
          service_account: deploy@my-project.iam.gserviceaccount.com

      - name: Set up Cloud SDK
        uses: google-github-actions/setup-gcloud@v2

      - name: Deploy to Cloud Run
        run: gcloud run deploy myapp --image gcr.io/my-project/myapp:${{ github.sha }}
```

### Azure via OIDC

```yaml
permissions:
  id-token: write
  contents: read

jobs:
  deploy:
    runs-on: ubuntu-latest
    steps:
      - uses: azure/login@v2
        with:
          client-id: ${{ secrets.AZURE_CLIENT_ID }}
          tenant-id: ${{ secrets.AZURE_TENANT_ID }}
          subscription-id: ${{ secrets.AZURE_SUBSCRIPTION_ID }}

      - uses: azure/webapps-deploy@v3
        with:
          app-name: my-azure-app
          package: dist/
```

## Built-In Environment Variables

GitHub populates dozens of variables automatically. Key ones:

### Commit and Repository

| Variable | Description | Example |
|---|---|---|
| `GITHUB_SHA` | Full commit SHA | `a1b2c3d4...` |
| `GITHUB_REF` | Full ref name | `refs/heads/main` |
| `GITHUB_REF_NAME` | Short ref name | `main` |
| `GITHUB_HEAD_REF` | Head branch for PRs | `feature/my-feature` |
| `GITHUB_BASE_REF` | Base branch for PRs | `main` |
| `GITHUB_REPOSITORY` | Owner/repo | `octocat/Hello-World` |
| `GITHUB_REPOSITORY_OWNER` | Owner name | `octocat` |
| `GITHUB_WORKSPACE` | Working directory | `/home/runner/work/repo/repo` |

### Workflow and Run

| Variable | Description |
|---|---|
| `GITHUB_WORKFLOW` | Workflow name |
| `GITHUB_RUN_ID` | Unique run ID (number) |
| `GITHUB_RUN_NUMBER` | Sequential run number for this workflow |
| `GITHUB_RUN_ATTEMPT` | Attempt number (re-runs) |
| `GITHUB_JOB` | Current job ID |
| `GITHUB_ACTION` | Currently executing action ID |
| `GITHUB_ACTOR` | User that triggered the workflow |
| `GITHUB_EVENT_NAME` | Event that triggered the workflow |
| `GITHUB_TRIGGERING_ACTOR` | User that triggered re-run |

### Runner

| Variable | Description | Example |
|---|---|---|
| `RUNNER_OS` | Runner operating system | `Linux`, `Windows`, `macOS` |
| `RUNNER_ARCH` | Runner architecture | `X64`, `ARM64` |
| `RUNNER_NAME` | Runner name | `Hosted Agent` |
| `RUNNER_TEMP` | Temp directory | `/tmp` |
| `RUNNER_TOOL_CACHE` | Tool cache directory | `/opt/hostedtoolcache` |

## Persisting Values Across Steps

### `$GITHUB_ENV` — Environment Variables

```yaml
- name: Set environment variables
  run: |
    echo "VERSION=1.2.3" >> $GITHUB_ENV
    echo "DEPLOY_TIME=$(date -u +%Y-%m-%dT%H:%M:%SZ)" >> $GITHUB_ENV

- name: Use persisted variable
  run: echo "Deploying version $VERSION at $DEPLOY_TIME"
```

Multi-line values use a heredoc delimiter:
```bash
echo "RELEASE_NOTES<<EOF" >> $GITHUB_ENV
echo "## Changes" >> $GITHUB_ENV
echo "- Fixed bug #123" >> $GITHUB_ENV
echo "EOF" >> $GITHUB_ENV
```

### `$GITHUB_PATH` — Add to PATH

```yaml
- name: Install custom tool
  run: |
    curl -sL https://example.com/tool.tar.gz | tar -xz -C /usr/local/bin/
    echo "/usr/local/bin" >> $GITHUB_PATH

- name: Use custom tool
  run: mytool --version    # /usr/local/bin is now on PATH
```

## Context Objects

GitHub Actions provides rich context objects accessible via `${{ context.property }}`.

### `github` Context

```yaml
${{ github.sha }}                  # commit SHA
${{ github.ref }}                  # full ref
${{ github.event_name }}           # triggering event
${{ github.actor }}                # who triggered
${{ github.repository }}           # owner/repo
${{ github.run_id }}               # run ID
${{ github.run_number }}           # run number
${{ github.workflow }}             # workflow name
${{ github.head_ref }}             # PR head branch
${{ github.base_ref }}             # PR base branch
${{ github.event.pull_request.number }}   # PR number
${{ github.event.pull_request.title }}    # PR title
${{ github.event.release.tag_name }}      # release tag
${{ toJSON(github.event) }}        # full event payload as JSON
```

### `env` Context

Access workflow/job/step-level env vars:
```yaml
${{ env.NODE_VERSION }}
${{ env.DATABASE_URL }}
```

### `vars` Context

Access repository, environment, or organization **variables** (non-secret configuration values):
```yaml
${{ vars.APP_URL }}
${{ vars.FEATURE_FLAGS }}
```

### `secrets` Context

```yaml
${{ secrets.GITHUB_TOKEN }}
${{ secrets.MY_API_KEY }}
```

### `needs` Context

Access outputs and status of upstream jobs:
```yaml
${{ needs.build.outputs.version }}
${{ needs.build.result }}         # success | failure | cancelled | skipped
```

### `job` Context

```yaml
${{ job.status }}          # success | failure | cancelled
${{ job.container.id }}    # container ID if running in container
```

### `steps` Context

```yaml
${{ steps.my-step-id.outputs.my-output }}
${{ steps.my-step-id.conclusion }}    # success | failure | skipped | cancelled
${{ steps.my-step-id.outcome }}       # success | failure | skipped | cancelled (pre-continue-on-error)
```

### `runner` Context

```yaml
${{ runner.os }}       # Linux | Windows | macOS
${{ runner.arch }}     # X64 | ARM64
${{ runner.name }}     # runner name
${{ runner.temp }}     # temp dir path
```

### `matrix` Context

```yaml
${{ matrix.node }}     # current matrix value
${{ matrix.os }}
```

## Expression Syntax

Expressions are enclosed in `${{ }}` and evaluated by GitHub Actions.

### Functions

```yaml
# String functions
${{ contains(github.ref, 'refs/heads/main') }}
${{ startsWith(github.ref, 'refs/tags/') }}
${{ endsWith(github.actor, '[bot]') }}
${{ format('Hello {0}!', github.actor) }}

# Array/join
${{ join(matrix.node, ', ') }}

# JSON
${{ toJSON(github.event) }}
${{ fromJSON(needs.setup.outputs.matrix) }}

# Files
${{ hashFiles('**/package-lock.json') }}
${{ hashFiles('src/**', 'package.json') }}

# Status functions (in if conditions)
${{ success() }}       # previous steps succeeded
${{ failure() }}       # any previous step failed
${{ always() }}        # always true
${{ cancelled() }}     # workflow was cancelled
```

### Operators

```yaml
# Comparison
${{ github.ref == 'refs/heads/main' }}
${{ github.run_number > 10 }}
${{ github.actor != 'dependabot[bot]' }}

# Logical
${{ github.ref == 'refs/heads/main' && github.event_name == 'push' }}
${{ github.event_name == 'push' || github.event_name == 'workflow_dispatch' }}
${{ !contains(github.actor, '[bot]') }}

# Ternary-style (using && and || as ternary)
${{ github.event_name == 'push' && 'true' || 'false' }}
```

### Example: Conditional Step with Expression

```yaml
- name: Deploy to production
  if: |
    github.ref == 'refs/heads/main' &&
    github.event_name == 'push' &&
    !contains(github.actor, '[bot]')
  run: ./deploy-prod.sh

- name: Notify on failure
  if: failure() && github.ref == 'refs/heads/main'
  uses: actions/github-script@v7
  with:
    script: |
      github.rest.issues.create({
        owner: context.repo.owner,
        repo: context.repo.repo,
        title: `CI failed on main: run #${{ github.run_number }}`,
        body: `Workflow: ${{ github.workflow }}\nRun: ${{ github.server_url }}/${{ github.repository }}/actions/runs/${{ github.run_id }}`,
        labels: ['ci-failure']
      })
```

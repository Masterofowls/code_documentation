## Secrets Hierarchy

GitHub Actions secrets are encrypted at rest and never exposed in logs. They exist at four levels:

### Repository Secrets

Set in `Settings > Secrets and variables > Actions > Repository secrets`. Available to all workflows in the repository.

```yaml
- run: deploy.sh
  env:
    API_KEY: ${{ secrets.API_KEY }}
    DATABASE_URL: ${{ secrets.DATABASE_URL }}
```

### Environment Secrets

Set per deployment environment (`Settings > Environments`). Only available when a job targets that environment with `environment: production`. Useful for keeping prod credentials separate from staging.

```yaml
jobs:
  deploy:
    environment: production # activates environment secrets
    runs-on: ubuntu-latest
    steps:
      - run: echo ${{ secrets.PROD_API_KEY }} # only available here
```

### Organization Secrets

Managed in `Organization Settings > Secrets`. Can be scoped to `all repositories`, `private repositories`, or a list of selected repos. Useful for shared service credentials (Slack webhooks, monitoring tokens).

### Dependabot Secrets

Secrets available only to Dependabot-triggered workflows. Set in `Settings > Secrets and variables > Dependabot`. Standard `secrets.*` are not accessible in Dependabot runs for security reasons.

```yaml
# Only runs when triggered by Dependabot
- if: github.actor == 'dependabot[bot]'
  run: ./auto-merge-if-safe.sh
  env:
    GH_TOKEN: ${{ secrets.DEPENDABOT_TOKEN }}
```

## GITHUB_TOKEN

Every workflow automatically receives a short-lived `GITHUB_TOKEN` scoped to the current repository. It expires when the workflow ends.

```yaml
permissions:
  contents: write # allow creating commits, tags, releases
  pull-requests: write # allow creating/updating PR comments
  issues: write # allow creating/updating issue comments
  packages: write # push to GitHub Container Registry
  id-token: write # enable OIDC for cloud providers
  deployments: write # create deployment records
  statuses: write # update commit statuses
  checks: write # create check runs

jobs:
  example:
    permissions:
      pull-requests: write # override at job level
    runs-on: ubuntu-latest
    steps:
      - uses: actions/github-script@v7
        with:
          github-token: ${{ secrets.GITHUB_TOKEN }}
          script: |
            github.rest.issues.createComment({
              issue_number: context.issue.number,
              owner: context.repo.owner,
              repo: context.repo.repo,
              body: '✅ Deployment complete'
            })
```

:::tip[GITHUB_TOKEN vs PAT]
`GITHUB_TOKEN` cannot trigger other GitHub Actions workflows (to prevent infinite loops). If you need a workflow to trigger another workflow, use a Personal Access Token (PAT) stored as a repository secret.
:::

## OIDC Cloud Authentication

OpenID Connect (OIDC) lets GitHub Actions authenticate directly to cloud providers **without storing long-lived credentials as secrets**. The workflow requests a short-lived token directly from the cloud provider's identity service.

### How OIDC Works

<Steps>
  1. Workflow requests a JWT from GitHub's OIDC provider 2. Job sends that JWT
  to the cloud provider (AWS, GCP, Azure) 3. Cloud provider validates the JWT
  against GitHub's public keys 4. Cloud provider issues a short-lived access
  token 5. Workflow uses that token to access cloud resources
</Steps>

### AWS OIDC Setup

First, create an IAM Identity Provider and Role in AWS (one-time setup):

```bash
# The provider URL for GitHub Actions
https://token.actions.githubusercontent.com

# Trust policy for the IAM role
{
  "Version": "2012-10-17",
  "Statement": [{
    "Effect": "Allow",
    "Principal": { "Federated": "arn:aws:iam::ACCOUNT:oidc-provider/token.actions.githubusercontent.com" },
    "Action": "sts:AssumeRoleWithWebIdentity",
    "Condition": {
      "StringEquals": {
        "token.actions.githubusercontent.com:aud": "sts.amazonaws.com",
        "token.actions.githubusercontent.com:sub": "repo:MY-ORG/MY-REPO:ref:refs/heads/main"
      }
    }
  }]
}
```

Then use in workflows:

```yaml
permissions:
  id-token: write # required for OIDC
  contents: read

jobs:
  deploy-aws:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Configure AWS credentials via OIDC
        uses: aws-actions/configure-aws-credentials@v4
        with:
          role-to-assume: arn:aws:iam::123456789012:role/github-actions-deploy
          aws-region: us-east-1
          role-session-name: GitHubActions-${{ github.run_id }}

      - name: Deploy to S3
        run: aws s3 sync ./dist s3://my-bucket --delete
```

### GCP OIDC Setup

```yaml
- name: Authenticate to Google Cloud
  uses: google-github-actions/auth@v2
  with:
    workload_identity_provider: 'projects/123/locations/global/workloadIdentityPools/github/providers/github'
    service_account: 'github-actions@my-project.iam.gserviceaccount.com'

- name: Deploy to Cloud Run
  uses: google-github-actions/deploy-cloudrun@v2
  with:
    service: my-service
    region: us-central1
    image: gcr.io/my-project/my-service:${{ github.sha }}
```

### Azure OIDC Setup

```yaml
- name: Azure login via OIDC
  uses: azure/login@v2
  with:
    client-id: ${{ secrets.AZURE_CLIENT_ID }}
    tenant-id: ${{ secrets.AZURE_TENANT_ID }}
    subscription-id: ${{ secrets.AZURE_SUBSCRIPTION_ID }}
    # No client-secret needed! OIDC handles auth
```

## Built-in Environment Variables

GitHub automatically sets these environment variables in every workflow run:

| Variable             | Description                          | Example                     |
| -------------------- | ------------------------------------ | --------------------------- |
| `GITHUB_SHA`         | Full commit SHA                      | `a2b3c4d5...`               |
| `GITHUB_REF`         | Full ref that triggered the workflow | `refs/heads/main`           |
| `GITHUB_REF_NAME`    | Short branch/tag name                | `main`, `v1.2.3`            |
| `GITHUB_HEAD_REF`    | Source branch of a PR                | `feature/my-feature`        |
| `GITHUB_BASE_REF`    | Target branch of a PR                | `main`                      |
| `GITHUB_ACTOR`       | Username that triggered the run      | `octocat`                   |
| `GITHUB_REPOSITORY`  | `owner/repo` slug                    | `my-org/my-repo`            |
| `GITHUB_WORKSPACE`   | Working directory path               | `/home/runner/work/repo`    |
| `GITHUB_RUN_ID`      | Unique run ID (workflow-scoped)      | `1234567890`                |
| `GITHUB_RUN_NUMBER`  | Sequential run number                | `42`                        |
| `GITHUB_RUN_ATTEMPT` | Attempt number (1 on first try)      | `1`                         |
| `GITHUB_SERVER_URL`  | GitHub server URL                    | `https://github.com`        |
| `GITHUB_API_URL`     | GitHub API URL                       | `https://api.github.com`    |
| `RUNNER_OS`          | Operating system of the runner       | `Linux`, `Windows`, `macOS` |
| `RUNNER_ARCH`        | CPU architecture                     | `X64`, `ARM64`              |
| `RUNNER_TEMP`        | Path to temp directory               | `/tmp`                      |
| `CI`                 | Always `true` in GitHub Actions      | `true`                      |

## `$GITHUB_ENV` — Persisting Variables Between Steps

```yaml
- name: Compute and persist variables
  run: |
    APP_VERSION=$(node -p "require('./package.json').version")
    BUILD_DATE=$(date -u +%Y-%m-%dT%H:%M:%SZ)
    DOCKER_TAG="$REGISTRY/$IMAGE_NAME:$APP_VERSION"

    echo "APP_VERSION=$APP_VERSION" >> $GITHUB_ENV
    echo "BUILD_DATE=$BUILD_DATE" >> $GITHUB_ENV
    echo "DOCKER_TAG=$DOCKER_TAG" >> $GITHUB_ENV

- name: Use persisted variables
  run: |
    echo "Building $APP_VERSION at $BUILD_DATE"
    docker build -t "$DOCKER_TAG" .
    docker push "$DOCKER_TAG"
```

## `$GITHUB_PATH` — Extending PATH

```yaml
- name: Install custom binary
  run: |
    curl -Lo /usr/local/bin/my-tool https://github.com/org/tool/releases/latest/download/tool-linux-amd64
    chmod +x /usr/local/bin/my-tool
    echo "/usr/local/bin" >> $GITHUB_PATH

- name: Use installed binary
  run: my-tool --version # now available in PATH
```

## Context Objects

All context data is accessible via `${{ context.property }}` in expressions.

### `github` Context

```yaml
${{ github.event_name }}          # push, pull_request, workflow_dispatch, etc.
${{ github.ref }}                 # refs/heads/main
${{ github.sha }}                 # full commit SHA
${{ github.repository }}          # owner/repo
${{ github.actor }}               # triggering username
${{ github.run_id }}              # workflow run ID
${{ github.run_number }}          # sequential run number
${{ github.workflow }}            # workflow name
${{ github.job }}                 # current job ID
${{ github.event.pull_request.number }}   # PR number (in PR contexts)
${{ github.event.release.tag_name }}      # release tag (in release contexts)
${{ toJSON(github) }}             # dump entire context for debugging
```

### `env` Context

```yaml
${{ env.MY_VARIABLE }} # access workflow/job/step env vars in expressions
```

### `vars` Context

Access non-secret configuration variables (set in Settings > Variables):

```yaml
${{ vars.DEPLOY_REGION }}
${{ vars.MAX_RETRIES }}
```

### `secrets` Context

```yaml
${{ secrets.API_KEY }}
${{ secrets.GITHUB_TOKEN }}
```

### `needs` Context

```yaml
${{ needs.build.outputs.version }}
${{ needs.build.result }} # success, failure, cancelled, skipped
```

### `job` Context

```yaml
${{ job.status }}               # success, failure, cancelled
${{ job.container.id }}         # container ID (if using container jobs)
```

### `steps` Context

```yaml
${{ steps.my-step-id.outputs.result }}
${{ steps.my-step-id.outcome }}   # success, failure, cancelled, skipped
${{ steps.my-step-id.conclusion }}
```

### `runner` Context

```yaml
${{ runner.os }}        # Linux, Windows, macOS
${{ runner.arch }}      # X64, ARM64
${{ runner.temp }}      # temp directory path
${{ runner.name }}      # runner name
```

### `matrix` Context

```yaml
${{ matrix.node }}
${{ matrix.os }}
${{ matrix.include.experimental }}
```

## Expression Functions

### String Functions

```yaml
# contains(searchString, searchValue)
if: contains(github.ref, 'release/')
if: contains(github.event.pull_request.labels.*.name, 'hotfix')

# startsWith(searchString, searchValue)
if: startsWith(github.ref, 'refs/tags/')

# endsWith(searchString, searchValue)
if: endsWith(github.actor, '[bot]')

# format(string, ...replaceValues)
run: echo "${{ format('Hello {0}, your PR is #{1}', github.actor, github.event.number) }}"

# join(array, separator)
run: echo "${{ join(github.event.pull_request.labels.*.name, ', ') }}"
```

### Type Conversion

```yaml
# toJSON(value) — convert to JSON string
run: echo "${{ toJSON(matrix) }}"

# fromJSON(value) — parse JSON string
if: fromJSON(steps.check.outputs.result).deploy == true
```

### Hash Functions

```yaml
# hashFiles(path) — SHA-256 of files (for cache keys)
key: ${{ runner.os }}-deps-${{ hashFiles('**/package-lock.json') }}
key: ${{ hashFiles('Cargo.lock', 'rust-toolchain.toml') }}
```

### Status Check Functions

These are used in `if:` conditions on steps/jobs to handle failures:

```yaml
# success() — true if all previous steps succeeded (default)
- if: success()
  run: echo "Everything passed"

# failure() — true if any previous step failed
- if: failure()
  run: ./notify-failure.sh

# always() — always runs, regardless of previous step status
- if: always()
  uses: actions/upload-artifact@v4
  with:
    name: test-results
    path: test-results/

# cancelled() — true if the workflow was cancelled
- if: cancelled()
  run: echo "Workflow was cancelled"

# Combined: run only if a specific step failed
- if: failure() && steps.tests.outcome == 'failure'
  run: echo "Tests specifically failed"
```

## Complete Secrets and OIDC Example

```yaml
name: Deploy with OIDC

on:
  push:
    branches: [main]

permissions:
  id-token: write
  contents: read

env:
  AWS_REGION: us-east-1
  ECR_REGISTRY: 123456789012.dkr.ecr.us-east-1.amazonaws.com
  IMAGE_NAME: my-app

jobs:
  build-and-push:
    runs-on: ubuntu-latest
    outputs:
      image-uri: ${{ steps.build.outputs.uri }}
    steps:
      - uses: actions/checkout@v4

      - name: Configure AWS via OIDC
        uses: aws-actions/configure-aws-credentials@v4
        with:
          role-to-assume: ${{ vars.AWS_DEPLOY_ROLE_ARN }}
          aws-region: ${{ env.AWS_REGION }}

      - name: Login to Amazon ECR
        id: login-ecr
        uses: aws-actions/amazon-ecr-login@v2

      - name: Build and push
        id: build
        run: |
          URI="$ECR_REGISTRY/$IMAGE_NAME:${{ github.sha }}"
          docker build -t "$URI" .
          docker push "$URI"
          echo "uri=$URI" >> $GITHUB_OUTPUT

  deploy:
    needs: build-and-push
    runs-on: ubuntu-latest
    environment: production
    steps:
      - name: Configure AWS via OIDC
        uses: aws-actions/configure-aws-credentials@v4
        with:
          role-to-assume: ${{ vars.AWS_DEPLOY_ROLE_ARN }}
          aws-region: ${{ env.AWS_REGION }}

      - name: Deploy to ECS
        run: |
          aws ecs update-service \
            --cluster production \
            --service my-app \
            --force-new-deployment \
            --region $AWS_REGION
          echo "Deployed image: ${{ needs.build-and-push.outputs.image-uri }}"
```

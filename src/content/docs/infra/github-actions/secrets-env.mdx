Secure, flexible configuration is critical in CI/CD pipelines. This page covers GitHub's secrets system, OIDC-based cloud authentication, the full set of predefined environment variables, context objects, and the expression syntax.

## Secrets

### Repository Secrets

Repository secrets are encrypted values stored at the repository level. They are available to all workflows unless overridden by environment secrets.

<Steps>
1. Go to **Settings → Secrets and variables → Actions**
2. Click **New repository secret**
3. Enter a name (e.g., `DATABASE_URL`) and value
4. Click **Add secret**
</Steps>

Access in workflows:

```yaml
steps:
  - name: Use secret
    env:
      DB_URL: ${{ secrets.DATABASE_URL }}
    run: ./connect.sh
```

### Environment Secrets

Environment secrets are scoped to a specific deployment environment (e.g., `staging`, `production`). They override repository secrets of the same name.

```yaml
jobs:
  deploy:
    runs-on: ubuntu-latest
    environment: production   # Activates production environment secrets
    steps:
      - run: ./deploy.sh
        env:
          API_KEY: ${{ secrets.API_KEY }}   # Reads from production environment
```

### Organization Secrets

Organization-level secrets can be shared across multiple repositories. Admins can restrict which repositories can access each secret.

```yaml
# Same syntax — GitHub resolves the scope automatically
env:
  ORG_WIDE_KEY: ${{ secrets.SHARED_KEY }}
```

### Dependabot Secrets

Dependabot PRs cannot access regular repository secrets for security reasons. Create separate **Dependabot secrets** under **Settings → Secrets and variables → Dependabot**.

:::caution[Secret visibility]
Secrets are masked in logs but are passed to `run:` steps as plain environment variables. Never echo a secret directly: `run: echo "${{ secrets.MY_SECRET }}"` will be masked, but could be exposed by external tools or subprocesses.
:::

## `GITHUB_TOKEN`

Every workflow run receives an auto-generated `GITHUB_TOKEN` without any setup. It authenticates as the `github-actions[bot]` app on your repository.

```yaml
jobs:
  release:
    runs-on: ubuntu-latest
    permissions:
      contents: write
    steps:
      - uses: actions/checkout@v4
        with:
          token: ${{ secrets.GITHUB_TOKEN }}

      - name: Create a release
        run: gh release create v1.0.0 --generate-notes
        env:
          GH_TOKEN: ${{ secrets.GITHUB_TOKEN }}
```

Default `GITHUB_TOKEN` permissions vary by repository setting. You can restrict the default to read-only in **Settings → Actions → General → Workflow permissions** and grant specific permissions per workflow.

## OIDC Cloud Authentication

OpenID Connect (OIDC) lets GitHub Actions authenticate to cloud providers without storing long-lived credentials as secrets. The workflow exchanges a short-lived OIDC token for cloud credentials.

### AWS with OIDC

<Steps>
1. Create an OIDC identity provider in AWS IAM:
   - Provider URL: `https://token.actions.githubusercontent.com`
   - Audience: `sts.amazonaws.com`

2. Create an IAM role with a trust policy allowing your repository:
   ```json
   {
     "Condition": {
       "StringLike": {
         "token.actions.githubusercontent.com:sub":
           "repo:OWNER/REPO:ref:refs/heads/main"
       }
     }
   }
   ```

3. Use in your workflow:
   ```yaml
   jobs:
     deploy:
       runs-on: ubuntu-latest
       permissions:
         id-token: write   # Required for OIDC
         contents: read
       steps:
         - uses: aws-actions/configure-aws-credentials@v4
           with:
             role-to-assume: arn:aws:iam::123456789:role/GitHubActionsRole
             aws-region: us-east-1

         - run: aws s3 ls
   ```
</Steps>

### GCP with OIDC

```yaml
jobs:
  deploy:
    runs-on: ubuntu-latest
    permissions:
      id-token: write
      contents: read

    steps:
      - uses: google-github-actions/auth@v2
        with:
          workload_identity_provider: 'projects/123/locations/global/workloadIdentityPools/my-pool/providers/my-provider'
          service_account: 'my-service-account@my-project.iam.gserviceaccount.com'

      - uses: google-github-actions/setup-gcloud@v2

      - run: gcloud run deploy my-service --image gcr.io/my-project/my-image
```

### Azure with OIDC

```yaml
steps:
  - uses: azure/login@v2
    with:
      client-id: ${{ secrets.AZURE_CLIENT_ID }}
      tenant-id: ${{ secrets.AZURE_TENANT_ID }}
      subscription-id: ${{ secrets.AZURE_SUBSCRIPTION_ID }}
    # No client secret needed — uses federated OIDC credential

  - run: az webapp deploy --name myapp --resource-group mygroup
```

:::tip[OIDC is more secure than static credentials]
OIDC tokens are short-lived (scoped to a single workflow run) and cannot be replayed. With static credentials, a leaked secret stays valid until rotated. Prefer OIDC for all cloud provider authentication.
:::

## Predefined Environment Variables

GitHub provides environment variables in every workflow run:

| Variable | Description | Example |
|---|---|---|
| `GITHUB_SHA` | Commit SHA that triggered the run | `a1b2c3d4...` |
| `GITHUB_REF` | Full ref that triggered the run | `refs/heads/main` |
| `GITHUB_REF_NAME` | Short branch or tag name | `main` |
| `GITHUB_HEAD_REF` | Source branch of a PR | `feature/my-feature` |
| `GITHUB_BASE_REF` | Target branch of a PR | `main` |
| `GITHUB_ACTOR` | Username that triggered the run | `octocat` |
| `GITHUB_REPOSITORY` | Owner and repo name | `owner/repo` |
| `GITHUB_WORKSPACE` | Working directory path | `/home/runner/work/repo/repo` |
| `GITHUB_RUN_ID` | Unique run ID | `1658821493` |
| `GITHUB_RUN_NUMBER` | Sequential run number | `42` |
| `GITHUB_RUN_ATTEMPT` | Attempt number (1 on first try) | `1` |
| `GITHUB_EVENT_NAME` | Triggering event | `push`, `pull_request` |
| `GITHUB_SERVER_URL` | GitHub server URL | `https://github.com` |
| `GITHUB_API_URL` | GitHub API URL | `https://api.github.com` |
| `RUNNER_OS` | Runner OS | `Linux`, `Windows`, `macOS` |
| `RUNNER_ARCH` | Runner architecture | `X64`, `ARM64` |
| `RUNNER_TEMP` | Temp directory (cleaned after job) | `/home/runner/work/_temp` |
| `RUNNER_TOOL_CACHE` | Tool cache directory | `/opt/hostedtoolcache` |

```yaml
steps:
  - name: Print run info
    run: |
      echo "Triggered by: $GITHUB_ACTOR"
      echo "Commit: $GITHUB_SHA"
      echo "Branch: $GITHUB_REF_NAME"
      echo "Run #$GITHUB_RUN_NUMBER (attempt $GITHUB_RUN_ATTEMPT)"
```

## Context Objects

Contexts are objects accessible via the `${{ }}` expression syntax. They contain rich metadata about the current run.

### `github` Context

```yaml
${{ github.event_name }}        # 'push', 'pull_request', etc.
${{ github.ref }}               # 'refs/heads/main'
${{ github.sha }}               # Full commit SHA
${{ github.actor }}             # User or bot that triggered the workflow
${{ github.repository }}        # 'owner/repo'
${{ github.run_id }}            # Unique run ID
${{ github.run_number }}        # Sequential run number
${{ github.workflow }}          # Workflow name
${{ github.job }}               # Current job ID
${{ github.event.pull_request.number }}  # PR number (in PR events)
${{ github.event.head_commit.message }}  # Commit message (in push events)
```

### `env` Context

Access environment variables defined in the workflow:

```yaml
env:
  APP_NAME: myapp

steps:
  - run: echo "${{ env.APP_NAME }}"
```

### `vars` Context

Repository/org variables (non-secret configuration values):

```yaml
# Set in Settings → Secrets and variables → Variables
${{ vars.DEPLOY_REGION }}
${{ vars.NODE_VERSION }}
```

### `secrets` Context

```yaml
${{ secrets.GITHUB_TOKEN }}
${{ secrets.MY_API_KEY }}
```

### `needs` Context

Access outputs from dependency jobs:

```yaml
${{ needs.build.outputs.image-tag }}
${{ needs.build.result }}    # success | failure | cancelled | skipped
```

### `job` Context

```yaml
${{ job.status }}    # success | failure | cancelled
```

### `steps` Context

Access outputs from previous steps in the same job:

```yaml
- id: my-step
  run: echo "result=42" >> $GITHUB_OUTPUT

- run: echo "${{ steps.my-step.outputs.result }}"
          # Also: steps.my-step.outcome, steps.my-step.conclusion
```

### `matrix` Context

```yaml
strategy:
  matrix:
    os: [ubuntu-latest, windows-latest]

steps:
  - run: echo "Running on ${{ matrix.os }}"
```

### `runner` Context

```yaml
${{ runner.os }}       # Linux | Windows | macOS
${{ runner.arch }}     # X64 | ARM64
${{ runner.temp }}     # Temp directory path
${{ runner.name }}     # Runner name
```

## Expression Syntax and Functions

Expressions use `${{ }}` syntax and support a rich set of functions.

### Comparison Operators

```yaml
${{ github.ref == 'refs/heads/main' }}
${{ github.event_name != 'schedule' }}
${{ github.run_number > 100 }}
${{ contains(github.ref, 'release') }}
```

### String Functions

```yaml
# contains(haystack, needle)
${{ contains(github.ref, 'hotfix') }}
${{ contains(toJSON(github.event.labels.*.name), 'deploy') }}

# startsWith / endsWith
${{ startsWith(github.ref, 'refs/tags/') }}
${{ endsWith(github.ref, '-rc') }}

# format
${{ format('Hello {0}, you triggered run {1}', github.actor, github.run_number) }}

# join
${{ join(matrix.*, ', ') }}
```

### JSON Functions

```yaml
# toJSON — serialize to JSON string
- run: echo "${{ toJSON(github.event) }}"

# fromJSON — parse JSON string
strategy:
  matrix: ${{ fromJSON(needs.setup.outputs.matrix) }}
```

### Hash Functions

```yaml
# hashFiles — hash one or more file globs (used for cache keys)
key: ${{ hashFiles('**/package-lock.json') }}
key: ${{ hashFiles('**/pom.xml', '**/build.gradle') }}
```

### Status Functions (for `if:` conditions)

```yaml
# success() — true if all previous steps succeeded (default)
if: success()

# failure() — true if any previous step failed
if: failure()

# always() — always runs (even if cancelled)
if: always()

# cancelled() — true if the workflow was cancelled
if: cancelled()
```

### Combining Functions

```yaml
# Run only on main, but always upload logs
- name: Upload failure logs
  if: failure() && github.ref == 'refs/heads/main'
  uses: actions/upload-artifact@v4
  with:
    name: failure-logs
    path: logs/

# Run if any dependency failed
- name: Notify on failure
  if: contains(needs.*.result, 'failure')
  needs: [lint, test, build]
  run: ./notify.sh
```

## Practical Secrets Patterns

### Dynamic Token Generation

```yaml
- name: Generate temp token
  id: get-token
  run: |
    TOKEN=$(./scripts/generate-deploy-token.sh)
    echo "::add-mask::$TOKEN"         # Mask in all subsequent logs
    echo "token=$TOKEN" >> $GITHUB_OUTPUT

- name: Deploy with token
  run: ./deploy.sh
  env:
    DEPLOY_TOKEN: ${{ steps.get-token.outputs.token }}
```

### Conditional Secrets

```yaml
- name: Set environment-based secret
  run: |
    if [ "${{ github.ref }}" = "refs/heads/main" ]; then
      echo "API_URL=${{ secrets.PROD_API_URL }}" >> $GITHUB_ENV
    else
      echo "API_URL=${{ secrets.STAGING_API_URL }}" >> $GITHUB_ENV
    fi
```

### Validate Required Secrets

```yaml
- name: Validate required secrets are set
  run: |
    required_secrets=("${{ secrets.AWS_ROLE_ARN }}" "${{ secrets.ECR_REGISTRY }}")
    for secret in "${required_secrets[@]}"; do
      if [ -z "$secret" ]; then
        echo "Required secret is not set"
        exit 1
      fi
    done
    echo "All required secrets are present"
```

:::tip[Use `vars` for non-sensitive config]
Repository variables (`vars.*`) are like secrets but visible in the UI and not masked in logs. Use them for configuration like `NODE_VERSION`, `AWS_REGION`, `SLACK_CHANNEL` — values that are not sensitive but change between environments.
:::

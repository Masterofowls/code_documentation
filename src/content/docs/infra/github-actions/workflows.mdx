## Trigger Events In Detail

The `on:` key defines what causes your workflow to run. You can specify one event, a list of events, or a map with filters.

### `push`

Triggered when commits are pushed to matching branches or tags.

```yaml
on:
  push:
    branches:
      - main
      - develop
      - 'feature/**'          # glob: all feature branches
      - '!feature/wip-*'      # exclude WIP branches (negate with !)
    tags:
      - 'v[0-9]+.[0-9]+.[0-9]+'  # semver tags only
    paths:
      - 'src/**'
      - 'package.json'
      - 'package-lock.json'
    paths-ignore:
      - '**/*.md'
      - 'docs/**'
```

### `pull_request`

Triggered by activity on pull requests. Runs in the context of the **merge commit** (base + head merged).

```yaml
on:
  pull_request:
    branches: [main, develop]
    types:
      - opened          # PR created
      - synchronize     # new commits pushed
      - reopened        # re-opened after close
      - labeled         # label added
      - ready_for_review  # converted from draft
    paths:
      - 'src/**'
```

:::caution[pull_request and forks]
For pull requests from forks, `GITHUB_TOKEN` has read-only permissions and secrets are not available. Use `pull_request_target` with caution for fork-PR workflows that need elevated permissions — it runs in the context of the base branch and has access to secrets.
:::

### `pull_request_target`

Runs in the context of the **base** branch. Secrets are available. Used for workflows that need to comment on PRs from forks.

```yaml
on:
  pull_request_target:
    types: [opened, synchronize]

jobs:
  label-pr:
    runs-on: ubuntu-latest
    permissions:
      pull-requests: write
    steps:
      - uses: actions/labeler@v5
        with:
          repo-token: ${{ secrets.GITHUB_TOKEN }}
```

### `workflow_dispatch`

Allows manual triggering from the GitHub UI, CLI, or REST API.

```yaml
on:
  workflow_dispatch:
    inputs:
      environment:
        description: 'Deployment target'
        type: choice
        required: true
        options: [dev, staging, production]
      version:
        description: 'Semver version (e.g., 1.2.3)'
        type: string
        required: false
      run_tests:
        description: 'Run test suite before deploy'
        type: boolean
        default: true
```

Reference inputs with `${{ inputs.environment }}`.

### `workflow_call`

Makes a workflow callable by other workflows (reusable workflow). See the [Reusable Workflows](/infra/github-actions/reusable) page.

```yaml
on:
  workflow_call:
    inputs:
      environment:
        type: string
        required: true
    secrets:
      deploy_key:
        required: true
    outputs:
      deployed_url:
        description: 'URL of the deployed application'
        value: ${{ jobs.deploy.outputs.url }}
```

### `schedule`

Runs on a cron schedule. Uses UTC timezone.

```yaml
on:
  schedule:
    - cron: '0 2 * * *'       # 2am UTC daily
    - cron: '0 9 * * 1'       # 9am UTC every Monday
    - cron: '*/15 * * * *'    # every 15 minutes (minimum interval)
```

Cron fields: `minute(0-59) hour(0-23) day(1-31) month(1-12) weekday(0-7)`

### `release`

```yaml
on:
  release:
    types:
      - created     # new release created (draft or published)
      - published   # release published (not draft)
      - prereleased # pre-release published
      - released    # non-pre-release published
```

### `issues` and `issue_comment`

```yaml
on:
  issues:
    types: [opened, labeled, assigned]
  issue_comment:
    types: [created]
```

Useful for chatops — trigger deployments or actions by posting specific comments.

### `workflow_run`

Triggered when another workflow run completes. Useful for separating CI (which runs on forks without secrets) from CD (which needs secrets).

```yaml
on:
  workflow_run:
    workflows: ['CI']         # exact workflow name(s)
    types: [completed]
    branches: [main]

jobs:
  deploy:
    runs-on: ubuntu-latest
    if: ${{ github.event.workflow_run.conclusion == 'success' }}
    steps:
      - run: echo "CI passed, deploying..."
```

### `repository_dispatch`

Triggered by an external HTTP POST to the GitHub API. Useful for triggering workflows from external systems.

```yaml
on:
  repository_dispatch:
    types: [deploy-request, build-complete]

jobs:
  handle:
    runs-on: ubuntu-latest
    steps:
      - run: echo "Event type: ${{ github.event.action }}"
      - run: echo "Payload: ${{ toJSON(github.event.client_payload) }}"
```

Trigger it via API:
```bash
curl -X POST \
  -H "Authorization: token $PAT" \
  -H "Accept: application/vnd.github+json" \
  https://api.github.com/repos/ORG/REPO/dispatches \
  -d '{"event_type":"deploy-request","client_payload":{"env":"staging"}}'
```

### Push Tags for Releases

A common pattern: trigger release workflows only on semver tags.

```yaml
on:
  push:
    tags:
      - 'v[0-9]+.[0-9]+.[0-9]+'       # v1.2.3
      - 'v[0-9]+.[0-9]+.[0-9]+-*'     # v1.2.3-beta.1
```

## Concurrency Control

Prevent duplicate workflow runs with `concurrency`:

```yaml
concurrency:
  group: ${{ github.workflow }}-${{ github.ref }}
  cancel-in-progress: true
```

The `group` key is a string that identifies concurrent runs. Any new run cancels the existing in-progress run in the same group.

### Common Concurrency Patterns

```yaml
# Per-branch CI — cancel old runs when new push arrives
concurrency:
  group: ci-${{ github.ref }}
  cancel-in-progress: true

# Per-environment deployment — don't cancel; queue them
concurrency:
  group: deploy-production
  cancel-in-progress: false

# Per-PR — cancel when new commits pushed
concurrency:
  group: pr-${{ github.event.pull_request.number }}
  cancel-in-progress: true
```

## Permissions

The `permissions:` block sets the permissions granted to `GITHUB_TOKEN` for this workflow or job. Best practice: request only the permissions you need.

```yaml
permissions:
  contents: read          # read repository contents
  pull-requests: write    # create/update PR comments
  issues: write           # create/update issues
  checks: write           # create check runs
  statuses: write         # update commit status
  packages: write         # push to GitHub Packages
  id-token: write         # OIDC token for cloud auth
  deployments: write      # create deployment records
  security-events: write  # upload SARIF results
```

### Setting Default Permissions

```yaml
# Restrictive default — opt-in to what you need
permissions:
  contents: read

jobs:
  build:
    permissions:
      contents: read
      packages: write    # only this job can push packages
    runs-on: ubuntu-latest
    steps:
      - run: echo "building..."
```

:::tip[Least privilege]
Set `permissions: read-all` or `permissions: {}` at the workflow level and grant only necessary permissions per job. This limits blast radius if a step is compromised.
:::

## Matrix Strategy

Run the same job with multiple configurations:

### Basic Matrix

```yaml
jobs:
  test:
    runs-on: ${{ matrix.os }}
    strategy:
      matrix:
        os: [ubuntu-latest, windows-latest, macos-latest]
        node: ['18', '20', '22']
      fail-fast: false      # don't cancel all if one fails
      max-parallel: 6       # limit concurrent jobs
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: ${{ matrix.node }}
      - run: npm test
```

This creates 3 × 3 = 9 parallel jobs.

### Include: Add Extra Combinations

```yaml
strategy:
  matrix:
    os: [ubuntu-latest, windows-latest]
    node: ['18', '20']
    include:
      - os: ubuntu-latest
        node: '20'
        experimental: true      # extra property for this combination
      - os: macos-latest
        node: '20'              # adds a new combination not in the base
```

### Exclude: Remove Combinations

```yaml
strategy:
  matrix:
    os: [ubuntu-latest, windows-latest, macos-latest]
    node: ['18', '20']
    exclude:
      - os: windows-latest
        node: '18'    # skip Windows + Node 18
```

### Dynamic Matrix from Job Output

```yaml
jobs:
  setup:
    runs-on: ubuntu-latest
    outputs:
      matrix: ${{ steps.set-matrix.outputs.matrix }}
    steps:
      - id: set-matrix
        run: |
          echo 'matrix={"service":["api","frontend","worker"]}' >> $GITHUB_OUTPUT

  test:
    needs: setup
    runs-on: ubuntu-latest
    strategy:
      matrix: ${{ fromJSON(needs.setup.outputs.matrix) }}
    steps:
      - run: echo "Testing ${{ matrix.service }}"
```

## `if` Conditions at Workflow Level

Skip jobs or steps based on conditions:

```yaml
jobs:
  deploy:
    runs-on: ubuntu-latest
    # Only deploy from main branch, not PRs
    if: github.ref == 'refs/heads/main' && github.event_name == 'push'
    steps:
      - run: echo "Deploying..."

  notify-failure:
    runs-on: ubuntu-latest
    needs: deploy
    # Run even if deploy fails
    if: always() && needs.deploy.result == 'failure'
    steps:
      - run: echo "Deploy failed! Sending alert..."
```

### Common `if` Conditions

```yaml
# Only on main branch push
if: github.ref == 'refs/heads/main'

# Only on PRs
if: github.event_name == 'pull_request'

# Only on tag push
if: startsWith(github.ref, 'refs/tags/v')

# Skip dependabot PRs
if: github.actor != 'dependabot[bot]'

# Only on manual trigger with specific input
if: inputs.environment == 'production'

# Run after failure
if: failure()

# Always run (even after cancelled)
if: always()
```

## Filtering with `paths`

Avoid running workflows when unrelated files change:

```yaml
on:
  push:
    branches: [main]
    paths:
      - 'backend/**'
      - 'shared/**'
      - '.github/workflows/backend.yml'

  pull_request:
    paths-ignore:
      - '*.md'
      - 'docs/**'
      - '.github/CODEOWNERS'
```

:::caution[Required status checks]
If your workflow is set as a required status check for merging, and a PR only changes ignored paths (so the workflow doesn't run), the PR will be blocked. Use `paths` filters carefully with required checks, or use a workaround job that always passes.
:::

## Full Workflow Example

```yaml
name: Full CI/CD

on:
  push:
    branches: [main]
    paths: ['src/**', 'package*.json']
  pull_request:
    branches: [main]
    types: [opened, synchronize]
  workflow_dispatch:
    inputs:
      skip_tests:
        type: boolean
        default: false

concurrency:
  group: ${{ github.workflow }}-${{ github.ref }}
  cancel-in-progress: ${{ github.event_name == 'pull_request' }}

permissions:
  contents: read
  pull-requests: write
  id-token: write

jobs:
  lint:
    name: Lint
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: '20'
          cache: 'npm'
      - run: npm ci && npm run lint

  test:
    name: Test (Node ${{ matrix.node }})
    runs-on: ubuntu-latest
    if: ${{ !inputs.skip_tests }}
    strategy:
      matrix:
        node: ['18', '20']
      fail-fast: false
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: ${{ matrix.node }}
          cache: 'npm'
      - run: npm ci && npm test

  deploy:
    name: Deploy
    runs-on: ubuntu-latest
    needs: [lint, test]
    if: github.ref == 'refs/heads/main' && github.event_name == 'push'
    environment: production
    steps:
      - uses: actions/checkout@v4
      - run: echo "Deploying to production..."
```

Workflows are the top-level unit of automation in GitHub Actions. This page covers every trigger event available, how to control workflow concurrency, permission scoping, matrix strategies, and advanced patterns.

## Trigger Events (`on:`)

### `push`

Triggered when commits are pushed to a branch or tag is pushed.

```yaml
on:
  push:
    branches:
      - main
      - 'release/**'
      - '!release/old-*'      # Negate with !
    branches-ignore:
      - 'dependabot/**'
    tags:
      - 'v*.*.*'              # Semantic version tags
      - '!v*-beta'            # Exclude beta tags
    paths:
      - 'src/**'
      - 'package.json'
      - 'Dockerfile'
    paths-ignore:
      - '**.md'
      - '.github/CODEOWNERS'
```

### `pull_request`

Triggered on pull request lifecycle events. Runs in the context of the **merge commit** (what the branch would look like if merged). For PRs from forks, secrets are not available — use `pull_request_target` carefully.

```yaml
on:
  pull_request:
    types:
      - opened          # PR created
      - synchronize     # New commits pushed to PR
      - reopened        # Closed PR reopened
      - ready_for_review
      - labeled
      - unlabeled
      - assigned
    branches:
      - main
      - develop
    paths:
      - 'src/**'
```

### `pull_request_target`

Runs in the context of the **base branch** (not the PR branch). Has access to secrets and write permissions. Use for workflows that need to comment on or label PRs from forks.

:::caution[Security risk]
`pull_request_target` runs arbitrary code from fork branches with access to your secrets. Always pin the checkout to `github.event.pull_request.head.sha` and be extremely careful about what you execute.
:::

```yaml
on:
  pull_request_target:
    types: [opened, labeled]

jobs:
  label:
    runs-on: ubuntu-latest
    permissions:
      pull-requests: write
    steps:
      - uses: actions/labeler@v5
        with:
          repo-token: ${{ secrets.GITHUB_TOKEN }}
```

### `workflow_dispatch`

Manual trigger with optional user inputs. Dispatched from the GitHub UI (Actions tab → workflow → Run workflow) or the API.

```yaml
on:
  workflow_dispatch:
    inputs:
      environment:
        description: 'Deployment environment'
        required: true
        type: choice
        options:
          - staging
          - production
        default: staging
      version:
        description: 'Version tag to deploy'
        required: true
        type: string
      skip_tests:
        description: 'Skip test suite'
        required: false
        type: boolean
        default: false
      notify_channel:
        description: 'Slack channel for notifications'
        required: false
        type: string
        default: '#deployments'

jobs:
  deploy:
    runs-on: ubuntu-latest
    environment: ${{ inputs.environment }}
    steps:
      - run: |
          echo "Deploying version ${{ inputs.version }}"
          echo "Skip tests: ${{ inputs.skip_tests }}"
```

### `workflow_call`

Makes a workflow callable by other workflows (reusable workflows). See the [Reusable Workflows](/infra/github-actions/reusable) page for full coverage.

```yaml
on:
  workflow_call:
    inputs:
      environment:
        required: true
        type: string
    secrets:
      DEPLOY_KEY:
        required: true
    outputs:
      deployed-url:
        description: 'URL of deployed app'
        value: ${{ jobs.deploy.outputs.url }}
```

### `schedule`

Cron-based triggers. All times are UTC. Schedules only run on the repository's default branch.

```yaml
on:
  schedule:
    - cron: '30 5 * * 1,3'   # 5:30 AM UTC, Mon and Wed
    - cron: '0 0 1 * *'      # Midnight UTC, first of each month
```

Common cron patterns:

| Pattern | Description |
|---|---|
| `0 * * * *` | Every hour |
| `0 9 * * 1-5` | 9 AM UTC, weekdays |
| `0 0 * * 0` | Midnight Sunday |
| `*/15 * * * *` | Every 15 minutes |
| `0 2 * * *` | 2 AM UTC daily |

### `release`

Triggered on GitHub Release lifecycle events.

```yaml
on:
  release:
    types:
      - published       # New release published (most common)
      - created
      - edited
      - prereleased
      - released        # Pre-release promoted to release
```

### `issues` and `issue_comment`

```yaml
on:
  issues:
    types: [opened, labeled, assigned]

  issue_comment:
    types: [created]

jobs:
  triage:
    if: github.event_name == 'issues' && github.event.action == 'opened'
    runs-on: ubuntu-latest
    steps:
      - run: echo "New issue opened"
```

### `workflow_run`

Triggered when another workflow run completes. Useful for post-processing after CI.

```yaml
on:
  workflow_run:
    workflows: ['CI']        # Name of the triggering workflow
    types:
      - completed
    branches:
      - main

jobs:
  notify:
    runs-on: ubuntu-latest
    if: ${{ github.event.workflow_run.conclusion == 'success' }}
    steps:
      - run: echo "CI passed, running post-CI tasks"
```

### `repository_dispatch`

External webhook trigger. Allows third-party systems to trigger workflows via the GitHub API.

```yaml
on:
  repository_dispatch:
    types: [deploy-request, data-refresh]

jobs:
  process:
    runs-on: ubuntu-latest
    steps:
      - run: |
          echo "Event type: ${{ github.event.action }}"
          echo "Client payload: ${{ toJSON(github.event.client_payload) }}"
```

Trigger via API:
```bash
curl -X POST \
  -H "Authorization: token $GITHUB_TOKEN" \
  -H "Accept: application/vnd.github.v3+json" \
  https://api.github.com/repos/OWNER/REPO/dispatches \
  -d '{"event_type":"deploy-request","client_payload":{"version":"1.2.3"}}'
```

## Concurrency Control

Prevent redundant workflow runs — e.g., cancel in-progress CI runs when new commits are pushed to the same PR.

```yaml
concurrency:
  group: ${{ github.workflow }}-${{ github.ref }}
  cancel-in-progress: true
```

The `group` is a string — runs with the same group string are treated as concurrent. Common group patterns:

```yaml
# Per-branch: cancel old runs on same branch
group: ci-${{ github.ref }}

# Per-PR: cancel old runs for same PR
group: pr-${{ github.event.pull_request.number }}

# Per-workflow + branch (multiple workflows on same branch don't conflict)
group: ${{ github.workflow }}-${{ github.ref }}

# Production deploys: queue, never cancel
concurrency:
  group: deploy-production
  cancel-in-progress: false   # Queue instead of cancel
```

:::tip[Production deployments]
Set `cancel-in-progress: false` for deployment jobs targeting production. You want new deploys to queue, not cancel an in-flight deployment that might leave the system in an inconsistent state.
:::

## Permissions

The `permissions:` key scopes the `GITHUB_TOKEN` for a workflow or job. Setting it to a minimal set is a security best practice.

```yaml
# Set at workflow level (applies to all jobs, unless overridden)
permissions:
  contents: read
  pull-requests: write

jobs:
  release:
    runs-on: ubuntu-latest
    # Override at job level
    permissions:
      contents: write       # Create releases
      packages: write       # Push to GHCR
      id-token: write       # OIDC for cloud auth
```

Available permission scopes:

| Scope | Description |
|---|---|
| `actions` | Manage workflow runs |
| `checks` | Create check runs |
| `contents` | Repository contents (code, releases) |
| `deployments` | Manage deployments |
| `id-token` | Required for OIDC authentication |
| `issues` | Read/write issues |
| `packages` | GitHub Packages (GHCR) |
| `pages` | GitHub Pages |
| `pull-requests` | PR read/write |
| `repository-projects` | Projects |
| `security-events` | Code scanning results |
| `statuses` | Commit statuses |

Set the entire workflow to read-only, then grant specific write permissions per-job:

```yaml
permissions: read-all   # Deny all writes globally

jobs:
  test:
    runs-on: ubuntu-latest
    permissions:
      checks: write     # Allow only check creation
```

## Matrix Strategy

Run a job multiple times with different parameter combinations. Matrices create parallel jobs automatically.

```yaml
jobs:
  test:
    runs-on: ${{ matrix.os }}
    strategy:
      fail-fast: false      # Don't cancel other matrix jobs on failure
      max-parallel: 4       # Limit concurrent jobs
      matrix:
        os: [ubuntu-latest, windows-latest, macos-latest]
        node: [18, 20, 22]
        # Creates 9 jobs: 3 OS × 3 Node versions

    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: ${{ matrix.node }}
      - run: npm test
```

### Include and Exclude

```yaml
strategy:
  matrix:
    os: [ubuntu-latest, windows-latest]
    node: [18, 20]
    include:
      # Add extra properties to an existing combination
      - os: ubuntu-latest
        node: 20
        experimental: true

      # Add entirely new combinations
      - os: macos-latest
        node: 20

    exclude:
      # Remove specific combinations
      - os: windows-latest
        node: 18
```

### Dynamic Matrix from JSON

Generate the matrix dynamically in a setup job:

```yaml
jobs:
  setup:
    runs-on: ubuntu-latest
    outputs:
      matrix: ${{ steps.set-matrix.outputs.matrix }}
    steps:
      - id: set-matrix
        run: |
          echo 'matrix={"include":[{"node":"18"},{"node":"20"}]}' >> $GITHUB_OUTPUT

  test:
    needs: setup
    runs-on: ubuntu-latest
    strategy:
      matrix: ${{ fromJSON(needs.setup.outputs.matrix) }}
    steps:
      - run: echo "Node ${{ matrix.node }}"
```

## Filtering with `if` at the Workflow Level

You can use `if:` on jobs to conditionally run them based on context:

```yaml
jobs:
  deploy-staging:
    if: github.ref == 'refs/heads/develop'
    runs-on: ubuntu-latest
    steps:
      - run: echo "Deploy to staging"

  deploy-prod:
    if: github.ref == 'refs/heads/main' && github.event_name == 'push'
    runs-on: ubuntu-latest
    steps:
      - run: echo "Deploy to production"

  notify:
    if: always() && contains(needs.*.result, 'failure')
    needs: [deploy-staging, deploy-prod]
    runs-on: ubuntu-latest
    steps:
      - run: echo "A deployment failed"
```

## Path Filtering

Trigger workflows only when specific paths change:

```yaml
on:
  push:
    paths:
      - 'backend/**'
      - 'shared/**'
      - 'package.json'

  pull_request:
    paths:
      - 'frontend/**'
      - '!frontend/**.md'   # Exclude docs changes in frontend
```

:::tip[Separate frontend and backend workflows]
Create separate workflow files — one for frontend changes and one for backend changes using `paths:` filtering. This prevents a CSS change from triggering your entire backend test suite.
:::

## Complete Advanced Workflow Example

```yaml
name: Advanced CI

on:
  push:
    branches: [main]
    tags: ['v*']
  pull_request:
    branches: [main]
  schedule:
    - cron: '0 3 * * 1'   # Weekly dependency audit
  workflow_dispatch:
    inputs:
      skip-deploy:
        type: boolean
        default: false

concurrency:
  group: ${{ github.workflow }}-${{ github.ref }}
  cancel-in-progress: ${{ github.ref != 'refs/heads/main' }}

permissions:
  contents: read
  pull-requests: write

jobs:
  lint:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: '20'
          cache: 'npm'
      - run: npm ci --ignore-scripts
      - run: npm run lint

  test:
    runs-on: ${{ matrix.os }}
    strategy:
      fail-fast: false
      matrix:
        os: [ubuntu-latest]
        node: [18, 20, 22]
        include:
          - os: windows-latest
            node: 20
    needs: lint
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: ${{ matrix.node }}
          cache: 'npm'
      - run: npm ci
      - run: npm test

  security-audit:
    runs-on: ubuntu-latest
    if: github.event_name == 'schedule' || github.event_name == 'push'
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: '20'
          cache: 'npm'
      - run: npm audit --audit-level=high
```

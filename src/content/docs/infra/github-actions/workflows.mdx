## Trigger Events (`on:`)

GitHub Actions supports dozens of event triggers. Understanding the right trigger for your workflow is critical for both correctness and cost efficiency.

## Code Events

### `push`

Fires when commits are pushed to any matching branch, tag, or path.

```yaml
on:
  push:
    branches:
      - main
      - 'release/**'
      - '!release/draft-*' # exclude with !
    branches-ignore:
      - 'dependabot/**'
    tags:
      - 'v[0-9]+.[0-9]+.[0-9]+' # regex-like glob: v1.2.3
    paths:
      - 'src/**'
      - 'tests/**'
      - 'package*.json'
    paths-ignore:
      - '**/*.md'
      - 'docs/**'
```

### `pull_request`

Fires for pull request activity. This trigger runs in the context of the **merge commit** and uses the target branch's permissions.

```yaml
on:
  pull_request:
    branches: [main, develop]
    types:
      - opened # PR created
      - synchronize # new commits pushed to the PR branch
      - reopened # PR reopened after close
      - ready_for_review # draft → ready
      - labeled # label added
      - unlabeled # label removed
      - closed # PR closed (check merged with github.event.pull_request.merged)
```

### `pull_request_target`

Runs in the **base branch context** (with write permissions). Used for workflows that need secrets for fork PRs — but use with extreme care.

```yaml
on:
  pull_request_target:
    types: [opened, synchronize]
```

:::caution[pull_request_target Security Risk]
`pull_request_target` has write access to the base repository and secrets. Never checkout the PR's code and then use it in a `run:` step — this is a common supply-chain attack vector. Only use it for labeling, commenting, or other metadata operations that don't execute PR code.
:::

### `workflow_run`

Triggers when another workflow completes, starts, or is requested. Runs in the base branch context with full permissions.

```yaml
on:
  workflow_run:
    workflows: [CI] # name of the triggering workflow
    types: [completed]
    branches: [main]

jobs:
  deploy:
    if: ${{ github.event.workflow_run.conclusion == 'success' }}
    runs-on: ubuntu-latest
    steps:
      - run: echo "CI passed, deploying..."
```

## Manual and External Triggers

### `workflow_dispatch` — Manual Trigger

```yaml
on:
  workflow_dispatch:
    inputs:
      environment:
        description: 'Deployment target'
        required: true
        type: choice
        options: [development, staging, production]
      version:
        description: 'Git tag or branch to deploy'
        required: true
        type: string
        default: main
      dry_run:
        description: 'Simulate without applying changes'
        type: boolean
        default: false
      log_level:
        description: 'Logging verbosity'
        type: choice
        options: [info, debug, trace]
        default: info

jobs:
  deploy:
    runs-on: ubuntu-latest
    steps:
      - name: Show inputs
        run: |
          echo "Environment: ${{ inputs.environment }}"
          echo "Version:     ${{ inputs.version }}"
          echo "Dry run:     ${{ inputs.dry_run }}"
```

### `workflow_call` — Reusable Workflow Trigger

```yaml
on:
  workflow_call:
    inputs:
      environment:
        required: true
        type: string
      image-tag:
        required: true
        type: string
    secrets:
      DEPLOY_KEY:
        required: true
    outputs:
      deploy-url:
        description: 'The deployed application URL'
        value: ${{ jobs.deploy.outputs.url }}
```

### `repository_dispatch` — External API Trigger

Triggered via the GitHub API from external systems:

```yaml
on:
  repository_dispatch:
    types: [deploy-request, cache-bust]

jobs:
  handle:
    runs-on: ubuntu-latest
    steps:
      - name: Process dispatch event
        run: |
          echo "Event type: ${{ github.event.action }}"
          echo "Client payload: ${{ toJSON(github.event.client_payload) }}"
```

```bash
# Trigger from external system via GitHub API
curl -X POST \
  -H "Accept: application/vnd.github+json" \
  -H "Authorization: Bearer $GITHUB_TOKEN" \
  https://api.github.com/repos/OWNER/REPO/dispatches \
  -d '{"event_type":"deploy-request","client_payload":{"version":"1.2.3","env":"staging"}}'
```

## Scheduled Workflows

```yaml
on:
  schedule:
    - cron: '0 0 * * *' # midnight UTC every day
    - cron: '*/15 * * * *' # every 15 minutes (minimum: 5)
    - cron: '0 8 * * 1' # 8 AM UTC every Monday
    - cron: '0 2 1 * *' # 2 AM UTC on the 1st of each month
```

Cron syntax: `minute hour day-of-month month day-of-week`

## GitHub Event Triggers

```yaml
on:
  # When a GitHub Release is published/created/edited
  release:
    types: [published, created, edited, prereleased, released]

  # When an issue is opened/closed/labeled/etc.
  issues:
    types: [opened, closed, labeled, assigned]

  # When someone comments on an issue or PR
  issue_comment:
    types: [created, edited]

  # When a branch or tag is created
  create: {}

  # When a branch or tag is deleted
  delete: {}

  # When a fork is created
  fork: {}

  # When someone stars the repo
  watch:
    types: [started]

  # When a check run completes
  check_run:
    types: [completed]
```

## Concurrency Controls

Prevent multiple runs of the same workflow from interfering with each other:

```yaml
concurrency:
  # Unique group key — same branch, same workflow
  group: ${{ github.workflow }}-${{ github.ref }}
  # Cancel any in-progress run in this group when a new one starts
  cancel-in-progress: true
```

For deployments, you want to queue rather than cancel:

```yaml
concurrency:
  group: deploy-${{ github.ref }}
  cancel-in-progress: false # queue; don't cancel in-flight deployments
```

Job-level concurrency:

```yaml
jobs:
  deploy:
    concurrency:
      group: production-deploy
      cancel-in-progress: false
```

## Permissions Block

Explicitly declare what `GITHUB_TOKEN` permissions your workflow needs. This follows the principle of least privilege and is required for OIDC cloud authentication.

```yaml
# At workflow level — applies to all jobs
permissions:
  contents: read # read repo content
  pull-requests: write # create/update PR comments
  issues: write # create/update issue comments
  packages: write # publish to GitHub Container Registry
  id-token: write # required for OIDC (AWS, GCP, Azure)
  deployments: write # create deployment records
  statuses: write # set commit status checks
  checks: write # create/update check runs
  actions: read # read workflow runs (for workflow_run trigger)
  security-events: write # upload SARIF results (code scanning)
```

Override at job level:

```yaml
jobs:
  read-only-job:
    permissions:
      contents: read # this job only needs read
    runs-on: ubuntu-latest
    steps:
      - run: echo "Can only read"

  write-job:
    permissions:
      contents: write
      pull-requests: write
    runs-on: ubuntu-latest
```

:::tip[Least Privilege Default]
Set `permissions: {}` (empty — all denied) at the workflow level, then grant only what each job needs. This limits blast radius if a step is compromised.
:::

## Matrix Strategy

Run a job multiple times with different parameter combinations:

```yaml
jobs:
  test:
    runs-on: ${{ matrix.os }}
    strategy:
      fail-fast: false # don't cancel other matrix jobs if one fails
      max-parallel: 4 # limit concurrent jobs
      matrix:
        os: [ubuntu-latest, windows-latest, macos-latest]
        node: [18, 20, 22]
        # Results in 9 combinations: 3 OS × 3 Node versions

    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: ${{ matrix.node }}
      - run: npm ci && npm test
```

### Matrix with Include and Exclude

```yaml
strategy:
  matrix:
    os: [ubuntu-latest, windows-latest]
    node: [18, 20]
    include:
      # Add extra jobs beyond the matrix
      - os: ubuntu-latest
        node: 22
        experimental: true
      # Add properties to existing combinations
      - os: windows-latest
        node: 20
        runs-extra-checks: true
    exclude:
      # Remove specific combinations
      - os: windows-latest
        node: 18
```

### Dynamic Matrix from JSON

```yaml
jobs:
  setup:
    runs-on: ubuntu-latest
    outputs:
      matrix: ${{ steps.set-matrix.outputs.matrix }}
    steps:
      - id: set-matrix
        run: |
          echo 'matrix={"include":[{"env":"staging","url":"https://staging.example.com"},{"env":"prod","url":"https://example.com"}]}' >> $GITHUB_OUTPUT

  deploy:
    needs: setup
    runs-on: ubuntu-latest
    strategy:
      matrix: ${{ fromJSON(needs.setup.outputs.matrix) }}
    steps:
      - run: echo "Deploying to ${{ matrix.env }} at ${{ matrix.url }}"
```

## Workflow-Level `if` Conditions

```yaml
on:
  push:
    branches: [main]
  pull_request:

# Skip the entire workflow for docs-only changes
jobs:
  build:
    if: |
      github.event_name == 'push' ||
      (github.event_name == 'pull_request' && !contains(github.event.pull_request.labels.*.name, 'skip-ci'))
    runs-on: ubuntu-latest
    steps:
      - run: npm run build
```

## Path Filtering with `dorny/paths-filter`

For fine-grained path-based job skipping:

```yaml
jobs:
  changes:
    runs-on: ubuntu-latest
    outputs:
      backend: ${{ steps.filter.outputs.backend }}
      frontend: ${{ steps.filter.outputs.frontend }}
    steps:
      - uses: actions/checkout@v4
      - uses: dorny/paths-filter@v3
        id: filter
        with:
          filters: |
            backend:
              - 'api/**'
              - 'packages/server/**'
            frontend:
              - 'web/**'
              - 'packages/ui/**'

  test-backend:
    needs: changes
    if: ${{ needs.changes.outputs.backend == 'true' }}
    runs-on: ubuntu-latest
    steps:
      - run: npm run test:api

  test-frontend:
    needs: changes
    if: ${{ needs.changes.outputs.frontend == 'true' }}
    runs-on: ubuntu-latest
    steps:
      - run: npm run test:web
```

## Complete Advanced Workflow Example

```yaml
name: Advanced CI/CD

on:
  push:
    branches: [main]
    tags: ['v*']
  pull_request:
    branches: [main]
    types: [opened, synchronize, reopened]
  workflow_dispatch:
    inputs:
      skip-tests:
        type: boolean
        default: false

concurrency:
  group: ${{ github.workflow }}-${{ github.ref }}
  cancel-in-progress: ${{ github.ref != 'refs/heads/main' }}

permissions:
  contents: read
  pull-requests: write

env:
  NODE_VERSION: '20'
  REGISTRY: ghcr.io

jobs:
  lint:
    runs-on: ubuntu-latest
    timeout-minutes: 5
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: ${{ env.NODE_VERSION }}
          cache: npm
      - run: npm ci --ignore-scripts
      - run: npm run lint
      - run: npm run type-check

  test:
    runs-on: ubuntu-latest
    timeout-minutes: 15
    if: ${{ !inputs.skip-tests }}
    strategy:
      matrix:
        shard: [1, 2, 3, 4] # split tests into 4 parallel shards
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: ${{ env.NODE_VERSION }}
          cache: npm
      - run: npm ci
      - run: npm test -- --shard=${{ matrix.shard }}/4

  build:
    runs-on: ubuntu-latest
    needs: [lint, test]
    timeout-minutes: 15
    outputs:
      image-tag: ${{ steps.meta.outputs.tags }}
    steps:
      - uses: actions/checkout@v4
      - run: npm ci && npm run build
      - uses: actions/upload-artifact@v4
        with:
          name: dist
          path: dist/
```

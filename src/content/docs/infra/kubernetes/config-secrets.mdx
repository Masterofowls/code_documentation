# Config & Secrets

Kubernetes separates configuration from application code using **ConfigMaps** for non-sensitive data and **Secrets** for sensitive data. This enables the same container image to run in development, staging, and production with different configurations.

---

## ConfigMaps

A ConfigMap stores configuration data as key-value pairs.

### Creating ConfigMaps

```bash
# From literal values
kubectl create configmap app-config \
  --from-literal=NODE_ENV=production \
  --from-literal=PORT=3000 \
  --from-literal=LOG_LEVEL=info

# From a file (key = filename, value = file contents)
kubectl create configmap nginx-config \
  --from-file=nginx.conf

# From a file with a custom key
kubectl create configmap nginx-config \
  --from-file=config=nginx.conf

# From a directory (each file becomes a key)
kubectl create configmap app-configs \
  --from-file=./config/

# From an .env file
kubectl create configmap app-env \
  --from-env-file=.env.production
```

### ConfigMap YAML

```yaml
apiVersion: v1
kind: ConfigMap
metadata:
  name: app-config
  namespace: production
data:
  # Simple key-value pairs
  NODE_ENV: production
  PORT: "3000"
  LOG_LEVEL: info
  DATABASE_POOL_SIZE: "10"

  # Multi-line file content
  nginx.conf: |
    server {
      listen 80;
      location / {
        proxy_pass http://api:3000;
      }
    }

  # JSON config
  app.json: |
    {
      "featureFlags": {
        "newUI": true,
        "betaAPI": false
      }
    }
```

### Using ConfigMaps as Environment Variables

```yaml
spec:
  containers:
    - name: app
      image: my-app:1.0.0

      # Inject all keys as env vars
      envFrom:
        - configMapRef:
            name: app-config
            optional: false       # Fail if configmap doesn't exist

      # Inject specific keys
      env:
        - name: NODE_ENV
          valueFrom:
            configMapKeyRef:
              name: app-config
              key: NODE_ENV
        - name: CUSTOM_NAME
          valueFrom:
            configMapKeyRef:
              name: app-config
              key: PORT           # Can rename the key
```

### Using ConfigMaps as Volumes

```yaml
spec:
  containers:
    - name: nginx
      image: nginx:alpine
      volumeMounts:
        # Mount entire configmap as directory
        - name: nginx-config
          mountPath: /etc/nginx/conf.d
          readOnly: true

        # Mount specific key as a file
        - name: app-config
          mountPath: /app/config/app.json
          subPath: app.json        # Mount only this key as this file
          readOnly: true

  volumes:
    - name: nginx-config
      configMap:
        name: nginx-config
        # Optionally map keys to filenames
        items:
          - key: nginx.conf
            path: default.conf
            mode: 0644

    - name: app-config
      configMap:
        name: app-config
```

:::tip[ConfigMap volume hot-reloading]
When you mount a ConfigMap as a volume, Kubernetes **automatically updates** the files when the ConfigMap changes (within ~1 minute). This enables configuration hot-reloading without pod restarts. Environment variable injection does NOT hot-reload.
:::

### Immutable ConfigMaps

```yaml
apiVersion: v1
kind: ConfigMap
metadata:
  name: app-config-v1
immutable: true             # Cannot be modified after creation
data:
  version: "1.0.0"
```

Immutable ConfigMaps improve cluster performance — the kubelet doesn't need to watch them for changes.

## Secrets

Secrets store sensitive data like passwords, tokens, and TLS certificates. Data is stored **base64-encoded** (not encrypted by default).

:::caution[Secrets are not encrypted by default]
By default, Secrets are stored as base64-encoded values in etcd — **not encrypted**. Anyone with access to etcd or the Kubernetes API can read them. For production, enable **encryption at rest** for etcd, or use an external secret management solution.
:::

### Secret Types

| Type | Use Case |
|---|---|
| `Opaque` | Arbitrary user-defined data (default) |
| `kubernetes.io/tls` | TLS certificate and key |
| `kubernetes.io/dockerconfigjson` | Docker registry credentials |
| `kubernetes.io/service-account-token` | Service account token |
| `kubernetes.io/basic-auth` | Basic authentication credentials |
| `kubernetes.io/ssh-auth` | SSH private key |

### Creating Secrets

```bash
# From literal values (recommended — base64 encoding is automatic)
kubectl create secret generic db-credentials \
  --from-literal=username=admin \
  --from-literal=password='S3cr3t!P@ssw0rd'

# From files
kubectl create secret generic tls-certs \
  --from-file=tls.crt=./certs/server.crt \
  --from-file=tls.key=./certs/server.key

# TLS secret
kubectl create secret tls my-tls-secret \
  --cert=./certs/server.crt \
  --key=./certs/server.key

# Docker registry credentials
kubectl create secret docker-registry ghcr-credentials \
  --docker-server=ghcr.io \
  --docker-username=myusername \
  --docker-password=$GITHUB_TOKEN
```

### Secret YAML

```yaml
apiVersion: v1
kind: Secret
metadata:
  name: db-credentials
  namespace: production
type: Opaque
# Values must be base64-encoded
data:
  username: YWRtaW4=          # echo -n 'admin' | base64
  password: UzNjcjN0IQ==      # echo -n 'S3cr3t!' | base64

# OR use stringData (Kubernetes auto-encodes these)
stringData:
  username: admin
  password: "S3cr3t!P@ssw0rd"  # Written as plaintext; stored as base64
```

```bash
# Decode a secret value
kubectl get secret db-credentials -o jsonpath='{.data.password}' | base64 -d
```

### Using Secrets as Environment Variables

```yaml
spec:
  containers:
    - name: app
      # Inject all keys from secret
      envFrom:
        - secretRef:
            name: db-credentials

      # Inject specific key
      env:
        - name: DB_PASSWORD
          valueFrom:
            secretKeyRef:
              name: db-credentials
              key: password
              optional: false
```

### Using Secrets as Volumes

```yaml
spec:
  containers:
    - name: app
      volumeMounts:
        - name: tls-certs
          mountPath: /etc/ssl/certs
          readOnly: true

        - name: db-credentials
          mountPath: /run/secrets/db
          readOnly: true

  volumes:
    - name: tls-certs
      secret:
        secretName: my-tls-secret
        defaultMode: 0400           # Restrictive permissions

    - name: db-credentials
      secret:
        secretName: db-credentials
        items:
          - key: password
            path: db-password
            mode: 0400
```

## Sealed Secrets (Bitnami) — GitOps Safe

[Sealed Secrets](https://github.com/bitnami-labs/sealed-secrets) encrypt Secrets so they can safely be committed to Git.

```bash
# Install controller
kubectl apply -f https://github.com/bitnami-labs/sealed-secrets/releases/latest/download/controller.yaml

# Install kubeseal CLI
brew install kubeseal

# Create a regular Secret
kubectl create secret generic db-credentials \
  --from-literal=password=mypassword \
  --dry-run=client -o yaml > secret.yaml

# Seal the Secret (only your cluster can decrypt it)
kubeseal --format yaml < secret.yaml > sealed-secret.yaml

# Now safe to commit sealed-secret.yaml to git!
# The controller decrypts it back to a Secret in cluster
kubectl apply -f sealed-secret.yaml
```

## External Secrets Operator

[External Secrets Operator](https://external-secrets.io) syncs secrets from external providers (AWS SSM, Vault, GCP Secret Manager, Azure Key Vault) into Kubernetes Secrets.

```yaml
# SecretStore — connect to AWS SSM Parameter Store
apiVersion: external-secrets.io/v1beta1
kind: SecretStore
metadata:
  name: aws-ssm
  namespace: production
spec:
  provider:
    aws:
      service: ParameterStore
      region: us-east-1
      auth:
        jwt:
          serviceAccountRef:
            name: external-secrets-sa
---
# ExternalSecret — define which secrets to sync
apiVersion: external-secrets.io/v1beta1
kind: ExternalSecret
metadata:
  name: db-credentials
  namespace: production
spec:
  refreshInterval: 1h           # Resync every hour
  secretStoreRef:
    name: aws-ssm
    kind: SecretStore
  target:
    name: db-credentials        # Name of the Kubernetes Secret to create
    creationPolicy: Owner
  data:
    - secretKey: password        # Key in K8s Secret
      remoteRef:
        key: /production/db/password  # Path in Parameter Store
    - secretKey: username
      remoteRef:
        key: /production/db/username
```

## The Downward API

The Downward API lets containers access information about themselves and the cluster without calling the Kubernetes API.

### As Environment Variables

```yaml
spec:
  containers:
    - name: app
      env:
        # Pod metadata
        - name: POD_NAME
          valueFrom:
            fieldRef:
              fieldPath: metadata.name
        - name: POD_NAMESPACE
          valueFrom:
            fieldRef:
              fieldPath: metadata.namespace
        - name: POD_IP
          valueFrom:
            fieldRef:
              fieldPath: status.podIP
        - name: NODE_NAME
          valueFrom:
            fieldRef:
              fieldPath: spec.nodeName
        - name: SERVICE_ACCOUNT
          valueFrom:
            fieldRef:
              fieldPath: spec.serviceAccountName

        # Resource limits (container's own limits)
        - name: MEMORY_LIMIT
          valueFrom:
            resourceFieldRef:
              containerName: app
              resource: limits.memory
        - name: CPU_REQUEST
          valueFrom:
            resourceFieldRef:
              containerName: app
              resource: requests.cpu
```

### As Volume Files

```yaml
spec:
  containers:
    - name: app
      volumeMounts:
        - name: pod-info
          mountPath: /etc/pod-info
  volumes:
    - name: pod-info
      downwardAPI:
        items:
          - path: labels
            fieldRef:
              fieldPath: metadata.labels
          - path: annotations
            fieldRef:
              fieldPath: metadata.annotations
          - path: memory-limit
            resourceFieldRef:
              containerName: app
              resource: limits.memory
```

## RBAC for Secrets

By default, any service account can read Secrets in its namespace. Restrict access with RBAC.

```yaml
# Role — read only specific secrets
apiVersion: rbac.authorization.k8s.io/v1
kind: Role
metadata:
  name: secret-reader
  namespace: production
rules:
  - apiGroups: [""]
    resources: ["secrets"]
    resourceNames: ["db-credentials", "api-keys"]   # Only these secrets
    verbs: ["get", "list"]
---
# RoleBinding — bind role to service account
apiVersion: rbac.authorization.k8s.io/v1
kind: RoleBinding
metadata:
  name: app-secret-binding
  namespace: production
subjects:
  - kind: ServiceAccount
    name: my-app
    namespace: production
roleRef:
  kind: Role
  apiGrouping: rbac.authorization.k8s.io
  name: secret-reader
```

```bash
# Verify RBAC permissions
kubectl auth can-i get secrets --as=system:serviceaccount:production:my-app -n production

# List all permissions for a service account
kubectl auth can-i --list --as=system:serviceaccount:production:my-app -n production
```

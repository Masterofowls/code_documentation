# ConfigMaps & Secrets

ConfigMaps and Secrets decouple configuration from container images, enabling the same image to run in development, staging, and production with different settings.

## ConfigMap

ConfigMaps store non-sensitive configuration data as key-value pairs.

### Creating ConfigMaps

<Tabs>
  <TabItem label="From literals">
    ```bash
    kubectl create configmap app-config \
      --from-literal=LOG_LEVEL=info \
      --from-literal=CACHE_TTL=300 \
      --from-literal=MAX_CONNECTIONS=100

    # Generate YAML
    kubectl create configmap app-config \
      --from-literal=LOG_LEVEL=info \
      --dry-run=client -o yaml
    ```

</TabItem>
  <TabItem label="From file">
    ```bash
    # Single file — key is filename
    kubectl create configmap nginx-config \
      --from-file=nginx.conf

    # Single file with custom key
    kubectl create configmap nginx-config \
      --from-file=config=nginx.conf

    # Multiple files
    kubectl create configmap app-config \
      --from-file=config/app.properties \
      --from-file=config/logging.properties
    ```

</TabItem>
  <TabItem label="From directory">
    ```bash
    # All files in directory become keys
    kubectl create configmap app-configs \
      --from-file=./config/

    # Result: each filename is a key, file content is the value
    ```

</TabItem>
  <TabItem label="From env file">
    ```bash
    # .env format file (KEY=value per line)
    kubectl create configmap app-config \
      --from-env-file=.env.production

    # .env format
    # LOG_LEVEL=info
    # PORT=3000
    # MAX_CONNECTIONS=100
    ```

</TabItem>
</Tabs>

### ConfigMap YAML

```yaml
apiVersion: v1
kind: ConfigMap
metadata:
  name: app-config
  namespace: production
data:
  # Simple key-value pairs
  LOG_LEVEL: 'info'
  PORT: '3000'
  MAX_CONNECTIONS: '100'
  FEATURE_FLAGS: 'new-ui,dark-mode'

  # Multiline value (file content)
  app.properties: |
    server.port=3000
    log.level=info
    cache.ttl=300
    database.pool.size=10

  nginx.conf: |
    server {
      listen 80;
      location / {
        proxy_pass http://app:3000;
      }
    }

  # JSON value
  config.json: |
    {
      "features": {
        "newUI": true,
        "darkMode": false
      },
      "limits": {
        "maxConnections": 100
      }
    }

# binaryData: for binary values (base64 encoded in YAML)
binaryData:
  font.woff2: <base64-encoded-content>
```

### Using ConfigMap as Environment Variables

```yaml
spec:
  containers:
    - name: myapp
      # ── All keys from ConfigMap as env vars ──────────────────
      envFrom:
        - configMapRef:
            name: app-config
        - configMapRef:
            name: feature-flags
            optional: true # don't fail if ConfigMap doesn't exist

      # ── Specific keys ────────────────────────────────────────
      env:
        - name: LOG_LEVEL
          valueFrom:
            configMapKeyRef:
              name: app-config
              key: LOG_LEVEL
        - name: MAX_CONN
          valueFrom:
            configMapKeyRef:
              name: app-config
              key: MAX_CONNECTIONS
              optional: true # don't fail if key doesn't exist
```

### Using ConfigMap as Volume (Files)

```yaml
spec:
  containers:
    - name: myapp
      volumeMounts:
        # Mount entire ConfigMap as directory
        - name: app-config-vol
          mountPath: /etc/app
          readOnly: true

        # Mount specific key as a file (subPath)
        - name: nginx-config-vol
          mountPath: /etc/nginx/nginx.conf
          subPath: nginx.conf
          readOnly: true

  volumes:
    # Mount all keys as files
    - name: app-config-vol
      configMap:
        name: app-config
        defaultMode: 0644

    # Mount only selected keys
    - name: nginx-config-vol
      configMap:
        name: nginx-config
        items:
          - key: nginx.conf
            path: nginx.conf
            mode: 0644
```

:::tip[Hot Reload for ConfigMap Volumes]
When a ConfigMap is mounted as a volume (not env vars), Kubernetes automatically updates the files when the ConfigMap changes — typically within 1–2 minutes. Many applications can watch these files and hot-reload without restarting. Environment variables do NOT update; the pod must restart.
:::

### Immutable ConfigMaps

```yaml
apiVersion: v1
kind: ConfigMap
metadata:
  name: app-config-v1
immutable: true # cannot be changed; prevents accidental modification
data:
  VERSION: '1.2.3'
```

## Secret

Secrets store sensitive data like passwords, tokens, and certificates.

:::caution[Secrets are Not Encrypted by Default]
Secrets in etcd are only base64-encoded by default — not encrypted. Anyone with access to etcd (or a cluster backup) can read them. For production, enable **encryption at rest** in the API server config, or use an external secrets manager (Vault, AWS SSM, External Secrets Operator).
:::

### Secret Types

| Type                                  | Use Case                              |
| ------------------------------------- | ------------------------------------- |
| `Opaque`                              | Arbitrary user-defined data (default) |
| `kubernetes.io/tls`                   | TLS certificate and key               |
| `kubernetes.io/dockerconfigjson`      | Docker registry credentials           |
| `kubernetes.io/service-account-token` | Service account token (auto-managed)  |
| `kubernetes.io/ssh-auth`              | SSH authentication                    |
| `kubernetes.io/basic-auth`            | Basic authentication                  |
| `bootstrap.kubernetes.io/token`       | Bootstrap tokens                      |

### Creating Secrets

```bash
# Opaque secret from literals
kubectl create secret generic app-secrets \
  --from-literal=db-password=supersecret \
  --from-literal=api-key=abc123xyz

# From files
kubectl create secret generic tls-certs \
  --from-file=tls.crt=./certs/server.crt \
  --from-file=tls.key=./certs/server.key

# TLS secret (from existing cert files)
kubectl create secret tls myapp-tls \
  --cert=server.crt \
  --key=server.key

# Docker registry credentials
kubectl create secret docker-registry regcred \
  --docker-server=ghcr.io \
  --docker-username=myuser \
  --docker-password="$GITHUB_TOKEN" \
  --docker-email=me@example.com
```

### Secret YAML

```yaml
apiVersion: v1
kind: Secret
metadata:
  name: app-secrets
  namespace: production
type: Opaque
# Values must be base64 encoded
data:
  db-password: c3VwZXJzZWNyZXQ= # echo -n 'supersecret' | base64
  api-key: YWJjMTIzeHl6 # echo -n 'abc123xyz' | base64

# stringData: plain text (Kubernetes encodes at write time)
stringData:
  db-url: 'postgresql://myuser:supersecret@db:5432/mydb'
  config.json: |
    {
      "apiKey": "abc123",
      "endpoint": "https://api.example.com"
    }
```

```bash
# Decode a secret value
kubectl get secret app-secrets -o jsonpath='{.data.db-password}' | base64 -d
```

### Using Secrets

```yaml
spec:
  containers:
    - name: myapp
      # All keys as env vars
      envFrom:
        - secretRef:
            name: app-secrets

      # Specific keys
      env:
        - name: DB_PASSWORD
          valueFrom:
            secretKeyRef:
              name: app-secrets
              key: db-password
        - name: API_KEY
          valueFrom:
            secretKeyRef:
              name: app-secrets
              key: api-key

      # Mount as files
      volumeMounts:
        - name: secrets-vol
          mountPath: /run/secrets
          readOnly: true

  volumes:
    - name: secrets-vol
      secret:
        secretName: app-secrets
        defaultMode: 0400 # owner read-only
        items:
          - key: config.json
            path: config.json

  # Pull images from private registry
  imagePullSecrets:
    - name: regcred
```

## Sealed Secrets (GitOps)

Sealed Secrets encrypt Secrets so they can be safely committed to Git. Only the controller in the cluster can decrypt them.

```bash
# Install controller
kubectl apply -f https://github.com/bitnami-labs/sealed-secrets/releases/download/v0.26.0/controller.yaml

# Install kubeseal CLI
brew install kubeseal

# Create a regular Secret YAML (do not apply yet)
kubectl create secret generic app-secrets \
  --from-literal=db-password=supersecret \
  --dry-run=client -o yaml > secret.yaml

# Seal it (fetches public key from controller)
kubeseal --format yaml < secret.yaml > sealed-secret.yaml

# Commit sealed-secret.yaml to Git (safe — encrypted)
git add sealed-secret.yaml
git commit -m "Add sealed secrets for app"

# Apply to cluster (controller decrypts and creates the Secret)
kubectl apply -f sealed-secret.yaml

# The controller creates the real Secret automatically
kubectl get secret app-secrets
```

## External Secrets Operator

ESO syncs secrets from external vaults (AWS Secrets Manager, HashiCorp Vault, GCP Secret Manager, Azure Key Vault) into Kubernetes Secrets.

```bash
# Install via Helm
helm repo add external-secrets https://charts.external-secrets.io
helm install external-secrets external-secrets/external-secrets -n external-secrets-system --create-namespace
```

```yaml
# SecretStore — defines the connection to the external secret store
apiVersion: external-secrets.io/v1beta1
kind: ClusterSecretStore
metadata:
  name: aws-secrets-manager
spec:
  provider:
    aws:
      service: SecretsManager
      region: us-east-1
      auth:
        jwt:
          serviceAccountRef:
            name: external-secrets-sa
            namespace: external-secrets-system
---
# ExternalSecret — specifies which secrets to sync
apiVersion: external-secrets.io/v1beta1
kind: ExternalSecret
metadata:
  name: app-secrets
  namespace: production
spec:
  refreshInterval: 1h # how often to sync
  secretStoreRef:
    name: aws-secrets-manager
    kind: ClusterSecretStore
  target:
    name: app-secrets # name of the K8s Secret to create
    creationPolicy: Owner
    template:
      type: Opaque
      data:
        DB_URL: 'postgresql://{{ .username }}:{{ .password }}@{{ .host }}/{{ .dbname }}'
  data:
    - secretKey: db-password
      remoteRef:
        key: production/myapp/db
        property: password
    - secretKey: db-username
      remoteRef:
        key: production/myapp/db
        property: username
  dataFrom:
    - extract:
        key: production/myapp/config # sync all properties as separate keys
```

## Downward API

Inject Pod metadata (name, namespace, labels, resource limits) into containers without needing a service account.

```yaml
spec:
  containers:
    - name: myapp
      env:
        # Pod metadata
        - name: MY_POD_NAME
          valueFrom:
            fieldRef:
              fieldPath: metadata.name
        - name: MY_POD_NAMESPACE
          valueFrom:
            fieldRef:
              fieldPath: metadata.namespace
        - name: MY_POD_IP
          valueFrom:
            fieldRef:
              fieldPath: status.podIP
        - name: MY_NODE_NAME
          valueFrom:
            fieldRef:
              fieldPath: spec.nodeName
        - name: MY_SERVICE_ACCOUNT
          valueFrom:
            fieldRef:
              fieldPath: spec.serviceAccountName

        # Resource limits as env vars
        - name: MY_CPU_LIMIT
          valueFrom:
            resourceFieldRef:
              resource: limits.cpu
              divisor: 1m # milliCPU
        - name: MY_MEM_LIMIT
          valueFrom:
            resourceFieldRef:
              resource: limits.memory
              divisor: 1Mi

      volumeMounts:
        - name: podinfo
          mountPath: /etc/podinfo

  volumes:
    # Labels and annotations as files (updated live when changed)
    - name: podinfo
      downwardAPI:
        items:
          - path: labels
            fieldRef:
              fieldPath: metadata.labels
          - path: annotations
            fieldRef:
              fieldPath: metadata.annotations
          - path: cpu_limit
            resourceFieldRef:
              containerName: myapp
              resource: limits.cpu
```

## RBAC for Secret Access

```yaml
# Role — allows reading specific secrets in a namespace
apiVersion: rbac.authorization.k8s.io/v1
kind: Role
metadata:
  name: secret-reader
  namespace: production
rules:
  - apiGroups: ['']
    resources: ['secrets']
    resourceNames: ['app-secrets', 'db-credentials'] # specific secrets only
    verbs: ['get', 'list']
---
# RoleBinding — binds role to service account
apiVersion: rbac.authorization.k8s.io/v1
kind: RoleBinding
metadata:
  name: app-secret-binding
  namespace: production
subjects:
  - kind: ServiceAccount
    name: myapp-sa
    namespace: production
roleRef:
  kind: Role
  name: secret-reader
  apiGroup: rbac.authorization.k8s.io
---
# ServiceAccount for the application
apiVersion: v1
kind: ServiceAccount
metadata:
  name: myapp-sa
  namespace: production
automountServiceAccountToken: false # disable default token mount
```

:::tip[Secret Management Best Practices]

1. Enable etcd encryption at rest in production clusters
2. Use External Secrets Operator to source secrets from dedicated vaults
3. Use Sealed Secrets for GitOps workflows
4. Apply RBAC — restrict which service accounts can read which secrets
5. Rotate secrets regularly — ESO refresh intervals make this automatic
6. Never log secret values in application code
7. Use `immutable: true` on Secrets that should never change (prevents accidental modification)
   :::

# ConfigMaps & Secrets

Kubernetes separates application code from configuration using ConfigMaps (non-sensitive) and Secrets (sensitive). Both can be consumed as environment variables or volume-mounted files.

## ConfigMap

A ConfigMap stores non-sensitive key-value configuration data.

### Creating ConfigMaps

```bash
# From literal values
kubectl create configmap app-config \
  --from-literal=NODE_ENV=production \
  --from-literal=PORT=3000 \
  --from-literal=LOG_LEVEL=info

# From a file (key = filename, value = file contents)
kubectl create configmap nginx-conf \
  --from-file=nginx.conf

# From a file with custom key
kubectl create configmap nginx-conf \
  --from-file=default.conf=./nginx/nginx.conf

# From a directory (each file becomes a key)
kubectl create configmap app-configs \
  --from-file=./configs/

# From a .env file
kubectl create configmap app-env \
  --from-env-file=.env.production

# Show resulting ConfigMap
kubectl get configmap app-config -o yaml
```

### Declarative ConfigMap

```yaml
apiVersion: v1
kind: ConfigMap
metadata:
  name: app-config
  namespace: myapp
data:
  # Simple key-value pairs
  NODE_ENV: production
  PORT: "3000"
  LOG_LEVEL: info
  DB_HOST: postgres.myapp.svc.cluster.local
  DB_PORT: "5432"

  # Multi-line values (use | for literal block, > for folded)
  nginx.conf: |
    server {
      listen 80;
      location / {
        proxy_pass http://api:3000;
      }
    }

  # JSON value
  feature-flags.json: |
    {
      "newDashboard": true,
      "betaApi": false
    }
```

### Using ConfigMap as Environment Variables

```yaml
spec:
  containers:
    - name: app
      image: myapp:latest

      # Inject all keys as environment variables
      envFrom:
        - configMapRef:
            name: app-config

      # Inject a specific key
      env:
        - name: LOG_LEVEL
          valueFrom:
            configMapKeyRef:
              name: app-config
              key: LOG_LEVEL
              optional: false   # fail if key missing
```

### Using ConfigMap as a Volume

```yaml
spec:
  containers:
    - name: nginx
      image: nginx:alpine
      volumeMounts:
        # Mount entire ConfigMap as directory
        - name: nginx-config
          mountPath: /etc/nginx/conf.d
          readOnly: true

        # Mount a single key as a specific file path
        - name: app-config
          mountPath: /app/config/features.json
          subPath: feature-flags.json   # mount only this key
          readOnly: true

  volumes:
    - name: nginx-config
      configMap:
        name: nginx-conf
        # Mount specific files with custom names/permissions
        items:
          - key: nginx.conf
            path: default.conf
            mode: 0444

    - name: app-config
      configMap:
        name: app-config
```

:::tip[Volume-mounted ConfigMaps update automatically]
When a ConfigMap is mounted as a volume, Kubernetes updates the file automatically within ~1 minute when the ConfigMap changes (via symlink rotation). Environment variable injection does NOT update automatically — a pod restart is required.
:::

## Secrets

Secrets store sensitive data. Values are base64-encoded (not encrypted by default — enable encryption at rest via KMS).

:::caution[Secrets are not secret by default]
Base64 is encoding, not encryption. Anyone with read access to the Secret can decode it instantly. Restrict access with RBAC. Enable encryption at rest and consider tools like Sealed Secrets or External Secrets Operator for GitOps.
:::

### Secret Types

| Type | Use for |
|---|---|
| `Opaque` (default) | Generic key-value secrets |
| `kubernetes.io/tls` | TLS certificates and private keys |
| `kubernetes.io/dockerconfigjson` | Docker registry credentials |
| `kubernetes.io/service-account-token` | Service account tokens |
| `kubernetes.io/basic-auth` | Basic authentication credentials |
| `kubernetes.io/ssh-auth` | SSH private keys |

### Creating Secrets

```bash
# From literal (base64 encoded automatically)
kubectl create secret generic app-secrets \
  --from-literal=DB_PASSWORD=supersecretpassword \
  --from-literal=JWT_SECRET=my-jwt-signing-key

# From a file
kubectl create secret generic ssl-certs \
  --from-file=tls.crt=./certs/server.crt \
  --from-file=tls.key=./certs/server.key

# TLS secret
kubectl create secret tls api-tls \
  --cert=./certs/tls.crt \
  --key=./certs/tls.key

# Docker registry secret
kubectl create secret docker-registry regcred \
  --docker-server=ghcr.io \
  --docker-username=myuser \
  --docker-password=$GITHUB_TOKEN \
  --docker-email=user@example.com
```

### Declarative Secret

```yaml
apiVersion: v1
kind: Secret
metadata:
  name: app-secrets
  namespace: myapp
type: Opaque
# Values must be base64-encoded
# echo -n "supersecretpassword" | base64
data:
  DB_PASSWORD: c3VwZXJzZWNyZXRwYXNzd29yZA==
  JWT_SECRET: bXktand0LXNpZ25pbmcta2V5

# Alternatively, use stringData (plaintext, auto-encoded)
stringData:
  API_KEY: "my-api-key-value"    # stored as base64 in etcd
```

### Using Secrets

```yaml
spec:
  # Pull image from private registry
  imagePullSecrets:
    - name: regcred

  containers:
    - name: app
      image: ghcr.io/myorg/myapp:latest

      # All keys as environment variables
      envFrom:
        - secretRef:
            name: app-secrets

      # Specific key
      env:
        - name: DB_PASSWORD
          valueFrom:
            secretKeyRef:
              name: app-secrets
              key: DB_PASSWORD

      # Mount as files (most secure — app reads from /run/secrets/)
      volumeMounts:
        - name: secrets
          mountPath: /run/secrets
          readOnly: true

  volumes:
    - name: secrets
      secret:
        secretName: app-secrets
        defaultMode: 0400   # read-only, owner only
        items:
          - key: DB_PASSWORD
            path: db_password
```

## Sealed Secrets (GitOps)

[Bitnami Sealed Secrets](https://github.com/bitnami-labs/sealed-secrets) encrypts Secrets so they can be safely committed to Git.

```bash
# Install the controller
helm repo add sealed-secrets https://bitnami-labs.github.io/sealed-secrets
helm install sealed-secrets sealed-secrets/sealed-secrets -n kube-system

# Install kubeseal CLI
brew install kubeseal

# Seal a regular secret
kubectl create secret generic app-secrets \
  --from-literal=DB_PASSWORD=secret \
  --dry-run=client -o yaml | \
  kubeseal --format yaml > sealed-secret.yaml

# Apply — only the controller can decrypt it
kubectl apply -f sealed-secret.yaml
```

```yaml
# sealed-secret.yaml (safe to commit to Git)
apiVersion: bitnami.com/v1alpha1
kind: SealedSecret
metadata:
  name: app-secrets
  namespace: myapp
spec:
  encryptedData:
    DB_PASSWORD: AgB3m9X2Kv8...  # encrypted ciphertext
```

## External Secrets Operator

[External Secrets Operator (ESO)](https://external-secrets.io) syncs secrets from external stores (AWS SSM, Vault, GCP Secret Manager, Azure Key Vault) into Kubernetes Secrets.

```bash
# Install
helm repo add external-secrets https://charts.external-secrets.io
helm install external-secrets external-secrets/external-secrets -n external-secrets --create-namespace
```

```yaml
# Configure AWS SSM as a secret store
apiVersion: external-secrets.io/v1beta1
kind: SecretStore
metadata:
  name: aws-ssm
  namespace: myapp
spec:
  provider:
    aws:
      service: ParameterStore
      region: us-east-1
      auth:
        jwt:
          serviceAccountRef:
            name: external-secrets-sa

---
# Sync a secret from SSM
apiVersion: external-secrets.io/v1beta1
kind: ExternalSecret
metadata:
  name: app-secrets
  namespace: myapp
spec:
  refreshInterval: 5m
  secretStoreRef:
    name: aws-ssm
    kind: SecretStore
  target:
    name: app-secrets        # name of K8s Secret to create
    creationPolicy: Owner
  data:
    - secretKey: DB_PASSWORD
      remoteRef:
        key: /myapp/production/db_password
    - secretKey: JWT_SECRET
      remoteRef:
        key: /myapp/production/jwt_secret
```

## Downward API

The Downward API exposes Pod metadata and resource information as environment variables or files — without querying the API server.

```yaml
spec:
  containers:
    - name: app
      env:
        # Pod metadata
        - name: POD_NAME
          valueFrom:
            fieldRef:
              fieldPath: metadata.name
        - name: POD_NAMESPACE
          valueFrom:
            fieldRef:
              fieldPath: metadata.namespace
        - name: POD_IP
          valueFrom:
            fieldRef:
              fieldPath: status.podIP
        - name: NODE_NAME
          valueFrom:
            fieldRef:
              fieldPath: spec.nodeName

        # Resource limits (useful for JVM heap sizing, etc.)
        - name: CPU_REQUEST
          valueFrom:
            resourceFieldRef:
              resource: requests.cpu
              divisor: 1m   # in millicores
        - name: MEMORY_LIMIT
          valueFrom:
            resourceFieldRef:
              resource: limits.memory
              divisor: 1Mi  # in mebibytes

      volumeMounts:
        - name: podinfo
          mountPath: /etc/podinfo

  volumes:
    - name: podinfo
      downwardAPI:
        items:
          - path: labels
            fieldRef:
              fieldPath: metadata.labels
          - path: annotations
            fieldRef:
              fieldPath: metadata.annotations
```

## Immutable ConfigMaps and Secrets

Mark as immutable to prevent accidental changes and improve performance (kubelet stops watching for updates):

```yaml
apiVersion: v1
kind: ConfigMap
metadata:
  name: app-config-v1
immutable: true
data:
  version: "1.0.0"

---
apiVersion: v1
kind: Secret
metadata:
  name: app-secrets-v1
immutable: true
data:
  API_KEY: bXlhcGlrZXk=
```

## RBAC for Secret Access

Restrict which ServiceAccounts can read Secrets:

```yaml
# Role — access only specific secrets in one namespace
apiVersion: rbac.authorization.k8s.io/v1
kind: Role
metadata:
  name: secret-reader
  namespace: myapp
rules:
  - apiGroups: [""]
    resources: ["secrets"]
    resourceNames: ["app-secrets"]   # only this specific secret
    verbs: ["get"]

---
apiVersion: rbac.authorization.k8s.io/v1
kind: RoleBinding
metadata:
  name: app-secret-reader
  namespace: myapp
subjects:
  - kind: ServiceAccount
    name: myapp-sa
    namespace: myapp
roleRef:
  kind: Role
  name: secret-reader
  apiGroup: rbac.authorization.k8s.io
```

```bash
# Check what a service account can do
kubectl auth can-i get secrets --as=system:serviceaccount:myapp:myapp-sa -n myapp
```

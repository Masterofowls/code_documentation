# Getting Started with Kubernetes

This guide covers installing kubectl, setting up a local cluster, and mastering the essential commands needed for daily Kubernetes work.

---

## Installing kubectl

<Tabs>
  <TabItem label="macOS">
    ```bash
    # Homebrew (recommended)
    brew install kubectl

    # Verify
    kubectl version --client
    ```
  </TabItem>
  <TabItem label="Linux">
    ```bash
    # Download the latest stable release
    curl -LO "https://dl.k8s.io/release/$(curl -sL https://dl.k8s.io/release/stable.txt)/bin/linux/amd64/kubectl"

    # Make executable and move to PATH
    chmod +x kubectl
    sudo mv kubectl /usr/local/bin/

    # Verify
    kubectl version --client
    ```
  </TabItem>
  <TabItem label="Windows">
    ```bash
    # Chocolatey
    choco install kubernetes-cli

    # Winget
    winget install Kubernetes.kubectl

    # Verify
    kubectl version --client
    ```
  </TabItem>
</Tabs>

## Local Cluster Options

| Tool | Nodes | Speed | Best For |
|---|---|---|---|
| **minikube** | Single | Medium | Local dev, beginners |
| **kind** | Multi | Fast | CI, multi-node testing |
| **k3d** | Multi | Very fast | Dev, multi-node, ARM |
| **Docker Desktop K8s** | Single | Slow startup | macOS/Windows convenience |

### minikube

```bash
# Install
brew install minikube      # macOS
choco install minikube     # Windows

# Start a cluster
minikube start

# Start with specific resources
minikube start --cpus=4 --memory=8192 --driver=docker

# Enable addons
minikube addons enable ingress
minikube addons enable metrics-server
minikube addons enable dashboard

# Open dashboard
minikube dashboard

# Stop cluster
minikube stop

# Delete cluster
minikube delete
```

### kind (Kubernetes in Docker)

```bash
# Install
brew install kind

# Create a basic cluster
kind create cluster

# Create with a config (multi-node)
cat <<EOF > kind-config.yaml
kind: Cluster
apiVersion: kind.x-k8s.io/v1alpha4
nodes:
  - role: control-plane
  - role: worker
  - role: worker
EOF
kind create cluster --config kind-config.yaml --name dev

# List clusters
kind get clusters

# Load a local image into the cluster
kind load docker-image myapp:latest --name dev

# Delete
kind delete cluster --name dev
```

### k3d

```bash
# Install
brew install k3d

# Create a cluster
k3d cluster create dev \
  --agents 2 \
  --port "80:80@loadbalancer" \
  --port "443:443@loadbalancer"

# Import a local image
k3d image import myapp:latest --cluster dev

# Delete
k3d cluster delete dev
```

## kubeconfig and Contexts

kubectl uses a `kubeconfig` file (`~/.kube/config` by default) to store cluster credentials and connection info.

```bash
# View current context
kubectl config current-context

# List all contexts
kubectl config get-contexts

# Switch to a different context
kubectl config use-context my-cluster

# Set a default namespace for the current context
kubectl config set-context --current --namespace myapp

# View full kubeconfig
kubectl config view

# Merge multiple kubeconfig files
KUBECONFIG=~/.kube/config:~/.kube/another-config \
  kubectl config view --flatten > ~/.kube/merged-config

# Use a specific kubeconfig file
kubectl --kubeconfig ~/.kube/staging-config get pods
KUBECONFIG=~/.kube/staging get pods
```

## Essential kubectl Commands

### Getting Resources

```bash
# Get Pods (current namespace)
kubectl get pods

# Get Pods in all namespaces
kubectl get pods -A
kubectl get pods --all-namespaces

# Get Pods in a specific namespace
kubectl get pods -n kube-system

# Wide output (more columns including node name and IP)
kubectl get pods -o wide

# YAML output for a resource
kubectl get pod my-pod -o yaml

# JSON output
kubectl get pod my-pod -o json

# Custom columns
kubectl get pods -o custom-columns="NAME:.metadata.name,STATUS:.status.phase"

# Watch for changes in real time
kubectl get pods -w

# Get all common resources at once
kubectl get all

# Get a specific resource by name
kubectl get deployment my-deployment
```

### Describing Resources

```bash
# Describe a Pod (events, state, conditions)
kubectl describe pod my-pod

# Describe a Deployment
kubectl describe deployment my-deployment

# Describe a Node
kubectl describe node my-node-name
```

### Creating and Applying Resources

```bash
# Apply a manifest file (create or update — idempotent)
kubectl apply -f deployment.yaml

# Apply all manifests in a directory
kubectl apply -f ./manifests/

# Apply from a URL
kubectl apply -f https://raw.githubusercontent.com/org/repo/main/manifest.yaml

# Create a resource (fails if already exists)
kubectl create -f pod.yaml

# Preview changes before applying
kubectl diff -f deployment.yaml

# Dry run — validate without applying
kubectl apply -f deployment.yaml --dry-run=client

# Server-side dry run (more accurate)
kubectl apply -f deployment.yaml --dry-run=server
```

### Deleting Resources

```bash
# Delete by manifest
kubectl delete -f deployment.yaml

# Delete a specific resource
kubectl delete pod my-pod
kubectl delete deployment my-deployment

# Delete all Pods in a namespace
kubectl delete pods --all -n my-namespace

# Force delete a stuck Pod
kubectl delete pod my-pod --grace-period=0 --force
```

### Logs

```bash
# Pod logs
kubectl logs my-pod

# Follow logs in real time
kubectl logs -f my-pod

# Logs for a specific container in a multi-container Pod
kubectl logs my-pod -c my-container

# Logs for all Pods matching a label
kubectl logs -l app=myapp

# Last 50 lines
kubectl logs --tail=50 my-pod

# Logs since 1 hour ago
kubectl logs --since=1h my-pod

# Previous container instance (useful after crash)
kubectl logs my-pod --previous
```

### Executing Commands

```bash
# Open an interactive shell in a Pod
kubectl exec -it my-pod -- sh

# Exec with bash
kubectl exec -it my-pod -- bash

# Run a single command
kubectl exec my-pod -- ls /app

# Exec in a specific container
kubectl exec -it my-pod -c sidecar -- sh
```

### Port Forwarding

```bash
# Forward local port 8080 to Pod port 80
kubectl port-forward pod/my-pod 8080:80

# Forward to a Service
kubectl port-forward svc/my-service 8080:80

# Forward to a Deployment (picks one Pod)
kubectl port-forward deployment/my-deployment 8080:80

# Bind to all interfaces (accessible from network)
kubectl port-forward pod/my-pod 8080:80 --address=0.0.0.0
```

### Rollouts

```bash
# Check rollout status
kubectl rollout status deployment/my-deployment

# View rollout history
kubectl rollout history deployment/my-deployment

# Roll back to previous version
kubectl rollout undo deployment/my-deployment

# Roll back to a specific revision
kubectl rollout undo deployment/my-deployment --to-revision=2

# Pause a rolling update
kubectl rollout pause deployment/my-deployment

# Resume a paused rollout
kubectl rollout resume deployment/my-deployment
```

### Scaling

```bash
# Scale a Deployment to 5 replicas
kubectl scale deployment my-deployment --replicas=5

# Scale a StatefulSet
kubectl scale statefulset my-statefulset --replicas=3
```

### Resource Usage (requires metrics-server)

```bash
# Node resource usage
kubectl top nodes

# Pod resource usage
kubectl top pods

# Pod resource usage with containers
kubectl top pods --containers

# Pod resource in a namespace
kubectl top pods -n production
```

## Namespaces

Namespaces provide logical isolation within a cluster.

```bash
# List namespaces
kubectl get namespaces
kubectl get ns

# Create a namespace
kubectl create namespace staging
kubectl create ns staging

# Apply a manifest in a specific namespace
kubectl apply -f app.yaml -n staging

# Get resources in a namespace
kubectl get pods -n staging

# Get resources across ALL namespaces
kubectl get pods -A

# Set default namespace for current context
kubectl config set-context --current --namespace staging

# Delete a namespace (DELETES ALL RESOURCES IN IT)
kubectl delete namespace staging
```

## Imperative vs Declarative

Kubernetes supports both imperative (direct commands) and declarative (manifest files) approaches.

```bash
# ── IMPERATIVE ────────────────────────────────────────────────────────────
# Create a Deployment directly
kubectl create deployment myapp --image=myapp:1.2.3 --replicas=3

# Expose it as a Service
kubectl expose deployment myapp --port=80 --target-port=3000

# Set a new image (update)
kubectl set image deployment/myapp myapp=myapp:1.2.4

# ── DECLARATIVE ───────────────────────────────────────────────────────────
# Apply a manifest — idempotent, tracked in cluster
kubectl apply -f deployment.yaml
```

:::tip[Always use declarative in production]
Declarative manifests (`kubectl apply -f`) stored in Git are the foundation of GitOps. They are auditable, reviewable, and repeatable. Use imperative commands only for quick experiments and debugging.
:::

## kubectl explain

`kubectl explain` is the built-in API documentation tool.

```bash
# Explain a resource type
kubectl explain deployment

# Explain a specific field
kubectl explain deployment.spec.strategy

# Explain recursively (all fields)
kubectl explain deployment --recursive

# Explain a Pod container field
kubectl explain pod.spec.containers.resources
```

## kubectl api-resources

```bash
# List all available API resource types
kubectl api-resources

# List only namespaced resources
kubectl api-resources --namespaced=true

# List resources for a specific API group
kubectl api-resources --api-group=apps

# Find the short names
kubectl api-resources | grep -E "^NAME|deploy|svc|ing|cm|secret"
```

## Quick Debugging Workflow

```bash
# 1. Check Pod status
kubectl get pods -n myapp

# 2. If CrashLoopBackOff or Error, check logs
kubectl logs -f my-pod-abc123 -n myapp
kubectl logs my-pod-abc123 --previous -n myapp  # Previous crash

# 3. Describe for events and conditions
kubectl describe pod my-pod-abc123 -n myapp

# 4. Exec into a running pod
kubectl exec -it my-pod-abc123 -n myapp -- sh

# 5. Check resource limits
kubectl top pod my-pod-abc123 -n myapp

# 6. Check events in namespace
kubectl get events -n myapp --sort-by='.lastTimestamp'
```

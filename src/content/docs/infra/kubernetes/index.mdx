# Kubernetes

Kubernetes (K8s) is an open-source container orchestration platform that automates deploying, scaling, and managing containerized applications. Originally developed by Google and donated to the CNCF, it is now the industry standard for running containers at scale.

---

## The Container Orchestration Problem

Running a single container on one machine is simple. Running **hundreds of containers across dozens of machines** in production is a different challenge entirely:

- How do you ensure containers restart if they crash?
- How do you roll out new versions with zero downtime?
- How do you scale up under load and scale down to save costs?
- How do you route traffic to healthy containers?
- How do you distribute containers across machines to use resources efficiently?
- How do you manage configuration and secrets across environments?

Kubernetes solves all of these problems.

## Kubernetes Architecture

Kubernetes follows a **control plane / data plane** architecture.

```
┌────────────────────────────────────────────────────────────────────┐
│                        CONTROL PLANE                               │
│                                                                    │
│  ┌──────────────┐  ┌─────────────┐  ┌────────────────────────┐   │
│  │  API Server  │  │    etcd     │  │  Controller Manager    │   │
│  │(kube-apiserver)│ │ (key-value │  │  (node, replication,   │   │
│  │              │  │  store)     │  │   endpoints, etc.)     │   │
│  └──────┬───────┘  └─────────────┘  └────────────────────────┘   │
│         │                                                          │
│  ┌──────┴──────────────┐                                           │
│  │     Scheduler       │                                           │
│  │  (kube-scheduler)   │                                           │
│  └─────────────────────┘                                           │
└────────────────────────────────────────────────────────────────────┘
           │
           │ kubelet (watches API server)
           ▼
┌────────────────────────────────────────────────────────────────────┐
│                    DATA PLANE (Worker Nodes)                        │
│                                                                    │
│  ┌──────────────────────────────────────────────────────────────┐ │
│  │                       Node 1                                  │ │
│  │  ┌──────────┐  ┌─────────────────────────────────────────┐  │ │
│  │  │  kubelet │  │              kube-proxy                  │  │ │
│  │  └──────────┘  └─────────────────────────────────────────┘  │ │
│  │                                                               │ │
│  │  ┌─────────────────────┐  ┌─────────────────────────────┐   │ │
│  │  │  Pod (containers)   │  │     Pod (containers)        │   │ │
│  │  └─────────────────────┘  └─────────────────────────────┘   │ │
│  └──────────────────────────────────────────────────────────────┘ │
└────────────────────────────────────────────────────────────────────┘
```

### Control Plane Components

**kube-apiserver**
The front end to the Kubernetes control plane. All communication goes through the API server — `kubectl`, controllers, kubelets, and external tools all talk to it via REST. It validates and processes requests, then persists state to etcd.

**etcd**
A distributed, highly available key-value store used as Kubernetes' backing store for all cluster data. Think of it as the cluster's source of truth. Back it up regularly.

**kube-scheduler**
Watches for newly created Pods with no assigned node and selects the best node to run them on, based on resource requirements, affinity rules, taints/tolerations, and other constraints.

**kube-controller-manager**
Runs controller processes in a single binary. Each controller is a control loop that watches cluster state and makes changes to move the current state toward the desired state. Examples: Node controller, Replication controller, Endpoints controller, Service Account controller.

**cloud-controller-manager**
Embeds cloud-specific control logic. Lets you link your cluster into your cloud provider's API (for LoadBalancer services, PersistentVolumes via cloud disks, etc.).

### Data Plane (Worker Node) Components

**kubelet**
An agent that runs on every worker node. It watches the API server for Pods assigned to its node and ensures they are running and healthy. It talks to the container runtime (containerd/CRI-O) to manage containers.

**kube-proxy**
Maintains network rules on nodes that implement the Kubernetes Service concept. It enables communication to Pods from inside or outside the cluster using iptables or IPVS rules.

**Container Runtime**
The software responsible for running containers. Kubernetes supports any CRI-compliant runtime: `containerd` (default), `CRI-O`, or others.

## Managed Kubernetes Services

Running your own control plane is complex and risky. Managed services handle the control plane for you.

| Service | Provider | Notes |
|---|---|---|
| **EKS** | AWS | Elastic Kubernetes Service. Tight AWS integration (IAM, ALB, EBS, EFS). |
| **GKE** | Google Cloud | Google Kubernetes Engine. Best managed experience; Autopilot mode available. |
| **AKS** | Azure | Azure Kubernetes Service. Free control plane; strong Windows container support. |
| **DOKS** | DigitalOcean | Simple, affordable. Good for smaller workloads. |
| **k3s** | Rancher/SUSE | Lightweight K8s. Runs on edge, IoT, Raspberry Pi. |
| **OpenShift** | Red Hat | Enterprise K8s platform with built-in CI/CD and developer tools. |

:::tip[Start with managed]
Unless you have specific on-prem requirements or enjoy infrastructure operations, use a managed service. The control plane is not where you add value — your applications are.
:::

## Kubernetes vs Alternatives

| Feature | Kubernetes | Docker Swarm | HashiCorp Nomad |
|---|---|---|---|
| **Maturity** | Very high | Stable | High |
| **Learning curve** | Steep | Low | Medium |
| **Ecosystem** | Enormous (CNCF) | Small | Medium |
| **Multi-tenancy** | Excellent (namespaces, RBAC) | Basic | Good |
| **Autoscaling** | HPA, VPA, KEDA, Cluster Autoscaler | Manual | Autoscaler plugin |
| **Storage** | Rich (CSI, many drivers) | Basic | Good |
| **Non-container workloads** | Pods only | Containers only | VMs, binaries, containers |
| **Community** | Massive | Shrinking | Active |
| **Managed offering** | EKS, GKE, AKS, etc. | None cloud-managed | HCP Nomad |
| **Config complexity** | High (YAML-heavy) | Low | Medium |

## Core Kubernetes Concepts

A brief overview of the objects you'll encounter most often:

**Pod**
The smallest deployable unit. A Pod wraps one or more containers that share network and storage. You rarely create Pods directly — controllers manage them.

**Deployment**
Manages a replicated set of Pods. Handles rolling updates, rollbacks, and scaling. The most common way to run stateless applications.

**Service**
A stable network endpoint for a set of Pods. Provides load balancing and DNS-based discovery. Types: ClusterIP, NodePort, LoadBalancer.

**Ingress**
Manages external HTTP/HTTPS access to Services. Defines routing rules based on hostname and path. Requires an Ingress Controller (nginx-ingress, Traefik, etc.).

**ConfigMap**
Stores non-sensitive configuration data as key-value pairs. Can be injected into Pods as environment variables or mounted as files.

**Secret**
Stores sensitive data (passwords, tokens, keys) in base64-encoded form. Can be injected as environment variables or volumes.

**PersistentVolumeClaim (PVC)**
A request for storage. Kubernetes provisions a PersistentVolume matching the claim's requirements (size, access mode, StorageClass).

**Namespace**
A mechanism to divide cluster resources between multiple users or teams. Provides scope for names, resource quotas, and RBAC policies.

**StatefulSet**
Like a Deployment, but for stateful applications. Provides stable network identities and persistent storage per replica. Used for databases, message queues, etc.

**DaemonSet**
Ensures a copy of a Pod runs on every (or selected) node. Used for log collectors, monitoring agents, network plugins.

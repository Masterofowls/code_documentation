# Kubernetes

Kubernetes (K8s) is an open-source container orchestration system that automates the deployment, scaling, and management of containerised applications. Originally designed by Google, it is now maintained by the Cloud Native Computing Foundation (CNCF).

## The Container Orchestration Problem

Running a single container on a single server is easy. Production workloads need much more:

- **High availability** — if a server crashes, containers must restart elsewhere
- **Scaling** — automatically add instances when traffic spikes, remove them when it drops
- **Rolling updates** — deploy new versions without downtime
- **Service discovery** — containers need to find each other without hardcoded IPs
- **Load balancing** — distribute traffic across multiple container instances
- **Secret management** — inject credentials without baking them into images
- **Storage orchestration** — attach persistent storage from cloud providers or NFS
- **Self-healing** — restart failed containers, replace unresponsive ones
- **Resource management** — schedule containers to nodes with sufficient CPU and memory

Kubernetes solves all of these problems with a declarative API: you describe the **desired state** and Kubernetes continuously reconciles actual state to match it.

### Control Plane Components

**kube-apiserver**
The front door to Kubernetes. All communication — from `kubectl`, from nodes, from controllers — goes through the REST API server. It validates and processes API objects, and is the only component that talks to etcd.

**etcd**
A distributed, strongly consistent key-value store. The single source of truth for all cluster state: what nodes exist, what pods are running, what ConfigMaps and Secrets contain. Always run etcd in HA (3 or 5 nodes) in production.

**kube-scheduler**
Watches for newly created Pods with no assigned node and selects a node for them to run on. Considers: resource requirements, hardware/software constraints, affinity rules, taints and tolerations, data locality.

**kube-controller-manager**
Runs controller loops that watch state and take action. Includes: Node Controller (notices when nodes go down), Replication Controller (maintains correct pod count), Endpoints Controller, Service Account Controller, and many more.

### Node Components

**kubelet**
An agent on every node. Receives PodSpec (from the API server) and ensures the described containers are running and healthy. Communicates pod status back to the control plane.

**kube-proxy**
A network proxy on every node. Implements the Service concept by maintaining iptables/ipvs rules to route traffic to the correct pods. Handles load balancing across pod endpoints.

**Container runtime**
The software that actually runs containers. Kubernetes supports any OCI-compliant runtime: containerd (most common), CRI-O, Docker (via cri-dockerd shim).

## Managed Kubernetes Services

| Service       | Provider     | Notes                                                              |
| ------------- | ------------ | ------------------------------------------------------------------ |
| **EKS**       | AWS          | EC2 or Fargate worker nodes; integrates with IAM, ALB, EBS         |
| **GKE**       | Google Cloud | Autopilot mode for fully managed nodes; best Kubernetes experience |
| **AKS**       | Azure        | Deep Azure AD integration; free control plane                      |
| **DOKS**      | DigitalOcean | Simple, affordable, good for smaller workloads                     |
| **k3s**       | Rancher/SUSE | Lightweight K8s, great for edge and on-prem                        |
| **k0s**       | Mirantis     | Single binary, no external dependencies                            |
| **OpenShift** | Red Hat      | Enterprise K8s with additional developer tools                     |

## Kubernetes vs Docker Swarm vs Nomad

| Feature                     | Kubernetes                      | Docker Swarm     | Nomad                    |
| --------------------------- | ------------------------------- | ---------------- | ------------------------ |
| **Complexity**              | High                            | Low              | Medium                   |
| **Ecosystem**               | Massive (CNCF)                  | Small            | Medium (HashiCorp)       |
| **Scaling**                 | Sophisticated (HPA, VPA, KEDA)  | Basic            | Good                     |
| **Networking**              | CNI plugins (Calico, Cilium...) | Built-in overlay | Consul integration       |
| **Storage**                 | PV/PVC/CSI                      | Volume plugins   | Task volumes             |
| **Learning curve**          | Steep                           | Gentle           | Moderate                 |
| **Non-container workloads** | Pods only                       | Docker only      | VMs, JARs, binaries      |
| **Production readiness**    | Industry standard               | Limited use      | Good for mixed workloads |
| **GPU support**             | Yes (device plugins)            | Limited          | Yes                      |

## Core Concepts Overview

| Concept                   | Purpose                                                                               |
| ------------------------- | ------------------------------------------------------------------------------------- |
| **Pod**                   | Smallest deployable unit; one or more containers sharing network and storage          |
| **Deployment**            | Manages ReplicaSets; handles rolling updates and rollbacks                            |
| **StatefulSet**           | Like Deployment but for stateful apps; stable network identity and ordered deployment |
| **DaemonSet**             | Runs exactly one pod per node (logging agents, monitoring)                            |
| **Service**               | Stable IP/DNS endpoint for a set of pods; load balances traffic                       |
| **Ingress**               | HTTP(S) routing rules; exposes services to the internet                               |
| **ConfigMap**             | Non-sensitive configuration data; injected as env vars or files                       |
| **Secret**                | Sensitive data (passwords, tokens); base64 encoded in etcd                            |
| **PersistentVolumeClaim** | Request for persistent storage; bound to a PersistentVolume                           |
| **Namespace**             | Virtual cluster; isolates resources within the same physical cluster                  |
| **ServiceAccount**        | Identity for pods; controls API and resource access                                   |
| **RBAC**                  | Role-Based Access Control; who can do what with which resources                       |

## Quick Reference

```bash
# Get everything in a namespace
kubectl get all -n myapp

# Apply a manifest
kubectl apply -f deployment.yaml

# Get pod logs
kubectl logs -f deployment/myapp

# Shell into a pod
kubectl exec -it pod/myapp-abc123 -- sh

# Port forward to local machine
kubectl port-forward svc/myapp 8080:80

# Scale a deployment
kubectl scale deployment/myapp --replicas=5

# Watch rolling update
kubectl rollout status deployment/myapp

# Roll back
kubectl rollout undo deployment/myapp
```

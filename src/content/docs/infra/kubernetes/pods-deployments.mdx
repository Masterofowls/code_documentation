# Pods & Deployments

Pods and Deployments are the foundation of running workloads in Kubernetes. Understanding their spec, lifecycle, and update mechanisms is essential for operating applications reliably.

---

## Pods

A **Pod** is the smallest deployable unit in Kubernetes. It wraps one or more containers that share the same network namespace and storage volumes.

### Basic Pod Spec

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: my-pod
  namespace: default
  labels:
    app: my-app
    version: "1.0"
  annotations:
    description: "My application pod"
spec:
  containers:
    - name: app
      image: my-app:1.0.0
      ports:
        - name: http
          containerPort: 3000
          protocol: TCP

      # Environment variables
      env:
        - name: NODE_ENV
          value: production
        - name: DB_PASSWORD
          valueFrom:
            secretKeyRef:
              name: db-secret
              key: password
        - name: POD_NAME
          valueFrom:
            fieldRef:
              fieldPath: metadata.name

      # Environment from ConfigMap or Secret
      envFrom:
        - configMapRef:
            name: app-config
        - secretRef:
            name: app-secrets
            optional: true

      # Resource requests and limits
      resources:
        requests:
          memory: "128Mi"
          cpu: "100m"
        limits:
          memory: "512Mi"
          cpu: "500m"

      # Volume mounts
      volumeMounts:
        - name: app-data
          mountPath: /app/data
        - name: config-vol
          mountPath: /app/config
          readOnly: true
        - name: tmp
          mountPath: /tmp

  volumes:
    - name: app-data
      persistentVolumeClaim:
        claimName: my-pvc
    - name: config-vol
      configMap:
        name: app-config
    - name: tmp
      emptyDir: {}

  restartPolicy: Always          # Always | OnFailure | Never
  terminationGracePeriodSeconds: 30
```

## Probes

Kubernetes uses probes to determine the health state of containers.

| Probe | Purpose | Failure Action |
|---|---|---|
| `livenessProbe` | Is the container alive? | Restart container |
| `readinessProbe` | Is the container ready to serve traffic? | Remove from Service endpoints |
| `startupProbe` | Has the container finished starting? | Disable liveness/readiness until passed |

### Probe Types

```yaml
# HTTP GET probe
livenessProbe:
  httpGet:
    path: /health
    port: 3000
    httpHeaders:
      - name: X-Health-Check
        value: "1"
  initialDelaySeconds: 10
  periodSeconds: 30
  timeoutSeconds: 5
  failureThreshold: 3
  successThreshold: 1

# TCP Socket probe
readinessProbe:
  tcpSocket:
    port: 5432
  initialDelaySeconds: 5
  periodSeconds: 10
  failureThreshold: 3

# Exec probe (command exit code 0 = success)
livenessProbe:
  exec:
    command:
      - sh
      - -c
      - "pg_isready -U postgres || exit 1"
  initialDelaySeconds: 30
  periodSeconds: 20

# gRPC probe (K8s 1.24+)
livenessProbe:
  grpc:
    port: 5000
    service: "liveness"
  periodSeconds: 10
```

### Startup Probe for Slow Applications

```yaml
# Use startupProbe for apps that take a long time to start
# Liveness/Readiness probes are disabled until startupProbe succeeds
startupProbe:
  httpGet:
    path: /health
    port: 3000
  failureThreshold: 30      # Allow up to 30 * 10s = 5 minutes to start
  periodSeconds: 10

livenessProbe:
  httpGet:
    path: /health
    port: 3000
  periodSeconds: 30
  failureThreshold: 3
```

## Pod Lifecycle

```
                  ┌─────────┐
      Pending ──► │ Running │ ──► Succeeded
                  └─────────┘ ──► Failed
                       │
                   (Unknown — node communication lost)
```

| Phase | Description |
|---|---|
| `Pending` | Pod accepted by cluster; containers not yet running (scheduling or image pull) |
| `Running` | At least one container is running or starting |
| `Succeeded` | All containers exited with code 0 (common for Jobs) |
| `Failed` | At least one container exited non-zero and won't restart |
| `Unknown` | Pod state cannot be obtained (node issues) |

```bash
# Check pod phase
kubectl get pod my-pod -o jsonpath='{.status.phase}'

# Check container states
kubectl describe pod my-pod | grep -A 5 "State:"

# Check events for crash diagnosis
kubectl describe pod crashing-pod | grep -A 20 Events
```

### Container States

| State | Description |
|---|---|
| `Waiting` | Not yet running (pulling image, pending resource) |
| `Running` | Executing normally |
| `Terminated` | Finished execution (check exit code) |

### RestartPolicy

| Policy | Use With | Behavior |
|---|---|---|
| `Always` | Deployments, StatefulSets, DaemonSets | Restart on any exit |
| `OnFailure` | Jobs | Restart only on non-zero exit |
| `Never` | One-shot tasks | Never restart |

## Init Containers

Init containers run **before** the main containers start. They must complete successfully before the Pod proceeds.

```yaml
spec:
  initContainers:
    - name: wait-for-db
      image: busybox:1.35
      command:
        - sh
        - -c
        - |
          until nc -z db 5432; do
            echo "Waiting for database..."
            sleep 2
          done

    - name: run-migrations
      image: my-app:1.0.0
      command: ["npm", "run", "db:migrate"]
      env:
        - name: DATABASE_URL
          valueFrom:
            secretKeyRef:
              name: db-secret
              key: url

  containers:
    - name: app
      image: my-app:1.0.0
      # Main app only starts after both init containers succeed
```

## Sidecar Containers (K8s 1.29+)

Sidecar containers run alongside the main container and have their own lifecycle management.

```yaml
spec:
  initContainers:
    - name: log-collector
      image: fluent/fluent-bit:latest
      restartPolicy: Always     # Makes this a sidecar (K8s 1.29+)
      volumeMounts:
        - name: log-volume
          mountPath: /logs

  containers:
    - name: app
      image: my-app:1.0.0
      volumeMounts:
        - name: log-volume
          mountPath: /app/logs
```

## Deployments

A **Deployment** provides declarative updates for Pods. It manages a ReplicaSet under the hood, which manages the desired number of Pod replicas.

### Deployment Spec

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: my-app
  namespace: production
  labels:
    app: my-app
spec:
  replicas: 3

  # Selector must match pod template labels
  selector:
    matchLabels:
      app: my-app

  # Update strategy
  strategy:
    type: RollingUpdate
    rollingUpdate:
      maxSurge: 1           # Extra pods during update (can be %)
      maxUnavailable: 0     # Never go below desired replicas

  # How long to wait before considering pod available
  minReadySeconds: 10

  # Number of old ReplicaSets to retain for rollback
  revisionHistoryLimit: 10

  template:
    metadata:
      labels:
        app: my-app
        version: "1.2.3"
    spec:
      containers:
        - name: app
          image: my-app:1.2.3
          ports:
            - containerPort: 3000
          resources:
            requests:
              cpu: "100m"
              memory: "128Mi"
            limits:
              cpu: "500m"
              memory: "512Mi"
          readinessProbe:
            httpGet:
              path: /ready
              port: 3000
            initialDelaySeconds: 5
            periodSeconds: 5
          livenessProbe:
            httpGet:
              path: /health
              port: 3000
            initialDelaySeconds: 15
            periodSeconds: 30
```

### Update Strategy

**RollingUpdate** (default): Gradually replaces old Pods with new ones. Zero downtime.

```yaml
strategy:
  type: RollingUpdate
  rollingUpdate:
    maxSurge: 1          # Allow 1 extra pod during rollout
    maxUnavailable: 0    # Keep all desired pods available (slower but safer)
```

**Recreate**: Kills all existing Pods before creating new ones. Has downtime.

```yaml
strategy:
  type: Recreate
```

### Rollout Commands

```bash
# Check rollout status (wait for completion)
kubectl rollout status deployment/my-app

# View rollout history
kubectl rollout history deployment/my-app

# View specific revision details
kubectl rollout history deployment/my-app --revision=3

# Undo the last rollout
kubectl rollout undo deployment/my-app

# Undo to a specific revision
kubectl rollout undo deployment/my-app --to-revision=2

# Pause rollout (useful for canary-style manual control)
kubectl rollout pause deployment/my-app

# Resume paused rollout
kubectl rollout resume deployment/my-app

# Trigger a rollout without changing the image (force restart)
kubectl rollout restart deployment/my-app

# Watch rollout in real time
kubectl get pods -w --selector app=my-app
```

## StatefulSets

StatefulSets are for stateful applications that need **stable network identity** and **persistent storage** per replica.

```yaml
apiVersion: apps/v1
kind: StatefulSet
metadata:
  name: postgres
spec:
  serviceName: postgres          # Headless Service name (required)
  replicas: 3
  selector:
    matchLabels:
      app: postgres
  template:
    metadata:
      labels:
        app: postgres
    spec:
      containers:
        - name: postgres
          image: postgres:16-alpine
          env:
            - name: POSTGRES_PASSWORD
              valueFrom:
                secretKeyRef:
                  name: postgres-secret
                  key: password
          volumeMounts:
            - name: data
              mountPath: /var/lib/postgresql/data
  # Each replica gets its own PVC (not shared)
  volumeClaimTemplates:
    - metadata:
        name: data
      spec:
        accessModes: ["ReadWriteOnce"]
        storageClassName: fast-ssd
        resources:
          requests:
            storage: 10Gi
```

**StatefulSet stable identities:**
- Pod names: `postgres-0`, `postgres-1`, `postgres-2`
- DNS: `postgres-0.postgres.namespace.svc.cluster.local`
- Scaling: Pods are created in order (0, 1, 2) and deleted in reverse

## DaemonSets

A DaemonSet ensures a copy of a Pod runs on **every node** (or a subset).

```yaml
apiVersion: apps/v1
kind: DaemonSet
metadata:
  name: log-collector
  namespace: kube-system
spec:
  selector:
    matchLabels:
      app: log-collector
  template:
    metadata:
      labels:
        app: log-collector
    spec:
      tolerations:
        - key: node-role.kubernetes.io/control-plane
          effect: NoSchedule           # Also run on control-plane nodes
      containers:
        - name: fluent-bit
          image: fluent/fluent-bit:2.2
          volumeMounts:
            - name: varlog
              mountPath: /var/log
              readOnly: true
      volumes:
        - name: varlog
          hostPath:
            path: /var/log
```

Common DaemonSet uses: log collectors, metrics agents, network plugins, storage drivers.

## Jobs and CronJobs

### Job

A Job creates one or more Pods and retries until a specified number of successful completions.

```yaml
apiVersion: batch/v1
kind: Job
metadata:
  name: db-migration
spec:
  completions: 1              # Number of successful completions needed
  parallelism: 1              # Run N pods in parallel
  backoffLimit: 3             # Max retries before marking Job as failed
  activeDeadlineSeconds: 300  # Kill Job after 5 minutes
  ttlSecondsAfterFinished: 3600  # Auto-delete 1 hour after completion

  template:
    spec:
      restartPolicy: OnFailure     # Required for Jobs (not Always)
      containers:
        - name: migration
          image: my-app:1.0.0
          command: ["npm", "run", "db:migrate"]
          env:
            - name: DATABASE_URL
              valueFrom:
                secretKeyRef:
                  name: db-secret
                  key: url
```

### CronJob

```yaml
apiVersion: batch/v1
kind: CronJob
metadata:
  name: backup
spec:
  schedule: "0 2 * * *"            # Every day at 2:00 AM (cron syntax)
  timeZone: "America/New_York"      # K8s 1.27+
  concurrencyPolicy: Forbid         # Allow | Forbid | Replace
  successfulJobsHistoryLimit: 3
  failedJobsHistoryLimit: 1
  startingDeadlineSeconds: 60       # Skip if missed by >60s
  jobTemplate:
    spec:
      template:
        spec:
          restartPolicy: OnFailure
          containers:
            - name: backup
              image: my-backup-tool:latest
              command: ["/bin/sh", "-c", "backup.sh"]
```

```bash
# Manually trigger a CronJob
kubectl create job --from=cronjob/backup manual-backup-$(date +%Y%m%d)

# Check Job status
kubectl get jobs
kubectl describe job db-migration

# Check CronJob
kubectl get cronjobs
kubectl describe cronjob backup
```

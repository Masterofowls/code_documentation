# Pods & Deployments

Pods are the smallest deployable unit in Kubernetes. Deployments, StatefulSets, and DaemonSets are higher-level workload controllers that manage pods.

## Pod Specification

A Pod runs one or more containers that share a network namespace, hostname, and optionally storage volumes.

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: my-pod
  namespace: myapp
  labels:
    app: myapp
    version: "1.2.3"
  annotations:
    prometheus.io/scrape: "true"
    prometheus.io/port: "9090"
spec:
  containers:
    - name: app
      image: myapp/api:1.2.3
      imagePullPolicy: IfNotPresent   # Always | Never | IfNotPresent

      # ── Ports ────────────────────────────────────────────────
      ports:
        - name: http
          containerPort: 3000
          protocol: TCP

      # ── Environment Variables ─────────────────────────────────
      env:
        - name: PORT
          value: "3000"
        - name: NODE_ENV
          value: production
        - name: DB_HOST
          valueFrom:
            configMapKeyRef:
              name: app-config
              key: db_host
        - name: DB_PASSWORD
          valueFrom:
            secretKeyRef:
              name: app-secrets
              key: db_password

      # ── Env from ConfigMap / Secret ────────────────────────────
      envFrom:
        - configMapRef:
            name: app-config
        - secretRef:
            name: app-secrets

      # ── Resources ─────────────────────────────────────────────
      resources:
        requests:
          cpu: 100m       # 100 millicores = 0.1 CPU
          memory: 128Mi
        limits:
          cpu: 500m
          memory: 512Mi

      # ── Volume Mounts ──────────────────────────────────────────
      volumeMounts:
        - name: config
          mountPath: /app/config
          readOnly: true
        - name: tmp
          mountPath: /tmp

      # ── Probes ────────────────────────────────────────────────
      livenessProbe:
        httpGet:
          path: /health/live
          port: 3000
        initialDelaySeconds: 30
        periodSeconds: 15
        timeoutSeconds: 5
        failureThreshold: 3

      readinessProbe:
        httpGet:
          path: /health/ready
          port: 3000
        initialDelaySeconds: 10
        periodSeconds: 10
        timeoutSeconds: 3
        failureThreshold: 3

      startupProbe:
        httpGet:
          path: /health/startup
          port: 3000
        failureThreshold: 30    # allows 30 × 10s = 5min to start
        periodSeconds: 10

  volumes:
    - name: config
      configMap:
        name: app-config
    - name: tmp
      emptyDir: {}

  restartPolicy: Always   # Always | OnFailure | Never
```

## Probe Types

Kubernetes supports three probe mechanisms:

```yaml
# HTTP GET — most common for web services
livenessProbe:
  httpGet:
    path: /health
    port: 3000
    httpHeaders:
      - name: X-Health-Check
        value: "true"

# TCP Socket — for non-HTTP services
livenessProbe:
  tcpSocket:
    port: 5432

# Exec — run a command inside the container
livenessProbe:
  exec:
    command:
      - sh
      - -c
      - "pg_isready -U app -d mydb"
```

### Probe Roles

| Probe | Action on failure | Use for |
|---|---|---|
| `livenessProbe` | Restart the container | Detect deadlocks, hung processes |
| `readinessProbe` | Remove from Service endpoints | Signal "not ready for traffic" during startup/overload |
| `startupProbe` | Restart if never succeeds | Slow-starting apps — replaces `initialDelaySeconds` |

:::tip[startupProbe prevents false restarts]
Without a `startupProbe`, a slow-starting app may be killed by `livenessProbe` before it finishes initialising. Use `startupProbe` with a generous `failureThreshold` to give the app time to start, then hand over to `livenessProbe`.
:::

## Init Containers

Init containers run to completion before the main containers start. Useful for migrations, waiting for dependencies, or fetching secrets.

```yaml
spec:
  initContainers:
    # Wait for the database to be ready
    - name: wait-for-db
      image: busybox:1.36
      command:
        - sh
        - -c
        - |
          until nc -z postgres 5432; do
            echo "Waiting for postgres..."; sleep 2
          done

    # Run database migrations
    - name: migrate
      image: myapp/api:1.2.3
      command: ["node", "dist/migrate.js"]
      env:
        - name: DATABASE_URL
          valueFrom:
            secretKeyRef:
              name: app-secrets
              key: database_url

  containers:
    - name: api
      image: myapp/api:1.2.3
```

## Sidecar Containers

Sidecar containers run alongside the main container in the same Pod, sharing its network and storage.

```yaml
spec:
  containers:
    - name: app
      image: myapp/api:1.2.3
      volumeMounts:
        - name: logs
          mountPath: /app/logs

    # Sidecar: ship logs to centralised logging
    - name: log-shipper
      image: fluent/fluent-bit:latest
      volumeMounts:
        - name: logs
          mountPath: /var/log/app
          readOnly: true

  volumes:
    - name: logs
      emptyDir: {}
```

## Pod Lifecycle

```
┌─────────┐    ┌─────────┐    ┌───────────┐
│ Pending │───►│ Running │───►│ Succeeded │
└─────────┘    └────┬────┘    └───────────┘
                    │
                    ▼
               ┌─────────┐
               │  Failed │
               └─────────┘
```

| Phase | Meaning |
|---|---|
| `Pending` | Accepted by cluster; containers not yet running (scheduling, image pulling) |
| `Running` | At least one container running or starting |
| `Succeeded` | All containers exited with code 0 (Jobs) |
| `Failed` | At least one container exited with non-zero code |
| `Unknown` | Cannot communicate with the node |

```bash
# Check pod phase
kubectl get pod my-pod -o jsonpath='{.status.phase}'

# Detailed status
kubectl describe pod my-pod | grep -A 10 "Conditions"

# Check container states
kubectl get pod my-pod -o jsonpath='{.status.containerStatuses[*].state}'
```

## Deployment

A Deployment manages a ReplicaSet, which manages Pods. It provides declarative updates with rollout and rollback capabilities.

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: myapp
  namespace: production
  labels:
    app: myapp
spec:
  replicas: 3

  # Must match pod template labels
  selector:
    matchLabels:
      app: myapp

  # Rolling update strategy
  strategy:
    type: RollingUpdate
    rollingUpdate:
      maxSurge: 1          # extra pods during update (absolute or %)
      maxUnavailable: 0    # keep all replicas up (zero-downtime)
  # Recreate strategy — kill all old pods before starting new ones
  # strategy:
  #   type: Recreate

  template:
    metadata:
      labels:
        app: myapp
        version: "1.2.3"
    spec:
      # Allow at most 1 unavailable pod during node disruptions
      topologySpreadConstraints:
        - maxSkew: 1
          topologyKey: kubernetes.io/hostname
          whenUnsatisfiable: DoNotSchedule
          labelSelector:
            matchLabels:
              app: myapp

      containers:
        - name: app
          image: myapp/api:1.2.3
          ports:
            - containerPort: 3000
          resources:
            requests:
              cpu: 100m
              memory: 128Mi
            limits:
              cpu: 500m
              memory: 512Mi
          readinessProbe:
            httpGet:
              path: /health
              port: 3000
            initialDelaySeconds: 10
            periodSeconds: 10
```

### Rollout Commands

```bash
# Update image (triggers rolling update)
kubectl set image deployment/myapp app=myapp/api:1.3.0

# Apply manifest changes
kubectl apply -f deployment.yaml

# Check rollout status
kubectl rollout status deployment/myapp
# Waiting for deployment "myapp" rollout to finish: 1 out of 3 updated...

# View rollout history
kubectl rollout history deployment/myapp

# Rollback to previous
kubectl rollout undo deployment/myapp

# Rollback to specific revision
kubectl rollout undo deployment/myapp --to-revision=3

# Pause rollout (to make multiple changes)
kubectl rollout pause deployment/myapp
kubectl set image deployment/myapp app=myapp/api:1.3.0
kubectl set env deployment/myapp LOG_LEVEL=debug
kubectl rollout resume deployment/myapp

# Annotate a rollout (shows in history)
kubectl annotate deployment/myapp kubernetes.io/change-cause="release v1.3.0"
```

## StatefulSet

StatefulSets manage stateful applications that require:
- Stable, unique network identities (e.g., `pod-0`, `pod-1`)
- Stable persistent storage per pod
- Ordered deployment and scaling

```yaml
apiVersion: apps/v1
kind: StatefulSet
metadata:
  name: postgres
spec:
  serviceName: postgres   # headless service name (required)
  replicas: 3
  selector:
    matchLabels:
      app: postgres
  template:
    metadata:
      labels:
        app: postgres
    spec:
      containers:
        - name: postgres
          image: postgres:16-alpine
          env:
            - name: POSTGRES_PASSWORD
              valueFrom:
                secretKeyRef:
                  name: postgres-secret
                  key: password
          volumeMounts:
            - name: data
              mountPath: /var/lib/postgresql/data

  # Each pod gets its own PVC from this template
  volumeClaimTemplates:
    - metadata:
        name: data
      spec:
        accessModes: ["ReadWriteOnce"]
        storageClassName: fast
        resources:
          requests:
            storage: 10Gi
```

Pod names are predictable: `postgres-0`, `postgres-1`, `postgres-2`. DNS: `postgres-0.postgres.namespace.svc.cluster.local`.

## DaemonSet

Runs exactly one Pod per node (or per selected nodes). Used for cluster-level services like log collectors and monitoring agents.

```yaml
apiVersion: apps/v1
kind: DaemonSet
metadata:
  name: node-exporter
  namespace: monitoring
spec:
  selector:
    matchLabels:
      app: node-exporter
  template:
    metadata:
      labels:
        app: node-exporter
    spec:
      hostNetwork: true
      hostPID: true
      containers:
        - name: node-exporter
          image: prom/node-exporter:latest
          ports:
            - containerPort: 9100
          volumeMounts:
            - name: proc
              mountPath: /host/proc
              readOnly: true
            - name: sys
              mountPath: /host/sys
              readOnly: true
      volumes:
        - name: proc
          hostPath:
            path: /proc
        - name: sys
          hostPath:
            path: /sys
      tolerations:
        - effect: NoSchedule
          operator: Exists
```

## Job

A Job creates one or more Pods and ensures a specified number complete successfully.

```yaml
apiVersion: batch/v1
kind: Job
metadata:
  name: db-migration
spec:
  completions: 1
  parallelism: 1
  backoffLimit: 3       # retry up to 3 times on failure
  activeDeadlineSeconds: 300   # fail if not complete in 5 minutes
  template:
    spec:
      restartPolicy: OnFailure   # Never | OnFailure
      containers:
        - name: migrate
          image: myapp/api:1.2.3
          command: ["node", "dist/migrate.js"]
          env:
            - name: DATABASE_URL
              valueFrom:
                secretKeyRef:
                  name: app-secrets
                  key: database_url
```

## CronJob

A CronJob creates Jobs on a schedule (standard cron syntax).

```yaml
apiVersion: batch/v1
kind: CronJob
metadata:
  name: nightly-backup
spec:
  schedule: "0 2 * * *"          # Every day at 2:00 AM UTC
  timeZone: "America/New_York"   # Kubernetes 1.27+
  concurrencyPolicy: Forbid      # Allow | Forbid | Replace
  successfulJobsHistoryLimit: 3
  failedJobsHistoryLimit: 3
  startingDeadlineSeconds: 300   # skip if missed by >5 minutes
  jobTemplate:
    spec:
      template:
        spec:
          restartPolicy: OnFailure
          containers:
            - name: backup
              image: myapp/backup:latest
              command: ["./backup.sh"]
              env:
                - name: S3_BUCKET
                  value: my-backup-bucket
```

```bash
# Manually trigger a CronJob
kubectl create job --from=cronjob/nightly-backup manual-backup-$(date +%s)

# View job status
kubectl get jobs
kubectl describe job nightly-backup-123456
```

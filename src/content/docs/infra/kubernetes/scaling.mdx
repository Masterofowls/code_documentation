# Scaling in Kubernetes

Kubernetes provides multiple layers of scaling — from manual replica adjustment to sophisticated event-driven and predictive autoscaling systems.

## Manual Scaling

```bash
# Scale a Deployment to 5 replicas
kubectl scale deployment myapp --replicas=5

# Scale multiple deployments at once
kubectl scale deployment api worker --replicas=3

# Scale to zero (stop without deleting)
kubectl scale deployment myapp --replicas=0

# Conditional scale (only if current replicas = 2)
kubectl scale deployment myapp --replicas=5 --current-replicas=2
```

## Horizontal Pod Autoscaler (HPA)

HPA automatically scales the number of Pod replicas based on observed metrics.

### Install metrics-server

HPA requires `metrics-server` for CPU/memory metrics:

```bash
# Install via Helm
helm repo add metrics-server https://kubernetes-sigs.github.io/metrics-server/
helm upgrade --install metrics-server metrics-server/metrics-server \
  --namespace kube-system

# Verify
kubectl top pods
kubectl top nodes
```

### Basic CPU-Based HPA

```yaml
apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata:
  name: myapp-hpa
  namespace: myapp
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: myapp
  minReplicas: 2
  maxReplicas: 20
  metrics:
    - type: Resource
      resource:
        name: cpu
        target:
          type: Utilization
          averageUtilization: 70    # target 70% CPU utilisation
    - type: Resource
      resource:
        name: memory
        target:
          type: AverageValue
          averageValue: 400Mi       # target 400Mi memory per pod
```

### Custom Metrics HPA

```yaml
apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata:
  name: myapp-hpa-custom
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: myapp
  minReplicas: 2
  maxReplicas: 50
  metrics:
    # Custom metric from Prometheus adapter
    - type: Pods
      pods:
        metric:
          name: http_requests_per_second
        target:
          type: AverageValue
          averageValue: "100"    # 100 req/s per pod

    # External metric (queue depth, etc.)
    - type: External
      external:
        metric:
          name: sqs_queue_depth
          selector:
            matchLabels:
              queue: myapp-jobs
        target:
          type: AverageValue
          averageValue: "30"    # 30 messages per pod
```

### HPA Scaling Behaviour

Control the speed of scale-up and scale-down:

```yaml
spec:
  behavior:
    scaleUp:
      stabilizationWindowSeconds: 60    # don't scale up again within 60s
      policies:
        - type: Percent
          value: 100      # double replicas per period
          periodSeconds: 30
        - type: Pods
          value: 4        # add max 4 pods per period
          periodSeconds: 30
      selectPolicy: Max   # use the policy that allows more scaling

    scaleDown:
      stabilizationWindowSeconds: 300   # wait 5min before scaling down
      policies:
        - type: Percent
          value: 25       # remove max 25% per period
          periodSeconds: 60
      selectPolicy: Min   # use the policy that allows less scaling
```

```bash
# View HPA status
kubectl get hpa
kubectl describe hpa myapp-hpa

# Watch HPA in action
kubectl get hpa myapp-hpa -w
```

## Vertical Pod Autoscaler (VPA)

VPA automatically adjusts CPU/memory requests and limits based on actual usage. Requires the [VPA admission controller](https://github.com/kubernetes/autoscaler/tree/master/vertical-pod-autoscaler).

```bash
# Install VPA
git clone https://github.com/kubernetes/autoscaler.git
cd autoscaler/vertical-pod-autoscaler
./hack/vpa-install.sh
```

```yaml
apiVersion: autoscaling.k8s.io/v1
kind: VerticalPodAutoscaler
metadata:
  name: myapp-vpa
spec:
  targetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: myapp
  updatePolicy:
    updateMode: "Auto"    # Off | Initial | Recreate | Auto
  resourcePolicy:
    containerPolicies:
      - containerName: app
        minAllowed:
          cpu: 50m
          memory: 64Mi
        maxAllowed:
          cpu: 2
          memory: 2Gi
        controlledResources: ["cpu", "memory"]
```

| VPA Mode | Behaviour |
|---|---|
| `Off` | Recommendations only — no automatic changes |
| `Initial` | Set requests when Pod is first created |
| `Recreate` | Update requests by evicting and recreating Pods |
| `Auto` | Same as Recreate (may support in-place updates in future) |

:::caution[Don't use HPA and VPA together on CPU]
Using HPA (scaling replicas by CPU) and VPA (adjusting CPU requests) simultaneously on the same metric creates conflicts. Use HPA for CPU/memory scaling and VPA for fine-tuning, or use VPA only in `Off` mode for recommendations.
:::

## KEDA (Kubernetes Event-Driven Autoscaling)

KEDA extends HPA to support scaling based on external event sources — message queues, databases, HTTP traffic, cron schedules, and more.

```bash
# Install KEDA
helm repo add kedacore https://kedacore.github.io/charts
helm install keda kedacore/keda --namespace keda --create-namespace
```

### Scale on AWS SQS Queue Depth

```yaml
apiVersion: keda.sh/v1alpha1
kind: ScaledObject
metadata:
  name: sqs-scaler
  namespace: myapp
spec:
  scaleTargetRef:
    name: worker-deployment
  minReplicaCount: 0    # scale to zero!
  maxReplicaCount: 50
  pollingInterval: 30
  cooldownPeriod: 300
  triggers:
    - type: aws-sqs-queue
      metadata:
        queueURL: https://sqs.us-east-1.amazonaws.com/123456/my-queue
        queueLength: "5"    # 1 replica per 5 messages
        awsRegion: us-east-1
      authenticationRef:
        name: aws-secret-provider
```

### Scale on Redis List Length

```yaml
triggers:
  - type: redis
    metadata:
      address: redis.myapp.svc.cluster.local:6379
      listName: job-queue
      listLength: "10"   # 1 replica per 10 items
```

### Scale on Kafka Consumer Lag

```yaml
triggers:
  - type: kafka
    metadata:
      bootstrapServers: kafka.myapp.svc:9092
      consumerGroup: myapp-consumers
      topic: events
      lagThreshold: "100"   # 1 replica per 100 messages behind
```

### Scale on HTTP Request Rate

```yaml
# Requires KEDA HTTP add-on
triggers:
  - type: http
    metadata:
      url: http://myapp.myapp.svc.cluster.local/
      targetPendingRequests: "100"
```

### Scale on Cron Schedule

```yaml
triggers:
  - type: cron
    metadata:
      timezone: America/New_York
      start: "0 8 * * 1-5"   # scale up: 8 AM weekdays
      end: "0 18 * * 1-5"    # scale down: 6 PM weekdays
      desiredReplicas: "10"
```

## Cluster Autoscaler

Cluster Autoscaler adds or removes nodes based on pending Pods and underutilised nodes.

```bash
# Install on AWS EKS
helm repo add autoscaler https://kubernetes.github.io/autoscaler
helm install cluster-autoscaler autoscaler/cluster-autoscaler \
  --namespace kube-system \
  --set autoDiscovery.clusterName=my-cluster \
  --set awsRegion=us-east-1 \
  --set rbac.serviceAccount.annotations."eks\.amazonaws\.com/role-arn"=arn:aws:iam::ACCOUNT:role/ClusterAutoscalerRole
```

Node group annotation for auto-discovery:

```bash
# Tag the Auto Scaling Group
k8s.io/cluster-autoscaler/enabled: "true"
k8s.io/cluster-autoscaler/my-cluster: "owned"
```

## Pod Disruption Budgets (PDB)

PDBs limit the number of Pods that can be voluntarily disrupted simultaneously (node drains, rolling updates).

```yaml
# Ensure at least 2 replicas always available
apiVersion: policy/v1
kind: PodDisruptionBudget
metadata:
  name: myapp-pdb
  namespace: myapp
spec:
  selector:
    matchLabels:
      app: myapp
  minAvailable: 2      # absolute count OR percentage "80%"

---
# Alternatively, set maximum unavailable
apiVersion: policy/v1
kind: PodDisruptionBudget
metadata:
  name: myapp-pdb-max
spec:
  selector:
    matchLabels:
      app: myapp
  maxUnavailable: 1    # at most 1 pod disrupted at once
```

```bash
kubectl get pdb
kubectl describe pdb myapp-pdb
```

## Pod Affinity and Anti-Affinity

### Anti-Affinity (spread pods across nodes)

```yaml
spec:
  affinity:
    podAntiAffinity:
      # Hard rule — never schedule two myapp pods on the same node
      requiredDuringSchedulingIgnoredDuringExecution:
        - labelSelector:
            matchExpressions:
              - key: app
                operator: In
                values: ["myapp"]
          topologyKey: kubernetes.io/hostname

      # Soft rule — prefer different availability zones
      preferredDuringSchedulingIgnoredDuringExecution:
        - weight: 100
          podAffinityTerm:
            labelSelector:
              matchLabels:
                app: myapp
            topologyKey: topology.kubernetes.io/zone
```

### Pod Affinity (co-locate related pods)

```yaml
spec:
  affinity:
    podAffinity:
      preferredDuringSchedulingIgnoredDuringExecution:
        - weight: 100
          podAffinityTerm:
            labelSelector:
              matchLabels:
                app: redis-cache
            topologyKey: kubernetes.io/hostname
```

## Taints and Tolerations

Taints prevent Pods from being scheduled on a node unless the Pod has a matching toleration.

```bash
# Taint a node (e.g., reserve for GPU workloads)
kubectl taint nodes gpu-node-1 gpu=true:NoSchedule

# Remove a taint
kubectl taint nodes gpu-node-1 gpu=true:NoSchedule-

# Taint effects:
# NoSchedule — don't schedule new pods without toleration
# PreferNoSchedule — avoid scheduling, but allow if necessary
# NoExecute — evict running pods without toleration
```

```yaml
# Pod with matching toleration
spec:
  tolerations:
    - key: gpu
      operator: Equal
      value: "true"
      effect: NoSchedule

  # Schedule specifically on GPU nodes
  nodeSelector:
    gpu: "true"
```

## Node Affinity

```yaml
spec:
  affinity:
    nodeAffinity:
      # Hard requirement
      requiredDuringSchedulingIgnoredDuringExecution:
        nodeSelectorTerms:
          - matchExpressions:
              - key: topology.kubernetes.io/zone
                operator: In
                values: ["us-east-1a", "us-east-1b"]

      # Preference
      preferredDuringSchedulingIgnoredDuringExecution:
        - weight: 100
          preference:
            matchExpressions:
              - key: node.kubernetes.io/instance-type
                operator: In
                values: ["m5.large", "m5.xlarge"]
```

## TopologySpreadConstraints

Distribute Pods evenly across topology domains (nodes, AZs, regions).

```yaml
spec:
  topologySpreadConstraints:
    # Spread across availability zones
    - maxSkew: 1
      topologyKey: topology.kubernetes.io/zone
      whenUnsatisfiable: DoNotSchedule
      labelSelector:
        matchLabels:
          app: myapp

    # Spread across nodes
    - maxSkew: 1
      topologyKey: kubernetes.io/hostname
      whenUnsatisfiable: ScheduleAnyway   # soft
      labelSelector:
        matchLabels:
          app: myapp
```

## Resource Quotas and LimitRange

### ResourceQuota (namespace-level limits)

```yaml
apiVersion: v1
kind: ResourceQuota
metadata:
  name: myapp-quota
  namespace: myapp
spec:
  hard:
    pods: "50"
    requests.cpu: "10"
    requests.memory: 20Gi
    limits.cpu: "20"
    limits.memory: 40Gi
    persistentvolumeclaims: "10"
    requests.storage: 200Gi
    services.loadbalancers: "2"
```

### LimitRange (per-pod defaults)

```yaml
apiVersion: v1
kind: LimitRange
metadata:
  name: myapp-limits
  namespace: myapp
spec:
  limits:
    - type: Container
      default:          # applied if limits not specified
        cpu: 200m
        memory: 256Mi
      defaultRequest:   # applied if requests not specified
        cpu: 50m
        memory: 64Mi
      max:
        cpu: "2"
        memory: 2Gi
      min:
        cpu: 10m
        memory: 32Mi
```

# Services & Ingress

Kubernetes Services provide stable networking for Pods. Ingress manages external HTTP/HTTPS access. Together they form the core of Kubernetes networking for applications.

---

## Why Services?

Pods are ephemeral — they get new IP addresses when they restart, scale, or roll out updates. A **Service** provides a stable virtual IP (ClusterIP) and DNS name that automatically load balances across all healthy matching Pods.

```
Client Request
      │
      ▼
  Service (stable IP + DNS)
      │
      ├──► Pod A (10.0.0.5)
      ├──► Pod B (10.0.0.6)
      └──► Pod C (10.0.0.7)  ← Pods come and go; Service IP stays
```

## Service Types

### ClusterIP (Default)

Exposes the Service on a cluster-internal IP. Only reachable from within the cluster.

```yaml
apiVersion: v1
kind: Service
metadata:
  name: my-api
  namespace: default
spec:
  type: ClusterIP             # Default; can be omitted
  selector:
    app: my-api               # Routes to pods with this label
  ports:
    - name: http
      protocol: TCP
      port: 80                # Service port (what clients connect to)
      targetPort: 3000        # Pod port (where app listens)
```

### NodePort

Exposes the Service on each Node's IP at a static port (30000–32767).

```yaml
spec:
  type: NodePort
  selector:
    app: my-api
  ports:
    - port: 80
      targetPort: 3000
      nodePort: 30080         # Omit to auto-assign a port in range
```

```bash
# Access via any node IP
curl http://<node-ip>:30080
```

:::tip[NodePort and cloud environments]
NodePort is rarely used directly in production cloud environments. Use LoadBalancer or Ingress instead. NodePort is useful for local clusters (minikube, kind) and bare-metal setups.
:::

### LoadBalancer

Provisions an external load balancer (from the cloud provider — AWS ELB, GCP CLB, Azure LB).

```yaml
spec:
  type: LoadBalancer
  selector:
    app: my-api
  ports:
    - port: 80
      targetPort: 3000
  loadBalancerSourceRanges:
    - 10.0.0.0/8              # Restrict to internal networks
```

```bash
# Watch for external IP assignment
kubectl get service my-api --watch
```

:::caution[LoadBalancer costs]
Each `LoadBalancer` Service typically provisions a separate cloud load balancer, which can be expensive. For HTTP/HTTPS traffic, use a single Ingress with one load balancer for all services.
:::

### ExternalName

Maps a Service to an external DNS name. Useful for accessing external services with the same DNS pattern as internal services.

```yaml
spec:
  type: ExternalName
  externalName: my-database.example.com
```

### Headless Service

A Service with `clusterIP: None`. Used with StatefulSets for direct Pod DNS addressing.

```yaml
spec:
  clusterIP: None             # Headless — no virtual IP
  selector:
    app: postgres
  ports:
    - port: 5432
```

DNS entries for individual pods: `postgres-0.postgres.default.svc.cluster.local`

## Service DNS

Kubernetes includes a DNS server (CoreDNS) that automatically creates DNS records for Services.

```
Full DNS name:
  <service-name>.<namespace>.svc.cluster.local

Within the same namespace (short name works):
  my-api → resolves to my-api ClusterIP

Cross-namespace:
  my-api.other-namespace → resolves to my-api in other-namespace

StatefulSet pod DNS:
  postgres-0.postgres.default.svc.cluster.local
```

```bash
# Test DNS from within a pod
kubectl run dns-test --image=busybox:1.35 --rm -it -- sh
nslookup my-api
nslookup my-api.default.svc.cluster.local
nslookup postgres-0.postgres.default.svc.cluster.local
```

## Multi-Port Services

```yaml
spec:
  selector:
    app: my-app
  ports:
    - name: http
      port: 80
      targetPort: 3000
    - name: https
      port: 443
      targetPort: 3443
    - name: metrics
      port: 9090
      targetPort: 9090
```

:::tip[Named ports]
When a Service has multiple ports, all ports must be named. Named ports can also be referenced by name in targetPort, Ingress rules, and probes — making configuration more readable and resilient to port number changes.
:::

## Endpoints and EndpointSlices

Services select Pods via label selectors. The selected Pod IPs are stored in **Endpoints** (or **EndpointSlices** for scalability).

```bash
# View endpoints for a service
kubectl get endpoints my-api

# View EndpointSlices
kubectl get endpointslices -l kubernetes.io/service-name=my-api

# Manual endpoint (Service without selector — external backend)
```

```yaml
# Service without selector
apiVersion: v1
kind: Service
metadata:
  name: external-db
spec:
  ports:
    - port: 5432
---
# Manual Endpoints
apiVersion: v1
kind: Endpoints
metadata:
  name: external-db
subsets:
  - addresses:
      - ip: 192.168.1.100
    ports:
      - port: 5432
```

## Ingress

An **Ingress** manages external access to Services, typically HTTP/HTTPS. It provides load balancing, SSL termination, and name-based virtual hosting in a single resource.

:::caution[Ingress Controller required]
An Ingress resource alone does nothing. You need an **Ingress Controller** installed in your cluster. Popular options:
- **ingress-nginx** — most widely used
- **Traefik** — dynamic routing, great defaults
- **AWS ALB Ingress Controller** — provisions AWS Application Load Balancers
- **HAProxy Ingress** — high performance
:::

### Basic Ingress

```yaml
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: my-ingress
  namespace: default
  annotations:
    nginx.ingress.kubernetes.io/rewrite-target: /
spec:
  ingressClassName: nginx          # Which IngressClass to use
  rules:
    - host: api.example.com
      http:
        paths:
          - path: /
            pathType: Prefix
            backend:
              service:
                name: my-api
                port:
                  name: http       # Use service port name

    - host: admin.example.com
      http:
        paths:
          - path: /
            pathType: Exact
            backend:
              service:
                name: admin-ui
                port:
                  number: 80

  # Default backend for unmatched requests
  defaultBackend:
    service:
      name: default-backend
      port:
        number: 80
```

### Path Types

| pathType | Behavior |
|---|---|
| `Exact` | Matches exactly the path |
| `Prefix` | Matches paths with given prefix |
| `ImplementationSpecific` | Behavior depends on IngressClass implementation |

### TLS Termination

```yaml
spec:
  ingressClassName: nginx
  tls:
    - hosts:
        - api.example.com
        - admin.example.com
      secretName: example-tls-secret   # TLS cert Secret
  rules:
    - host: api.example.com
      # ...
```

```bash
# Manually create TLS secret
kubectl create secret tls example-tls-secret \
  --cert=tls.crt \
  --key=tls.key \
  --namespace=default
```

## cert-manager and Let's Encrypt

[cert-manager](https://cert-manager.io) automates TLS certificate management.

<Steps>
1. **Install cert-manager**

   ```bash
   kubectl apply -f https://github.com/cert-manager/cert-manager/releases/latest/download/cert-manager.yaml
   ```

2. **Create a ClusterIssuer for Let's Encrypt**

   ```yaml
   apiVersion: cert-manager.io/v1
   kind: ClusterIssuer
   metadata:
     name: letsencrypt-prod
   spec:
     acme:
       server: https://acme-v02.api.letsencrypt.org/directory
       email: admin@example.com
       privateKeySecretRef:
         name: letsencrypt-prod-key
       solvers:
         - http01:
             ingress:
               ingressClassName: nginx
   ```

3. **Annotate your Ingress**

   ```yaml
   metadata:
     annotations:
       cert-manager.io/cluster-issuer: letsencrypt-prod
   spec:
     tls:
       - hosts:
           - api.example.com
         secretName: api-tls             # cert-manager creates this
     rules:
       - host: api.example.com
         # ...
   ```
</Steps>

## nginx Ingress Annotations

```yaml
metadata:
  annotations:
    # Rewrite URL path
    nginx.ingress.kubernetes.io/rewrite-target: /$2

    # Rate limiting
    nginx.ingress.kubernetes.io/limit-rps: "10"
    nginx.ingress.kubernetes.io/limit-connections: "5"

    # CORS
    nginx.ingress.kubernetes.io/enable-cors: "true"
    nginx.ingress.kubernetes.io/cors-allow-origin: "https://app.example.com"

    # Body size limit (default: 1m)
    nginx.ingress.kubernetes.io/proxy-body-size: "50m"

    # WebSocket support
    nginx.ingress.kubernetes.io/proxy-read-timeout: "3600"
    nginx.ingress.kubernetes.io/proxy-send-timeout: "3600"

    # Auth
    nginx.ingress.kubernetes.io/auth-url: "https://auth.example.com/verify"
    nginx.ingress.kubernetes.io/auth-signin: "https://auth.example.com/login"

    # Force HTTPS
    nginx.ingress.kubernetes.io/ssl-redirect: "true"
    nginx.ingress.kubernetes.io/force-ssl-redirect: "true"

    # Custom nginx config
    nginx.ingress.kubernetes.io/configuration-snippet: |
      add_header X-Frame-Options DENY;
      add_header X-Content-Type-Options nosniff;
```

## Gateway API (Modern Replacement for Ingress)

The **Gateway API** is the next generation of Kubernetes traffic routing, offering more expressiveness and role separation than Ingress.

```yaml
# GatewayClass (cluster-admin sets up once)
apiVersion: gateway.networking.k8s.io/v1
kind: GatewayClass
metadata:
  name: nginx
spec:
  controllerName: k8s.nginx.org/nginx-gateway-controller
---
# Gateway (cluster-admin or infra team)
apiVersion: gateway.networking.k8s.io/v1
kind: Gateway
metadata:
  name: main-gateway
  namespace: infra
spec:
  gatewayClassName: nginx
  listeners:
    - name: https
      port: 443
      protocol: HTTPS
      tls:
        certificateRefs:
          - name: wildcard-cert
---
# HTTPRoute (application team — namespaced)
apiVersion: gateway.networking.k8s.io/v1
kind: HTTPRoute
metadata:
  name: api-route
  namespace: production
spec:
  parentRefs:
    - name: main-gateway
      namespace: infra
  hostnames:
    - api.example.com
  rules:
    - matches:
        - path:
            type: PathPrefix
            value: /v1
      backendRefs:
        - name: my-api
          port: 80
          weight: 100
```

## external-dns

[external-dns](https://github.com/kubernetes-sigs/external-dns) automatically creates DNS records for Services and Ingresses in external DNS providers (Route53, Cloud DNS, Cloudflare, etc.).

```yaml
# Annotation on Ingress or Service to manage DNS
metadata:
  annotations:
    external-dns.alpha.kubernetes.io/hostname: api.example.com
    external-dns.alpha.kubernetes.io/ttl: "60"
```

## Internal Load Balancing

```yaml
# AWS — provision internal NLB (not public-facing)
metadata:
  annotations:
    service.beta.kubernetes.io/aws-load-balancer-internal: "true"
    service.beta.kubernetes.io/aws-load-balancer-type: "nlb"
spec:
  type: LoadBalancer
```

```yaml
# GKE — internal load balancer
metadata:
  annotations:
    cloud.google.com/load-balancer-type: "Internal"
spec:
  type: LoadBalancer
```

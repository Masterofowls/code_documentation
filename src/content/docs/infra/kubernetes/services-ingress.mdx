# Services & Ingress

Services provide stable network endpoints for Pods, while Ingress routes external HTTP/HTTPS traffic into the cluster.

---

## Services

A Service is a stable virtual IP (ClusterIP) and DNS name that routes traffic to a dynamic set of Pods based on label selectors.

### Why Services Are Needed

Pods are ephemeral — they can be rescheduled, restarted, and replaced at any time, changing their IP addresses. Services provide a stable endpoint that always routes to the current healthy Pods.

---

## Service Types

### ClusterIP (Default)

Exposes the Service on an internal cluster IP. Only reachable from within the cluster.

```yaml
apiVersion: v1
kind: Service
metadata:
  name: api-service
  namespace: default
spec:
  type: ClusterIP             # Default — omit type to use ClusterIP
  selector:
    app: myapp                # Routes to Pods with this label
  ports:
    - name: http
      port: 80                # Port the Service listens on
      targetPort: 3000        # Port on the Pod
      protocol: TCP
```

### NodePort

Exposes the Service on a static port on each node's IP. Accessible from outside the cluster at `NodeIP:NodePort`.

```yaml
spec:
  type: NodePort
  selector:
    app: myapp
  ports:
    - port: 80
      targetPort: 3000
      nodePort: 30080         # Must be 30000–32767 (or auto-assigned)
```

:::caution[NodePort for production]
NodePort exposes a port on every node. For production, prefer LoadBalancer type (cloud managed) or Ingress for HTTP/HTTPS traffic. NodePort is useful for development clusters and bare-metal setups without a cloud load balancer.
:::

### LoadBalancer

Provisions an external load balancer from the cloud provider (AWS NLB, GCP NLB, Azure Load Balancer).

```yaml
spec:
  type: LoadBalancer
  selector:
    app: myapp
  ports:
    - port: 80
      targetPort: 3000
  # Cloud-specific annotations
  # AWS:
  # annotations:
  #   service.beta.kubernetes.io/aws-load-balancer-type: external
  #   service.beta.kubernetes.io/aws-load-balancer-nlb-target-type: ip
```

### ExternalName

Maps a Service to an external DNS name. Useful for referencing external services inside the cluster.

```yaml
spec:
  type: ExternalName
  externalName: my-database.us-east-1.rds.amazonaws.com
  # Now accessible inside cluster as: db-service.default.svc.cluster.local
```

### Headless Service

A ClusterIP Service with `clusterIP: None`. Returns individual Pod IPs instead of a virtual IP. Used with StatefulSets for direct Pod addressing.

```yaml
spec:
  clusterIP: None             # Headless — no virtual IP
  selector:
    app: postgres
  ports:
    - port: 5432
# DNS: postgres-0.postgres-headless.default.svc.cluster.local
```

## Service DNS

Kubernetes provides DNS for Services automatically via CoreDNS.

```
# Full DNS name format
<service>.<namespace>.svc.cluster.local

# Examples
api-service.default.svc.cluster.local
db.production.svc.cluster.local

# Short forms (within same namespace)
api-service
api-service.default

# From inside a Pod — test DNS resolution
kubectl exec -it my-pod -- nslookup api-service
kubectl exec -it my-pod -- curl http://api-service:80/health
```

## Multi-Port Services

```yaml
spec:
  selector:
    app: myapp
  ports:
    - name: http
      port: 80
      targetPort: 3000
    - name: grpc
      port: 9090
      targetPort: 9090
    - name: metrics
      port: 9091
      targetPort: 9091
```

:::tip[Name your ports]
Always name Service ports. This allows Ingress rules and health checks to reference the port by name rather than number, making refactoring easier.
:::

## Session Affinity

Route requests from the same client to the same Pod.

```yaml
spec:
  sessionAffinity: ClientIP
  sessionAffinityConfig:
    clientIP:
      timeoutSeconds: 10800   # 3 hours
```

## Endpoints and EndpointSlices

Kubernetes automatically creates `Endpoints` (or `EndpointSlices` in newer clusters) that track the IPs of ready Pods matching the Service selector.

```bash
# View endpoints for a Service
kubectl get endpoints api-service

# View EndpointSlices (K8s 1.21+)
kubectl get endpointslices -l kubernetes.io/service-name=api-service

# Manual endpoint (for external services without DNS)
kubectl apply -f - <<EOF
apiVersion: v1
kind: Endpoints
metadata:
  name: external-db
subsets:
  - addresses:
      - ip: 192.168.1.100
    ports:
      - port: 5432
EOF
```

## Ingress

Ingress manages external HTTP/HTTPS access to Services based on rules. It requires an **Ingress controller** to be running in the cluster.

### Installing an Ingress Controller

<Tabs>
  <TabItem label="nginx (most common)">
    ```bash
    # Install nginx ingress controller via Helm
    helm repo add ingress-nginx https://kubernetes.github.io/ingress-nginx
    helm repo update
    helm install ingress-nginx ingress-nginx/ingress-nginx \
      --namespace ingress-nginx \
      --create-namespace

    # Verify
    kubectl get pods -n ingress-nginx
    kubectl get svc -n ingress-nginx
    ```
  </TabItem>
  <TabItem label="Traefik">
    ```bash
    helm repo add traefik https://traefik.github.io/charts
    helm repo update
    helm install traefik traefik/traefik \
      --namespace traefik \
      --create-namespace
    ```
  </TabItem>
  <TabItem label="AWS ALB Controller">
    ```bash
    helm repo add eks https://aws.github.io/eks-charts
    helm install aws-load-balancer-controller eks/aws-load-balancer-controller \
      -n kube-system \
      --set clusterName=my-cluster \
      --set serviceAccount.create=false \
      --set serviceAccount.name=aws-load-balancer-controller
    ```
  </TabItem>
</Tabs>

### Ingress Resource

```yaml
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: my-ingress
  namespace: default
  annotations:
    # nginx annotations
    nginx.ingress.kubernetes.io/rewrite-target: /
    nginx.ingress.kubernetes.io/proxy-body-size: "50m"
    nginx.ingress.kubernetes.io/proxy-read-timeout: "60"
    nginx.ingress.kubernetes.io/proxy-connect-timeout: "10"
    nginx.ingress.kubernetes.io/use-regex: "true"
    nginx.ingress.kubernetes.io/limit-rps: "100"
spec:
  ingressClassName: nginx      # Which Ingress controller to use

  # TLS termination
  tls:
    - hosts:
        - api.example.com
        - www.example.com
      secretName: example-tls   # TLS cert/key stored in a Secret

  rules:
    # Route by hostname and path
    - host: api.example.com
      http:
        paths:
          - path: /v1
            pathType: Prefix           # Prefix | Exact | ImplementationSpecific
            backend:
              service:
                name: api-v1-service
                port:
                  name: http

          - path: /v2
            pathType: Prefix
            backend:
              service:
                name: api-v2-service
                port:
                  name: http

    - host: www.example.com
      http:
        paths:
          - path: /
            pathType: Prefix
            backend:
              service:
                name: frontend-service
                port:
                  number: 80

  # Default backend if no rules match
  defaultBackend:
    service:
      name: default-http-backend
      port:
        number: 80
```

### Path Types

| Path Type | Behavior |
|---|---|
| `Exact` | Path must match exactly |
| `Prefix` | Path must match prefix (split by `/`) |
| `ImplementationSpecific` | Behavior defined by the Ingress controller |

## IngressClass

```yaml
apiVersion: networking.k8s.io/v1
kind: IngressClass
metadata:
  name: nginx
  annotations:
    ingressclass.kubernetes.io/is-default-class: "true"  # Default for all Ingress
spec:
  controller: k8s.io/ingress-nginx
```

## TLS with cert-manager

cert-manager automates TLS certificate provisioning from Let's Encrypt (or other ACME providers).

```bash
# Install cert-manager
kubectl apply -f https://github.com/cert-manager/cert-manager/releases/download/v1.14.0/cert-manager.yaml

# Wait for it to be ready
kubectl wait --for=condition=Available deployment -l app.kubernetes.io/instance=cert-manager \
  -n cert-manager --timeout=120s
```

Create a ClusterIssuer for Let's Encrypt:

```yaml
apiVersion: cert-manager.io/v1
kind: ClusterIssuer
metadata:
  name: letsencrypt-prod
spec:
  acme:
    server: https://acme-v02.api.letsencrypt.org/directory
    email: admin@example.com
    privateKeySecretRef:
      name: letsencrypt-prod-account-key
    solvers:
      - http01:
          ingress:
            class: nginx
```

Annotate Ingress for automatic TLS:

```yaml
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: my-ingress
  annotations:
    cert-manager.io/cluster-issuer: letsencrypt-prod
spec:
  tls:
    - hosts:
        - api.example.com
      secretName: api-example-com-tls    # cert-manager creates this Secret
  rules:
    - host: api.example.com
      http:
        paths:
          - path: /
            pathType: Prefix
            backend:
              service:
                name: api-service
                port:
                  name: http
```

## Gateway API (Modern Replacement)

The Gateway API is the next generation of Kubernetes ingress routing. It is more expressive, extensible, and role-oriented than Ingress.

```yaml
# GatewayClass (cluster-wide — configured by platform team)
apiVersion: gateway.networking.k8s.io/v1
kind: GatewayClass
metadata:
  name: nginx
spec:
  controllerName: k8s.io/ingress-nginx

---
# Gateway (namespace-level — configured by platform/ops team)
apiVersion: gateway.networking.k8s.io/v1
kind: Gateway
metadata:
  name: main-gateway
  namespace: infra
spec:
  gatewayClassName: nginx
  listeners:
    - name: https
      protocol: HTTPS
      port: 443
      tls:
        mode: Terminate
        certificateRefs:
          - name: example-tls

---
# HTTPRoute (application team configures their own routes)
apiVersion: gateway.networking.k8s.io/v1
kind: HTTPRoute
metadata:
  name: api-route
  namespace: default
spec:
  parentRefs:
    - name: main-gateway
      namespace: infra
  hostnames:
    - api.example.com
  rules:
    - matches:
        - path:
            type: PathPrefix
            value: /api
      backendRefs:
        - name: api-service
          port: 80
          weight: 100
    - matches:
        - path:
            type: PathPrefix
            value: /
      backendRefs:
        - name: frontend-service
          port: 80
```

## external-dns

external-dns automatically creates DNS records in Route53, Cloudflare, etc. when Ingress or Services are created.

```yaml
# Annotation to trigger external-dns record creation
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  annotations:
    external-dns.alpha.kubernetes.io/hostname: api.example.com
    external-dns.alpha.kubernetes.io/ttl: "60"
```

## Troubleshooting Services and Ingress

```bash
# Check Service endpoints (are Pods ready?)
kubectl get endpoints api-service
kubectl describe service api-service

# Test ClusterIP from inside the cluster
kubectl run -it --rm debug --image=nicolaka/netshoot -- \
  curl http://api-service.default.svc.cluster.local/health

# Check Ingress status
kubectl get ingress
kubectl describe ingress my-ingress

# Check Ingress controller logs
kubectl logs -n ingress-nginx -l app.kubernetes.io/name=ingress-nginx

# Check cert-manager certificate status
kubectl get certificate -A
kubectl describe certificate api-example-com-tls

# Check TLS secret
kubectl get secret api-example-com-tls -o jsonpath='{.data.tls\.crt}' | base64 -d | openssl x509 -noout -dates
```

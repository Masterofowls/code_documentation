# Services & Ingress

Kubernetes Services provide stable network endpoints for Pods. Ingress and the Gateway API manage external HTTP/HTTPS traffic routing into the cluster.

## Why Services?

Pods are ephemeral — they can be rescheduled to different IPs at any time. A Service provides a stable virtual IP (ClusterIP) and DNS name that transparently load-balances to healthy Pod endpoints, regardless of how many replicas exist or where they run.

## Service Types

### ClusterIP (default)

Reachable only from within the cluster. Most services should be ClusterIP.

```yaml
apiVersion: v1
kind: Service
metadata:
  name: api
  namespace: myapp
spec:
  type: ClusterIP
  selector:
    app: api         # matches pods with this label
  ports:
    - name: http
      port: 80          # port on the Service
      targetPort: 3000  # port on the Pod
      protocol: TCP
```

```bash
# From within the cluster, reach this service at:
# api                                 (within same namespace)
# api.myapp                           (namespace.svc shorthand)
# api.myapp.svc.cluster.local         (fully qualified)
# 10.96.x.x                           (ClusterIP)
```

### NodePort

Opens a port (default range 30000–32767) on every node. External traffic can reach the service via `<NodeIP>:<NodePort>`.

```yaml
apiVersion: v1
kind: Service
metadata:
  name: api-nodeport
spec:
  type: NodePort
  selector:
    app: api
  ports:
    - port: 80
      targetPort: 3000
      nodePort: 30080   # omit to auto-assign
```

:::caution[NodePort limitations]
NodePort exposes the service on *every* node. It is not recommended for production external access — use a LoadBalancer or Ingress instead.
:::

### LoadBalancer

Provisions a cloud load balancer (AWS ELB, GCP LB, Azure LB) automatically.

```yaml
apiVersion: v1
kind: Service
metadata:
  name: api-lb
  annotations:
    # AWS: internal load balancer
    service.beta.kubernetes.io/aws-load-balancer-internal: "true"
    # AWS: NLB instead of CLB
    service.beta.kubernetes.io/aws-load-balancer-type: nlb
spec:
  type: LoadBalancer
  selector:
    app: api
  ports:
    - port: 443
      targetPort: 3000
```

```bash
# Get the external IP / hostname
kubectl get service api-lb
# NAME     TYPE           CLUSTER-IP    EXTERNAL-IP                          PORT(S)
# api-lb   LoadBalancer   10.96.0.1     abc123.us-east-1.elb.amazonaws.com   443:32100/TCP
```

### ExternalName

Maps a Service to an external DNS name. No proxying — just a CNAME.

```yaml
apiVersion: v1
kind: Service
metadata:
  name: external-db
spec:
  type: ExternalName
  externalName: mydb.rds.amazonaws.com
```

Pods can use `external-db.myapp.svc.cluster.local` and the cluster DNS resolves it to the external RDS hostname.

### Headless Service

A Service with `clusterIP: None`. Returns all Pod IPs directly via DNS — no load balancing. Required by StatefulSets for per-pod DNS.

```yaml
apiVersion: v1
kind: Service
metadata:
  name: postgres-headless
spec:
  clusterIP: None    # headless
  selector:
    app: postgres
  ports:
    - port: 5432
```

DNS query for `postgres-headless` returns all Pod IPs. Individual pods addressable as `postgres-0.postgres-headless.myapp.svc.cluster.local`.

## Multi-Port Services

```yaml
apiVersion: v1
kind: Service
metadata:
  name: api
spec:
  selector:
    app: api
  ports:
    - name: http
      port: 80
      targetPort: 3000
    - name: metrics
      port: 9090
      targetPort: 9090
    - name: grpc
      port: 50051
      targetPort: 50051
```

## Session Affinity

Route requests from the same client IP to the same Pod:

```yaml
spec:
  sessionAffinity: ClientIP
  sessionAffinityConfig:
    clientIP:
      timeoutSeconds: 3600
```

## Endpoints and EndpointSlices

Services use Endpoints (or EndpointSlices in K8s 1.21+) to track Pod IP addresses.

```bash
# View endpoints for a service
kubectl get endpoints api
kubectl get endpointslices -l kubernetes.io/service-name=api

# If no endpoints, check that pod labels match service selector
kubectl get pods --show-labels
kubectl describe service api
```

## Ingress

Ingress manages external HTTP/HTTPS access to Services. A single Ingress can route traffic to multiple services based on host and path rules.

:::tip[Ingress requires a controller]
Ingress resources do nothing without an Ingress Controller installed in the cluster. Popular options: nginx-ingress, Traefik, AWS ALB Ingress Controller, GKE Ingress, Contour.
:::

```bash
# Install nginx-ingress controller
helm upgrade --install ingress-nginx ingress-nginx \
  --repo https://kubernetes.github.io/ingress-nginx \
  --namespace ingress-nginx --create-namespace
```

### Basic Ingress

```yaml
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: myapp-ingress
  namespace: myapp
  annotations:
    nginx.ingress.kubernetes.io/rewrite-target: /
spec:
  ingressClassName: nginx
  rules:
    - host: api.example.com
      http:
        paths:
          - path: /
            pathType: Prefix
            backend:
              service:
                name: api
                port:
                  number: 80
    - host: admin.example.com
      http:
        paths:
          - path: /
            pathType: Exact
            backend:
              service:
                name: admin
                port:
                  number: 80
  defaultBackend:
    service:
      name: not-found
      port:
        number: 80
```

### Path Types

| pathType | Behaviour |
|---|---|
| `Exact` | Matches the exact URL path (e.g., `/api` only, not `/api/v1`) |
| `Prefix` | Matches the path prefix (e.g., `/api` matches `/api`, `/api/v1`) |
| `ImplementationSpecific` | Up to the Ingress controller to decide |

### TLS Termination

```yaml
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: myapp-ingress-tls
  annotations:
    cert-manager.io/cluster-issuer: letsencrypt-prod
spec:
  ingressClassName: nginx
  tls:
    - hosts:
        - api.example.com
      secretName: api-tls-cert   # cert-manager will populate this
  rules:
    - host: api.example.com
      http:
        paths:
          - path: /
            pathType: Prefix
            backend:
              service:
                name: api
                port:
                  number: 80
```

### Nginx Annotations

```yaml
metadata:
  annotations:
    # Rate limiting
    nginx.ingress.kubernetes.io/limit-rps: "10"

    # Enable CORS
    nginx.ingress.kubernetes.io/enable-cors: "true"
    nginx.ingress.kubernetes.io/cors-allow-origin: "https://app.example.com"

    # Proxy timeouts
    nginx.ingress.kubernetes.io/proxy-read-timeout: "120"
    nginx.ingress.kubernetes.io/proxy-send-timeout: "120"

    # Client max body size (for file uploads)
    nginx.ingress.kubernetes.io/proxy-body-size: "50m"

    # SSL redirect
    nginx.ingress.kubernetes.io/ssl-redirect: "true"
    nginx.ingress.kubernetes.io/force-ssl-redirect: "true"

    # Basic auth
    nginx.ingress.kubernetes.io/auth-type: basic
    nginx.ingress.kubernetes.io/auth-secret: basic-auth
```

## cert-manager for Let's Encrypt

cert-manager automates TLS certificate issuance and renewal.

```bash
# Install cert-manager
helm repo add jetstack https://charts.jetstack.io
helm upgrade --install cert-manager jetstack/cert-manager \
  --namespace cert-manager \
  --create-namespace \
  --set crds.enabled=true
```

```yaml
# ClusterIssuer for Let's Encrypt
apiVersion: cert-manager.io/v1
kind: ClusterIssuer
metadata:
  name: letsencrypt-prod
spec:
  acme:
    server: https://acme-v02.api.letsencrypt.org/directory
    email: admin@example.com
    privateKeySecretRef:
      name: letsencrypt-prod-account-key
    solvers:
      - http01:
          ingress:
            ingressClassName: nginx
```

Once the ClusterIssuer exists, annotate any Ingress with `cert-manager.io/cluster-issuer: letsencrypt-prod` and cert-manager handles the rest.

## Gateway API (Modern Replacement for Ingress)

The Gateway API is the successor to Ingress, with richer routing semantics and role-based configuration.

```yaml
# GatewayClass — one per infrastructure type
apiVersion: gateway.networking.k8s.io/v1
kind: GatewayClass
metadata:
  name: nginx
spec:
  controllerName: k8s.nginx.org/nginx-gateway-controller

---
# Gateway — defines listeners (ports/protocols)
apiVersion: gateway.networking.k8s.io/v1
kind: Gateway
metadata:
  name: prod-gateway
  namespace: infra
spec:
  gatewayClassName: nginx
  listeners:
    - name: http
      port: 80
      protocol: HTTP
    - name: https
      port: 443
      protocol: HTTPS
      tls:
        mode: Terminate
        certificateRefs:
          - name: api-tls-cert

---
# HTTPRoute — routes for application teams
apiVersion: gateway.networking.k8s.io/v1
kind: HTTPRoute
metadata:
  name: api-route
  namespace: myapp
spec:
  parentRefs:
    - name: prod-gateway
      namespace: infra
  hostnames:
    - api.example.com
  rules:
    - matches:
        - path:
            type: PathPrefix
            value: /v2
      backendRefs:
        - name: api-v2
          port: 80
    - matches:
        - path:
            type: PathPrefix
            value: /
      backendRefs:
        - name: api-v1
          port: 80
          weight: 90
        - name: api-v2
          port: 80
          weight: 10   # canary: 10% to v2
```

## Service DNS Reference

```
<service>.<namespace>.svc.cluster.local
```

| Expression | Resolves from |
|---|---|
| `api` | same namespace |
| `api.myapp` | any namespace |
| `api.myapp.svc` | any namespace |
| `api.myapp.svc.cluster.local` | any namespace (FQDN) |
| `postgres-0.postgres.myapp.svc.cluster.local` | StatefulSet pod FQDN |

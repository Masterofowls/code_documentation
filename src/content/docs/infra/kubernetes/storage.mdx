# Kubernetes Storage

Kubernetes provides a rich storage subsystem that abstracts underlying storage infrastructure behind a uniform API. Understanding the PV/PVC model is essential for running stateful workloads.

## The PV/PVC Model

```
StorageClass ──► PersistentVolume (PV)
                       ▲
                       │ bound
                 PersistentVolumeClaim (PVC)
                       ▲
                       │ uses
                     Pod
```

- **PersistentVolume (PV)** — a piece of storage provisioned by an admin or dynamically. Lives independently of pods.
- **PersistentVolumeClaim (PVC)** — a request for storage by a user. Specifies size and access mode.
- **StorageClass** — defines how dynamic PVs are provisioned (which provisioner, performance tier, reclaim policy).

## StorageClass

StorageClasses enable **dynamic provisioning** — PVs are created automatically when a PVC is submitted.

```yaml
apiVersion: storage.k8s.io/v1
kind: StorageClass
metadata:
  name: fast-ssd
  annotations:
    storageclass.kubernetes.io/is-default-class: 'true' # make default

# AWS EBS (gp3)
provisioner: ebs.csi.aws.com
parameters:
  type: gp3
  iops: '3000'
  throughput: '125'
  encrypted: 'true'
  kmsKeyId: arn:aws:kms:us-east-1:123456:key/...

reclaimPolicy: Delete # Delete | Retain
volumeBindingMode: WaitForFirstConsumer # Immediate | WaitForFirstConsumer
allowVolumeExpansion: true
mountOptions:
  - debug
```

```yaml
# GKE — standard persistent disk
apiVersion: storage.k8s.io/v1
kind: StorageClass
metadata:
  name: standard
provisioner: pd.csi.storage.gke.io
parameters:
  type: pd-ssd
reclaimPolicy: Delete
volumeBindingMode: WaitForFirstConsumer
allowVolumeExpansion: true
```

```yaml
# Azure Managed Disk
apiVersion: storage.k8s.io/v1
kind: StorageClass
metadata:
  name: azure-ssd
provisioner: disk.csi.azure.com
parameters:
  skuName: Premium_LRS
  kind: Managed
reclaimPolicy: Delete
volumeBindingMode: WaitForFirstConsumer
allowVolumeExpansion: true
```

### ReclaimPolicy

| Policy    | Behaviour                                                                   |
| --------- | --------------------------------------------------------------------------- |
| `Delete`  | PV and underlying storage deleted when PVC is deleted (default for dynamic) |
| `Retain`  | PV kept after PVC deletion; manual cleanup required; data preserved         |
| `Recycle` | Deprecated — basic scrub and reuse                                          |

### VolumeBindingMode

| Mode                   | Behaviour                                                          |
| ---------------------- | ------------------------------------------------------------------ |
| `Immediate`            | PV provisioned immediately when PVC created (before pod scheduled) |
| `WaitForFirstConsumer` | PV provisioned after pod scheduled (ensures same AZ as pod)        |

:::tip[Use WaitForFirstConsumer]
Always use `volumeBindingMode: WaitForFirstConsumer` for cloud volumes. `Immediate` may provision the volume in a different availability zone than where the pod is scheduled, causing mount failures.
:::

## PersistentVolume (PV)

Usually created by dynamic provisioning, but can be pre-provisioned by an admin:

```yaml
apiVersion: v1
kind: PersistentVolume
metadata:
  name: manual-pv-001
  labels:
    type: local
spec:
  storageClassName: manual
  capacity:
    storage: 100Gi
  accessModes:
    - ReadWriteOnce
  persistentVolumeReclaimPolicy: Retain

  # NFS example
  nfs:
    path: /exports/data
    server: 192.168.1.100

  # Local volume (node-specific)
  local:
    path: /mnt/data
  nodeAffinity:
    required:
      nodeSelectorTerms:
        - matchExpressions:
            - key: kubernetes.io/hostname
              operator: In
              values: [node-01]
```

## PersistentVolumeClaim (PVC)

```yaml
apiVersion: v1
kind: PersistentVolumeClaim
metadata:
  name: postgres-data
  namespace: production
  labels:
    app: postgres
spec:
  storageClassName: fast-ssd # omit to use default StorageClass
  accessModes:
    - ReadWriteOnce
  resources:
    requests:
      storage: 50Gi
  # Optionally select a specific PV by label
  selector:
    matchLabels:
      type: ssd
```

### Using PVC in a Pod

```yaml
spec:
  containers:
    - name: postgres
      image: postgres:16
      volumeMounts:
        - name: data
          mountPath: /var/lib/postgresql/data

  volumes:
    - name: data
      persistentVolumeClaim:
        claimName: postgres-data
        readOnly: false
```

### Check PVC Status

```bash
# PVC lifecycle: Pending → Bound
kubectl get pvc -n production
kubectl describe pvc postgres-data -n production

# Check which PV is bound to the PVC
kubectl get pvc postgres-data -o jsonpath='{.spec.volumeName}'
```

## Access Modes

| Mode               | Short | Multiple Nodes                   |
| ------------------ | ----- | -------------------------------- |
| `ReadWriteOnce`    | RWO   | One node reads/writes            |
| `ReadOnlyMany`     | ROX   | Many nodes read                  |
| `ReadWriteMany`    | RWX   | Many nodes read/write            |
| `ReadWriteOncePod` | RWOP  | One pod reads/writes (K8s 1.22+) |

:::caution[RWX Storage]
Most block storage (EBS, Azure Disk, GCE PD) only supports `ReadWriteOnce`. For `ReadWriteMany` (shared storage), you need NFS, CephFS, Azure Files, or a cloud-native shared filesystem.
:::

## StatefulSet volumeClaimTemplates

Each pod in a StatefulSet gets its own PVC via `volumeClaimTemplates`:

```yaml
apiVersion: apps/v1
kind: StatefulSet
metadata:
  name: postgres-cluster
spec:
  serviceName: postgres-headless
  replicas: 3
  selector:
    matchLabels:
      app: postgres
  template:
    metadata:
      labels:
        app: postgres
    spec:
      containers:
        - name: postgres
          image: postgres:16
          volumeMounts:
            - name: data # references the volumeClaimTemplate name
              mountPath: /var/lib/postgresql/data

  volumeClaimTemplates:
    - metadata:
        name: data
      spec:
        accessModes: [ReadWriteOnce]
        storageClassName: fast-ssd
        resources:
          requests:
            storage: 20Gi
```

This creates PVCs: `data-postgres-cluster-0`, `data-postgres-cluster-1`, `data-postgres-cluster-2`

## CSI Drivers

Container Storage Interface (CSI) is the standard for plugging storage into Kubernetes. Popular drivers:

| Driver                       | Storage                     |
| ---------------------------- | --------------------------- |
| `ebs.csi.aws.com`            | AWS EBS                     |
| `efs.csi.aws.com`            | AWS EFS (NFS, RWX)          |
| `pd.csi.storage.gke.io`      | GCP Persistent Disk         |
| `disk.csi.azure.com`         | Azure Managed Disk          |
| `file.csi.azure.com`         | Azure Files (RWX)           |
| `nfs.csi.k8s.io`             | NFS                         |
| `driver.longhorn.io`         | Longhorn (Rancher, on-prem) |
| `rook-ceph.rbd.csi.ceph.com` | Rook/Ceph (on-prem)         |

## Volume Types for Pods

### emptyDir

Ephemeral storage that lives as long as the pod. Shared between all containers in the pod.

```yaml
volumes:
  - name: cache
    emptyDir: {}

  # Size-limited emptyDir (uses node storage)
  - name: tmp
    emptyDir:
      sizeLimit: 1Gi

  # Memory-backed (tmpfs) — fast but uses node RAM
  - name: ramdisk
    emptyDir:
      medium: Memory
      sizeLimit: 256Mi
```

Use cases:

- Shared scratch space between containers in the same pod
- Caching downloaded data
- Checkpoint data during long computations

### hostPath

Mounts a directory from the host node's filesystem.

```yaml
volumes:
  - name: docker-socket
    hostPath:
      path: /var/run/docker.sock
      type: Socket

  - name: host-proc
    hostPath:
      path: /proc
      type: Directory
```

:::caution[hostPath Security Risk]
`hostPath` mounts give containers direct access to the host filesystem — a major security risk. Avoid unless absolutely necessary (e.g., DaemonSet log collectors, privileged system agents). Kubernetes Pod Security Standards restrict `hostPath` in restricted mode.
:::

### ConfigMap and Secret as Volumes

```yaml
volumes:
  - name: config
    configMap:
      name: app-config
      defaultMode: 0644

  - name: certs
    secret:
      secretName: tls-certs
      defaultMode: 0400 # read-only by owner (important for TLS keys)
```

## Volume Snapshots

Volume Snapshots provide point-in-time copies of PVCs.

```yaml
# VolumeSnapshotClass — defines the snapshot driver
apiVersion: snapshot.storage.k8s.io/v1
kind: VolumeSnapshotClass
metadata:
  name: ebs-vsc
driver: ebs.csi.aws.com
deletionPolicy: Delete # Delete | Retain
```

```yaml
# Create a snapshot from a PVC
apiVersion: snapshot.storage.k8s.io/v1
kind: VolumeSnapshot
metadata:
  name: postgres-snapshot-2024-01-15
spec:
  volumeSnapshotClassName: ebs-vsc
  source:
    persistentVolumeClaimName: postgres-data
```

```yaml
# Restore from snapshot — create new PVC from snapshot
apiVersion: v1
kind: PersistentVolumeClaim
metadata:
  name: postgres-data-restored
spec:
  storageClassName: fast-ssd
  accessModes: [ReadWriteOnce]
  resources:
    requests:
      storage: 50Gi
  dataSource:
    name: postgres-snapshot-2024-01-15
    kind: VolumeSnapshot
    apiGroup: snapshot.storage.k8s.io
```

```bash
# Check snapshot status
kubectl get volumesnapshot
kubectl describe volumesnapshot postgres-snapshot-2024-01-15

# Should show ReadyToUse: true when complete
```

## Expanding Volumes

If `allowVolumeExpansion: true` in the StorageClass:

```bash
# Edit the PVC to increase size
kubectl edit pvc postgres-data
# Change .spec.resources.requests.storage from 50Gi to 100Gi

# Or patch directly
kubectl patch pvc postgres-data -p '{"spec":{"resources":{"requests":{"storage":"100Gi"}}}}'

# Check expansion status
kubectl get pvc postgres-data
# CAPACITY column will show new size once complete (pod may need restart for filesystem expand)
```

## Longhorn (On-Premises Distributed Storage)

Longhorn is a cloud-native distributed block storage system for Kubernetes, ideal for on-premises clusters.

```bash
# Install Longhorn
kubectl apply -f https://raw.githubusercontent.com/longhorn/longhorn/v1.6.0/deploy/longhorn.yaml

# Access dashboard
kubectl port-forward -n longhorn-system svc/longhorn-frontend 8080:80
```

```yaml
# Longhorn StorageClass
apiVersion: storage.k8s.io/v1
kind: StorageClass
metadata:
  name: longhorn
provisioner: driver.longhorn.io
parameters:
  numberOfReplicas: '3' # data replicated across 3 nodes
  staleReplicaTimeout: '2880'
  fromBackup: ''
reclaimPolicy: Delete
volumeBindingMode: Immediate
allowVolumeExpansion: true
```

## Storage Capacity Tracking

Kubernetes 1.24+ tracks available storage per node, helping the scheduler make better pod placement decisions:

```bash
# Check storage capacity per node
kubectl get csistoragecapacities -A

# Get storage class capacity info
kubectl get csistoragecapacities -n kube-system \
  -o custom-columns="CLASS:.storageClassName,CAPACITY:.capacity,NODE:.nodeTopology.matchLabels"
```

## Useful Storage Commands

```bash
# List all PVs in the cluster
kubectl get pv
kubectl get pv --sort-by=.spec.capacity.storage

# List all PVCs in all namespaces
kubectl get pvc -A

# See which pods are using a PVC
kubectl get pods -A -o json | jq '.items[] | select(.spec.volumes[]?.persistentVolumeClaim.claimName == "my-pvc") | .metadata.name'

# Manually release a retained PV (to reuse it)
kubectl patch pv my-pv -p '{"spec":{"claimRef":null}}'

# Delete a stuck PVC (remove finalizer)
kubectl patch pvc my-pvc -p '{"metadata":{"finalizers":null}}'

# Force delete a stuck PV
kubectl delete pv my-pv --grace-period=0 --force
```

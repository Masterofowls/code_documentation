# Storage in Kubernetes

Kubernetes decouples storage provisioning from consumption. Administrators create PersistentVolumes (PVs) or configure dynamic provisioning via StorageClasses. Developers request storage with PersistentVolumeClaims (PVCs).

## Volume Types Overview

| Volume Type | Lifetime | Shared | Use Case |
|---|---|---|---|
| `emptyDir` | Pod lifetime | Between containers in same pod | Scratch space, caches, sidecar sharing |
| `hostPath` | Node lifetime | Pods on same node | DaemonSets, node-level data |
| `configMap` | K8s object | All pods mounting it | Configuration files |
| `secret` | K8s object | All pods mounting it | Credentials, certificates |
| `persistentVolumeClaim` | PV lifetime | Access mode dependent | Databases, stateful apps |
| `nfs` | NFS server | All pods | Shared read/write storage |
| `csi` | Driver-defined | Driver-defined | Cloud disks, SAN, distributed storage |

## emptyDir

An `emptyDir` volume is created when a Pod starts and deleted when it stops. Containers in the same Pod can share it.

```yaml
spec:
  containers:
    - name: app
      volumeMounts:
        - name: cache
          mountPath: /app/cache
        - name: shared-logs
          mountPath: /var/log/app

    - name: log-shipper
      volumeMounts:
        - name: shared-logs
          mountPath: /var/log/app
          readOnly: true

  volumes:
    - name: cache
      emptyDir:
        sizeLimit: 1Gi          # optional size limit
        medium: ""              # "" = disk, "Memory" = RAM (tmpfs)

    - name: shared-logs
      emptyDir: {}

    # In-memory emptyDir (tmpfs)
    - name: secure-tmp
      emptyDir:
        medium: Memory
        sizeLimit: 128Mi
```

## hostPath

Mounts a file or directory from the host node. Use sparingly — it tightly couples the Pod to a specific node.

```yaml
volumes:
  - name: host-data
    hostPath:
      path: /var/lib/myapp/data
      type: DirectoryOrCreate   # Directory | DirectoryOrCreate | File | FileOrCreate | Socket | CharDevice | BlockDevice
```

:::caution[hostPath security risk]
`hostPath` volumes bypass container isolation and give the Pod direct access to the host filesystem. Prefer PersistentVolumes for stateful data. DaemonSets (e.g., log collectors) are the primary legitimate use case.
:::

## PersistentVolume (PV)

A PersistentVolume is a piece of storage provisioned by an administrator or dynamically by a StorageClass. PVs exist independently of any Pod.

```yaml
apiVersion: v1
kind: PersistentVolume
metadata:
  name: manual-pv
spec:
  capacity:
    storage: 20Gi
  volumeMode: Filesystem    # Filesystem | Block
  accessModes:
    - ReadWriteOnce
  persistentVolumeReclaimPolicy: Retain   # Retain | Delete | Recycle
  storageClassName: manual
  nfs:
    server: nfs.example.com
    path: /exports/myapp
```

## PersistentVolumeClaim (PVC)

A PVC is a request for storage by a user. Kubernetes binds it to a matching PV.

```yaml
apiVersion: v1
kind: PersistentVolumeClaim
metadata:
  name: postgres-data
  namespace: myapp
spec:
  accessModes:
    - ReadWriteOnce
  storageClassName: fast    # "" = use default, "manual" = static PV
  volumeMode: Filesystem
  resources:
    requests:
      storage: 20Gi
```

### Access Modes

| Mode | Abbrev | Description |
|---|---|---|
| `ReadWriteOnce` | RWO | Single node read-write |
| `ReadOnlyMany` | ROX | Many nodes read-only |
| `ReadWriteMany` | RWX | Many nodes read-write |
| `ReadWriteOncePod` | RWOP | Single pod read-write (K8s 1.22+) |

:::tip[EBS only supports RWO]
AWS EBS volumes only support `ReadWriteOnce`. For shared read-write storage on AWS, use EFS (NFS) or a distributed storage layer like Longhorn.
:::

### Using PVC in a Pod

```yaml
spec:
  containers:
    - name: postgres
      image: postgres:16-alpine
      volumeMounts:
        - name: data
          mountPath: /var/lib/postgresql/data
  volumes:
    - name: data
      persistentVolumeClaim:
        claimName: postgres-data
        readOnly: false
```

## StorageClass

StorageClasses enable dynamic PV provisioning. When a PVC references a StorageClass, a PV is automatically created.

```yaml
apiVersion: storage.k8s.io/v1
kind: StorageClass
metadata:
  name: fast
  annotations:
    storageclass.kubernetes.io/is-default-class: "true"   # set as default
provisioner: ebs.csi.aws.com      # CSI driver
reclaimPolicy: Delete             # Delete | Retain
allowVolumeExpansion: true
volumeBindingMode: WaitForFirstConsumer   # Immediate | WaitForFirstConsumer
parameters:
  type: gp3
  iops: "3000"
  throughput: "125"
  encrypted: "true"
```

### volumeBindingMode

| Mode | Behaviour |
|---|---|
| `Immediate` | PV provisioned as soon as PVC is created |
| `WaitForFirstConsumer` | PV provisioned only when a Pod is scheduled — ensures same AZ |

:::tip[Use WaitForFirstConsumer]
`WaitForFirstConsumer` is almost always preferable for cloud environments. `Immediate` may provision the disk in a different availability zone than the pod, causing scheduling failures.
:::

```bash
# List storage classes
kubectl get storageclasses
kubectl get sc

# Get default storage class
kubectl get sc -o jsonpath='{.items[?(@.metadata.annotations.storageclass\.kubernetes\.io/is-default-class=="true")].metadata.name}'

# Check PVC status (Pending = not yet bound)
kubectl get pvc -n myapp
kubectl describe pvc postgres-data -n myapp
```

## Dynamic Provisioning Flow

```
Developer creates PVC
        │
        ▼
Kubernetes finds matching StorageClass
        │
        ▼
StorageClass provisioner called (e.g., EBS CSI driver)
        │
        ▼
Cloud disk created and PV created
        │
        ▼
PVC bound to PV (STATUS: Bound)
        │
        ▼
Pod scheduled; disk attached to node and mounted
```

## StatefulSet volumeClaimTemplates

Each StatefulSet replica gets its own PVC automatically from the template.

```yaml
apiVersion: apps/v1
kind: StatefulSet
metadata:
  name: mongo
spec:
  serviceName: mongo
  replicas: 3
  selector:
    matchLabels:
      app: mongo
  template:
    metadata:
      labels:
        app: mongo
    spec:
      containers:
        - name: mongo
          image: mongo:7
          volumeMounts:
            - name: data
              mountPath: /data/db
            - name: config
              mountPath: /data/configdb

  # Kubernetes creates one PVC per replica
  # PVCs: data-mongo-0, data-mongo-1, data-mongo-2
  volumeClaimTemplates:
    - metadata:
        name: data
      spec:
        accessModes: ["ReadWriteOnce"]
        storageClassName: fast
        resources:
          requests:
            storage: 50Gi
    - metadata:
        name: config
      spec:
        accessModes: ["ReadWriteOnce"]
        storageClassName: standard
        resources:
          requests:
            storage: 1Gi
```

## CSI Drivers

Container Storage Interface (CSI) is the standard plugin interface for storage systems.

| Driver | Provider | Volumes |
|---|---|---|
| `ebs.csi.aws.com` | AWS | EBS gp2/gp3/io1/st1 |
| `efs.csi.aws.com` | AWS | EFS (NFS, RWX) |
| `pd.csi.storage.gke.io` | GCP | Persistent Disk |
| `disk.csi.azure.com` | Azure | Azure Disk |
| `file.csi.azure.com` | Azure | Azure Files (NFS, RWX) |
| `driver.longhorn.io` | Self-hosted | Distributed block |
| `rook-ceph.rbd.csi.ceph.com` | Self-hosted | Ceph RBD block |
| `rook-ceph.cephfs.csi.ceph.com` | Self-hosted | CephFS (RWX) |
| `nfs.csi.k8s.io` | NFS server | NFS (RWX) |

### Installing AWS EBS CSI Driver

```bash
# Via Helm
helm repo add aws-ebs-csi-driver https://kubernetes-sigs.github.io/aws-ebs-csi-driver
helm upgrade --install aws-ebs-csi-driver aws-ebs-csi-driver/aws-ebs-csi-driver \
  --namespace kube-system \
  --set controller.serviceAccount.annotations."eks\.amazonaws\.com/role-arn"=arn:aws:iam::ACCOUNT:role/EBSCSIRole
```

## Volume Snapshots

Volume snapshots allow point-in-time copies of PVCs.

```yaml
# VolumeSnapshotClass
apiVersion: snapshot.storage.k8s.io/v1
kind: VolumeSnapshotClass
metadata:
  name: ebs-snapshots
driver: ebs.csi.aws.com
deletionPolicy: Delete

---
# Create a snapshot of a PVC
apiVersion: snapshot.storage.k8s.io/v1
kind: VolumeSnapshot
metadata:
  name: postgres-snapshot-20240115
  namespace: myapp
spec:
  volumeSnapshotClassName: ebs-snapshots
  source:
    persistentVolumeClaimName: postgres-data

---
# Restore from snapshot into a new PVC
apiVersion: v1
kind: PersistentVolumeClaim
metadata:
  name: postgres-restored
spec:
  accessModes: ["ReadWriteOnce"]
  storageClassName: fast
  resources:
    requests:
      storage: 20Gi
  dataSource:
    name: postgres-snapshot-20240115
    kind: VolumeSnapshot
    apiGroup: snapshot.storage.k8s.io
```

```bash
# List snapshots
kubectl get volumesnapshots -n myapp
kubectl describe volumesnapshot postgres-snapshot-20240115 -n myapp
```

## Expanding Volumes

```bash
# Edit PVC to request more storage (requires allowVolumeExpansion: true)
kubectl edit pvc postgres-data -n myapp
# Change: resources.requests.storage from 20Gi to 40Gi

# Check expansion status
kubectl get pvc postgres-data -n myapp
kubectl describe pvc postgres-data -n myapp | grep -A 5 "Conditions"
```

## ConfigMap and Secret as Volumes

```yaml
volumes:
  # ConfigMap as directory of files
  - name: app-config
    configMap:
      name: app-config
      defaultMode: 0444

  # Secret as directory of files
  - name: tls-certs
    secret:
      secretName: api-tls
      defaultMode: 0400
      items:
        - key: tls.crt
          path: server.crt
        - key: tls.key
          path: server.key
```

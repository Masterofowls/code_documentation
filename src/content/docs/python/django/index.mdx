Django is a high-level Python web framework that encourages rapid development and clean, pragmatic design. Built by experienced developers, it takes care of much of the hassle of web development so you can focus on writing your app without needing to reinvent the wheel.

## What Is Django?

Django follows the **"batteries-included"** philosophy — it ships with everything you need to build a full-featured web application out of the box:

- An **ORM** for database interaction without writing raw SQL
- A **built-in admin interface** auto-generated from your models
- A robust **authentication system** with users, groups, and permissions
- **Form handling** with validation and rendering
- A **template engine** with inheritance and filters
- **Migrations** to manage database schema changes over time
- **URL routing**, **middleware**, **caching**, **i18n**, and much more

> Django was originally created in 2003 at the Lawrence Journal-World newspaper and open-sourced in 2005. It powers Instagram, Pinterest, Disqus, Mozilla, and the Washington Post.

## MTV Architecture

Django uses the **Model–Template–View** pattern (analogous to MVC):

```
┌─────────────────────────────────────────────────────────────────┐
│                        HTTP Request                             │
└───────────────────────────┬─────────────────────────────────────┘
                            │
                            ▼
┌─────────────────────────────────────────────────────────────────┐
│                        URL Dispatcher                           │
│              urls.py  ──  path() / re_path()                    │
└───────────────────────────┬─────────────────────────────────────┘
                            │ matches URL pattern
                            ▼
┌─────────────────────────────────────────────────────────────────┐
│                           VIEW                                  │
│         views.py  ──  function or class-based view              │
│   • Receives HttpRequest                                        │
│   • Calls Model layer to read/write data                        │
│   • Passes context dict to Template                             │
│   • Returns HttpResponse                                        │
└──────────┬────────────────────────────────────────┬────────────┘
           │ queries                                │ renders
           ▼                                        ▼
┌──────────────────────┐              ┌─────────────────────────┐
│        MODEL         │              │        TEMPLATE         │
│  models.py           │              │  *.html / *.mjml        │
│  • Python classes    │              │  • Django template lang │
│  • ORM query API     │              │  • {{ variables }}      │
│  • DB abstraction    │              │  • {% tags %}           │
│  • Migrations        │              │  • Inheritance          │
└──────────────────────┘              └─────────────────────────┘
           │                                        │
           ▼                                        ▼
┌──────────────────────┐              ┌─────────────────────────┐
│      DATABASE        │              │      HTTP Response       │
│  PostgreSQL / SQLite │              │   HTML / JSON / File     │
└──────────────────────┘              └─────────────────────────┘
```

**Model** — Defines data structure and handles database interaction via the ORM.

**Template** — Defines how data is presented (HTML with Django template language).

**View** — Contains business logic; reads from models, renders templates.

## Key Features at a Glance

| Feature | Description |
|---------|-------------|
| **ORM** | Map Python classes to database tables; write queries in Python |
| **Admin** | Auto-generated CRUD interface for all registered models |
| **Auth** | Users, groups, permissions, password hashing, session management |
| **Forms** | Declarative form classes with validation and HTML rendering |
| **Migrations** | Version-controlled, auto-generated schema change scripts |
| **Middleware** | Hooks into request/response lifecycle globally |
| **Caching** | Pluggable backends: in-memory, Redis, Memcached, DB |
| **i18n / l10n** | Built-in translation framework and locale formatting |
| **Security** | CSRF, XSS, SQL injection, clickjacking protection by default |
| **Signals** | Decoupled notification system between components |
| **Static files** | Management and serving of CSS, JS, images |
| **Testing** | TestCase classes built on unittest with request factory |

## Django vs Flask vs FastAPI

<Aside type="note" title="Choosing a Framework">
All three frameworks are production-ready. Your choice depends on project size, team experience, and required features.
</Aside>

| Criteria | Django | Flask | FastAPI |
|----------|--------|-------|---------|
| **Philosophy** | Batteries-included | Micro-framework | Modern async micro |
| **Learning curve** | Moderate | Low | Low–Moderate |
| **ORM** | Built-in (powerful) | Extension (SQLAlchemy) | Extension (SQLAlchemy) |
| **Admin panel** | ✅ Auto-generated | ❌ Extension needed | ❌ Extension needed |
| **Auth system** | ✅ Built-in | ❌ Flask-Login | ❌ Depends on library |
| **Async support** | Partial (Django 3.1+) | Limited | ✅ First-class |
| **Performance** | Good | Good | Excellent |
| **REST API** | DRF extension | Flask-RESTful / manual | ✅ Built-in |
| **Type hints** | Optional | Optional | ✅ Required / core |
| **Auto docs** | DRF Spectacular | Flasgger | ✅ Built-in Swagger |
| **Best for** | Full-stack apps, CMS, dashboards | Small APIs, prototypes | High-perf APIs, ML services |
| **Companies** | Instagram, Mozilla | Netflix (some), LinkedIn | Uber (some), Microsoft |

:::tip[When to Choose Django]
Choose Django when you need a complete solution: user authentication, admin interface, ORM with migrations, and forms — all working together out of the box with sensible defaults.
:::

:::caution[When NOT to Choose Django]
If you're building a purely async, high-throughput microservice or a project where you want full control over every component, consider FastAPI or Flask instead.
:::

## Hello World in 5 Minutes

<Steps>

1. **Install Django**

   ```bash
   pip install django
   django-admin startproject mysite
   cd mysite
   python manage.py startapp core
   ```

2. **Write your first view** — `core/views.py`

   ```python
   from django.http import HttpResponse


   def hello_world(request):
       name = request.GET.get("name", "World")
       return HttpResponse(f"<h1>Hello, {name}!</h1>")
   ```

3. **Wire up the URL** — `core/urls.py` (create this file)

   ```python
   from django.urls import path
   from . import views

   urlpatterns = [
       path("hello/", views.hello_world, name="hello"),
   ]
   ```

4. **Include in project URLs** — `mysite/urls.py`

   ```python
   from django.contrib import admin
   from django.urls import path, include

   urlpatterns = [
       path("admin/", admin.site.urls),
       path("", include("core.urls")),
   ]
   ```

5. **Register the app and run**

   ```python
   # mysite/settings.py
   INSTALLED_APPS = [
       # ... default apps ...
       "core",  # add this
   ]
   ```

   ```bash
   python manage.py runserver
   # Visiting http://127.0.0.1:8000/hello/?name=Django
   # → <h1>Hello, Django!</h1>
   ```

</Steps>

## A Slightly More Real Example

Here's a complete mini blog view with a model, a template, and URL — showing how all three layers interact:

```python
# core/models.py
from django.db import models


class Post(models.Model):
    title = models.CharField(max_length=200)
    content = models.TextField()
    published = models.BooleanField(default=False)
    created_at = models.DateTimeField(auto_now_add=True)

    class Meta:
        ordering = ["-created_at"]

    def __str__(self):
        return self.title
```

```python
# core/views.py
from django.shortcuts import render
from .models import Post


def post_list(request):
    posts = Post.objects.filter(published=True)
    return render(request, "core/post_list.html", {"posts": posts})
```

```html
<!-- core/templates/core/post_list.html -->
<!DOCTYPE html>
<html>
<head><title>Blog</title></head>
<body>
  <h1>Latest Posts</h1>
  {% for post in posts %}
    <article>
      <h2>{{ post.title }}</h2>
      <p>{{ post.content }}</p>
      <small>{{ post.created_at|date:"N j, Y" }}</small>
    </article>
  {% empty %}
    <p>No posts yet.</p>
  {% endfor %}
</body>
</html>
```

```python
# core/urls.py
from django.urls import path
from . import views

urlpatterns = [
    path("posts/", views.post_list, name="post-list"),
]
```

:::tip[Run Migrations]
After creating models, always run `python manage.py makemigrations` and `python manage.py migrate` before starting the server.
:::

## Quick Reference: Most Used Commands

```bash
# Project & app management
django-admin startproject <name>
python manage.py startapp <name>

# Database
python manage.py makemigrations
python manage.py migrate
python manage.py showmigrations
python manage.py sqlmigrate <app> <migration>

# Development server
python manage.py runserver
python manage.py runserver 0.0.0.0:8000

# Shell
python manage.py shell
python manage.py shell_plus  # with django-extensions

# Admin user
python manage.py createsuperuser

# Static files
python manage.py collectstatic

# Testing
python manage.py test
python manage.py test <app>

# Deployment check
python manage.py check --deploy
```

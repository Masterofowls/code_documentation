## Virtual Environment Best Practice

Always isolate Django projects in a virtual environment. This prevents dependency conflicts between projects.

<Tabs>
<TabItem label="venv (built-in)">
```bash
# Create a virtual environment
python -m venv .venv

# Activate it
# Windows
.venv\Scripts\activate

# macOS / Linux
source .venv/bin/activate

# Confirm Python path
which python
# /path/to/project/.venv/bin/python

# Deactivate when done
deactivate
```
</TabItem>
<TabItem label="uv (faster)">
```bash
# Install uv
pip install uv

# Create and activate virtual environment
uv venv .venv
source .venv/bin/activate  # or .venv\Scripts\activate on Windows

# Install packages (much faster than pip)
uv pip install django
```
</TabItem>
<TabItem label="pyenv + venv">
```bash
# Install a specific Python version
pyenv install 3.12.3
pyenv local 3.12.3

# Create venv with that Python
python -m venv .venv
source .venv/bin/activate
```
</TabItem>
</Tabs>

:::tip[Use .python-version]
Add a `.python-version` file to your project root with the Python version string (e.g., `3.12.3`) so `pyenv` automatically switches when you enter the directory.
:::

## Installing Django

```bash
# Install latest stable Django
pip install django

# Install a specific version
pip install "django==4.2.*"   # LTS — recommended for production
pip install "django==5.1.*"   # Latest

# Verify installation
python -m django --version
# 5.1.4

# Save dependencies
pip freeze > requirements.txt
```

A minimal `requirements.txt` for a new project:

```text
Django==5.1.4
python-decouple==3.8
psycopg[binary]==3.2.3
gunicorn==23.0.0
```

## Creating a Project

```bash
django-admin startproject mysite
cd mysite
```

:::caution[Project Naming]
Avoid naming your project `django` (conflicts with the framework itself) or names that conflict with Python built-ins like `test`, `site`, or `types`.
:::

This creates the following structure:

```
mysite/
├── manage.py              ← CLI utility for this project
└── mysite/                ← Python package for project config
    ├── __init__.py        ← Marks this as a Python package
    ├── asgi.py            ← ASGI entry point (async servers)
    ├── settings.py        ← All configuration lives here
    ├── urls.py            ← Root URL configuration
    └── wsgi.py            ← WSGI entry point (gunicorn, uWSGI)
```

## Creating an Application

A Django **project** is a collection of **apps**. Each app is a self-contained module with its own models, views, templates, and URLs.

```bash
python manage.py startapp blog
```

This creates:

```
blog/
├── __init__.py            ← Python package marker
├── admin.py               ← Register models with admin
├── apps.py                ← AppConfig class
├── migrations/            ← Auto-generated migration files
│   └── __init__.py
├── models.py              ← Database models
├── tests.py               ← Test cases
└── views.py               ← View functions / classes
```

Typical additions you'll create manually:

```
blog/
├── urls.py                ← App-level URL patterns
├── forms.py               ← Form classes
├── serializers.py         ← DRF serializers
├── signals.py             ← Django signal handlers
├── managers.py            ← Custom model managers
└── templates/
    └── blog/              ← Namespace templates by app name
        ├── post_list.html
        └── post_detail.html
```

### Register the App

```python
# mysite/settings.py
INSTALLED_APPS = [
    "django.contrib.admin",
    "django.contrib.auth",
    "django.contrib.contenttypes",
    "django.contrib.sessions",
    "django.contrib.messages",
    "django.contrib.staticfiles",
    # ↓ Your apps
    "blog",
]
```

:::tip[Using AppConfig]
Prefer using the dotted path to the `AppConfig` class for better configurability:

```python
INSTALLED_APPS = [
    # ...
    "blog.apps.BlogConfig",
]
```

The `BlogConfig` class in `blog/apps.py` lets you configure app-specific settings and run startup code.
:::

## Project Structure (Full)

A real-world project typically looks like this:

```
mysite/
├── .env                   ← Secret environment variables (NOT in git)
├── .env.example           ← Template for .env (committed to git)
├── .gitignore
├── manage.py
├── requirements.txt
├── requirements-dev.txt
│
├── mysite/                ← Project config package
│   ├── __init__.py
│   ├── settings/          ← Split settings (optional but recommended)
│   │   ├── __init__.py
│   │   ├── base.py        ← Shared settings
│   │   ├── development.py ← Dev overrides
│   │   └── production.py  ← Prod overrides
│   ├── urls.py
│   ├── asgi.py
│   └── wsgi.py
│
├── blog/                  ← Example app
│   ├── migrations/
│   ├── templates/blog/
│   ├── admin.py
│   ├── apps.py
│   ├── forms.py
│   ├── models.py
│   ├── urls.py
│   └── views.py
│
├── users/                 ← Custom user app (recommended)
│   ├── migrations/
│   ├── models.py
│   └── ...
│
├── static/                ← Project-level static files
│   ├── css/
│   ├── js/
│   └── images/
│
├── media/                 ← User-uploaded files
│
└── templates/             ← Project-level templates
    └── base.html
```

## settings.py Walkthrough

Django's settings file controls every aspect of your project. Here's each critical section explained:

### Security Keys

```python
# mysite/settings.py
import os
from pathlib import Path

BASE_DIR = Path(__file__).resolve().parent.parent

# SECURITY WARNING: keep the secret key used in production secret!
# This key signs cookies, sessions, CSRF tokens, and password reset links.
SECRET_KEY = "django-insecure-xxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxx"

# SECURITY WARNING: don't run with debug turned on in production!
DEBUG = True

# Hosts/domain names that Django can serve
# In production: ALLOWED_HOSTS = ["myapp.com", "www.myapp.com"]
ALLOWED_HOSTS = []
```

### Installed Applications

```python
INSTALLED_APPS = [
    # Django built-in apps
    "django.contrib.admin",          # Admin site
    "django.contrib.auth",           # Authentication framework
    "django.contrib.contenttypes",   # Content type framework
    "django.contrib.sessions",       # Session framework
    "django.contrib.messages",       # Flash messaging
    "django.contrib.staticfiles",    # Static file management
    # Third-party
    "rest_framework",                # Django REST Framework (if installed)
    # Your apps
    "blog.apps.BlogConfig",
]
```

### Middleware

```python
MIDDLEWARE = [
    "django.middleware.security.SecurityMiddleware",        # Security headers
    "django.contrib.sessions.middleware.SessionMiddleware", # Session handling
    "django.middleware.common.CommonMiddleware",             # URL normalization
    "django.middleware.csrf.CsrfViewMiddleware",            # CSRF protection
    "django.contrib.auth.middleware.AuthenticationMiddleware", # User on request
    "django.contrib.messages.middleware.MessageMiddleware", # Flash messages
    "django.middleware.clickjacking.XFrameOptionsMiddleware", # Clickjacking
]
```

### Database Configuration

<Tabs>
<TabItem label="SQLite (development)">
```python
DATABASES = {
    "default": {
        "ENGINE": "django.db.backends.sqlite3",
        "NAME": BASE_DIR / "db.sqlite3",
    }
}
```
</TabItem>
<TabItem label="PostgreSQL">
```python
DATABASES = {
    "default": {
        "ENGINE": "django.db.backends.postgresql",
        "NAME": "mydb",
        "USER": "myuser",
        "PASSWORD": "mypassword",
        "HOST": "localhost",
        "PORT": "5432",
        "OPTIONS": {
            "connect_timeout": 10,
        },
    }
}
```
</TabItem>
<TabItem label="DATABASE_URL (recommended)">
```python
import dj_database_url

DATABASES = {
    "default": dj_database_url.config(
        default="sqlite:///db.sqlite3",
        conn_max_age=600,
        conn_health_checks=True,
    )
}
# Set DATABASE_URL=postgres://user:pass@host:5432/dbname in .env
```
</TabItem>
</Tabs>

### Templates

```python
TEMPLATES = [
    {
        "BACKEND": "django.template.backends.django.DjangoTemplates",
        # Look for templates in each app's templates/ folder
        "DIRS": [BASE_DIR / "templates"],  # Project-level templates
        "APP_DIRS": True,  # Also search <app>/templates/
        "OPTIONS": {
            "context_processors": [
                "django.template.context_processors.debug",
                "django.template.context_processors.request",
                "django.contrib.auth.context_processors.auth",
                "django.contrib.messages.context_processors.messages",
            ],
        },
    },
]
```

### Static and Media Files

```python
# URL prefix for static files served in development
STATIC_URL = "/static/"

# Where collectstatic gathers files for production
STATIC_ROOT = BASE_DIR / "staticfiles"

# Extra directories to search for static files
STATICFILES_DIRS = [BASE_DIR / "static"]

# URL prefix for user-uploaded media
MEDIA_URL = "/media/"

# Where uploaded files are stored
MEDIA_ROOT = BASE_DIR / "media"
```

### Authentication

```python
# Redirect after login/logout
LOGIN_REDIRECT_URL = "/"
LOGOUT_REDIRECT_URL = "/"
LOGIN_URL = "/accounts/login/"

# Password validation rules
AUTH_PASSWORD_VALIDATORS = [
    {"NAME": "django.contrib.auth.password_validation.UserAttributeSimilarityValidator"},
    {"NAME": "django.contrib.auth.password_validation.MinimumLengthValidator",
     "OPTIONS": {"min_length": 8}},
    {"NAME": "django.contrib.auth.password_validation.CommonPasswordValidator"},
    {"NAME": "django.contrib.auth.password_validation.NumericPasswordValidator"},
]
```

### Internationalization

```python
LANGUAGE_CODE = "en-us"
TIME_ZONE = "UTC"
USE_I18N = True   # Internationalization
USE_TZ = True     # Timezone-aware datetimes (always keep True)

DEFAULT_AUTO_FIELD = "django.db.models.BigAutoField"  # Default PK type
```

## Using python-decouple for Settings

Never hardcode secrets. Use `python-decouple` to load them from a `.env` file:

```bash
pip install python-decouple
```

```bash
# .env  (add to .gitignore)
SECRET_KEY=your-very-secret-key-here
DEBUG=True
DATABASE_URL=postgres://user:pass@localhost:5432/mydb
ALLOWED_HOSTS=localhost,127.0.0.1
```

```bash
# .env.example  (commit this to git)
SECRET_KEY=change-me
DEBUG=True
DATABASE_URL=sqlite:///db.sqlite3
ALLOWED_HOSTS=localhost,127.0.0.1
```

```python
# mysite/settings.py
from decouple import config, Csv
import dj_database_url

SECRET_KEY = config("SECRET_KEY")
DEBUG = config("DEBUG", default=False, cast=bool)
ALLOWED_HOSTS = config("ALLOWED_HOSTS", default="localhost", cast=Csv())

DATABASES = {
    "default": dj_database_url.config(
        default=config("DATABASE_URL", default="sqlite:///db.sqlite3")
    )
}
```

:::caution[Never Commit .env]
Add `.env` to your `.gitignore` immediately. Leaked `SECRET_KEY` values compromise your entire application's security.

```bash
echo ".env" >> .gitignore
```
:::

## manage.py Commands Reference

The `manage.py` script is your main interface for project management tasks.

| Command | Description |
|---------|-------------|
| `runserver` | Start the development server on `127.0.0.1:8000` |
| `runserver 0.0.0.0:8080` | Listen on all interfaces, port 8080 |
| `startapp <name>` | Create a new application |
| `makemigrations` | Detect model changes and create migration files |
| `makemigrations <app>` | Only for a specific app |
| `migrate` | Apply pending migrations to the database |
| `migrate <app> <migration>` | Migrate to a specific migration (rollback too) |
| `showmigrations` | List all migrations and their applied status |
| `sqlmigrate <app> <num>` | Print the SQL a migration would execute |
| `createsuperuser` | Create an admin superuser interactively |
| `changepassword <user>` | Change a user's password |
| `shell` | Open an interactive Python shell with Django loaded |
| `shell_plus` | Enhanced shell (requires `django-extensions`) |
| `dbshell` | Open the database's native shell (psql, sqlite3) |
| `collectstatic` | Copy all static files to `STATIC_ROOT` |
| `test` | Run the test suite |
| `test --verbosity 2` | Verbose test output |
| `check` | Validate the project for common issues |
| `check --deploy` | Validate production-readiness |
| `flush` | Delete all data from the database |
| `loaddata <fixture>` | Load data from a fixture file |
| `dumpdata <app>` | Export data to a JSON fixture |
| `diffsettings` | Show difference from default settings |
| `inspectdb` | Generate models from existing database |

## Split Settings Pattern (Advanced)

For projects with multiple environments, split `settings.py` into separate files:

```python
# mysite/settings/__init__.py
# Empty — lets Python treat this as a package

# mysite/settings/base.py
from pathlib import Path

BASE_DIR = Path(__file__).resolve().parent.parent.parent

SECRET_KEY = config("SECRET_KEY")
INSTALLED_APPS = [...]
MIDDLEWARE = [...]
# ... all shared settings

# mysite/settings/development.py
from .base import *  # noqa: F401, F403

DEBUG = True
ALLOWED_HOSTS = ["*"]
DATABASES = {
    "default": {"ENGINE": "django.db.backends.sqlite3", "NAME": BASE_DIR / "db.sqlite3"}
}

# mysite/settings/production.py
from .base import *  # noqa: F401, F403

DEBUG = False
ALLOWED_HOSTS = config("ALLOWED_HOSTS", cast=Csv())
DATABASES = {"default": dj_database_url.config()}
SECURE_SSL_REDIRECT = True
SESSION_COOKIE_SECURE = True
CSRF_COOKIE_SECURE = True
```

Run with a specific settings module:

```bash
# Set in environment
export DJANGO_SETTINGS_MODULE=mysite.settings.production

# Or pass directly
python manage.py runserver --settings=mysite.settings.development
```

:::tip[DJANGO_SETTINGS_MODULE]
Add `DJANGO_SETTINGS_MODULE=mysite.settings.development` to your `.env` file so `python-decouple` or your shell environment always uses the right settings file automatically.
:::

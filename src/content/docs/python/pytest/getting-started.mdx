This guide walks you from a bare Python environment to a fully structured pytest project with passing tests and proper configuration.

## Installation

<Steps>

1. **Create and activate a virtual environment**

   ```bash
   python -m venv .venv

   # Windows
   .venv\Scripts\activate

   # macOS / Linux
   source .venv/bin/activate
   ```

2. **Install pytest**

   ```bash
   pip install pytest
   ```

3. **Verify the installation**

   ```bash
   pytest --version
   # pytest 8.1.0
   ```

</Steps>

:::tip[Pin your pytest version]
Add pytest to `requirements-dev.txt` or `pyproject.toml` so every contributor uses the same version:

```
pytest>=8.0,<9.0
```
:::

## Recommended Project Structure

A conventional layout separates source code from tests, making both easier to navigate and preventing accidental imports during testing.

```
my_project/
├── src/
│   └── myapp/
│       ├── __init__.py
│       ├── calculator.py
│       └── utils.py
├── tests/
│   ├── __init__.py          # Optional but recommended for imports
│   ├── conftest.py          # Shared fixtures — covered in the Fixtures guide
│   ├── test_calculator.py
│   └── test_utils.py
├── pyproject.toml           # Project metadata and pytest configuration
└── README.md
```

:::tip[The src layout]
Placing your package under `src/` prevents Python from accidentally importing it directly from the project root during tests. Install your package in editable mode with `pip install -e .` so tests always import the real package.
:::

For simpler projects without a `src/` layout:

```
my_project/
├── calculator.py
├── utils.py
├── tests/
│   ├── test_calculator.py
│   └── test_utils.py
└── pyproject.toml
```

## Writing Your First Tests

Create the module under test:

```python
# src/myapp/calculator.py

def add(a: float, b: float) -> float:
    """Return the sum of a and b."""
    return a + b


def subtract(a: float, b: float) -> float:
    """Return the difference of a and b."""
    return a - b


def multiply(a: float, b: float) -> float:
    """Return the product of a and b."""
    return a * b


def divide(a: float, b: float) -> float:
    """Return a divided by b.

    Raises:
        ZeroDivisionError: If b is zero.
    """
    if b == 0:
        raise ZeroDivisionError("Cannot divide by zero")
    return a / b
```

Now create the test file:

```python
# tests/test_calculator.py
import pytest
from myapp.calculator import add, subtract, multiply, divide


# ---- Basic assertions ----

def test_add_two_integers():
    assert add(2, 3) == 5


def test_add_negative_numbers():
    assert add(-1, -2) == -3


def test_add_with_zero():
    assert add(0, 5) == 5
    assert add(5, 0) == 5


def test_add_floats():
    # Use pytest.approx for floating-point comparisons
    assert add(0.1, 0.2) == pytest.approx(0.3)


def test_subtract():
    assert subtract(10, 4) == 6


def test_multiply():
    assert multiply(3, 4) == 12


def test_divide():
    assert divide(10, 2) == 5.0
    assert divide(7, 2) == pytest.approx(3.5)


# ---- Exception testing ----

def test_divide_by_zero_raises():
    with pytest.raises(ZeroDivisionError, match="Cannot divide by zero"):
        divide(10, 0)


def test_divide_by_zero_error_type():
    with pytest.raises(ZeroDivisionError) as exc_info:
        divide(5, 0)
    assert "zero" in str(exc_info.value).lower()
```

Run the tests:

```bash
pytest tests/test_calculator.py -v
```

## Understanding Test Output

When a test fails, pytest introspects the assertion and shows you exactly what went wrong:

```python
def test_add_wrong_expected():
    assert add(2, 3) == 6  # wrong expected value
```

```
FAILED tests/test_calculator.py::test_add_wrong_expected

=========================== FAILURES ===========================
_________________ test_add_wrong_expected __________________

    def test_add_wrong_expected():
>       assert add(2, 3) == 6
E       AssertionError: assert 5 == 6
E        +  where 5 = add(2, 3)

tests/test_calculator.py:8: AssertionError
```

pytest shows the line that failed (`>`), the actual value, and where that value came from. No special assertion methods needed.

:::tip[Custom failure messages]
Add a message as the second argument to `assert` for extra context:

```python
result = compute(data)
assert result.success, f"compute() failed: {result.error!r} (input size={len(data)})"
```
:::

## Running Tests

### Basic Commands

```bash
# Run all tests discovered from the current directory
pytest

# Run tests in a specific file
pytest tests/test_calculator.py

# Run a specific test function
pytest tests/test_calculator.py::test_add_two_integers

# Run a specific method in a class
pytest tests/test_calculator.py::TestMath::test_add
```

### Verbosity and Tracebacks

```bash
# Verbose: show each test name and result
pytest -v

# Extra verbose: show more information on failures
pytest -vv

# Short traceback (great for CI — only the relevant lines)
pytest --tb=short

# One-line traceback
pytest --tb=line

# Full traceback with local variables
pytest -l --tb=long

# No traceback at all — just pass/fail counts
pytest --tb=no
```

### Filtering Tests

```bash
# Run tests whose names contain a keyword
pytest -k "add"

# Boolean logic
pytest -k "add or subtract"
pytest -k "divide and not zero"
pytest -k "not slow"

# Run tests with a specific marker
pytest -m slow
pytest -m "integration and not slow"
pytest -m "unit or smoke"
```

### Stopping Early

```bash
# Stop after the first failure
pytest -x

# Stop after N failures
pytest --maxfail=3
```

### Output Capture

```bash
# Show print() output even for passing tests
pytest -s

# Equivalent long form
pytest --capture=no
```

### Parallel Execution

```bash
# Requires: pip install pytest-xdist
pytest -n 4        # 4 worker processes
pytest -n auto     # use all available CPU cores
```

### Useful Utilities

```bash
# List all collected tests without running them
pytest --collect-only
pytest --co              # short form

# Re-run only the tests that failed last run
pytest --lf
pytest --last-failed     # long form

# Run last-failed first, then the rest
pytest --ff
pytest --failed-first    # long form

# Show the 10 slowest tests
pytest --durations=10

# Show all test durations (including fast ones)
pytest --durations=0
```

## Configuration Files

pytest reads configuration from `pyproject.toml`, `pytest.ini`, or `setup.cfg`. The recommended approach is `pyproject.toml`.

<Tabs>
  <TabItem label="pyproject.toml">

  ```toml
  [tool.pytest.ini_options]
  # Where to look for tests
  testpaths = ["tests"]

  # Minimum required pytest version
  minversion = "8.0"

  # Flags added to every pytest invocation
  addopts = "-v --tb=short --strict-markers"

  # Register custom markers (prevents PytestUnknownMarkWarning)
  markers = [
      "slow: marks tests as slow-running (deselect with '-m \"not slow\"')",
      "integration: marks tests that require external services",
      "unit: marks fast unit tests",
      "smoke: marks smoke tests for post-deployment verification",
  ]

  # Control warning behavior
  filterwarnings = [
      "error",
      "ignore::DeprecationWarning",
      "ignore::PendingDeprecationWarning",
  ]

  # Log level for captured log output
  log_cli_level = "WARNING"
  ```

  </TabItem>
  <TabItem label="pytest.ini">

  ```ini
  [pytest]
  testpaths = tests
  minversion = 8.0
  addopts = -v --tb=short --strict-markers
  markers =
      slow: marks tests as slow-running
      integration: marks tests requiring external services
      unit: marks fast unit tests
  filterwarnings =
      error
      ignore::DeprecationWarning
  log_cli_level = WARNING
  ```

  </TabItem>
  <TabItem label="setup.cfg">

  ```ini
  [tool:pytest]
  testpaths = tests
  minversion = 8.0
  addopts = -v --tb=short --strict-markers
  markers =
      slow: marks tests as slow-running
      integration: marks tests requiring external services
  filterwarnings =
      error
      ignore::DeprecationWarning
  ```

  </TabItem>
</Tabs>

:::caution[One config source only]
Do not define pytest configuration in multiple files simultaneously (e.g., both `pyproject.toml` and `pytest.ini`). pytest raises an error if it detects configuration in more than one location. Pick one file and use it exclusively.
:::

## Test Classes

You can group related tests in a class. pytest collects classes starting with `Test` (no `__init__` needed):

```python
# tests/test_calculator.py

class TestAddition:
    def test_positive_numbers(self):
        assert add(1, 2) == 3

    def test_negative_numbers(self):
        assert add(-1, -2) == -3

    def test_with_zero(self):
        assert add(0, 99) == 99


class TestDivision:
    def test_integer_result(self):
        assert divide(10, 2) == 5.0

    def test_float_result(self):
        assert divide(7, 2) == pytest.approx(3.5)

    def test_raises_on_zero(self):
        with pytest.raises(ZeroDivisionError):
            divide(1, 0)
```

:::tip[Classes vs functions]
Prefer plain functions for most tests — they're simpler. Use classes when you want to group tests that share setup via class-level fixtures, or when testing a single unit with many edge cases.
:::

## Introduction to conftest.py

`conftest.py` is a special file that pytest discovers automatically. It lets you share fixtures, hooks, and plugins across multiple test files without explicit imports.

```
tests/
├── conftest.py            # Available to all tests/
├── test_calculator.py
├── test_utils.py
└── integration/
    ├── conftest.py        # Available only to tests in integration/
    └── test_api.py
```

A minimal example:

```python
# tests/conftest.py
import pytest
from myapp.calculator import Calculator


@pytest.fixture
def calculator():
    """Provide a fresh Calculator instance for each test."""
    return Calculator()


@pytest.fixture(scope="session")
def config():
    """Load application configuration once for the entire test session."""
    return {"debug": True, "timeout": 30, "api_url": "https://api.test.example.com"}
```

Test files receive fixtures automatically — no import needed:

```python
# tests/test_calculator.py

def test_add_via_instance(calculator):        # injected from conftest.py
    assert calculator.add(1, 2) == 3


def test_config_timeout(config):              # injected from conftest.py
    assert config["timeout"] == 30
```

The full power of fixtures is covered in the [Fixtures](/python/pytest/fixtures) guide.

## Using Markers

Markers let you categorize tests and run subsets selectively.

```python
import pytest


@pytest.mark.slow
def test_heavy_computation():
    result = run_heavy_algorithm(iterations=1_000_000)
    assert result > 0


@pytest.mark.integration
def test_database_round_trip(db):
    db.insert({"id": 1, "name": "Alice"})
    record = db.get(1)
    assert record["name"] == "Alice"


@pytest.mark.unit
def test_format_price():
    assert format_price(9.99) == "$9.99"
```

```bash
# Run only fast unit tests during development
pytest -m "not slow"

# Run only integration tests in CI
pytest -m integration

# Skip both slow and integration tests
pytest -m "not slow and not integration"
```

:::caution[Register your markers]
Add every custom marker to `pyproject.toml` under `[tool.pytest.ini_options] markers`. Without registration, pytest emits `PytestUnknownMarkWarning`. With `--strict-markers` in `addopts`, unregistered markers cause an **error** — the recommended setting for clean projects.
:::

Pytest is the de-facto standard for Python testing, trusted by projects of all sizes — from hobby scripts to large-scale enterprise codebases. It combines an **approachable syntax** with **deep, extensible power** via fixtures, parametrize, and a rich plugin ecosystem.

## Why Pytest Over unittest?

Python's standard library includes `unittest`, which is perfectly capable. But the community has largely converged on pytest because it removes friction at every step.

| Feature          | pytest                                    | unittest                                      |
| ---------------- | ----------------------------------------- | --------------------------------------------- |
| Assertions       | Plain `assert` with rich diffs            | `self.assertEqual()`, `self.assertIn()`, etc. |
| Failure messages | Detailed introspected diffs               | Generic `AssertionError`                      |
| Fixtures         | Declarative, composable, scoped           | Rigid `setUp` / `tearDown` per-class          |
| Parametrize      | `@pytest.mark.parametrize`                | Manual looping or `subTest`                   |
| Discovery        | Fully automatic (`test_*.py`)             | Requires class inheritance or manual loader   |
| Plugins          | 1,000+ on PyPI                            | Minimal third-party ecosystem                 |
| Compatibility    | Runs `unittest.TestCase` and `nose` tests | unittest only                                 |

:::tip[Backwards compatible]
pytest runs your existing `unittest.TestCase` tests without modification. You can adopt pytest incrementally, file by file, without rewriting anything.
:::

## A Taste of pytest

Here is a representative example covering the most common patterns:

```python
# tests/test_calculator.py
import pytest
from myapp.calculator import add, divide


# --- Simple assertions ---

def test_add_two_positive_numbers():
    assert add(2, 3) == 5


def test_add_returns_zero_for_inverses():
    assert add(-4, 4) == 0


# --- Testing exceptions ---

def test_divide_raises_on_zero_divisor():
    with pytest.raises(ZeroDivisionError, match="Cannot divide by zero"):
        divide(10, 0)


# --- Floating point ---

def test_float_precision():
    assert 0.1 + 0.2 == pytest.approx(0.3)


# --- Parametrize: one function, many inputs ---

@pytest.mark.parametrize("a, b, expected", [
    (1, 1, 2),
    (0, 0, 0),
    (-1, -1, -2),
    (100, 200, 300),
])
def test_add_parametrized(a, b, expected):
    assert add(a, b) == expected
```

## Installation

```bash
pip install pytest
```

Verify the installation:

```bash
pytest --version
# pytest 8.1.0
```

A solid baseline for most projects:

```bash
pip install pytest pytest-cov pytest-mock pytest-xdist
```

## How pytest Discovers Tests

By default, pytest collects tests by:

1. Starting from the current directory (or paths passed on the command line)
2. Recursing into directories not listed in `norecursedirs`
3. Collecting files matching `test_*.py` or `*_test.py`
4. Within those files, collecting functions, methods, and classes whose names start with `test`

:::caution[Naming matters]
If your test file or function does not follow the `test_` prefix convention, pytest will not discover it automatically. Use `pytest --collect-only` to verify what gets collected before your first run.
:::

## The pytest Feedback Loop

```
Write test → Run pytest → Read failure output → Fix code → Repeat
```

The feedback loop is fast, the output is human-readable, and the framework stays out of your way. Let's get started.

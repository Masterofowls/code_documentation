**Next.js** is the most popular React framework, built and maintained by Vercel. It extends React with server-side rendering, static site generation, file-based routing, API routes, image optimization, and a full-stack architecture — all with zero configuration.

## What is Next.js?

Next.js is a **full-stack React framework** that handles:

- **Rendering** — SSR, SSG, ISR, and CSR in the same app
- **Routing** — file-based routing with layouts, loading states, and error boundaries
- **Data fetching** — built-in `fetch` with caching, revalidation, and streaming
- **Optimization** — automatic code splitting, image/font/script optimization
- **API layer** — Route Handlers that run on the server or Edge runtime
- **Deployment** — optimized builds for Vercel, Node.js, Docker, and static hosts

The current major version is **Next.js 15**, released in 2024, which defaults to the **App Router** and brings async cookies/headers, React 19 support, and Turbopack as the default bundler.

## App Router vs Pages Router

Next.js supports two routing paradigms. The **App Router** (introduced in Next.js 13) is the future of Next.js. The **Pages Router** remains fully supported for backward compatibility.

| Feature | App Router (`app/`) | Pages Router (`pages/`) |
|---|---|---|
| **React Server Components** | ✅ Default | ❌ Client-only |
| **Layouts** | Nested, persistent | `_app.js` (global only) |
| **Loading states** | `loading.tsx` (Suspense) | Manual with `next/router` |
| **Error boundaries** | `error.tsx` | `_error.js` |
| **Data fetching** | `fetch` in components | `getServerSideProps` / `getStaticProps` |
| **Streaming** | ✅ Built-in with Suspense | ❌ Not supported |
| **Server Actions** | ✅ | ❌ |
| **Metadata** | `metadata` export / `generateMetadata` | `next/head` |
| **Stability** | Stable since 13.4 | Battle-tested |

:::tip[Which Router to Choose?]
For new projects, always use the **App Router**. It provides better performance through React Server Components, built-in streaming, and a cleaner composable layout system. Only use Pages Router if you're maintaining a legacy project or need a specific Pages Router feature.
:::

## SSR vs SSG vs ISR vs CSR

```
Decision Tree for Rendering Strategy:
─────────────────────────────────────────────────────────────
Is the content user-specific or requires auth?
  ├─ YES → SSR (Server-Side Rendering) or Client-Side with API
  └─ NO → Is the content static or changes infrequently?
            ├─ YES, static → SSG (Static Site Generation)
            ├─ YES, but updates hourly/daily → ISR (Incremental Static Regen)
            └─ Changes in real time → SSR or Client-Side Fetching
─────────────────────────────────────────────────────────────
```

| Strategy | When generated | Caching | Best for |
|---|---|---|---|
| **SSG** | Build time | CDN indefinitely | Blog, docs, landing pages |
| **ISR** | Build + background regen | CDN with TTL | Product catalog, news |
| **SSR** | Each request | No CDN (or short TTL) | User dashboards, auth pages |
| **CSR** | Client-side after load | Browser cache | Real-time data, SPA sections |

```typescript
// SSG (default in App Router — cached at build time):
async function BlogPost({ params }: { params: { slug: string } }) {
  const post = await fetchPost(params.slug); // cached indefinitely
  return <article>{post.content}</article>;
}

// SSR (opt-in with cache: 'no-store'):
async function Dashboard() {
  const data = await fetch('/api/user', { cache: 'no-store' }); // fresh per request
  return <Dashboard data={await data.json()} />;
}

// ISR (revalidate after N seconds):
async function ProductPage({ params }: { params: { id: string } }) {
  const product = await fetch(`/api/products/${params.id}`, {
    next: { revalidate: 3600 }, // refresh every 1 hour
  });
  return <ProductDetail product={await product.json()} />;
}
```

## Next.js vs Remix vs Astro

| Criterion | Next.js 15 | Remix | Astro |
|---|---|---|---|
| **Primary use case** | Full-stack React apps | Data-centric React apps | Content-heavy sites |
| **Rendering** | SSR / SSG / ISR / CSR | SSR-first | SSG-first, islands architecture |
| **Router** | File-based (app/) | File-based (routes/) | File-based (src/pages/) |
| **Data fetching** | fetch + Server Components | loaders + actions | Astro.props, endpoints |
| **Client JS** | React, zero if SC only | React | Zero by default (islands opt-in) |
| **Forms / mutations** | Server Actions | form + action pattern | API routes |
| **Learning curve** | Medium | Medium | Low |
| **Ecosystem** | Massive | Growing | Growing |
| **Hosting** | Vercel, Node, static | Cloudflare, Fly, Vercel | Any static host |
| **Best for** | SaaS, e-commerce, apps | Data-heavy CRUD apps | Blogs, docs, marketing |

## Full-Stack Capabilities

Next.js blurs the line between frontend and backend:

```typescript
// app/api/products/route.ts — runs on the server
export async function GET(request: Request) {
  const products = await db.product.findMany({ take: 20 });
  return Response.json(products);
}

// app/products/page.tsx — Server Component with direct DB access
import { db } from '@/lib/db';

export default async function ProductsPage() {
  // No API call needed — directly query the database:
  const products = await db.product.findMany({
    orderBy: { createdAt: 'desc' },
    take: 20,
  });
  return <ProductGrid products={products} />;
}

// Server Action in a form — no API route needed:
async function createProduct(formData: FormData) {
  'use server';
  const name = formData.get('name') as string;
  await db.product.create({ data: { name } });
  revalidatePath('/products');
}
```

## Vercel Hosting

Next.js was created by Vercel and has first-class deployment integration:

- **Zero-config deploys** — push to GitHub and it deploys automatically
- **Preview Deployments** — every PR gets a unique preview URL
- **Edge Network** — static assets and SSG pages served from 100+ PoPs worldwide
- **Analytics** — Core Web Vitals tracking built in
- **Speed Insights** — real user performance monitoring
- **Environment Variables** — managed per environment (development, preview, production)
- **ISR** — works natively without any special configuration

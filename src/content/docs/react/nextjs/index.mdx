Next.js is the most popular full-stack React framework, maintained by Vercel. It extends React with file-based routing, multiple rendering strategies (SSR, SSG, ISR, CSR), built-in image/font optimization, and a full-stack API layer — all in one cohesive tool.

## What is Next.js?

Next.js turns a React application into a production-ready full-stack web application. Key features:

- **App Router** — React Server Components, layouts, streaming, and nested routing via the file system
- **Multiple rendering modes** — SSR, SSG, ISR, and CSR, per route
- **Server Actions** — write server-side mutations directly in React components
- **Built-in optimizations** — automatic image, font, and script optimization
- **Full-stack API layer** — Route Handlers (API endpoints) alongside UI code
- **TypeScript-first** — full TypeScript support with zero configuration
- **Vercel integration** — first-class deployment with edge network, analytics, and CI/CD

```bash
# Create a new Next.js app (TypeScript, Tailwind, App Router)
npx create-next-app@latest my-app \
  --typescript \
  --tailwind \
  --app \
  --src-dir \
  --import-alias "@/*"
```

## App Router vs Pages Router

Next.js has two routing systems. The **App Router** (introduced in Next.js 13) is the current default and uses React Server Components.

| Feature | App Router (`app/`) | Pages Router (`pages/`) |
|---|---|---|
| React Server Components | ✅ Default | ❌ No |
| Streaming / Suspense | ✅ Built-in | Limited |
| Nested layouts | ✅ Native | Manual (_app.tsx) |
| Server Actions | ✅ Yes | ❌ No |
| `loading.tsx` / `error.tsx` | ✅ Yes | Manual |
| `getServerSideProps` | ❌ (use RSC) | ✅ |
| `getStaticProps` | ❌ (use `fetch`) | ✅ |
| Middleware | ✅ Both | ✅ Both |
| Learning curve | Higher | Lower |
| Status | Current (recommended) | Stable (maintained) |

:::tip[Use App Router for New Projects]
All new Next.js projects should use the App Router. The Pages Router is still fully supported but will not receive new features.
:::

## SSR vs SSG vs ISR vs CSR

Next.js lets you choose the rendering strategy **per route**, not per project.

```
Decision Tree:
─────────────
Does the page need real-time data per request?
├── YES → SSR (Server-Side Rendering)
│         fetch(url, { cache: 'no-store' })
│
└── NO → Does the content change occasionally?
         ├── NEVER → SSG (Static Site Generation)
         │           fetch(url) with default cache
         │
         ├── YES, PERIODICALLY → ISR (Incremental Static Regeneration)
         │                        fetch(url, { next: { revalidate: 60 } })
         │
         └── YES, CLIENT-SIDE ONLY → CSR
                                      useEffect + SWR/React Query
```

```tsx
// SSR — fresh data on every request
async function UserProfile({ params }: { params: { id: string } }) {
  const user = await fetch(`/api/users/${params.id}`, {
    cache: 'no-store',
  }).then(r => r.json());
  return <div>{user.name}</div>;
}

// SSG — built once, served from CDN
async function BlogPost({ params }: { params: { slug: string } }) {
  const post = await fetch(`/api/posts/${params.slug}`).then(r => r.json());
  return <article>{post.content}</article>;
}

// ISR — rebuilt every 60 seconds max
async function ProductPage({ params }: { params: { id: string } }) {
  const product = await fetch(`/api/products/${params.id}`, {
    next: { revalidate: 60 },
  }).then(r => r.json());
  return <div>{product.name}</div>;
}
```

## Next.js vs Remix vs Astro

| Feature | Next.js 15 | Remix v2 | Astro 4 |
|---|---|---|---|
| **Primary use case** | Full-stack web apps | Full-stack web apps | Content sites, blogs, docs |
| **Rendering** | SSR, SSG, ISR, CSR, PPR | SSR-first (CSR opt-in) | Static-first (Islands) |
| **React Server Components** | ✅ Yes | ❌ No (as of v2) | ✅ Partial |
| **Routing** | File-based (App Router) | File-based (Remix) | File-based |
| **Data loading** | fetch in RSC, Server Actions | `loader`, `action` functions | Frontmatter, content collections |
| **Streaming** | ✅ React Suspense | ✅ Built-in | ✅ Partial |
| **Multi-framework** | React only | React only | React, Vue, Svelte, etc. |
| **Bundle size** | Larger | Moderate | Smallest (0 JS default) |
| **Vercel integration** | ⭐⭐⭐⭐⭐ | ⭐⭐⭐ | ⭐⭐⭐ |
| **Maturity** | Very mature (2016) | Mature (2021) | Moderate (2021) |
| **Learning curve** | Moderate-High | Moderate | Low-Moderate |

### When to Choose Next.js
- Building a full-stack web application with dynamic data
- Need multiple rendering strategies in one project
- Team already knows React
- Want Vercel's deployment pipeline
- Need SEO + performance + developer experience

### When to Choose Remix
- Prefer progressive enhancement and HTML-first philosophy
- Want simpler data loading with `loader`/`action` functions
- Building forms-heavy applications

### When to Choose Astro
- Primarily content-focused (blog, docs, marketing)
- Want maximum performance with minimal JavaScript
- Using multiple frontend frameworks on one site

## Full-Stack Capabilities

```typescript
// app/api/users/route.ts — API endpoint
import { NextRequest, NextResponse } from 'next/server';
import { db } from '@/lib/db';

export async function GET() {
  const users = await db.query('SELECT * FROM users LIMIT 100');
  return NextResponse.json(users);
}

export async function POST(req: NextRequest) {
  const body = await req.json();
  const user = await db.insert('users', body);
  return NextResponse.json(user, { status: 201 });
}
```

```tsx
// app/users/page.tsx — Server Component reads DB directly
import { db } from '@/lib/db';

export default async function UsersPage() {
  const users = await db.query('SELECT * FROM users LIMIT 100');
  return (
    <ul>
      {users.map(user => (
        <li key={user.id}>{user.name}</li>
      ))}
    </ul>
  );
}
```

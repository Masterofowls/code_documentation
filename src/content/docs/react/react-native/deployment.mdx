Getting a React Native app into users' hands involves configuring code signing, building native binaries, and submitting to app stores. EAS (Expo Application Services) simplifies this process dramatically.

## EAS Build Setup

<Steps>

1. **Install EAS CLI**

   ```bash
   npm install -g eas-cli
   eas login
   ```

2. **Initialize EAS in your project**

   ```bash
   eas build:configure
   ```

   This creates `eas.json` and updates `app.json` with your Expo project ID.

3. **Verify your app.json**

   ```json
   {
     "expo": {
       "name": "MyApp",
       "slug": "my-app",
       "version": "1.0.0",
       "runtimeVersion": {
         "policy": "appVersion"
       },
       "owner": "your-expo-username",
       "extra": {
         "eas": {
           "projectId": "xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx"
         }
       }
     }
   }
   ```

</Steps>

## iOS Code Signing

### Automatic (Recommended with EAS)

```bash
# EAS handles provisioning profiles and certificates automatically
eas credentials

# Build for iOS (EAS manages signing)
eas build --platform ios --profile production
```

### Manual Setup

If you need manual control:

```bash
# Create or fetch credentials interactively
eas credentials --platform ios

# Options EAS will ask:
# 1. Distribution Certificate (create new or use existing)
# 2. Provisioning Profile (App Store or Ad Hoc)
```

:::tip[EAS Credentials Storage]
EAS securely stores your signing credentials in the cloud so any team member or CI machine can build without needing the certificates locally. For sensitive projects, you can opt for local credentials storage.
:::

## Android Keystore

```bash
# Generate a new keystore (if you don't have one)
keytool -genkey -v \
  -keystore my-release-key.keystore \
  -alias my-key-alias \
  -keyalg RSA \
  -keysize 2048 \
  -validity 10000

# Upload keystore to EAS
eas credentials --platform android
```

:::caution[Never Lose Your Keystore]
Your Android keystore is irreplaceable. If you lose it, you cannot update your app on the Play Store — you would have to create a new app listing. Back it up securely (password manager, encrypted cloud storage). EAS can store it for you.
:::

## OTA Updates with EAS Update

EAS Update lets you push JavaScript and asset updates to users without going through the App Store review process.

```bash
# Install expo-updates
npx expo install expo-updates

# Configure in app.config.ts (already shown above in updates block)

# Publish an update to the production channel
eas update --branch production --message "Fix login screen crash"

# Publish to a specific channel
eas update --branch preview --message "New feature preview"

# List recent updates
eas update:list
```

```typescript
// Manual update check in your app (optional)
import * as Updates from 'expo-updates';

export async function checkForUpdates() {
  if (!Updates.isEnabled) return;

  try {
    const update = await Updates.checkForUpdateAsync();
    if (update.isAvailable) {
      await Updates.fetchUpdateAsync();
      await Updates.reloadAsync(); // Restart app with new update
    }
  } catch (error) {
    console.error('Update check failed:', error);
  }
}
```

:::caution[OTA Update Limitations]
OTA updates can only update your JavaScript bundle and assets (images, fonts). They cannot add new native modules, change native code, or modify app permissions. Changes requiring native code need a full store build.
:::

## Building and Submitting

```bash
# Build for production
eas build --platform ios --profile production
eas build --platform android --profile production
eas build --platform all --profile production

# Check build status
eas build:list

# Submit to App Store (after build completes)
eas submit --platform ios --latest

# Submit to Play Store
eas submit --platform android --latest

# Build and submit in one command
eas build --platform all --profile production --auto-submit
```

## App Store Submission Checklist

Before submitting to the Apple App Store:

- [ ] **App Icon** — 1024x1024 PNG, no alpha channel, no rounded corners (Apple adds them)
- [ ] **Screenshots** — Required sizes: 6.7" (iPhone 14 Pro Max), 5.5" (iPhone 8 Plus), 12.9" iPad
- [ ] **App Preview Video** — Optional but recommended (max 30 seconds)
- [ ] **Privacy Policy URL** — Required for all apps
- [ ] **Age Rating** — Complete the content questionnaire
- [ ] **App Review Information** — Demo account credentials if login is required
- [ ] **Export Compliance** — Declare encryption usage
- [ ] **App Description** — Up to 4000 characters, keyword-optimized
- [ ] **Keywords** — 100 characters max, comma-separated
- [ ] **Support URL** — Must be accessible
- [ ] **Build** — Uploaded via EAS or Transporter
- [ ] **TestFlight** — Test with external testers before production submission
- [ ] **Permissions** — All `NSUsageDescription` strings are descriptive and accurate
- [ ] **Deep Links** — Universal links configured and verified

## Google Play Store Submission Checklist

Before submitting to Google Play:

- [ ] **App Icon** — 512x512 PNG
- [ ] **Feature Graphic** — 1024x500 PNG/JPEG
- [ ] **Screenshots** — Minimum 2, max 8 (phone + tablet)
- [ ] **Short Description** — Max 80 characters
- [ ] **Full Description** — Max 4000 characters
- [ ] **App Category** — Select appropriate category
- [ ] **Content Rating** — Complete rating questionnaire
- [ ] **Privacy Policy** — URL required
- [ ] **App Signing** — Enrolled in Play App Signing
- [ ] **Target API Level** — Must target Android 14 (API 34) for new apps
- [ ] **AAB Format** — Upload `.aab` file (not `.apk`) for Play Store
- [ ] **Permissions** — Justify any sensitive permissions in declaration
- [ ] **Data Safety** — Complete the Data Safety form (what data you collect and why)
- [ ] **Internal Testing Track** — Test before promoting to production

## GitHub Actions CI/CD

```yaml
# .github/workflows/eas-build.yml
name: EAS Build and Submit

on:
  push:
    branches: [main]
    tags: ['v*']

jobs:
  build:
    name: Build and Submit
    runs-on: ubuntu-latest
    steps:
      - name: Checkout code
        uses: actions/checkout@v4

      - name: Setup Node.js
        uses: actions/setup-node@v4
        with:
          node-version: 20
          cache: npm

      - name: Install dependencies
        run: npm ci

      - name: Setup Expo
        uses: expo/expo-github-action@v8
        with:
          expo-version: latest
          eas-version: latest
          token: ${{ secrets.EXPO_TOKEN }}

      - name: Run type check
        run: npx tsc --noEmit

      - name: Build production (tagged releases only)
        if: startsWith(github.ref, 'refs/tags/v')
        run: eas build --platform all --profile production --non-interactive --auto-submit

      - name: Publish OTA update (main branch)
        if: github.ref == 'refs/heads/main' && !startsWith(github.ref, 'refs/tags/')
        run: eas update --branch production --message "${{ github.event.head_commit.message }}" --non-interactive
```

## Fastlane Alternative

For teams already using Fastlane with native iOS/Android projects:

```ruby
# fastlane/Fastfile
lane :build_ios do
  setup_ci if ENV['CI']
  match(type: 'appstore', readonly: is_ci)
  build_app(
    workspace: 'ios/MyApp.xcworkspace',
    scheme: 'MyApp',
    export_method: 'app-store',
    output_directory: './build',
  )
end

lane :deploy_ios do
  build_ios
  deliver(
    skip_screenshots: true,
    submit_for_review: false,
    automatic_release: false,
  )
end

lane :build_android do
  gradle(
    task: 'bundle',
    build_type: 'Release',
    project_dir: 'android/',
  )
end
```

:::tip[EAS vs Fastlane]
For Expo projects, EAS Build is the better choice — it handles the entire native toolchain in the cloud. For complex bare React Native apps with deep native customization, Fastlane gives more control and integrates well with existing native CI pipelines.
:::

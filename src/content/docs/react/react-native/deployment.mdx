Shipping a React Native app to the App Store and Play Store involves configuring native build settings, managing code signing credentials, and submitting binaries for review. EAS (Expo Application Services) automates the most complex parts of this process.

## EAS Build Setup

EAS Build runs your native builds in the cloud — no Xcode or Android Studio required locally (for CI/CD or non-macOS developers).

<Steps>

1. **Install and authenticate EAS CLI**

   ```bash
   npm install -g eas-cli
   eas login
   ```

2. **Initialize EAS in your project**

   ```bash
   eas build:configure
   ```

   This creates `eas.json` with default build profiles.

3. **Verify your project is linked to Expo**

   ```bash
   eas project:info
   ```

</Steps>

### Build Commands

```bash
# Development build (requires physical device or simulator):
eas build --profile development --platform ios
eas build --profile development --platform android

# Preview build for internal testing:
eas build --profile preview --platform all

# Production build:
eas build --profile production --platform ios
eas build --profile production --platform android
eas build --profile production --platform all

# Build locally (requires Xcode / Android Studio):
eas build --profile production --platform ios --local
```

## iOS Code Signing

EAS manages iOS provisioning profiles and certificates automatically.

```bash
# Let EAS handle credentials (recommended):
eas credentials

# View existing credentials:
eas credentials --platform ios

# Generate new credentials (fresh project):
# EAS will prompt you to create or use existing Apple Developer credentials
eas build --profile production --platform ios
# EAS auto-creates: Distribution Certificate + Provisioning Profile
```

### Manual Certificate Management

If you need to manage credentials yourself:

<Steps>

1. **Create a Distribution Certificate** in the Apple Developer portal
2. **Download and install** the `.p12` certificate on your Mac
3. **Create a Provisioning Profile** (App Store distribution) linked to your App ID and certificate
4. **Upload to EAS:**

   ```bash
   eas credentials --platform ios
   # Select: Add new certificate / Add new provisioning profile
   ```

</Steps>

:::caution[Certificate Expiration]
Apple Distribution Certificates expire after 1 year. Set a calendar reminder 1 month before expiration to renew. Expired certificates will block App Store submissions.
:::

## Android Keystore

Every Android app must be signed with a keystore. **Never lose your production keystore** — you cannot update an app on the Play Store if you lose the signing key.

```bash
# Generate a keystore:
keytool -genkey -v -keystore my-release-key.keystore \
  -alias my-key-alias \
  -keyalg RSA \
  -keysize 2048 \
  -validity 10000

# Upload to EAS (EAS stores it securely):
eas credentials --platform android
# Select: Add new keystore → Upload existing keystore
```

:::tip[Google Play App Signing]
Enable **Google Play App Signing** — Google stores your production key and you provide an upload key. This protects your app if your upload key is compromised: you can request a new upload key from Google without losing your app's signing identity.
:::

## OTA Updates with EAS Update

EAS Update lets you push JavaScript/asset changes without a full App Store/Play Store release — ideal for hotfixes and rapid iteration.

```bash
# Install:
npx expo install expo-updates

# Configure in eas.json (each build profile maps to an update channel)
# Then publish an update:
eas update --channel production --message "Fix login crash"

# Preview channel:
eas update --channel preview --message "WIP: new onboarding flow"

# Branch-based update (auto deploy from CI):
eas update --auto
```

```typescript
// Manually check and apply updates in code:
import * as Updates from 'expo-updates';

async function checkForUpdate() {
  try {
    const update = await Updates.checkForUpdateAsync();
    if (update.isAvailable) {
      await Updates.fetchUpdateAsync();
      await Updates.reloadAsync(); // restarts the app with new JS
    }
  } catch (err) {
    console.error('Update check failed:', err);
  }
}
```

:::caution[OTA Update Limitations]
EAS Update can only deliver JavaScript and asset changes. Changes to native code (new Expo modules, modified `app.json` plugins, or native dependencies) require a new binary build submitted to the stores.
:::

## Environment Variables with EAS

```bash
# Set a secret in EAS (server-side only, not in JS bundle):
eas secret:create --scope project --name STRIPE_SECRET_KEY --value sk_live_xxx

# Set a public variable (available in JS with EXPO_PUBLIC_ prefix):
eas env:create --name EXPO_PUBLIC_API_URL --value https://api.example.com --environment production
```

In code:

```typescript
// Public env vars (compiled into the JS bundle, visible to users):
const apiUrl = process.env.EXPO_PUBLIC_API_URL; // accessible in JS

// Secret env vars (available in EAS Build workers only, NOT in JS bundle):
// Use for API keys needed at build time (e.g., fetching config)
// Access via eas.json env or EAS secrets
```

:::caution[Never Put Secrets in EXPO_PUBLIC_ Variables]
Anything prefixed with `EXPO_PUBLIC_` is compiled into your JavaScript bundle and visible to anyone who decompiles your app. Use EAS Secrets for API keys, private tokens, and credentials.
:::

## App Store Submission Checklist

Before submitting to the Apple App Store:

- [ ] **App icon** — 1024×1024 PNG, no transparency, no rounded corners (Apple applies them)
- [ ] **Screenshots** — required sizes: 6.7″, 6.1″, 5.5″, 12.9″ iPad; optional: 4.7″, iPad 2nd gen
- [ ] **App Privacy Policy URL** — required for all apps
- [ ] **App category** — primary and optional secondary
- [ ] **Age rating** — complete the questionnaire accurately
- [ ] **In-app purchases** — all IAP configured in App Store Connect before submission
- [ ] **Permissions** — every `NS*UsageDescription` has a clear, honest reason string
- [ ] **Version string** matches `app.json` `version`
- [ ] **Build number** is higher than previous approved build
- [ ] **No private APIs** — App Store review scans for private API usage
- [ ] **TestFlight** — run a TestFlight beta for at least 1 week before submitting for review
- [ ] **Crash-free** — test on real devices, especially oldest supported iOS version

```bash
# Submit to App Store via EAS:
eas submit --platform ios --latest
# Or specify a build ID:
eas submit --platform ios --id <build-id>
```

## Play Store Submission Checklist

Before submitting to the Google Play Store:

- [ ] **App icon** — 512×512 PNG
- [ ] **Feature graphic** — 1024×500 JPEG or PNG
- [ ] **Screenshots** — minimum 2, maximum 8; phone, 7″ tablet, 10″ tablet
- [ ] **Short description** ≤ 80 chars; **Full description** ≤ 4000 chars
- [ ] **Content rating** — complete the IARC questionnaire
- [ ] **Target API level** — must target the latest required Android API (Google updates this annually)
- [ ] **Privacy Policy URL** — required
- [ ] **Data safety** section completed honestly
- [ ] **App Bundle** (`.aab`) used, not APK, for production releases
- [ ] **versionCode** incremented from previous release
- [ ] Tested on multiple Android versions and screen sizes

```bash
# Submit AAB to Play Store:
eas submit --platform android --latest
```

## CI/CD with GitHub Actions

```yaml
# .github/workflows/eas-build.yml
name: EAS Production Build

on:
  push:
    tags:
      - 'v*'

jobs:
  build:
    name: EAS Build
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - uses: actions/setup-node@v4
        with:
          node-version: 20
          cache: npm

      - name: Install dependencies
        run: npm ci

      - name: Setup EAS
        uses: expo/expo-github-action@v8
        with:
          eas-version: latest
          token: ${{ secrets.EXPO_TOKEN }}

      - name: Build iOS
        run: eas build --platform ios --profile production --non-interactive

      - name: Build Android
        run: eas build --platform android --profile production --non-interactive

      - name: Submit to stores
        run: eas submit --platform all --latest --non-interactive
```

## Fastlane Alternative

For teams that prefer Fastlane (especially in bare React Native projects):

```ruby
# ios/fastlane/Fastfile
default_platform(:ios)

platform :ios do
  lane :beta do
    increment_build_number(xcodeproj: "MyApp.xcodeproj")
    build_app(scheme: "MyApp", export_method: "app-store")
    upload_to_testflight(skip_waiting_for_build_processing: true)
  end

  lane :release do
    increment_build_number(xcodeproj: "MyApp.xcodeproj")
    build_app(scheme: "MyApp", export_method: "app-store")
    upload_to_app_store(
      skip_metadata: false,
      submit_for_review: false,
      force: true,
    )
  end
end
```

```ruby
# android/fastlane/Fastfile
default_platform(:android)

platform :android do
  lane :beta do
    gradle(task: "bundle", build_type: "Release")
    upload_to_play_store(track: "internal")
  end

  lane :release do
    gradle(task: "bundle", build_type: "Release")
    upload_to_play_store(track: "production", release_status: "draft")
  end
end
```

:::tip[EAS vs Fastlane]
EAS Build is significantly easier to set up and maintain, especially for Expo projects. Choose Fastlane if you have an existing Fastlane setup, need Fastlane's plugins (Slack notifications, Firebase distribution, etc.), or are working in a bare RN project without Expo.
:::

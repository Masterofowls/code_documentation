React Native lets you build real, natively rendering mobile apps for iOS and Android using React and JavaScript. Unlike hybrid frameworks that wrap a WebView, React Native bridges JavaScript logic to actual native platform components — giving your app the look, feel, and performance that users expect.

## What Is React Native?

React Native was created by Facebook (now Meta) in 2015. It uses React's declarative component model, but instead of rendering HTML elements, it renders native UI components:

- `<View>` → `UIView` (iOS) / `android.view.View` (Android)
- `<Text>` → `UILabel` / `android.widget.TextView`
- `<Image>` → `UIImageView` / `android.widget.ImageView`

This means your app is **not** a web app running inside a browser. It's a real native app compiled from your JavaScript code.

```tsx
import { View, Text, StyleSheet } from 'react-native';

export default function HelloWorld() {
  return (
    <View style={styles.container}>
      <Text style={styles.heading}>Hello, React Native!</Text>
    </View>
  );
}

const styles = StyleSheet.create({
  container: { flex: 1, justifyContent: 'center', alignItems: 'center' },
  heading: { fontSize: 24, fontWeight: 'bold' },
});
```

## React Native vs Expo vs Bare Workflow

There are three main ways to work with React Native. Choosing the right one depends on your team's needs.

| | **Expo (Managed)** | **Expo (Bare Workflow)** | **React Native CLI** |
|---|---|---|---|
| Setup difficulty | Very easy | Moderate | Moderate–Hard |
| Native code access | Via plugins | Full access | Full access |
| Over-the-air updates | ✅ EAS Update | ✅ EAS Update | Manual |
| Build service | ✅ EAS Build | ✅ EAS Build | Self-managed |
| Custom native modules | Via config plugins | ✅ Full | ✅ Full |
| Best for | New apps, prototypes | Most production apps | Legacy / existing native code |

:::tip[Start with Expo]
Unless you have a specific reason to use bare React Native CLI, start every new project with Expo. The managed workflow handles certificates, builds, and updates for you, and you can always eject to a bare workflow later.
:::

**Expo Managed Workflow** is the simplest path. You write only JavaScript/TypeScript, use pre-built native modules from Expo, and use `expo-dev-client` or EAS Build for custom modules.

**Expo Bare Workflow** gives you the full native project (`ios/` and `android/` directories) while still using Expo's tooling like EAS Build and EAS Update.

**React Native CLI** generates the full native project from scratch. Use this when integrating React Native into an existing native app or when you need maximum control.

## The New Architecture

React Native's "New Architecture" is a complete overhaul of the communication layer between JavaScript and native code. It shipped as stable in React Native 0.74+.

### Old Architecture

In the old architecture, JavaScript and native code communicated over an **asynchronous bridge**. Every call had to serialize data to JSON, pass it across the bridge, and deserialize it on the other side. This caused jank in complex animations and gesture interactions.

### New Architecture Components

**JSI (JavaScript Interface)**

JSI replaces the bridge with a lightweight C++ layer that lets JavaScript hold a reference to native objects directly — synchronous, no serialization required.

```cpp
// JSI allows JS to call native C++ directly
// No JSON serialization, no async round-trips
jsi::Value result = nativeFunction.call(runtime, args);
```

**Fabric (New Renderer)**

Fabric is a reimplementation of the UI rendering layer. It supports:
- Synchronous rendering for priority updates
- Concurrent React features (Suspense, transitions)
- Multithreaded layout calculations

**TurboModules**

TurboModules replace the old NativeModules system. They are lazily loaded (modules load on demand, not all at startup) and use JSI for synchronous calls.

```typescript
// TurboModule interface definition
import type { TurboModule } from 'react-native';
import { TurboModuleRegistry } from 'react-native';

export interface Spec extends TurboModule {
  multiply(a: number, b: number): Promise<number>;
}

export default TurboModuleRegistry.getEnforcing<Spec>('Calculator');
```

**Codegen**

Codegen automatically generates C++ type-safe glue code from TypeScript/Flow specs, reducing errors at the JS-native boundary.

:::caution[Migration Warning]
Some third-party libraries have not yet migrated to the New Architecture. Check the [React Native Directory](https://reactnative.directory) for compatibility before adding a dependency.
:::

## React Native vs Competitors

| Feature | **React Native** | **Flutter** | **Ionic** | **Capacitor** |
|---|---|---|---|---|
| Language | JavaScript / TypeScript | Dart | HTML/CSS/JS | HTML/CSS/JS |
| Rendering | Native components | Custom canvas (Skia) | WebView | WebView |
| Performance | Near-native | Near-native | Web-level | Web-level |
| Web sharing | React (partial) | Partial | ✅ Full | ✅ Full |
| Native feel | ✅ Platform-native | Custom widgets | Limited | Limited |
| Ecosystem | Huge (npm) | Growing | Huge (npm) | Huge (npm) |
| Hot reload | ✅ Fast Refresh | ✅ Hot reload | ✅ | ✅ |
| Companies using | Meta, Microsoft, Shopify | Google, BMW | Marketwatch | Capacitor apps |

**When to choose React Native:**
- Your team already knows React/JavaScript
- You want native UI components (not a custom widget system)
- You need access to a massive npm ecosystem
- You want to share business logic with a React web app

**When to choose Flutter:**
- You need pixel-perfect custom UI that looks identical on every platform
- Your team is comfortable learning Dart
- You need excellent performance on low-end Android devices

**When to choose Ionic/Capacitor:**
- You want to ship a web app to the App Store with minimal effort
- Native look-and-feel is not a priority

## Essential Community Libraries

These are the battle-tested libraries used in virtually every production React Native app.

### Navigation
- **[React Navigation](https://reactnavigation.org/)** — The standard navigation library. Supports Stack, Tab, Drawer, and more.
- **[Expo Router](https://expo.github.io/router)** — File-based routing built on React Navigation, similar to Next.js.

### Animations & Gestures
- **[React Native Reanimated](https://docs.swmansion.com/react-native-reanimated/)** — Runs animations on the UI thread via Worklets; enables 60fps gesture-driven animations.
- **[React Native Gesture Handler](https://docs.swmansion.com/react-native-gesture-handler/)** — Native-powered gesture system; pairs with Reanimated.

### State & Data
- **[Zustand](https://zustand-demo.pmnd.rs/)** — Minimal, hook-based global state.
- **[TanStack Query](https://tanstack.com/query)** — Server state management with caching, background refresh, and offline support.
- **[MMKV](https://github.com/mrousavy/react-native-mmkv)** — C++ backed key-value storage, 10x faster than AsyncStorage.

### UI Components
- **[React Native Paper](https://reactnativepaper.com/)** — Material Design 3 component library.
- **[NativeWind](https://nativewind.dev/)** — Tailwind CSS for React Native.
- **[Shopify Flash List](https://shopify.github.io/flash-list/)** — High-performance list component replacing FlatList.

### Media & Device
- **[Expo Camera](https://docs.expo.dev/versions/latest/sdk/camera/)** — Camera access.
- **[Expo Image](https://docs.expo.dev/versions/latest/sdk/image/)** — Optimized image component with caching.
- **[React Native Vision Camera](https://mrousavy.com/react-native-vision-camera/)** — High-performance camera with frame processors.

### Testing
- **[Jest](https://jestjs.io/)** + **[@testing-library/react-native](https://callstack.github.io/react-native-testing-library/)** — Unit and component tests.
- **[Detox](https://wix.github.io/Detox/)** — End-to-end testing on real simulators/emulators.

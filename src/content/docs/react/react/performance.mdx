React is fast by default, but complex applications can accumulate performance regressions. This guide shows how to measure first, then systematically eliminate bottlenecks using the right tools and patterns.

:::tip[Golden Rule: Measure Before Optimising]
Never add `React.memo`, `useMemo`, or `useCallback` without first using the React DevTools Profiler to confirm a genuine problem. Premature optimisation adds complexity and can actually make things slower.
:::

## React DevTools Profiler

The Profiler records component render timing and shows what caused each render.

<Steps>

1. **Install React DevTools**

   ```bash
   # Chrome / Firefox extension — search "React Developer Tools"
   ```

2. **Enable the Profiler**

   Open DevTools → React tab → click **Profiler** → press ▶ Record.

3. **Interact with your app** — click buttons, navigate pages, type in inputs.

4. **Stop recording** — press ⏹. React shows a flame graph of every commit.

5. **Identify slow components**

   - Yellow/red bars = slow renders
   - Grey = did not render
   - Hover a bar to see props that changed

</Steps>

### Programmatic Profiling

```tsx
import { Profiler, ProfilerOnRenderCallback } from 'react';

const onRenderCallback: ProfilerOnRenderCallback = (
  id,            // component tree identifier
  phase,         // 'mount' | 'update' | 'nested-update'
  actualDuration, // time for committed render
  baseDuration,  // estimated time to render without memoisation
  startTime,
  commitTime
) => {
  if (actualDuration > 16) { // longer than one frame at 60fps
    console.warn(`Slow render in "${id}": ${actualDuration.toFixed(2)}ms (${phase})`);
  }
};

function App() {
  return (
    <Profiler id="Dashboard" onRender={onRenderCallback}>
      <Dashboard />
    </Profiler>
  );
}
```

## React.memo

Wrapping a component in `React.memo` prevents re-renders when its **props have not changed** (shallow comparison):

```tsx
import { memo, useState } from 'react';

interface ListItemProps {
  id: string;
  name: string;
  price: number;
  onSelect: (id: string) => void;
}

// ✅ Only re-renders when id, name, price, or onSelect changes
const ListItem = memo(function ListItem({ id, name, price, onSelect }: ListItemProps) {
  console.log(`Rendering ListItem ${id}`); // shows in console only on prop change
  return (
    <li onClick={() => onSelect(id)}>
      <span>{name}</span>
      <span>${price.toFixed(2)}</span>
    </li>
  );
});

// Parent with 1000 items — only changed items re-render
function ProductList({ products }: { products: Product[] }) {
  const [selectedId, setSelectedId] = useState<string | null>(null);

  return (
    <ul>
      {products.map(p => (
        <ListItem
          key={p.id}
          id={p.id}
          name={p.name}
          price={p.price}
          onSelect={setSelectedId} // ⚠️ this is a new function on every render!
        />
      ))}
    </ul>
  );
}
```

:::caution[React.memo + Inline Functions]
The example above has a bug: `onSelect={setSelectedId}` creates a **new function reference** on every parent render, causing all memoised `ListItem`s to re-render anyway. Always wrap callbacks in `useCallback` when passing to memoised children.
:::

### Custom Comparison Function

```tsx
const ExpensiveChart = memo(
  function Chart({ data, title }: ChartProps) {
    return <canvas>{/* rendering logic */}</canvas>;
  },
  // Custom compare: only re-render if data length or title changes
  (prev, next) =>
    prev.title === next.title &&
    prev.data.length === next.data.length
);
```

## useMemo

Memoises a computed value so it's only recalculated when dependencies change:

```tsx
import { useState, useMemo } from 'react';

function DataTable({ rows }: { rows: Row[] }) {
  const [filter, setFilter]       = useState('');
  const [sortField, setSortField] = useState<keyof Row>('name');
  const [sortDir, setSortDir]     = useState<'asc' | 'desc'>('asc');

  // Without useMemo: re-runs every render even when rows/filter/sort haven't changed
  // With useMemo: only re-runs when these deps change
  const processedRows = useMemo(() => {
    const lower = filter.toLowerCase();
    return rows
      .filter(r => r.name.toLowerCase().includes(lower) ||
                   r.email.toLowerCase().includes(lower))
      .sort((a, b) => {
        const aVal = a[sortField];
        const bVal = b[sortField];
        const cmp  = String(aVal).localeCompare(String(bVal));
        return sortDir === 'asc' ? cmp : -cmp;
      });
  }, [rows, filter, sortField, sortDir]);

  return (
    <>
      <input value={filter} onChange={e => setFilter(e.target.value)} placeholder="Filter…" />
      <table>
        <thead>
          <tr>
            {(['name', 'email', 'role'] as const).map(field => (
              <th
                key={field}
                onClick={() => {
                  if (sortField === field) setSortDir(d => d === 'asc' ? 'desc' : 'asc');
                  else { setSortField(field); setSortDir('asc'); }
                }}
              >
                {field} {sortField === field ? (sortDir === 'asc' ? '↑' : '↓') : ''}
              </th>
            ))}
          </tr>
        </thead>
        <tbody>
          {processedRows.map(r => (
            <tr key={r.id}>
              <td>{r.name}</td>
              <td>{r.email}</td>
              <td>{r.role}</td>
            </tr>
          ))}
        </tbody>
      </table>
      <p>{processedRows.length} of {rows.length} rows shown</p>
    </>
  );
}
```

## useCallback

Memoises a function reference so it's stable across renders:

```tsx
import { useState, useCallback, memo } from 'react';

// Memoised child — only re-renders when onToggle changes
const TodoItem = memo(({ text, done, onToggle }: {
  text: string;
  done: boolean;
  onToggle: () => void;
}) => (
  <li style={{ textDecoration: done ? 'line-through' : 'none' }} onClick={onToggle}>
    {text}
  </li>
));

function TodoList() {
  const [todos, setTodos] = useState([
    { id: '1', text: 'Learn React', done: false },
    { id: '2', text: 'Build something', done: false },
  ]);

  // ✅ Stable function reference — memoised TodoItem doesn't re-render unnecessarily
  const handleToggle = useCallback((id: string) => {
    setTodos(prev =>
      prev.map(t => t.id === id ? { ...t, done: !t.done } : t)
    );
  }, []); // empty deps — setTodos is stable

  return (
    <ul>
      {todos.map(todo => (
        <TodoItem
          key={todo.id}
          text={todo.text}
          done={todo.done}
          onToggle={() => handleToggle(todo.id)}
          // ⚠️ This arrow function still creates a new ref each render!
          // Fix: pass id as a prop and use handleToggle directly
        />
      ))}
    </ul>
  );
}
```

## Virtualisation

Render only the rows visible in the viewport — essential for lists with 1000+ items:

```bash
npm install @tanstack/react-virtual
```

```tsx
import { useVirtualizer } from '@tanstack/react-virtual';
import { useRef } from 'react';

function VirtualList({ items }: { items: { id: string; name: string }[] }) {
  const parentRef = useRef<HTMLDivElement>(null);

  const virtualizer = useVirtualizer({
    count:         items.length,
    getScrollElement: () => parentRef.current,
    estimateSize:  () => 50,       // estimated row height in px
    overscan:      10,             // extra items to render above/below viewport
  });

  const virtualItems = virtualizer.getVirtualItems();

  return (
    <div
      ref={parentRef}
      style={{ height: '600px', overflow: 'auto' }}
    >
      {/* Total height container so scrollbar is correct */}
      <div style={{ height: `${virtualizer.getTotalSize()}px`, position: 'relative' }}>
        {virtualItems.map(virtualRow => (
          <div
            key={virtualRow.key}
            data-index={virtualRow.index}
            ref={virtualizer.measureElement}
            style={{
              position:  'absolute',
              top:       0,
              left:      0,
              width:     '100%',
              transform: `translateY(${virtualRow.start}px)`,
            }}
          >
            <div className="list-item">
              {items[virtualRow.index]?.name}
            </div>
          </div>
        ))}
      </div>
    </div>
  );
}
```

**Console output with 10,000 items:**
```
DOM nodes in viewport: ~22 (instead of 10,000)
Scroll FPS: 60fps consistent
Initial render time: 12ms (vs 800ms without virtualisation)
```

## Code Splitting

Split your JavaScript bundle so users only download code they need:

```tsx
import { lazy, Suspense } from 'react';
import { Routes, Route } from 'react-router-dom';

// Each page becomes a separate JS chunk
const HomePage      = lazy(() => import('./pages/HomePage'));
const DashboardPage = lazy(() => import('./pages/DashboardPage'));
const SettingsPage  = lazy(() => import('./pages/SettingsPage'));
// Named export
const AnalyticsPage = lazy(() =>
  import('./pages/AnalyticsPage').then(m => ({ default: m.AnalyticsPage }))
);

function App() {
  return (
    <Suspense fallback={<GlobalSpinner />}>
      <Routes>
        <Route path="/"           element={<HomePage />} />
        <Route path="/dashboard"  element={<DashboardPage />} />
        <Route path="/settings"   element={<SettingsPage />} />
        <Route path="/analytics"  element={<AnalyticsPage />} />
      </Routes>
    </Suspense>
  );
}
```

## Automatic Batching in React 18

React 18 batches state updates from all sources — not just React event handlers:

```tsx
function AutoBatchingDemo() {
  const [count, setCount] = useState(0);
  const [flag, setFlag]   = useState(false);

  // React 18: all three setters in one tick → single re-render
  const handleAsync = () => {
    setTimeout(() => {
      setCount(c => c + 1);   // \
      setFlag(f => !f);       //  > batched → 1 render
      setCount(c => c + 1);   // /
    }, 0);
  };

  const handleFetch = async () => {
    const data = await fetchSomething();
    // React 18: batched even after await
    setCount(c => c + 1);
    setFlag(f => !f);
    // → 1 render (was 2 in React 17)
  };

  // Opt out of batching when needed
  const handleSyncUpdate = () => {
    import { flushSync } from 'react-dom';
    flushSync(() => setCount(c => c + 1)); // forces immediate render
    flushSync(() => setFlag(f => !f));     // another immediate render
  };

  return (
    <div>
      <p>Count: {count} | Flag: {String(flag)}</p>
      <button onClick={handleAsync}>Async update</button>
    </div>
  );
}
```

## Concurrent Features

### useTransition for Expensive UI Updates

```tsx
import { useState, useTransition, startTransition } from 'react';

function FilterableTable({ data }: { data: Row[] }) {
  const [filter,   setFilter]   = useState('');
  const [filtered, setFiltered] = useState(data);
  const [isPending, startTransition] = useTransition();

  const handleFilterChange = (value: string) => {
    // Urgent: update the input immediately
    setFilter(value);

    // Non-urgent: expensive filter — deferred so input stays responsive
    startTransition(() => {
      const lower = value.toLowerCase();
      setFiltered(
        data.filter(row =>
          Object.values(row).some(v =>
            String(v).toLowerCase().includes(lower)
          )
        )
      );
    });
  };

  return (
    <div>
      <input
        value={filter}
        onChange={e => handleFilterChange(e.target.value)}
        placeholder="Filter table…"
      />
      {isPending && <span className="spinner" aria-label="Filtering…" />}
      <table style={{ opacity: isPending ? 0.7 : 1 }}>
        <tbody>
          {filtered.map(row => <TableRow key={row.id} row={row} />)}
        </tbody>
      </table>
    </div>
  );
}
```

## Avoiding Reconciliation Pitfalls

```tsx
// ❌ Inline object — new reference every render, forces child re-render
function Parent() {
  return <Child style={{ margin: '8px' }} />;
}

// ✅ Stable reference outside component
const CHILD_STYLE = { margin: '8px' };
function Parent() {
  return <Child style={CHILD_STYLE} />;
}

// ❌ Unstable key — array index causes subtle bugs on reorder
{items.map((item, index) => <Item key={index} item={item} />)}

// ✅ Stable, unique key
{items.map(item => <Item key={item.id} item={item} />)}

// ❌ Context causes all consumers to re-render
const value = { user, theme }; // new object every render
<AppContext.Provider value={value}>

// ✅ Memoised context value
const value = useMemo(() => ({ user, theme }), [user, theme]);
<AppContext.Provider value={value}>
```

## Web Vitals in React Apps

```bash
npm install web-vitals
```

```tsx
// src/reportWebVitals.ts
import { onCLS, onINP, onLCP, onFCP, onTTFB } from 'web-vitals';

type ReportFn = (metric: { name: string; value: number; rating: string }) => void;

export function reportWebVitals(report: ReportFn = console.log) {
  onCLS(report);  // Cumulative Layout Shift   < 0.1  (good)
  onINP(report);  // Interaction to Next Paint  < 200ms (good)
  onLCP(report);  // Largest Contentful Paint   < 2.5s (good)
  onFCP(report);  // First Contentful Paint
  onTTFB(report); // Time to First Byte
}

// main.tsx
reportWebVitals(metric => {
  if (metric.rating === 'poor') {
    console.warn(`Poor Web Vital: ${metric.name} = ${metric.value}`);
    // Send to analytics
    sendToAnalytics(metric);
  }
});
```

## Performance Checklist

<Aside type="note" title="Before You Ship">
- [ ] Run React DevTools Profiler — no component takes > 16ms to render
- [ ] Bundle size analysed — no unexpected large dependencies
- [ ] Route-level code splitting in place
- [ ] Lists with > 100 items use virtualisation
- [ ] `React.memo` added only where Profiler confirmed unnecessary re-renders
- [ ] Lighthouse performance score > 90 on mobile
- [ ] LCP < 2.5s, INP < 200ms, CLS < 0.1
- [ ] No inline objects/functions passed to memoised child components
- [ ] `useTransition` used for any filter/search that updates > 10k DOM nodes
</Aside>

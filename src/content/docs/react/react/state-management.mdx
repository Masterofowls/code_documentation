Choosing the right state management solution is one of the most impactful architectural decisions in a React application. This guide covers every tier of the state management spectrum with complete, production-ready examples.

## State Categories

Before choosing a library, categorise your state:

| Category | Description | Solution |
|---|---|---|
| **Local UI state** | Open/closed, hover, form input | `useState` / `useReducer` |
| **Lifted state** | Shared between siblings | Lift to common parent |
| **Global UI state** | Theme, locale, modal stack | Context API or Zustand |
| **Server state** | Fetched data, caches, mutations | TanStack Query / SWR |
| **URL state** | Filters, pagination, selected item | `useSearchParams` |
| **Form state** | Input values, validation, submit | React Hook Form |

## Local State with useState

Keep state as close to where it's used as possible:

```tsx
import { useState } from 'react';

function Dropdown({ options }: { options: string[] }) {
  const [isOpen, setIsOpen]       = useState(false);
  const [selected, setSelected]   = useState<string | null>(null);

  return (
    <div className="dropdown">
      <button onClick={() => setIsOpen(o => !o)}>
        {selected ?? 'Select an option'} ▾
      </button>
      {isOpen && (
        <ul className="dropdown__menu">
          {options.map(opt => (
            <li
              key={opt}
              onClick={() => { setSelected(opt); setIsOpen(false); }}
              className={opt === selected ? 'active' : ''}
            >
              {opt}
            </li>
          ))}
        </ul>
      )}
    </div>
  );
}
```

## Lifting State Up

When two sibling components need to share state, lift it to their nearest common ancestor:

```tsx
// ✅ Shared state lifted to parent
function TemperatureCalculator() {
  const [celsius, setCelsius] = useState<number | ''>('');

  const fahrenheit = celsius !== '' ? celsius * 9/5 + 32 : '';

  return (
    <div>
      <TemperatureInput
        scale="Celsius"
        value={celsius}
        onChange={setCelsius}
      />
      <TemperatureInput
        scale="Fahrenheit"
        value={fahrenheit}
        onChange={f => setCelsius(f !== '' ? (f - 32) * 5/9 : '')}
      />
      {celsius !== '' && (
        <p>
          {celsius}°C = {fahrenheit}°F
          {Number(celsius) >= 100 && ' — Water boils! 🌊'}
        </p>
      )}
    </div>
  );
}
```

## Context API

Context is ideal for low-frequency global values like theme, locale, or auth user:

```tsx
import { createContext, useContext, useState, useMemo, useCallback } from 'react';

// --- Types ---
interface User {
  id: string;
  name: string;
  email: string;
  role: 'admin' | 'user';
}

interface AuthContextValue {
  user: User | null;
  login:  (credentials: { email: string; password: string }) => Promise<void>;
  logout: () => void;
  isAuthenticated: boolean;
}

// --- Context ---
const AuthContext = createContext<AuthContextValue | null>(null);

export function useAuth(): AuthContextValue {
  const ctx = useContext(AuthContext);
  if (!ctx) throw new Error('useAuth must be used within AuthProvider');
  return ctx;
}

// --- Provider ---
export function AuthProvider({ children }: { children: React.ReactNode }) {
  const [user, setUser] = useState<User | null>(null);

  const login = useCallback(async ({ email, password }: { email: string; password: string }) => {
    const res  = await fetch('/api/auth/login', {
      method: 'POST',
      headers: { 'Content-Type': 'application/json' },
      body: JSON.stringify({ email, password }),
    });
    const data = await res.json() as User;
    setUser(data);
  }, []);

  const logout = useCallback(() => {
    setUser(null);
    fetch('/api/auth/logout', { method: 'POST' });
  }, []);

  // Memoised value prevents all consumers re-rendering when unrelated state changes
  const value = useMemo<AuthContextValue>(
    () => ({ user, login, logout, isAuthenticated: user !== null }),
    [user, login, logout]
  );

  return <AuthContext.Provider value={value}>{children}</AuthContext.Provider>;
}
```

### Context Performance Pitfalls

:::caution[Every Consumer Re-renders on Value Change]
When a Context value changes, **every component** that calls `useContext` with that context re-renders — even if the part of the value they care about didn't change. Avoid this with:
1. **Context splitting** — separate contexts for separate concerns
2. **Memoised values** — `useMemo` in the provider
3. **Selectors** — use Zustand or Jotai instead for high-frequency updates
:::

```tsx
// ❌ One context for everything — all consumers re-render on any change
const AppContext = createContext({ user, theme, cart, notifications });

// ✅ Split contexts — consumers only re-render when their slice changes
const UserContext         = createContext<UserValue | null>(null);
const ThemeContext        = createContext<ThemeValue | null>(null);
const CartContext         = createContext<CartValue | null>(null);
const NotificationContext = createContext<NotificationValue | null>(null);
```

## Zustand

Zustand is a minimal, fast, scalable state management library with an excellent TypeScript experience:

```bash
npm install zustand
```

### Basic Store

```tsx
import { create } from 'zustand';
import { devtools, persist } from 'zustand/middleware';

interface CartItem {
  id: string;
  name: string;
  price: number;
  qty: number;
}

interface CartStore {
  items: CartItem[];
  total: number;
  addItem:    (item: Omit<CartItem, 'qty'>) => void;
  removeItem: (id: string) => void;
  updateQty:  (id: string, qty: number) => void;
  clearCart:  () => void;
}

function calcTotal(items: CartItem[]) {
  return items.reduce((sum, i) => sum + i.price * i.qty, 0);
}

export const useCartStore = create<CartStore>()(
  devtools(
    persist(
      (set) => ({
        items: [],
        total: 0,

        addItem: (item) =>
          set((state) => {
            const exists = state.items.find(i => i.id === item.id);
            const items = exists
              ? state.items.map(i => i.id === item.id ? { ...i, qty: i.qty + 1 } : i)
              : [...state.items, { ...item, qty: 1 }];
            return { items, total: calcTotal(items) };
          }),

        removeItem: (id) =>
          set((state) => {
            const items = state.items.filter(i => i.id !== id);
            return { items, total: calcTotal(items) };
          }),

        updateQty: (id, qty) =>
          set((state) => {
            const items = state.items.map(i => i.id === id ? { ...i, qty } : i);
            return { items, total: calcTotal(items) };
          }),

        clearCart: () => set({ items: [], total: 0 }),
      }),
      { name: 'cart-storage' }    // persists to localStorage
    ),
    { name: 'CartStore' }         // Redux DevTools label
  )
);
```

### Using the Store with Selectors

```tsx
// ✅ Selector — only re-renders when items.length changes
function CartIcon() {
  const itemCount = useCartStore(state => state.items.length);
  return <span className="cart-icon">🛒 {itemCount}</span>;
}

// ✅ Multiple values — use shallow to prevent unnecessary re-renders
import { useShallow } from 'zustand/react/shallow';

function CartTotal() {
  const { total, clearCart } = useCartStore(
    useShallow(state => ({ total: state.total, clearCart: state.clearCart }))
  );
  return (
    <div>
      <p>Total: ${total.toFixed(2)}</p>
      <button onClick={clearCart}>Clear</button>
    </div>
  );
}
```

## Jotai

Jotai uses atomic state — each piece of state is an `atom`, and components subscribe only to the atoms they use:

```bash
npm install jotai
```

```tsx
import { atom, useAtom, useAtomValue, useSetAtom } from 'jotai';
import { atomWithStorage } from 'jotai/utils';

// --- Primitive atoms ---
const countAtom    = atom(0);
const themeAtom    = atomWithStorage<'light' | 'dark'>('theme', 'light');

// --- Derived (read-only) atom ---
const doubledAtom  = atom((get) => get(countAtom) * 2);

// --- Writable derived atom ---
const countWithLog = atom(
  (get) => get(countAtom),
  (get, set, value: number) => {
    console.log(`Count changed: ${get(countAtom)} → ${value}`);
    set(countAtom, value);
  }
);

// --- Usage ---
function Counter() {
  const [count, setCount] = useAtom(countAtom);
  const doubled           = useAtomValue(doubledAtom);

  return (
    <div>
      <p>Count: {count} (doubled: {doubled})</p>
      <button onClick={() => setCount(c => c + 1)}>+</button>
    </div>
  );
}

function ThemeToggle() {
  const setTheme = useSetAtom(themeAtom);
  return (
    <button onClick={() => setTheme(t => t === 'light' ? 'dark' : 'light')}>
      Toggle theme
    </button>
  );
}
```

## TanStack Query (React Query)

TanStack Query is the gold standard for **server state** — it handles caching, background refetching, loading/error states, and optimistic updates:

```bash
npm install @tanstack/react-query
npm install --save-dev @tanstack/react-query-devtools
```

### Setup

```tsx
// main.tsx
import { QueryClient, QueryClientProvider } from '@tanstack/react-query';
import { ReactQueryDevtools } from '@tanstack/react-query-devtools';

const queryClient = new QueryClient({
  defaultOptions: {
    queries: {
      staleTime:       60 * 1000,   // data fresh for 1 minute
      gcTime:          5 * 60 * 1000, // garbage collected after 5 minutes
      retry:           2,
      refetchOnWindowFocus: true,
    },
  },
});

createRoot(document.getElementById('root')!).render(
  <QueryClientProvider client={queryClient}>
    <App />
    <ReactQueryDevtools initialIsOpen={false} />
  </QueryClientProvider>
);
```

### useQuery

```tsx
import { useQuery } from '@tanstack/react-query';

interface Post {
  id: number;
  title: string;
  body: string;
}

async function fetchPosts(page: number): Promise<Post[]> {
  const res = await fetch(`/api/posts?page=${page}&limit=10`);
  if (!res.ok) throw new Error(`HTTP ${res.status}`);
  return res.json();
}

function PostList() {
  const [page, setPage] = useState(1);

  const { data: posts, isLoading, isError, error, isFetching } = useQuery({
    queryKey: ['posts', page],          // cache key — unique per page
    queryFn:  () => fetchPosts(page),
    placeholderData: (prev) => prev,    // keep previous data while fetching next page
  });

  if (isLoading) return <Spinner />;
  if (isError)   return <p>Error: {(error as Error).message}</p>;

  return (
    <div>
      {isFetching && <p style={{ opacity: 0.5 }}>Refreshing…</p>}
      <ul>
        {posts?.map(post => (
          <li key={post.id}>
            <h3>{post.title}</h3>
            <p>{post.body}</p>
          </li>
        ))}
      </ul>
      <button onClick={() => setPage(p => p - 1)} disabled={page === 1}>Previous</button>
      <button onClick={() => setPage(p => p + 1)}>Next</button>
    </div>
  );
}
```

### useMutation

```tsx
import { useMutation, useQueryClient } from '@tanstack/react-query';

interface CreatePostInput {
  title: string;
  body: string;
}

async function createPost(input: CreatePostInput): Promise<Post> {
  const res = await fetch('/api/posts', {
    method: 'POST',
    headers: { 'Content-Type': 'application/json' },
    body: JSON.stringify(input),
  });
  if (!res.ok) throw new Error(`HTTP ${res.status}`);
  return res.json();
}

function CreatePostForm() {
  const queryClient = useQueryClient();
  const [title, setTitle] = useState('');
  const [body, setBody]   = useState('');

  const mutation = useMutation({
    mutationFn: createPost,
    onSuccess: (newPost) => {
      // Invalidate the posts list so it refetches
      queryClient.invalidateQueries({ queryKey: ['posts'] });
      // Or optimistically add to cache:
      queryClient.setQueryData<Post[]>(['posts', 1], (old = []) => [newPost, ...old]);
      setTitle(''); setBody('');
    },
    onError: (error: Error) => {
      alert(`Failed: ${error.message}`);
    },
  });

  return (
    <form onSubmit={(e) => {
      e.preventDefault();
      mutation.mutate({ title, body });
    }}>
      <input value={title} onChange={e => setTitle(e.target.value)} placeholder="Title" />
      <textarea value={body} onChange={e => setBody(e.target.value)} />
      <button type="submit" disabled={mutation.isPending}>
        {mutation.isPending ? 'Posting…' : 'Create Post'}
      </button>
    </form>
  );
}
```

:::tip[staleTime vs gcTime]
- **`staleTime`**: How long data is considered fresh. During this window, React Query won't refetch in the background. Default: `0` (always stale).
- **`gcTime`** (formerly `cacheTime`): How long unused/inactive query data stays in cache before garbage collection. Default: `5 minutes`.

Set `staleTime` to `Infinity` for truly static data like config or feature flags.
:::

## Solution Comparison

<Tabs>
  <TabItem label="Comparison Table">

| Criteria | Context API | Zustand | Jotai | Redux Toolkit | TanStack Query |
|---|---|---|---|---|---|
| **Bundle size** | 0kb (built-in) | ~1.1kb | ~3kb | ~14kb | ~13kb |
| **Purpose** | Any global state | Client state | Atomic state | Complex client state | Server state |
| **DevTools** | ❌ | ✅ Redux DevTools | ✅ Jotai DevTools | ✅ Redux DevTools | ✅ TQ DevTools |
| **TypeScript** | Good | Excellent | Excellent | Excellent | Excellent |
| **Learning curve** | Low | Low | Low-Medium | Medium-High | Medium |
| **Async/server state** | Manual | Manual | Manual | RTK Query | ✅ Built-in |
| **Selectors / derived** | Limited | ✅ | ✅ atoms | ✅ | ✅ select option |
| **Persistence** | Manual | `persist` middleware | `atomWithStorage` | redux-persist | Background sync |
| **Best for** | Auth, theme, locale | UI global state | Fine-grained atoms | Large enterprise apps | API data |

</TabItem>
  <TabItem label="When to Use Each">

**Context API** — when:
- You have low-frequency updates (auth status, theme, locale)
- You don't want any third-party dependencies
- Less than ~10 consumers exist

**Zustand** — when:
- You need simple global state with minimal boilerplate
- You want Redux-like patterns without Redux overhead
- You need `persist`, `devtools`, or `immer` middleware

**Jotai** — when:
- Your state is naturally atomic (many small independent pieces)
- You want Recoil-like state but smaller and simpler
- You're building complex derived state graphs

**Redux Toolkit** — when:
- Your organisation already uses Redux
- You need time-travel debugging and complex middleware
- Multiple teams need strong conventions enforced by tooling

**TanStack Query** — when:
- You're fetching data from APIs (this is almost always)
- You need caching, background refresh, pagination, or optimistic updates
- You want to eliminate `useEffect` + `useState` data fetching patterns

</TabItem>
</Tabs>

<Aside type="tip" title="The Recommended Stack">
For most React applications, combine:
- **Zustand** for global client state (auth user cached in memory, UI state)
- **TanStack Query** for all server/API state
- **Context API** only for static config (theme tokens, i18n strings)

This combination covers 95% of state needs with minimal boilerplate.
</Aside>

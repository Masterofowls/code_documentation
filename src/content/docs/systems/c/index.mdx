# C Programming Language

C is one of the most influential programming languages ever created. Designed for systems programming, it gives developers direct control over hardware and memory while remaining portable across platforms. Nearly every operating system kernel, embedded firmware, and high-performance runtime traces its roots to C.

---

## A Brief History

C was developed by **Dennis Ritchie** at **Bell Labs** starting in **1972**, initially as a systems programming language to write the Unix operating system. Before C, Unix had been written in assembly — C made the OS portable across different hardware architectures for the first time.

The name derives from its predecessor **B**, which was itself derived from **BCPL**. Ritchie and Brian Kernighan documented the language in the landmark book *The C Programming Language* (1978), commonly known as **K&R C**, which became the de facto standard before formal standardization.

```
1969 — B language developed at Bell Labs
1972 — C developed by Dennis Ritchie
1978 — K&R C ("The C Programming Language" book)
1989 — ANSI C / C89 standardized by ANSI
1990 — C90 adopted by ISO (same language, different body)
1999 — C99 introduces bool, inline, VLAs, designated initializers
2011 — C11 adds threads, atomics, _Generic, _Static_assert
2017 — C17 (C18) — bug-fix release, no new features
2023 — C23 — typeof, #embed, nullptr, bool as keyword, bit utilities
```

## C Standards

### C89 / C90
The first official standard. Introduced the familiar function prototype syntax, `void` keyword, `const`, `volatile`, and the standard library. C89 and C90 are the same language — ANSI published it in 1989, ISO ratified it in 1990.

### C99
A significant update adding:
- `//` single-line comments
- `bool` type via `<stdbool.h>`
- Variable-length arrays (VLAs)
- Designated initializers: `struct Point p = { .x = 1, .y = 2 };`
- Compound literals
- `inline` keyword
- `restrict` pointer qualifier
- `long long` integer type
- `<stdint.h>` with `int32_t`, `uint64_t`, etc.
- Flexible array members
- Mixed declarations and code (not just top of block)

### C11
- `_Generic` for type-generic macros
- `_Static_assert` for compile-time assertions
- `_Noreturn` function specifier
- `_Alignas` / `_Alignof`
- Anonymous structs and unions
- `<threads.h>` — portable threading (thrd_t, mtx_t)
- `<stdatomic.h>` — atomic operations (_Atomic, memory order)
- `gets()` removed (replaced by `fgets`)
- `<uchar.h>` for Unicode support

### C17 / C18
A maintenance release. Clarified undefined behaviors and corrected defects in C11 without introducing new features.

### C23
The latest standard, ratified 2023:
- `bool`, `true`, `false` are now keywords (no `<stdbool.h>` needed)
- `nullptr` constant (typed as `nullptr_t`)
- `typeof` and `typeof_unqual` operators
- `#embed` preprocessor directive for binary file inclusion
- `#elifdef` / `#elifndef`
- `0b` binary integer literals
- `[[nodiscard]]`, `[[deprecated]]`, `[[maybe_unused]]` attributes
- `constexpr` for constants (different from C++ constexpr)
- Improved bit manipulation utilities in `<stdbit.h>`
- `_BitInt(N)` arbitrary-width integers

## C vs C++ Comparison

| Feature | C | C++ |
|---|---|---|
| Object-oriented | ❌ | ✅ Classes, inheritance, polymorphism |
| Templates / generics | ❌ (macros only) | ✅ Templates, concepts |
| RAII | ❌ (manual) | ✅ Constructors/destructors |
| Standard library | `libc` | `libstdc++` / `libc++` |
| Operator overloading | ❌ | ✅ |
| Exceptions | ❌ (`setjmp`/`longjmp`) | ✅ |
| Namespaces | ❌ | ✅ |
| ABI stability | Simpler | Complex (mangling) |
| Compile times | Fast | Can be slow (templates) |
| Interop with C | — | C-compatible via `extern "C"` |
| Use cases | OS kernels, firmware, embedded | Game engines, compilers, HPC |

:::tip[When to Choose C]
Prefer C for OS kernels, embedded microcontrollers, bootloaders, firmware, language runtimes (Python/Ruby/Lua are implemented in C), and any context where a minimal footprint and stable ABI are paramount.
:::

## C's Role Today

Despite being over 50 years old, C remains indispensable:

- **Operating system kernels** — Linux, Windows NT kernel, macOS XNU, FreeBSD
- **Embedded systems and firmware** — microcontrollers (Arduino, STM32, ESP32), RTOS (FreeRTOS, Zephyr)
- **Language runtimes** — CPython, MRI Ruby, Lua, PHP (core)
- **Databases** — SQLite, portions of PostgreSQL, Redis
- **Compilers and interpreters** — LLVM (partially), many interpreters start in C
- **Networking** — nginx, cURL, OpenSSL, libuv
- **Scientific computing** — BLAS/LAPACK, HDF5
- **WebAssembly** — compiled via Emscripten, C is a primary target

## The Compilation Pipeline

A C source file passes through four distinct stages before becoming an executable:

```
source.c → [Preprocessor] → source.i → [Compiler] → source.s → [Assembler] → source.o → [Linker] → executable
```

### 1. Preprocessing
The C preprocessor (`cpp`) handles all `#` directives:
- `#include` — textually pastes header file content
- `#define` — expands macros
- `#ifdef` / `#ifndef` — conditional compilation
- Strips comments
- Produces a `.i` translation unit

```bash
gcc -E source.c -o source.i    # stop after preprocessing
```

### 2. Compilation
The compiler translates preprocessed C into assembly language for the target CPU architecture. This stage performs:
- Lexing and parsing (AST construction)
- Semantic analysis and type checking
- Optimization passes
- Code generation

```bash
gcc -S source.c -o source.s    # stop after compilation, emit assembly
```

### 3. Assembly
The assembler (`as`) converts assembly text into machine code, producing an **object file** (`.o` or `.obj`). Object files contain:
- Machine code sections (`.text`)
- Initialized data (`.data`)
- Uninitialized data (`.bss`)
- A symbol table with unresolved references

```bash
gcc -c source.c -o source.o    # compile and assemble, stop before linking
```

### 4. Linking
The linker (`ld`) combines one or more object files with libraries to produce the final executable:
- Resolves symbol references between object files
- Links against the C standard library (`libc`)
- Sets up the program entry point (`_start` → `main`)
- Produces a relocatable or position-independent executable

```bash
gcc source.o -o program        # link object file into executable
```

## Hello, World — Every Line Explained

```c
#include <stdio.h>

int main(void) {
    printf("Hello, World!\n");
    return 0;
}
```

| Line | Explanation |
|---|---|
| `#include <stdio.h>` | Preprocessor directive. Pastes the standard I/O header into the translation unit, providing the declaration of `printf`. Angle brackets `<>` search system include paths. |
| `int main(void)` | The program entry point. `int` is the return type (exit code). `void` explicitly states no parameters are accepted. The alternate form `int main(int argc, char *argv[])` accepts command-line arguments. |
| `{` | Opens the function body block. |
| `printf("Hello, World!\n");` | Calls the formatted print function. The string literal contains `\n` — the newline escape sequence. The semicolon terminates the statement. |
| `return 0;` | Returns exit code 0 to the OS, conventionally indicating success. Non-zero values indicate error. In `main`, `return 0` can be omitted in C99+ (implicitly falls off the end). |
| `}` | Closes the function body block. |

## Undefined, Implementation-Defined, and Unspecified Behavior

Understanding these three categories is essential for writing portable, correct C code.

### Undefined Behavior (UB)
The C standard places **no constraints** on what the compiler or runtime must do. The compiler is free to assume UB never occurs — and use that assumption to optimize aggressively. Common sources:

```c
int arr[5];
arr[10] = 42;        // ❌ buffer overflow — UB

int x;
printf("%d\n", x);   // ❌ uninitialized read — UB

int *p = NULL;
*p = 5;              // ❌ null pointer dereference — UB

int overflow = INT_MAX + 1;  // ❌ signed integer overflow — UB
```

:::caution[Never Rely on Undefined Behavior]
Compilers exploit UB for optimizations that can silently delete code, invert logic, or produce security vulnerabilities. Use `-fsanitize=undefined` (UBSanitizer) to detect UB at runtime during development.
:::

### Implementation-Defined Behavior
The standard allows the implementation to choose, but requires it to document its choice:

```c
int n = -7 % 3;      // Implementation-defined in C89; defined as truncation-toward-zero in C99+
char c = 200;        // char may be signed or unsigned — implementation-defined
sizeof(int);         // 4 on most 32/64-bit systems, but not guaranteed
```

### Unspecified Behavior
The standard allows several valid behaviors; the implementation need not be consistent or document which it uses:

```c
int i = 0;
printf("%d %d\n", i++, i++);  // Unspecified: order of argument evaluation
```

## C11 Selected Features at a Glance

```c
#include <stdio.h>
#include <stdalign.h>
#include <stdatomic.h>

// _Static_assert — compile-time assertion
_Static_assert(sizeof(int) >= 4, "int must be at least 4 bytes");

// _Alignas — aligned variable
_Alignas(64) char cache_line[64];  // aligned to 64-byte cache line

// _Generic — type-generic macro (poor man's overloading)
#define type_name(x) _Generic((x), \
    int:    "int",                  \
    float:  "float",                \
    double: "double",               \
    char *: "char *",               \
    default: "unknown"              \
)

// Anonymous struct in union (C11)
typedef struct {
    union {
        struct { float x, y, z; };   // anonymous struct — access as .x .y .z directly
        float v[3];                   // same memory, array access
    };
} Vec3;

// _Atomic (C11)
_Atomic int counter = 0;

int main(void) {
    Vec3 pos = { .x = 1.0f, .y = 2.0f, .z = 3.0f };
    printf("Type of 3.14: %s\n", type_name(3.14));
    printf("Type of 42:   %s\n", type_name(42));
    printf("pos.x = %.1f, pos.v[0] = %.1f\n", pos.x, pos.v[0]);

    // Atomic increment
    atomic_fetch_add(&counter, 1);
    printf("Counter: %d\n", atomic_load(&counter));
    return 0;
}
```

## C in the Modern Era

C continues to evolve carefully, maintaining backward compatibility while adopting features that make safe, expressive code easier to write. C23 takes significant strides: keywords for boolean and null, `typeof`, and `#embed` modernize the language without bloating it.

The language's simplicity is a feature, not a limitation. When you write C, you know exactly what the machine is doing — there is no hidden runtime, no garbage collector, no ABI surprise. That predictability is precisely why C will remain essential for decades to come.

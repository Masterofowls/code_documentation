# C Programming Language

C is one of the most influential programming languages ever created. Born at Bell Labs in the early 1970s, it remains the lingua franca of systems programming, operating system kernels, embedded firmware, and performance-critical applications more than fifty years after its creation.

## History

Dennis Ritchie developed C at Bell Labs between **1969 and 1973** as part of the effort to rewrite the Unix operating system. C evolved from an earlier language called B (itself derived from BCPL), which Ken Thompson had written. The decision to rewrite Unix in a high-level language rather than assembly was revolutionary — it produced a portable, maintainable codebase that could be moved across hardware architectures.

The first book, _The C Programming Language_ by Brian Kernighan and Dennis Ritchie (commonly called "K&R C"), was published in 1978 and effectively served as the informal standard for years.

```c
/* Classic K&R Hello World — the program that introduced a generation */
#include <stdio.h>

main()
{
    printf("hello, world\n");
}
```

## C Standards

C has been standardized by ANSI and ISO multiple times, with each revision adding features and clarifying behavior:

| Standard             | Year      | Key Additions                                                                                       |
| -------------------- | --------- | --------------------------------------------------------------------------------------------------- |
| **K&R C**            | 1978      | Original; implicit int, old-style function params                                                   |
| **C89/C90** (ANSI C) | 1989/1990 | First formal standard; function prototypes, `void`, `const`, `volatile`                             |
| **C99**              | 1999      | `//` comments, `stdint.h`, VLAs, designated initializers, `_Bool`, `inline`, `restrict`             |
| **C11**              | 2011      | `_Atomic`, `_Thread_local`, `_Generic`, anonymous structs/unions, `static_assert`, `_Noreturn`      |
| **C17/C18**          | 2018      | Bug-fix release; no new features, clarified undefined behavior                                      |
| **C23**              | 2024      | `typeof`, `nullptr`, `bool` as keyword, `#embed`, `[[attributes]]`, binary literals `0b`, `_BitInt` |

:::tip[Which Standard to Use?]
For most new projects, target **C11** or **C17**. They offer the best balance of features and compiler support. Use C23 when your compiler fully supports it — GCC 13+ and Clang 16+ have substantial C23 support.
:::

## C vs C++ Comparison

| Feature               | C                                      | C++                                       |
| --------------------- | -------------------------------------- | ----------------------------------------- |
| Paradigm              | Procedural                             | Multi-paradigm (OOP, generic, functional) |
| Classes               | No (structs only)                      | Yes                                       |
| Namespaces            | No                                     | Yes                                       |
| Templates             | No                                     | Yes                                       |
| Exceptions            | No (use setjmp/longjmp or error codes) | Yes                                       |
| RAII                  | Manual                                 | Automatic (destructors)                   |
| Standard library size | Small                                  | Large (STL)                               |
| Compilation speed     | Fast                                   | Slow (especially templates)               |
| ABI stability         | Relatively stable                      | Fragile (name mangling)                   |
| Learning curve        | Moderate                               | Steep                                     |
| Header-only libraries | Common                                 | Very common                               |

C is not a "subset" of C++ — there are programs that are valid C but not valid C++, and many C idioms have different or undefined behavior in C++.

## Compilers

### GCC — GNU Compiler Collection

The most widely used C compiler on Linux and embedded targets.

```bash
# Install on Debian/Ubuntu
sudo apt install gcc

# Check version
gcc --version

# Compile a file
gcc -std=c11 -Wall -Wextra -o hello hello.c
```

### Clang — LLVM C Compiler

Excellent error messages, fast compilation, and drop-in GCC replacement.

```bash
# Install on macOS (via Xcode Command Line Tools)
xcode-select --install

# Install on Ubuntu
sudo apt install clang

# Compile
clang -std=c11 -Wall -Wextra -o hello hello.c
```

### MSVC — Microsoft Visual C++

The compiler for Windows development, available through Visual Studio or Build Tools.

```bash
# Command line (from Developer Command Prompt)
cl /std:c11 /W4 hello.c
```

### TCC — Tiny C Compiler

Extremely fast compilation, useful for scripting and rapid prototyping.

```bash
tcc -run hello.c      # Compile and run in one step
tcc -o hello hello.c  # Compile to binary
```

## The Compilation Pipeline

C source code passes through four distinct stages before becoming an executable:

<Steps>

1. **Preprocessing** — The C preprocessor (`cpp`) processes `#include`, `#define`, and `#if` directives. The output is a single expanded translation unit of pure C source.

   ```bash
   gcc -E hello.c -o hello.i   # Stop after preprocessing
   ```

2. **Compilation** — The compiler translates the preprocessed C source to assembly language specific to the target architecture.

   ```bash
   gcc -S hello.i -o hello.s   # Stop after compilation (assembly output)
   ```

3. **Assembly** — The assembler (`as`) converts the assembly text to an object file containing machine code and a symbol table.

   ```bash
   gcc -c hello.s -o hello.o   # Produce object file
   # Or directly from C:
   gcc -c hello.c -o hello.o
   ```

4. **Linking** — The linker (`ld`, invoked by `gcc`) combines object files and libraries, resolves symbol references, and produces the final executable.

   ```bash
   gcc hello.o -o hello        # Link to executable
   ```

</Steps>

You can observe each stage:

```bash
# See the preprocessed output (thousands of lines from stdio.h)
gcc -E hello.c | head -50

# See the assembly output
gcc -S -O2 hello.c && cat hello.s

# See symbols in object file
nm hello.o

# See dynamic library dependencies
ldd hello    # Linux
otool -L hello  # macOS
```

## Hello World — Every Line Explained

```c
// hello.c — A fully annotated Hello World in C11

// 1. Include directive: tells the preprocessor to copy the contents of
//    stdio.h into this translation unit. stdio.h declares printf, scanf,
//    FILE, stdin, stdout, stderr, and related constants.
#include <stdio.h>

// 2. int main(void) — the entry point of every C program.
//    'int' is the return type — main returns an integer status code to
//    the operating system. 'void' in the parameter list means the program
//    ignores command-line arguments. The alternative is:
//    int main(int argc, char *argv[])
//    where argc is the argument count and argv is the argument vector.
int main(void)
{
    // 3. printf is a variadic function from stdio.h. The first argument
    //    is a format string. "Hello, World!\n" is a string literal stored
    //    in the read-only data segment. \n is the newline escape sequence.
    //    printf returns the number of characters written (we ignore it here).
    printf("Hello, World!\n");

    // 4. return 0 signals success to the operating system.
    //    Non-zero values indicate errors (by convention).
    //    In C99+, reaching the end of main() implicitly returns 0,
    //    but explicit return is good practice.
    return 0;
}
```

Compile and run:

```bash
gcc -std=c11 -Wall -Wextra -o hello hello.c
./hello
# Output: Hello, World!
```

## C11 Key Features

C11 brought significant additions to the language:

```c
#include <stdio.h>
#include <stdatomic.h>
#include <threads.h>
#include <stdalign.h>
#include <stdbool.h>
#include <assert.h>

// 1. Static assertions (checked at compile time)
static_assert(sizeof(int) >= 4, "int must be at least 32 bits");
static_assert(sizeof(void*) == 8, "Expected 64-bit pointers");

// 2. _Generic — type-generic expressions (pseudo-overloading)
#define type_name(x) _Generic((x),  \
    int:    "int",                   \
    float:  "float",                 \
    double: "double",                \
    char*:  "string",                \
    default: "unknown")

// 3. _Noreturn — hint that function never returns
_Noreturn void fatal(const char *msg) {
    fprintf(stderr, "Fatal: %s\n", msg);
    exit(1);
}

// 4. alignas / alignof from stdalign.h
struct AlignedBuffer {
    alignas(64) char data[64];   // 64-byte aligned (cache line)
};

// 5. Anonymous structs and unions
typedef struct {
    union {
        struct { float x, y, z; };   // Access as vec.x, vec.y, vec.z
        float v[3];                   // Also access as vec.v[0]
    };
} Vec3;

// 6. Atomic operations
atomic_int counter = ATOMIC_VAR_INIT(0);

int main(void) {
    printf("%s\n", type_name(42));       // "int"
    printf("%s\n", type_name(3.14f));    // "float"
    printf("%s\n", type_name("hello"));  // "string"

    Vec3 v = { .x = 1.0f, .y = 2.0f, .z = 3.0f };
    printf("%.1f %.1f %.1f\n", v.v[0], v.v[1], v.v[2]);

    atomic_fetch_add(&counter, 1);
    printf("Counter: %d\n", atomic_load(&counter));

    struct AlignedBuffer buf;
    printf("Alignment: %zu\n", alignof(buf));  // 64

    return 0;
}
```

## Behavior Categories

Understanding how C defines (or doesn't define) behavior is critical for writing correct code:

:::caution[Undefined Behavior is Dangerous]
Undefined behavior (UB) is not just "implementation-defined" — it means the compiler can assume it never happens and will optimize accordingly, potentially producing completely wrong code. UB is the root cause of countless security vulnerabilities.
:::

### Undefined Behavior (UB)

The standard says nothing about what happens. The compiler may do _anything_, including:

```c
int x = INT_MAX;
x = x + 1;          // UB: signed integer overflow

int arr[10];
arr[10] = 5;        // UB: out-of-bounds write

int *p = NULL;
*p = 42;            // UB: null pointer dereference

int i = 0;
i = i++ + ++i;      // UB: multiple modifications without sequence point

char *s = "hello";
s[0] = 'H';         // UB: modifying string literal
```

**Tools to detect UB:**

```bash
gcc -fsanitize=undefined -fsanitize=address -g program.c -o program
```

### Implementation-Defined Behavior

The behavior is specified but varies by implementation. The implementation must document it.

```c
// Size of fundamental types (varies by platform/ABI)
printf("sizeof(int) = %zu\n", sizeof(int));   // 4 on most 32/64-bit systems

// Right-shifting signed negative integers
int n = -8;
int r = n >> 1;   // Implementation-defined: arithmetic or logical shift
```

### Unspecified Behavior

The behavior is one of a finite set of options, and the implementation needn't document which.

```c
// Order of evaluation of function arguments
int a = 1;
printf("%d %d\n", a++, a++);   // Unspecified: which ++ happens first
```

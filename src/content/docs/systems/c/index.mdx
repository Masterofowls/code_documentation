# C Programming Language

C is one of the most influential programming languages ever created. Nearly every modern operating system kernel, embedded firmware, database engine, and runtime virtual machine has C at its core. Understanding C means understanding the machine.

## History

C was developed by **Dennis Ritchie** at **Bell Labs** between 1969 and 1973, primarily as a language to rewrite the Unix operating system. It evolved from an earlier language called **B** (itself derived from BCPL), inheriting its low-level philosophy while adding data types.

The first edition of *The C Programming Language* by Brian Kernighan and Dennis Ritchie — universally known as "K&R C" — was published in 1978 and served as the de-facto standard for a decade.

```c
/* Classic K&R Hello World — 1978 */
main()
{
    printf("hello, world\n");
}
```

The language spread rapidly because Unix spread rapidly — and Unix was portable precisely because it was written in C.

## C Standards

| Standard | Year | Key Additions |
|----------|------|---------------|
| K&R C    | 1978 | Original definition |
| **C89/C90** | 1989/1990 | First ANSI/ISO standard; function prototypes; `void`; standardized library |
| **C99**  | 1999 | `//` comments; `_Bool`; `long long`; VLAs; `restrict`; `stdint.h`; compound literals; designated initializers; inline functions |
| **C11**  | 2011 | `_Alignas`/`_Alignof`; `_Atomic`; `_Generic`; `_Noreturn`; `_Static_assert`; threads (`<threads.h>`); anonymous structs/unions; `gets` removed |
| **C17**  | 2018 | Bug fixes only — no new features; defect resolutions |
| **C23**  | 2023 | `bool`/`true`/`false` as keywords; binary literals `0b`; `#embed`; `nullptr`; attributes `[[nodiscard]]`; `typeof`; `_BitInt(N)` |

:::tip[Choosing a Standard]
Use `-std=c11` as your default. It offers a well-supported feature set across GCC, Clang, and MSVC. Use `-std=c99` only for maximum portability with older compilers. C23 support is still maturing in compilers as of 2024.
:::

## C vs C++ Comparison

| Feature | C | C++ |
|---------|---|-----|
| Paradigm | Procedural | Multi-paradigm (OOP, generic, procedural) |
| Classes | No | Yes |
| Templates | No | Yes |
| Standard library | Minimal | Rich (STL) |
| RAII | Manual | Yes |
| Compile speed | Fast | Slower |
| ABI stability | Very stable | Fragile (name mangling) |
| Embedded use | Excellent | Good (with care) |
| Learning curve | Moderate | Steep |
| VLAs | Yes (C99, optional C11) | No (not in standard) |

C++ is largely a superset of C, but they are not identical. Valid C code can subtly mean something different in C++.

## Compilers

<Tabs>
  <TabItem label="GCC">
    The **GNU Compiler Collection** — the reference implementation for C on Linux. Supports all standards through C23 (partial). Available on all major platforms.

    ```bash
    gcc -std=c11 -Wall -Wextra -O2 -o program main.c
    ```
  </TabItem>
  <TabItem label="Clang">
    The LLVM-based compiler. Excellent error messages, fast compile times, built-in static analysis. Default on macOS.

    ```bash
    clang -std=c11 -Wall -Wextra -fsanitize=address -o program main.c
    ```
  </TabItem>
  <TabItem label="MSVC">
    Microsoft Visual C++ compiler — required for Windows Driver Kit and many Windows APIs. C11/C17 support added in VS 2019 16.8+.

    ```bash
    cl /std:c11 /Wall /O2 main.c
    ```
  </TabItem>
  <TabItem label="TCC">
    **Tiny C Compiler** — compiles in milliseconds, supports C99. Useful for scripting-style C programs, not production use.

    ```bash
    tcc -run main.c
    ```
  </TabItem>
</Tabs>

## The Compilation Pipeline

C source code goes through four distinct stages before becoming an executable:

<Steps>
1. **Preprocessing** — The preprocessor (`cpp`) handles all `#` directives. It expands macros, includes header files, strips comments, and processes `#ifdef`/`#endif` guards. Output is a single expanded `.i` file.

   ```bash
   gcc -E main.c -o main.i
   ```

2. **Compilation** — The compiler proper translates preprocessed C into **assembly language** specific to the target architecture (x86-64, ARM, RISC-V, etc.). Output is a `.s` file.

   ```bash
   gcc -S main.i -o main.s
   ```

3. **Assembly** — The assembler (`as`) converts assembly mnemonics into **machine code** (binary object code). Output is a relocatable `.o` (object) file. Symbols are unresolved placeholders at this stage.

   ```bash
   gcc -c main.s -o main.o
   ```

4. **Linking** — The linker (`ld`) combines one or more `.o` files with library archives (`.a` static, `.so`/`.dll` dynamic), resolves all symbol references, assigns final memory addresses, and produces the executable.

   ```bash
   gcc main.o -o program
   ```
</Steps>

You can observe all stages with:
```bash
gcc -v -std=c11 main.c -o program   # verbose — shows every sub-command
```

## Your First C Program — Explained

```c
/* main.c — annotated Hello World */

// 1. Preprocessor directive: include the Standard I/O header.
//    <stdio.h> declares printf, scanf, FILE, etc.
//    Angle brackets search the system include path.
#include <stdio.h>

// 2. main() is the program entry point.
//    int return type: convention — 0 = success, non-zero = error.
//    void means we accept no command-line arguments here.
int main(void)
{
    // 3. printf: formatted print to stdout.
    //    "Hello, World!\n" — string literal; \n is a newline escape.
    //    Returns number of characters written (we ignore it here).
    printf("Hello, World!\n");

    // 4. Return 0 to the operating system — signals success.
    //    The OS can read this with $? (bash) or %ERRORLEVEL% (cmd).
    return 0;
}
```

Compile and run:
```bash
gcc -std=c11 -Wall -o hello main.c
./hello
# Hello, World!
```

## C11 Standard Highlights

C11 brought several important additions that are now widely expected in modern C code:

```c
#include <stdio.h>
#include <stdint.h>
#include <stdalign.h>   // _Alignas, _Alignof
#include <stdatomic.h>  // _Atomic
#include <threads.h>    // thrd_*, mtx_*, cnd_*
#include <stdbool.h>    // bool, true, false (C99, but confirmed in C11)

// Static assertions — checked at compile time, not runtime
_Static_assert(sizeof(int) == 4, "int must be 4 bytes on this platform");
// C11 shorthand:
static_assert(sizeof(long) >= 4, "long must be at least 4 bytes");

// Type-generic macros via _Generic
#define type_name(x) _Generic((x),          \
    int:       "int",                        \
    long:      "long",                       \
    float:     "float",                      \
    double:    "double",                     \
    default:   "other")

// Aligned allocation
alignas(64) uint8_t cache_line_buffer[64];   // 64-byte aligned

// Atomic types
_Atomic int shared_counter = 0;
// or with the macro form:
atomic_int another_counter;

// Anonymous structs and unions (C11)
typedef struct {
    union {
        struct { float x, y, z; };   // anonymous struct inside union
        float v[3];
    };
} Vec3;

int main(void)
{
    Vec3 pos = { .x = 1.0f, .y = 2.0f, .z = 3.0f };
    printf("pos.v[0] = %.1f\n", pos.v[0]);  // same memory as pos.x

    printf("Type of 42:   %s\n", type_name(42));
    printf("Type of 3.14: %s\n", type_name(3.14));
    return 0;
}
```

## Behavior Classifications

Understanding these categories is essential to writing correct, portable C:

| Category | Definition | Example |
|----------|------------|---------|
| **Defined Behavior** | The standard specifies exactly what happens | `1 + 1 == 2`, `malloc(4)` returns 4-byte buffer or NULL |
| **Implementation-defined** | Behavior varies by compiler but must be documented | `sizeof(int)`, `char` signedness, endianness |
| **Unspecified behavior** | Behavior is not defined but must be one of a set of valid choices | Order of evaluation of function arguments |
| **Undefined behavior (UB)** | The standard places no requirements; compiler may do *anything* | Signed integer overflow, null pointer dereference, out-of-bounds access, data race |

:::caution[Undefined Behavior is Not "Do Nothing"]
Modern compilers aggressively optimize around UB. If a branch can only be reached via UB, the compiler eliminates it entirely. UB does not mean "it will probably work" — it means the program is **incorrect**, and optimized builds may behave radically differently from debug builds.

```c
// UNDEFINED BEHAVIOR — signed overflow
int x = INT_MAX;
if (x + 1 < 0)         // Compiler may delete this branch entirely!
    puts("overflow");   // This may never print in optimized builds
```
:::

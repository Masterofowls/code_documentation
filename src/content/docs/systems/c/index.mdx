# C Programming Language

C is one of the most influential programming languages ever created. Born in the early 1970s at Bell Labs, it remains a cornerstone of systems programming, embedded firmware, OS kernels, and performance-critical applications more than fifty years later.

## History

C was developed by **Dennis Ritchie** at Bell Laboratories between 1969 and 1973. It grew out of an earlier language called **B**, which Ken Thompson had written for the PDP-7. The primary motivation was to rewrite the Unix operating system in a portable, high-level language rather than assembly.

Key milestones:

| Year | Event |
|------|-------|
| 1972 | Dennis Ritchie develops C at Bell Labs |
| 1973 | Unix kernel rewritten in C |
| 1978 | *The C Programming Language* (K&R C) published by Kernighan and Ritchie |
| 1989 | ANSI C (C89) ratified — the first formal standard |
| 1990 | ISO adopts C89 as ISO/IEC 9899:1990 (C90) |
| 1999 | C99 standard published — adds `//` comments, VLAs, `stdint.h`, `stdbool.h`, designated initializers |
| 2011 | C11 standard — adds threading (`threads.h`), atomics, `_Generic`, anonymous structs/unions, `_Static_assert` |
| 2017 | C17 (C18) — a bug-fix release with no new features |
| 2023 | C23 — adds `#embed`, `typeof`, bit-precise integers `_BitInt(N)`, `nullptr`, improved type inference |

:::tip[K&R vs ANSI C]
If you encounter old C code without function prototypes like `int add(a, b) int a, b; { ... }`, that is K&R style. Always write ANSI C or later. Modern compilers warn about K&R definitions.
:::

## C Standards Comparison

```c
// C89: no inline comments, no stdbool.h, no stdint.h
int x = 0;
unsigned long flags = 0UL;

// C99: inline comments, stdbool.h, stdint.h, VLAs, designated init
#include <stdbool.h>
#include <stdint.h>
bool done = false;
uint32_t count = 0;
int arr[n]; // VLA — size known at runtime

// C11: _Generic, _Static_assert, _Alignas, _Thread_local, atomics
_Static_assert(sizeof(int) == 4, "int must be 4 bytes");
_Alignas(16) float simd_data[4];
#include <stdatomic.h>
atomic_int counter = ATOMIC_VAR_INIT(0);

// C23: typeof, nullptr, #embed, _BitInt(N), improved auto
typeof(int) x2 = 42;         // typeof returns the type of an expression
int *p = nullptr;             // nullptr constant (was NULL before)
_BitInt(7) tiny = 63;         // 7-bit integer
```

## C vs C++

| Feature | C | C++ |
|---------|---|-----|
| Object-Oriented Programming | No | Yes (classes, inheritance, polymorphism) |
| Templates / Generics | Macros only | Full template system |
| RAII / Constructors/Destructors | No | Yes |
| Exception Handling | No (use setjmp/longjmp) | Yes (try/catch/throw) |
| Standard Library | Minimal (libc) | Rich STL |
| Name Mangling | No | Yes (extern "C" needed for interop) |
| Compilation Speed | Fast | Slower (especially with templates) |
| ABI Stability | Stable | Complex |
| Use Cases | OS, embedded, firmware | Applications, games, tools |
| Learning Curve | Moderate | Steep |

:::caution[C is not a subset of C++]
While most C code compiles as C++, there are important differences. In C, you can assign a `void*` to any pointer without a cast. In C++, this is a compile error. Additionally, C and C++ have different rules for struct tags, VLAs, and designated initializers.
:::

## C's Role Today

Despite being over 50 years old, C continues to thrive in specific domains:

- **Operating System Kernels** — Linux, Windows NT, macOS (XNU), FreeBSD are written primarily in C
- **Embedded Systems** — Microcontrollers (AVR, ARM Cortex-M, PIC) with limited resources
- **Firmware** — BIOS/UEFI, device drivers, bootloaders
- **Compilers and Interpreters** — CPython, Lua, SQLite are written in C
- **High-Performance Libraries** — OpenBLAS, FFTW, libjpeg, zlib
- **Network Infrastructure** — nginx, Redis, memcached, OpenSSL
- **Database Engines** — SQLite, PostgreSQL (core), parts of MySQL/MariaDB
- **WebAssembly Toolchain** — Emscripten compiles C to WASM

## Compilation Pipeline

A C source file goes through four distinct stages before becoming an executable binary:

```
main.c ──► Preprocessor ──► main.i ──► Compiler ──► main.s ──► Assembler ──► main.o ──► Linker ──► main
```

<Steps>
1. **Preprocessing** — The C preprocessor (`cpp`) handles all directives beginning with `#`. It expands `#include` files, substitutes `#define` macros, evaluates `#ifdef`/`#endif` conditional compilation blocks, and strips comments. Output is a `.i` file (expanded C source).

2. **Compilation** — The compiler parses the preprocessed C source, performs semantic analysis, type-checking, and optimization, then generates architecture-specific assembly code (`.s` file).

3. **Assembly** — The assembler (`as`) converts the assembly text into machine-code object files (`.o` on Unix, `.obj` on Windows). Each source file becomes one object file (translation unit).

4. **Linking** — The linker (`ld`) combines all object files and required library archives (`.a`, `.lib`) to resolve symbol references and produce the final executable or shared library.
</Steps>

You can inspect each stage:

```bash
# Run only the preprocessor
gcc -E main.c -o main.i

# Compile to assembly only
gcc -S main.c -o main.s

# Compile to object file (no linking)
gcc -c main.c -o main.o

# Full compilation and link
gcc main.c -o main

# View symbols in object file
nm main.o

# View disassembly
objdump -d main
```

## Hello, World — Every Line Explained

```c
#include <stdio.h>      /* (1) Include the standard I/O header */

int main(void)          /* (2) Entry point: returns int, takes no arguments */
{
    printf("Hello, World!\n");  /* (3) Print to stdout */
    return 0;                   /* (4) Return 0 to the OS: success */
}
```

1. `#include <stdio.h>` — A preprocessor directive that inserts the contents of the system header `stdio.h`. This provides the declaration of `printf`. Angle brackets `<>` tell the preprocessor to search system include paths; quotes `""` search the local directory first.

2. `int main(void)` — The program entry point. `int` is the return type (exit code). `void` explicitly states no command-line arguments are accepted. Alternatively, `int main(int argc, char *argv[])` accepts arguments. Note: in C89 an empty parameter list `()` means "any arguments" — use `void` to mean "no arguments".

3. `printf("Hello, World!\n")` — Calls the `printf` function from `<stdio.h>`. The string literal `"Hello, World!\n"` is the format string. `\n` is the newline escape sequence. `printf` returns the number of characters written.

4. `return 0;` — Exits `main` with status code 0, which by convention signals success to the calling process (shell). Non-zero return values indicate errors.

## C11 Key Features

```c
#include <stdio.h>
#include <stdalign.h>   // alignas, alignof
#include <stdnoreturn.h> // noreturn
#include <stdatomic.h>  // atomic types
#include <threads.h>    // thrd_t, mtx_t, cnd_t

// _Static_assert: compile-time assertion
_Static_assert(sizeof(long) >= 4, "long must be at least 4 bytes");

// _Alignas / _Alignof: alignment control
_Alignas(64) char cache_line[64]; // aligned to 64-byte cache line
size_t al = _Alignof(double);     // query alignment requirement

// _Generic: compile-time type-based selection (like overloading)
#define abs_val(x) _Generic((x),    \
    int:    abs(x),                  \
    long:   labs(x),                 \
    float:  fabsf(x),                \
    double: fabs(x),                 \
    default: fabs(x))

// _Noreturn: function never returns
_Noreturn void fatal_error(const char *msg) {
    fprintf(stderr, "Fatal: %s\n", msg);
    exit(EXIT_FAILURE);
}

// Anonymous structs and unions (C11)
typedef struct {
    int type;
    union {               // anonymous union — members accessible directly
        int   i_val;
        float f_val;
        char *s_val;
    };
} Value;

Value v = { .type = 1, .i_val = 42 };
printf("%d\n", v.i_val); // no v.u.i_val needed
```

## Behavior Categories in C

Understanding the three categories of non-standard behavior is essential for writing portable, correct C:

### Undefined Behavior (UB)

The C standard places no requirements on the behavior — the compiler may do anything: crash, corrupt data, produce wrong output, or apparently work correctly. UB allows compilers to apply aggressive optimizations.

```c
// Examples of undefined behavior — DO NOT do these
int arr[5];
arr[10] = 0;          // UB: out-of-bounds write
int *p = NULL;
*p = 5;               // UB: null pointer dereference
int x = INT_MAX;
x = x + 1;            // UB: signed integer overflow
int y;
printf("%d\n", y);    // UB: uninitialized variable read
char *s = "hello";
s[0] = 'H';           // UB: modifying a string literal
```

:::caution[Undefined Behavior and Optimizations]
Modern compilers use UB as a license to optimize. If you signed-integer-overflow check `if (x + 1 < x)`, the compiler may eliminate it entirely because signed overflow is UB and therefore "cannot happen." Use `-fsanitize=undefined` during development to catch UB at runtime.
:::

### Implementation-Defined Behavior

The behavior is defined by the implementation (compiler + platform) and must be documented. It is consistent and predictable on a given platform but may differ between platforms.

```c
// Size of types — implementation-defined
printf("sizeof(int) = %zu\n", sizeof(int));  // 4 on most 32/64-bit platforms
printf("sizeof(long) = %zu\n", sizeof(long)); // 4 on Windows 64-bit, 8 on Linux 64-bit

// Right-shifting a negative integer — implementation-defined
int n = -8;
int r = n >> 1;  // Arithmetic shift (-4) on most platforms, but not guaranteed
```

### Unspecified Behavior

The standard provides two or more options and does not require the implementation to document which it uses. The behavior may vary from call to call.

```c
// Order of evaluation of function arguments — unspecified
int i = 0;
printf("%d %d\n", i++, i++); // unspecified: either "0 1" or "1 0"

// Order of evaluation of sub-expressions — unspecified (unless short-circuit)
int a = 1;
int b = (a = 2) + (a = 3); // unspecified order of assignments
```

## Standards Compliance Flags

```bash
# GCC/Clang: specify standard and enable strict conformance
gcc -std=c11 -pedantic -Wall -Wextra -Werror main.c

# -std=c99     Use C99
# -std=c11     Use C11  
# -std=c17     Use C17 (same as c18)
# -std=c23     Use C23 (GCC 13+, Clang 17+)
# -pedantic    Warn for extensions not in the standard
# -Wall        Enable most useful warnings
# -Wextra      Enable extra warnings not covered by -Wall
# -Werror      Treat all warnings as errors
```

Code coverage measures how much of your source code is exercised by your test suite. Jest has built-in coverage support powered by Istanbul (Babel transform) or V8 (native Node.js coverage) — no additional tools required.

## Running Coverage

```bash
# Run all tests and collect coverage
npx jest --coverage

# Run coverage for specific files
npx jest --coverage --testPathPattern=src/services

# Run coverage and open the HTML report
npx jest --coverage && open coverage/lcov-report/index.html
```

Jest outputs a summary table to the terminal:

```
--------------------|---------|----------|---------|---------|-------------------
File                | % Stmts | % Branch | % Funcs | % Lines | Uncovered Line #s
--------------------|---------|----------|---------|---------|-------------------
All files           |   87.50 |    75.00 |   90.00 |   88.46 |
 src/               |         |          |         |         |
  math.ts           |  100.00 |   100.00 |  100.00 |  100.00 |
  utils.ts          |   80.00 |    50.00 |   83.33 |   81.82 | 24-26, 48
  api/              |         |          |         |         |
   client.ts        |   75.00 |    66.67 |   80.00 |   76.92 | 45, 67-71, 89
--------------------|---------|----------|---------|---------|-------------------
```

## The Four Coverage Dimensions

| Metric | What is measured | Example |
|---|---|---|
| **Statements** | Every executable statement executed at least once | `const x = getValue();` |
| **Branches** | Every conditional branch taken (both true and false paths) | `if (x > 0) { ... } else { ... }` |
| **Functions** | Every declared function called at least once | `function validate(input) { ... }` |
| **Lines** | Every source line touched at least once | Line 42 in source file |

**Branch coverage is the most valuable metric.** A function can have 100% statement coverage but only 50% branch coverage — meaning half of your conditional logic is untested.

```typescript
// This function has 100% statement coverage but only 50% branch coverage
// if you only test with a positive number
const describeNumber = (n: number): string => {
  if (n > 0) {
    return 'positive'; // ✅ covered
  } else {
    return 'non-positive'; // ❌ not covered (missed branch)
  }
};
```

## Coverage Reporters

Configure reporters in `jest.config.ts`:

```typescript
const config: Config = {
  coverageReporters: [
    'text',         // Terminal summary table (default)
    'text-summary', // Single-line summary only
    'lcov',         // Industry-standard LCOV format (for CI tools, Codecov, Coveralls)
    'html',         // Human-readable HTML report at coverage/lcov-report/
    'json',         // Raw JSON at coverage/coverage-final.json
    'json-summary', // Summary JSON at coverage/coverage-summary.json
    'clover',       // Clover XML format (for SonarQube, Jenkins)
    'cobertura',    // Cobertura XML (for Azure DevOps, GitLab)
  ],
  coverageDirectory: 'coverage', // Where to write reports (default)
};
```

## Coverage Thresholds

Enforce minimum coverage levels to prevent coverage from declining over time. Tests fail if coverage drops below the threshold.

```typescript
const config: Config = {
  coverageThreshold: {
    // Global thresholds — across all files
    global: {
      statements: 80,
      branches: 75,
      functions: 80,
      lines: 80,
    },
    // Per-file thresholds
    './src/core/auth.ts': {
      statements: 95,
      branches: 90,
      functions: 100,
      lines: 95,
    },
    // Glob pattern — all files in a directory
    './src/services/**/*.ts': {
      statements: 85,
      branches: 80,
      functions: 85,
      lines: 85,
    },
  },
};
```

When coverage falls below the threshold, Jest exits with a non-zero exit code:

```
Jest: "global" coverage threshold for statements (80%) not met: 74%
Jest: "global" coverage threshold for branches (75%) not met: 62%
```

:::caution[Thresholds Must Be Intentional]
Avoid setting thresholds at or just below your current coverage percentage. The purpose of thresholds is to prevent regression. Start at a realistic floor and raise it gradually as coverage improves.
:::

## Coverage Providers

Jest supports two coverage collection engines:

```typescript
const config: Config = {
  // 'babel' (default) or 'v8'
  coverageProvider: 'babel',
};
```

| Feature | `babel` (Istanbul) | `v8` (Node.js native) |
|---|---|---|
| Requires transform | Yes | No (works on native code) |
| Speed | Moderate | Faster (native instrumentation) |
| Accuracy | High | Higher for branch coverage |
| Source maps | Excellent | Good |
| TypeScript support | Via ts-jest / babel | Good with source maps |
| Recommended for | Most projects | Pure Node.js, ESM |

:::tip[Which Provider to Choose?]
Use `babel` (the default) for most TypeScript and React projects. Use `v8` if your project uses native ESM, you're on Node.js 18+, or you notice the babel coverage is slower than acceptable.
:::

## Specifying Which Files to Cover

By default, Jest only reports coverage for files that are `require()`d by your tests. Files that are never imported will not appear in the report, making coverage look better than it is.

```typescript
const config: Config = {
  // Collect coverage from ALL source files, even unimported ones
  collectCoverageFrom: [
    'src/**/*.{ts,tsx}',
    '!src/**/*.d.ts',                 // Exclude type declaration files
    '!src/**/*.stories.{ts,tsx}',     // Exclude Storybook stories
    '!src/**/__mocks__/**',            // Exclude mock files
    '!src/**/index.{ts,tsx}',         // Exclude barrel files (re-exports only)
    '!src/main.tsx',                  // Exclude application entry point
    '!src/types/**',                  // Exclude type-only files
  ],
};
```

### Excluding Files with Comments

You can exclude specific blocks of code from coverage using inline comments:

```typescript
/* istanbul ignore file */  // Exclude entire file from coverage

function handler() {
  /* istanbul ignore next */
  if (process.env.NODE_ENV === 'production') {
    // This branch is hard to test in unit tests
    sendMetrics();
  }
}

// Ignore the next branch
/* istanbul ignore else */
if (isFeatureEnabled) {
  newFeature();
} else {
  legacyFeature();
}
```

## Reading the HTML Report

The HTML report provides line-by-line coverage visualization:

- 🟢 **Green lines** — executed by at least one test
- 🔴 **Red lines** — never executed
- 🟡 **Yellow lines** — partially covered (e.g., `if/else` where only one branch ran)
- **Branch indicators** — `E` (else branch not taken) and `I` (if branch not taken) appear inline

Navigate to a specific file to see exactly which lines need additional tests:

```
  42 |  const processPayment = async (amount: number, method: string) => {
  43 |    if (amount <= 0) {                     // ← Branch: else not taken (E)
  44 |      throw new ValidationError('...');    // ← Red: never executed
  45 |    }
  46 |    if (method === 'crypto') {             // ← Branch: if not taken (I)
  47 |      return cryptoGateway.charge(amount); // ← Red: never executed
  48 |    }
  49 |    return stripeGateway.charge(amount);   // ← Green: executed
  50 |  };
```

## Testing Uncovered Branches

When the HTML report reveals uncovered branches, add targeted tests:

```typescript
// Before: only the happy path was tested
it('processes a valid payment', async () => {
  const result = await processPayment(100, 'card');
  expect(result.success).toBe(true);
});

// After: added tests for all uncovered branches
it('throws ValidationError for zero amount', async () => {
  await expect(processPayment(0, 'card')).rejects.toThrow(ValidationError);
});

it('throws ValidationError for negative amount', async () => {
  await expect(processPayment(-50, 'card')).rejects.toThrow(ValidationError);
});

it('routes to crypto gateway when method is crypto', async () => {
  const mockCharge = jest.spyOn(cryptoGateway, 'charge').mockResolvedValue({ success: true });
  await processPayment(100, 'crypto');
  expect(mockCharge).toHaveBeenCalledWith(100);
});
```

## 100% Coverage — Pitfalls and Realistic Goals

:::caution[100% Coverage Can Be Misleading]
100% statement coverage does not mean your code is bug-free. You can have 100% coverage and still miss:
- Edge cases (empty strings, null inputs, boundary values)
- Concurrent execution bugs
- Integration mismatches between modules
- Incorrect business logic that passes wrong inputs to functions

Cover behaviour, not lines.
:::

A healthy coverage strategy:

| Code type | Suggested threshold |
|---|---|
| Core business logic | 90–100% branches |
| API handlers / controllers | 85%+ |
| Utility functions | 95–100% |
| React components | 80–90% |
| Configuration files | Exclude |
| Migrations | Exclude |
| Generated code | Exclude |

## CI Integration

```yaml
# .github/workflows/test.yml
- name: Run tests with coverage
  run: npm run test:ci
  env:
    CI: true

- name: Upload coverage to Codecov
  uses: codecov/codecov-action@v4
  with:
    files: ./coverage/lcov.info
    fail_ci_if_error: true
```

```json
// package.json
{
  "scripts": {
    "test:ci": "jest --ci --coverage --coverageReporters=lcov --coverageReporters=text-summary"
  }
}
```

In your `jest.config.ts`, the `--ci` flag:
- Disables watch mode
- Fails when snapshots are out of date (instead of updating them)
- Uses a reporter optimised for CI output
- Does not write to the interactive terminal

```typescript
const config: Config = {
  // Fail CI if coverage drops below these levels
  coverageThreshold: {
    global: {
      statements: 80,
      branches: 75,
      functions: 80,
      lines: 80,
    },
  },
};
```

When coverage thresholds are not met in CI, the job exits with code 1, blocking the merge:

```
FAIL coverage threshold not met:
  statements: 74% (required: 80%)

npm ERR! Test failed. See above for more details.
```

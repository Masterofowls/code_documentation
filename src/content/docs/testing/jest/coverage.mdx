Code coverage measures how much of your source code is exercised by your test suite. Jest integrates Istanbul (the industry-standard JavaScript coverage tool) and optionally V8's native coverage engine — both without any extra dependencies.

## Running Coverage

```bash
# Run all tests and generate a coverage report
npx jest --coverage

# or via npm script
npm run test:coverage
```

Jest generates a summary table in the terminal:

```
 PASS  src/utils/math.test.ts
 PASS  src/services/user.service.test.ts

-----------|---------|----------|---------|---------|-------------------
File       | % Stmts | % Branch | % Funcs | % Lines | Uncovered Line #s
-----------|---------|----------|---------|---------|-------------------
All files  |   87.50 |    75.00 |   85.71 |   87.50 |
 math.ts   |   100   |    100   |   100   |   100   |
 user.ts   |   75.00 |    50.00 |   71.43 |   75.00 | 34-38, 52
-----------|---------|----------|---------|---------|-------------------
```

## Coverage Metrics Explained

Jest reports four distinct coverage metrics:

| Metric | What it Measures | Example |
|---|---|---|
| **Statements** | Every executable statement | `const x = 1;`, `return x;` |
| **Branches** | Each branch of control flow | Both sides of `if/else`, `? :`, `&&`, `||` |
| **Functions** | Each declared function/method | Whether the function was ever called |
| **Lines** | Physical lines of code containing statements | Similar to statements but line-based |

### Understanding Branch Coverage

Branches are the trickiest metric. Every `if`, `else`, `&&`, `||`, `??`, and `? :` creates a branch:

```typescript
// This function has 4 branches:
// - if condition: true and false
// - permission check: true and false
function canAccess(user: User | null, resource: string): boolean {
  if (!user) return false;                    // branch 1 (no user)
  return user.permissions.includes(resource); // branch 2 (user exists)
}

// This has 2 branches:
function getDisplayName(user: User): string {
  return user.displayName ?? user.email;  // ?? creates 2 branches
}

// This has 4 branches:
function classifyAge(age: number): string {
  if (age < 18) return 'minor';        // branch 1
  else if (age < 65) return 'adult';   // branch 2
  else return 'senior';                // branch 3
  // unreachable — would be branch 4 if explicit
}
```

To achieve 100% branch coverage, every possible path must be exercised:

```typescript
describe('canAccess', () => {
  it('returns false when user is null', () => {
    expect(canAccess(null, 'admin-panel')).toBe(false); // covers branch 1
  });

  it('returns false when user lacks the permission', () => {
    const user = { permissions: ['read'] };
    expect(canAccess(user, 'write')).toBe(false); // covers branch 2 (false path)
  });

  it('returns true when user has the permission', () => {
    const user = { permissions: ['write'] };
    expect(canAccess(user, 'write')).toBe(true); // covers branch 2 (true path)
  });
});
```

## Coverage Reporters

Configure which report formats Jest generates:

```typescript
// jest.config.ts
const config: Config = {
  coverageReporters: [
    'text',          // terminal summary table (always useful)
    'text-summary',  // compact one-line summary
    'lcov',          // lcov.info file — used by CI, SonarQube, Codecov
    'html',          // interactive HTML report in coverage/
    'json',          // machine-readable JSON
    'json-summary',  // compact JSON summary
    'clover',        // Clover XML (used by some CI tools)
  ],
};
```

| Reporter | Output | Use Case |
|---|---|---|
| `text` | Terminal table | Default — always include |
| `text-summary` | One-line terminal | CI log brevity |
| `html` | `coverage/index.html` | Local exploration of uncovered lines |
| `lcov` | `coverage/lcov.info` | Codecov, Coveralls, SonarQube integration |
| `json` | `coverage/coverage-final.json` | Custom processing / dashboards |
| `clover` | `coverage/clover.xml` | Atlassian Bamboo, older CI systems |

## Coverage Thresholds

Enforce minimum coverage levels — Jest will fail with a non-zero exit code when thresholds are not met, blocking CI merges.

```typescript
// jest.config.ts
const config: Config = {
  coverageThreshold: {
    global: {
      statements: 80,
      branches: 75,
      functions: 85,
      lines: 80,
    },
  },
};
```

### Per-File Thresholds

Set stricter thresholds on critical files:

```typescript
const config: Config = {
  coverageThreshold: {
    // Global minimum
    global: {
      branches: 70,
      functions: 80,
      lines: 80,
      statements: 80,
    },
    // Stricter thresholds for critical business logic
    './src/services/payment.service.ts': {
      branches: 95,
      functions: 100,
      lines: 95,
      statements: 95,
    },
    './src/utils/validation.ts': {
      branches: 100,
      functions: 100,
      lines: 100,
      statements: 100,
    },
    // Glob patterns also work
    './src/services/**/*.ts': {
      branches: 85,
      functions: 90,
      lines: 90,
      statements: 90,
    },
  },
};
```

:::tip[Start with Ratcheting]
Rather than starting with an ambitious threshold, configure a "ratchet" strategy: find your current coverage percentage, set the threshold slightly below it, then gradually raise it over time. This prevents regression without creating a large upfront debt.
:::

## Choosing a Coverage Provider

Jest supports two coverage engines:

```typescript
const config: Config = {
  coverageProvider: 'babel',  // default — instrumented at transform time
  // coverageProvider: 'v8',  // Node.js built-in coverage (faster, less accurate for branches)
};
```

| Provider | How it Works | Pros | Cons |
|---|---|---|---|
| `babel` (Istanbul) | Instruments source code at transform time | Accurate branch detection, widely supported | Slightly slower, larger overhead |
| `v8` | Uses Node.js V8 engine's native coverage | Faster, no instrumentation overhead | Less granular branch coverage |

:::caution[v8 Coverage with TypeScript]
When using the `v8` provider with TypeScript, coverage is reported against the compiled JavaScript output, not the original TypeScript source. Use `babel` / Istanbul for accurate TypeScript source coverage.
:::

## Configuring What to Collect

By default, Jest only collects coverage from files that are actually imported by your tests. Files with zero tests won't appear in the report. Use `collectCoverageFrom` to include all source files:

```typescript
const config: Config = {
  collectCoverage: false, // don't collect unless --coverage flag is passed
  collectCoverageFrom: [
    'src/**/*.{ts,tsx}',       // all TypeScript files
    '!src/**/*.d.ts',          // exclude type declaration files
    '!src/**/*.test.{ts,tsx}', // exclude test files themselves
    '!src/**/*.spec.{ts,tsx}',
    '!src/**/index.ts',        // exclude barrel files
    '!src/test-utils/**',      // exclude test utilities
    '!src/generated/**',       // exclude auto-generated code
  ],
  coverageDirectory: 'coverage',
  coveragePathIgnorePatterns: [
    '/node_modules/',
    '/__tests__/',
    '/dist/',
  ],
};
```

## Reading the HTML Report

The HTML report (`coverage/index.html`) is the most useful tool for identifying coverage gaps.

<Steps>
1. Run `npx jest --coverage`
2. Open `coverage/index.html` in your browser
3. Click a file to see line-by-line coverage highlights
4. **Green** lines are covered
5. **Red** lines are not covered
6. **Yellow** lines have partial branch coverage
</Steps>

The HTML report shows exactly which lines and branches are uncovered, making it easy to write targeted tests.

## Excluding Files from Coverage

Several mechanisms exist for excluding files:

```typescript
// 1. In jest.config.ts — collectCoverageFrom exclusions
collectCoverageFrom: [
  'src/**/*.ts',
  '!src/generated/**',    // generated code
  '!src/**/*.stories.ts', // Storybook stories
  '!src/**/*.mock.ts',    // mock files
]

// 2. Inline file comment — exclude entire file
/* istanbul ignore file */

// 3. Inline block comment — exclude a specific block
/* istanbul ignore next */
function legacyWorkaround() {
  // This function is excluded from coverage analysis
}

// 4. Inline comment for a branch
function parseConfig(value: unknown) {
  if (typeof value === 'undefined') {
    /* istanbul ignore next */
    return defaultConfig; // fallback only hit in edge cases
  }
  return value;
}
```

## Testing Untested Branches

When the HTML report shows uncovered branches (yellow highlighting), write tests that explicitly exercise them:

```typescript
// Source: src/utils/format.ts
export function formatFileSize(bytes: number): string {
  if (bytes === 0) return '0 Bytes';               // branch 1
  if (bytes < 1024) return `${bytes} Bytes`;       // branch 2
  if (bytes < 1024 * 1024) return `${(bytes / 1024).toFixed(1)} KB`;  // branch 3
  return `${(bytes / (1024 * 1024)).toFixed(1)} MB`; // branch 4
}
```

```typescript
// Test covering all branches:
describe('formatFileSize', () => {
  it('returns "0 Bytes" for zero', () => {
    expect(formatFileSize(0)).toBe('0 Bytes');        // branch 1
  });

  it('formats bytes under 1KB', () => {
    expect(formatFileSize(512)).toBe('512 Bytes');    // branch 2
  });

  it('formats kilobytes', () => {
    expect(formatFileSize(2048)).toBe('2.0 KB');      // branch 3
  });

  it('formats megabytes', () => {
    expect(formatFileSize(5_242_880)).toBe('5.0 MB'); // branch 4
  });
});
```

## Meaningful Coverage vs Vanity Metrics

:::caution[100% Coverage Is Not the Goal]
High coverage numbers do not guarantee correct tests. A test can execute every line without making a single meaningful assertion. Focus on testing behavior and edge cases, not on achieving a percentage.
:::

```typescript
// ❌ Achieves 100% coverage but tests nothing meaningful
it('runs the function', () => {
  processOrder({ id: 1, items: [] }); // no assertion — but 100% lines covered
});

// ✅ Meaningful test with clear behavior assertion
it('returns empty order total for empty items array', () => {
  const result = processOrder({ id: 1, items: [] });
  expect(result.total).toBe(0);
  expect(result.lineItems).toHaveLength(0);
});
```

| Coverage % | What it means |
|---|---|
| &lt; 50% | Large areas of code are completely untested — high risk |
| 50–70% | Basic happy-path tests exist — missing error paths |
| 70–85% | Good coverage — some edge cases missing |
| 85–95% | Strong coverage — most paths exercised |
| 95–100% | Very high coverage — appropriate for critical business logic |
| 100% | Excellent for utility functions; may be overkill for UI components |

## CI Integration

```yaml
# .github/workflows/test.yml
- name: Run tests with coverage
  run: npm run test:ci

- name: Upload coverage to Codecov
  uses: codecov/codecov-action@v4
  with:
    files: ./coverage/lcov.info
    fail_ci_if_error: true
    token: ${{ secrets.CODECOV_TOKEN }}
```

```json
// package.json
{
  "scripts": {
    "test:ci": "jest --ci --coverage --coverageReporters=text-summary,lcov"
  }
}
```

:::tip[--ci Flag in CI]
The `--ci` flag changes Jest's behavior for CI environments: it disables interactive prompts, fails immediately when a snapshot is outdated (instead of updating), and runs tests more strictly. Always use `--ci` in automated pipelines.
:::

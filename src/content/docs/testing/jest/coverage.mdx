Code coverage measures what percentage of your source code is exercised by your test suite. Jest integrates Istanbul (via Babel) or V8 to collect this data with zero additional tooling.

## Enabling Coverage

Run Jest with the `--coverage` flag:

```bash
npx jest --coverage
# or
npm test -- --coverage
```

After running, Jest prints a coverage table and writes report files to the `coverage/` directory.

## Reading the Coverage Table

```
---------------------|---------|----------|---------|---------|-------------------
File                 | % Stmts | % Branch | % Funcs | % Lines | Uncovered Line #s
---------------------|---------|----------|---------|---------|-------------------
All files            |   87.65 |    76.19 |   91.30 |   87.65 |
 src/utils           |         |          |         |         |
  math.ts            |   100.0 |    100.0 |   100.0 |   100.0 |
  string.ts          |    92.3 |     75.0 |   100.0 |    92.3 | 34, 67-68
  validators.ts      |    66.6 |     50.0 |    75.0 |    66.6 | 12-18, 45
---------------------|---------|----------|---------|---------|-------------------
```

### Four Coverage Metrics

| Metric | Meaning |
|--------|---------|
| **Statements** | Every executable statement executed at least once |
| **Branches** | Every code branch (if/else, ternary, switch case, `&&`, `\|\|`, `??`) taken at least once |
| **Functions** | Every function/method called at least once |
| **Lines** | Every source line executed at least once (similar to statements) |

**Branch coverage** is the most important metric. You can achieve 100% statement coverage while missing entire `else` branches.

## Coverage Reporters

Configure reporters in `jest.config.ts`:

```typescript
const config: Config = {
  coverageReporters: [
    'text',         // terminal table (default)
    'text-summary', // brief summary line only
    'lcov',         // lcov format for CI tools (SonarQube, Codecov, Coveralls)
    'html',         // interactive HTML report in coverage/lcov-report/
    'json',         // machine-readable JSON in coverage/coverage-final.json
    'json-summary', // summary JSON (useful for threshold CI checks)
    'clover',       // Clover XML (for IntelliJ / Atlassian)
  ],
};
```

:::tip[Recommended Reporters]
For local development use `['text', 'html']`. For CI pipelines, add `'lcov'` to integrate with coverage tools like Codecov, Coveralls, or SonarQube.
:::

## Coverage Thresholds

Enforce minimum coverage percentages — Jest fails with a non-zero exit code if any threshold is not met:

```typescript
const config: Config = {
  coverageThreshold: {
    // Global thresholds — applied to all collected files
    global: {
      branches: 80,
      functions: 90,
      lines: 85,
      statements: 85,
    },

    // Per-file thresholds — useful for critical modules
    './src/utils/auth.ts': {
      branches: 100,
      functions: 100,
      lines: 100,
      statements: 100,
    },

    // Directory glob — all files in src/api/ must have at least 70% branch coverage
    './src/api/**/*.ts': {
      branches: 70,
    },
  },
};
```

Threshold failure output:

```
Jest: "global" coverage threshold for branches (80%) not met: 71%
Jest: "/src/utils/auth.ts" coverage threshold for lines (100%) not met: 92%
```

:::caution[Ratcheting Coverage]
As you add new code, gradually increase thresholds to prevent regressions. A common approach is to set thresholds slightly below your current coverage and raise them after adding new tests. Never lower a threshold without a documented reason.
:::

## Coverage Providers

Jest supports two coverage instrumentation providers:

<Tabs>
  <TabItem label="babel (default)">

    Uses Istanbul's Babel plugin for instrumentation. Compatible with all Jest transform configurations.

    ```typescript
    const config: Config = {
      coverageProvider: 'babel', // default
    };
    ```

    **Pros**: Battle-tested, works with any transform setup  
    **Cons**: Adds instrumentation code via AST transformation; slightly slower

  </TabItem>
  <TabItem label="v8">

    Uses Node.js V8's native coverage counters. No AST transformation required.

    ```typescript
    const config: Config = {
      coverageProvider: 'v8',
    };
    ```

    **Pros**: Faster collection; more accurate for minified/transpiled code  
    **Cons**: Requires Node.js 12+; may show different numbers for TypeScript source maps

  </TabItem>
</Tabs>

## Collecting Coverage From Specific Files

By default, Jest only instruments files that are imported during tests. Files with zero test coverage are invisible unless you explicitly include them:

```typescript
const config: Config = {
  collectCoverageFrom: [
    'src/**/*.{ts,tsx}',          // include all TypeScript source files
    '!src/**/*.d.ts',             // exclude type declaration files
    '!src/**/*.stories.tsx',      // exclude Storybook stories
    '!src/**/index.ts',           // exclude barrel export files
    '!src/types/**',              // exclude type-only files
    '!src/main.tsx',              // exclude entry point
  ],
};
```

This ensures files that are **never imported** in any test still appear in coverage reports with 0% coverage — making it clear they're completely untested.

## Excluding Files from Coverage

### Via config

```typescript
const config: Config = {
  coveragePathIgnorePatterns: [
    '/node_modules/',
    '/dist/',
    '/coverage/',
    '\\.stories\\.tsx$',
    '\\.mock\\.ts$',
  ],
};
```

### Via inline comments

Use Istanbul's inline comments to exclude specific lines or blocks:

```typescript
/* istanbul ignore file */   // Exclude the entire file

function criticalFunction() {
  /* istanbul ignore next */  // Exclude the next statement/branch
  if (process.env.NODE_ENV === 'development') {
    debugLog('dev mode');
  }

  const value = condition
    ? computeA()
    : /* istanbul ignore next */ computeB(); // exclude the else branch
}
```

:::caution[Don't Abuse ignore Comments]
Istanbul ignore comments are a tool for genuinely untestable code (OS-specific error paths, unreachable fallbacks). Using them to inflate coverage numbers defeats the purpose. Code reviewers should scrutinise every `/* istanbul ignore */`.
:::

## Reading the HTML Report

The HTML report at `coverage/lcov-report/index.html` is invaluable for understanding exactly which lines and branches are covered:

<Steps>
1. Run `npx jest --coverage` to generate the report.

2. Open `coverage/lcov-report/index.html` in your browser.

3. Navigate into a specific file to see line-by-line coverage:
   - **Green**: covered (executed at least once)
   - **Red**: not covered (never executed)
   - **Yellow**: partially covered branch

4. Click on any yellow branch indicator to see which path was never taken.

5. Focus on **branch coverage** gaps — these represent untested code paths that could contain bugs.
</Steps>

## Testing Untested Branches

Branch coverage < 100% means some conditions were never evaluated. Common patterns:

```typescript
// Low branch coverage example
export function getStatusLabel(status: 'active' | 'inactive' | 'pending'): string {
  if (status === 'active') return 'Active';    // covered
  if (status === 'inactive') return 'Inactive'; // covered
  if (status === 'pending') return 'Pending';   // ← not covered!
  return 'Unknown'; // ← not covered either!
}
```

Add tests for missing branches:

```typescript
it.each([
  ['active', 'Active'],
  ['inactive', 'Inactive'],
  ['pending', 'Pending'],       // adds branch coverage
] as const)('returns "%s" for status "%s"', (status, expected) => {
  expect(getStatusLabel(status)).toBe(expected);
});

it('returns "Unknown" for an unexpected status', () => {
  // TypeScript guards prevent this at compile time, but tests can force it
  expect(getStatusLabel('banned' as any)).toBe('Unknown');
});
```

## 100% Coverage Pitfalls

:::caution[100% Coverage Is Not the Goal]
100% code coverage does not mean 100% correctness. You can achieve complete coverage with tests that never actually assert anything meaningful. Coverage is a tool for finding gaps, not a quality guarantee.

```typescript
// 100% coverage but completely meaningless
it('covers the function', () => {
  divide(10, 2); // runs the code but asserts nothing!
  expect(true).toBe(true);
});
```
:::

### Meaningful Coverage vs Vanity Metrics

| Good coverage practice | Bad coverage practice |
|------------------------|----------------------|
| Tests that verify correct output | Tests that just call functions |
| Tests covering edge cases (zero, null, empty) | Tests duplicating what TypeScript already enforces |
| Tests for each error path | Tests for trivial getters/setters |
| Tests documenting business rules | Tests written solely to hit a number |

A practical target for most projects is **80% line coverage + 70% branch coverage**, with 100% coverage on critical paths (authentication, payment processing, data validation).

## CI Integration

Fail the CI build when coverage drops below thresholds:

```yaml
# .github/workflows/test.yml
- name: Run tests with coverage
  run: npm run test:ci

- name: Upload coverage report
  uses: codecov/codecov-action@v4
  with:
    file: ./coverage/lcov.info
    fail_ci_if_error: true
    token: ${{ secrets.CODECOV_TOKEN }}
```

```typescript
// package.json
{
  "scripts": {
    "test:ci": "jest --ci --coverage --forceExit --maxWorkers=2"
  }
}
```

The `--ci` flag:
- Disables the interactive watch mode
- Fails immediately if snapshot files are out of date (instead of updating them)
- Does not write a new snapshot if a new one is encountered

```typescript
// jest.config.ts — strict CI thresholds
const config: Config = {
  coverageThreshold: {
    global: {
      branches: 75,
      functions: 85,
      lines: 85,
      statements: 85,
    },
  },
};
```

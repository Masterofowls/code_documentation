Code coverage measures how much of your source code is executed when your test suite runs. Jest integrates Istanbul (and optionally V8) to collect coverage without any additional tooling.

## The `--coverage` Flag

Run tests with coverage by passing `--coverage`:

```bash
npx jest --coverage
```

Jest produces a summary in the terminal and writes detailed reports to the `coverage/` directory:

```
--------------------|---------|----------|---------|---------|---------------------------
File                | % Stmts | % Branch | % Funcs | % Lines | Uncovered Line #s
--------------------|---------|----------|---------|---------|---------------------------
All files           |   87.50 |    75.00 |   90.00 |   87.50 |
 src/               |         |          |         |         |
  math.ts           |  100.00 |   100.00 |  100.00 |  100.00 |
  string.ts         |   83.33 |    66.67 |   80.00 |   83.33 | 24-27, 45
  api.ts            |   76.19 |    57.14 |   88.89 |   76.19 | 89, 103-108, 122
--------------------|---------|----------|---------|---------|---------------------------
```

## What Counts as Covered?

Jest measures four coverage dimensions:

| Dimension      | What it measures                                 | Example of a gap                         |
| -------------- | ------------------------------------------------ | ---------------------------------------- |
| **Statements** | Individual executable statements                 | An `if` block that is never entered      |
| **Branches**   | Decision points (if/else, ternary, `&&`, `\|\|`) | An `else` branch never executed          |
| **Functions**  | Function declarations called at least once       | A utility function never called in tests |
| **Lines**      | Physical lines containing at least one statement | Any line with untested code              |

Branch coverage is the most valuable metric. A function can have 100% statement coverage but only 50% branch coverage if one side of every `if` condition is always taken.

```typescript
// This function has 100% statement coverage if tested with positive numbers
// but only 50% branch coverage — the `throw` branch is never hit
function sqrt(n: number): number {
  if (n < 0) throw new RangeError('Cannot sqrt a negative number'); // untested branch
  return Math.sqrt(n);
}
```

## Coverage Reporters

Configure which reporters to generate in `jest.config.ts`:

```typescript
const config: Config = {
  coverageReporters: [
    'text', // terminal table output (default)
    'text-summary', // brief terminal summary
    'lcov', // for SonarQube, Codecov, Coveralls
    'html', // browsable HTML report at coverage/index.html
    'json', // raw JSON data at coverage/coverage-final.json
    'json-summary', // compact summary JSON
    'clover', // XML format (used by some CI tools)
  ],
};
```

:::tip[Recommended reporter set]
For most projects, use `['text', 'lcov', 'html']`:

- `text` gives immediate feedback in the terminal
- `lcov` integrates with Codecov, Coveralls, and SonarQube
- `html` provides the interactive report for detailed investigation
  :::

## Coverage Providers

Jest supports two coverage instrumentation engines:

<Tabs>
  <TabItem label="babel (default)">
    Istanbul's Babel-based instrumentation. It transforms your source code to inject counter statements before running tests.

    ```typescript
    const config: Config = {
      coverageProvider: 'babel', // default
    };
    ```

    **Pros**: Mature, widely supported, works with Babel transforms
    **Cons**: Slightly slower, can produce inaccurate branch coverage for complex TypeScript

</TabItem>
  <TabItem label="v8">
    Uses Node.js's built-in V8 coverage engine via `--experimental-vm-modules`. No instrumentation step needed.

    ```typescript
    const config: Config = {
      coverageProvider: 'v8',
    };
    ```

    **Pros**: More accurate branch detection, faster test execution, no code transformation
    **Cons**: Requires Node.js 14+, counts switch cases differently, some edge cases in reporting

</TabItem>
</Tabs>

## Setting Coverage Thresholds

Fail the test run if coverage drops below a minimum:

```typescript
const config: Config = {
  coverageThreshold: {
    // Global thresholds — applied across all files combined
    global: {
      branches: 80,
      functions: 80,
      lines: 80,
      statements: 80,
    },

    // Per-file thresholds — applied to individual files
    './src/utils/': {
      branches: 90,
      functions: 100,
      lines: 90,
      statements: 90,
    },

    // Specific file threshold
    './src/core/auth.ts': {
      branches: 95,
      functions: 100,
      lines: 95,
      statements: 95,
    },
  },
};
```

When coverage falls below a threshold, Jest exits with a non-zero code and prints:

```
Jest: "global" coverage threshold for branches (80%) not met: 75%
```

This makes the CI pipeline fail automatically.

:::caution[Threshold pitfalls]
Setting thresholds at exactly the current level (e.g., 73%) provides no safety net. Set them at a meaningful floor (e.g., 80%) and treat any violation as a reason to add tests, not lower the threshold.
:::

## Specifying Which Files to Collect Coverage From

By default, Jest only collects coverage from files that are imported in at least one test. To report coverage for all source files (including untested ones), use `collectCoverageFrom`:

```typescript
const config: Config = {
  collectCoverage: true, // always collect, not just with --coverage flag
  collectCoverageFrom: [
    'src/**/*.{ts,tsx}', // all TypeScript source files
    '!src/**/*.d.ts', // exclude type declarations
    '!src/**/*.stories.{ts,tsx}', // exclude Storybook stories
    '!src/**/index.ts', // exclude barrel files
    '!src/mocks/**', // exclude mock files
    '!src/setupTests.ts', // exclude test setup
    '!src/**/__tests__/**', // exclude test files themselves
  ],
};
```

### Excluding files with `coveragePathIgnorePatterns`

```typescript
const config: Config = {
  coveragePathIgnorePatterns: [
    '/node_modules/',
    '/dist/',
    '/coverage/',
    '\\.stories\\.tsx?$',
    '/src/generated/',
  ],
};
```

### Excluding specific lines or blocks

Use Istanbul comments to exclude specific code from coverage:

```typescript
// Exclude a single line
const value =
  process.env.NODE_ENV === 'production' /* istanbul ignore next */
    ? 'prod'
    : 'dev';

// Exclude a block
/* istanbul ignore next */
if (process.env.DEBUG) {
  console.log('Debug mode enabled');
}

// Exclude the else branch only
if (condition) {
  doSomething();
} else /* istanbul ignore else */ {
  // This branch is impossible in practice
  throw new Error('Unreachable');
}
```

:::tip[Use sparingly]
Reserve Istanbul ignore comments for code that is genuinely untestable (platform-specific code paths, debug-only branches, generated code). Don't use them to inflate coverage numbers.
:::

## Reading the HTML Report

After running `jest --coverage`, open `coverage/index.html` in a browser:

<Steps>
  1. **Summary page** — shows all files with coloured bars for each coverage
  dimension. Red = below threshold, yellow = partial coverage, green = well
  covered. 2. **File view** — click a file to see line-by-line coverage. Red
  lines are untested. Yellow lines have partial branch coverage (shown with
  `I`/`E` for if/else). 3. **Branch indicators** — hover over a branch indicator
  to see which side is covered and which is not. 4. **Function list** — scroll
  to the function list at the bottom to see which functions have never been
  called.
</Steps>

## Testing Untested Branches

After identifying uncovered branches in the HTML report, write targeted tests:

```typescript
// src/utils/formatDate.ts
export function formatDate(
  date: Date | string | null,
  format: 'short' | 'long' = 'short',
): string {
  if (!date) return '—'; // branch: null/undefined
  const d = typeof date === 'string' ? new Date(date) : date; // branch: string vs Date
  if (isNaN(d.getTime())) return 'Invalid date'; // branch: invalid date string

  if (format === 'long') {
    return d.toLocaleDateString('en-GB', {
      day: 'numeric',
      month: 'long',
      year: 'numeric',
    });
  }
  return d.toLocaleDateString('en-GB');
}
```

```typescript
// Branch-targeted test cases
describe('formatDate', () => {
  it('returns em-dash for null', () => expect(formatDate(null)).toBe('—')); // covers null branch
  it('returns em-dash for undefined', () =>
    expect(formatDate(undefined as any)).toBe('—'));
  it('accepts a string input', () =>
    expect(formatDate('2024-01-15')).toBeTruthy()); // covers string branch
  it('accepts a Date input', () =>
    expect(formatDate(new Date('2024-01-15'))).toBeTruthy());
  it('returns Invalid date for bad string', () =>
    expect(formatDate('not-a-date')).toBe('Invalid date'));
  it('uses long format when specified', () =>
    expect(formatDate(new Date('2024-01-15'), 'long')).toContain('January'));
});
```

## 100% Coverage Pitfalls

Achieving 100% code coverage does not mean your code is bug-free. A test can execute every line without making any useful assertions.

```typescript
// This test gives 100% coverage but tests nothing meaningful
test('bad test', () => {
  formatDate(new Date()); // line is hit
  formatDate('2024-01-15'); // line is hit
  formatDate(null); // line is hit
  expect(true).toBe(true); // useless assertion
});
```

**Meaningful coverage means:**

- Each test has clear `expect()` assertions tied to the specific branch being tested
- Edge cases are tested, not just the happy path
- Negative test cases (invalid inputs, error conditions) are covered

:::caution[100% coverage can be counterproductive]
Chasing 100% coverage on every file leads to brittle tests and wasted effort. Focus on critical business logic, utility functions with complex branching, and error handling paths. Allow lower coverage on configuration files, simple data models, and generated code.
:::

## CI Integration

A complete CI workflow with coverage:

```yaml
# .github/workflows/test.yml
- name: Run tests with coverage
  run: npx jest --ci --coverage --maxWorkers=2

- name: Upload coverage to Codecov
  uses: codecov/codecov-action@v4
  with:
    files: ./coverage/lcov.info
    fail_ci_if_error: true
    token: ${{ secrets.CODECOV_TOKEN }}

- name: Upload HTML coverage report
  uses: actions/upload-artifact@v4
  if: always()
  with:
    name: coverage-report
    path: coverage/
    retention-days: 30
```

### `--ci` flag differences

The `--ci` flag makes Jest behave differently for CI environments:

| Default mode                        | CI mode (`--ci`)                  |
| ----------------------------------- | --------------------------------- |
| Re-runs changed tests first         | Runs all tests (no priority)      |
| Interactive watch mode available    | Non-interactive                   |
| Creates new snapshots automatically | Fails if new snapshots are needed |
| Slower with `--maxWorkers=auto`     | Same                              |

Always use `--ci` in your CI pipeline to catch snapshot drift and ensure deterministic runs.

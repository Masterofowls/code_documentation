Code coverage measures how much of your source code is executed during tests. Jest integrates Istanbul (default) and V8 coverage providers directly — no additional tools required.

## Running Coverage

```bash
# Run tests and collect coverage
jest --coverage

# Or via npm script
npm run test:coverage

# Coverage for a specific file
jest --coverage --collectCoverageFrom="src/utils/math.ts"
```

Sample output:


## Coverage Reporters

Configure which report formats Jest generates:

```typescript
// jest.config.ts
export default {
  coverageReporters: [
    'text',          // terminal summary table (always include)
    'text-summary',  // brief summary line only
    'lcov',          // lcov.info file — consumed by Codecov, Coveralls, SonarQube
    'html',          // browsable HTML report in coverage/
    'json',          // machine-readable JSON
    'json-summary',  // summary JSON for CI badge services
    'clover',        // Clover XML — consumed by some CI systems
    'cobertura',     // Cobertura XML — used by Jenkins, Azure DevOps
  ],
};
```

| Reporter | Output | Use case |
|---|---|---|
| `text` | Terminal table | Always-on feedback during development |
| `text-summary` | One-line summary | Compact CI log output |
| `lcov` | `coverage/lcov.info` | Codecov, Coveralls, SonarQube integration |
| `html` | `coverage/index.html` | Human review, drilling into uncovered lines |
| `json` | `coverage/coverage-final.json` | Custom tooling and dashboards |
| `json-summary` | `coverage/coverage-summary.json` | Coverage badges, CI threshold checks |
| `cobertura` | `coverage/cobertura-coverage.xml` | Jenkins, Azure Pipelines |

:::tip[Minimal CI Reporter Setup]
For CI pipelines, use `['text-summary', 'lcov', 'json-summary']`. `text-summary` gives a quick pass/fail log line, `lcov` feeds your coverage service, and `json-summary` enables coverage badge automation.
:::

## coverageThreshold

Fail the Jest run if coverage drops below specified thresholds:

```typescript
// jest.config.ts
export default {
  coverageThreshold: {
    // Global thresholds — applied across all files
    global: {
      statements: 80,
      branches: 75,
      functions: 90,
      lines: 80,
    },
    // Per-file or per-directory thresholds
    './src/core/': {
      statements: 95,
      branches: 90,
      functions: 100,
      lines: 95,
    },
    './src/utils/math.ts': {
      statements: 100,
      branches: 100,
      functions: 100,
      lines: 100,
    },
  },
};
```

When a threshold is breached, Jest exits with a non-zero code and prints:

```
Jest: "global" coverage threshold for statements (80%) not met: 72%
```

:::caution[Threshold Granularity]
Per-file thresholds use glob patterns. The `./src/core/` pattern applies to the entire directory. Setting thresholds too high for third-party-heavy integration files can make CI brittle — apply stricter thresholds to pure logic modules, looser ones to integration layers.
:::

## Coverage Providers

Jest supports two coverage providers:

<Tabs>
  <TabItem label="babel (default)">

Uses Istanbul's instrumentation approach — Babel transforms inject coverage counters into your source code at compile time.

```typescript
// jest.config.ts
export default {
  coverageProvider: 'babel', // default
};
```

**Pros:** Mature, widely used, works with any transform pipeline.  
**Cons:** Slower — requires full Babel transform pass even without coverage.

  </TabItem>
  <TabItem label="v8">

Uses Node.js's built-in V8 JavaScript engine coverage APIs. No source transformation required.

```typescript
// jest.config.ts
export default {
  coverageProvider: 'v8',
};
```

**Pros:** Faster collection (no transform overhead), more accurate for native async code.  
**Cons:** Less accurate for highly transpiled code (TypeScript decorators, class fields).

  </TabItem>
</Tabs>

## What Counts as Covered

| Coverage type | What it measures |
|---|---|
| **Statements** | Each statement executed at least once |
| **Branches** | Each branch of conditional logic (true/false for `if`, each `case` in `switch`, ternary arms, `&&` / `\|\|` short-circuits) |
| **Functions** | Each function/method called at least once |
| **Lines** | Each line of code executed at least once |

Branch coverage is the hardest to achieve and the most valuable. A function with `if (a && b)` has four branches: `a=true,b=true`, `a=true,b=false`, `a=false` (b not evaluated), and the implicit `else`.

```typescript
// This function has 4 branches
function getDiscount(user: User): number {
  if (user.isPremium && user.yearsActive > 2) {
    return 0.20;  // branch: isPremium=true AND yearsActive>2
  }
  return user.isPremium ? 0.10 : 0;  // branches: isPremium=true, isPremium=false
}

// Tests needed for 100% branch coverage:
// 1. isPremium=true, yearsActive=3  → 0.20
// 2. isPremium=true, yearsActive=1  → 0.10
// 3. isPremium=false                → 0
// Note: yearsActive check short-circuits when isPremium=false
```

## Reading the HTML Report

The HTML report (`coverage/index.html`) is the best tool for finding uncovered code:

<Steps>
1. Run `jest --coverage` to generate the report
2. Open `coverage/index.html` in your browser
3. Click on a file to see line-by-line coverage highlighting:
   - **Green** — executed
   - **Red** — not executed
   - **Yellow** — partially covered branch
4. Look for red `E` (else) markers on branch statements — these indicate uncovered else paths
5. Click the `Branches` column to sort by lowest branch coverage
</Steps>

## Excluding Files from Coverage

```typescript
// jest.config.ts
export default {
  // Explicitly define which files to collect coverage from
  collectCoverageFrom: [
    'src/**/*.{ts,tsx}',           // include all TS/TSX source files
    '!src/**/*.d.ts',              // exclude type declaration files
    '!src/**/*.stories.{ts,tsx}',  // exclude Storybook stories
    '!src/**/index.ts',            // exclude barrel files
    '!src/test-utils/**',          // exclude test utilities
    '!src/generated/**',           // exclude auto-generated code
  ],

  // Patterns to exclude from coverage collection
  coveragePathIgnorePatterns: [
    '/node_modules/',
    '/dist/',
    '/coverage/',
    '<rootDir>/src/migrations/',
  ],
};
```

:::tip[collectCoverageFrom is Critical]
Without `collectCoverageFrom`, Jest only includes files that are actually imported by tests. This means untested files are invisible to the coverage report — your 90% coverage might actually be much lower. Always set `collectCoverageFrom` explicitly.
:::

## Testing Untested Branches

When the HTML report reveals uncovered branches, write targeted tests:

```typescript
// Before: 60% branch coverage on parseConfig
// Uncovered: default values, error paths, empty config

describe('parseConfig — branch coverage', () => {
  it('uses default timeout when not specified', () => {
    const config = parseConfig({ apiUrl: 'https://api.example.com' });
    expect(config.timeout).toBe(5000); // default value branch
  });

  it('throws when apiUrl is missing', () => {
    expect(() => parseConfig({})).toThrow('apiUrl is required'); // error branch
  });

  it('handles an empty features array', () => {
    const config = parseConfig({ apiUrl: 'https://api.example.com', features: [] });
    expect(config.features).toEqual([]); // empty array branch
  });

  it('handles null input', () => {
    expect(() => parseConfig(null as never)).toThrow(); // null guard branch
  });
});
```

## CI Integration

### GitHub Actions

```yaml
# .github/workflows/test.yml
- name: Run tests with coverage
  run: npm run test:ci

- name: Upload coverage to Codecov
  uses: codecov/codecov-action@v4
  with:
    files: ./coverage/lcov.info
    fail_ci_if_error: true

- name: Upload HTML coverage report as artifact
  if: always()
  uses: actions/upload-artifact@v4
  with:
    name: coverage-report
    path: coverage/
    retention-days: 14
```

### Failing CI Below Threshold

The `coverageThreshold` configuration in `jest.config.ts` automatically causes Jest to exit with a non-zero code when thresholds are not met, which fails the CI pipeline:

```typescript
// jest.config.ts
export default {
  coverageThreshold: {
    global: {
      statements: 80,
      branches: 75,
      functions: 90,
      lines: 80,
    },
  },
};
```

```bash
# CI command — exits with non-zero code on threshold failure
jest --ci --coverage --forceExit
```

## Coverage: Meaningful vs Vanity Metrics

:::caution[100% Coverage Does Not Mean Bug-Free]
Coverage measures whether code was executed — not whether it was tested correctly. A test that calls every function but never asserts anything will show 100% coverage while testing nothing. Focus on writing meaningful assertions, not on hitting a number.
:::

```typescript
// 100% coverage — but tests nothing meaningful
it('executes processOrder without crashing', () => {
  expect(() => processOrder({ items: [], userId: 1 })).not.toThrow();
  // No assertions about the result — useless test
});

// Meaningful test with real assertions
it('creates an order with correct total and pending status', () => {
  const order = processOrder({
    userId: 1,
    items: [{ productId: 'p1', quantity: 2, price: 9.99 }],
  });

  expect(order.total).toBe(19.98);
  expect(order.status).toBe('pending');
  expect(order.userId).toBe(1);
  expect(order.lineItems).toHaveLength(1);
});
```

Focus coverage efforts on:
- **Core business logic** — aim for 90–100% with meaningful assertions
- **Error handling paths** — often missed by happy-path tests
- **Edge cases** — empty inputs, boundary values, null checks
- **Integration glue code** — lower priority, harder to unit test meaningfully

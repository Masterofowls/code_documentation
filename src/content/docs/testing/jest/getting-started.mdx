This guide walks you through installing Jest in a TypeScript project, setting up configuration, and writing your first tests. By the end you will have a fully working test setup with TypeScript support, proper file discovery, and npm scripts.

## Installation

Jest requires a transform layer to handle TypeScript. There are two popular approaches: **ts-jest** (TypeScript-first, precise type checking) and **babel-jest** (faster, no type checking at test time).

<Tabs>
  <TabItem label="ts-jest (recommended)">

```bash
npm install --save-dev jest @types/jest ts-jest typescript
```

`ts-jest` compiles TypeScript using the TypeScript compiler directly. Type errors in your tests will cause test failures, which helps catch mistakes early.

  </TabItem>
  <TabItem label="babel-jest">

```bash
npm install --save-dev jest @types/jest babel-jest @babel/core @babel/preset-env @babel/preset-typescript
```

Then create a `babel.config.js`:

```javascript
// babel.config.js
module.exports = {
  presets: [
    ['@babel/preset-env', { targets: { node: 'current' } }],
    '@babel/preset-typescript',
  ],
};
```

  </TabItem>
</Tabs>

:::tip[Which Transform Should I Use?]
Use **ts-jest** for projects where TypeScript type safety matters in tests (recommended for most projects). Use **babel-jest** when test speed is critical or when you already use Babel for your build pipeline and want consistency.
:::

## Creating jest.config.ts

Create a `jest.config.ts` file in your project root:

<Tabs>
  <TabItem label="ts-jest">

```typescript
// jest.config.ts
import type { Config } from 'jest';

const config: Config = {
  preset: 'ts-jest',
  testEnvironment: 'node',
  roots: ['<rootDir>/src'],
  testMatch: [
    '**/__tests__/**/*.+(ts|tsx)',
    '**/?(*.)+(spec|test).+(ts|tsx)',
  ],
  transform: {
    '^.+\\.(ts|tsx)$': 'ts-jest',
  },
  collectCoverageFrom: [
    'src/**/*.{ts,tsx}',
    '!src/**/*.d.ts',
    '!src/**/index.ts',
  ],
  coverageDirectory: 'coverage',
  verbose: true,
};

export default config;
```

  </TabItem>
  <TabItem label="babel-jest">

```typescript
// jest.config.ts
import type { Config } from 'jest';

const config: Config = {
  testEnvironment: 'node',
  roots: ['<rootDir>/src'],
  testMatch: [
    '**/__tests__/**/*.+(ts|tsx|js)',
    '**/?(*.)+(spec|test).+(ts|tsx|js)',
  ],
  transform: {
    '^.+\\.(ts|tsx|js|jsx)$': 'babel-jest',
  },
  collectCoverageFrom: [
    'src/**/*.{ts,tsx}',
    '!src/**/*.d.ts',
  ],
  verbose: true,
};

export default config;
```

  </TabItem>
</Tabs>

## testMatch Patterns

Jest uses glob patterns to discover test files. The default patterns cover the two most common conventions:

| Pattern | Matches |
|---|---|
| `**/__tests__/**/*.{ts,tsx}` | Any `.ts` or `.tsx` file inside a `__tests__` folder |
| `**/*.test.{ts,tsx}` | Files ending in `.test.ts` or `.test.tsx` |
| `**/*.spec.{ts,tsx}` | Files ending in `.spec.ts` or `.spec.tsx` |

```typescript
// jest.config.ts — custom testMatch example
testMatch: [
  '<rootDir>/src/**/__tests__/**/*.ts',  // files in __tests__/
  '<rootDir>/src/**/*.test.ts',          // *.test.ts files
  '<rootDir>/src/**/*.spec.ts',          // *.spec.ts files
  '!<rootDir>/src/**/*.d.ts',            // exclude declaration files
],
```

:::caution[testMatch vs testRegex]
Do not use both `testMatch` and `testRegex` at the same time — they are mutually exclusive. `testMatch` uses glob patterns and is easier to read; `testRegex` uses regular expressions and offers more power for complex setups.
:::

## npm Scripts

Add these scripts to your `package.json`:

```json
{
  "scripts": {
    "test": "jest",
    "test:watch": "jest --watch",
    "test:watchAll": "jest --watchAll",
    "test:coverage": "jest --coverage",
    "test:ci": "jest --ci --coverage --forceExit",
    "test:verbose": "jest --verbose"
  }
}
```

| Script | Purpose |
|---|---|
| `npm test` | Run all tests once |
| `npm run test:watch` | Watch changed files only (requires git) |
| `npm run test:watchAll` | Watch all files regardless of git status |
| `npm run test:coverage` | Run tests and generate coverage report |
| `npm run test:ci` | CI-optimised run — no interactive mode, fails on threshold violations |

## Running Specific Tests

### Run a single file

```bash
npx jest src/utils/math.test.ts
npx jest math              # partial path match
```

### Run tests matching a name pattern

Use `--testNamePattern` (or `-t`) to run only tests whose description matches a string or regex:

```bash
npx jest --testNamePattern="formatCurrency"
npx jest -t "formats a positive"
npx jest -t "^add"         # regex: tests starting with "add"
```

### Run tests in a specific directory

```bash
npx jest src/api/
npx jest --testPathPattern="src/api"
```

### Combine file and name filters

```bash
npx jest math --testNamePattern="negative"
```

## Watch Mode

Watch mode re-runs tests automatically as files change. It is the most productive way to do TDD.

```bash
npm run test:watch
```

In watch mode, press the following keys to filter interactively:

| Key | Action |
|---|---|
| `f` | Run only failing tests |
| `p` | Filter by filename pattern |
| `t` | Filter by test name pattern |
| `a` | Run all tests |
| `q` | Quit watch mode |
| `Enter` | Re-run tests |

:::tip[Watch Mode Requires Git]
`--watch` only re-runs tests related to changed files and requires a git repository. Use `--watchAll` if you are not in a git repository or want to run all tests on every change.
:::

## Writing Your First Test File

### File Naming Conventions

Jest supports two common conventions — pick one and stick to it:

| Convention | Example |
|---|---|
| Co-located `*.test.ts` | `src/utils/math.ts` + `src/utils/math.test.ts` |
| Centralised `__tests__/` | `src/utils/math.ts` + `src/__tests__/utils/math.test.ts` |

Co-located tests are more common in modern projects because they keep source and test files visible together.

### Your First Test File

```typescript
// src/utils/math.ts
export const add = (a: number, b: number): number => a + b;

export const divide = (a: number, b: number): number => {
  if (b === 0) throw new Error('Division by zero');
  return a / b;
};

export const clamp = (value: number, min: number, max: number): number =>
  Math.min(Math.max(value, min), max);
```

```typescript
// src/utils/math.test.ts
import { add, divide, clamp } from './math';

describe('add', () => {
  it('returns the sum of two positive numbers', () => {
    expect(add(2, 3)).toBe(5);
  });

  it('handles negative numbers', () => {
    expect(add(-1, 1)).toBe(0);
  });

  it('handles floating point numbers', () => {
    expect(add(0.1, 0.2)).toBeCloseTo(0.3);
  });
});

describe('divide', () => {
  it('divides two numbers', () => {
    expect(divide(10, 2)).toBe(5);
  });

  it('throws on division by zero', () => {
    expect(() => divide(10, 0)).toThrow('Division by zero');
  });
});

describe('clamp', () => {
  it('returns the value when within range', () => {
    expect(clamp(5, 0, 10)).toBe(5);
  });

  it('returns min when value is below range', () => {
    expect(clamp(-5, 0, 10)).toBe(0);
  });

  it('returns max when value is above range', () => {
    expect(clamp(15, 0, 10)).toBe(10);
  });
});
```

## Jest Globals and Lifecycle Hooks

Jest provides global functions for organizing tests and managing setup/teardown. These do not need to be imported — they are available in every test file automatically.

### describe and test / it

`describe` creates a logical grouping (a test suite). `it` and `test` are identical — use whichever reads more naturally.

```typescript
describe('UserService', () => {
  describe('createUser', () => {
    it('creates a user with valid data', () => { /* ... */ });
    it('rejects a user with a duplicate email', () => { /* ... */ });
  });

  describe('deleteUser', () => {
    test('removes the user from the database', () => { /* ... */ });
    test('throws when user does not exist', () => { /* ... */ });
  });
});
```

### Lifecycle Hooks

```typescript
describe('DatabaseService', () => {
  let db: Database;

  // Runs once before all tests in this describe block
  beforeAll(async () => {
    db = await Database.connect('sqlite::memory:');
    await db.migrate();
  });

  // Runs once after all tests in this describe block
  afterAll(async () => {
    await db.disconnect();
  });

  // Runs before each individual test
  beforeEach(async () => {
    await db.seed({ users: testUsers });
  });

  // Runs after each individual test
  afterEach(async () => {
    await db.truncate();
  });

  it('finds a user by id', async () => {
    const user = await db.users.findById(1);
    expect(user).toBeDefined();
    expect(user.email).toBe('alice@example.com');
  });

  it('returns null for unknown id', async () => {
    const user = await db.users.findById(99999);
    expect(user).toBeNull();
  });
});
```

### Hook Execution Order

When hooks are nested inside multiple `describe` blocks, they execute in this order:

```
beforeAll (outer)
  beforeAll (inner)
    beforeEach (outer)
      beforeEach (inner)
        test
      afterEach (inner)
    afterEach (outer)
  afterAll (inner)
afterAll (outer)
```

:::tip[Prefer beforeEach Over beforeAll for State]
Use `beforeAll` only for expensive setup that is truly shared and read-only (like establishing a database connection). Use `beforeEach` for state that tests might mutate, to ensure each test starts with a clean slate and tests remain independent.
:::

### Skipping and Focusing Tests

```typescript
// Skip a single test
it.skip('this test is skipped', () => { /* ... */ });
xit('also skipped', () => { /* ... */ });

// Skip an entire describe block
describe.skip('skipped suite', () => { /* ... */ });

// Focus — only run this test (never commit!)
it.only('only this test runs', () => { /* ... */ });
fit('also focused', () => { /* ... */ });

// Mark as todo — shows in output as a reminder
it.todo('handle rate limiting');
```

:::caution[Never Commit .only]
`it.only` and `describe.only` cause all other tests to be skipped. If accidentally committed, your CI will pass even though 99% of tests are not running. Use the `jest-no-only` ESLint rule to catch this automatically.
:::

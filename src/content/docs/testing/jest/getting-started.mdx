This guide walks you through setting up Jest in a TypeScript project from scratch — from installation through your first passing test suite.

## Installation

<Tabs>
  <TabItem label="ts-jest (recommended)">

`ts-jest` compiles TypeScript files directly via the TypeScript compiler. Type errors in your tests will cause test failures, giving you type-safe tests.

```bash
npm install --save-dev jest ts-jest @types/jest typescript
```

  </TabItem>
  <TabItem label="babel-jest">

`babel-jest` uses Babel to strip TypeScript types without type-checking. Faster startup but no type-checking during tests.

```bash
npm install --save-dev jest babel-jest @types/jest @babel/core @babel/preset-env @babel/preset-typescript
```

Create a `babel.config.js`:

```javascript
module.exports = {
  presets: [
    ['@babel/preset-env', { targets: { node: 'current' } }],
    '@babel/preset-typescript',
  ],
};
```

  </TabItem>
</Tabs>

## jest.config.ts

Create a `jest.config.ts` at the root of your project:

<Tabs>
  <TabItem label="ts-jest">

```typescript
import type { Config } from 'jest';

const config: Config = {
  // Use ts-jest to handle TypeScript files
  preset: 'ts-jest',

  // Run in a Node.js environment (use 'jsdom' for browser/React projects)
  testEnvironment: 'node',

  // Only collect coverage from source files (not tests or node_modules)
  collectCoverageFrom: [
    'src/**/*.{ts,tsx}',
    '!src/**/*.d.ts',
    '!src/**/*.stories.{ts,tsx}',
  ],

  // Map path aliases to match tsconfig.json paths
  moduleNameMapper: {
    '^@/(.*)$': '<rootDir>/src/$1',
  },

  // Files to run before each test suite (after test framework is installed)
  setupFilesAfterFramework: ['<rootDir>/jest.setup.ts'],
};

export default config;
```

  </TabItem>
  <TabItem label="babel-jest">

```typescript
import type { Config } from 'jest';

const config: Config = {
  // babel-jest is automatic if babel.config.js is present
  testEnvironment: 'node',

  // Explicit transform for .ts and .tsx files
  transform: {
    '^.+\\.(ts|tsx)$': 'babel-jest',
  },

  moduleNameMapper: {
    '^@/(.*)$': '<rootDir>/src/$1',
  },
};

export default config;
```

  </TabItem>
</Tabs>

:::tip[TypeScript for jest.config]
Using `jest.config.ts` (TypeScript) rather than `jest.config.js` gives you autocomplete and type checking on all configuration options. Make sure `ts-node` is installed for Jest to parse the TypeScript config file: `npm install --save-dev ts-node`.
:::

## Test File Discovery — testMatch Patterns

Jest discovers test files using `testMatch` glob patterns. The defaults cover the most common conventions:

```typescript
// Default testMatch patterns (you rarely need to override these)
testMatch: [
  '**/__tests__/**/*.[jt]s?(x)',  // files inside __tests__ directories
  '**/?(*.)+(spec|test).[jt]s?(x)', // files ending in .spec.ts or .test.ts
],
```

Common project layouts and the patterns that match them:

| Layout | Pattern | Example |
|---|---|---|
| `__tests__` directory | `**/__tests__/**/*.ts` | `src/__tests__/math.test.ts` |
| Co-located spec files | `**/*.spec.ts` | `src/math.spec.ts` |
| Co-located test files | `**/*.test.ts` | `src/math.test.ts` |
| Nested `__tests__` | `**/__tests__/**/*.ts` | `src/api/__tests__/client.test.ts` |

```typescript
// Restricting to only .test.ts files (excludes .spec.ts)
const config: Config = {
  testMatch: ['**/*.test.ts', '**/*.test.tsx'],
};
```

## npm Test Scripts

Add these scripts to `package.json`:

```json
{
  "scripts": {
    "test": "jest",
    "test:watch": "jest --watch",
    "test:watchAll": "jest --watchAll",
    "test:coverage": "jest --coverage",
    "test:ci": "jest --ci --coverage --runInBand"
  }
}
```

| Script | Use case |
|---|---|
| `test` | Run all tests once |
| `test:watch` | Re-run tests affected by changed files |
| `test:watchAll` | Re-run all tests on any file change |
| `test:coverage` | Run tests and generate coverage report |
| `test:ci` | Optimised for CI — no watch, coverage, serial execution |

:::caution[--runInBand in CI]
`--runInBand` forces tests to run serially in a single process. Use it in CI environments with limited memory or when tests share stateful external resources (like a real database). For most CI pipelines, parallel execution is faster and preferred.
:::

## Running Tests from the CLI

```bash
# Run all tests
npx jest

# Run a single test file
npx jest src/math.test.ts

# Run all tests matching a path pattern
npx jest auth

# Run tests whose names match a pattern (--testNamePattern / -t)
npx jest -t "should return 401"

# Run tests in a specific file matching a name pattern
npx jest src/api.test.ts -t "POST /users"

# Run only tests that failed in the last run
npx jest --onlyFailures

# Run tests and bail after the first failure
npx jest --bail

# Run tests and bail after N failures
npx jest --bail=3

# Show verbose test names (even passing tests)
npx jest --verbose
```

## Watch Mode

Watch mode is the fastest way to develop with tests. It tracks file changes and only re-runs the tests affected by those changes.

```bash
npx jest --watch
```

While in watch mode, press these keys for interactive control:

| Key | Action |
|---|---|
| `f` | Run only failed tests |
| `a` | Run all tests |
| `p` | Filter by filename pattern |
| `t` | Filter by test name pattern |
| `u` | Update snapshots |
| `i` | Run one failed test interactively |
| `q` | Quit watch mode |
| `Enter` | Trigger a test run |

:::tip[--watchAll for Non-Git Projects]
`--watch` requires a Git repository to determine which files changed. In projects without Git, use `--watchAll` to watch all files instead.
:::

## Your First Test File

Create `src/math.ts`:

```typescript
export const add = (a: number, b: number): number => a + b;

export const divide = (a: number, b: number): number => {
  if (b === 0) {
    throw new Error('Division by zero is not allowed');
  }
  return a / b;
};

export const clamp = (value: number, min: number, max: number): number =>
  Math.min(Math.max(value, min), max);
```

Create `src/math.test.ts`:

```typescript
import { add, divide, clamp } from './math';

// describe() groups related tests into a named suite
describe('add', () => {
  it('returns the sum of two positive numbers', () => {
    expect(add(2, 3)).toBe(5);
  });

  it('handles negative numbers', () => {
    expect(add(-1, 1)).toBe(0);
  });

  it('handles floating point numbers', () => {
    expect(add(0.1, 0.2)).toBeCloseTo(0.3);
  });
});

describe('divide', () => {
  it('divides two numbers', () => {
    expect(divide(10, 2)).toBe(5);
  });

  it('throws when dividing by zero', () => {
    expect(() => divide(10, 0)).toThrow('Division by zero is not allowed');
  });
});

describe('clamp', () => {
  it('returns the value when within bounds', () => {
    expect(clamp(5, 0, 10)).toBe(5);
  });

  it('clamps to the minimum', () => {
    expect(clamp(-5, 0, 10)).toBe(0);
  });

  it('clamps to the maximum', () => {
    expect(clamp(15, 0, 10)).toBe(10);
  });
});
```

Run the tests:

```bash
npx jest src/math.test.ts --verbose
```

Expected output:

```
  add
    ✓ returns the sum of two positive numbers (2 ms)
    ✓ handles negative numbers
    ✓ handles floating point numbers (1 ms)

  divide
    ✓ divides two numbers
    ✓ throws when dividing by zero

  clamp
    ✓ returns the value when within bounds
    ✓ clamps to the minimum
    ✓ clamps to the maximum

Test Suites: 1 passed, 1 total
Tests:       8 passed, 8 total
```

## Test File Naming Conventions

| Convention | Example | Common in |
|---|---|---|
| `.test.ts` co-located | `src/utils.test.ts` | React, Vite projects |
| `.spec.ts` co-located | `src/utils.spec.ts` | Angular projects |
| `__tests__` directory | `src/__tests__/utils.ts` | Legacy projects |
| Component-collocated | `Button/Button.test.tsx` | Component libraries |
| Feature-grouped | `__tests__/auth/login.test.ts` | Large backends |

Most modern React and TypeScript projects use co-located `.test.ts` files alongside the source file they test.

## Lifecycle Hooks — Jest Globals

Jest provides four lifecycle hooks for setting up and tearing down state around your tests.

```typescript
import { db } from './database';

// Runs once before all tests in this describe block / file
beforeAll(async () => {
  await db.connect();
  await db.migrate();
});

// Runs once after all tests complete
afterAll(async () => {
  await db.close();
});

// Runs before EACH individual test
beforeEach(() => {
  jest.clearAllMocks();
  // Reset any shared state
});

// Runs after EACH individual test
afterEach(async () => {
  await db.truncateAllTables();
});
```

Lifecycle hooks scope to the `describe` block they are defined in:

```typescript
describe('UserService', () => {
  let service: UserService;

  // Only runs before tests in this describe block
  beforeEach(() => {
    service = new UserService(mockDb);
  });

  it('creates a user', async () => {
    const user = await service.create({ name: 'Alice' });
    expect(user.id).toBeDefined();
  });

  describe('when user exists', () => {
    // Runs additionally before tests in this nested describe
    beforeEach(async () => {
      await service.create({ name: 'Existing User' });
    });

    it('throws on duplicate email', async () => {
      await expect(
        service.create({ name: 'Existing User' }),
      ).rejects.toThrow('Email already in use');
    });
  });
});
```

:::tip[Hook Execution Order]
When hooks are nested, the order is: outer `beforeAll` → outer `beforeEach` → inner `beforeEach` → test → inner `afterEach` → outer `afterEach` → outer `afterAll`. Understanding this order is essential when debugging setup/teardown issues.
:::

## Skipping and Focusing Tests

```typescript
// Skip this test (it still shows in output as "skipped")
it.skip('work in progress', () => {
  // ...
});

// Only run this test — useful for debugging a single case
it.only('the one I care about right now', () => {
  // ...
});

// Skip an entire describe block
describe.skip('FeatureFlag: new-dashboard', () => {
  // all tests inside are skipped
});

// Focus an entire describe block
describe.only('critical path', () => {
  // only these tests run
});

// todo — marks a test as planned but not yet written
it.todo('should handle concurrent requests');
```

:::caution[Don't Commit .only]
Committing `it.only` or `describe.only` to your repository means the rest of your test suite is silently skipped in CI. Use a linter rule (`jest/no-focused-tests` from `eslint-plugin-jest`) to catch these before they reach main.
:::

## Test and `it` — Are They Different?

`test()` and `it()` are **identical** — they are aliases for each other. Choose whichever reads more naturally for your team:

```typescript
// Both are equivalent
test('returns 5 when adding 2 and 3', () => {
  expect(add(2, 3)).toBe(5);
});

it('returns 5 when adding 2 and 3', () => {
  expect(add(2, 3)).toBe(5);
});
```

The convention is that `it()` reads like English: _"it returns 5 when adding 2 and 3"_, while `test()` is more explicit. Most teams pick one and stick with it.

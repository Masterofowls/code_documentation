This guide walks you through installing Jest, configuring TypeScript support, writing your first tests, and mastering the essential CLI commands you'll use every day.

## Installation

<Tabs>
  <TabItem label="ts-jest (recommended)">
    `ts-jest` compiles TypeScript using the TypeScript compiler directly, giving you accurate type checking during tests.

    ```bash
    npm install --save-dev jest @types/jest ts-jest
    ```

    Generate a baseline config:

    ```bash
    npx ts-jest config:init
    ```

    This creates a `jest.config.ts` file pre-configured for TypeScript.
</TabItem>
  <TabItem label="babel-jest">
    `babel-jest` uses Babel to strip TypeScript types (no type checking, but faster).

    ```bash
    npm install --save-dev jest @types/jest babel-jest \
      @babel/core @babel/preset-env @babel/preset-typescript
    ```

    Create `babel.config.js`:

    ```javascript
    module.exports = {
      presets: [
        ['@babel/preset-env', { targets: { node: 'current' } }],
        '@babel/preset-typescript',
      ],
    };
    ```
</TabItem>
  <TabItem label="React / Next.js">
    For React projects, add `@testing-library/react` and `jest-environment-jsdom`:

    ```bash
    npm install --save-dev jest @types/jest ts-jest \
      @testing-library/react @testing-library/jest-dom \
      @testing-library/user-event jest-environment-jsdom
    ```
</TabItem>
</Tabs>

## jest.config.ts Setup

Create a `jest.config.ts` file at the root of your project:

```typescript
import type { Config } from 'jest';

const config: Config = {
  // Use ts-jest to handle TypeScript files
  preset: 'ts-jest',

  // 'node' for backend, 'jsdom' for frontend/React
  testEnvironment: 'node',

  // Root directories Jest searches for tests
  roots: ['<rootDir>/src'],

  // File patterns that Jest treats as test files
  testMatch: [
    '**/__tests__/**/*.{ts,tsx}',
    '**/*.{spec,test}.{ts,tsx}',
  ],

  // Map path aliases from tsconfig.json
  moduleNameMapper: {
    '^@/(.*)$': '<rootDir>/src/$1',
  },

  // Run this file after the test framework is set up
  setupFilesAfterFramework: ['<rootDir>/src/test-setup.ts'],

  // Collect coverage from these files
  collectCoverageFrom: [
    'src/**/*.{ts,tsx}',
    '!src/**/*.d.ts',
    '!src/**/index.ts',
  ],
};

export default config;
```

:::tip[Using a Preset]
`preset: 'ts-jest'` is a shorthand that sets `transform`, `moduleFileExtensions`, and several other options for TypeScript. You can override any individual option after setting the preset.
:::

For React projects, update `testEnvironment`:

```typescript
import type { Config } from 'jest';
import nextJest from 'next/jest.js'; // for Next.js projects

const config: Config = {
  preset: 'ts-jest',
  testEnvironment: 'jsdom',
  setupFilesAfterFramework: ['<rootDir>/src/test-setup.ts'],
};

export default config;
```

Create `src/test-setup.ts` to import custom matchers:

```typescript
import '@testing-library/jest-dom';
```

## Test File Naming Conventions

Jest discovers test files using the `testMatch` pattern. The two most common conventions are:

| Convention | Example Path | Common In |
|---|---|---|
| `__tests__` directory | `src/__tests__/math.test.ts` | Legacy projects |
| Co-located test file | `src/math.test.ts` | Modern projects |
| Spec suffix | `src/math.spec.ts` | Angular, BDD style |
| Co-located in `__tests__` | `src/utils/__tests__/math.test.ts` | Large monorepos |

:::tip[Co-location is Preferred]
Place test files next to the source files they test. This makes it easy to see at a glance which files have tests, and simplifies imports (no `../../../../` relative paths).
:::

```
src/
├── utils/
│   ├── math.ts
│   ├── math.test.ts        ← co-located test
│   ├── string-utils.ts
│   └── string-utils.test.ts
├── components/
│   ├── Button.tsx
│   └── Button.test.tsx     ← co-located component test
└── services/
    ├── api.ts
    └── api.test.ts
```

## npm Scripts

Add these scripts to your `package.json`:

```json
{
  "scripts": {
    "test": "jest",
    "test:watch": "jest --watch",
    "test:coverage": "jest --coverage",
    "test:ci": "jest --ci --coverage --runInBand",
    "test:debug": "node --inspect-brk node_modules/.bin/jest --runInBand"
  }
}
```

| Script | Purpose |
|---|---|
| `npm test` | Run all tests once |
| `npm run test:watch` | Run in watch mode (re-runs on file changes) |
| `npm run test:coverage` | Run with coverage report |
| `npm run test:ci` | Optimized for CI — no watch, serial execution, coverage |
| `npm run test:debug` | Launch Node.js inspector for debugging |

## Running Specific Tests

```bash
# Run a single test file
npx jest src/utils/math.test.ts

# Run all tests in a directory
npx jest src/utils/

# Run tests whose file path matches a pattern
npx jest math

# Run tests whose name matches a pattern (--testNamePattern or -t)
npx jest --testNamePattern="adds two numbers"
npx jest -t "adds"

# Run tests matching a file AND name pattern
npx jest math -t "addition"

# Run only tests that failed on the last run
npx jest --lastFailed

# Run only tests in files changed since last commit
npx jest --onlyChanged
```

:::caution[--runInBand for Debugging]
By default Jest runs test files in parallel worker processes. If you're debugging with `console.log` or a Node.js inspector, add `--runInBand` to run all tests serially in the main process. This makes output readable and allows the debugger to attach properly.
:::

## Watch Mode

Watch mode is the most efficient way to work during development. Jest monitors file changes and automatically re-runs relevant tests.

```bash
npx jest --watch
```

Inside watch mode, press these keys:

| Key | Action |
|---|---|
| `f` | Run only failed tests |
| `a` | Run all tests |
| `p` | Filter by file name pattern |
| `t` | Filter by test name pattern |
| `u` | Update all failing snapshots |
| `i` | Update failing snapshots interactively |
| `q` | Quit watch mode |
| `Enter` | Re-run current test suite |

Use `--watchAll` instead of `--watch` if you're not in a Git repository (watch mode uses Git to detect changed files by default).

## Your First Test File

Create `src/utils/math.ts`:

```typescript
export const add = (a: number, b: number): number => a + b;

export const subtract = (a: number, b: number): number => a - b;

export const multiply = (a: number, b: number): number => a * b;

export const divide = (a: number, b: number): number => {
  if (b === 0) {
    throw new Error('Division by zero is not allowed');
  }
  return a / b;
};

export const clamp = (value: number, min: number, max: number): number =>
  Math.min(Math.max(value, min), max);
```

Create `src/utils/math.test.ts`:

```typescript
import { add, subtract, multiply, divide, clamp } from './math';

// describe() groups related tests into a named suite
describe('math utilities', () => {

  // Nested describe blocks for finer organization
  describe('add', () => {
    it('adds two positive numbers', () => {
      expect(add(2, 3)).toBe(5);
    });

    it('adds a positive and negative number', () => {
      expect(add(10, -3)).toBe(7);
    });

    it('adds two negative numbers', () => {
      expect(add(-5, -3)).toBe(-8);
    });

    it('handles zero', () => {
      expect(add(0, 42)).toBe(42);
    });
  });

  describe('subtract', () => {
    it('subtracts two numbers', () => {
      expect(subtract(10, 4)).toBe(6);
    });
  });

  describe('multiply', () => {
    it('multiplies two numbers', () => {
      expect(multiply(3, 4)).toBe(12);
    });

    it('multiplying by zero returns zero', () => {
      expect(multiply(99, 0)).toBe(0);
    });
  });

  describe('divide', () => {
    it('divides two numbers', () => {
      expect(divide(10, 2)).toBe(5);
    });

    it('returns a float for non-integer results', () => {
      expect(divide(7, 2)).toBe(3.5);
    });

    it('throws when dividing by zero', () => {
      expect(() => divide(10, 0)).toThrow('Division by zero is not allowed');
    });
  });

  describe('clamp', () => {
    it('returns value when within range', () => {
      expect(clamp(5, 0, 10)).toBe(5);
    });

    it('returns min when value is below range', () => {
      expect(clamp(-5, 0, 10)).toBe(0);
    });

    it('returns max when value is above range', () => {
      expect(clamp(15, 0, 10)).toBe(10);
    });
  });
});
```

## Test Lifecycle Hooks

Jest provides four lifecycle hooks to run setup and teardown code around your tests:

```typescript
import { createDatabaseConnection, closeDatabaseConnection } from './db';
import type { Database } from './db';

describe('UserRepository', () => {
  let db: Database;

  // Runs ONCE before all tests in this describe block
  beforeAll(async () => {
    db = await createDatabaseConnection(':memory:');
    await db.runMigrations();
  });

  // Runs ONCE after all tests in this describe block
  afterAll(async () => {
    await closeDatabaseConnection(db);
  });

  // Runs before EACH individual test
  beforeEach(async () => {
    await db.run('DELETE FROM users');  // clean slate for each test
    await db.run("INSERT INTO users (id, name) VALUES (1, 'Alice')");
  });

  // Runs after EACH individual test
  afterEach(() => {
    jest.clearAllMocks();  // reset mock state between tests
  });

  it('finds a user by id', async () => {
    const user = await UserRepository.findById(db, 1);
    expect(user).toEqual({ id: 1, name: 'Alice' });
  });

  it('returns null when user does not exist', async () => {
    const user = await UserRepository.findById(db, 999);
    expect(user).toBeNull();
  });
});
```

### Lifecycle Hook Execution Order

When nesting `describe` blocks, hooks execute in this order:

```
beforeAll (outer)
  beforeAll (inner)
    beforeEach (outer)
    beforeEach (inner)
      → test runs
    afterEach (inner)
    afterEach (outer)
  afterAll (inner)
afterAll (outer)
```

:::tip[Prefer beforeEach Over beforeAll for State]
Use `beforeAll` only for expensive one-time setup (database connections, server startup). For resetting test state, always use `beforeEach` — using `beforeAll` for state reset is a common source of test interdependency bugs.
:::

## Skipping and Focusing Tests

```typescript
// Skip a single test
it.skip('this test is temporarily disabled', () => {
  // ...
});

// Skip an entire describe block
describe.skip('WIP feature', () => {
  // all tests skipped
});

// Focus on a single test (ONLY this test runs)
it.only('this is the only test that runs', () => {
  // ...
});

// Focus on a describe block
describe.only('focused suite', () => {
  // only tests in this block run
});

// Run a test conditionally
it.skipIf(process.env.CI === 'true')('skipped in CI', () => {
  // ...
});
```

:::caution[Never Commit .only()]
Committing `it.only()` or `describe.only()` to version control will silently skip all other tests in CI. Configure ESLint with the `jest/no-focused-tests` rule to catch this automatically.
:::

## TypeScript `testMatch` Patterns

Common `testMatch` patterns for different project layouts:

```typescript
// Standard co-located tests
testMatch: ['**/*.{spec,test}.{ts,tsx}']

// Tests only in __tests__ directories
testMatch: ['**/__tests__/**/*.{ts,tsx}']

// Both patterns (most permissive)
testMatch: [
  '**/__tests__/**/*.{ts,tsx}',
  '**/*.{spec,test}.{ts,tsx}',
]

// Exclude specific directories
testPathIgnorePatterns: [
  '/node_modules/',
  '/dist/',
  '/build/',
  '/.next/',
]
```

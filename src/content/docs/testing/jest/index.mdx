Jest is a delightful JavaScript testing framework developed and maintained by Meta. It delivers a complete, zero-configuration testing solution: a fast test runner, a comprehensive assertion library, a powerful mocking system, and built-in code coverage — all in one package.

## What is Jest?

Before Jest, JavaScript developers manually assembled multiple libraries:

- **Mocha** or **Jasmine** as the test runner
- **Chai** or Node's built-in `assert` for assertions
- **Sinon** for spies, stubs, and mocks
- **Istanbul / nyc** for code coverage reporting

Jest bundles all of these capabilities into one cohesive framework with sensible defaults that **just work** for most JavaScript and TypeScript projects without any configuration.

```typescript
// Your first Jest test — no configuration required
import { formatCurrency } from './utils';

describe('formatCurrency', () => {
  it('formats a positive number with dollar sign', () => {
    expect(formatCurrency(1234.56)).toBe('$1,234.56');
  });

  it('handles zero correctly', () => {
    expect(formatCurrency(0)).toBe('$0.00');
  });

  it('handles negative values', () => {
    expect(formatCurrency(-99.99)).toBe('-$99.99');
  });
});
```

:::tip[Why Jest Remains Popular]
Despite newer alternatives like Vitest, Jest remains the most widely deployed JavaScript testing framework. It has first-class support in Create React App, Next.js, NestJS, and dozens of other frameworks — meaning your knowledge transfers directly across projects.
:::

## Jest vs Vitest vs Mocha

| Feature | **Jest** | **Vitest** | **Mocha** |
|---|---|---|---|
| Bundler integration | Babel / ts-jest | Vite (ESM-native) | None — bring your own |
| Assertion library | Built-in `expect` | Built-in `expect` (Chai-compatible) | Bring your own (Chai, assert) |
| Mocking system | Built-in | Built-in | Sinon (external) |
| Coverage | Built-in (Istanbul / V8) | Built-in (Istanbul / V8) | Istanbul (external) |
| Parallel execution | Yes (worker threads) | Yes (worker threads) | Limited |
| Speed | Fast | Very fast (Vite HMR) | Moderate |
| Zero-config | ✅ | ✅ (needs Vite) | ❌ |
| Watch mode | `--watch` | `--watch` | External plugin |
| Snapshot testing | Built-in | Built-in | Plugin required |
| Browser simulation | jsdom | jsdom / happy-dom | External |
| React Testing Library | Excellent | Excellent | Good |
| ESM support | Limited (transform required) | Native | Yes |
| TypeScript support | via ts-jest / babel-jest | Native via Vite | Bring your own |
| Ecosystem maturity | ⭐⭐⭐⭐⭐ | ⭐⭐⭐⭐ | ⭐⭐⭐⭐⭐ |
| Community size | Very large | Growing rapidly | Large |

:::caution[Jest and Native ESM]
Jest has historically required transformation of ES Modules via Babel or ts-jest. If your project uses pure native ESM without a build step, Vitest may offer a smoother setup. Jest 29+ improves ESM support with the `--experimental-vm-modules` flag, but it still requires care.
:::

## Jest Architecture

Jest is built on four integrated pillars that work seamlessly together.

### 1. Test Runner

Jest's test runner discovers and executes test files in parallel using Node.js worker threads. Each test file runs in its own isolated sandboxed environment, preventing global state leaks between test suites.

```
Project
├── src/
│   ├── math.ts
│   ├── string-utils.ts
│   └── api-client.ts
└── __tests__/
    ├── math.test.ts         ← Isolated Worker 1
    ├── string-utils.test.ts ← Isolated Worker 2
    └── api-client.test.ts   ← Isolated Worker 3
```

The test runner provides a rich CLI experience:

```bash
# Run all tests
npx jest

# Run with verbose output
npx jest --verbose

# Run only changed files (requires Git)
npx jest --onlyChanged

# Run in watch mode
npx jest --watch
```

### 2. Assertion Library

Jest's `expect()` API provides over 40 built-in matchers covering equality, truthiness, numbers, strings, arrays, objects, errors, and async values.

```typescript
// Equality
expect(result).toBe(42);                        // strict (===)
expect(obj).toEqual({ name: 'Alice' });         // deep equality

// Collections
expect(arr).toContain('item');
expect(arr).toHaveLength(3);

// Strings
expect(str).toMatch(/pattern/);

// Errors
expect(() => fn()).toThrow('error message');

// Async
await expect(promise).resolves.toBe('success');
await expect(failingFn()).rejects.toThrow(Error);
```

### 3. Mock Library

Jest's mocking system can replace modules, functions, timers, and the entire module registry — enabling complete isolation of the unit under test.

```typescript
// Mock an entire module
jest.mock('./database');

// Create a mock function
const mockFetch = jest.fn().mockResolvedValue({ data: [] });

// Spy on an existing method without replacing it
jest.spyOn(console, 'warn').mockImplementation(() => {});

// Mock the system clock
jest.useFakeTimers();
jest.advanceTimersByTime(5000);
```

### 4. Coverage Engine

Jest integrates Istanbul (or optionally V8) to collect code coverage with zero additional tooling required.

```bash
npx jest --coverage

# Output:
# ----------------------|---------|----------|---------|---------|
# File                  | % Stmts | % Branch | % Funcs | % Lines |
# ----------------------|---------|----------|---------|---------|
# src/math.ts           |   100   |    100   |   100   |   100   |
# src/string-utils.ts   |    85   |     75   |    90   |    85   |
# ----------------------|---------|----------|---------|---------|
```

## Zero-Configuration Philosophy

Jest follows the principle that **most projects should work without any configuration file**. Running `npx jest` in a project with Jest installed will automatically:

<Steps>
1. Discover test files matching `**/__tests__/**/*.{js,ts}` or `**/*.{spec,test}.{js,ts}`
2. Use Babel to transform files if `@babel/core` is present in the project
3. Set `testEnvironment` to `node` (or `jsdom` when `jest-environment-jsdom` is installed)
4. Run test files in parallel across available CPU cores
5. Generate a summary report in the terminal
</Steps>

For TypeScript or React projects, a small configuration file is needed — but it is far less work than assembling separate tools manually.

## Snapshot Testing

Snapshot testing captures the serialized "shape" of a value the first time a test runs, then fails future tests if that shape changes unexpectedly. It is particularly powerful for React component output and large data structures.

```typescript
import { render } from '@testing-library/react';
import { UserCard } from './UserCard';

it('renders the user card correctly', () => {
  const { container } = render(<UserCard name="Alice" role="Admin" />);
  expect(container.firstChild).toMatchSnapshot();
});
```

On first run, Jest creates a `.snap` file alongside your test:

```
// __snapshots__/UserCard.test.tsx.snap
exports[`renders the user card correctly 1`] = `
<div
  class="user-card"
>
  <h2>
    Alice
  </h2>
  <span
    class="role badge"
  >
    Admin
  </span>
</div>
`;
```

On subsequent runs, Jest compares the current render to the stored snapshot. If they differ, the test fails and you must either fix the regression or intentionally update the snapshot.

:::tip[Updating Snapshots Safely]
When you intentionally change a component, run `jest --updateSnapshot` (alias: `jest -u`) to regenerate `.snap` files. Always review the diff in your Git client before committing updated snapshots — they are part of your test suite.
:::

For primitive values and non-React data, use inline snapshots to keep snapshot content visible in the test file:

```typescript
it('serializes a user object', () => {
  const user = createUser({ name: 'Bob', age: 30 });
  expect(user).toMatchInlineSnapshot(`
    Object {
      "age": 30,
      "id": Any<String>,
      "name": "Bob",
    }
  `);
});
```

## jsdom: Browser Simulation

For frontend code that uses DOM APIs (`document`, `window`, `localStorage`, `fetch`), Jest provides **jsdom** — a JavaScript implementation of browser APIs that runs in Node.js.

When `testEnvironment: 'jsdom'` is set (or via the `@jest-environment jsdom` docblock comment), your tests run in a simulated browser environment without launching a real browser.

```typescript
/**
 * @jest-environment jsdom
 */

it('updates the DOM element text', () => {
  document.body.innerHTML = `
    <div id="greeting">Hello</div>
    <button id="btn">Click me</button>
  `;

  const greeting = document.getElementById('greeting')!;
  const btn = document.getElementById('btn')!;

  btn.addEventListener('click', () => {
    greeting.textContent = 'World';
  });

  btn.click();
  expect(greeting.textContent).toBe('World');
});
```

:::caution[jsdom Limitations]
jsdom simulates the DOM API but does **not** execute CSS, render pixels, calculate layout, or handle browser-specific quirks. For visual regression testing, full user-flow testing, or anything that depends on real browser rendering, use [Playwright](/testing/playwright) instead.
:::

## Ecosystem Integration

Jest integrates with the broader JavaScript ecosystem:

- **@testing-library/react** — Accessibility-first React component testing
- **@testing-library/user-event** — Realistic user interaction simulation
- **jest-extended** — Additional matchers (`toBeArray`, `toBeString`, etc.)
- **@testing-library/jest-dom** — DOM-specific matchers (`toBeInTheDocument`, `toHaveClass`)
- **jest-junit** — JUnit XML reporter for CI systems
- **msw** (Mock Service Worker) — API mocking at the network layer

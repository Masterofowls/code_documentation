Jest is a delightful JavaScript testing framework developed and maintained by Meta. It delivers a complete, zero-configuration testing solution: a fast test runner, a comprehensive assertion library, a powerful mocking system, and built-in code coverage — all in one package.

## What is Jest?

Before Jest, JavaScript developers had to manually stitch together multiple libraries to build a functional test suite:

- **Mocha** or **Jasmine** as the test runner
- **Chai** or **assert** for assertions
- **Sinon** for spies, stubs, and mocks
- **Istanbul / nyc** for code coverage

Jest bundles all of these capabilities into a single, cohesive framework with sensible defaults that **just work** for most JavaScript and TypeScript projects.

```typescript
// Your first Jest test — no additional configuration required
import { formatCurrency } from './utils';

describe('formatCurrency', () => {
  it('formats a positive number with a dollar sign', () => {
    expect(formatCurrency(1234.56)).toBe('$1,234.56');
  });

  it('handles zero correctly', () => {
    expect(formatCurrency(0)).toBe('$0.00');
  });

  it('handles negative values', () => {
    expect(formatCurrency(-99.99)).toBe('-$99.99');
  });
});
```

Jest's API is intentionally readable — tests describe behaviour in plain English, making them useful as documentation as well as quality gates.

## Jest vs Vitest vs Mocha

| Feature | **Jest** | **Vitest** | **Mocha** |
|---|---|---|---|
| Bundler / Transformer | Babel / ts-jest | Vite (ESM-native) | None (bring your own) |
| Assertion library | Built-in (`expect`) | Built-in (Chai-compatible `expect`) | Bring your own (Chai, assert) |
| Mocking | Built-in | Built-in | Sinon (external) |
| Coverage | Built-in (Istanbul / V8) | Built-in (Istanbul / V8) | Istanbul (external) |
| Speed | Fast (parallel workers) | Very fast (Vite HMR cache) | Moderate |
| Configuration required | Minimal | Minimal (reuses Vite config) | Moderate |
| Watch mode | `--watch` built-in | `--watch` built-in | External watcher |
| Snapshot testing | Built-in | Built-in | Plugin required |
| Browser simulation | jsdom (simulated) | jsdom / happy-dom | External |
| React / Next.js support | Excellent | Excellent | Good (plugins) |
| Ecosystem maturity | ⭐⭐⭐⭐⭐ | ⭐⭐⭐⭐ | ⭐⭐⭐⭐⭐ |
| Native ESM | Requires configuration | Native | Yes |
| Parallel execution | Worker threads | Worker threads | `--parallel` flag |

:::tip[When to Choose Jest]
Choose Jest when you need battle-tested stability, the largest ecosystem of plugins and integrations, and seamless support for React, Next.js, and Node.js projects. It remains the most widely deployed JavaScript testing framework in production codebases.
:::

:::caution[Jest and Native ESM]
Jest has historically required transformation of ES Modules via Babel or ts-jest. Pure ESM projects without a build step may experience friction. Jest 29+ ships improved experimental ESM support, but still requires `--experimental-vm-modules` and additional configuration. Vitest may be smoother for fully ESM-native projects.
:::

## Jest Architecture

Jest is built on four core pillars that work together seamlessly.

### 1. Test Runner

Jest's test runner executes test files in parallel using Node.js worker threads (or child processes). Each test file runs in its own isolated environment, preventing global state from leaking between suites.

```
Project structure
├── src/
│   ├── math.ts
│   ├── string-utils.ts
│   └── api-client.ts
└── __tests__/
    ├── math.test.ts         ← Worker 1 (isolated Node.js process)
    ├── string-utils.test.ts ← Worker 2 (isolated Node.js process)
    └── api-client.test.ts   ← Worker 3 (isolated Node.js process)
```

The test runner handles:
- Test file discovery via glob patterns
- Parallel execution with configurable worker count
- Retry logic for flaky tests
- Bail-on-first-failure mode
- Watch mode with intelligent re-running of affected tests

### 2. Assertion Library

Jest's `expect()` API provides over 40 built-in matchers for every common testing scenario.

```typescript
// Equality
expect(result).toBe(42);
expect(obj).toEqual({ name: 'Alice', age: 30 });
expect(obj).toStrictEqual({ name: 'Alice', age: 30 });

// Collections
expect(arr).toContain('item');
expect(arr).toHaveLength(3);
expect(arr).toEqual(expect.arrayContaining(['a', 'b']));

// Objects
expect(obj).toHaveProperty('user.name', 'Alice');
expect(obj).toMatchObject({ status: 'active' });

// Errors
expect(() => riskyFn()).toThrow('expected message');
expect(promise).rejects.toThrow(ValidationError);

// Async
expect(fetchUser()).resolves.toMatchObject({ id: 1 });
```

### 3. Mock Library

Jest's mocking system can replace modules, functions, timers, and even entire dependency trees — enabling complete isolation of the unit under test without touching production code.

```typescript
// Mock an entire module
jest.mock('./database');

// Create a mock function
const mockFn = jest.fn().mockReturnValue(42);

// Spy on an existing method
const spy = jest.spyOn(console, 'error').mockImplementation(() => {});

// Replace the clock
jest.useFakeTimers();
jest.runAllTimers();

// Restore everything after tests
afterEach(() => {
  jest.restoreAllMocks();
  jest.useRealTimers();
});
```

### 4. Coverage Engine

Jest integrates Istanbul (or V8) to collect code coverage metrics without any additional tooling. Running `jest --coverage` generates a detailed report across four dimensions:

| Metric | What it measures |
|---|---|
| **Statements** | Every executable statement hit at least once |
| **Branches** | Every `if`/`else`/`ternary`/`switch` branch taken |
| **Functions** | Every declared function called at least once |
| **Lines** | Every source line executed at least once |

```bash
jest --coverage

# Output:
# ----------|---------|----------|---------|---------|
# File      | % Stmts | % Branch | % Funcs | % Lines |
# ----------|---------|----------|---------|---------|
# math.ts   |   95.24 |    87.50 |  100.00 |   95.24 |
# utils.ts  |  100.00 |   100.00 |  100.00 |  100.00 |
# ----------|---------|----------|---------|---------|
```

## Zero-Configuration Philosophy

Jest follows the principle that **most projects should work without any configuration file**. Running `npx jest` in a project that has Jest installed will automatically:

1. Discover test files matching `**/__tests__/**/*.{js,ts}` or `**/*.{spec,test}.{js,ts}`
2. Transform files with Babel if `@babel/core` is present
3. Set `testEnvironment` to `node` by default
4. Run all tests in parallel with an optimal worker count
5. Report results in a readable terminal format

For TypeScript or React projects, minimal configuration is still needed — but it's far less than assembling separate tools from scratch.

```typescript
// jest.config.ts — the minimal TypeScript setup
import type { Config } from 'jest';

const config: Config = {
  preset: 'ts-jest',
  testEnvironment: 'node',
};

export default config;
```

## Snapshot Testing

Snapshot testing captures the serialized output of a value — typically a rendered React component — and stores it in a `.snap` file. Future test runs compare the current output against the stored snapshot, failing if anything changed unexpectedly.

```typescript
import { render } from '@testing-library/react';
import { UserCard } from './UserCard';

it('renders the user card correctly', () => {
  const { asFragment } = render(<UserCard name="Alice" role="Admin" />);
  expect(asFragment()).toMatchSnapshot();
});
```

On first run, Jest creates a `__snapshots__/UserCard.test.tsx.snap` file:

```
exports[`renders the user card correctly 1`] = `
<DocumentFragment>
  <div class="user-card">
    <h2>Alice</h2>
    <span class="role badge badge--admin">Admin</span>
  </div>
</DocumentFragment>
`;
```

On subsequent runs, Jest compares the fresh render to the stored snapshot and fails if they diverge.

:::tip[Updating Snapshots]
When you intentionally change a component's output, run `jest --updateSnapshot` (or `jest -u`) to regenerate the `.snap` files. Always review the diff carefully before committing — snapshot updates are easy to rubber-stamp but hide real regressions.
:::

:::caution[Snapshot Test Anti-Patterns]
Avoid snapshotting large, deeply nested component trees. They become noisy and reviewers stop scrutinising changes. Instead, snapshot small, focused components and prefer targeted `toHaveText` / `toHaveAttribute` assertions for verifying specific properties of larger components.
:::

## jsdom: Browser API Simulation

For frontend code that relies on DOM APIs (`document`, `window`, `navigator`, `localStorage`, `fetch`), Jest provides **jsdom** — a pure-JavaScript implementation of the browser's DOM and CSSOM standards. Configuring `testEnvironment: 'jsdom'` gives your tests a simulated browser context without spinning up a real browser.

```typescript
/**
 * @jest-environment jsdom
 */
it('reads and writes localStorage', () => {
  localStorage.setItem('theme', 'dark');
  expect(localStorage.getItem('theme')).toBe('dark');
  localStorage.clear();
  expect(localStorage.getItem('theme')).toBeNull();
});
```

jsdom limitations to be aware of:

- No CSS layout computation — computed styles are not realistic
- No canvas or WebGL rendering
- No network requests (must be mocked)
- No real browser event loop behaviour

For full browser fidelity — including visual rendering, real network requests, and multi-browser testing — use **Playwright** instead.

Jest is a delightful JavaScript testing framework developed and maintained by Meta. It delivers a complete, zero-configuration testing solution: a fast test runner, a rich assertion library, a powerful mocking system, and built-in code coverage — all in a single package.

## What is Jest?

Before Jest, JavaScript developers had to manually wire together multiple independent libraries:

- **Mocha** or **Jasmine** as the test runner
- **Chai** or Node's built-in `assert` for assertions
- **Sinon** for spies, stubs, and mocks
- **Istanbul / nyc** for code coverage reporting

Jest bundles all of these capabilities into one cohesive framework with sensible defaults that **just work** for most JavaScript and TypeScript projects — no configuration required to get started.

```typescript
// Your first Jest test — no configuration needed
import { formatCurrency } from './utils';

describe('formatCurrency', () => {
  it('formats a positive number with a dollar sign', () => {
    expect(formatCurrency(1234.56)).toBe('$1,234.56');
  });

  it('handles zero', () => {
    expect(formatCurrency(0)).toBe('$0.00');
  });

  it('handles negative values', () => {
    expect(formatCurrency(-99.99)).toBe('-$99.99');
  });
});
```

| Feature                | **Jest**                   | **Vitest**                    | **Mocha**                     |
|------------------------|----------------------------|-------------------------------|-------------------------------|
| Bundler / transform    | Babel / ts-jest            | Vite (ESM-native)             | None (bring your own)         |
| Assertion library      | Built-in (`expect`)        | Built-in (Chai-compatible)    | Bring your own (Chai, assert) |
| Mocking                | Built-in                   | Built-in                      | Sinon (external)              |
| Coverage               | Built-in (Istanbul / V8)   | Built-in (Istanbul / V8)      | Istanbul (external)           |
| Speed                  | Fast (parallel workers)    | Very fast (Vite HMR cache)    | Moderate                      |
| Config required        | Minimal                    | Minimal (Vite config)         | Moderate                      |
| Watch mode             | Yes (`--watch`)            | Yes (`--watch`)               | External plugin               |
| Snapshot testing       | Built-in                   | Built-in                      | Plugin required               |
| Browser simulation     | jsdom (built-in)           | jsdom / happy-dom             | Browser-specific only         |
| React / Next.js        | Excellent (official)       | Excellent                     | Community plugins             |
| ESM support            | Limited (needs transform)  | Native                        | Yes                           |
| Ecosystem maturity     | ⭐⭐⭐⭐⭐                 | ⭐⭐⭐⭐                     | ⭐⭐⭐⭐⭐                   |

:::tip[When to Choose Jest]
Choose Jest when you need battle-tested stability, the largest ecosystem of plugins and integrations, and out-of-the-box compatibility with React, React Native, and Next.js. It remains the most widely-deployed JavaScript testing framework.
:::

:::caution[Jest and Native ESM]
Jest has historically required transformation of ES Modules via Babel or ts-jest. Projects using pure ESM without a build step may find Vitest easier to configure. Jest 29+ improves ESM support, but `"type": "module"` projects still need additional setup.
:::

## Jest Architecture

Jest is built on four core pillars that work together seamlessly.

### 1. Test Runner

Jest's runner executes test files in parallel using Node.js worker processes. Each test file gets its own isolated Node.js environment, preventing global state leakage between suites.

```
Project
├── src/
│   ├── math.ts
│   ├── string.ts
│   └── api.ts
└── __tests__/
    ├── math.test.ts    ← Worker 1 (isolated)
    ├── string.test.ts  ← Worker 2 (isolated)
    └── api.test.ts     ← Worker 3 (isolated)
```

Jest uses a smart scheduler: it runs the slowest test files first to minimise total wall-clock time, and caches transformed files between runs.

### 2. Assertion Library

Jest's `expect()` API provides over 40 built-in matchers covering every common assertion type:

```typescript
expect(value).toBe(42);                          // strict reference equality (===)
expect(obj).toEqual({ name: 'Alice' });          // deep structural equality
expect(arr).toContain('item');                   // array / string inclusion
expect(fn).toThrow('error message');             // error assertion
expect(promise).resolves.toBe('success');        // async assertion
expect(obj).toMatchObject({ status: 'ok' });     // partial object match
```

### 3. Mock Library

Jest's mocking system lets you replace modules, functions, timers, and entire module registries — enabling complete isolation of the unit under test:

```typescript
jest.mock('./database');                  // auto-mock entire module
const mockFn = jest.fn().mockReturnValue(42);   // mock a standalone function
jest.useFakeTimers();                     // take control of setTimeout/setInterval
jest.spyOn(console, 'error');            // spy on existing method without replacing it
```

### 4. Coverage Engine

Jest integrates Istanbul (Babel mode) or V8 (native mode) to collect code coverage with zero additional tooling:

```bash
jest --coverage
# ✓ Produces: statements, branches, functions, and lines coverage
# ✓ Generates HTML report in coverage/lcov-report/index.html
```

## Zero-Configuration Philosophy

Jest follows the principle that **most projects should work without any explicit configuration**. Running `npx jest` in a project with Jest installed will automatically:

1. Discover test files matching `**/__tests__/**/*.{js,ts}` or `**/*.{spec,test}.{js,ts}`
2. Transform files with Babel (if `@babel/core` is present) or ts-jest (if configured)
3. Set `testEnvironment` to `node` (or detect `jsdom` from project dependencies)
4. Run tests in parallel across all available CPU cores

For TypeScript or React projects you need minimal configuration — still far less than assembling a toolchain from scratch.

## Snapshot Testing

Snapshot testing captures the serialised form of a value at a point in time and fails future tests if that shape changes unexpectedly. It is particularly powerful for React component output and complex data structures.

```typescript
import { render } from '@testing-library/react';
import { UserCard } from './UserCard';

it('renders the user card correctly', () => {
  const { container } = render(<UserCard name="Alice" role="Admin" />);
  expect(container.firstChild).toMatchSnapshot();
});
```

On first run, Jest writes a `.snap` file alongside your test:

```
// __snapshots__/UserCard.test.tsx.snap
exports[`renders the user card correctly 1`] = `
<div
  class="user-card"
>
  <h2>
    Alice
  </h2>
  <span
    class="role"
  >
    Admin
  </span>
</div>
`;
```

On subsequent runs, Jest compares the current output against the stored snapshot and fails the test if they differ — catching unintentional UI regressions.

:::tip[Updating Snapshots]
When you intentionally change a component's output, run `jest --updateSnapshot` (or `jest -u`) to regenerate the `.snap` files. Always review the diff with `git diff` before committing updated snapshots to ensure the change was intentional.
:::

### Inline Snapshots

For small values, inline snapshots keep the expected value directly in the test file:

```typescript
it('formats the greeting', () => {
  expect(greet('Alice')).toMatchInlineSnapshot(`"Hello, Alice!"`);
});
```

## jsdom: Browser Environment Simulation

For frontend code that uses DOM APIs (`document`, `window`, `localStorage`, `fetch`), Jest provides **jsdom** — a headless JavaScript implementation of browser APIs that runs inside Node.js. When `testEnvironment: 'jsdom'` is configured, your tests operate in a simulated browser without launching a real browser process.

```typescript
/**
 * @jest-environment jsdom
 */
it('updates the DOM element on button click', () => {
  document.body.innerHTML = `
    <div id="counter">0</div>
    <button id="increment">+1</button>
  `;

  const counter = document.getElementById('counter')!;
  const button = document.getElementById('increment')!;

  button.addEventListener('click', () => {
    counter.textContent = String(Number(counter.textContent) + 1);
  });

  button.click();
  button.click();

  expect(counter.textContent).toBe('2');
});
```

:::caution[jsdom is Not a Real Browser]
jsdom does not execute CSS, render pixels, perform layout, or support all browser APIs. For tests that depend on real browser behaviour — scroll positions, computed styles, CSS animations, or browser-specific APIs — use Playwright instead.
:::

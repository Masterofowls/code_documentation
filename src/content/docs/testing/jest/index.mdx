Jest is a delightful JavaScript testing framework developed and maintained by Meta. It delivers a complete, zero-configuration testing solution: a fast parallel test runner, a comprehensive assertion library, a powerful mocking system, and built-in code coverage — all in one package.

## What is Jest?

Before Jest, JavaScript developers had to manually stitch together multiple libraries:

- **Mocha** or **Jasmine** as the test runner
- **Chai** or Node's built-in `assert` for assertions
- **Sinon** for spies, stubs, and mocks
- **Istanbul / nyc** for code coverage reporting

Jest bundles all of these capabilities into a single, cohesive framework with sensible defaults that **just work** for most JavaScript and TypeScript projects.

```typescript
// Your first Jest test — no configuration required
import { formatCurrency } from './utils';

describe('formatCurrency', () => {
  it('formats a positive number with a dollar sign', () => {
    expect(formatCurrency(1234.56)).toBe('$1,234.56');
  });

  it('handles zero', () => {
    expect(formatCurrency(0)).toBe('$0.00');
  });

  it('handles negative values', () => {
    expect(formatCurrency(-99.99)).toBe('-$99.99');
  });
});
```

Run it with:

```bash
npx jest
```

## Jest vs Vitest vs Mocha

| Feature            | **Jest**                   | **Vitest**                 | **Mocha**             |
| ------------------ | -------------------------- | -------------------------- | --------------------- |
| Bundler            | Babel / ts-jest            | Vite (ESM-native)          | None (bring your own) |
| Assertions         | Built-in (`expect`)        | Built-in (Chai-compatible) | Bring your own (Chai) |
| Mocking            | Built-in                   | Built-in                   | Sinon (external)      |
| Coverage           | Built-in (Istanbul / V8)   | Built-in (Istanbul / V8)   | Istanbul (external)   |
| Speed              | Fast (parallel workers)    | Very fast (Vite HMR)       | Moderate              |
| Config required    | Minimal                    | Minimal (Vite config)      | Moderate              |
| Watch mode         | `--watch` / `--watchAll`   | `--watch`                  | Plugin required       |
| Snapshot testing   | Built-in                   | Built-in                   | Plugin required       |
| Browser simulation | jsdom (built-in)           | jsdom / happy-dom          | Browser-specific      |
| React / Next.js    | Excellent support          | Excellent support          | Good (plugins)        |
| Ecosystem maturity | ⭐⭐⭐⭐⭐                 | ⭐⭐⭐⭐                   | ⭐⭐⭐⭐⭐            |
| Native ESM         | Limited (transform needed) | Native                     | Yes                   |
| TypeScript         | ts-jest or babel-jest      | Native (esbuild)           | ts-node / register    |
| Parallel execution | Yes (workers)              | Yes (workers)              | Yes (--parallel)      |

:::tip[When to Choose Jest]
Choose Jest when you need battle-tested stability, an extensive ecosystem (especially for React, Next.js, and NestJS projects), and a single tool that covers all testing needs without additional configuration. It is the most widely deployed JavaScript testing framework in production codebases.
:::

:::caution[Jest and Native ESM]
Jest has historically required transformation of ES Modules via Babel or ts-jest. If your project uses pure ESM without a build step, Vitest may offer a smoother experience. Jest 29+ improves ESM support via the `--experimental-vm-modules` flag, but it still requires additional configuration steps.
:::

## Jest Architecture

Jest is built on four core pillars that work together seamlessly:

### 1. Test Runner

Jest's test runner executes test files in parallel using Node.js worker processes. Each test file gets its own isolated Node.js environment, preventing global state pollution between test suites. Jest uses a priority queue to run the slowest tests first, minimising total wall-clock time.

```
Project
├── src/
│   ├── math.ts
│   ├── string.ts
│   └── api.ts
└── __tests__/
    ├── math.test.ts    ← Worker 1 (isolated VM context)
    ├── string.test.ts  ← Worker 2 (isolated VM context)
    └── api.test.ts     ← Worker 3 (isolated VM context)
```

### 2. Assertion Library

Jest's `expect()` API provides over 40 built-in matchers covering equality, truthiness, numbers, strings, arrays, objects, errors, and asynchronous values. The matcher API is chainable and produces clear, human-readable failure messages.

```typescript
expect(42).toBe(42); // strict equality (===)
expect({ a: 1 }).toEqual({ a: 1 }); // deep equality
expect(['a', 'b']).toContain('a'); // array contains
expect(() => risky()).toThrow('expected error'); // error thrown
await expect(promise).resolves.toBe('success'); // promise resolves
await expect(failingPromise).rejects.toThrow(); // promise rejects
```

### 3. Mock Library

Jest's mocking system can replace functions, modules, timers, and even the entire module registry — enabling complete isolation of the unit under test without touching the file system.

```typescript
// Mock an entire module
jest.mock('./database');

// Mock a single function
const mockFetch = jest.fn().mockResolvedValue({ ok: true, json: () => ({}) });

// Spy on an existing method without replacing it
const spy = jest.spyOn(console, 'error').mockImplementation(() => {});

// Replace timers globally
jest.useFakeTimers();
jest.advanceTimersByTime(5000);
```

### 4. Coverage Engine

Jest integrates Istanbul (or the V8 coverage engine) to collect code coverage without any additional tooling. Coverage data is collected at the same time tests run — no separate instrumentation step needed.

```bash
jest --coverage

# Output:
# -------------------|---------|----------|---------|---------|
# File               | % Stmts | % Branch | % Funcs | % Lines |
# -------------------|---------|----------|---------|---------|
# src/math.ts        |   100   |    100   |   100   |   100   |
# src/api.ts         |   87.5  |    75    |   100   |   87.5  |
```

## Zero-Configuration Philosophy

Jest follows the principle that **most projects should work without any configuration**. Running `npx jest` in a project that has Jest installed will automatically:

<Steps>
1. Discover test files matching `**/__tests__/**/*.{js,ts,jsx,tsx}` or `**/*.{spec,test}.{js,ts,jsx,tsx}`
2. Apply Babel transformation if `@babel/core` is installed, or fall back to plain Node.js execution
3. Set `testEnvironment` to `node` by default (or `jsdom` when `@testing-library/dom` is detected)
4. Run tests in parallel with a worker count matching available CPU cores
5. Display a coloured summary with pass/fail counts and timing information
</Steps>

For TypeScript or React projects, minimal configuration is needed — but it is still far less than assembling separate tools.

```json
// Minimum package.json configuration for TypeScript
{
  "jest": {
    "preset": "ts-jest"
  }
}
```

## Snapshot Testing

Snapshot testing captures the "shape" of a value at a point in time and then fails future tests if that shape changes unexpectedly. It is particularly powerful for React component output but works for any serialisable JavaScript value.

```typescript
import { render } from '@testing-library/react';
import { UserCard } from './UserCard';

it('renders the user card correctly', () => {
  const { container } = render(<UserCard name="Alice" role="Admin" />);
  expect(container).toMatchSnapshot();
});
```

On first run, Jest creates a `.snap` file alongside the test file:

```
// __snapshots__/UserCard.test.tsx.snap
exports[`renders the user card correctly 1`] = `
<div>
  <div
    class="user-card"
  >
    <h2>
      Alice
    </h2>
    <span
      class="role badge"
    >
      Admin
    </span>
  </div>
</div>
`;
```

On subsequent runs, Jest compares the current output to the stored snapshot and fails if they differ. This catches accidental UI regressions automatically.

:::tip[Updating Snapshots]
When you intentionally change a component's output, run `jest --updateSnapshot` (or `jest -u`) to regenerate all `.snap` files. Always review the diff shown in the terminal before committing updated snapshots — the goal is to catch _accidental_ changes, not block _intentional_ ones.
:::

Inline snapshots keep the snapshot directly in the test file for smaller values:

```typescript
expect({ id: 1, name: 'Alice' }).toMatchInlineSnapshot(`
  Object {
    "id": 1,
    "name": "Alice",
  }
`);
```

## jsdom: Browser Simulation

For frontend code that uses DOM APIs (`document`, `window`, `localStorage`, `fetch`), Jest provides **jsdom** — a headless JavaScript implementation of the W3C DOM and HTML standards. When `testEnvironment: 'jsdom'` is configured, your tests run in a simulated browser environment without launching a real browser.

```typescript
/**
 * @jest-environment jsdom
 */
it('toggles the button state on click', () => {
  document.body.innerHTML = `
    <button id="toggle" aria-pressed="false">Toggle</button>
  `;

  const button = document.getElementById('toggle') as HTMLButtonElement;
  button.addEventListener('click', () => {
    const pressed = button.getAttribute('aria-pressed') === 'true';
    button.setAttribute('aria-pressed', String(!pressed));
  });

  button.click();
  expect(button.getAttribute('aria-pressed')).toBe('true');

  button.click();
  expect(button.getAttribute('aria-pressed')).toBe('false');
});
```

:::caution[jsdom is Not a Real Browser]
jsdom does not execute CSS, render pixels, perform layout calculations, or support all browser APIs. For tests that depend on computed styles, animations, or actual rendering, use Playwright's browser-based testing instead.
:::

jsdom is ideal for:

- Testing DOM manipulation utilities
- Testing React, Vue, and Angular components with `@testing-library`
- Testing `localStorage` / `sessionStorage` interactions
- Testing custom event dispatching and handling

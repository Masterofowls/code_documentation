Jest is a delightful JavaScript testing framework developed and maintained by Meta. It provides a complete, zero-configuration testing solution: a fast test runner, a comprehensive assertion library, a powerful mocking system, and built-in code coverage reporting — all in one package.

## What is Jest?

Before Jest, JavaScript developers had to manually stitch together multiple independent libraries just to get a working test suite:

- **Mocha** or **Jasmine** as the test runner
- **Chai** or **assert** for assertions
- **Sinon** for spies, stubs, and mocks
- **Istanbul / nyc** for code coverage
- **Proxyquire** or **rewire** for module mocking

Jest bundles all of these capabilities into a single cohesive framework with sensible defaults that work for the vast majority of JavaScript and TypeScript projects — from small utilities to large React applications.

```typescript
// A complete Jest test — no configuration required
import { formatCurrency } from './utils';

describe('formatCurrency', () => {
  it('formats a positive number with a dollar sign', () => {
    expect(formatCurrency(1234.56)).toBe('$1,234.56');
  });

  it('handles zero correctly', () => {
    expect(formatCurrency(0)).toBe('$0.00');
  });

  it('handles negative values', () => {
    expect(formatCurrency(-99.99)).toBe('-$99.99');
  });
});
```

:::tip[Origin Story]
Jest was originally built by Christoph Pojer at Facebook and open-sourced in 2014. It was designed specifically for testing React applications but has since become the most widely used JavaScript testing framework regardless of the frontend framework used.
:::

## Jest vs Vitest vs Mocha

| Feature | **Jest** | **Vitest** | **Mocha** |
|---|---|---|---|
| Bundler / Transform | Babel / ts-jest | Vite (ESM-native) | None (BYO) |
| Assertion library | Built-in (`expect`) | Built-in (Chai-compatible) | BYO (Chai, assert) |
| Mocking | Built-in | Built-in | Sinon (external) |
| Coverage | Built-in (Istanbul / V8) | Built-in (Istanbul / V8) | Istanbul (external) |
| Speed | Fast (parallel workers) | Very fast (Vite HMR) | Moderate |
| Config required | Minimal | Minimal | Moderate |
| Watch mode | Yes (`--watch`) | Yes (`--watch`) | Yes (external) |
| Snapshot testing | Built-in | Built-in | Plugin required |
| Browser testing | jsdom (simulated) | jsdom / happy-dom | Browser-specific |
| React / Next.js | Excellent support | Excellent support | Good (plugins) |
| ESM support | Limited (transform required) | Native ESM | Yes |
| Ecosystem maturity | ⭐⭐⭐⭐⭐ | ⭐⭐⭐⭐ | ⭐⭐⭐⭐⭐ |
| Community size | Very large | Growing fast | Very large |

:::tip[When to Choose Jest]
Choose Jest when you need battle-tested stability, an extensive ecosystem (especially for React and Next.js projects), and a single dependency that handles everything. It has the most answers on Stack Overflow and the largest collection of third-party integrations.
:::

:::caution[Jest and Native ESM]
Jest has historically required transformation of ES Modules via Babel or ts-jest. If your project uses pure ESM without a build step, Vitest may be a smoother choice. Jest 29+ improves ESM support via `--experimental-vm-modules` but still requires explicit configuration.
:::

## Jest Architecture

Jest is built on four core pillars that work together seamlessly:

### 1. Test Runner

Jest's test runner discovers and executes test files in parallel using Node.js worker processes. Each test file gets its own isolated Node.js module registry, preventing global state from leaking between test suites.

```
Project
├── src/
│   ├── math.ts
│   ├── string.ts
│   └── api.ts
└── __tests__/
    ├── math.test.ts    ← Worker 1 (isolated environment)
    ├── string.test.ts  ← Worker 2 (isolated environment)
    └── api.test.ts     ← Worker 3 (isolated environment)
```

The runner collects results from all workers and produces a unified report. By default, Jest runs as many workers as there are CPU cores minus one, keeping your machine responsive during test runs.

### 2. Assertion Library

Jest's `expect()` API provides over 40 built-in matchers covering equality, truthiness, numbers, strings, arrays, objects, errors, and asynchronous values. The matchers produce clear, human-readable failure messages.

```typescript
expect(value).toBe(42);                           // Strict equality (===)
expect(obj).toEqual({ name: 'Alice' });           // Deep equality
expect(arr).toContain('item');                    // Array containment
expect(fn).toThrow('error message');              // Error thrown
expect(asyncFn).resolves.toBe('success');         // Promise resolves
expect(asyncFn).rejects.toThrow('Network error'); // Promise rejects
```

### 3. Mock Library

Jest's mocking system can replace modules, functions, class instances, and timers — giving you complete control over the dependencies of the unit under test.

```typescript
jest.mock('./database');                    // Auto-mock entire module
const mockFn = jest.fn().mockReturnValue(42); // Mock a function
jest.useFakeTimers();                       // Mock setTimeout/setInterval
jest.spyOn(console, 'error').mockImplementation(() => {}); // Spy on method
```

### 4. Coverage Engine

Jest integrates Istanbul (default) or V8 to collect code coverage without any additional tooling or configuration.

```bash
jest --coverage

# Output:
# -------------------|---------|----------|---------|---------|
# File               | % Stmts | % Branch | % Funcs | % Lines |
# -------------------|---------|----------|---------|---------|
# All files          |   92.31 |    83.33 |     100 |   92.31 |
#  src/utils.ts      |   92.31 |    83.33 |     100 |   92.31 |
# -------------------|---------|----------|---------|---------|
```

## Zero-Configuration Philosophy

Jest follows the principle that **most projects should work without any configuration**. Running `npx jest` in a project with Jest installed will:

1. Discover test files matching `**/__tests__/**/*.{js,ts}` or `**/*.{spec,test}.{js,ts}`
2. Use Babel to transform files (if `@babel/core` is installed) or fall back to plain Node.js
3. Set `testEnvironment` to `node` by default
4. Run tests in parallel with an optimal worker count

For TypeScript or React projects, minimal configuration is needed (a `jest.config.ts` file and one or two packages), but it is far less than assembling separate tools from scratch.

```bash
# Works without any jest.config.ts
npx jest

# With TypeScript — needs ts-jest or babel-jest
npx jest --passWithNoTests
```

## Snapshot Testing

Snapshot testing captures the serialized output of a value at a point in time, then fails future tests if that output changes unexpectedly. It is most powerful for React component rendering.

```typescript
import { render } from '@testing-library/react';
import { UserCard } from './UserCard';

it('renders the user card correctly', () => {
  const { asFragment } = render(<UserCard name="Alice" role="Admin" />);
  expect(asFragment()).toMatchSnapshot();
});
```

On the first run, Jest creates a `.snap` file alongside the test:

```
// __snapshots__/UserCard.test.tsx.snap
exports[`renders the user card correctly 1`] = `
<DocumentFragment>
  <div class="user-card">
    <h2>Alice</h2>
    <span class="badge badge--admin">Admin</span>
  </div>
</DocumentFragment>
`;
```

On subsequent runs, Jest compares the current output to the stored snapshot. Any unexpected change causes the test to fail, prompting a deliberate review.

:::tip[Inline Snapshots]
Use `toMatchInlineSnapshot()` to store the snapshot directly in the test file instead of a separate `.snap` file. This keeps small snapshots visible alongside the test logic and works well for simple values.
:::

:::caution[Snapshot Abuse]
Avoid snapshotting large, complex components as a substitute for meaningful assertions. A 300-line snapshot diff is hard to review and easy to blindly update. Prefer targeted `toMatchObject` assertions for component output and reserve snapshots for stable, simple structures.
:::

## jsdom: Browser Simulation

For frontend code that uses DOM APIs (`document`, `window`, `localStorage`, `fetch`), Jest can run in a simulated browser environment using **jsdom** — a JavaScript implementation of browser APIs that runs in Node.js.

```typescript
/**
 * @jest-environment jsdom
 */
it('updates the DOM element text content', () => {
  document.body.innerHTML = '<button id="btn">Click me</button>';

  const button = document.getElementById('btn') as HTMLButtonElement;
  button.textContent = 'Clicked!';

  expect(button.textContent).toBe('Clicked!');
});
```

You can set `testEnvironment: 'jsdom'` globally in `jest.config.ts` or per-file using the docblock comment shown above.

:::caution[jsdom Limitations]
jsdom does not execute CSS, render visual layouts, or handle real network requests. It is a JavaScript approximation of a browser — not a real one. For tests that require actual browser rendering, CSS evaluation, or cross-browser verification, use **Playwright** instead.
:::

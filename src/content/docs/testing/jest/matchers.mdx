Matchers are the heart of Jest assertions. Every assertion starts with `expect(receivedValue)` and chains a matcher method that describes the expected condition. Jest ships with over 40 built-in matchers covering virtually every assertion need.

## Equality Matchers

Understanding the difference between Jest's equality matchers prevents subtle bugs.

### toBe — Strict Equality

`toBe` uses `Object.is()` (similar to `===`) for comparison. It compares **by reference** for objects and arrays.

```typescript
// ✅ Works for primitives
expect(2 + 2).toBe(4);
expect('hello').toBe('hello');
expect(true).toBe(true);
expect(null).toBe(null);

// ❌ Fails for objects — different references
expect({ a: 1 }).toBe({ a: 1 });   // FAIL — different object instances

// ❌ Fails for NaN — use toBeNaN() instead
expect(NaN).toBe(NaN);  // FAIL — NaN !== NaN in JS
```

### toEqual — Deep Equality

`toEqual` recursively checks that two objects have the same structure and values. It ignores `undefined` properties.

```typescript
// ✅ Works for objects and arrays
expect({ a: 1, b: 2 }).toEqual({ a: 1, b: 2 });
expect([1, 2, 3]).toEqual([1, 2, 3]);

// ✅ Nested objects
expect({ user: { name: 'Alice', age: 30 } })
  .toEqual({ user: { name: 'Alice', age: 30 } });

// ✅ Ignores undefined properties
expect({ a: 1, b: undefined }).toEqual({ a: 1 }); // PASS

// ❌ Fails for type differences
expect({ a: '1' }).toEqual({ a: 1 }); // FAIL — string vs number
```

### toStrictEqual — Strict Deep Equality

`toStrictEqual` is the strictest option — it checks structure, values, **and** type. It does **not** ignore `undefined` properties and distinguishes between object instances of different classes.

```typescript
// toStrictEqual catches these cases that toEqual misses:

// ❌ Different class instances
class Dog {}
class Cat {}
expect(new Dog()).toEqual(new Cat());       // PASS (toEqual)
expect(new Dog()).toStrictEqual(new Cat()); // FAIL (toStrictEqual)

// ❌ undefined properties matter
expect({ a: 1, b: undefined }).toEqual({ a: 1 });       // PASS
expect({ a: 1, b: undefined }).toStrictEqual({ a: 1 }); // FAIL

// ❌ Sparse arrays vs dense arrays
expect([1, , 3]).toEqual([1, undefined, 3]);       // PASS
expect([1, , 3]).toStrictEqual([1, undefined, 3]); // FAIL
```

| Matcher | Reference check | Type check | `undefined` props | Class distinction |
|---|---|---|---|---|
| `toBe` | ✅ (by reference) | ✅ | ✅ | ✅ |
| `toEqual` | ❌ (by value) | ✅ | ❌ (ignored) | ❌ |
| `toStrictEqual` | ❌ (by value) | ✅ | ✅ | ✅ |

:::tip[Which Equality Matcher to Use?]
- Use `toBe` for primitive values (numbers, strings, booleans)
- Use `toEqual` for plain objects and arrays when you don't care about `undefined` fields
- Use `toStrictEqual` when you need exact structural fidelity, especially in data transformation tests
:::

## Truthiness Matchers

```typescript
// toBeTruthy — passes for any truthy value (not: false, 0, '', null, undefined, NaN)
expect(1).toBeTruthy();
expect('hello').toBeTruthy();
expect([]).toBeTruthy();         // empty array is truthy!
expect({}).toBeTruthy();         // empty object is truthy!

// toBeFalsy — passes for any falsy value
expect(0).toBeFalsy();
expect('').toBeFalsy();
expect(null).toBeFalsy();
expect(undefined).toBeFalsy();
expect(NaN).toBeFalsy();
expect(false).toBeFalsy();

// toBeNull — specifically null
expect(null).toBeNull();
expect(undefined).not.toBeNull(); // undefined is NOT null

// toBeUndefined — specifically undefined
expect(undefined).toBeUndefined();
expect(null).not.toBeUndefined();

// toBeDefined — anything except undefined
expect(null).toBeDefined();    // null IS defined
expect(0).toBeDefined();
expect('').toBeDefined();
expect(undefined).not.toBeDefined();
```

:::caution[toBeTruthy vs toBeDefined]
These are not equivalent. `toBeDefined` only rejects `undefined`, while `toBeTruthy` rejects `null`, `0`, `''`, `false`, and `NaN` too. Use the most specific matcher for your intent.
:::

## Number Matchers

```typescript
const result = calculateTotal(items);

// Greater/less than comparisons
expect(result).toBeGreaterThan(0);
expect(result).toBeGreaterThanOrEqual(10);
expect(result).toBeLessThan(1000);
expect(result).toBeLessThanOrEqual(999.99);

// Floating point — use toBeCloseTo to avoid precision issues
expect(0.1 + 0.2).not.toBe(0.3);            // PASS — floating point!
expect(0.1 + 0.2).toBeCloseTo(0.3);         // PASS — close enough
expect(0.1 + 0.2).toBeCloseTo(0.3, 5);      // second arg = decimal places (default: 2)

// NaN
expect(NaN).toBeNaN();
expect(parseInt('not-a-number')).toBeNaN();
expect(42).not.toBeNaN();

// Integer check (custom, using modulo)
expect(result % 1).toBe(0); // is an integer

// Range check
expect(result).toBeGreaterThanOrEqual(0);
expect(result).toBeLessThanOrEqual(100);
```

## String Matchers

```typescript
const message = 'Hello, World! Welcome to Jest testing.';

// toContain — substring match (case-sensitive)
expect(message).toContain('Hello');
expect(message).toContain('Jest');
expect(message).not.toContain('Mocha');

// toMatch — regex or string pattern
expect(message).toMatch('Hello');               // string
expect(message).toMatch(/World/);              // regex
expect(message).toMatch(/^Hello/);             // starts with
expect(message).toMatch(/testing\.$/);         // ends with
expect(message).toMatch(/\bJest\b/);           // whole word
expect('hello@example.com').toMatch(/^\S+@\S+\.\S+$/); // email pattern

// toHaveLength — string length
expect('Hello').toHaveLength(5);
expect('').toHaveLength(0);

// Combining with not
expect(message).not.toMatch(/error/i);
expect(message).not.toContain('Mocha');
```

## Array Matchers

```typescript
const fruits = ['apple', 'banana', 'cherry', 'date'];
const numbers = [1, 2, 3, 4, 5];

// toContain — exact element match
expect(fruits).toContain('banana');
expect(fruits).not.toContain('mango');

// toHaveLength — array length
expect(fruits).toHaveLength(4);
expect([]).toHaveLength(0);

// toEqual with expect.arrayContaining — subset match
// "received array contains ALL expected items (order doesn't matter)"
expect(fruits).toEqual(
  expect.arrayContaining(['cherry', 'apple'])  // subset, any order
);

// NOT arrayContaining
expect(fruits).not.toEqual(
  expect.arrayContaining(['mango'])  // mango is not in fruits
);

// Check every element (using toEqual on each)
numbers.forEach(n => expect(n).toBeGreaterThan(0));

// Check the first/last element
expect(fruits[0]).toBe('apple');
expect(fruits.at(-1)).toBe('date');

// Tuple-style exact array match
expect(numbers).toEqual([1, 2, 3, 4, 5]);

// Array of objects
const users = [
  { id: 1, name: 'Alice' },
  { id: 2, name: 'Bob' },
];

expect(users).toEqual(
  expect.arrayContaining([
    expect.objectContaining({ name: 'Alice' }),
  ])
);
```

## Object Matchers

```typescript
const user = {
  id: 1,
  name: 'Alice',
  email: 'alice@example.com',
  address: {
    city: 'Portland',
    country: 'US',
    zip: '97201',
  },
  roles: ['admin', 'user'],
};

// toHaveProperty — check key existence
expect(user).toHaveProperty('name');
expect(user).toHaveProperty('address.city');         // nested path
expect(user).toHaveProperty(['address', 'zip']);     // array path

// toHaveProperty with value
expect(user).toHaveProperty('name', 'Alice');
expect(user).toHaveProperty('address.country', 'US');
expect(user).toHaveProperty('roles', ['admin', 'user']);

// toMatchObject — partial object match
// "received object contains at LEAST these properties"
expect(user).toMatchObject({
  name: 'Alice',
  address: { city: 'Portland' },  // nested partial match
});

// expect.objectContaining — inline partial match
expect(user).toEqual(
  expect.objectContaining({
    id: expect.any(Number),
    name: expect.any(String),
    email: expect.stringMatching(/@example\.com$/),
  })
);

// Asymmetric matchers
expect(user).toMatchObject({
  id: expect.any(Number),
  name: expect.any(String),
  roles: expect.arrayContaining(['admin']),
});
```

### Asymmetric Matchers Reference

| Matcher | Description |
|---|---|
| `expect.any(Constructor)` | Matches any value of that type |
| `expect.anything()` | Matches anything except `null` and `undefined` |
| `expect.arrayContaining(arr)` | Array contains all elements in `arr` |
| `expect.objectContaining(obj)` | Object contains all keys in `obj` |
| `expect.stringContaining(str)` | String contains `str` |
| `expect.stringMatching(regex)` | String matches regex pattern |
| `expect.not.arrayContaining(arr)` | Negated array check |
| `expect.not.objectContaining(obj)` | Negated object check |

## Error Matchers

```typescript
// toThrow — function throws any error
expect(() => JSON.parse('invalid json')).toThrow();

// toThrow with string — error message contains the string
expect(() => divide(10, 0)).toThrow('Division by zero');

// toThrow with regex — error message matches pattern
expect(() => validateAge(-1)).toThrow(/must be.*positive/i);

// toThrow with Error class — throws specific error type
expect(() => requireAuth(null)).toThrow(TypeError);
expect(() => fetchData('bad-url')).toThrow(NetworkError);

// toThrow with error object — class AND message
expect(() => login('', 'pass')).toThrow(
  new ValidationError('Username is required')
);

// Testing async errors — use rejects
await expect(asyncFn()).rejects.toThrow('async error message');
await expect(asyncFn()).rejects.toBeInstanceOf(CustomError);
```

:::caution[Always Wrap in Arrow Function]
`expect(fn())` calls the function immediately and captures the return value. `expect(() => fn())` passes the function without calling it, allowing Jest to catch the thrown error. **Always wrap functions that should throw in an arrow function.**
:::

```typescript
// ❌ This evaluates fn() immediately — error is uncaught by Jest
expect(fn()).toThrow();

// ✅ This passes the function reference — Jest can catch the error
expect(() => fn()).toThrow();
```

## Custom Matchers with `expect.extend()`

When built-in matchers aren't expressive enough, create domain-specific matchers:

```typescript
// src/test-utils/matchers.ts

expect.extend({
  toBeValidEmail(received: string) {
    const emailRegex = /^[^\s@]+@[^\s@]+\.[^\s@]+$/;
    const pass = emailRegex.test(received);
    return {
      pass,
      message: () =>
        pass
          ? `Expected "${received}" not to be a valid email address`
          : `Expected "${received}" to be a valid email address`,
    };
  },

  toBeWithinRange(received: number, floor: number, ceiling: number) {
    const pass = received >= floor && received <= ceiling;
    return {
      pass,
      message: () =>
        `Expected ${received} ${pass ? 'not ' : ''}to be within range [${floor}, ${ceiling}]`,
    };
  },

  toHaveBeenCalledWithObject(received: jest.Mock, expected: object) {
    const calls = received.mock.calls;
    const pass = calls.some(args =>
      args.some(arg =>
        typeof arg === 'object' &&
        Object.entries(expected).every(([k, v]) => arg[k] === v)
      )
    );
    return {
      pass,
      message: () =>
        `Expected mock function ${pass ? 'not ' : ''}to have been called with an object matching ${JSON.stringify(expected)}`,
    };
  },
});

// TypeScript: extend the Jest type declarations
declare global {
  namespace jest {
    interface Matchers<R> {
      toBeValidEmail(): R;
      toBeWithinRange(floor: number, ceiling: number): R;
      toHaveBeenCalledWithObject(expected: object): R;
    }
  }
}
```

Register custom matchers in `setupFilesAfterFramework`:

```typescript
// jest.config.ts
const config: Config = {
  setupFilesAfterFramework: [
    '<rootDir>/src/test-utils/matchers.ts',
  ],
};
```

Use them in tests:

```typescript
it('validates email format', () => {
  expect('alice@example.com').toBeValidEmail();
  expect('not-an-email').not.toBeValidEmail();
});

it('response time is within acceptable range', () => {
  const responseTime = measureResponseTime();
  expect(responseTime).toBeWithinRange(0, 500); // 0-500ms
});
```

:::tip[Reusable Matcher Libraries]
`jest-extended` provides 70+ additional matchers like `toBeArray()`, `toBeString()`, `toBeEmpty()`, `toBePositive()`, `toSatisfy()`, and more. Install it with `npm install --save-dev jest-extended` and add it to `setupFilesAfterFramework`.
:::

## Negating Matchers

Every matcher can be negated with `.not`:

```typescript
expect(value).not.toBe(null);
expect(arr).not.toHaveLength(0);
expect(fn).not.toThrow();
expect(str).not.toMatch(/error/i);
expect(obj).not.toHaveProperty('deletedAt');
```

## Combining Multiple Expectations

```typescript
// Test multiple properties on the same value
it('creates a valid user object', () => {
  const user = createUser({ name: 'Alice', email: 'alice@example.com' });

  expect(user).toHaveProperty('id');
  expect(user.id).toEqual(expect.any(String));
  expect(user.name).toBe('Alice');
  expect(user.email).toBeValidEmail();
  expect(user.createdAt).toBeInstanceOf(Date);
  expect(user.role).toBe('user');
  expect(user.isActive).toBe(true);
});
```

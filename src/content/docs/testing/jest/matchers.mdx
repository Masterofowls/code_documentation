Jest's `expect()` API provides a rich set of matchers that let you validate values in tests. Every matcher call produces a clear, human-readable failure message when the assertion fails.

## Equality Matchers

The three equality matchers behave differently and choosing the right one matters:

### `toBe` — strict reference equality

`toBe` uses `Object.is()` (equivalent to `===` with correct handling of `NaN` and `+0`/`-0`). Use it for primitives.

```typescript
expect(1 + 1).toBe(2);
expect('hello').toBe('hello');
expect(true).toBe(true);
expect(null).toBe(null);

// FAILS — different object references
expect({ a: 1 }).toBe({ a: 1 });
```

### `toEqual` — deep structural equality

`toEqual` recursively checks that two values have the same structure and values. It ignores `undefined` object properties and array items.

```typescript
expect({ a: 1, b: { c: 2 } }).toEqual({ a: 1, b: { c: 2 } });
expect([1, 2, 3]).toEqual([1, 2, 3]);

// toEqual ignores undefined properties — these PASS
expect({ a: 1, b: undefined }).toEqual({ a: 1 });
expect([1, , 3]).toEqual([1, undefined, 3]);
```

### `toStrictEqual` — deep equality including `undefined`

`toStrictEqual` is stricter than `toEqual`: it checks for `undefined` properties, distinguishes sparse arrays from arrays with `undefined`, and checks object types (instances of different classes are not equal).

```typescript
// These FAIL with toStrictEqual but PASS with toEqual
expect({ a: 1, b: undefined }).toStrictEqual({ a: 1 }); // FAILS
expect([1, , 3]).toStrictEqual([1, undefined, 3]); // FAILS

// Class instances — toStrictEqual checks constructor
class Point {
  constructor(
    public x: number,
    public y: number,
  ) {}
}
expect(new Point(1, 2)).toStrictEqual({ x: 1, y: 2 }); // FAILS — different types
expect(new Point(1, 2)).toEqual({ x: 1, y: 2 }); // PASSES
```

:::tip[Which equality matcher to use?]

- Use `toBe` for primitives (numbers, strings, booleans, null, undefined).
- Use `toEqual` for plain objects and arrays where undefined properties don't matter.
- Use `toStrictEqual` when you want to enforce exact structural equality including class types.
  :::

## Truthiness Matchers

```typescript
// Truthy / Falsy
expect(1).toBeTruthy();
expect('hello').toBeTruthy();
expect({}).toBeTruthy();
expect([]).toBeTruthy();

expect(0).toBeFalsy();
expect('').toBeFalsy();
expect(null).toBeFalsy();
expect(undefined).toBeFalsy();
expect(false).toBeFalsy();
expect(NaN).toBeFalsy();

// Specific null / undefined checks
expect(null).toBeNull();
expect(undefined).toBeUndefined();
expect('value').toBeDefined(); // not undefined
expect(null).toBeDefined(); // null IS defined — use toBeNull() instead
```

| Matcher           | Passes when value is                                             |
| ----------------- | ---------------------------------------------------------------- |
| `toBeTruthy()`    | Any truthy value (`1`, `'text'`, `{}`, `[]`)                     |
| `toBeFalsy()`     | Any falsy value (`0`, `''`, `null`, `undefined`, `false`, `NaN`) |
| `toBeNull()`      | Exactly `null`                                                   |
| `toBeUndefined()` | Exactly `undefined`                                              |
| `toBeDefined()`   | Anything except `undefined`                                      |

## Number Matchers

```typescript
expect(5).toBeGreaterThan(4);
expect(5).toBeGreaterThanOrEqual(5);
expect(3).toBeLessThan(4);
expect(3).toBeLessThanOrEqual(3);

// NaN
expect(NaN).toBeNaN();
expect(1).not.toBeNaN();

// Infinity
expect(Infinity).toBeGreaterThan(Number.MAX_SAFE_INTEGER);
```

### `toBeCloseTo` for floating-point arithmetic

Never use `toBe` for floating-point comparisons due to rounding errors:

```typescript
// FAILS — floating point imprecision
expect(0.1 + 0.2).toBe(0.3); // 0.30000000000000004

// PASSES — compares to a number of decimal places (default: 2)
expect(0.1 + 0.2).toBeCloseTo(0.3);
expect(0.1 + 0.2).toBeCloseTo(0.3, 5); // 5 decimal places
expect(Math.PI).toBeCloseTo(3.14159, 4);
```

## String Matchers

```typescript
// Contains substring
expect('Hello, World!').toContain('World');
expect('Hello, World!').not.toContain('foo');

// Matches regex
expect('Hello, World!').toMatch(/World/);
expect('user@example.com').toMatch(/^[\w.]+@[\w.]+\.\w+$/);
expect('The quick brown fox').toMatch('quick'); // also accepts string

// String length
expect('hello').toHaveLength(5);
expect('').toHaveLength(0);

// Starts / ends with
expect('Hello').toMatch(/^Hello/);
expect('Hello').toMatch(/o$/);
```

:::tip[Prefer regex for complex string assertions]
When validating email addresses, URLs, phone numbers, or other structured strings, use a regex pattern with `toMatch` rather than checking individual substrings. This produces clearer error messages and documents the expected format.
:::

## Array Matchers

```typescript
const fruits = ['apple', 'banana', 'cherry'];

// Contains a specific item (uses deep equality)
expect(fruits).toContain('banana');
expect([{ id: 1 }, { id: 2 }]).toContainEqual({ id: 1 });

// Array length
expect(fruits).toHaveLength(3);
expect([]).toHaveLength(0);

// Array contains all expected items (order doesn't matter)
expect(fruits).toEqual(expect.arrayContaining(['banana', 'apple']));

// Array does NOT contain these items
expect(fruits).not.toEqual(expect.arrayContaining(['mango']));

// Each item passes a custom assertion
expect([2, 4, 6, 8]).toEqual(
  expect.arrayContaining(expect.arrayContaining([])),
);
```

### `expect.arrayContaining` for partial array matching

```typescript
const users = [
  { id: 1, name: 'Alice', role: 'admin' },
  { id: 2, name: 'Bob', role: 'user' },
  { id: 3, name: 'Carol', role: 'user' },
];

// Check that the array contains at least these items
expect(users).toEqual(
  expect.arrayContaining([
    expect.objectContaining({ name: 'Alice', role: 'admin' }),
    expect.objectContaining({ name: 'Bob' }),
  ]),
);
```

## Object Matchers

### `toHaveProperty` — check a property exists or has a value

```typescript
const user = {
  id: 1,
  profile: {
    name: 'Alice',
    address: {
      city: 'Paris',
      zip: '75001',
    },
  },
};

// Check existence
expect(user).toHaveProperty('id');
expect(user).toHaveProperty('profile.name');

// Check value
expect(user).toHaveProperty('id', 1);
expect(user).toHaveProperty('profile.name', 'Alice');
expect(user).toHaveProperty('profile.address.city', 'Paris');

// Array index path
const data = { items: ['a', 'b', 'c'] };
expect(data).toHaveProperty('items[0]', 'a');
expect(data).toHaveProperty(['items', 1], 'b');
```

### `toMatchObject` — partial object matching

```typescript
expect({ id: 1, name: 'Alice', createdAt: new Date() }).toMatchObject({
  id: 1,
  name: 'Alice',
  // createdAt is not checked — extra properties are allowed
});

// Works with arrays of objects
expect([
  { id: 1, name: 'Alice' },
  { id: 2, name: 'Bob' },
]).toMatchObject([{ id: 1 }, { id: 2 }]);
```

### `expect.objectContaining` — asymmetric object matcher

Use this inside other matchers when you only care about certain properties:

```typescript
const mockFn = jest.fn();
mockFn({ userId: 42, action: 'login', timestamp: Date.now() });

// Only assert on the properties you care about
expect(mockFn).toHaveBeenCalledWith(
  expect.objectContaining({ userId: 42, action: 'login' }),
);
```

### Asymmetric matchers combined

```typescript
expect(response).toMatchObject({
  status: 200,
  data: expect.arrayContaining([
    expect.objectContaining({
      id: expect.any(Number),
      name: expect.any(String),
      email: expect.stringMatching(/^[\w.]+@[\w.]+\.\w+$/),
    }),
  ]),
  meta: expect.objectContaining({
    total: expect.any(Number),
  }),
});
```

Available asymmetric matchers:

- `expect.any(Constructor)` — matches any instance of the given class
- `expect.anything()` — matches anything except `null` or `undefined`
- `expect.arrayContaining(array)` — matches arrays that include these items
- `expect.objectContaining(object)` — matches objects with at least these properties
- `expect.stringContaining(string)` — matches strings containing this substring
- `expect.stringMatching(regex)` — matches strings matching this pattern
- `expect.not.anything()` — the negated form

## Error Matchers

Testing that errors are thrown correctly is critical for robust error handling:

```typescript
function parseAge(value: string): number {
  const age = parseInt(value, 10);
  if (isNaN(age)) throw new TypeError(`Invalid age: "${value}"`);
  if (age < 0 || age > 150) throw new RangeError(`Age out of range: ${age}`);
  return age;
}

// Test that an error is thrown
expect(() => parseAge('abc')).toThrow();

// Test the error message (string match)
expect(() => parseAge('abc')).toThrow('Invalid age');
expect(() => parseAge('abc')).toThrow('Invalid age: "abc"');

// Test with regex
expect(() => parseAge('abc')).toThrow(/Invalid age/);

// Test the error type
expect(() => parseAge('abc')).toThrow(TypeError);
expect(() => parseAge('200')).toThrow(RangeError);

// Test both type and message
expect(() => parseAge('abc')).toThrow(new TypeError('Invalid age: "abc"'));
```

:::caution[Always wrap the throwing call in a function]
`expect(parseAge('abc')).toThrow()` will **not** work — the function throws before `expect()` can wrap it. Always use `expect(() => parseAge('abc')).toThrow()` with an arrow function.
:::

### Testing async errors

```typescript
// Async functions that throw
async function fetchUser(id: number) {
  if (id <= 0) throw new Error('Invalid user ID');
  // ...
}

await expect(fetchUser(-1)).rejects.toThrow('Invalid user ID');
await expect(fetchUser(-1)).rejects.toThrow(Error);
await expect(fetchUser(-1)).rejects.toMatchObject({
  message: 'Invalid user ID',
});
```

## Negating Matchers

Any matcher can be negated with `.not`:

```typescript
expect(5).not.toBe(6);
expect('hello').not.toContain('world');
expect([1, 2, 3]).not.toHaveLength(0);
expect(() => add(1, 2)).not.toThrow();
expect(null).not.toBeDefined();
```

## Custom Matchers with `expect.extend()`

When the built-in matchers are not sufficient, you can add your own with `expect.extend()`:

```typescript
// src/testUtils/matchers.ts
import type { MatcherContext } from '@jest/expect-utils';

expect.extend({
  toBeWithinRange(
    this: MatcherContext,
    received: number,
    floor: number,
    ceiling: number,
  ) {
    const pass = received >= floor && received <= ceiling;
    if (pass) {
      return {
        message: () =>
          `expected ${received} not to be within range ${floor} - ${ceiling}`,
        pass: true,
      };
    } else {
      return {
        message: () =>
          `expected ${received} to be within range ${floor} - ${ceiling}`,
        pass: false,
      };
    }
  },

  toBeValidEmail(this: MatcherContext, received: string) {
    const emailRegex = /^[^\s@]+@[^\s@]+\.[^\s@]+$/;
    const pass = emailRegex.test(received);
    return {
      pass,
      message: () =>
        pass
          ? `expected "${received}" not to be a valid email`
          : `expected "${received}" to be a valid email`,
    };
  },
});

// TypeScript: extend the expect interface
declare global {
  namespace jest {
    interface Matchers<R> {
      toBeWithinRange(floor: number, ceiling: number): R;
      toBeValidEmail(): R;
    }
  }
}
```

Register the custom matchers in your `setupFilesAfterFramework`:

```typescript
// src/setupTests.ts
import './testUtils/matchers';
```

Use them like any built-in matcher:

```typescript
expect(15).toBeWithinRange(10, 20); // PASSES
expect(5).not.toBeWithinRange(10, 20); // PASSES
expect('user@example.com').toBeValidEmail(); // PASSES
expect('not-an-email').not.toBeValidEmail(); // PASSES
```

## Quick Reference

| Category       | Matchers                                                                                                     |
| -------------- | ------------------------------------------------------------------------------------------------------------ |
| **Equality**   | `toBe`, `toEqual`, `toStrictEqual`                                                                           |
| **Truthiness** | `toBeTruthy`, `toBeFalsy`, `toBeNull`, `toBeUndefined`, `toBeDefined`                                        |
| **Numbers**    | `toBeGreaterThan`, `toBeGreaterThanOrEqual`, `toBeLessThan`, `toBeLessThanOrEqual`, `toBeCloseTo`, `toBeNaN` |
| **Strings**    | `toMatch`, `toContain`, `toHaveLength`                                                                       |
| **Arrays**     | `toContain`, `toContainEqual`, `toHaveLength`, `toEqual(expect.arrayContaining(...))`                        |
| **Objects**    | `toHaveProperty`, `toMatchObject`, `toEqual(expect.objectContaining(...))`                                   |
| **Errors**     | `toThrow`, `toThrowError`                                                                                    |
| **Async**      | `resolves.toBe`, `rejects.toThrow`                                                                           |
| **Snapshots**  | `toMatchSnapshot`, `toMatchInlineSnapshot`                                                                   |
| **Mock calls** | `toHaveBeenCalled`, `toHaveBeenCalledTimes`, `toHaveBeenCalledWith`, `toHaveBeenLastCalledWith`              |

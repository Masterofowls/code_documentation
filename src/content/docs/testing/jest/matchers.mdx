Matchers are the methods chained onto `expect()` that define what condition must hold for a test to pass. Jest ships with over 40 built-in matchers and supports extending the set with custom matchers via `expect.extend()`.

## Negating Matchers

Every matcher can be negated by chaining `.not`:

```typescript
expect(42).not.toBe(0);
expect([]).not.toContain('item');
expect(() => safeFunc()).not.toThrow();
```

## Equality Matchers

### `toBe`

Uses `Object.is` (strict equality, `===`) to compare primitives. **Do not use for objects or arrays** — it compares references, not structure.

```typescript
expect(1 + 1).toBe(2);
expect('hello').toBe('hello');
expect(null).toBe(null);

const obj = { a: 1 };
expect(obj).toBe(obj);       // ✓ same reference
expect(obj).not.toBe({ a: 1 }); // ✓ different reference
```

### `toEqual`

Recursively checks that two objects have the same structure and values. **Use for objects and arrays.** Ignores `undefined` properties and array holes.

```typescript
expect({ a: 1, b: { c: 2 } }).toEqual({ a: 1, b: { c: 2 } });
expect([1, 2, 3]).toEqual([1, 2, 3]);

// toEqual ignores undefined properties
expect({ a: 1, b: undefined }).toEqual({ a: 1 }); // ✓ passes
```

### `toStrictEqual`

Like `toEqual` but **does not ignore** `undefined` properties and distinguishes between object types (e.g., class instances vs plain objects).

```typescript
class Person {
  constructor(public name: string) {}
}

// toEqual passes — same shape
expect(new Person('Alice')).toEqual({ name: 'Alice' });

// toStrictEqual fails — different types
expect(new Person('Alice')).not.toStrictEqual({ name: 'Alice' });

// toStrictEqual catches undefined
expect({ a: 1, b: undefined }).not.toStrictEqual({ a: 1 });
```

### When to Use Each

| Matcher | Use case |
|---------|----------|
| `toBe` | Primitives (numbers, strings, booleans, null, undefined) |
| `toEqual` | Plain objects and arrays where class identity doesn't matter |
| `toStrictEqual` | Class instances and when `undefined` properties are significant |

## Truthiness Matchers

```typescript
// toBeTruthy — any value that coerces to true
expect(1).toBeTruthy();
expect('hello').toBeTruthy();
expect([]).toBeTruthy();      // empty array is truthy!
expect({}).toBeTruthy();      // empty object is truthy!

// toBeFalsy — any value that coerces to false
expect(0).toBeFalsy();
expect('').toBeFalsy();
expect(null).toBeFalsy();
expect(undefined).toBeFalsy();
expect(NaN).toBeFalsy();

// Specific null / undefined checks
expect(null).toBeNull();
expect(undefined).toBeUndefined();
expect('hello').not.toBeNull();
expect(42).toBeDefined();       // not undefined
expect(undefined).toBeUndefined();
```

:::tip[Prefer Specific Matchers]
Use `toBeNull()` instead of `toBe(null)`, and `toBeUndefined()` instead of `toBe(undefined)`. Specific matchers produce clearer error messages and communicate intent better.
:::

## Number Matchers

```typescript
const price = 19.99;

expect(price).toBeGreaterThan(10);
expect(price).toBeGreaterThanOrEqual(19.99);
expect(price).toBeLessThan(100);
expect(price).toBeLessThanOrEqual(19.99);

// Floating point precision — never use toBe for float arithmetic
expect(0.1 + 0.2).toBeCloseTo(0.3);           // default 2 decimal places
expect(0.1 + 0.2).toBeCloseTo(0.3, 5);        // 5 decimal places precision
expect(Math.PI).toBeCloseTo(3.14159, 4);

// NaN
expect(NaN).toBeNaN();
expect(parseInt('not a number')).toBeNaN();
expect(42).not.toBeNaN();

// Finite / Infinite (use jest-extended for these)
expect(Infinity).not.toBeFinite?.();           // if jest-extended is installed
```

:::caution[Never Use toBe for Floating Point]
Due to IEEE 754 floating-point arithmetic, `0.1 + 0.2` equals `0.30000000000000004` in JavaScript. Always use `toBeCloseTo` when comparing the results of floating-point calculations.
:::

## String Matchers

### `toMatch`

Tests a string against a regular expression or substring:

```typescript
const email = 'user@example.com';

// With a regex
expect(email).toMatch(/^[\w.-]+@[\w.-]+\.\w+$/);
expect(email).toMatch(/@example\.com$/);

// With a substring
expect(email).toMatch('example.com');
expect('Hello World').toMatch('World');
expect('Hello World').not.toMatch('Goodbye');
```

### `toContain` (strings)

```typescript
expect('Hello, World!').toContain('World');
expect('Testing with Jest').toContain('Jest');
```

### `toHaveLength`

Works for both strings and arrays:

```typescript
expect('hello').toHaveLength(5);
expect('').toHaveLength(0);
```

## Array Matchers

### `toContain`

Checks that an array contains a specific item using `===` equality:

```typescript
const fruits = ['apple', 'banana', 'cherry'];

expect(fruits).toContain('banana');
expect(fruits).not.toContain('grape');
```

### `toHaveLength`

```typescript
expect([1, 2, 3]).toHaveLength(3);
expect([]).toHaveLength(0);
```

### `toEqual` with `expect.arrayContaining`

`expect.arrayContaining` checks that the array contains all specified items (in any order), regardless of extra items:

```typescript
const result = ['dog', 'cat', 'bird', 'fish'];

// Passes — result contains all of these
expect(result).toEqual(expect.arrayContaining(['dog', 'fish']));

// Also useful with toMatchObject
expect({ animals: result }).toMatchObject({
  animals: expect.arrayContaining(['cat', 'bird']),
});

// Strict — only these items, no extras
expect(['a', 'b']).toEqual(['a', 'b']); // exact match via toEqual
```

### Checking Every Element

```typescript
const numbers = [2, 4, 6, 8];
numbers.forEach(n => expect(n % 2).toBe(0));
```

## Object Matchers

### `toHaveProperty`

Checks that an object has a property at the given path (dot-notation or array syntax), optionally matching its value:

```typescript
const user = {
  id: 1,
  name: 'Alice',
  address: {
    city: 'London',
    postcode: 'EC1A 1BB',
  },
  roles: ['admin', 'editor'],
};

// Property existence
expect(user).toHaveProperty('name');
expect(user).toHaveProperty('address.city');     // nested path
expect(user).toHaveProperty(['address', 'city']); // array path syntax

// Property value
expect(user).toHaveProperty('name', 'Alice');
expect(user).toHaveProperty('address.city', 'London');
expect(user).toHaveProperty('roles[0]', 'admin'); // array index
```

### `toMatchObject`

Checks that a received object contains the expected subset of properties. Extra properties on the received object are ignored:

```typescript
const apiResponse = {
  status: 200,
  data: { id: 1, name: 'Alice', createdAt: '2024-01-01' },
  headers: { 'content-type': 'application/json' },
};

// Only checks the specified properties — extra properties are fine
expect(apiResponse).toMatchObject({
  status: 200,
  data: { name: 'Alice' },
});

// Works with arrays of objects
const users = [
  { id: 1, name: 'Alice', role: 'admin' },
  { id: 2, name: 'Bob', role: 'user' },
];

expect(users).toMatchObject([
  { name: 'Alice' },
  { name: 'Bob' },
]);
```

### `expect.objectContaining`

An asymmetric matcher that can be nested inside `toEqual` or `toMatchObject`:

```typescript
const event = { type: 'click', target: '#btn', timestamp: Date.now() };

// timestamp changes every run — use objectContaining to ignore it
expect(event).toEqual(expect.objectContaining({
  type: 'click',
  target: '#btn',
}));
```

## Error Matchers

### `toThrow`

Test that a function throws an error. **The function must be wrapped in an arrow function**:

```typescript
function parseJson(input: string) {
  return JSON.parse(input); // throws SyntaxError on invalid input
}

// Check that it throws (any error)
expect(() => parseJson('invalid')).toThrow();

// Check error message (substring)
expect(() => parseJson('invalid')).toThrow('JSON');

// Check error message (exact)
expect(() => parseJson('{')).toThrow('Unexpected end of JSON input');

// Check error message (regex)
expect(() => parseJson('{')).toThrow(/Unexpected/);

// Check error type
expect(() => parseJson('invalid')).toThrow(SyntaxError);

// Check both type and message
expect(() => parseJson('invalid')).toThrow(new SyntaxError());
```

:::caution[Wrap Throwing Functions in Arrow Functions]
If you write `expect(parseJson('invalid')).toThrow()`, the function executes immediately and throws _before_ Jest can catch it, causing the test to fail with an unhandled error. Always wrap: `expect(() => parseJson('invalid')).toThrow()`.
:::

### `toThrowError`

An alias for `toThrow` — they behave identically.

### Async errors

For async functions that reject, use `rejects`:

```typescript
async function fetchUser(id: number) {
  if (id <= 0) throw new Error('Invalid ID');
  return { id, name: 'Alice' };
}

await expect(fetchUser(-1)).rejects.toThrow('Invalid ID');
await expect(fetchUser(-1)).rejects.toBeInstanceOf(Error);
```

## Custom Matchers with `expect.extend`

You can add project-specific matchers to improve test readability:

```typescript
// jest.setup.ts (or a matchers.ts file imported in setup)

interface CustomMatchers<R = unknown> {
  toBeWithinRange(min: number, max: number): R;
  toBeValidEmail(): R;
  toHaveBeenCalledExactlyOnce(): R;
}

declare global {
  namespace jest {
    interface Expect extends CustomMatchers {}
    interface Matchers<R> extends CustomMatchers<R> {}
    interface InverseAsymmetricMatchers extends CustomMatchers {}
  }
}

expect.extend({
  toBeWithinRange(received: number, min: number, max: number) {
    const pass = received >= min && received <= max;
    return {
      pass,
      message: () =>
        pass
          ? `expected ${received} not to be within range [${min}, ${max}]`
          : `expected ${received} to be within range [${min}, ${max}]`,
    };
  },

  toBeValidEmail(received: string) {
    const emailRegex = /^[^\s@]+@[^\s@]+\.[^\s@]+$/;
    const pass = emailRegex.test(received);
    return {
      pass,
      message: () =>
        pass
          ? `expected "${received}" not to be a valid email address`
          : `expected "${received}" to be a valid email address`,
    };
  },
});
```

Usage in tests:

```typescript
expect(42).toBeWithinRange(1, 100);         // ✓
expect(150).not.toBeWithinRange(1, 100);    // ✓
expect('user@example.com').toBeValidEmail(); // ✓
expect('not-an-email').not.toBeValidEmail(); // ✓
```

:::tip[jest-extended]
The `jest-extended` package provides 70+ ready-to-use additional matchers like `toBeArray()`, `toBeString()`, `toStartWith()`, `toEndWith()`, `toBePositive()`, `toBeNegative()`, and many more — saving you from writing custom matchers for common cases.

```bash
npm install --save-dev jest-extended
```

```typescript
// jest.setup.ts
import 'jest-extended';
```
:::

## Asymmetric Matchers Reference

Asymmetric matchers can be used _inside_ `toEqual`, `toMatchObject`, or anywhere a value is expected — they match without requiring exact equality.

```typescript
// expect.anything() — matches anything except null or undefined
expect({ id: 1, token: 'xyz123' }).toMatchObject({
  id: 1,
  token: expect.anything(),
});

// expect.any(Constructor) — matches any instance of the given class
expect({ createdAt: new Date() }).toMatchObject({
  createdAt: expect.any(Date),
});

// expect.stringContaining — matches strings containing the substring
expect({ message: 'Hello, World!' }).toMatchObject({
  message: expect.stringContaining('World'),
});

// expect.stringMatching — matches strings matching a regex
expect({ email: 'user@example.com' }).toMatchObject({
  email: expect.stringMatching(/^[\w.-]+@[\w.-]+\.\w+$/),
});

// expect.arrayContaining — subset match for arrays
expect({ tags: ['a', 'b', 'c'] }).toMatchObject({
  tags: expect.arrayContaining(['a', 'c']),
});

// expect.objectContaining — subset match for objects
expect({ user: { id: 1, name: 'Alice', role: 'admin' } }).toMatchObject({
  user: expect.objectContaining({ name: 'Alice' }),
});

// expect.not.arrayContaining — array must not contain these items
expect([1, 2, 3]).toEqual(expect.not.arrayContaining([4, 5]));
```

React Testing Library (RTL) combined with Jest is the standard for testing React components. RTL encourages testing from the user's perspective — interacting with components the way a real user would — rather than testing implementation details.

## Setup

```bash
npm install --save-dev @testing-library/react @testing-library/user-event @testing-library/jest-dom
```

Configure `jest.config.ts` to use jsdom and load jest-dom matchers:

```typescript
import type { Config } from 'jest';

const config: Config = {
  testEnvironment: 'jsdom',
  setupFilesAfterFramework: ['<rootDir>/jest.setup.ts'],
};

export default config;
```

```typescript
// jest.setup.ts
import '@testing-library/jest-dom';
```

`@testing-library/jest-dom` adds custom matchers like `toBeInTheDocument()`, `toHaveTextContent()`, `toBeVisible()`, `toBeDisabled()`, and more.

## `render()`

`render()` mounts a component into a real DOM node (provided by jsdom) and returns a set of query utilities scoped to that component.

```typescript
import { render, screen } from '@testing-library/react';
import { UserCard } from './UserCard';

it('displays the user name and role', () => {
  render(<UserCard name="Alice Chen" role="Administrator" />);

  expect(screen.getByText('Alice Chen')).toBeInTheDocument();
  expect(screen.getByText('Administrator')).toBeInTheDocument();
});
```

`render()` returns several utilities you can destructure:

```typescript
const {
  getByText,        // Direct query (scoped to rendered component)
  asFragment,       // Serialize the current DOM for snapshot testing
  container,        // The raw DOM node (avoid using this)
  rerender,         // Re-render with new props
  unmount,          // Unmount the component
} = render(<MyComponent />);

// Prefer screen.* over destructured queries — screen is always up to date
```

## Screen Queries

`screen` provides query functions that search the entire rendered DOM. Queries come in three families:

| Family | Throws if missing? | Waits for async? | Use case |
|---|---|---|---|
| `getBy*` | ✅ Yes | ❌ No | Element that must be present synchronously |
| `queryBy*` | ❌ Returns null | ❌ No | Asserting an element is NOT present |
| `findBy*` | ✅ Yes (after timeout) | ✅ Yes (awaitable) | Element that appears asynchronously |
| `getAllBy*` | ✅ Yes (needs ≥1) | ❌ No | Multiple elements (returns array) |
| `queryAllBy*` | ❌ Returns [] | ❌ No | Multiple elements, may return empty |
| `findAllBy*` | ✅ Yes (after timeout) | ✅ Yes | Multiple async elements |

### Query Priority — Accessibility First

RTL recommends this priority order for queries. Higher priority = more accessible and resilient tests:

```typescript
// 1. getByRole — BEST: matches ARIA roles and accessible names
screen.getByRole('button', { name: 'Submit Order' });
screen.getByRole('heading', { level: 1 });
screen.getByRole('textbox', { name: 'Email address' });
screen.getByRole('checkbox', { name: 'Accept terms' });
screen.getByRole('combobox', { name: 'Country' });
screen.getByRole('link', { name: 'Learn more' });
screen.getByRole('dialog', { name: 'Confirm deletion' });
screen.getByRole('listitem');

// 2. getByLabelText — for form fields associated with a label
screen.getByLabelText('Email address');
screen.getByLabelText(/password/i);

// 3. getByPlaceholderText — only when no label is available
screen.getByPlaceholderText('Search...');

// 4. getByText — for non-interactive elements with text content
screen.getByText('Your order has been placed');
screen.getByText(/welcome/i);

// 5. getByDisplayValue — for pre-populated form elements
screen.getByDisplayValue('alice@example.com');

// 6. getByAltText — for images
screen.getByAltText('User profile photo');

// 7. getByTitle — for title attributes (tooltip text)
screen.getByTitle('Close dialog');

// 8. getByTestId — LAST RESORT: use data-testid attribute
screen.getByTestId('loading-spinner');
```

:::tip[Use getByRole First]
`getByRole` is the most powerful query because it maps to how assistive technologies like screen readers see your page. If you can query by role + accessible name, your component is accessible to all users.
:::

```typescript
// Real component test using role queries
it('submits the login form', async () => {
  const user = userEvent.setup();
  render(<LoginForm onSuccess={jest.fn()} />);

  await user.type(screen.getByRole('textbox', { name: /email/i }), 'alice@example.com');
  await user.type(screen.getByLabelText(/password/i), 'securepassword');
  await user.click(screen.getByRole('button', { name: /sign in/i }));

  expect(screen.queryByText(/invalid credentials/i)).not.toBeInTheDocument();
});
```

## `userEvent` vs `fireEvent`

| | `userEvent` (v14+) | `fireEvent` |
|---|---|---|
| Accuracy | High — simulates real browser events | Low — dispatches single DOM events |
| Event sequence | Full sequence (pointerover, pointerdown, focus, keydown, input, keyup, click, …) | Single event only |
| Async | `await user.click()` | Synchronous |
| Typing | Fires keydown, keypress, input, keyup per character | Single `change` event |
| Setup required | `const user = userEvent.setup()` | None |
| Use case | All new tests | Only when you specifically need a single DOM event |

```typescript
import userEvent from '@testing-library/user-event';

// Always set up userEvent at the top of describe/test
const user = userEvent.setup();

it('handles user interactions correctly', async () => {
  render(<SearchInput onSearch={mockSearch} />);

  // Type characters one at a time (fires all keyboard events)
  await user.type(screen.getByRole('searchbox'), 'react testing');

  // Click a button
  await user.click(screen.getByRole('button', { name: /search/i }));

  // Clear a field
  await user.clear(screen.getByRole('searchbox'));

  // Select an option from a <select>
  await user.selectOptions(
    screen.getByRole('combobox', { name: /sort by/i }),
    'relevance',
  );

  // Check/uncheck a checkbox
  await user.click(screen.getByRole('checkbox', { name: /remember me/i }));

  // Type into a field then press Enter
  await user.type(screen.getByRole('textbox', { name: /email/i }), 'alice@example.com');
  await user.keyboard('{Enter}');
});
```

:::caution[Don't Forget to await userEvent]
`userEvent` v14+ is fully async — all interactions return promises. Forgetting `await` leads to tests that pass unexpectedly because assertions run before the interactions complete.
:::

## `waitFor` and `waitForElementToBeRemoved`

Use these utilities to wait for asynchronous DOM changes:

```typescript
import { render, screen, waitFor, waitForElementToBeRemoved } from '@testing-library/react';

it('displays user list after loading', async () => {
  render(<UserList />);

  // Wait for the loading spinner to disappear
  await waitForElementToBeRemoved(() => screen.queryByRole('progressbar'));

  // Now assert on the loaded content
  expect(screen.getByText('Alice Chen')).toBeInTheDocument();
  expect(screen.getByText('Bob Smith')).toBeInTheDocument();
});

it('shows error message when fetch fails', async () => {
  mockFetch.mockRejectedValueOnce(new Error('Network error'));

  render(<UserList />);

  // Wait for the error to appear
  await waitFor(() => {
    expect(screen.getByRole('alert')).toBeInTheDocument();
  });

  expect(screen.getByText(/failed to load/i)).toBeInTheDocument();
});

// waitFor with custom timeout
await waitFor(
  () => expect(screen.getByText('Data loaded')).toBeInTheDocument(),
  { timeout: 5000, interval: 100 },
);
```

## Testing Forms

```typescript
it('validates required fields on submit', async () => {
  const user = userEvent.setup();
  const onSubmit = jest.fn();

  render(<RegistrationForm onSubmit={onSubmit} />);

  // Submit without filling required fields
  await user.click(screen.getByRole('button', { name: /register/i }));

  // Validation errors should appear
  expect(screen.getByText(/name is required/i)).toBeInTheDocument();
  expect(screen.getByText(/email is required/i)).toBeInTheDocument();
  expect(onSubmit).not.toHaveBeenCalled();
});

it('calls onSubmit with form data when valid', async () => {
  const user = userEvent.setup();
  const onSubmit = jest.fn();

  render(<RegistrationForm onSubmit={onSubmit} />);

  await user.type(screen.getByLabelText(/full name/i), 'Alice Chen');
  await user.type(screen.getByLabelText(/email/i), 'alice@example.com');
  await user.type(screen.getByLabelText(/^password$/i), 'SecurePass1!');
  await user.type(screen.getByLabelText(/confirm password/i), 'SecurePass1!');
  await user.click(screen.getByLabelText(/accept terms/i));

  await user.click(screen.getByRole('button', { name: /register/i }));

  expect(onSubmit).toHaveBeenCalledWith({
    name: 'Alice Chen',
    email: 'alice@example.com',
    password: 'SecurePass1!',
    acceptedTerms: true,
  });
});
```

## Testing Hooks with `renderHook`

```typescript
import { renderHook, act } from '@testing-library/react';
import { useCounter } from './useCounter';

it('initializes with the provided value', () => {
  const { result } = renderHook(() => useCounter(10));
  expect(result.current.count).toBe(10);
});

it('increments the count', () => {
  const { result } = renderHook(() => useCounter(0));

  act(() => {
    result.current.increment();
  });

  expect(result.current.count).toBe(1);
});

it('resets to the initial value', () => {
  const { result } = renderHook(() => useCounter(5));

  act(() => {
    result.current.increment();
    result.current.increment();
    result.current.reset();
  });

  expect(result.current.count).toBe(5);
});
```

### Testing Async Hooks

```typescript
import { renderHook, waitFor } from '@testing-library/react';
import { useUserData } from './useUserData';

it('fetches and returns user data', async () => {
  mockFetch.mockResolvedValueOnce({
    ok: true,
    json: async () => ({ id: 1, name: 'Alice' }),
  } as Response);

  const { result } = renderHook(() => useUserData(1));

  // Initially loading
  expect(result.current.isLoading).toBe(true);
  expect(result.current.data).toBeNull();

  // Wait for data to load
  await waitFor(() => {
    expect(result.current.isLoading).toBe(false);
  });

  expect(result.current.data).toEqual({ id: 1, name: 'Alice' });
  expect(result.current.error).toBeNull();
});
```

## Testing Context

```typescript
// ThemeContext.tsx
export const ThemeContext = React.createContext<{ theme: string; toggle: () => void } | null>(null);

// Test helper: wrap component with context provider
const renderWithTheme = (
  ui: React.ReactElement,
  { theme = 'light' } = {},
) => {
  const toggle = jest.fn();
  return render(
    <ThemeContext.Provider value={{ theme, toggle }}>
      {ui}
    </ThemeContext.Provider>,
  );
};

it('applies dark theme class when theme is dark', () => {
  renderWithTheme(<ThemeToggle />, { theme: 'dark' });
  expect(screen.getByRole('button')).toHaveClass('toggle--dark');
});

it('calls toggle when button is clicked', async () => {
  const user = userEvent.setup();
  const mockToggle = jest.fn();

  render(
    <ThemeContext.Provider value={{ theme: 'light', toggle: mockToggle }}>
      <ThemeToggle />
    </ThemeContext.Provider>,
  );

  await user.click(screen.getByRole('button', { name: /toggle theme/i }));
  expect(mockToggle).toHaveBeenCalledTimes(1);
});
```

## Testing with React Router

```typescript
import { MemoryRouter, Routes, Route } from 'react-router-dom';

const renderWithRouter = (
  ui: React.ReactElement,
  { initialEntries = ['/'] } = {},
) => {
  return render(
    <MemoryRouter initialEntries={initialEntries}>
      {ui}
    </MemoryRouter>,
  );
};

it('renders the profile page for /profile/:id', async () => {
  renderWithRouter(
    <Routes>
      <Route path="/profile/:id" element={<ProfilePage />} />
    </Routes>,
    { initialEntries: ['/profile/42'] },
  );

  await waitFor(() => {
    expect(screen.getByRole('heading', { name: 'Alice Chen' })).toBeInTheDocument();
  });
});

it('navigates to login when accessing a protected route unauthenticated', () => {
  renderWithRouter(<ProtectedRoute component={<Dashboard />} />, {
    initialEntries: ['/dashboard'],
  });

  expect(screen.getByRole('heading', { name: /sign in/i })).toBeInTheDocument();
});
```

## The `act()` Wrapper

React requires that state updates happening outside of its event system are wrapped in `act()`. RTL wraps its interactions automatically, but sometimes you need to wrap direct imperative calls.

```typescript
import { act } from '@testing-library/react';

it('updates when an external store changes', () => {
  render(<Counter store={externalStore} />);

  // External state change must be wrapped in act()
  act(() => {
    externalStore.dispatch({ type: 'INCREMENT' });
  });

  expect(screen.getByText('Count: 1')).toBeInTheDocument();
});

// Async act for async state updates
it('fetches data on mount', async () => {
  await act(async () => {
    render(<DataLoader />);
  });

  expect(screen.getByText('Data loaded')).toBeInTheDocument();
});
```

:::tip[When You See the act() Warning]
If Jest warns: _"An update to Component inside a test was not wrapped in act(…)"_, it usually means an async state update happened after your test assertions ran. Fix by using `waitFor()` to wait for the update, or wrapping the trigger in `await act(async () => { ... })`.
:::

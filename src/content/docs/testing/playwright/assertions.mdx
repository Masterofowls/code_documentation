Playwright's `expect` API provides **web-first assertions** that automatically retry until the condition is met or the timeout expires. This is the key difference from traditional assertions — you never need explicit waits before an assertion.

## Web-First Assertions

Traditional assertions fail immediately if the condition isn't met. Web-first assertions poll the DOM repeatedly until success or timeout:

```typescript
// ❌ Brittle — fails if DOM hasn't updated yet
const text = await page.getByRole('status').textContent();
expect(text).toBe('Saved'); // may run before the save completes

// ✅ Web-first — retries until 'Saved' appears or timeout
await expect(page.getByRole('status')).toHaveText('Saved');
```

Configure the default assertion timeout in `playwright.config.ts`:

```typescript
export default defineConfig({
  expect: {
    timeout: 5000, // 5 seconds (default)
  },
});
```

Override per-assertion:

```typescript
await expect(page.getByRole('progressbar')).toBeHidden({ timeout: 30000 });
```

## Visibility Assertions

```typescript
// Element is visible in the viewport
await expect(page.getByRole('dialog')).toBeVisible();
await expect(page.getByText('Welcome back!')).toBeVisible();

// Element is hidden (or not in DOM)
await expect(page.getByRole('dialog')).toBeHidden();
await expect(page.getByTestId('loading-spinner')).toBeHidden();

// Element is attached to the DOM (may be hidden)
await expect(page.getByTestId('hidden-section')).toBeAttached();
await expect(page.getByTestId('removed-section')).not.toBeAttached();

// Wait for element to be visible then assert
await expect(page.getByRole('alert')).toBeVisible({ timeout: 10000 });
```

## State Assertions

```typescript
// Enabled / Disabled
await expect(page.getByRole('button', { name: 'Submit' })).toBeEnabled();
await expect(page.getByRole('button', { name: 'Submit' })).toBeDisabled();

// Checked / Unchecked (checkbox, radio)
await expect(page.getByLabel('Remember me')).toBeChecked();
await expect(page.getByLabel('Subscribe')).not.toBeChecked();
await expect(page.getByLabel('Monthly')).toBeChecked({ checked: true });

// Editable (not readonly, not disabled)
await expect(page.getByLabel('Name')).toBeEditable();
await expect(page.getByLabel('Account ID')).not.toBeEditable();

// Empty (no value, no text content)
await expect(page.getByLabel('Search')).toBeEmpty();
await expect(page.getByRole('listbox')).not.toBeEmpty();

// Focused
await expect(page.getByLabel('Email')).toBeFocused();
```

## Text Assertions

### toHaveText

Checks the text content of an element:

```typescript
// Exact match (default when string)
await expect(page.getByRole('heading')).toHaveText('Dashboard');

// Substring match with regex
await expect(page.getByRole('status')).toHaveText(/saved/i);

// Partial match (string — normalized whitespace)
await expect(page.getByRole('paragraph')).toHaveText('Hello, Alice');

// Array of texts (for multiple elements)
const items = page.getByRole('listitem');
await expect(items).toHaveText(['Apple', 'Banana', 'Cherry']);
await expect(items).toHaveText([/apple/i, /banana/i, /cherry/i]);
```

### toContainText

Checks that the element contains the given text (substring):

```typescript
await expect(page.getByRole('article')).toContainText('Alice');
await expect(page.getByRole('article')).toContainText(/posted \d+ days ago/);

// Multiple elements — each must contain corresponding text
await expect(page.getByRole('listitem')).toContainText(['Apple', 'Cherry']);
```

### toHaveValue

Checks the value of an `<input>`, `<textarea>`, or `<select>`:

```typescript
await expect(page.getByLabel('Email')).toHaveValue('alice@example.com');
await expect(page.getByLabel('Bio')).toHaveValue(/experienced developer/);
await expect(page.getByLabel('Country')).toHaveValue('US');

// Empty input
await expect(page.getByLabel('Search')).toHaveValue('');
```

### toHaveValues

Checks the selected values of a multi-select:

```typescript
await expect(page.getByLabel('Skills')).toHaveValues([
  'typescript',
  'react',
  'testing',
]);
```

## Attribute and Class Assertions

### toHaveAttribute

```typescript
// Check an attribute value
await expect(page.getByRole('link', { name: 'Home' })).toHaveAttribute(
  'href',
  '/',
);
await expect(page.getByRole('img')).toHaveAttribute('alt', 'Profile photo');
await expect(page.getByRole('textbox')).toHaveAttribute('type', 'email');

// Regex match
await expect(page.getByRole('link')).toHaveAttribute('href', /\/products\/\d+/);

// Attribute exists (any value)
await expect(page.getByRole('button')).toHaveAttribute('aria-expanded');

// Attribute does not exist
await expect(page.getByRole('button')).not.toHaveAttribute('disabled');
```

### toHaveClass

```typescript
// Has a specific class
await expect(page.getByRole('button')).toHaveClass('btn-primary');

// Has multiple classes (checks for all)
await expect(page.getByRole('button')).toHaveClass('btn btn-primary active');

// Regex match
await expect(page.getByRole('alert')).toHaveClass(/error/);

// Exact class attribute match
await expect(page.getByRole('button')).toHaveClass('btn btn-primary', {
  exact: true,
});
```

## Count Assertion

```typescript
// Check the number of matching elements
await expect(page.getByRole('listitem')).toHaveCount(5);
await expect(page.getByRole('row')).toHaveCount(11); // 10 data rows + 1 header

// At least one element
await expect(page.getByRole('option')).not.toHaveCount(0);
```

## URL and Title Assertions

```typescript
// Check the page URL
await expect(page).toHaveURL('/dashboard');
await expect(page).toHaveURL('https://myapp.example.com/dashboard');
await expect(page).toHaveURL(/\/users\/\d+/);

// Check the page title
await expect(page).toHaveTitle('Dashboard — My App');
await expect(page).toHaveTitle(/Dashboard/);
```

## Viewport Visibility

```typescript
// Check element is within the visible viewport
await expect(page.getByRole('banner')).toBeInViewport();
await expect(page.getByRole('contentinfo')).not.toBeInViewport();

// With ratio (% of element that must be in viewport)
await expect(page.getByRole('img')).toBeInViewport({ ratio: 0.5 }); // at least 50% visible
```

## Visual Screenshot Comparison

Playwright's built-in screenshot assertion compares pixel-by-pixel against stored reference screenshots:

```typescript
// Full page screenshot
await expect(page).toHaveScreenshot('home-page.png');

// Element screenshot
await expect(page.getByRole('article')).toHaveScreenshot('product-card.png');

// With tolerance options
await expect(page).toHaveScreenshot('dashboard.png', {
  maxDiffPixels: 200, // allow up to 200 different pixels
  threshold: 0.2, // 20% color difference tolerance per pixel
  maxDiffPixelRatio: 0.01, // allow 1% of pixels to differ
});

// Mask dynamic regions (timestamps, ads, avatars)
await expect(page).toHaveScreenshot('profile.png', {
  mask: [
    page.getByTestId('last-updated'),
    page.getByRole('img', { name: 'Avatar' }),
  ],
});

// Clip to a specific area
await expect(page).toHaveScreenshot('header.png', {
  clip: { x: 0, y: 0, width: 1280, height: 80 },
});

// Full page (scrolls and stitches)
await expect(page).toHaveScreenshot('full-page.png', { fullPage: true });
```

Update reference screenshots when the UI intentionally changes:

```bash
npx playwright test --update-snapshots
```

:::tip[Snapshot Storage]
Reference screenshots are stored in `tests/__snapshots__/` (or alongside the test file). Commit them to source control. Review screenshot diffs in CI the same way you review code diffs — they represent visual changes to your UI.
:::

## API Response Assertions

```typescript
import { expect, request } from '@playwright/test';

// In an API test using the request fixture
test('API returns correct user', async ({ request }) => {
  const response = await request.get('/api/users/1');

  await expect(response).toBeOK(); // status 200-299
  expect(response.status()).toBe(200);
  expect(response.headers()['content-type']).toContain('application/json');

  const body = await response.json();
  expect(body).toMatchObject({ id: 1, name: 'Alice' });
});

// Combined page + API assertion
test('creating a user reflects in the UI', async ({ page, request }) => {
  const response = await request.post('/api/users', {
    data: { name: 'Bob', email: 'bob@example.com' },
  });
  await expect(response).toBeOK();

  await page.goto('/admin/users');
  await expect(page.getByText('Bob')).toBeVisible();
});
```

## Soft Assertions — Non-Blocking Failures

Soft assertions continue test execution even when the assertion fails. All soft assertion failures are reported together at the end:

```typescript
test('product page has all required elements', async ({ page }) => {
  await page.goto('/products/42');

  // These won't stop the test if they fail
  await expect.soft(page.getByRole('heading')).toBeVisible();
  await expect.soft(page.getByRole('img', { name: /product/i })).toBeVisible();
  await expect.soft(page.getByText(/\$\d+\.\d{2}/)).toBeVisible();
  await expect
    .soft(page.getByRole('button', { name: 'Add to cart' }))
    .toBeEnabled();
  await expect
    .soft(page.getByRole('button', { name: 'Add to cart' }))
    .toBeVisible();

  // This IS a hard assertion — stops the test if navigation doesn't work
  await page.getByRole('button', { name: 'Add to cart' }).click();
  await expect(page).toHaveURL('/cart');
});
```

:::tip[When to Use Soft Assertions]
Use `expect.soft` when checking multiple independent properties of the same page where you want a full picture of what's failing. Use hard `expect` for assertions that are prerequisites for subsequent steps — no point continuing if the button isn't there.
:::

## Custom Expect Messages

Add a description to any assertion for clearer failure messages:

```typescript
await expect(
  page.getByRole('button', { name: 'Checkout' }),
  'Checkout button should be visible in the cart page header',
).toBeVisible();

await expect(
  page.getByLabel('Total'),
  `Cart total should show $${expectedTotal}`,
).toHaveValue(`$${expectedTotal}`);
```

When the assertion fails, the custom message appears in the error output:

```
Error: Checkout button should be visible in the cart page header

Call log:
  - expect.toBeVisible with timeout 5000ms
  - waiting for getByRole('button', { name: 'Checkout' })
```

## Polling with expect.poll

For conditions that aren't tied to a DOM element, use `expect.poll`:

```typescript
// Poll a function until it returns a truthy value
await expect
  .poll(
    async () => {
      const response = await page.request.get('/api/job/status');
      const body = await response.json();
      return body.status;
    },
    {
      message: 'Job should complete within 60 seconds',
      timeout: 60000,
      intervals: [1000, 2000, 5000], // custom polling intervals
    },
  )
  .toBe('completed');
```

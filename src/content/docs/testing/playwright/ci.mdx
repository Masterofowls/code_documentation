Running Playwright in CI requires installing browser binaries, configuring parallelism, enabling retries, and uploading reports and traces as artifacts. This guide covers a production-ready CI setup.

## GitHub Actions — Complete Workflow

```yaml
# .github/workflows/playwright.yml
name: Playwright Tests

on:
  push:
    branches: [main, develop]
  pull_request:
    branches: [main]

jobs:
  test:
    timeout-minutes: 60
    runs-on: ubuntu-latest

    strategy:
      fail-fast: false
      matrix:
        shard: [1, 2, 3, 4] # run 4 shards in parallel

    steps:
      - uses: actions/checkout@v4

      - uses: actions/setup-node@v4
        with:
          node-version: '20'
          cache: 'npm'

      - name: Install dependencies
        run: npm ci

      # Cache Playwright browser binaries
      - name: Get Playwright version
        id: playwright-version
        run: echo "version=$(npx playwright --version | cut -d ' ' -f 2)" >> $GITHUB_OUTPUT

      - name: Cache Playwright browsers
        uses: actions/cache@v4
        id: playwright-cache
        with:
          path: ~/.cache/ms-playwright
          key: playwright-browsers-${{ steps.playwright-version.outputs.version }}-${{ runner.os }}

      - name: Install Playwright browsers
        if: steps.playwright-cache.outputs.cache-hit != 'true'
        run: npx playwright install --with-deps

      - name: Install OS dependencies (if cache hit)
        if: steps.playwright-cache.outputs.cache-hit == 'true'
        run: npx playwright install-deps

      - name: Run Playwright tests (shard ${{ matrix.shard }}/4)
        run: npx playwright test --shard=${{ matrix.shard }}/4
        env:
          BASE_URL: ${{ vars.STAGING_URL || 'http://localhost:3000' }}
          TEST_USER_EMAIL: ${{ secrets.TEST_USER_EMAIL }}
          TEST_USER_PASSWORD: ${{ secrets.TEST_USER_PASSWORD }}
          CI: true

      # Upload test results (even on failure)
      - name: Upload blob report
        if: ${{ !cancelled() }}
        uses: actions/upload-artifact@v4
        with:
          name: blob-report-${{ matrix.shard }}
          path: blob-report/
          retention-days: 7

  # Merge shard reports into one HTML report
  merge-reports:
    if: ${{ !cancelled() }}
    needs: test
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: '20'
          cache: 'npm'
      - run: npm ci

      - name: Download blob reports from all shards
        uses: actions/download-artifact@v4
        with:
          pattern: blob-report-*
          path: all-blob-reports
          merge-multiple: true

      - name: Merge reports
        run: npx playwright merge-reports --reporter html ./all-blob-reports

      - name: Upload HTML report
        uses: actions/upload-artifact@v4
        with:
          name: playwright-html-report
          path: playwright-report/
          retention-days: 30
```

## playwright.config.ts — CI Settings

```typescript
import { defineConfig, devices } from '@playwright/test';
import os from 'os';

export default defineConfig({
  testDir: './tests',
  timeout: 30 * 1000,
  fullyParallel: true,

  // Fail build if test.only is committed
  forbidOnly: !!process.env.CI,

  // Retry on CI — surface flaky tests
  retries: process.env.CI ? 2 : 0,

  // Parallelize across workers
  workers: process.env.CI ? '50%' : undefined,

  reporter: process.env.CI
    ? [
        ['blob'], // for shard merging
        ['github'], // GitHub PR annotations
        ['junit', { outputFile: 'test-results/junit.xml' }],
      ]
    : [['html', { open: 'never' }], ['list']],

  use: {
    baseURL: process.env.BASE_URL ?? 'http://localhost:3000',
    trace: 'on-first-retry',
    video: 'on-first-retry',
    screenshot: 'only-on-failure',
    headless: true,
  },

  projects: [
    { name: 'chromium', use: { ...devices['Desktop Chrome'] } },
    { name: 'firefox', use: { ...devices['Desktop Firefox'] } },
    { name: 'webkit', use: { ...devices['Desktop Safari'] } },
  ],
});
```

## Retries in CI

Retries re-run failed tests automatically — helping distinguish actual failures from transient flakiness:

```typescript
// playwright.config.ts
retries: process.env.CI ? 2 : 0,
```

With `retries: 2`:

1. Test runs for the first time → if it fails:
2. Test retries once (`attempt 1 of 2`) → if it still fails:
3. Test retries again (`attempt 2 of 2`) → marked as failed

A test that passes on any retry is marked as **flaky** in the HTML report — still a pass, but flagged for investigation.

:::tip[Retries Don't Fix Bugs]
Retries hide flakiness — they don't fix it. Use the flaky test report to identify timing issues, race conditions, or external dependencies that need to be made deterministic. The goal is zero flaky tests, not retrying them away.
:::

## Sharding — Parallel Execution Across Machines

Sharding splits your test suite across multiple CI jobs, dramatically reducing total wall-clock time:

```bash
# Job 1: runs tests 1–25% of total
npx playwright test --shard=1/4

# Job 2: runs tests 26–50%
npx playwright test --shard=2/4

# Job 3: runs tests 51–75%
npx playwright test --shard=3/4

# Job 4: runs tests 76–100%
npx playwright test --shard=4/4
```

In GitHub Actions, use a matrix strategy (see the full workflow above). Playwright automatically distributes tests evenly by file.

### Choosing Shard Count

| Test suite size | Recommended shards | Expected CI time |
| --------------- | ------------------ | ---------------- |
| < 50 tests      | 1 (no sharding)    | ~2–5 min         |
| 50–200 tests    | 2–4 shards         | ~3–8 min         |
| 200–500 tests   | 4–8 shards         | ~5–10 min        |
| 500+ tests      | 8–16 shards        | ~8–15 min        |

## Workers — Parallelism Within a Machine

Within a single CI job, `workers` controls how many browser contexts run in parallel:

```typescript
workers: process.env.CI ? '50%' : undefined,
// '50%' = use half the available CPUs
// On a 4-core machine: 2 workers
// On an 8-core machine: 4 workers
```

```typescript
// Fine-tuned options:
workers: 4,              // exactly 4 workers
workers: '50%',          // 50% of CPUs
workers: os.cpus().length - 1,  // all cores minus 1
workers: 1,              // sequential — useful for debugging or shared state
```

:::caution[Workers and Shared State]
When using multiple workers, tests run in parallel and cannot share in-process state. Use the `storageState` pattern and API-based setup/teardown for anything that needs to be shared.
:::

## HTML Report Artifacts

Upload the HTML report as a CI artifact so it's accessible after the run:

```yaml
- name: Upload Playwright HTML report
  uses: actions/upload-artifact@v4
  if: ${{ !cancelled() }}
  with:
    name: playwright-report-${{ github.run_id }}
    path: playwright-report/
    retention-days: 30
```

To view the report, download the artifact and run:

```bash
npx playwright show-report playwright-report/
```

Or host it temporarily:

```bash
npx http-server playwright-report/ -p 9323
```

## JUnit XML Reporter for CI Feedback

JUnit XML is supported by most CI platforms for inline test results display:

```typescript
reporter: [
  ['junit', {
    outputFile: 'test-results/junit.xml',
    stripANSIControlSequences: true,
  }],
],
```

In GitHub Actions, publish the JUnit report:

```yaml
- name: Publish test results
  uses: dorny/test-reporter@v1
  if: ${{ !cancelled() }}
  with:
    name: Playwright Results
    path: test-results/junit.xml
    reporter: java-junit
    fail-on-error: false
```

## Docker Image

Microsoft provides an official Docker image with all browsers and dependencies pre-installed:

```dockerfile
# Use the official Playwright Docker image
FROM mcr.microsoft.com/playwright:v1.44.0-jammy

WORKDIR /app
COPY package*.json ./
RUN npm ci
COPY . .
CMD ["npx", "playwright", "test"]
```

In GitHub Actions:

```yaml
jobs:
  test:
    runs-on: ubuntu-latest
    container:
      image: mcr.microsoft.com/playwright:v1.44.0-jammy
    steps:
      - uses: actions/checkout@v4
      - run: npm ci
      - run: npx playwright test
        env:
          HOME: /root # required in Docker
```

Available image variants:

| Tag                   | OS               | Size   |
| --------------------- | ---------------- | ------ |
| `v1.44.0-jammy`       | Ubuntu 22.04     | ~1.9GB |
| `v1.44.0-focal`       | Ubuntu 20.04     | ~1.8GB |
| `v1.44.0-jammy-arm64` | Ubuntu 22.04 ARM | ~1.7GB |

## Environment Variables and Secrets

```typescript
// playwright.config.ts — read from environment
export default defineConfig({
  use: {
    baseURL: process.env.BASE_URL ?? 'http://localhost:3000',
    extraHTTPHeaders: {
      'x-api-key': process.env.API_KEY ?? '',
    },
  },
});
```

```yaml
# GitHub Actions — pass secrets to tests
- name: Run Playwright tests
  run: npx playwright test
  env:
    BASE_URL: ${{ vars.STAGING_URL }}
    TEST_USER_EMAIL: ${{ secrets.TEST_USER_EMAIL }}
    TEST_USER_PASSWORD: ${{ secrets.TEST_USER_PASSWORD }}
    API_KEY: ${{ secrets.TEST_API_KEY }}
```

:::caution[Never Hardcode Secrets]
Never commit real credentials, API keys, or passwords to your test files. Always use environment variables sourced from your CI secrets manager. Use dedicated test accounts with minimal permissions.
:::

## Running Specific Projects or Tags in CI

```bash
# Run only Chromium tests
npx playwright test --project=chromium

# Run only tests with a specific tag
npx playwright test --grep @smoke

# Combine: smoke tests on all browsers
npx playwright test --grep @smoke

# Exclude slow tests
npx playwright test --grep-invert @slow
```

Tag tests in the test file:

```typescript
test('critical checkout flow @smoke @critical', async ({ page }) => {
  // ...
});

test.describe('@smoke', () => {
  test('homepage loads', async ({ page }) => {
    /* ... */
  });
  test('login works', async ({ page }) => {
    /* ... */
  });
});
```

```yaml
# Run only smoke tests on PR, full suite on main
- name: Run tests
  run: |
    if [ "${{ github.event_name }}" == "pull_request" ]; then
      npx playwright test --grep @smoke
    else
      npx playwright test
    fi
```

## Flaky Test Detection

Playwright's `--repeat-each` flag runs each test multiple times to detect flakiness:

```bash
# Run each test 3 times to detect flakes
npx playwright test --repeat-each=3

# Run only failed tests multiple times
npx playwright test --only-failures --repeat-each=5
```

Identify historically flaky tests from the HTML report's "Flaky tests" section. For tests marked as flaky (passed on retry), investigate:

1. **Timing issues** — add `await expect(element).toBeVisible()` before interacting
2. **Race conditions** — wait for specific UI state, not arbitrary `waitForTimeout`
3. **Test isolation** — ensure tests don't depend on execution order or shared state
4. **External dependencies** — mock third-party APIs that may be slow or down

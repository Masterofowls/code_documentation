Running Playwright reliably in CI requires the right configuration for parallelism, retries, artifact collection, and browser installation. This guide covers everything from basic GitHub Actions setup to advanced sharding across multiple runners.

## GitHub Actions — Complete Workflow

```yaml
# .github/workflows/playwright.yml
name: Playwright Tests

on:
  push:
    branches: [main, develop]
  pull_request:
    branches: [main, develop]

jobs:
  playwright:
    name: 'Playwright Tests'
    runs-on: ubuntu-latest
    timeout-minutes: 60

    strategy:
      fail-fast: false
      matrix:
        shard: [1, 2, 3, 4]

    steps:
      - uses: actions/checkout@v4

      - uses: actions/setup-node@v4
        with:
          node-version: 20
          cache: 'npm'

      - name: Install dependencies
        run: npm ci

      - name: Cache Playwright browsers
        uses: actions/cache@v4
        id: playwright-cache
        with:
          path: ~/.cache/ms-playwright
          key: playwright-${{ hashFiles('**/package-lock.json') }}
          restore-keys: |
            playwright-

      - name: Install Playwright browsers
        if: steps.playwright-cache.outputs.cache-hit != 'true'
        run: npx playwright install --with-deps

      - name: Install OS dependencies (if browsers were cached)
        if: steps.playwright-cache.outputs.cache-hit == 'true'
        run: npx playwright install-deps

      - name: Run Playwright tests (shard ${{ matrix.shard }} of 4)
        run: npx playwright test --shard=${{ matrix.shard }}/4
        env:
          CI: 'true'
          BASE_URL: ${{ vars.STAGING_URL }}
          AUTH_TOKEN: ${{ secrets.TEST_AUTH_TOKEN }}

      - name: Upload test results on failure
        if: failure()
        uses: actions/upload-artifact@v4
        with:
          name: playwright-results-shard-${{ matrix.shard }}
          path: |
            test-results/
            playwright-report/
          retention-days: 14

  merge-reports:
    name: 'Merge Playwright Reports'
    needs: playwright
    runs-on: ubuntu-latest
    if: always()

    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: 20
          cache: 'npm'
      - run: npm ci

      - name: Download all shard artifacts
        uses: actions/download-artifact@v4
        with:
          pattern: playwright-results-shard-*
          merge-multiple: true
          path: playwright-results

      - name: Merge reports
        run: npx playwright merge-reports --reporter html,junit ./playwright-results

      - name: Upload merged HTML report
        uses: actions/upload-artifact@v4
        with:
          name: playwright-report-merged
          path: playwright-report/
          retention-days: 30
```

## Retries in CI

Playwright can automatically retry flaky tests before marking them as failed:

```typescript
// playwright.config.ts
export default defineConfig({
  // Retry failed tests up to 2 times in CI
  retries: process.env.CI ? 2 : 0,

  // Mark tests as "flaky" instead of "failed" if they pass on retry
  // (reported in the HTML report for visibility)
});
```

```bash
# Retry from command line
npx playwright test --retries=2
```

:::tip[Retries Mask Flakiness]
Retries are a safety net, not a cure. Monitor the `playwright-report` for tests marked "flaky" — they indicate timing issues, race conditions, or missing await statements. Fix the root cause rather than increasing retries indefinitely.
:::

## Sharding — Parallelism Across Machines

Sharding distributes your test suite across multiple CI runners, reducing total execution time:

```bash
# Run shard 1 of 4 on runner 1
npx playwright test --shard=1/4

# Run shard 2 of 4 on runner 2
npx playwright test --shard=2/4

# ... and so on for shards 3 and 4

# Merge reports from all shards
npx playwright merge-reports --reporter html ./all-blob-reports
```

Playwright distributes test files evenly across shards. With 40 test files and 4 shards, each runner gets 10 files.

For GitHub Actions with a matrix strategy:

```yaml
strategy:
  matrix:
    shard: [1, 2, 3, 4]
steps:
  - run: npx playwright test --shard=${{ matrix.shard }}/4
```

## Worker Parallelism Within a Machine

Within a single runner, Playwright uses worker processes for parallelism:

```typescript
// playwright.config.ts
export default defineConfig({
  // Automatically determined (usually CPUs / 2)
  workers: undefined,

  // In CI with limited CPUs
  workers: process.env.CI ? 2 : undefined,

  // Half of available CPUs (percentage string)
  workers: '50%',

  // Fully serial execution (safest for shared state)
  workers: 1,
});
```

```bash
# Set workers from CLI
npx playwright test --workers=4
npx playwright test --workers=1  # equivalent to --serial (sort of)
```

:::caution[Parallel Tests Must Be Independent]
Parallel workers run tests concurrently. Tests that share a database, modify global state, or depend on specific port availability will fail randomly. Each test should create its own data and clean up after itself, or use Playwright's storage state for authentication.
:::

## Reporters for CI

Configure reporters optimized for CI output:

```typescript
// playwright.config.ts
export default defineConfig({
  reporter: process.env.CI
    ? [
        // Minimal terminal output (one dot per test)
        ['dot'],
        // HTML report — saved to playwright-report/
        ['html', { open: 'never', outputFolder: 'playwright-report' }],
        // JUnit XML — consumed by GitHub Actions, Jenkins, GitLab CI
        ['junit', { outputFile: 'test-results/junit.xml' }],
        // Blob report — for merging shard results
        ['blob', { outputDir: 'blob-report' }],
      ]
    : [
        // Beautiful list output for local development
        ['list'],
        // HTML report that auto-opens on failure
        ['html', { open: 'on-failure' }],
      ],
});
```

## Docker — Official Playwright Image

Use the official Playwright Docker image for a consistent, pre-installed environment:

```dockerfile
# Dockerfile.playwright
FROM mcr.microsoft.com/playwright:v1.48.0-jammy

WORKDIR /app

COPY package*.json ./
RUN npm ci

COPY . .

CMD ["npx", "playwright", "test"]
```

```yaml
# .github/workflows/playwright.yml
jobs:
  playwright:
    runs-on: ubuntu-latest
    container:
      image: mcr.microsoft.com/playwright:v1.48.0-jammy
    steps:
      - uses: actions/checkout@v4
      - run: npm ci
      # No browser install needed — they're pre-installed in the image
      - run: npx playwright test
```

Available image variants:
- `mcr.microsoft.com/playwright:v1.48.0-jammy` — Ubuntu 22.04 LTS
- `mcr.microsoft.com/playwright:v1.48.0-focal` — Ubuntu 20.04 LTS
- `mcr.microsoft.com/playwright:v1.48.0-jammy-arm64` — ARM64

## Environment Variables

```typescript
// playwright.config.ts
export default defineConfig({
  use: {
    baseURL: process.env.BASE_URL ?? 'http://localhost:3000',
    extraHTTPHeaders: {
      'Authorization': `Bearer ${process.env.API_TOKEN ?? ''}`,
    },
  },
});
```

```yaml
# GitHub Actions — secrets and variables
- name: Run Playwright tests
  run: npx playwright test
  env:
    BASE_URL: ${{ vars.STAGING_BASE_URL }}          # Repository variable (public)
    API_TOKEN: ${{ secrets.TEST_API_TOKEN }}         # Repository secret (encrypted)
    DATABASE_URL: ${{ secrets.TEST_DATABASE_URL }}
    CI: 'true'
```

:::caution[Never hardcode secrets in test files]
Test files in version control must never contain API keys, passwords, or tokens. Use environment variables accessed via `process.env`. In GitHub Actions, store sensitive values in Repository Secrets (Settings → Secrets and variables → Actions).
:::

## Caching Browser Binaries

Browser downloads are large (300-500 MB total) and slow. Cache them between CI runs:

```yaml
- name: Get Playwright version
  id: playwright-version
  run: echo "version=$(node -e "console.log(require('./node_modules/@playwright/test/package.json').version)")" >> $GITHUB_OUTPUT

- name: Cache Playwright browsers
  uses: actions/cache@v4
  id: playwright-cache
  with:
    path: ~/.cache/ms-playwright
    key: playwright-browsers-${{ steps.playwright-version.outputs.version }}

- name: Install Playwright browsers
  if: steps.playwright-cache.outputs.cache-hit != 'true'
  run: npx playwright install --with-deps chromium firefox webkit

- name: Install system dependencies (if using cached browsers)
  if: steps.playwright-cache.outputs.cache-hit == 'true'
  run: npx playwright install-deps
```

:::tip[Cache Key Strategy]
Use the Playwright version as the cache key. Playwright downloads new browser binaries with each version release, so version-keyed caches are automatically invalidated when you update Playwright.
:::

## Running Specific Projects and Tags

```bash
# Run only one browser in CI
npx playwright test --project=chromium

# Run tests with a specific tag
npx playwright test --grep "@smoke"
npx playwright test --grep "@critical"

# Exclude slow tests from the main pipeline
npx playwright test --grep-invert "@slow"
```

Tag tests in your code:

```typescript
test('critical checkout flow @smoke @critical', async ({ page }) => {
  // ...
});

test.describe('@slow', () => {
  test('loads thousands of records', async ({ page }) => {
    // ...
  });
});
```

```yaml
# Run smoke tests on every push, full suite nightly
jobs:
  smoke:
    runs-on: ubuntu-latest
    steps:
      - run: npx playwright test --grep "@smoke" --project=chromium

  full-suite:
    if: github.event_name == 'schedule'
    runs-on: ubuntu-latest
    steps:
      - run: npx playwright test --shard=${{ matrix.shard }}/4
```

## Flaky Test Detection

```bash
# Run each test multiple times to detect flakiness
npx playwright test --repeat-each=3

# Use built-in flakiness report
npx playwright test --reporter=html
# Open the report and filter by "Flaky" status
```

Configure flaky test handling in `playwright.config.ts`:

```typescript
export default defineConfig({
  retries: 2,         // Retry up to 2 times
  // A test is marked "flaky" if it fails initially but passes on retry
  // Flaky tests appear in the HTML report for investigation
});
```

## Full CI Configuration Example

```typescript
// playwright.config.ts — production-ready CI configuration
import { defineConfig, devices } from '@playwright/test';

const isCI = !!process.env.CI;

export default defineConfig({
  testDir: './tests',
  timeout: 30_000,
  expect: { timeout: 5_000 },
  retries: isCI ? 2 : 0,
  workers: isCI ? 2 : undefined,
  fullyParallel: true,

  reporter: isCI
    ? [
        ['dot'],
        ['html', { open: 'never' }],
        ['junit', { outputFile: 'test-results/junit.xml' }],
        ['blob', { outputDir: 'blob-report' }],
      ]
    : [['list'], ['html', { open: 'on-failure' }]],

  use: {
    baseURL: process.env.BASE_URL ?? 'http://localhost:3000',
    trace: isCI ? 'on-first-retry' : 'off',
    screenshot: isCI ? 'only-on-failure' : 'off',
    video: isCI ? 'on-first-retry' : 'off',
    headless: isCI,
  },

  projects: isCI
    ? [
        { name: 'chromium', use: { ...devices['Desktop Chrome'] } },
        { name: 'firefox', use: { ...devices['Desktop Firefox'] } },
        { name: 'webkit', use: { ...devices['Desktop Safari'] } },
      ]
    : [
        { name: 'chromium', use: { ...devices['Desktop Chrome'] } },
      ],

  webServer: {
    command: 'npm run start',
    url: 'http://localhost:3000',
    reuseExistingServer: !isCI,
    timeout: 120_000,
  },
});
```

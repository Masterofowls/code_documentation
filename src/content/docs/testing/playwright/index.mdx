Playwright is a cross-browser end-to-end testing framework developed by Microsoft. It provides a single API to automate Chromium, Firefox, and WebKit — covering the three major browser engines — with built-in auto-waiting, trace recording, network interception, and visual comparison.

## What is Playwright?

Playwright controls real browsers programmatically. Unlike jsdom-based tools, Playwright tests run against the actual rendering engine, executing CSS, JavaScript, and layout exactly as users experience them.

```typescript
import { test, expect } from '@playwright/test';

test('user can log in and see their dashboard', async ({ page }) => {
  await page.goto('https://myapp.example.com/login');

  await page.getByLabel('Email').fill('alice@example.com');
  await page.getByLabel('Password').fill('supersecret');
  await page.getByRole('button', { name: 'Sign in' }).click();

  await expect(page).toHaveURL('/dashboard');
  await expect(
    page.getByRole('heading', { name: 'Welcome, Alice' }),
  ).toBeVisible();
});
```

Playwright is not a unit testing tool — it is designed for **integration**, **end-to-end**, **visual regression**, and **API testing** scenarios.

## Cross-Browser Testing

Playwright tests run against all three major browser engines out of the box:

| Browser  | Engine                  | Platform             |
| -------- | ----------------------- | -------------------- |
| Chromium | Blink (V8)              | Chrome, Edge, Brave  |
| Firefox  | Gecko (SpiderMonkey)    | Firefox              |
| WebKit   | WebKit (JavaScriptCore) | Safari, iOS browsers |

```typescript
// playwright.config.ts
import { defineConfig, devices } from '@playwright/test';

export default defineConfig({
  projects: [
    { name: 'chromium', use: { ...devices['Desktop Chrome'] } },
    { name: 'firefox', use: { ...devices['Desktop Firefox'] } },
    { name: 'webkit', use: { ...devices['Desktop Safari'] } },
    { name: 'Mobile Chrome', use: { ...devices['Pixel 5'] } },
    { name: 'Mobile Safari', use: { ...devices['iPhone 13'] } },
  ],
});
```

## Playwright vs Cypress vs Selenium

| Feature                | **Playwright**                           | **Cypress**                        | **Selenium**                 |
| ---------------------- | ---------------------------------------- | ---------------------------------- | ---------------------------- |
| Browser engines        | Chromium, Firefox, WebKit                | Chromium-based only (Firefox beta) | All browsers via WebDriver   |
| Language support       | TypeScript, JavaScript, Python, Java, C# | JavaScript / TypeScript            | All major languages          |
| Architecture           | Out-of-process (CDP/WebSocket)           | In-process (runs in browser)       | WebDriver protocol           |
| Auto-waiting           | Yes (smart, auto-retry)                  | Yes                                | No (explicit waits required) |
| Network interception   | Yes, first-class                         | Yes                                | Limited (proxy-based)        |
| iFrame support         | Yes                                      | Limited                            | Yes                          |
| Multiple tabs/windows  | Yes                                      | No (single tab)                    | Yes                          |
| File uploads/downloads | Yes                                      | Yes                                | Yes                          |
| Visual comparisons     | Built-in                                 | Plugin (cypress-image-snapshot)    | Plugin                       |
| Component testing      | Yes (experimental)                       | Yes (mature)                       | No                           |
| API testing            | Yes (built-in `request` fixture)         | Yes (cy.request)                   | No                           |
| Trace/debugging        | Trace Viewer, UI mode                    | Cypress Studio                     | No built-in                  |
| CI Docker image        | Official (mcr.microsoft.com/playwright)  | Official                           | WebDriver in path            |
| Setup complexity       | Low                                      | Low                                | High                         |
| Speed                  | Fast                                     | Moderate                           | Slow                         |
| Cost                   | Free, open source                        | Free + paid Cloud                  | Free + infrastructure cost   |

:::tip[When to Choose Playwright]
Choose Playwright when you need: cross-browser testing (including Safari/WebKit), multiple tabs or windows, iFrame testing, robust API testing, or when you want a single tool for E2E and API tests.
:::

:::caution[Playwright is Not a Unit Test Tool]
Playwright is optimized for browser automation, not unit testing. Use Jest or Vitest for unit and component logic. Use Playwright for the full user journey — login through checkout, file uploads, multi-step forms.
:::

## Playwright Architecture

Playwright communicates with browsers through the **Chrome DevTools Protocol (CDP)** for Chromium, and native protocols for Firefox and WebKit. This out-of-process model gives Playwright:

- **Isolation**: each test gets its own browser context (equivalent to a fresh incognito window)
- **Speed**: contexts are cheap to create — no full browser restart needed per test
- **Reliability**: auto-waiting on all actions eliminates explicit `sleep()` calls

```
Your Test Code
     │
     ▼
@playwright/test
     │  (Node.js process)
     ▼
Playwright Server  ──────────────────┐
     │                               │
     ▼                               ▼
Chromium (CDP)    Firefox (native)   WebKit (native)
```

### Browser Hierarchy

```
Browser                  ← one Chromium/Firefox/WebKit instance
  └── BrowserContext     ← isolated session (cookies, storage, permissions)
        └── Page         ← a single tab
              └── Frame  ← main frame or iframe
```

## Auto-Waiting Philosophy

Playwright's most important feature: **every action automatically waits** for the target element to be ready before proceeding.

When you call `page.getByRole('button', { name: 'Submit' }).click()`, Playwright:

1. Waits for the element to be **attached** to the DOM
2. Waits for the element to be **visible**
3. Waits for the element to be **stable** (not animating)
4. Waits for the element to receive **pointer events** (not covered by another element)
5. Waits for the element to be **enabled**
6. Then performs the click

This eliminates the most common source of test flakiness: timing issues.

```typescript
// ❌ Selenium / Cypress style — explicit waiting
await driver.wait(until.elementIsVisible(button), 5000);
await button.click();

// ✅ Playwright — implicit waiting built in
await page.getByRole('button', { name: 'Submit' }).click();
```

## Trace Viewer

Playwright records a trace of every test run — a complete recording of browser state, actions, network requests, and console output. Open the trace in the interactive viewer:

```bash
npx playwright show-trace trace.zip
```

The trace viewer shows:

- Timeline of all actions
- Screenshot before and after each action
- DOM snapshot at any point
- Network tab with all requests/responses
- Console logs

```typescript
// Enable tracing in playwright.config.ts
use: {
  trace: 'on-first-retry',   // record trace only when a test retries
  trace: 'on',               // always record
  trace: 'off',              // never record (default)
  trace: 'retain-on-failure', // save trace only for failed tests
}
```

## Codegen — Record & Playback

Playwright's codegen tool records your browser interactions and generates test code:

```bash
npx playwright codegen https://myapp.example.com
```

This opens a browser with a recording toolbar. Every click, type, and navigation is converted to Playwright code in real-time. Use it to:

- Bootstrap tests quickly
- Learn Playwright's locator API
- Generate selectors for complex UI elements

## Visual Comparisons

Playwright has built-in screenshot comparison for visual regression testing:

```typescript
test('product card matches design', async ({ page }) => {
  await page.goto('/products/42');

  // First run: creates the reference screenshot
  // Subsequent runs: compares against it
  await expect(page.getByTestId('product-card')).toHaveScreenshot(
    'product-card.png',
    {
      maxDiffPixels: 100, // allow up to 100 pixels difference
      threshold: 0.2, // 20% pixel color tolerance
      mask: [page.getByTestId('timestamp')], // mask dynamic content
    },
  );
});
```

Playwright is an open-source end-to-end testing framework developed by Microsoft. It enables reliable, cross-browser automation for Chromium, Firefox, and WebKit from a single API — testing your application the way real users experience it.

## What is Playwright?

Playwright drives real browsers — not simulated DOM environments. When a Playwright test clicks a button, a real Chromium, Firefox, or Safari engine processes that click, runs JavaScript, applies CSS, and renders pixels, exactly as it does for real users.

```typescript
import { test, expect } from '@playwright/test';

test('user can log in and see dashboard', async ({ page }) => {
  await page.goto('https://app.example.com/login');

  await page.getByLabel('Email').fill('alice@example.com');
  await page.getByLabel('Password').fill('securePassword123');
  await page.getByRole('button', { name: 'Sign in' }).click();

  await expect(page).toHaveURL('/dashboard');
  await expect(page.getByRole('heading', { name: 'Welcome, Alice' })).toBeVisible();
});
```

## Cross-Browser Support

| Browser | Engine | Supported |
|---|---|---|
| Chrome | Chromium | ✅ |
| Edge | Chromium | ✅ |
| Firefox | Gecko | ✅ |
| Safari | WebKit | ✅ |
| Mobile Chrome | Chromium (device emulation) | ✅ |
| Mobile Safari | WebKit (device emulation) | ✅ |

A single test runs against all configured browsers in your `playwright.config.ts`:

```typescript
// playwright.config.ts
export default defineConfig({
  projects: [
    { name: 'chromium', use: { ...devices['Desktop Chrome'] } },
    { name: 'firefox', use: { ...devices['Desktop Firefox'] } },
    { name: 'webkit', use: { ...devices['Desktop Safari'] } },
    { name: 'Mobile Chrome', use: { ...devices['Pixel 5'] } },
    { name: 'Mobile Safari', use: { ...devices['iPhone 13'] } },
  ],
});
```

## Playwright vs Cypress vs Selenium

| Feature | **Playwright** | **Cypress** | **Selenium** |
|---|---|---|---|
| Browsers | Chromium, Firefox, WebKit | Chromium only (Firefox/WebKit beta) | All browsers |
| Language support | TypeScript, JavaScript, Python, Java, .NET | JS/TS only | All major languages |
| Architecture | Out-of-process (CDP/WebSocket) | In-process (same JS context) | WebDriver protocol |
| Speed | Very fast (parallel, no network throttle) | Fast (limited parallelism in free tier) | Slow (WebDriver overhead) |
| Auto-waiting | Yes (built-in, configurable) | Yes (built-in) | No (manual waits) |
| Network interception | Yes (full route API) | Yes (cy.intercept) | Limited |
| iframes | Yes (frameLocator) | Limited | Yes |
| Multiple tabs | Yes | Limited | Yes |
| Mobile emulation | Yes | Limited | No |
| Component testing | Yes (experimental) | Yes (mature) | No |
| API testing | Yes (request fixture) | No | No |
| Visual testing | Built-in screenshots | Plugin | Plugin |
| Trace viewer | Yes | No | No |
| Codegen | Yes | No | No |
| CI speed | Excellent (sharding, parallelism) | Good (paid parallelism) | Moderate |
| Setup complexity | Low | Very low | High |
| License | Apache 2.0 (free) | MIT + paid plans | Apache 2.0 |

:::tip[When to Choose Playwright]
Choose Playwright when you need true cross-browser coverage (especially Safari/WebKit), parallel test execution without per-seat pricing, Python/Java/.NET support, or built-in network mocking. It's Microsoft-backed with strong long-term support.
:::

:::caution[Playwright vs Cypress Mental Model]
Playwright uses an async/await model — every action returns a promise. Cypress uses a queue-based model where commands are chained synchronously. If your team is deeply invested in Cypress, migration requires a mental model shift, not just a syntax change.
:::

## Playwright Architecture

Playwright communicates with browsers over the **Chrome DevTools Protocol (CDP)** and a custom WebSocket protocol for Firefox and WebKit. This out-of-process architecture means:

- Tests run in Node.js; browsers run as separate processes
- Multiple browser contexts can run in parallel within one test
- Network interception happens at the browser level (not a proxy)
- Tests are not affected by browser security restrictions on same-origin

```
Test Process (Node.js)
    │
    │ WebSocket / CDP
    ▼
Browser Process (Chromium / Firefox / WebKit)
    │
    ├── BrowserContext 1 (isolated session, cookies, storage)
    │       └── Page 1
    │       └── Page 2 (new tab)
    │
    └── BrowserContext 2 (different user, isolated)
            └── Page 3
```

### Key Abstractions

| Object | Description |
|---|---|
| `Browser` | A browser instance (Chromium, Firefox, or WebKit) |
| `BrowserContext` | An isolated browser session (like an incognito window) |
| `Page` | A single tab within a context |
| `Locator` | A lazy reference to DOM elements with auto-waiting |
| `Frame` | A frame/iframe within a page |
| `Request` / `Response` | Network request/response objects |

## Auto-Waiting Philosophy

Playwright's most important feature is **auto-waiting**: before performing any action, Playwright waits for the target element to be:

1. Attached to the DOM
2. Visible (not hidden by CSS)
3. Stable (not animating)
4. Enabled (not disabled)
5. Editable (for fill/type)
6. Receiving events (not obscured by another element)

This eliminates the vast majority of `waitFor` calls that plague Selenium and reduce flakiness:

```typescript
// Playwright auto-waits for the button to be visible, enabled, and stable
await page.getByRole('button', { name: 'Submit' }).click();

// Playwright auto-waits for the text to appear (no manual wait needed)
await expect(page.getByRole('alert')).toHaveText('Saved successfully');
```

## Developer Tools

### Trace Viewer

Playwright Trace captures a complete timeline of every test action, including:
- Screenshots before and after each step
- DOM snapshots (interactive, scrubable)
- Network requests and responses
- Console output
- Action timing

```bash
# Record traces in CI
npx playwright test --trace on

# Open the trace viewer
npx playwright show-trace trace.zip
```

### Codegen — Record Tests

Generate test code by clicking through your application:

```bash
npx playwright codegen https://app.example.com
```

Playwright opens a browser and records every interaction, generating test code in real time. Use it as a starting point, then refine the selectors to use `getByRole` and `getByLabel`.

### UI Mode

Playwright's UI mode provides a visual test runner with:
- Test file tree with pass/fail indicators
- Time-travel debugging through traces
- Watch mode that re-runs on file changes

```bash
npx playwright test --ui
```

### Visual Comparisons

```typescript
// Compare screenshots pixel-by-pixel
await expect(page).toHaveScreenshot('homepage.png');

// With options
await expect(page).toHaveScreenshot('button.png', {
  clip: { x: 0, y: 0, width: 200, height: 50 },
  threshold: 0.1, // allow 10% pixel difference
  maxDiffPixels: 50,
});
```

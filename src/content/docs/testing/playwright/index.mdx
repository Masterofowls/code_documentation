Playwright is Microsoft's end-to-end testing framework that enables reliable cross-browser testing across Chromium, Firefox, and WebKit from a single API. It is designed for modern web applications, with built-in support for auto-waiting, network interception, parallel execution, and visual comparisons.

## What is Playwright?

Playwright automates real browser instances — not simulated environments. It can interact with a web application exactly as a user would: clicking buttons, filling forms, navigating between pages, uploading files, and verifying visible results.

```typescript
import { test, expect } from '@playwright/test';

test('user can log in and see their dashboard', async ({ page }) => {
  await page.goto('https://app.example.com/login');

  await page.getByLabel('Email').fill('alice@example.com');
  await page.getByLabel('Password').fill('secret');
  await page.getByRole('button', { name: 'Sign In' }).click();

  await expect(page).toHaveURL(/\/dashboard/);
  await expect(page.getByRole('heading', { name: 'Welcome, Alice' })).toBeVisible();
});
```

## Cross-Browser Testing

Playwright tests run against all three major browser engines from a single test file:

| Engine | Browser | Notes |
|---|---|---|
| **Chromium** | Chrome, Edge (Blink engine) | Most widely used for development |
| **Firefox** | Firefox (Gecko engine) | Essential for cross-browser validation |
| **WebKit** | Safari, iOS browsers (WebKit engine) | Critical for Apple ecosystem testing |

```bash
# Run against all browsers
npx playwright test

# Run against a specific browser
npx playwright test --project=chromium
npx playwright test --project=firefox
npx playwright test --project=webkit

# Run mobile viewports
npx playwright test --project="Mobile Chrome"
npx playwright test --project="Mobile Safari"
```

## Playwright vs Cypress vs Selenium

| Feature | **Playwright** | **Cypress** | **Selenium** |
|---|---|---|---|
| Language support | JS/TS, Python, Java, C# | JS/TS only | All major languages |
| Browser support | Chromium, Firefox, WebKit | Chrome, Edge, Firefox (limited) | All browsers |
| Architecture | Out-of-process (CDP + WebSocket) | In-browser JavaScript | WebDriver protocol |
| Auto-waiting | ✅ Built-in | ✅ Built-in | ❌ Manual waits |
| Network mocking | ✅ page.route() | ✅ cy.intercept() | ❌ Limited |
| Parallel execution | ✅ Native (worker threads) | ✅ Paid plan | ❌ External tools |
| Visual snapshots | ✅ Built-in | ✅ Plugin | ❌ External |
| Component testing | ✅ Experimental | ✅ Built-in | ❌ |
| API testing | ✅ Built-in | ✅ Built-in | ❌ |
| Iframe support | ✅ Excellent | ⚠️ Limited | ✅ |
| File upload/download | ✅ Excellent | ⚠️ Workarounds | ✅ |
| Mobile emulation | ✅ Built-in | ⚠️ Limited | ❌ |
| Shadow DOM | ✅ Full support | ⚠️ Limited | ⚠️ |
| Test isolation | ✅ Per-test context | ⚠️ Per-spec file | ❌ |
| Setup complexity | Low | Very low | High |
| CI performance | Excellent | Good | Poor |
| License | Apache 2.0 (free) | MIT (free, cloud paid) | Apache 2.0 (free) |

:::tip[When to Choose Playwright]
Choose Playwright when you need real cross-browser testing (especially Safari/WebKit), Python/Java test suites, excellent parallel performance, superior iframe and shadow DOM support, or built-in API testing capabilities alongside E2E tests.
:::

:::caution[Playwright vs Unit Tests]
Playwright is for end-to-end and integration testing. It is not a replacement for Jest unit tests — use both. Unit tests are faster and more precise; E2E tests validate complete user workflows. A good testing strategy uses both.
:::

## Playwright Architecture

Understanding how Playwright works helps write better, more reliable tests.

### Browser, Context, and Page

```
Playwright
└── Browser (one Chromium/Firefox/WebKit instance)
    ├── BrowserContext (isolated session — like a new incognito window)
    │   ├── Page (a single browser tab)
    │   └── Page (another tab in the same session)
    └── BrowserContext (different user session — different cookies, storage)
        └── Page
```

Each `test()` in Playwright gets its own **BrowserContext** — completely isolated cookies, localStorage, and session state. This is why tests don't interfere with each other even when running in parallel.

```typescript
// Each test gets a fresh context and page automatically
test('user A sees their data', async ({ page }) => {
  await page.goto('/');
  // page is isolated from all other tests
});

test('user B sees their data', async ({ page }) => {
  await page.goto('/');
  // completely separate browser context — no shared state
});
```

### Auto-Waiting Philosophy

Playwright automatically waits for elements before interacting with them. Unlike Selenium (which requires explicit `WebDriverWait`) or raw Puppeteer (which requires manual `waitFor*` calls), Playwright's `click()`, `fill()`, `check()`, and other action methods all:

1. Wait for the element to exist in the DOM
2. Wait for the element to be visible
3. Wait for the element to be stable (not animating)
4. Wait for the element to be enabled
5. Wait for the element to be in the viewport
6. Then perform the action

```typescript
// Playwright waits automatically — no manual waiting needed
await page.getByRole('button', { name: 'Submit' }).click();
// Equivalent Selenium code would need:
// WebDriverWait(driver, 10).until(EC.element_to_be_clickable(...)).click()
```

## Trace Viewer

The Trace Viewer is Playwright's most powerful debugging tool. It records a complete timeline of test execution including DOM snapshots at every action, network requests, console logs, and screenshots.

```bash
# Record traces on first retry (recommended for CI)
npx playwright test --trace on-first-retry

# Always record traces
npx playwright test --trace on

# View a trace file
npx playwright show-trace trace.zip

# Or open the HTML report (which includes traces)
npx playwright show-report
```

The Trace Viewer shows:
- Before/after DOM snapshots for every action
- Network waterfall
- Console messages
- Errors with stack traces
- Video recording (if enabled)

## Codegen — Test Recorder

Generate test code by recording browser interactions:

```bash
npx playwright codegen https://app.example.com
```

Playwright opens a browser and the Codegen inspector. As you click, type, and navigate, Playwright generates test code in real time. Use the generated code as a starting point, then refine it.

## Visual Comparisons

Playwright has built-in screenshot comparison (visual regression testing):

```typescript
test('dashboard layout matches snapshot', async ({ page }) => {
  await page.goto('/dashboard');

  // Compare the full page to a stored screenshot
  await expect(page).toHaveScreenshot('dashboard.png');

  // Compare a specific element
  await expect(page.getByRole('main')).toHaveScreenshot('main-content.png', {
    threshold: 0.1,      // allow up to 10% pixel difference
    maxDiffPixels: 100,  // or allow up to 100 different pixels
  });
});
```

## API Testing

Playwright can test REST APIs directly without a browser:

```typescript
import { test, expect } from '@playwright/test';

test('creates a new user via API', async ({ request }) => {
  const response = await request.post('/api/users', {
    data: { name: 'Alice', email: 'alice@example.com' },
  });

  expect(response.status()).toBe(201);
  const body = await response.json();
  expect(body).toMatchObject({ name: 'Alice' });
});
```

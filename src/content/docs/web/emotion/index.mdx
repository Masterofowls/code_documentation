Emotion is a **performant and flexible CSS-in-JS library** that allows you to write CSS styles with JavaScript. It offers powerful features like automatic vendor prefixing, dynamic styles based on props, theming, and server-side rendering support — all with minimal runtime overhead.

## What is CSS-in-JS?

CSS-in-JS is a styling approach where CSS is authored directly inside JavaScript/TypeScript files. Styles are scoped to their component, co-located with the logic that drives them, and can leverage the full power of JavaScript — conditionals, loops, variables, and dynamic values.

```jsx
// Traditional CSS: styling lives in a separate file
import './Button.css';

function Button({ primary }) {
  return <button className={primary ? 'btn-primary' : 'btn-secondary'}>Click</button>;
}
```

```jsx
// CSS-in-JS with Emotion: styling lives with the component
/** @jsxImportSource @emotion/react */
import { css } from '@emotion/react';

const styles = {
  primary: css`
    background: #3b82f6;
    color: white;
    padding: 0.5rem 1rem;
    border-radius: 0.5rem;
  `,
  secondary: css`
    background: transparent;
    color: #3b82f6;
    border: 1px solid #3b82f6;
    padding: 0.5rem 1rem;
    border-radius: 0.5rem;
  `,
};

function Button({ primary }) {
  return <button css={primary ? styles.primary : styles.secondary}>Click</button>;
}
```

## The CSS-in-JS Philosophy

CSS-in-JS addresses several challenges with traditional stylesheet approaches:

- **Scoping:** Styles are automatically scoped to the component — no leakage or specificity battles
- **Dead code elimination:** When a component is deleted, its styles are deleted too
- **Dynamic styles:** Styles can respond to props, state, theme, or any JavaScript value at runtime
- **Colocation:** Styles live next to the code that uses them — easier to understand, maintain, and refactor
- **TypeScript:** Full type-safety for style values and props that influence styles

:::caution[Runtime vs Zero-Runtime]
Emotion is a *runtime* CSS-in-JS library — styles are processed and injected by JavaScript in the browser. This means there's a JavaScript bundle cost and a small runtime overhead. For zero-runtime alternatives, look at Vanilla Extract or Linaria. For most applications, Emotion's performance is excellent and the overhead is negligible.
:::

## Package Overview

Emotion is split into focused packages. Understanding which to use is essential.

### `@emotion/react`

The primary package for React projects. Provides:
- The `css` prop on any element (via JSX pragma or Babel plugin)
- The `css` template literal / object helper
- `keyframes` for animations
- `Global` for global styles
- `ThemeProvider` / `useTheme` for theming

```bash
npm install @emotion/react
```

```jsx
/** @jsxImportSource @emotion/react */
import { css } from '@emotion/react';

const style = css`
  color: hotpink;
  font-size: 1.25rem;
`;

function App() {
  return <div css={style}>Hello from Emotion!</div>;
}
```

### `@emotion/styled`

The `styled` API, similar to styled-components. Creates React components with styles baked in.

```bash
npm install @emotion/styled
```

```jsx
import styled from '@emotion/styled';

const Button = styled.button`
  background: #3b82f6;
  color: white;
  padding: 0.5rem 1rem;
  border-radius: 0.5rem;

  &:hover {
    background: #2563eb;
  }
`;

function App() {
  return <Button>Styled Button</Button>;
}
```

### `@emotion/css`

A framework-agnostic package that doesn't require React. Provides a `css()` function that returns a class name string.

```bash
npm install @emotion/css
```

```javascript
import { css } from '@emotion/css';

const className = css`
  color: hotpink;
  font-size: 1.5rem;
`;

document.getElementById('app').className = className;
```

## SSR Support

Emotion has first-class server-side rendering support through `CacheProvider` and `createCache`:

```tsx
import createCache from '@emotion/cache';
import { CacheProvider } from '@emotion/react';

// Create a cache with a custom nonce for Content Security Policy
const cache = createCache({ key: 'css', nonce: 'server-nonce' });

function App() {
  return (
    <CacheProvider value={cache}>
      <YourApp />
    </CacheProvider>
  );
}
```

For Next.js, a dedicated `_document.tsx` approach extracts critical CSS for SSR. See the [Performance guide](/web/emotion/performance) for the full setup.

## Performance Characteristics

Emotion is designed for performance:

| Characteristic | Details |
|---------------|---------|
| Style deduplication | Identical styles generate one class name, reused everywhere |
| Stable class names | Same styles always produce the same hash |
| Critical CSS extraction | SSR mode extracts only the styles needed for the initial render |
| No unused styles | Only styles for rendered components are generated |
| Composition | `cx()` composes class names without duplication |

:::tip[Benchmark Context]
In benchmarks, Emotion performs similarly to or faster than styled-components, especially in production mode where the Babel plugin optimizes style generation. The runtime cost is typically measured in single-digit milliseconds for most real applications.
:::

## Comparison: Emotion vs styled-components vs Tailwind

| Feature | Emotion | styled-components | Tailwind CSS |
|---------|---------|-------------------|--------------|
| Approach | CSS-in-JS (runtime) | CSS-in-JS (runtime) | Utility classes (build-time) |
| API style | `css` prop + `styled` | `styled` primarily | Class names |
| TypeScript | Excellent | Excellent | Good |
| Dynamic styles | Full JavaScript | Full JavaScript | Limited (JIT arbitrary values) |
| SSR support | Yes (CacheProvider) | Yes (ServerStyleSheet) | Yes (static CSS) |
| Bundle impact | ~7KB gzip | ~12KB gzip | ~5-30KB (purged CSS) |
| Learning curve | Medium | Low-Medium | Medium |
| Theming | ThemeProvider | ThemeProvider | config + CSS vars |
| Design systems | Excellent fit | Excellent fit | Good fit |
| Co-location | Yes — styles with code | Yes — styles with code | Sort of — classes in JSX |
| Composition | `cx()` | Tagged templates | Utility stacking |

## When to Use Emotion

**Emotion excels for:**

- Large design systems where component styles are tightly coupled to component logic
- Applications where styles change significantly at runtime (dynamic theming, user customization)
- TypeScript projects that benefit from typed theme tokens and styled props
- Teams already using styled-components who want better performance
- Projects that need powerful composition and style inheritance

**Consider Tailwind CSS instead when:**

- Your team prefers thinking in design tokens rather than arbitrary CSS
- Build-time CSS generation and zero runtime overhead is important
- The project doesn't require heavy runtime dynamic styling
- You're working with a large team and need strong conventions enforced by tooling

:::caution[Library Authors]
Avoid Emotion in published library packages. The `CacheProvider` can conflict with consumer applications, and the Babel plugin requirement adds friction. For libraries, prefer Vanilla Extract, CSS Modules, or plain CSS.
:::

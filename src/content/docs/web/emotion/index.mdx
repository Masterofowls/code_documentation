**Emotion** is a high-performance, flexible CSS-in-JS library that lets you write CSS styles using JavaScript or TypeScript. It supports both the `css` prop and `styled` components, making it adaptable to any workflow.

## What Is CSS-in-JS?

CSS-in-JS is an approach where styles are colocated with JavaScript components instead of being managed in separate `.css` files. The styles are injected into the document at runtime (or extracted at build time for SSR).

```jsx
// Traditional CSS
import './Button.css';
function Button({ children }) {
  return <button className="button button--primary">{children}</button>;
}

// CSS-in-JS with Emotion
import { css } from '@emotion/react';
const buttonStyle = css`
  background: #7c3aed;
  color: white;
  padding: 0.5rem 1rem;
  border-radius: 0.5rem;
  font-weight: 600;
`;
function Button({ children }) {
  return <button css={buttonStyle}>{children}</button>;
}
```

### Philosophy

CSS-in-JS solves several problems inherent in traditional CSS:

| Problem            | Traditional CSS        | CSS-in-JS                         |
| ------------------ | ---------------------- | --------------------------------- |
| Global scope       | Class name collisions  | Automatically scoped to component |
| Dead code          | Accumulates over time  | Removed with JS tree-shaking      |
| Conditional styles | Complex class toggling | Direct JS logic                   |
| Dynamic values     | CSS variables only     | Any JS expression                 |
| Co-location        | Separate files         | Styles live next to component     |
| TypeScript         | No type checking       | Full type safety for props        |

## Emotion Package Overview

Emotion ships three packages with different use cases:

### `@emotion/react`

The primary package. Provides the `css` prop, `Global`, `keyframes`, and `ThemeProvider`.

```bash
npm install @emotion/react
```

```jsx
/** @jsxImportSource @emotion/react */
import { css } from '@emotion/react';

const style = css`
  color: hotpink;
  font-size: 1.5rem;
`;

function Component() {
  return <div css={style}>Hello Emotion</div>;
}
```

### `@emotion/styled`

Provides the `styled` factory for creating styled components — similar to styled-components.

```bash
npm install @emotion/react @emotion/styled
```

```jsx
import styled from '@emotion/styled';

const Button = styled.button`
  background: #7c3aed;
  color: white;
  padding: 0.5rem 1rem;
  border-radius: 0.5rem;
`;

function App() {
  return <Button>Click me</Button>;
}
```

### `@emotion/css`

Framework-agnostic package — works outside of React (Angular, Vue, plain JS):

```bash
npm install @emotion/css
```

```js
import { css, cx, injectGlobal } from '@emotion/css';

const color = css`
  color: hotpink;
`;

const div = document.getElementById('root');
div.classList.add(color);
```

## SSR Support

Emotion has first-class server-side rendering support through `CacheProvider` and `createCache`:

```tsx
// Server-side: extract critical CSS
import createEmotionServer from '@emotion/server/create-instance';
import createCache from '@emotion/cache';

const cache = createCache({ key: 'css' });
const { extractCriticalToChunks, constructStyleTagsFromChunks } =
  createEmotionServer(cache);

// Client-side: hydrate from server-extracted styles
import { CacheProvider } from '@emotion/react';
import createCache from '@emotion/cache';

const clientCache = createCache({ key: 'css' });

function App({ children }) {
  return <CacheProvider value={clientCache}>{children}</CacheProvider>;
}
```

## Performance Characteristics

Emotion is one of the fastest CSS-in-JS libraries:

- **Zero-runtime option** with `@emotion/css` (generates static class names)
- **Incremental style injection** — only injects styles for rendered components
- **Style deduplication** — identical styles produce the same class name
- **Atomic CSS generation** — similar to Tailwind in approach when using `@emotion/css`
- **`babel-plugin-emotion`** — optional build-time optimization for labels and source maps

:::tip[Performance in production]
For SSR applications, always use `extractCritical` or `extractCriticalToChunks` to prevent a flash of unstyled content (FOUC) and improve Largest Contentful Paint (LCP) scores.
:::

## Emotion vs styled-components vs Tailwind

| Criterion         | Emotion                            | styled-components | Tailwind CSS                |
| ----------------- | ---------------------------------- | ----------------- | --------------------------- |
| Syntax            | Template literals or object styles | Template literals | Utility class names in HTML |
| Bundle size       | ~7 KB gzip                         | ~12 KB gzip       | ~5–15 KB (purged)           |
| Runtime cost      | Low                                | Low-Medium        | Zero (no runtime)           |
| Dynamic styles    | JS expressions                     | JS expressions    | Arbitrary values / inline   |
| TypeScript        | Excellent                          | Good              | Excellent                   |
| SSR               | Built-in                           | Built-in          | Not applicable              |
| Theming           | ThemeProvider                      | ThemeProvider     | CSS variables + config      |
| Tooling           | babel-plugin                       | babel-plugin      | VS Code extension           |
| Learning curve    | Low-Medium                         | Low               | Medium                      |
| Colocation        | Yes                                | Yes               | Yes (classes in JSX)        |
| Server components | Not compatible                     | Not compatible    | Compatible                  |

:::caution[React Server Components]
Emotion and styled-components require a client-side runtime and **cannot be used in React Server Components**. If you're building with Next.js App Router and want to maximize server rendering, consider Tailwind CSS or vanilla CSS modules instead.
:::

## When to Use Emotion

**Emotion is a great choice when:**

- You want styles strongly colocated with components
- Styles depend heavily on component props or application state
- Your team prefers writing CSS syntax within JS/TS files
- You need SSR with critical CSS extraction
- You're building a design system with ThemeProvider-based theming

**Consider alternatives when:**

- Using React Server Components (use CSS Modules or Tailwind)
- Bundle size is a hard constraint (Tailwind has zero runtime)
- The team is more comfortable with traditional CSS authoring

Web accessibility means building sites that everyone can use — including people with visual, motor, auditory, and cognitive disabilities. It is not optional: in many countries it is a legal requirement, and it makes your site better for all users.

## WCAG 2.1 AA Overview

The **Web Content Accessibility Guidelines (WCAG) 2.1** are the international standard. They are organised around four principles:

| Principle | Meaning | Example |
|---|---|---|
| **Perceivable** | Content must be presentable in ways users can perceive | Alt text for images, captions for video |
| **Operable** | Interface must be navigable and usable | Full keyboard access, sufficient time limits |
| **Understandable** | Content must be readable and predictable | Clear error messages, consistent navigation |
| **Robust** | Content must work with assistive technologies | Valid HTML, correct ARIA |

**Level AA** is the target for most websites — it is what most accessibility laws reference. Level AAA is stricter and not required for entire sites.

## ARIA Roles

ARIA (Accessible Rich Internet Applications) roles override or supplement the semantic meaning of HTML elements for assistive technologies.

### Landmark Roles

Landmark roles mirror the semantic HTML landmarks — use the HTML elements when possible, and roles only when you can't:

```html
<!-- Prefer native HTML landmarks -->
<header>...</header>     <!-- implicit role="banner" -->
<nav>...</nav>           <!-- implicit role="navigation" -->
<main>...</main>         <!-- implicit role="main" -->
<aside>...</aside>       <!-- implicit role="complementary" -->
<footer>...</footer>     <!-- implicit role="contentinfo" -->
<form>...</form>         <!-- implicit role="form" (when it has an accessible name) -->

<!-- ARIA landmarks when native elements can't be used (e.g. legacy apps) -->
<div role="banner">...</div>
<div role="navigation" aria-label="Main menu">...</div>
<div role="main">...</div>
```

### Widget Roles

Widget roles describe interactive components the browser doesn't natively support:

```html
<!-- Tabs -->
<div role="tablist" aria-label="Account settings">
  <button role="tab" id="tab-profile" aria-controls="panel-profile" aria-selected="true">
    Profile
  </button>
  <button role="tab" id="tab-security" aria-controls="panel-security" aria-selected="false" tabindex="-1">
    Security
  </button>
</div>
<div role="tabpanel" id="panel-profile" aria-labelledby="tab-profile">
  <!-- Profile settings content -->
</div>
<div role="tabpanel" id="panel-security" aria-labelledby="tab-security" hidden>
  <!-- Security settings content -->
</div>

<!-- Tree (hierarchical list) -->
<ul role="tree" aria-label="File explorer">
  <li role="treeitem" aria-expanded="true">
    <span>src/</span>
    <ul role="group">
      <li role="treeitem">index.html</li>
      <li role="treeitem">styles.css</li>
    </ul>
  </li>
</ul>

<!-- Slider -->
<div
  role="slider"
  aria-valuemin="0"
  aria-valuemax="100"
  aria-valuenow="60"
  aria-valuetext="60%"
  aria-label="Volume"
  tabindex="0"
></div>
```

## Essential ARIA Attributes

### Labelling

```html
<!-- aria-label: direct string label (no visible text needed) -->
<button aria-label="Close modal" onclick="modal.close()">×</button>

<!-- aria-labelledby: points to another element's text -->
<h2 id="billing-title">Billing Address</h2>
<section aria-labelledby="billing-title">
  <!-- content -->
</section>

<!-- aria-describedby: provides additional description -->
<input
  type="password"
  id="password"
  aria-describedby="password-hint"
/>
<p id="password-hint">
  Must be at least 12 characters with one uppercase letter and one number.
</p>
```

### State and Properties

```html
<!-- aria-hidden: remove from accessibility tree (decorative content) -->
<span aria-hidden="true">★★★★☆</span>
<span class="sr-only">4 out of 5 stars</span>

<!-- aria-expanded: whether a section is expanded/collapsed -->
<button
  aria-expanded="false"
  aria-controls="dropdown-menu"
  onclick="toggleDropdown()"
>
  Options ▾
</button>
<ul id="dropdown-menu" hidden>
  <li><a href="/edit">Edit</a></li>
  <li><a href="/delete">Delete</a></li>
</ul>

<!-- aria-live: announce dynamic content updates -->
<div aria-live="polite" aria-atomic="true" id="status-message">
  <!-- Content injected here will be announced by screen readers -->
</div>

<!-- aria-live="assertive" for critical interruptions -->
<div role="alert" aria-live="assertive" id="error-message"></div>

<!-- aria-current: marks the current item in a set -->
<nav aria-label="Breadcrumb">
  <ol>
    <li><a href="/">Home</a></li>
    <li><a href="/blog">Blog</a></li>
    <li><a href="/blog/html" aria-current="page">HTML Accessibility</a></li>
  </ol>
</nav>

<!-- aria-controls: identifies the element this controls -->
<button aria-controls="sidebar" aria-expanded="true" onclick="toggleSidebar()">
  Toggle Sidebar
</button>

<!-- aria-busy: element is loading -->
<div aria-busy="true" aria-label="Loading results...">
  <div class="spinner"></div>
</div>
```

## Tabindex

`tabindex` controls whether and where an element appears in the keyboard focus order:

```html
<!-- tabindex="0": add to natural tab order (for non-interactive elements that need focus) -->
<div
  role="button"
  tabindex="0"
  onclick="handleClick()"
  onkeydown="if(event.key === 'Enter' || event.key === ' ') handleClick()"
>
  Custom Button
</div>

<!-- tabindex="-1": remove from tab order but allow programmatic focus -->
<div id="error-summary" tabindex="-1">
  <h2>Please fix the following errors:</h2>
  <!-- Focused via JS after form submission failure -->
</div>
```

```js
// Programmatically focus an element with tabindex="-1"
document.getElementById('error-summary').focus();
```

:::caution[Avoid Positive `tabindex` Values]
`tabindex="1"`, `tabindex="2"`, etc. create a separate tab order that overrides the natural document order, confusing users and making maintenance a nightmare. Use `tabindex="0"` and `tabindex="-1"` only.
:::

## Focus Management

```html
<!-- Skip navigation link — first focusable element on page -->
<a href="#main-content" class="skip-link">Skip to main content</a>

<nav><!-- ... --></nav>

<main id="main-content" tabindex="-1">
  <h1>Page Title</h1>
</main>
```

```css
/* Visually hide skip link until focused */
.skip-link {
  position: absolute;
  top: -100%;
  left: 0;
  background: #000;
  color: #fff;
  padding: 0.5rem 1rem;
  font-size: 1rem;
  z-index: 9999;
  text-decoration: none;
  border-radius: 0 0 4px 0;
}

.skip-link:focus {
  top: 0;
}
```

### Modal Focus Trapping

When a modal opens, focus must be trapped inside it:

```js
function trapFocus(element) {
  const focusableElements = element.querySelectorAll(
    'a[href], button:not([disabled]), textarea, input:not([disabled]), ' +
    'select:not([disabled]), [tabindex="0"]'
  );

  const firstFocusable = focusableElements[0];
  const lastFocusable = focusableElements[focusableElements.length - 1];

  element.addEventListener('keydown', function(event) {
    if (event.key !== 'Tab') return;

    if (event.shiftKey) {
      // Shift+Tab: backwards
      if (document.activeElement === firstFocusable) {
        event.preventDefault();
        lastFocusable.focus();
      }
    } else {
      // Tab: forwards
      if (document.activeElement === lastFocusable) {
        event.preventDefault();
        firstFocusable.focus();
      }
    }
  });
}
```

:::tip[Use the Native `<dialog>` Element]
The native `<dialog>` element handles focus trapping automatically. Always prefer it over a custom `<div role="dialog">` implementation.
:::

## Keyboard Navigation Patterns

| Widget | Keys Required |
|---|---|
| Links / Buttons | `Enter` to activate; `Space` for buttons |
| Checkboxes | `Space` to toggle |
| Radio group | `Tab` into group, `Arrow keys` between options |
| Select | `Arrow keys` to change value |
| Tabs | `Arrow keys` between tabs, `Tab` into panel |
| Modal | `Escape` to close, focus trapped inside |
| Dropdown menu | `Escape` to close, `Arrow keys` to navigate items |
| Tree view | `Arrow keys` to navigate, `Enter` to activate |

## Screen Reader Testing

### Screen Readers to Test With

| Screen Reader | Platform | Browser |
|---|---|---|
| NVDA (free) | Windows | Chrome, Firefox |
| JAWS (paid) | Windows | Chrome, Edge, IE |
| VoiceOver | macOS / iOS | Safari |
| TalkBack | Android | Chrome |

### VoiceOver Quick Start (macOS)

- **Start/stop:** `Cmd + F5`
- **Read next:** `Ctrl + Option + Right Arrow`
- **Heading navigation:** `Ctrl + Option + Cmd + H`
- **Landmark navigation:** `Ctrl + Option + Cmd + U` then Arrow keys
- **Click:** `Ctrl + Option + Space`

## Colour Contrast

WCAG AA requires:
- **Normal text** (< 18pt regular, < 14pt bold): contrast ratio ≥ **4.5:1**
- **Large text** (≥ 18pt regular, ≥ 14pt bold): contrast ratio ≥ **3:1**
- **UI components and graphics**: contrast ratio ≥ **3:1**

```html
<!-- Bad: light grey text on white background (1.5:1 ratio) -->
<p style="color: #aaaaaa; background: #ffffff;">Hard to read</p>

<!-- Good: dark text on white background (7:1 ratio) -->
<p style="color: #374151; background: #ffffff;">Easy to read</p>
```

Test contrast with tools like [WebAIM Contrast Checker](https://webaim.org/resources/contrastchecker/) or browser DevTools.

## Accessible Forms

```html
<form novalidate>
  <!-- Associate every input with a visible label -->
  <div class="form-field">
    <label for="full-name">
      Full name
      <span aria-hidden="true" class="required-marker">*</span>
    </label>
    <input
      type="text"
      id="full-name"
      name="full-name"
      required
      aria-required="true"
      aria-describedby="full-name-error"
      autocomplete="name"
    />
    <!-- Error message hidden until validation fails -->
    <span id="full-name-error" role="alert" class="error-message" hidden>
      Please enter your full name.
    </span>
  </div>

  <!-- Inform screen readers about required fields -->
  <p aria-hidden="true"><span class="required-marker">*</span> Required fields</p>

  <button type="submit">Submit</button>
</form>
```

## Accessible Modals

```html
<dialog
  id="delete-modal"
  aria-labelledby="delete-title"
  aria-describedby="delete-desc"
>
  <h2 id="delete-title">Delete item?</h2>
  <p id="delete-desc">
    This action is permanent and cannot be undone.
  </p>
  <form method="dialog">
    <button type="button" onclick="this.closest('dialog').close('cancel')">
      Cancel
    </button>
    <button value="confirm" autofocus>Delete</button>
  </form>
</dialog>
```

```js
const modal = document.getElementById('delete-modal');
const triggerButton = document.getElementById('delete-trigger');

triggerButton.addEventListener('click', () => {
  modal.showModal();
});

modal.addEventListener('close', () => {
  // Return focus to the trigger when modal closes
  triggerButton.focus();
});
```

## Automated Testing with axe DevTools

Install the [axe DevTools browser extension](https://www.deque.com/axe/devtools/) for free automated accessibility testing. You can also integrate `axe-core` into your test suite:

```js
// Jest + axe-core
import { axe, toHaveNoViolations } from 'jest-axe';

expect.extend(toHaveNoViolations);

test('form has no accessibility violations', async () => {
  document.body.innerHTML = `
    <form>
      <label for="name">Name</label>
      <input type="text" id="name" name="name" />
      <button type="submit">Submit</button>
    </form>
  `;

  const results = await axe(document.body);
  expect(results).toHaveNoViolations();
});
```

:::tip[Automated Testing Finds ~30% of Issues]
Automated tools like axe are excellent for catching obvious violations, but cannot test everything. Always supplement with manual keyboard testing and screen reader testing.
:::

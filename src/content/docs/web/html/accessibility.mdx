Web accessibility means building sites that work for everyone — including the roughly **1 in 6 people** worldwide who live with some form of disability. It is also a legal requirement in many jurisdictions (ADA, WCAG, EN 301 549).

## WCAG 2.1 AA Overview

The **Web Content Accessibility Guidelines (WCAG) 2.1** organise requirements under four principles, often abbreviated as **POUR**:

| Principle | What it means |
|---|---|
| **Perceivable** | Information must be presentable to users in ways they can perceive (text alternatives, captions, sufficient contrast) |
| **Operable** | UI components must be operable — keyboard accessible, no seizure-triggering content, enough time |
| **Understandable** | Content and operation must be understandable — readable language, predictable behaviour, input assistance |
| **Robust** | Content must be robust enough for assistive technologies to parse it reliably |

WCAG defines three levels: **A** (minimum), **AA** (standard target), **AAA** (enhanced). Most legal standards reference **AA compliance**.

## ARIA Roles

ARIA (Accessible Rich Internet Applications) roles communicate the purpose of elements to assistive technologies. There are three categories:

### Landmark Roles

Landmark roles map to HTML5 sectioning elements and let screen reader users jump directly to regions:

```html
<!-- These elements implicitly have landmark roles -->
<header>...</header>      <!-- role="banner" (top-level only) -->
<nav>...</nav>            <!-- role="navigation" -->
<main>...</main>          <!-- role="main" -->
<aside>...</aside>        <!-- role="complementary" -->
<footer>...</footer>      <!-- role="contentinfo" (top-level only) -->

<!-- When you have multiple of the same landmark, label them -->
<nav aria-label="Primary navigation">...</nav>
<nav aria-label="Breadcrumb">...</nav>

<!-- Search landmark (no HTML5 equivalent) -->
<div role="search">
  <form action="/search">
    <label for="site-search">Search this site</label>
    <input type="search" id="site-search" name="q" />
    <button type="submit">Search</button>
  </form>
</div>
```

### Widget Roles

```html
<!-- Button that opens a menu -->
<button
  type="button"
  aria-expanded="false"
  aria-haspopup="menu"
  aria-controls="user-menu"
  id="user-menu-btn"
>
  My Account
</button>

<ul role="menu" id="user-menu" aria-labelledby="user-menu-btn" hidden>
  <li role="menuitem"><a href="/profile">Profile</a></li>
  <li role="menuitem"><a href="/settings">Settings</a></li>
  <li role="separator"></li>
  <li role="menuitem"><button type="button">Sign Out</button></li>
</ul>

<!-- Tab panel -->
<div role="tablist" aria-label="Product sections">
  <button role="tab" id="tab-overview" aria-selected="true" aria-controls="panel-overview">Overview</button>
  <button role="tab" id="tab-specs" aria-selected="false" aria-controls="panel-specs">Specs</button>
  <button role="tab" id="tab-reviews" aria-selected="false" aria-controls="panel-reviews">Reviews</button>
</div>

<div role="tabpanel" id="panel-overview" aria-labelledby="tab-overview">...</div>
<div role="tabpanel" id="panel-specs" aria-labelledby="tab-specs" hidden>...</div>
<div role="tabpanel" id="panel-reviews" aria-labelledby="tab-reviews" hidden>...</div>
```

## Essential ARIA Attributes

| Attribute | Purpose | Example value |
|---|---|---|
| `aria-label` | Provides a text label for an element without visible text | `aria-label="Close dialog"` |
| `aria-labelledby` | References the `id` of another element that labels this one | `aria-labelledby="dialog-title"` |
| `aria-describedby` | References element(s) that provide supplementary description | `aria-describedby="password-hint"` |
| `aria-hidden` | Removes element from accessibility tree (decorative content) | `aria-hidden="true"` |
| `aria-live` | Announces dynamic content changes | `aria-live="polite"` |
| `aria-expanded` | Current expanded/collapsed state of a toggleable element | `aria-expanded="false"` |
| `aria-controls` | References the element this control manages | `aria-controls="dropdown-menu"` |
| `aria-current` | Indicates current item in a set (nav, breadcrumb) | `aria-current="page"` |
| `aria-required` | Indicates a field is required (in addition to HTML `required`) | `aria-required="true"` |
| `aria-invalid` | Marks a field as having an invalid value | `aria-invalid="true"` |

```html
<!-- Accessible notification area -->
<div
  role="status"
  aria-live="polite"
  aria-atomic="true"
  id="notification"
  class="sr-only"
></div>

<script>
  function notify(message) {
    const el = document.getElementById('notification');
    el.textContent = '';
    // Small timeout forces screen readers to re-announce even identical messages
    requestAnimationFrame(() => {
      el.textContent = message;
    });
  }

  // Usage
  notify('Item added to cart');
</script>
```

## `tabindex`

Controls keyboard focus order:

```html
<!-- tabindex="0": add element to natural tab order -->
<div role="button" tabindex="0" class="custom-btn">
  Click Me
</div>

<!-- tabindex="-1": focusable via JS but not in tab order -->
<!-- Use for elements that receive programmatic focus (dialogs, error messages) -->
<div role="dialog" tabindex="-1" id="modal" aria-modal="true" aria-labelledby="modal-title">
  <h2 id="modal-title">Confirmation Required</h2>
  ...
</div>

<!-- tabindex > 0: AVOID — creates non-linear tab order -->
<!-- Only use in rare legacy situations -->
```

:::caution[Avoid Positive tabindex Values]
Using `tabindex="1"`, `tabindex="2"` etc. creates a separate tab order that fires before the natural DOM order — this confuses users and is very hard to maintain. Use `tabindex="0"` or reorganise the DOM instead.
:::

## Skip Navigation Link

A skip link lets keyboard users bypass repetitive navigation and jump straight to the main content:

```html
<!-- First focusable element on the page -->
<a href="#main-content" class="skip-link">Skip to main content</a>

<header>
  <nav><!-- many links --></nav>
</header>

<main id="main-content" tabindex="-1">
  <h1>Page Title</h1>
  <!-- content -->
</main>
```

```css
/* Hidden by default, visible on focus */
.skip-link {
  position: absolute;
  top: -100%;
  left: 1rem;
  z-index: 9999;
  padding: 0.5rem 1rem;
  background: #000;
  color: #fff;
  text-decoration: none;
  border-radius: 0 0 4px 4px;
  transition: top 0.2s;
}

.skip-link:focus {
  top: 0;
}
```

## Focus Management

When UI changes dynamically (modals open, content loads), move focus programmatically:

```js
// Modal focus management
function openModal(modalId) {
  const modal = document.getElementById(modalId);
  const focusableSelectors = [
    'a[href]', 'button:not([disabled])', 'textarea',
    'input:not([disabled])', 'select:not([disabled])',
    '[tabindex]:not([tabindex="-1"])'
  ].join(', ');

  modal.removeAttribute('hidden');
  modal.setAttribute('aria-modal', 'true');

  // Move focus to the modal container (or first focusable element)
  const firstFocusable = modal.querySelector(focusableSelectors);
  (firstFocusable || modal).focus();

  // Trap focus inside the modal
  modal.addEventListener('keydown', trapFocus);
}

function trapFocus(e) {
  if (e.key !== 'Tab') return;

  const focusable = [...e.currentTarget.querySelectorAll(
    'a[href], button:not([disabled]), input, textarea, select, [tabindex="0"]'
  )];

  const first = focusable[0];
  const last = focusable.at(-1);

  if (e.shiftKey && document.activeElement === first) {
    e.preventDefault();
    last.focus();
  } else if (!e.shiftKey && document.activeElement === last) {
    e.preventDefault();
    first.focus();
  }
}
```

## Keyboard Navigation Patterns

| Pattern | Keys |
|---|---|
| Any focusable element | `Tab` (forward), `Shift+Tab` (backward) |
| Buttons / links | `Enter` to activate; `Space` for buttons |
| Checkboxes | `Space` to toggle |
| Radio groups | Arrow keys to move between options |
| Dropdown menus | `Enter`/`Space` to open, `Escape` to close, Arrow keys to navigate |
| Dialogs | `Escape` to close |
| Carousels/sliders | Arrow keys |

## Accessible Forms

```html
<form>
  <!-- Always associate labels with inputs -->
  <div class="field">
    <label for="email">Email address</label>
    <input
      type="email"
      id="email"
      name="email"
      required
      aria-required="true"
      aria-describedby="email-hint email-error"
      autocomplete="email"
    />
    <p id="email-hint" class="hint">We'll never share your email.</p>
    <!-- Error message (shown on invalid submission) -->
    <p id="email-error" role="alert" aria-live="assertive" hidden>
      Please enter a valid email address.
    </p>
  </div>

  <!-- Group related inputs -->
  <fieldset>
    <legend>Preferred notification method</legend>
    <label>
      <input type="radio" name="notification" value="email" /> Email
    </label>
    <label>
      <input type="radio" name="notification" value="sms" /> SMS
    </label>
  </fieldset>
</form>
```

## Accessible Tables

```html
<table>
  <!-- Caption is the table's accessible name -->
  <caption>Monthly revenue by product line, Q1 2024</caption>

  <thead>
    <tr>
      <!-- scope="col" for column headers -->
      <th scope="col">Product</th>
      <th scope="col">January</th>
      <th scope="col">February</th>
      <th scope="col">March</th>
    </tr>
  </thead>

  <tbody>
    <tr>
      <!-- scope="row" for row headers -->
      <th scope="row">Widget A</th>
      <td>$12,400</td>
      <td>$13,100</td>
      <td>$11,900</td>
    </tr>
    <tr>
      <th scope="row">Widget B</th>
      <td>$8,700</td>
      <td>$9,200</td>
      <td>$10,400</td>
    </tr>
  </tbody>
</table>
```

## Color Contrast

WCAG 2.1 AA requires:

- **Normal text** (below 18pt / 14pt bold): **4.5:1** contrast ratio
- **Large text** (18pt+ / 14pt+ bold): **3:1** contrast ratio
- **UI components and graphics**: **3:1**

```html
<!-- Always test your colour combinations -->
<!-- Passing examples: -->
<!-- #000000 on #ffffff = 21:1 ✅ -->
<!-- #595959 on #ffffff = 7:0:1 ✅ -->
<!-- #767676 on #ffffff = 4.6:1 ✅ (just passes AA) -->

<!-- Failing examples: -->
<!-- #aaaaaa on #ffffff = 2.3:1 ❌ -->
<!-- #ff6600 on #ffffff = 3.0:1 ❌ (fails for normal text) -->
```

:::tip[Use axe DevTools for Automated Testing]
The [axe browser extension](https://www.deque.com/axe/) catches ~57% of accessibility issues automatically with zero false positives. Run it on every page. It integrates with Playwright and Jest for CI testing:

```js
// Playwright + axe-core
import { checkA11y } from 'axe-playwright';
await checkA11y(page, null, { runOnly: ['wcag2a', 'wcag2aa'] });
```
:::

## Screen Reader-Only Utility Class

Visually hide content while keeping it accessible:

```css
.sr-only {
  position: absolute;
  width: 1px;
  height: 1px;
  padding: 0;
  margin: -1px;
  overflow: hidden;
  clip: rect(0, 0, 0, 0);
  white-space: nowrap;
  border: 0;
}
```

```html
<!-- Provide context to screen reader users -->
<button type="button">
  <svg aria-hidden="true" focusable="false">...</svg>
  <span class="sr-only">Close dialog</span>
</button>

<!-- Table column counts -->
<p>
  Showing
  <span>10</span>
  <span class="sr-only">results out of</span>
  <span>247</span>
  <span class="sr-only">total results</span>
</p>
```

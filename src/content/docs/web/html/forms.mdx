HTML forms are the primary way users send data to your server. A well-built form is accessible, validates input before submission, and provides clear feedback on errors.

## The `<form>` Element

```html
<form
  action="/api/register"
  method="post"
  enctype="multipart/form-data"
  novalidate
>
  <!-- Form controls go here -->
</form>
```

| Attribute | Values | Purpose |
|---|---|---|
| `action` | URL | Where form data is submitted. Defaults to the current URL. |
| `method` | `get`, `post` | HTTP method. Use `get` for search forms; `post` for data that changes state. |
| `enctype` | `application/x-www-form-urlencoded` (default), `multipart/form-data`, `text/plain` | Encoding for POST data. **Must** be `multipart/form-data` when the form includes file uploads. |
| `novalidate` | (boolean) | Disables native browser validation — useful when you want full control via JavaScript. |
| `autocomplete` | `on`, `off` | Controls browser autofill for the whole form. |

:::tip[GET vs POST]
Use `method="get"` for search and filter forms — the query string is bookmarkable and shareable. Use `method="post"` for forms that create, update, or delete data, or that contain sensitive information like passwords.
:::

## Labels

Every form control **must** have an associated `<label>`. Labels provide an accessible name and increase the click/tap target.

```html
<!-- Explicit association (recommended) -->
<label for="email">Email address</label>
<input type="email" id="email" name="email" />

<!-- Implicit wrapping (also valid) -->
<label>
  Email address
  <input type="email" name="email" />
</label>

<!-- aria-label for controls where a visible label isn't possible -->
<input type="search" name="q" aria-label="Search the site" />

<!-- aria-labelledby for labels defined elsewhere -->
<h2 id="billing-heading">Billing Address</h2>
<input type="text" name="billing-street" aria-labelledby="billing-heading" />
```

:::caution[Never Use Placeholder as a Label]
`placeholder` text disappears when the user starts typing, leaving them unable to remember what the field is for. Always provide a visible `<label>`. Use `placeholder` only for format hints like `placeholder="MM/DD/YYYY"`.
:::

## Fieldset and Legend

Group related controls with `<fieldset>` and label the group with `<legend>`:

```html
<form action="/checkout" method="post">
  <fieldset>
    <legend>Personal Details</legend>
    <label for="first-name">First name</label>
    <input type="text" id="first-name" name="first-name" required autocomplete="given-name" />

    <label for="last-name">Last name</label>
    <input type="text" id="last-name" name="last-name" required autocomplete="family-name" />
  </fieldset>

  <fieldset>
    <legend>Delivery Address</legend>
    <label for="street">Street address</label>
    <input type="text" id="street" name="street" required autocomplete="street-address" />

    <label for="city">City</label>
    <input type="text" id="city" name="city" required autocomplete="address-level2" />

    <label for="postcode">Postcode</label>
    <input type="text" id="postcode" name="postcode" required autocomplete="postal-code" />
  </fieldset>

  <button type="submit">Place Order</button>
</form>
```

## Datalist (Autocomplete Suggestions)

`<datalist>` provides suggestions while still allowing free-form input:

```html
<label for="browser">Favourite browser</label>
<input
  type="text"
  id="browser"
  name="browser"
  list="browsers"
  placeholder="Start typing..."
/>
<datalist id="browsers">
  <option value="Chrome" />
  <option value="Firefox" />
  <option value="Safari" />
  <option value="Edge" />
  <option value="Opera" />
  <option value="Brave" />
</datalist>
```

## The `<output>` Element

`<output>` displays the result of a calculation or user action:

```html
<form oninput="result.value = parseInt(a.value) + parseInt(b.value)">
  <input type="number" id="a" name="a" value="10" />
  +
  <input type="number" id="b" name="b" value="20" />
  =
  <output name="result" for="a b">30</output>
</form>
```

## Built-in Form Validation

HTML5 provides client-side validation with no JavaScript required:

```html
<form action="/register" method="post">
  <!-- required — field must not be empty -->
  <input type="text" name="username" required />

  <!-- minlength / maxlength — character limits -->
  <input type="text" name="username" required minlength="3" maxlength="20" />

  <!-- min / max — numeric limits -->
  <input type="number" name="age" min="18" max="120" required />

  <!-- pattern — regular expression -->
  <input
    type="text"
    name="postcode"
    pattern="[A-Z]{1,2}[0-9][0-9A-Z]?\s?[0-9][A-Z]{2}"
    title="Enter a valid UK postcode"
    required
  />

  <!-- Email format validated automatically -->
  <input type="email" name="email" required />

  <!-- URL format validated automatically -->
  <input type="url" name="website" />

  <button type="submit">Register</button>
</form>
```

### CSS for Validation State

```css
/* Style valid and invalid fields */
input:valid {
  border-color: #22c55e;
}

input:invalid {
  border-color: #ef4444;
}

/* Only show invalid styles after user interaction */
input:not(:placeholder-shown):invalid {
  border-color: #ef4444;
  background-color: #fef2f2;
}

/* Focus styles always override validation colours */
input:focus {
  outline: 2px solid #3b82f6;
  outline-offset: 2px;
  border-color: #3b82f6;
}
```

## Custom Validity with JavaScript

For complex validation logic, use the Constraint Validation API:

```js
const form = document.getElementById('registration-form');
const password = document.getElementById('password');
const confirmPassword = document.getElementById('confirm-password');

// Custom validation: passwords must match
function validatePasswords() {
  if (password.value !== confirmPassword.value) {
    confirmPassword.setCustomValidity('Passwords do not match.');
  } else {
    confirmPassword.setCustomValidity(''); // Empty string clears the error
  }
}

password.addEventListener('input', validatePasswords);
confirmPassword.addEventListener('input', validatePasswords);

// Custom submit handler with validation
form.addEventListener('submit', (event) => {
  if (!form.checkValidity()) {
    event.preventDefault();
    // Show error summary
    const firstInvalid = form.querySelector(':invalid');
    firstInvalid?.focus();
  }
});

// Access validation state
const input = document.getElementById('email');
console.log(input.validity.valid);          // true / false
console.log(input.validity.valueMissing);   // true if required and empty
console.log(input.validity.typeMismatch);   // true if wrong format
console.log(input.validity.patternMismatch);// true if pattern fails
console.log(input.validationMessage);       // browser's error message string
```

### Validity State Properties

| Property | True When |
|---|---|
| `valueMissing` | Field is `required` and empty |
| `typeMismatch` | Value doesn't match the `type` (e.g., bad email format) |
| `patternMismatch` | Value doesn't match `pattern` |
| `tooShort` | Value is shorter than `minlength` |
| `tooLong` | Value is longer than `maxlength` |
| `rangeUnderflow` | Number is less than `min` |
| `rangeOverflow` | Number is greater than `max` |
| `stepMismatch` | Number doesn't match `step` |
| `customError` | `setCustomValidity()` was called with a non-empty string |
| `valid` | All constraints are satisfied |

:::tip[Combine Native Validation with Server Validation]
Native HTML validation is great for immediate user feedback, but **never rely on it alone** for security. Always validate input on the server — client-side validation can be bypassed by disabling JavaScript or sending raw HTTP requests.
:::

HTML (**HyperText Markup Language**) is the standard markup language for creating web pages. It provides the skeleton of every document rendered in a browser — defining structure and meaning through a system of nested **elements** represented by **tags**.

HTML is not a programming language. It describes content: headings, paragraphs, links, images, forms, and more. The browser parses this description and builds an interactive, visual document.

## HTML5 vs Older HTML

HTML has evolved significantly since its inception in the early 1990s. HTML5, finalized in 2014 and continually updated by the WHATWG as a **Living Standard**, represents a complete overhaul of the language.

### Key Improvements in HTML5

| Feature | HTML 4 / XHTML | HTML5 |
|---|---|---|
| Doctype | Long, complex DTD string | `<!doctype html>` (simple) |
| Semantic elements | None (`<div>` everywhere) | `<header>`, `<nav>`, `<main>`, `<article>`, `<section>`, `<footer>` |
| Multimedia | Required Flash plugin | Native `<video>` and `<audio>` |
| Forms | Basic inputs only | Date pickers, color inputs, validation, `<datalist>` |
| APIs | None built-in | Canvas, Web Storage, Geolocation, Web Workers, Service Workers |
| Error handling | Strict (XHTML) or inconsistent | Defined, consistent parsing algorithm |
| Character encoding | Complex declaration | `<meta charset="UTF-8">` |

:::tip[Always use the HTML5 doctype]
Every new project should start with `<!doctype html>`. It is case-insensitive, triggers standards mode in all browsers, and requires no DTD URL.
:::

## The Browser Rendering Pipeline

When a browser receives an HTML document, it processes it through a well-defined pipeline before pixels appear on screen.

<Steps>
1. **Navigation & Fetch** — The browser resolves the URL (DNS lookup, TCP/TLS handshake) and downloads the HTML bytes over HTTP.

2. **Parsing** — The HTML parser reads bytes, converts them to characters, tokenizes the stream into start tags, end tags, and text, and builds the **DOM (Document Object Model)** — a tree of node objects.

3. **Resource Discovery** — As the parser encounters `<link>`, `<script>`, `<img>`, and similar tags, it dispatches resource requests. The **preload scanner** runs speculatively ahead of the main parser.

4. **CSSOM Construction** — Downloaded CSS is parsed into the **CSS Object Model** — a tree mirroring the DOM but for style rules.

5. **JavaScript Execution** — Synchronous `<script>` tags block the parser. Modules and `defer`/`async` scripts execute at defined points.

6. **Render Tree** — The DOM and CSSOM are combined into the **Render Tree**, which contains only visible nodes with their computed styles.

7. **Layout (Reflow)** — The browser calculates the exact size and position of every render tree element in the viewport.

8. **Paint** — Pixels are drawn into layers — backgrounds, text, borders, shadows.

9. **Compositing** — Layers are assembled in the correct order (respecting z-index and opacity) and sent to the GPU for display.
</Steps>

:::caution[Parser-blocking resources hurt performance]
A `<script>` tag without `defer` or `async` stops HTML parsing completely until the script downloads and executes. Always use `defer` for non-critical scripts, and place scripts at the end of `<body>` or in `<head>` with `defer`.
:::

## HTML Document Structure

Every valid HTML5 document follows this structure:

```html
<!doctype html>
<html lang="en">
  <head>
    <!-- Document metadata — not rendered directly -->
    <meta charset="UTF-8" />
    <meta name="viewport" content="width=device-width, initial-scale=1.0" />
    <title>Page Title — shown in browser tab</title>
    <meta name="description" content="Page summary for search engines." />
    <link rel="stylesheet" href="/styles.css" />
  </head>
  <body>
    <!-- All visible content goes here -->
    <h1>Main Heading</h1>
    <p>A paragraph of text.</p>
    <script src="/app.js" defer></script>
  </body>
</html>
```

### The `<head>` Element

The `<head>` contains metadata that browsers, search engines, and social platforms consume:

- `<meta charset>` — character encoding declaration
- `<meta name="viewport">` — responsive layout instruction for mobile browsers
- `<title>` — tab title, bookmark name, and the default `<h1>` for screen readers when the page has no visible heading
- `<meta name="description">` — search engine snippet text
- `<link rel="stylesheet">` — external CSS
- `<link rel="icon">` — favicon
- `<link rel="preload">` — resource hints
- `<script>` — inline or external JavaScript (prefer `defer`)

### The `<body>` Element

Everything the user sees lives inside `<body>`. The body renders a single root node containing nested elements. Browsers are highly forgiving — they will correct many structural errors (missing closing tags, improperly nested elements) by following the HTML5 parsing spec's defined error recovery rules.

## The DOCTYPE Declaration

```html
<!doctype html>
```

The DOCTYPE is **not** an HTML tag — it is an instruction to the browser about which version of HTML the document uses. Without it, browsers enter **quirks mode**, which emulates bugs in very old browsers (Netscape 4, IE5) and produces inconsistent rendering.

With the HTML5 DOCTYPE, browsers enter **standards mode** (also called "no-quirks mode"), which renders according to current specifications.

:::tip[DOCTYPE must be the very first line]
Nothing — not even a blank line or BOM — should appear before `<!doctype html>`. Some older browsers enter quirks mode if any character precedes the DOCTYPE.
:::

## Character Encoding

```html
<meta charset="UTF-8" />
```

**UTF-8** encodes every Unicode character and is the universal standard for web documents. It is backward-compatible with ASCII. Always declare it as the **first element inside `<head>`** — browsers begin decoding bytes immediately and need the encoding declaration before reading any other text.

:::caution[Missing charset can cause security vulnerabilities]
Without a charset declaration, browsers may auto-detect encoding. In rare cases, this allows crafted content to be misinterpreted as a different encoding, enabling XSS attacks. Always declare `charset="UTF-8"`.
:::

## HTML vs XHTML

XHTML (Extensible HTML) applied XML syntax rules to HTML. It required:
- All tags lowercase
- All attributes quoted
- All elements closed (including void elements: `<br />`, `<img />`)
- A strict DOCTYPE with DTD
- Well-formed XML (no unclosed tags, no overlapping)

HTML5 made XHTML largely obsolete. Modern HTML5:
- Is **not** XML-based (though it can be served as `application/xhtml+xml`)
- Has lenient parsing with defined error recovery
- Self-closing slashes on void elements are **optional** (`<br>` and `<br />` are both valid)
- Attribute values are **optional** to quote in simple cases (but quoting is still best practice)

For new projects, use standard HTML5. XHTML remains relevant only when integrating HTML into XML pipelines (e.g., SVG documents, certain XML-based CMS systems).

## Void Elements

Void elements have no content and no closing tag. They cannot have child nodes:

```html
<br>        <!-- line break -->
<hr>        <!-- thematic break -->
<img src="photo.jpg" alt="Description">
<input type="text">
<link rel="stylesheet" href="style.css">
<meta charset="UTF-8">
<source src="video.mp4" type="video/mp4">
<track kind="subtitles" src="captions.vtt">
<wbr>       <!-- word break opportunity -->
```

The `<head>` element is invisible to most users but is the most important part of your page for search engines, social platforms, browsers, and performance. A well-configured `<head>` section dramatically improves discoverability, sharing appearance, and load speed.

## Essential Meta Tags

```html
<head>
  <!-- 1. Character encoding — must be first in <head> -->
  <meta charset="UTF-8" />

  <!-- 2. Viewport — essential for responsive design -->
  <meta name="viewport" content="width=device-width, initial-scale=1.0" />

  <!-- 3. Page title — appears in browser tab, bookmarks, and search results -->
  <title>How to Bake Sourdough Bread | The Bread Guide</title>

  <!-- 4. Meta description — appears as the snippet in search results -->
  <meta
    name="description"
    content="Step-by-step sourdough bread recipe with tips for beginners. Learn starter care, hydration ratios, scoring, and baking techniques for the perfect loaf."
  />

  <!-- 5. Robots directive -->
  <meta name="robots" content="index, follow" />

  <!-- 6. Author -->
  <meta name="author" content="Jane Doe" />
</head>
```

### Meta Description Best Practices

- **Length:** 150–160 characters (Google truncates longer ones in results)
- **Unique:** Every page should have a distinct description
- **Compelling:** Write for humans — this is your ad copy in search results
- **Keywords:** Include relevant terms naturally, but never stuff keywords

:::caution[Google May Override Your Description]
Google may choose to display different text from your page rather than your meta description. This is normal — write the best description you can, and Google will use it when it fits.
:::

## Canonical URL

The canonical tag prevents duplicate content issues when the same page is accessible via multiple URLs:

```html
<!-- The definitive URL for this page -->
<link rel="canonical" href="https://www.example.com/blog/sourdough-bread" />
```

Common scenarios requiring canonical tags:
- `http://` vs `https://`
- `www.` vs non-`www.`
- Trailing slash vs no trailing slash
- URL parameters (`?utm_source=newsletter`)
- Paginated pages

## Robots Meta Tag

```html
<!-- Default: index this page, follow all links -->
<meta name="robots" content="index, follow" />

<!-- Don't index, but follow links -->
<meta name="robots" content="noindex, follow" />

<!-- Don't index, don't follow links -->
<meta name="robots" content="noindex, nofollow" />

<!-- Specific crawlers -->
<meta name="googlebot" content="noindex" />
<meta name="bingbot" content="nosnippet" />
```

| Directive | Meaning |
|---|---|
| `index` / `noindex` | Whether to include in search results |
| `follow` / `nofollow` | Whether to follow links on this page |
| `nosnippet` | Don't show a text snippet in results |
| `noimageindex` | Don't index images on this page |
| `noarchive` | Don't show a cached copy link |
| `max-snippet:N` | Limit snippet to N characters |

## Open Graph Tags

Open Graph (OG) tags control how your page appears when shared on Facebook, LinkedIn, Slack, and most other platforms:

```html
<head>
  <!-- Basic Open Graph -->
  <meta property="og:title" content="How to Bake Sourdough Bread" />
  <meta
    property="og:description"
    content="Master sourdough bread with our step-by-step guide for beginners."
  />
  <meta property="og:image" content="https://www.example.com/images/sourdough-og.jpg" />
  <meta property="og:image:width" content="1200" />
  <meta property="og:image:height" content="630" />
  <meta property="og:image:alt" content="A freshly baked sourdough loaf on a wooden board" />
  <meta property="og:url" content="https://www.example.com/blog/sourdough-bread" />
  <meta property="og:type" content="article" />
  <meta property="og:site_name" content="The Bread Guide" />
  <meta property="og:locale" content="en_US" />

  <!-- Article-specific OG -->
  <meta property="article:published_time" content="2025-03-15T09:00:00Z" />
  <meta property="article:modified_time" content="2025-03-20T14:30:00Z" />
  <meta property="article:author" content="https://www.example.com/authors/jane-doe" />
  <meta property="article:section" content="Baking" />
  <meta property="article:tag" content="sourdough" />
  <meta property="article:tag" content="bread" />
  <meta property="article:tag" content="baking" />
</head>
```

### OG Image Specifications

| Requirement | Value |
|---|---|
| Recommended size | 1200 × 630 pixels |
| Minimum size | 600 × 315 pixels |
| Max file size | < 8 MB (smaller is better) |
| Format | JPEG or PNG recommended |
| Aspect ratio | 1.91:1 |

## Twitter Card Tags

Twitter uses its own meta tags (though it falls back to OG tags for some properties):

```html
<!-- Summary card with large image -->
<meta name="twitter:card" content="summary_large_image" />
<meta name="twitter:site" content="@BreadGuideSite" />
<meta name="twitter:creator" content="@JaneDoe" />
<meta name="twitter:title" content="How to Bake Sourdough Bread" />
<meta name="twitter:description" content="Master sourdough with our beginner's guide." />
<meta name="twitter:image" content="https://www.example.com/images/sourdough-twitter.jpg" />
<meta name="twitter:image:alt" content="A freshly baked sourdough loaf on a wooden board" />
```

| Card Type | Appearance |
|---|---|
| `summary` | Small image + title + description |
| `summary_large_image` | Large image + title + description |
| `app` | App download card |
| `player` | Video/audio player card |

## Favicons

A complete favicon setup covers all platforms:

```html
<head>
  <!-- Classic .ico (browsers that don't support SVG/PNG favicons) -->
  <link rel="icon" href="/favicon.ico" sizes="48x48" />

  <!-- SVG favicon (modern browsers, supports dark mode) -->
  <link rel="icon" href="/favicon.svg" type="image/svg+xml" />

  <!-- PNG fallback -->
  <link rel="icon" href="/favicon-96x96.png" type="image/png" sizes="96x96" />

  <!-- Apple Touch Icon (iOS home screen, Safari) -->
  <link rel="apple-touch-icon" href="/apple-touch-icon.png" /><!-- 180x180 -->

  <!-- Web App Manifest (PWA, Android home screen) -->
  <link rel="manifest" href="/site.webmanifest" />

  <!-- Tile colour for pinned Windows sites -->
  <meta name="msapplication-TileColor" content="#3b82f6" />
  <meta name="theme-color" content="#3b82f6" />
</head>
```

## Language and Internationalisation

```html
<!-- Primary language of the document -->
<html lang="en">

<!-- For a French page -->
<html lang="fr">

<!-- British English vs American English -->
<html lang="en-GB">
<html lang="en-US">
```

### `hreflang` for International Sites

Tell search engines which page to serve for each language/region:

```html
<head>
  <!-- Self-referencing canonical -->
  <link rel="canonical" href="https://www.example.com/en/about" />

  <!-- Language alternates -->
  <link rel="alternate" hreflang="en" href="https://www.example.com/en/about" />
  <link rel="alternate" hreflang="en-GB" href="https://www.example.co.uk/about" />
  <link rel="alternate" hreflang="fr" href="https://www.example.com/fr/a-propos" />
  <link rel="alternate" hreflang="de" href="https://www.example.com/de/ueber-uns" />
  <link rel="alternate" hreflang="ja" href="https://www.example.co.jp/about" />

  <!-- x-default: shown when no language matches the user's preference -->
  <link rel="alternate" hreflang="x-default" href="https://www.example.com/about" />
</head>
```

## Resource Hints

Resource hints speed up page loading by telling the browser about resources it will need:

```html
<head>
  <!-- preconnect: establish connection early (DNS + TCP + TLS) -->
  <!-- Use for critical third-party origins you'll definitely use -->
  <link rel="preconnect" href="https://fonts.googleapis.com" />
  <link rel="preconnect" href="https://fonts.gstatic.com" crossorigin />

  <!-- dns-prefetch: just DNS resolution (fallback for preconnect) -->
  <link rel="dns-prefetch" href="https://cdn.example.com" />

  <!-- preload: load a critical resource immediately, at high priority -->
  <!-- Use for fonts, hero images, critical CSS/JS -->
  <link rel="preload" href="/fonts/inter-var.woff2" as="font" type="font/woff2" crossorigin />
  <link rel="preload" href="/images/hero.webp" as="image" />

  <!-- prefetch: download a resource for likely future navigation (low priority) -->
  <link rel="prefetch" href="/page2.html" />
  <link rel="prefetch" href="/images/product-gallery.webp" />

  <!-- prerender: render an entire page in the background (use sparingly) -->
  <link rel="prerender" href="/checkout" />
</head>
```

### Resource Hint Decision Guide

| Hint | When to Use |
|---|---|
| `preconnect` | You will definitely load resources from this origin on this page |
| `dns-prefetch` | Likely (but not certain) to load from this origin |
| `preload` | Critical resource for this page's render path |
| `prefetch` | Very likely the user will navigate to this page next |

:::caution[Don't Over-preload]
`preload` forces the browser to fetch a resource regardless of whether it is needed. Preloading unused resources wastes bandwidth and competes with critical resources for network priority.
:::

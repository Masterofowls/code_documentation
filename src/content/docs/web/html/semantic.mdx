Semantic HTML means using elements that convey **meaning** — not just visual appearance. When you use `<article>` instead of `<div class="article">`, you are telling browsers, search engines, and assistive technologies exactly what that content represents.

## Why Semantic HTML Matters

<CardGrid>
  <Card title="Accessibility" icon="accessibility">
    Screen readers use semantic elements to navigate pages. Landmark elements like `<nav>` and `<main>` let users jump directly to key sections without reading every word.
  </Card>
  <Card title="SEO" icon="magnifier">
    Search engines understand your content better when you use appropriate elements. `<article>`, `<h1>`–`<h6>`, and structured content help your rankings.
  </Card>
  <Card title="Maintainability" icon="document">
    `<header>` is clearer than `<div id="top-section">`. Semantic code is self-documenting and easier for new developers to understand.
  </Card>
  <Card title="Styling" icon="pencil">
    Semantic elements provide natural CSS hooks. You can target `article > p` instead of `.content-area .text-wrapper p`.
  </Card>
</CardGrid>

## Page Landmark Elements

Landmark elements define the major regions of a page. Screen reader users can list all landmarks and jump directly to any one of them.

```html
<!DOCTYPE html>
<html lang="en">
  <head>
    <meta charset="UTF-8" />
    <title>News Site</title>
  </head>
  <body>
    <!-- Site-wide header -->
    <header>
      <a href="/" class="logo">My News</a>
      <!-- Primary site navigation -->
      <nav aria-label="Primary">
        <ul>
          <li><a href="/">Home</a></li>
          <li><a href="/world">World</a></li>
          <li><a href="/tech">Tech</a></li>
        </ul>
      </nav>
    </header>

    <!-- The single main content area (one per page) -->
    <main>
      <h1>Today's Top Stories</h1>

      <!-- Self-contained, independently distributable content -->
      <article>
        <h2>Scientists Discover New Exoplanet</h2>
        <p>Researchers at...</p>
      </article>

      <!-- Thematically grouped content within a larger context -->
      <section>
        <h2>Technology</h2>
        <article>
          <h3>New Chip Architecture Released</h3>
          <p>Silicon Valley's...</p>
        </article>
      </section>
    </main>

    <!-- Supplementary content tangentially related to the main content -->
    <aside aria-label="Trending topics">
      <h2>Trending</h2>
      <ul>
        <li><a href="#">Space Exploration</a></li>
      </ul>
    </aside>

    <!-- Site-wide footer -->
    <footer>
      <nav aria-label="Footer">
        <a href="/privacy">Privacy Policy</a>
        <a href="/contact">Contact</a>
      </nav>
      <p>&copy; 2025 My News</p>
    </footer>
  </body>
</html>
```

### Landmark Element Reference

| Element | Role | Notes |
|---|---|---|
| `<header>` | Introductory content | Can appear inside `<article>` or `<section>` too, not just `<body>` |
| `<nav>` | Navigation links | Use `aria-label` when multiple `<nav>` elements exist on a page |
| `<main>` | Dominant page content | Only **one** per page. Never nest inside `<article>`, `<aside>`, or `<footer>` |
| `<article>` | Self-contained content | Blog posts, news articles, forum posts, product cards — content that could stand alone |
| `<section>` | Thematic grouping | Groups related content that belongs together; should have a heading |
| `<aside>` | Tangentially related content | Sidebars, pull quotes, related links, advertising |
| `<footer>` | Footer for its nearest ancestor | Can appear inside `<article>` too |

:::tip[`<article>` vs `<section>` vs `<div>`]
Ask yourself: "Could this content be syndicated in an RSS feed or shared independently?" → `<article>`. "Is this a distinct thematic grouping that needs a heading?" → `<section>`. "Is this purely a styling container with no semantic meaning?" → `<div>`.
:::

## Figure and Figcaption

`<figure>` represents self-contained content referenced from the main flow — images, code listings, charts, or quotes. `<figcaption>` provides a caption or legend.

```html
<!-- Image with caption -->
<figure>
  <img
    src="/charts/q1-revenue.png"
    alt="Bar chart showing Q1 revenue by product category"
    width="800"
    height="450"
  />
  <figcaption>
    Figure 1: Q1 2025 revenue breakdown. Electronics led with 42% of total sales.
  </figcaption>
</figure>

<!-- Code listing with caption -->
<figure>
  <figcaption>Listing 1: Fibonacci sequence in Python</figcaption>
  <pre><code>def fib(n):
    a, b = 0, 1
    for _ in range(n):
        yield a
        a, b = b, a + b</code></pre>
</figure>

<!-- Blockquote as a figure -->
<figure>
  <blockquote>
    <p>The best way to predict the future is to invent it.</p>
  </blockquote>
  <figcaption>— Alan Kay</figcaption>
</figure>
```

## The `<time>` Element

`<time>` provides a machine-readable date or time alongside human-readable text:

```html
<!-- Article publish date -->
<article>
  <h2>Launch Announcement</h2>
  <p>Published on <time datetime="2025-03-15">March 15, 2025</time></p>
</article>

<!-- Time of day -->
<p>Doors open at <time datetime="19:30">7:30 PM</time>.</p>

<!-- Duration -->
<p>Total run time: <time datetime="PT2H30M">2 hours 30 minutes</time>.</p>

<!-- Date and time together -->
<p>
  The event starts
  <time datetime="2025-06-21T09:00:00-05:00">
    June 21, 2025 at 9:00 AM CDT
  </time>.
</p>
```

The `datetime` attribute uses the ISO 8601 format. Always include it so machines (and screen readers) understand the exact point in time even if the human-readable text says "yesterday" or "last Tuesday."

## The `<address>` Element

`<address>` marks up contact information for the nearest `<article>` or `<body>`:

```html
<footer>
  <address>
    Written by <a href="mailto:jane@example.com">Jane Doe</a>.<br />
    Visit us at:<br />
    Example Inc.<br />
    123 Main Street<br />
    Springfield, IL 62701
  </address>
</footer>
```

:::caution[`<address>` is for Contact Info Only]
`<address>` is not for all physical addresses — only for the contact information of the author or owner of the nearest article or document. Don't use it for an address inside a product listing, for example.
:::

## Text-Level Semantic Elements

```html
<!-- mark: highlighted/relevant text -->
<p>The search results show <mark>semantic HTML</mark> in context.</p>

<!-- abbr: abbreviation with expansion -->
<p>
  <abbr title="HyperText Markup Language">HTML</abbr> is the backbone of the web.
</p>

<!-- cite: title of a creative work -->
<p>According to <cite>The Art of ARIA</cite>, landmarks are essential.</p>

<!-- q: inline quotation -->
<p>
  The spec says <q cite="https://html.spec.whatwg.org">elements must not be nested
  in ways that are not permitted</q>.
</p>

<!-- blockquote: longer quotation from external source -->
<blockquote cite="https://www.w3.org/TR/WCAG21/">
  <p>
    Web content should be perceivable, operable, understandable, and robust.
  </p>
</blockquote>

<!-- strong: important text (bold semantically) -->
<p><strong>Warning:</strong> This action cannot be undone.</p>

<!-- em: stressed emphasis (italic semantically) -->
<p>I <em>really</em> want you to understand this.</p>

<!-- code: computer code -->
<p>Use the <code>display: flex</code> property to create a flex container.</p>

<!-- kbd: keyboard input -->
<p>Press <kbd>Ctrl</kbd> + <kbd>S</kbd> to save.</p>

<!-- samp: sample computer output -->
<p>The terminal displayed: <samp>Error: file not found</samp></p>

<!-- var: variable in math/programming -->
<p>The formula is <var>E</var> = <var>m</var><var>c</var>²</p>

<!-- sub / sup: subscript / superscript -->
<p>Water is H<sub>2</sub>O and E = mc<sup>2</sup></p>

<!-- small: fine print / side comments -->
<p>Price: $29.99 <small>(excluding tax)</small></p>
```

## `<details>` and `<summary>`

A native disclosure widget — no JavaScript required:

```html
<details>
  <summary>What is the return policy?</summary>
  <p>
    Items may be returned within 30 days of purchase with the original receipt.
    Sale items are final sale. See our full
    <a href="/returns">returns page</a> for details.
  </p>
</details>

<!-- Open by default -->
<details open>
  <summary>Technical Specifications</summary>
  <ul>
    <li>CPU: Apple M4</li>
    <li>RAM: 16 GB</li>
    <li>Storage: 512 GB SSD</li>
  </ul>
</details>
```

## The `<dialog>` Element

`<dialog>` is the native HTML modal — accessible, keyboard-trapped, and backed by browser behaviour:

```html
<button onclick="document.getElementById('confirm').showModal()">
  Delete Account
</button>

<dialog id="confirm">
  <h2>Confirm Deletion</h2>
  <p>Are you sure you want to delete your account? This cannot be undone.</p>
  <form method="dialog">
    <button value="cancel">Cancel</button>
    <button value="confirm" class="danger">Yes, delete my account</button>
  </form>
</dialog>
```

```js
const dialog = document.getElementById('confirm');

// Open as modal (traps focus, adds backdrop)
dialog.showModal();

// Open as non-modal (no backdrop, no focus trap)
dialog.show();

// Listen for close event
dialog.addEventListener('close', () => {
  console.log('User chose:', dialog.returnValue); // 'cancel' or 'confirm'
});
```

:::tip[Prefer `<dialog>` Over Custom Modals]
The native `<dialog>` element handles focus trapping, background scroll locking, the `Escape` key, and the `::backdrop` pseudo-element automatically. Custom modal implementations often miss one or more of these behaviours.
:::

## The `<template>` Element

`<template>` holds HTML that is not rendered immediately but can be cloned and inserted by JavaScript:

```html
<template id="card-template">
  <article class="card">
    <img class="card-image" src="" alt="" />
    <div class="card-body">
      <h3 class="card-title"></h3>
      <p class="card-description"></p>
    </div>
  </article>
</template>
```

```js
const template = document.getElementById('card-template');
const data = [
  { title: 'HTML', description: 'Structure', img: '/html.png' },
  { title: 'CSS', description: 'Style', img: '/css.png' },
];

data.forEach(item => {
  // Clone the template contents
  const clone = template.content.cloneNode(true);
  clone.querySelector('.card-title').textContent = item.title;
  clone.querySelector('.card-description').textContent = item.description;
  clone.querySelector('.card-image').src = item.img;
  clone.querySelector('.card-image').alt = item.title;
  document.querySelector('#card-container').appendChild(clone);
});
```

## `<div>` and `<span>` as a Last Resort

`<div>` and `<span>` have **no semantic meaning**. They are generic containers for when no semantic element fits. Use them only for styling or scripting hooks when there is no better option.

```html
<!-- ❌ Non-semantic -->
<div class="article">
  <div class="title">My Post</div>
  <div class="content"><p>Text here.</p></div>
</div>

<!-- ✅ Semantic -->
<article>
  <h2>My Post</h2>
  <p>Text here.</p>
</article>
```

## ARIA Roles When Semantics Aren't Enough

When native HTML semantics are insufficient (e.g. when building custom widgets), ARIA (Accessible Rich Internet Applications) roles supplement the element's meaning:

```html
<!-- Custom tab widget -->
<div role="tablist" aria-label="Product options">
  <button role="tab" aria-selected="true" aria-controls="panel-details" id="tab-details">
    Details
  </button>
  <button role="tab" aria-selected="false" aria-controls="panel-reviews" id="tab-reviews" tabindex="-1">
    Reviews
  </button>
</div>

<div role="tabpanel" id="panel-details" aria-labelledby="tab-details">
  <p>Product details content...</p>
</div>

<div role="tabpanel" id="panel-reviews" aria-labelledby="tab-reviews" hidden>
  <p>Customer reviews...</p>
</div>

<!-- Alert for important messages -->
<div role="alert" aria-live="assertive">
  Your session will expire in 5 minutes.
</div>

<!-- Progressbar -->
<div
  role="progressbar"
  aria-valuenow="65"
  aria-valuemin="0"
  aria-valuemax="100"
  aria-label="Upload progress"
>
  65%
</div>
```

:::caution[ARIA Rule #1: No ARIA is Better Than Bad ARIA]
Incorrect ARIA makes pages **less** accessible, not more. The first rule of ARIA is: if you can use a native HTML element with the required semantics, do so. ARIA should supplement, not replace, good semantic HTML.
:::

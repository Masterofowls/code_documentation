**Semantic HTML** means using elements that describe the *meaning* of content, not just its appearance. Instead of wrapping everything in `<div>` and `<span>`, you choose the element that best represents what the content *is*.

## Why Semantic HTML Matters

<CardGrid>
  <Card title="SEO" icon="magnifier">
    Search engines parse your HTML to understand what your page is about. Semantic elements give them clear signals — a `<article>` tells Google "this is a standalone piece of content".
</Card>
  <Card title="Accessibility" icon="heart">
    Screen readers and other assistive technologies use semantic elements to build a navigable page structure. Landmark regions, heading hierarchies, and widget roles all come from semantics.
</Card>
  <Card title="Maintainability" icon="pencil">
    Code that reads like English is easier to maintain. `<nav>` is instantly clearer than `<div id="nav-wrapper-outer">`. Future-you will thank present-you.
</Card>
  <Card title="Default Behaviour" icon="setting">
    Semantic elements come with built-in browser behaviour. `<button>` is keyboard-focusable and fires on Enter/Space. `<a>` is right-clickable. You get these for free.
</Card>
</CardGrid>

## Page Layout Landmarks

These elements define the major regions of a page. Assistive technologies expose them as "landmarks" that users can jump between directly.

```html
<!DOCTYPE html>
<html lang="en">
  <head>...</head>
  <body>

    <!-- Site-wide header: logo, site title, primary nav -->
    <header>
      <a href="/" aria-label="Home">
        <img src="logo.svg" alt="Acme Corp" width="120" height="40" />
      </a>

      <!-- Primary site navigation -->
      <nav aria-label="Primary">
        <ul>
          <li><a href="/products">Products</a></li>
          <li><a href="/blog">Blog</a></li>
          <li><a href="/contact">Contact</a></li>
        </ul>
      </nav>
    </header>

    <!-- The main content — only ONE per page -->
    <main id="main-content">

      <!-- A self-contained, independently distributable piece of content -->
      <article>
        <h1>Getting Started with Semantic HTML</h1>

        <!-- A thematic grouping within an article or page -->
        <section>
          <h2>What is Semantic HTML?</h2>
          <p>...</p>
        </section>

        <section>
          <h2>Why Does It Matter?</h2>
          <p>...</p>
        </section>
      </article>

      <!-- Content tangentially related to surrounding content -->
      <aside aria-label="Related articles">
        <h2>You might also like</h2>
        <ul>...</ul>
      </aside>

    </main>

    <!-- Site-wide footer -->
    <footer>
      <nav aria-label="Footer">
        <a href="/privacy">Privacy Policy</a>
        <a href="/terms">Terms of Service</a>
      </nav>
      <p><small>&copy; 2024 Acme Corp. All rights reserved.</small></p>
    </footer>

  </body>
</html>
```

### Element Reference

| Element | Landmark Role | Purpose |
|---|---|---|
| `<header>` | `banner` (when top-level) | Introductory content; site or section header |
| `<nav>` | `navigation` | A set of navigation links |
| `<main>` | `main` | The dominant content; one per page |
| `<article>` | `article` | Self-contained, redistributable content |
| `<section>` | `region` (if named) | Thematic grouping of related content |
| `<aside>` | `complementary` | Tangentially related content (sidebars) |
| `<footer>` | `contentinfo` (when top-level) | Footer for its nearest sectioning ancestor |

:::tip[article vs section]
Use `<article>` for content that makes sense on its own (a blog post, a product card, a comment). Use `<section>` to group related content within a larger context. When in doubt: if you could share it as a standalone item (RSS, email), it's an `<article>`.
:::

:::caution[Only One main Per Page]
The `<main>` element must not appear more than once per page and must not be a descendant of `<article>`, `<aside>`, `<footer>`, `<header>`, or `<nav>`.
:::

## `<figure>` and `<figcaption>`

Group media with its caption using `<figure>`. The caption is linked to the content semantically, unlike a plain `<p>`.

```html
<!-- Image with caption -->
<figure>
  <img
    src="architecture-diagram.png"
    alt="System architecture showing three microservices communicating via a message bus"
    width="900"
    height="500"
  />
  <figcaption>
    Figure 1: High-level architecture. The API gateway routes requests to
    individual microservices through a shared message bus.
  </figcaption>
</figure>

<!-- Code with caption -->
<figure>
  <pre><code class="language-js">
const greet = (name) => `Hello, ${name}!`;
  </code></pre>
  <figcaption>A simple arrow function using template literals.</figcaption>
</figure>

<!-- Blockquote with attribution -->
<figure>
  <blockquote>
    <p>The best way to predict the future is to invent it.</p>
  </blockquote>
  <figcaption>— Alan Kay</figcaption>
</figure>
```

## The `<time>` Element

The `<time>` element represents a specific time or duration. The `datetime` attribute provides a machine-readable value while the element content can be human-friendly.

```html
<!-- Date -->
<time datetime="2024-03-15">March 15, 2024</time>

<!-- Date and time (ISO 8601) -->
<time datetime="2024-03-15T14:30:00">2:30pm on 15 March</time>

<!-- Date with timezone offset -->
<time datetime="2024-03-15T14:30:00-05:00">March 15 at 2:30 PM EST</time>

<!-- Duration -->
<time datetime="PT2H30M">2 hours 30 minutes</time>

<!-- Month only -->
<time datetime="2024-03">March 2024</time>

<!-- In context -->
<article>
  <header>
    <h2>New Framework Released</h2>
    <p>
      Published by <a href="/authors/jane">Jane Smith</a> on
      <time datetime="2024-03-15">15 March 2024</time>
    </p>
  </header>
  <p>...</p>
</article>
```

## `<address>` Element

Represents contact information for the nearest `<article>` or `<body>` ancestor.

```html
<!-- Contact info for the document/site author -->
<address>
  <p>Written by <a href="mailto:jane@example.com">Jane Smith</a></p>
  <p>
    Acme Corp HQ<br />
    123 Tech Avenue<br />
    San Francisco, CA 94105
  </p>
</address>
```

:::caution[address is for Contact Info, Not Postal Addresses]
`<address>` is specifically for contact information related to the document or section — not for any arbitrary postal address in your content. Use `<p>` for addresses in running text.
:::

## Inline Semantic Elements

These elements add semantic meaning without creating block-level boxes.

<Tabs>
<TabItem label="Text Semantics">

```html
<!-- Highlight text of interest -->
<p>Search results for "JavaScript": found <mark>JavaScript</mark> frameworks.</p>

<!-- Abbreviation with expansion -->
<p>
  <abbr title="World Wide Web Consortium">W3C</abbr> maintains web standards.
</p>

<!-- Citation: title of a creative work -->
<p>I just finished reading <cite>Clean Code</cite> by Robert C. Martin.</p>

<!-- Short inline quotation (browser adds quote marks) -->
<p>Einstein said <q>Imagination is more important than knowledge.</q></p>

<!-- Inserted and deleted text (track changes) -->
<p>
  The price is <del>$29.99</del> <ins>$19.99</ins> — limited time offer!
</p>

<!-- Superscript and subscript -->
<p>E = mc<sup>2</sup></p>
<p>H<sub>2</sub>O is the chemical formula for water.</p>

<!-- Small print -->
<p><small>* Terms and conditions apply.</small></p>
```

</TabItem>

<!-- Long quotation with source -->

## `<details>` and `<summary>`

A native disclosure widget — no JavaScript required for basic expand/collapse:

```html
<details>
  <summary>What is the refund policy?</summary>
  <p>
    We offer a full refund within 30 days of purchase, no questions asked.
    After 30 days, we offer store credit at our discretion.
  </p>
</details>

<!-- Open by default -->
<details open>
  <summary>System Requirements</summary>
  <ul>
    <li>Node.js 20 or later</li>
    <li>4GB RAM minimum, 8GB recommended</li>
    <li>2GB free disk space</li>
  </ul>
</details>
```

## `<dialog>` Element

The native modal dialog element. Far better than `<div role="dialog">` for accessibility:

```html
<dialog id="confirm-dialog" aria-labelledby="dialog-title">
  <h2 id="dialog-title">Confirm Deletion</h2>
  <p>Are you sure you want to delete this item? This action cannot be undone.</p>
  <menu>
    <button type="button" id="cancel-btn">Cancel</button>
    <button type="button" id="confirm-btn" autofocus>Delete</button>
  </menu>
</dialog>

<button type="button" id="open-dialog">Delete Item</button>

<script>
  const dialog = document.getElementById('confirm-dialog');
  const openBtn = document.getElementById('open-dialog');
  const cancelBtn = document.getElementById('cancel-btn');

  // showModal() traps focus inside the dialog
  openBtn.addEventListener('click', () => dialog.showModal());
  cancelBtn.addEventListener('click', () => dialog.close());
</script>
```

:::tip[Use showModal() Not show()]
`dialog.showModal()` opens a true modal that traps keyboard focus inside and renders a `::backdrop` pseudo-element. `dialog.show()` opens a non-modal dialog that does not trap focus.
:::

## `<template>` Element

Content inside `<template>` is parsed but not rendered or executed. It acts as a client-side content template.

```html
<template id="card-template">
  <article class="card">
    <img src="" alt="" class="card-image" />
    <div class="card-body">
      <h2 class="card-title"></h2>
      <p class="card-description"></p>
      <a href="#" class="card-link">Read more</a>
    </div>
  </article>
</template>

<div id="card-container"></div>

<script>
  const template = document.getElementById('card-template');
  const container = document.getElementById('card-container');

  const data = [
    { title: 'Article One', description: 'A brief summary.', href: '/one' },
    { title: 'Article Two', description: 'Another summary.', href: '/two' },
  ];

  data.forEach(({ title, description, href }) => {
    // importNode clones the template content
    const clone = template.content.cloneNode(true);
    clone.querySelector('.card-title').textContent = title;
    clone.querySelector('.card-description').textContent = description;
    clone.querySelector('.card-link').href = href;
    container.appendChild(clone);
  });
</script>
```

## `<div>` and `<span>` as Last Resort

`<div>` and `<span>` are generic containers with **no semantic meaning**. They exist purely for styling and scripting hooks. Use them only when no semantic element fits.

```html
<!-- Bad: div soup -->
<div class="article">
  <div class="article-header">
    <div class="article-title">My Blog Post</div>
  </div>
  <div class="article-body">
    <div class="paragraph">Some text here.</div>
  </div>
</div>

<!-- Good: semantic HTML -->
<article>
  <header>
    <h1>My Blog Post</h1>
  </header>
  <p>Some text here.</p>
</article>
```

## ARIA Roles When Semantics Are Not Enough

Sometimes you must use a non-semantic element (e.g. a `<div>` that acts as a button in a component framework). ARIA roles fill the gap:

```html
<!-- Custom interactive component -->
<div
  role="button"
  tabindex="0"
  aria-pressed="false"
  class="toggle-btn"
  id="dark-mode-toggle"
>
  Dark Mode
</div>

<!-- Keyboard handler needed — native <button> handles this automatically -->
<script>
  const btn = document.getElementById('dark-mode-toggle');

  btn.addEventListener('click', toggle);
  btn.addEventListener('keydown', (e) => {
    if (e.key === 'Enter' || e.key === ' ') {
      e.preventDefault();
      toggle();
    }
  });

  function toggle() {
    const pressed = btn.getAttribute('aria-pressed') === 'true';
    btn.setAttribute('aria-pressed', String(!pressed));
  }
</script>
```

:::caution[First Rule of ARIA]
The first rule of ARIA is: **don't use ARIA if you can use a native HTML element instead**. `<button>` gives you keyboard focus, click events, and role "button" for free. `<div role="button">` requires you to re-implement all of that manually.
:::

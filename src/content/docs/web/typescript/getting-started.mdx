## Installation

TypeScript is distributed as an npm package. Install it as a development dependency — it is never needed at runtime.

<Tabs>
  <TabItem label="npm">
```bash
npm install --save-dev typescript
```
</TabItem>
  <TabItem label="pnpm">
```bash
pnpm add -D typescript
```
</TabItem>
  <TabItem label="yarn">
```bash
yarn add --dev typescript
```
</TabItem>
  <TabItem label="bun">
```bash
bun add -d typescript
```
</TabItem>
</Tabs>

Verify the installation:

```bash
npx tsc --version
# Version 5.5.4
```

:::tip[Global vs Local]
Prefer local (project-level) TypeScript installations over a global one. This ensures every contributor and CI environment uses the same compiler version, preventing "works on my machine" type errors.
:::

## Initialising `tsconfig.json`

The `tsconfig.json` file is the control centre for the TypeScript compiler. Generate one with sensible defaults:

```bash
npx tsc --init
```

This creates a `tsconfig.json` with every option documented inline. Below is a curated, annotated version for a modern Node.js project:

```ts
// tsconfig.json
{
  "compilerOptions": {
    // ── Output ────────────────────────────────────────────────
    "target": "ES2022",          // JavaScript version to emit
    "module": "NodeNext",        // Module system for emitted JS
    "moduleResolution": "NodeNext", // How imports are resolved
    "outDir": "./dist",          // Where compiled JS files go
    "rootDir": "./src",          // Root of TypeScript source files
    "declaration": true,         // Emit .d.ts declaration files
    "declarationMap": true,      // Emit .d.ts.map for go-to-definition
    "sourceMap": true,           // Emit .js.map source maps

    // ── Strict Type Checking ──────────────────────────────────
    "strict": true,              // Enable ALL strict checks below:
    //   noImplicitAny            – error on implicit any types
    //   strictNullChecks         – null/undefined are separate types
    //   strictFunctionTypes      – stricter function compatibility
    //   strictBindCallApply      – bind/call/apply are type-safe
    //   strictPropertyInitialization – class props must be initialised
    //   noImplicitThis           – error on implicit `this` type
    //   useUnknownInCatchVariables – catch clause var is `unknown`

    // ── Additional Checks ────────────────────────────────────
    "noUnusedLocals": true,       // Error on unused local variables
    "noUnusedParameters": true,   // Error on unused function params
    "noImplicitReturns": true,    // All code paths must return
    "noFallthroughCasesInSwitch": true, // switch case fall-through
    "noUncheckedIndexedAccess": true,   // arr[i] returns T | undefined
    "exactOptionalPropertyTypes": true, // ? props may not be set to undefined

    // ── Paths & Aliases ──────────────────────────────────────
    "baseUrl": ".",
    "paths": {
      "@/*": ["src/*"],           // @/utils → src/utils
      "@components/*": ["src/components/*"]
    },

    // ── Interop ─────────────────────────────────────────────
    "esModuleInterop": true,      // Allow default imports from CJS
    "allowSyntheticDefaultImports": true,
    "forceConsistentCasingInFileNames": true,
    "skipLibCheck": true          // Skip type-check of .d.ts files
  },

  // ── File Inclusion ────────────────────────────────────────
  "include": ["src/**/*", "tests/**/*"],
  "exclude": ["node_modules", "dist", "**/*.spec.ts"]
}
```

### Key Options Explained

| Option | Values | Purpose |
|---|---|---|
| `target` | `ES5`, `ES2020`, `ES2022`, `ESNext` | Output JS language level |
| `module` | `CommonJS`, `ESNext`, `NodeNext`, `Bundler` | Module format of emitted files |
| `moduleResolution` | `Node`, `NodeNext`, `Bundler` | Algorithm for resolving imports |
| `strict` | `true` / `false` | Bundle of strict type-checking flags |
| `outDir` | `"./dist"` | Destination for compiled output |
| `rootDir` | `"./src"` | TypeScript source root |
| `baseUrl` | `"."` | Base for non-relative module resolution |
| `paths` | `{ "@/*": ["src/*"] }` | Import aliases (requires bundler support too) |
| `declaration` | `true` / `false` | Emit `.d.ts` type declaration files |
| `skipLibCheck` | `true` / `false` | Skip checking `node_modules` `.d.ts` files |

:::caution[target vs module]
`target` controls the **JavaScript syntax** in the output (e.g., arrow functions, `class`, `async/await`). `module` controls the **module format** (`require()` vs `import`). They are independent settings — you can emit ES2022 syntax with CommonJS modules.
:::

## Your First TypeScript File

<Steps>
1. **Create a source directory and file**
   ```bash
   mkdir src
   ```

   ```ts
   // src/index.ts
   interface User {
     id: number;
     name: string;
     email: string;
     role: 'admin' | 'user' | 'guest';
   }

   function formatUser(user: User): string {
     return `[${user.role.toUpperCase()}] ${user.name} <${user.email}>`;
   }

   const alice: User = {
     id: 1,
     name: 'Alice Nguyen',
     email: 'alice@example.com',
     role: 'admin',
   };

   console.log(formatUser(alice));
   // [ADMIN] Alice Nguyen <alice@example.com>
   ```

2. **Compile to JavaScript**
   ```bash
   npx tsc
   # Outputs dist/index.js
   ```

3. **Run with Node.js**
   ```bash
   node dist/index.js
   # [ADMIN] Alice Nguyen <alice@example.com>
   ```
</Steps>

## Type Annotations

TypeScript infers types automatically in most cases. Explicit annotations are used when inference falls short or for documentation.

### Primitive Types

```ts
// Explicit annotations
const name: string = 'TypeScript';
const version: number = 5.5;
const isStable: boolean = true;
const nothing: null = null;
const unset: undefined = undefined;
const id: bigint = 9007199254740993n;
const sym: symbol = Symbol('unique');

// Type inference — annotation is redundant but valid
const inferred = 'TypeScript'; // type: string (inferred)
```

### Function Annotations

```ts
// Parameter types + return type
function add(a: number, b: number): number {
  return a + b;
}

// Arrow function
const multiply = (x: number, y: number): number => x * y;

// Optional parameter with default
function greet(name: string, greeting: string = 'Hello'): string {
  return `${greeting}, ${name}!`;
}

// Void return type
function logMessage(msg: string): void {
  console.log(msg);
  // Returning a value here is a type error:
  // return msg; // Error: Type 'string' is not assignable to type 'void'.
}

// Rest parameters
function sum(...nums: number[]): number {
  return nums.reduce((acc, n) => acc + n, 0);
}

console.log(add(2, 3));       // 5
console.log(sum(1, 2, 3, 4)); // 10
```

### Variable Declarations

```ts
// let — mutable, widened type
let count = 0;        // type: number
count = 10;           // OK
// count = 'ten';     // Error: Type 'string' is not assignable to type 'number'.

// const — immutable reference, narrower type
const MAX = 100;      // type: 100 (literal type, not number)
const PI = 3.14159;   // type: 3.14159

// Arrays
const numbers: number[] = [1, 2, 3];
const names: Array<string> = ['Alice', 'Bob'];

// Tuples — fixed-length, typed positions
const point: [number, number] = [10, 20];
const entry: [string, number] = ['age', 30];
```

## Running TypeScript Directly

For development, you often don't want a manual compile step. Two popular tools execute TypeScript files directly:

### ts-node

```bash
npm install --save-dev ts-node
npx ts-node src/index.ts
```

`ts-node` performs full type-checking before execution. Useful for scripts where type safety matters.

```ts
// scripts/seed.ts
import { createUser } from './db';

async function main(): Promise<void> {
  await createUser({ name: 'Alice', role: 'admin' });
  console.log('Database seeded.');
}

main().catch(console.error);
```

```bash
npx ts-node scripts/seed.ts
```

### tsx

```bash
npm install --save-dev tsx
npx tsx src/index.ts
npx tsx watch src/index.ts  # auto-restart on file change
```

:::tip[tsx for speed]
`tsx` uses esbuild to strip types and execute — it does **not** type-check. It is 10–50× faster than `ts-node` for scripts and dev servers. Run `tsc --noEmit` separately for type safety.
:::

## VS Code Integration

VS Code has first-class TypeScript support built in via the TypeScript Language Service. No extensions required for core functionality.

### IntelliSense Features

- **Autocomplete** — property names, method signatures, import paths
- **Hover types** — hover any symbol to see its inferred type
- **Error squiggles** — red underlines for type errors, yellow for warnings
- **Go to Definition** — `F12` or `Ctrl+Click` jumps to the type definition
- **Find All References** — see every usage of a symbol across the project
- **Rename Symbol** — `F2` renames a symbol and all its references

```ts
// VS Code will show:
// - Autocomplete after typing `user.`
// - A hover tooltip showing: (property) User.name: string
// - An error squiggle on `user.naem` with the message:
//   "Property 'naem' does not exist on type 'User'. Did you mean 'name'?"

interface User {
  name: string;
  age: number;
}

const user: User = { name: 'Alice', age: 30 };
console.log(user.naem);
//               ^^^^ red squiggle
```

### Selecting the Workspace TypeScript Version

By default, VS Code uses its bundled TypeScript. To use the project's installed version:

1. Open a `.ts` file
2. Press `Ctrl+Shift+P` → search **"TypeScript: Select TypeScript Version"**
3. Select **"Use Workspace Version"**

This ensures VS Code uses the same compiler version as your CI pipeline.

## `@types` Packages and Declaration Files

TypeScript understands JavaScript libraries through **declaration files** (`.d.ts`). These describe the shapes of values exported by a library without containing any implementation.

### Installing Type Declarations

Many libraries ship their own types. Others rely on the **DefinitelyTyped** community repository:

```bash
# Library ships its own types (no extra install needed):
npm install zod           # types included
npm install prisma        # types included

# Library needs @types:
npm install --save-dev @types/node
npm install --save-dev @types/react @types/react-dom
npm install --save-dev @types/lodash
npm install --save-dev @types/express
```

### Checking if a Package Includes Types

```bash
# Method 1: Check the package.json of the library
cat node_modules/lodash/package.json | grep '"types"'
# If missing, check DefinitelyTyped

# Method 2: Use the TypeScript playground or arethetypeswrong.github.io
```

### Writing Your Own Declaration File

When consuming a JavaScript library without existing types:

```ts
// types/my-legacy-lib.d.ts
declare module 'my-legacy-lib' {
  export interface Options {
    timeout: number;
    retries?: number;
  }

  export function connect(url: string, options?: Options): Promise<void>;
  export function disconnect(): void;

  export const version: string;
}
```

Then reference it in `tsconfig.json`:

```ts
// tsconfig.json
{
  "compilerOptions": {
    "typeRoots": ["./types", "./node_modules/@types"]
  }
}
```

### The `lib` Option

TypeScript includes built-in declarations for browser and runtime APIs. The `lib` option selects which ones to include:

```ts
// tsconfig.json
{
  "compilerOptions": {
    "target": "ES2022",
    // lib defaults to target-appropriate built-ins if omitted.
    // Override explicitly for fine-grained control:
    "lib": ["ES2022", "DOM", "DOM.Iterable"]
    // ES2022 – Promise.allSettled, Array.at, Object.hasOwn, etc.
    // DOM – window, document, HTMLElement, fetch, etc.
    // DOM.Iterable – for...of over NodeList, HTMLCollection, etc.
  }
}
```

## Project Structure Conventions

```
my-project/
├── src/
│   ├── index.ts          # Entry point
│   ├── types/            # Shared type definitions
│   │   └── index.ts
│   ├── utils/
│   │   └── format.ts
│   └── services/
│       └── user.ts
├── tests/
│   └── user.test.ts
├── dist/                 # Compiled output (git-ignored)
├── tsconfig.json
├── tsconfig.build.json   # Production build (excludes tests)
├── package.json
└── .gitignore
```

```ts
// tsconfig.build.json — used for production builds
{
  "extends": "./tsconfig.json",
  "exclude": ["tests/**/*", "**/*.test.ts", "**/*.spec.ts"]
}
```

```bash
# Development: full project with tests
npx tsc --noEmit

# Production build: exclude tests
npx tsc --project tsconfig.build.json
```

:::tip[Separate build config]
Always keep a `tsconfig.build.json` that excludes test files. This keeps your `dist/` clean and speeds up production builds. Your root `tsconfig.json` remains the source of truth for editor and CI type-checking.
:::

## Common First-Time Errors

```ts
// Error 1: Object is possibly 'null' (strictNullChecks)
const el = document.getElementById('app');
el.innerHTML = 'Hello'; // Error: Object is possibly 'null'.
// Fix:
el!.innerHTML = 'Hello';      // non-null assertion (use carefully)
el?.setAttribute('data-ready', 'true'); // optional chaining
if (el) { el.innerHTML = 'Hello'; }     // type guard (preferred)

// Error 2: Implicit any parameter
function double(n) { return n * 2; } // Error: Parameter 'n' implicitly has an 'any' type.
// Fix:
function double(n: number): number { return n * 2; }

// Error 3: Property does not exist
const obj = { name: 'Alice' };
console.log(obj.age); // Error: Property 'age' does not exist on type '{ name: string; }'.
// Fix: add the property to the type or use a type with that property

// Error 4: Type not assignable
const nums: number[] = [1, 2, 3];
nums.push('four'); // Error: Argument of type 'string' is not assignable to parameter of type 'number'.
```

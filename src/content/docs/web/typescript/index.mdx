TypeScript is an **open-source, statically typed superset of JavaScript** developed and maintained by Microsoft. Every valid JavaScript file is also a valid TypeScript file, but TypeScript adds optional static typing, interfaces, generics, decorators, and a rich type system that vanishes entirely at runtime — leaving behind clean, spec-compliant JavaScript.

## Why Static Typing?

Dynamic languages like JavaScript are fast to prototype but can hide whole classes of bugs until they surface in production. TypeScript catches those errors at **compile time**, before your code ever runs.

```ts
// JavaScript — no error until runtime
function greet(user) {
  return 'Hello, ' + user.naem; // typo: .naem instead of .name
}

// TypeScript — error at compile time
function greet(user: { name: string }): string {
  return 'Hello, ' + user.naem;
  //                       ^^^^
  // Error: Property 'naem' does not exist on type '{ name: string }'.
  // Did you mean 'name'?
}
```

### Key Benefits

| Benefit                   | Description                                                                       |
| ------------------------- | --------------------------------------------------------------------------------- |
| **Early error detection** | Catch typos, wrong argument types, and missing properties before runtime          |
| **IDE Intelligence**      | Autocomplete, refactoring, go-to-definition, and inline docs powered by types     |
| **Self-documenting code** | Function signatures act as inline contracts — no guessing what a function expects |
| **Safe refactoring**      | Rename a property and the compiler tells you every callsite that needs updating   |
| **Gradual adoption**      | Migrate file-by-file; `allowJs` lets TS and JS coexist in one project             |
| **Ecosystem breadth**     | Type definitions for thousands of npm packages via DefinitelyTyped (`@types/*`)   |

## The Compilation Pipeline

TypeScript source (`.ts`) is never executed directly by browsers or Node.js. It must first be **compiled** (or _transpiled_) to JavaScript.

```
  Source files (.ts / .tsx)
         │
         ▼
  TypeScript Compiler (tsc)
    ┌─────────────────┐
    │  1. Parse        │  → AST (Abstract Syntax Tree)
    │  2. Type-check   │  → Emit diagnostics / errors
    │  3. Emit         │  → .js  +  .d.ts  +  .js.map
    └─────────────────┘
         │
         ▼
  JavaScript output (.js)
  Declaration files (.d.ts)
  Source maps (.js.map)
```

:::tip[Source Maps]
Source maps link compiled `.js` lines back to original `.ts` lines, so stack traces and debugger breakpoints reference your TypeScript source — not the generated JavaScript.
:::

### Compilation Example

```ts
// src/hello.ts
interface User {
  name: string;
  age: number;
}

function greet(user: User): string {
  return `Hello, ${user.name}! You are ${user.age} years old.`;
}

const alice: User = { name: 'Alice', age: 30 };
console.log(greet(alice));
```

```bash
npx tsc src/hello.ts --target ES2020
```

```ts
// dist/hello.js  (compiled output — all types erased)
function greet(user) {
  return `Hello, ${user.name}! You are ${user.age} years old.`;
}
const alice = { name: 'Alice', age: 30 };
console.log(greet(alice));
```

## TypeScript vs JavaScript

<Tabs>
  <TabItem label="TypeScript">
```ts
// Explicit types, interfaces, and generics
interface Product {
  id: number;
  name: string;
  price: number;
  tags?: string[];
}

function getDiscount<T extends { price: number }>(
item: T,
percent: number
): number {
return item.price \* (1 - percent / 100);
}

const laptop: Product = {
id: 1,
name: 'ThinkPad X1',
price: 1499.99,
tags: ['laptop', 'business'],
};

// Error caught at compile time — not runtime:
// const bad: Product = { id: '1', name: 'x', price: 0 };
// Error: Type 'string' is not assignable to type 'number'.

console.log(getDiscount(laptop, 10)); // 1349.991

````
</TabItem>
  <TabItem label="JavaScript">
```ts
// No types — bugs surface at runtime
function getDiscount(item, percent) {
  return item.price * (1 - percent / 100);
}

const laptop = {
  id: 1,
  name: 'ThinkPad X1',
  price: 1499.99,
};

// No error here — silently wrong at runtime:
const bad = { id: '1', name: 'x', price: 0 };

console.log(getDiscount(laptop, 10));    // 1349.991
console.log(getDiscount(bad, 'ten'));    // NaN  ← silent bug
console.log(getDiscount(undefined, 10)); // TypeError at runtime
````

</TabItem>
</Tabs>

## TypeScript Version History

| Version | Release  | Headline Features                                                                        |
| ------- | -------- | ---------------------------------------------------------------------------------------- |
| **3.0** | Jul 2018 | Project references, unknown type, `--build` mode                                         |
| **3.7** | Nov 2019 | Optional chaining (`?.`), nullish coalescing (`??`), assertion functions                 |
| **3.9** | May 2020 | Speed improvements, `// @ts-expect-error`                                                |
| **4.0** | Aug 2020 | Variadic tuple types, labeled tuple elements, class static blocks                        |
| **4.1** | Nov 2020 | Template literal types, recursive conditional types, `remapped` mapped types             |
| **4.2** | Feb 2021 | Leading/middle rest elements in tuples, abstract construct signatures                    |
| **4.4** | Aug 2021 | Control flow analysis for aliased conditions, symbol index signatures                    |
| **4.5** | Nov 2021 | `Awaited&lt;T&gt;` type, `type` modifier on import names, tail-recursion elimination     |
| **4.7** | May 2022 | `NodeNext` / `Node16` module resolution, `infer` variance, instantiation expressions     |
| **4.9** | Nov 2022 | `satisfies` operator, `in` narrowing improvements, auto-accessors                        |
| **5.0** | Mar 2023 | `const` type parameters, `--moduleResolution bundler`, decorator standard (TC39 Stage 3) |
| **5.1** | May 2023 | Unrelated return types for `get`/`set`, JSX improved                                     |
| **5.2** | Aug 2023 | `using` keyword (Explicit Resource Management), decorator metadata                       |
| **5.3** | Nov 2023 | `import` attribute coercions, narrowing improvements for `switch(true)`                  |
| **5.4** | Mar 2024 | `NoInfer&lt;T&gt;` utility type, `Object.groupBy`, closure narrowing improvements        |
| **5.5** | Jun 2024 | Inferred type predicates, isolated declarations, JSDoc `@import`                         |

:::caution[Staying Current]
TypeScript ships new releases roughly every 3 months. Subscribe to the [TypeScript blog](https://devblogs.microsoft.com/typescript/) to stay informed about breaking changes before upgrading.
:::

## The TypeScript Ecosystem

### Core Tooling

<CardGrid>
  <Card title="tsc" icon="seti:typescript">
    The official TypeScript compiler. Use it for type-checking and emitting declarations. Configure via `tsconfig.json`.

    ```bash
    npx tsc --noEmit     # type-check only
    npx tsc --watch      # watch mode
    npx tsc --build      # project references
    ```

</Card>

  <Card title="ts-node" icon="seti:node">
    Run TypeScript directly in Node.js without a separate compile step. Great for scripts and REPLs.

    ```bash
    npx ts-node src/index.ts
    npx ts-node --esm src/index.ts
    ```

</Card>

  <Card title="tsx" icon="seti:typescript">
    A faster alternative to ts-node using esbuild under the hood. No type-checking — just executes.

    ```bash
    npx tsx src/index.ts
    npx tsx watch src/index.ts
    ```

</Card>

  <Card title="esbuild" icon="rocket">
    Extremely fast bundler/transpiler written in Go. Strips types but does **not** type-check.

    ```bash
    npx esbuild src/index.ts \
      --bundle --outfile=dist/index.js
    ```

</Card>
</CardGrid>

### Build Tooling Comparison

| Tool      | Type-checks?         | Speed     | Use case                           |
| --------- | -------------------- | --------- | ---------------------------------- |
| `tsc`     | ✅ Yes               | Moderate  | CI type-checking, declaration emit |
| `ts-node` | ✅ Yes (via tsc)     | Slower    | Development REPL, scripts          |
| `tsx`     | ❌ No                | Very fast | Dev server scripts, quick runs     |
| `esbuild` | ❌ No                | Fastest   | Production bundling                |
| `swc`     | ❌ No                | Very fast | Babel replacement in bundlers      |
| `Vite`    | ❌ No (uses esbuild) | Fast      | Frontend dev server + build        |

:::tip[Recommended Workflow]
Use **tsc** (or `tsc --noEmit`) in CI for type safety, and **tsx** or **Vite** in development for fast feedback. Never skip `tsc --noEmit` in your pipeline — esbuild and tsx silently ignore type errors.
:::

## How Types Are Erased

TypeScript types are purely a **compile-time construct**. They produce zero JavaScript output:

```ts
// TypeScript source
type UserId = string & { readonly __brand: unique symbol };
interface Config {
  apiUrl: string;
  timeout: number;
}
const x: Config = { apiUrl: 'https://api.example.com', timeout: 5000 };
```

```ts
// Compiled JavaScript — types completely erased
const x = { apiUrl: 'https://api.example.com', timeout: 5000 };
```

This means:

- **Zero runtime overhead** from types
- **No reflection** on types at runtime (without `reflect-metadata`)
- Type assertions (`as`) are also erased — they do not throw at runtime

## Quick Start

<Steps>
1. **Install TypeScript**
   ```bash
   npm install --save-dev typescript
   ```

2. **Generate a `tsconfig.json`**

   ```bash
   npx tsc --init
   ```

3. **Write your first `.ts` file**

   ```ts
   // src/index.ts
   const message: string = 'Hello, TypeScript!';
   console.log(message);
   ```

4. **Compile and run**
   ```bash
   npx tsc
   node dist/index.js
   ```
</Steps>

This guide walks through every syntactic building block of XML, from the opening declaration to namespaces and character entities. By the end you will be able to read and write any well-formed XML document.

---

## The XML Declaration

Every XML file should begin with an XML declaration. It is technically optional for XML 1.0 (it is assumed when omitted) but is mandatory for XML 1.1.

```xml
<?xml version="1.0" encoding="UTF-8"?>
```

| Attribute | Purpose | Common Values |
|---|---|---|
| `version` | XML specification version | `"1.0"` (most common), `"1.1"` |
| `encoding` | Character encoding | `"UTF-8"`, `"UTF-16"`, `"ISO-8859-1"` |
| `standalone` | Whether external DTD is required | `"yes"` or `"no"` |

```xml
<?xml version="1.0" encoding="UTF-8" standalone="yes"?>
```

:::tip[Always Use UTF-8]
Use `encoding="UTF-8"` unless you have a specific reason not to. UTF-8 can represent every Unicode character, is the default for modern tools, and avoids encoding mismatches between systems.
:::

## Elements

An **element** is the fundamental building block of XML. It consists of a start tag, content, and an end tag.

```xml
<title>The Pragmatic Programmer</title>
<!--  ^start tag   ^content      ^end tag -->
```

### Rules for Element Names

- Must start with a letter or underscore (`_`)
- Cannot start with a digit or hyphen
- Cannot start with `xml` (reserved for W3C)
- Can contain letters, digits, hyphens, underscores, and periods
- Case-sensitive: `<Title>` and `<title>` are different elements

```xml
<!-- Valid element names -->
<firstName>Alice</firstName>
<_internal>data</_internal>
<ns:element>value</ns:element>
<camelCase>ok</camelCase>
<kebab-case>ok</kebab-case>
<dot.separated>ok</dot.separated>

<!-- Invalid element names ❌ -->
<!-- <1name>bad</1name>      — starts with digit -->
<!-- <xml-reserved>bad</xml-reserved> — starts with xml -->
<!-- <my element>bad</my element>     — space in name -->
```

### Nesting Rules

XML elements must be **properly nested** — a child must close before its parent closes. Overlapping tags make a document ill-formed.

```xml
<!-- Correct nesting ✅ -->
<outer>
  <inner>
    <deepest>value</deepest>
  </inner>
</outer>

<!-- Overlapping — ill-formed ❌ -->
<bold><italic>text</bold></italic>
```

### The Root Element

Every XML document must have **exactly one root element** that contains all other elements.

```xml
<?xml version="1.0" encoding="UTF-8"?>
<catalog>          <!-- root element -->
  <product>...</product>
  <product>...</product>
</catalog>
```

---

## Self-Closing Tags

An element with no content can be written as a self-closing (empty-element) tag:

```xml
<!-- These are equivalent -->
<br></br>
<br/>

<!-- Real-world self-closing elements -->
<img src="photo.jpg" alt="A photo"/>
<input type="text" name="username"/>
<meta charset="UTF-8"/>
```

:::caution[XHTML vs HTML]
In XHTML (XML-based HTML), self-closing tags require the slash: `<br />`. In HTML5, `<br>` is sufficient. When writing XML-embedded XHTML, always use the slash form.
:::

## Attributes

Attributes are name-value pairs placed inside a start tag. Values must always be quoted (single or double quotes).

```xml
<product id="42" category="books" in-stock="true">
  <name>Clean Code</name>
  <price currency="USD">35.99</price>
</product>
```

### Attributes vs Child Elements

Both approaches can represent the same data. Choose based on semantics:

```xml
<!-- Attribute approach — good for metadata/identifiers -->
<user id="123" role="admin" created="2024-01-15"/>

<!-- Child element approach — good for data that is queried/transformed -->
<user>
  <id>123</id>
  <role>admin</role>
  <created>2024-01-15</created>
</user>
```

**General guideline:** Use attributes for short, atomic, identifier-like values. Use child elements for data that has sub-structure, may be repeated, or will be displayed/transformed.

---

## Text Content

Text between tags is called a **text node**:

```xml
<description>This is plain text content with a number: 42.</description>
```

Text content can be mixed with child elements — this is called **mixed content** and is common in document-centric XML:

```xml
<paragraph>This is <emphasis>very</emphasis> important text with a
<link href="https://example.com">hyperlink</link> inside.</paragraph>
```

## CDATA Sections

A CDATA (Character Data) section lets you embed text that contains characters that would normally need escaping (`<`, `&`, etc.) without escaping them. This is useful for embedding code or HTML snippets:

```xml
<script>
  <![CDATA[
    function compare(a, b) {
      if (a < b) return -1;
      if (a > b) return 1;
      return 0;
    }
    const result = compare(3 & 5, 10 && true);
  ]]>
</script>
```

CDATA sections begin with `<![CDATA[` and end with `]]>`. The only string that cannot appear inside a CDATA section is `]]>`.

---

## Comments

XML comments use the same syntax as HTML comments:

```xml
<!-- This is a comment -->

<!--
  Multi-line comment explaining
  the structure of this document
-->

<data>
  <!-- TODO: add validation -->
  <value>42</value>
</data>
```

:::caution[Comment Restrictions]
Comments cannot contain `--` (double hyphen) anywhere inside the comment body, except as the closing `-->`. The string `--` within a comment is illegal in XML 1.0.
:::

---

## Character Entities

Certain characters have special meaning in XML and must be escaped using **predefined character entities**:

| Character | Entity | When Required |
|---|---|---|
| `<` | `&lt;` | In element content and attribute values |
| `>` | `&gt;` | Technically only required after `]]` in content |
| `&` | `&amp;` | In element content and attribute values |
| `'` | `&apos;` | In attribute values delimited by single quotes |
| `"` | `&quot;` | In attribute values delimited by double quotes |

```xml
<!-- Using character entities -->
<expression>5 &lt; 10 &amp;&amp; 10 &gt; 3</expression>
<message>She said &quot;Hello&quot; and it&apos;s fine</message>

<!-- Numeric character references (decimal and hexadecimal) -->
<symbol>&#169;</symbol>   <!-- © copyright sign -->
<emoji>&#x1F600;</emoji>  <!-- 😀 grinning face -->
<euro>&#8364;</euro>      <!-- € euro sign -->
```

## Processing Instructions

Processing instructions (PIs) pass information to the application processing the XML document. They are not part of the data:

```xml
<?xml version="1.0" encoding="UTF-8"?>
<?xml-stylesheet type="text/xsl" href="transform.xsl"?>
<?xml-stylesheet type="text/css" href="style.css"?>

<document>
  <?php echo "This is a PHP PI (used historically)"; ?>
  <content>...</content>
</document>
```

The `<?xml-stylesheet?>` PI is standardised and tells parsers which stylesheet to apply to the document.

## Namespaces

XML namespaces prevent element and attribute name collisions when combining XML from different vocabularies.

### Default Namespace

A default namespace applies to all unprefixed elements in its scope:

```xml
<html xmlns="http://www.w3.org/1999/xhtml">
  <head>
    <title>XHTML Page</title>
  </head>
  <body>
    <p>All these elements are in the XHTML namespace.</p>
  </body>
</html>
```

### Namespace Prefixes

A prefix maps a short string to a namespace URI:

```xml
<?xml version="1.0" encoding="UTF-8"?>
<root
  xmlns:books="http://example.com/books"
  xmlns:store="http://example.com/store">

  <books:catalog>
    <books:book id="1">
      <books:title>Effective Java</books:title>
      <store:price currency="USD">49.99</store:price>
      <store:available>true</store:available>
    </books:book>
  </books:catalog>

</root>
```

:::tip[Namespace URIs Are Not URLs]
The namespace URI (like `http://example.com/books`) is just a unique identifier — it does not have to point to anything accessible on the internet. URIs are used because they are easy to make globally unique.
:::

### Combining Multiple Vocabularies

This is where namespaces shine — mixing XHTML, SVG, and MathML in a single document:

```xml
<?xml version="1.0" encoding="UTF-8"?>
<html xmlns="http://www.w3.org/1999/xhtml"
      xmlns:svg="http://www.w3.org/2000/svg"
      xmlns:math="http://www.w3.org/1998/Math/MathML">
  <body>
    <p>Here is a diagram:</p>
    <svg:svg width="100" height="100">
      <svg:circle cx="50" cy="50" r="40" fill="blue"/>
    </svg:svg>
    <p>And an equation:</p>
    <math:math>
      <math:msup>
        <math:mi>x</math:mi>
        <math:mn>2</math:mn>
      </math:msup>
    </math:math>
  </body>
</html>
```

## Well-Formedness Rules — Summary

An XML document is well-formed if and only if:

<Steps>
1. **Has an XML declaration** (recommended but optional for 1.0).
2. **Has exactly one root element** that contains all other content.
3. **All tags are closed** — either with a matching end tag or as `<tag/>`.
4. **Tags are properly nested** — no overlapping.
5. **Attribute values are quoted** — single or double quotes.
6. **No duplicate attributes** on the same start tag.
7. **Special characters are escaped** — `&lt;`, `&amp;`, etc.
8. **Element names are valid** — no spaces, start with letter or underscore.
9. **Comments and PIs** do not contain illegal sequences (`--` in comments).
10. **Encoding declared** matches the actual byte encoding of the file.
</Steps>

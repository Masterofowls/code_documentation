XML (eXtensible Markup Language) is a flexible, human-readable, machine-parseable text format for storing and transporting structured data. Defined by the W3C in 1998, XML underpins hundreds of internet standards — from RSS feeds and SOAP web services to SVG graphics and Android resource files.

Unlike HTML (which is a fixed-vocabulary markup language for documents), XML lets you **define your own tags** to describe any data domain you choose.

```xml
<?xml version="1.0" encoding="UTF-8"?>
<library>
  <book id="978-0-13-110362-7">
    <title>The C Programming Language</title>
    <author>Brian W. Kernighan</author>
    <year>1988</year>
    <available>true</available>
  </book>
</library>
```

## XML vs JSON vs YAML

Choosing the right serialization format depends on your use case. Here is a side-by-side comparison:

| Feature | XML | JSON | YAML |
|---|---|---|---|
| **Human readable** | ✅ Verbose but clear | ✅ Compact | ✅ Most readable |
| **Machine parseable** | ✅ | ✅ | ✅ |
| **Comments** | ✅ `<!-- -->` | ❌ None | ✅ `#` |
| **Namespaces** | ✅ Native support | ❌ | ❌ |
| **Schema validation** | ✅ DTD, XSD, RelaxNG | ✅ JSON Schema | ✅ (limited) |
| **Attributes vs elements** | ✅ Both | ❌ Keys only | ❌ Keys only |
| **Transformation language** | ✅ XSLT | ❌ | ❌ |
| **Query language** | ✅ XPath, XQuery | ✅ JSONPath (informal) | ❌ |
| **Binary encoding** | EXI | BSON, MessagePack | ❌ |
| **Mixed content** | ✅ (text + elements) | ❌ | ❌ |
| **Document-centric** | ✅ Excellent | ❌ | ⚠️ Limited |
| **Config files** | ⚠️ Verbose | ✅ | ✅ Excellent |
| **Web APIs** | SOAP, XML-RPC | ✅ REST default | ❌ Rare |
| **Streaming parse** | ✅ SAX, StAX | ✅ | ⚠️ |
| **File size** | Larger (tags repeat) | Moderate | Smallest |

:::tip[When to Choose XML]
Prefer XML when you need **namespaces**, **mixed content** (prose with inline markup), **XSLT transformation pipelines**, schema validation with XSD, or when working with existing XML-based standards like SOAP, SVG, or XHTML.
:::

:::caution[When to Avoid XML]
Avoid XML for simple REST APIs consumed by JavaScript frontends. JSON is lighter and natively parsed by browsers. For configuration files edited by humans, YAML is more ergonomic.
:::

## XML Use Cases

### Configuration Files

Many build tools and enterprise frameworks use XML for configuration:

```xml
<!-- Maven pom.xml -->
<project xmlns="http://maven.apache.org/POM/4.0.0">
  <modelVersion>4.0.0</modelVersion>
  <groupId>com.example</groupId>
  <artifactId>my-app</artifactId>
  <version>1.0.0</version>
  <dependencies>
    <dependency>
      <groupId>org.springframework.boot</groupId>
      <artifactId>spring-boot-starter-web</artifactId>
      <version>3.2.0</version>
    </dependency>
  </dependencies>
</project>
```

### Data Exchange — SOAP Web Services

SOAP (Simple Object Access Protocol) uses XML envelopes for inter-system communication:

```xml
<soap:Envelope xmlns:soap="http://schemas.xmlsoap.org/soap/envelope/">
  <soap:Header>
    <auth:Token xmlns:auth="http://example.com/auth">abc123</auth:Token>
  </soap:Header>
  <soap:Body>
    <GetWeather xmlns="http://example.com/weather">
      <City>London</City>
    </GetWeather>
  </soap:Body>
</soap:Envelope>
```

### SVG — Scalable Vector Graphics

SVG is pure XML and can be embedded directly in HTML:

```xml
<svg xmlns="http://www.w3.org/2000/svg" viewBox="0 0 100 100">
  <circle cx="50" cy="50" r="40" fill="#4f46e5" stroke="#312e81" stroke-width="3"/>
  <text x="50" y="55" text-anchor="middle" fill="white" font-size="14">Hello</text>
</svg>
```

### RSS / Atom Feeds

```xml
<?xml version="1.0" encoding="UTF-8"?>
<rss version="2.0">
  <channel>
    <title>My Blog</title>
    <link>https://example.com</link>
    <description>Latest articles</description>
    <item>
      <title>First Post</title>
      <link>https://example.com/posts/first</link>
      <pubDate>Mon, 01 Jan 2024 00:00:00 GMT</pubDate>
      <guid>https://example.com/posts/first</guid>
    </item>
  </channel>
</rss>
```

### Android Layouts & Resources

Android UI is defined in XML:

```xml
<?xml version="1.0" encoding="utf-8"?>
<LinearLayout xmlns:android="http://schemas.android.com/apk/res/android"
    android:layout_width="match_parent"
    android:layout_height="match_parent"
    android:orientation="vertical">

  <TextView
      android:id="@+id/heading"
      android:layout_width="wrap_content"
      android:layout_height="wrap_content"
      android:text="@string/app_name"
      android:textSize="24sp" />

  <Button
      android:id="@+id/submit"
      android:layout_width="match_parent"
      android:layout_height="wrap_content"
      android:text="@string/btn_submit" />
</LinearLayout>
```

## Well-Formed vs Valid XML

These are two distinct concepts that are frequently confused:

### Well-Formed

A well-formed XML document follows all **syntactic rules** of XML:

- Has exactly one root element
- All elements are properly nested (no overlapping)
- All elements are closed (or self-closing)
- All attribute values are quoted
- Case is consistent (`<Name>` ≠ `<name>`)
- Special characters are escaped (`&amp;`, `&lt;`, etc.)

```xml
<!-- Well-formed ✅ -->
<root>
  <child attr="value">text</child>
</root>

<!-- NOT well-formed ❌ — overlapping tags -->
<b><i>text</b></i>

<!-- NOT well-formed ❌ — unquoted attribute -->
<img src=photo.jpg />
```

### Valid

A **valid** XML document is well-formed AND conforms to a schema (DTD, XSD, or RelaxNG). Validity means the structure, element names, attribute names, data types, and cardinality all match the declared schema.

```xml
<!-- Valid against a schema requires:
     1. Being well-formed
     2. Element names matching schema declarations
     3. Attribute types correct
     4. Required elements present
     5. Data types matching (e.g., xs:integer for <year>) -->
```

:::tip[Parser Behaviour]
An XML parser that checks only well-formedness is called a **non-validating parser**. A **validating parser** also checks the document against a schema. Most browser XML parsers are non-validating.
:::

## The XML Ecosystem

XML is not a single technology — it is the foundation for an entire family of W3C standards:

| Standard | Purpose | Status |
|---|---|---|
| **XML 1.0 / 1.1** | Core specification | W3C Recommendation |
| **XML Namespaces** | Avoid name collisions | W3C Recommendation |
| **DTD** | Document Type Definition (schema) | Part of XML 1.0 spec |
| **XSD (XML Schema)** | Powerful schema language | W3C Recommendation |
| **RelaxNG** | Compact, elegant schema language | ISO standard |
| **Schematron** | Rule-based validation (XSLT-based) | ISO standard |
| **XPath 1.0 / 2.0 / 3.1** | Query language for XML trees | W3C Recommendation |
| **XQuery 1.0 / 3.1** | Full query language (like SQL for XML) | W3C Recommendation |
| **XSLT 1.0 / 2.0 / 3.0** | XML transformation language | W3C Recommendation |
| **DOM Level 3** | Tree-based XML API | W3C Recommendation |
| **SAX 2.0** | Event-based streaming parser API | De-facto standard |
| **StAX** | Pull-based streaming parser (Java) | JSR 173 |
| **XLink** | Linking between XML documents | W3C Recommendation |
| **XInclude** | XML file inclusion mechanism | W3C Recommendation |
| **XHTML** | HTML reformulated as XML | W3C Recommendation |
| **SVG** | Scalable Vector Graphics | W3C Recommendation |
| **MathML** | Mathematical Markup Language | W3C Recommendation |

### Parsing Models

```
┌─────────────────────────────────────────────────────┐
│                   XML Document                      │
└─────────────┬───────────────────┬───────────────────┘
              │                   │
    ┌─────────▼──────┐   ┌────────▼────────┐
    │  DOM Parser    │   │  SAX / StAX     │
    │  (tree in RAM) │   │  (event stream) │
    └─────────┬──────┘   └────────┬────────┘
              │                   │
    ┌─────────▼──────┐   ┌────────▼────────┐
    │ querySelector  │   │ startElement()  │
    │ XPathEvaluator │   │ endElement()    │
    │ XSLT transform │   │ characters()    │
    └────────────────┘   └─────────────────┘
```

**DOM** loads the entire document into memory as a traversable tree — best for random access and manipulation.

**SAX** fires events as the parser streams through the document — best for very large files where you cannot load everything into RAM.

**StAX** (Java) is a pull parser — your code drives the parse, requesting tokens one at a time — good balance of memory efficiency and usability.

## XML Versions

| Version | Year | Key Changes |
|---|---|---|
| **XML 1.0** (5th edition) | 1998 / 2008 | Original spec, most widely supported |
| **XML 1.1** | 2004 | Allows more Unicode name characters, end-of-line normalisation changes |
| **XML Namespaces 1.0** | 1999 | Added `xmlns` prefix mechanism |
| **XML Namespaces 1.1** | 2006 | Undeclaring namespace prefixes |

:::caution[Stick with XML 1.0]
The vast majority of parsers and tools target XML 1.0. Use XML 1.1 only if you specifically need to include Unicode characters from Plane 1+ in element and attribute names — which is extremely rare in practice.
:::

## XML in Modern Development

Despite JSON's dominance in REST APIs, XML is far from obsolete:

- **Build tools** — Maven, Ant, MSBuild, and Gradle all use XML
- **Office formats** — Every `.docx`, `.xlsx`, and `.pptx` is a ZIP of XML files
- **Configuration** — Spring Framework, Log4j, Android, `.NET`, and Tomcat all use XML config
- **Industry standards** — HL7/FHIR (healthcare), FpML (finance), ACORD (insurance), and MISMO (mortgages) use XML schemas
- **Publishing** — DITA, DocBook, TEI, and EPUB are XML-based publishing formats
- **Data integration** — EDI, SWIFT, and countless B2B data exchange standards use XML

:::tip[XML Is the Language of Interoperability]
When two enterprise systems from different vendors and different decades need to talk to each other, they almost always speak XML. Its longevity comes from its self-describing nature, strong schema tooling, and the existence of XSLT for adapting one vocabulary to another.
:::
